# 09. Instagram — Photo Sharing and Feed

**Type:** social-media
**Difficulty:** G4
**Frequency:** High — photo storage, CDN, feed ranking

---

## Problem Statement

Design Instagram. Users can upload photos and videos, follow other users, and see a feed of posts from people they follow. The system must handle 1B MAU, 100M photos uploaded daily, and serve the home feed in under 200ms with personalized ranking.

---

## Functional Requirements

**Must have:**
- Upload photo/video with caption
- Follow / unfollow users
- Home feed: posts from followed users, reverse-chronological or ranked
- Like and comment on posts
- User profile: grid of posts, follower/following count

**Should have:**
- Stories (24-hour disappearing posts)
- Explore page (discover new content)
- Image filters and basic editing

**Out of scope:**
- DMs (see WhatsApp design)
- Reels algorithm deep-dive
- Live video

---

## Non-Functional Requirements

- **Scale:** 1B MAU, 500M DAU, 100M uploads/day
- **Availability:** 99.99%
- **Latency:** Feed load < 200ms p99
- **Storage:** Photos and videos in object storage; infinite retention
- **Consistency:** Eventual — slight feed delay acceptable

---

## Capacity Estimation

```
Upload throughput:
  100M uploads/day ÷ 86,400 sec = 1,157/sec average; peak ~5,000/sec
  Photo size (original): avg 3MB (modern smartphone, HEIC/JPEG)
  Video size (short clip): avg 20MB
  Assuming 90% photos + 10% videos:
    90M × 3MB + 10M × 20MB = 270GB + 200GB = 470GB/day raw uploads

Storage after processing (multiple variants per image):
  Per photo (original + thumb 150px + medium 640px + large 1080px):
    ~3MB + 30KB + 300KB + 1MB = ~4.3MB total
  Per video: ~30MB (compressed + transcoded)
  Daily: 90M × 4.3MB + 10M × 30MB = 387GB + 300GB = 687GB/day
  10-year storage: 687GB × 3,650 = ~2.5PB (plus 30% growth = ~10PB with years)

CDN serving bandwidth:
  500M DAU × 50 photos viewed/day × 200KB avg (medium quality on CDN) = 5PB/day
  5PB/day ÷ 86,400 sec = 57.9GB/sec ≈ 463Gbps peak (with 3× peak factor)
  → Enterprise CDN (Fastly/Akamai) required

Social graph:
  1B users × avg 300 follows (follows, not followers) = 300B edges
  Per edge: 16 bytes (two UUIDs) = 4.8TB just for edges (manageable in sharded Cassandra)

Fan-out capacity:
  1,157 uploads/sec × avg 300 followers = 347K fan-out writes/sec to Redis caches
  Celebrity (50M followers) × 1 upload = 50M fan-out writes → use pull model
```

## High-Level Architecture

```
Upload Path:
[Client App]
  ├─ 1. POST /v1/media/upload-url ──→ [Upload API] ──→ [PostgreSQL: create post record]
  │                                               └── Returns S3 presigned URL + media_id
  ├─ 2. PUT {presigned_url} ──────────────────→ [S3: raw-uploads/]
  ├─ 3. POST /v1/posts ───────────────────────→ [Post API] ──→ [Kafka: post-created]
  │                                                                   │
  │                                                      [Media Processing Workers]
  │                                                       ├── Resize variants
  │                                                       ├── pHash dedup check
  │                                                       ├── Content moderation
  │                                                       └── Upload to CDN-backed S3
  │                                                                   │
  │                                                      [Fan-out Service]
  │                                                       └── ZADD to follower Redis caches

Feed Read Path:
[Client] ──→ [Feed API]
                │
                ├── Redis ZREVRANGEBYSCORE feed:{user_id} → post_ids (push content)
                ├── Cassandra: recent posts from celebrities followed (pull content)
                ├── Merge + deduplicate
                ├── [Ranking Service] → score 200 candidates → top 20
                └── [Post Service] → hydrate post data (batch fetch)
                     ├── Redis cache (hot posts < 1h)
                     └── PostgreSQL (older posts)
```

**Core insight:** Same hybrid push/pull pattern as Twitter. Push for regular users (< 1M followers), pull for celebrities (> 1M followers).

```

  id, created_at,

# Engagement
```

## Data Model

```sql
-- PostgreSQL: posts (primary metadata store)
posts (
  post_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES users(user_id),
  caption      TEXT,
  location     TEXT,
  like_count   INT DEFAULT 0,      -- denormalized for fast reads; updated async
  comment_count INT DEFAULT 0,
  status       TEXT DEFAULT 'published',  -- 'processing' | 'published' | 'deleted'
  created_at   TIMESTAMPTZ DEFAULT now(),
  INDEX (user_id, created_at DESC)     -- profile page: user's posts in order
)

-- Post media items (supports carousel posts with multiple images)
post_media (
  post_id    UUID REFERENCES posts(post_id),
  media_id   UUID NOT NULL,
  position   INT NOT NULL,          -- order in carousel (0-indexed)
  media_type TEXT NOT NULL,         -- 'photo' | 'video'
  s3_paths   JSONB NOT NULL,        -- { thumb: "s3://...", medium: "...", large: "..." }
  phash      TEXT,                  -- perceptual hash for dedup detection
  PRIMARY KEY (post_id, position)
)

-- Social graph: follows (Cassandra — 300B edges won't fit in PostgreSQL without heavy sharding)
follows (
  follower_id   UUID PARTITION KEY,
  followee_id   UUID CLUSTERING KEY,
  created_at    TIMESTAMP,
  PRIMARY KEY (follower_id, followee_id)
)
followers_of (
  followee_id   UUID PARTITION KEY,  -- "who follows me?" (needed for fan-out)
  follower_id   UUID CLUSTERING KEY,
  PRIMARY KEY (followee_id, follower_id)
)

-- Likes (Cassandra: write-heavy, large volume)
likes (
  post_id    UUID PARTITION KEY,
  user_id    UUID CLUSTERING KEY,
  created_at TIMESTAMP,
  PRIMARY KEY (post_id, user_id)
)

-- Redis: per-user feed cache
-- Key:    feed:{user_id}
-- Type:   Sorted Set (score = post_created_at Unix ms, member = post_id)
-- TTL:    7 days (inactive users' feeds expire; rebuild on next login)
-- Size:   Keep latest 1000 post IDs per user
ZADD feed:{user_id} {timestamp_ms} {post_id}
ZREMRANGEBYRANK feed:{user_id} 0 -1001   -- trim to 1000
EXPIRE feed:{user_id} 604800

-- Redis: post engagement counters (approximate, sync'd to DB every 30s)
Key:   post:likes:{post_id}     Type: String (INCR/DECR)
Key:   post:comments:{post_id}  Type: String
Key:   post:data:{post_id}      Type: Hash  (full post JSON for hot posts < 1h old)
```

## Deep Dive 1: Media Processing Pipeline

**Full pipeline from upload to CDN:**
```
Step 1: Client uploads raw photo to S3 via presigned URL
  S3 bucket: raw-uploads (private, no CDN)
  S3 event notification → SNS → SQS → Processing Workers

Step 2: Processing Worker downloads original from S3

Step 3: Generate image variants:
  Thumbnail: 150 × 150 pixels (square crop)
    → Profile grid, explore page, notification previews
  Medium:    640 pixels wide (maintains aspect ratio)
    → Home feed display (2× retina = 1280px actual)
  Large:     1080 pixels wide
    → Full-screen view, profile page large view
  Original:  Keep as-is (for potential future re-processing)

Step 4: Smart crop (thumbnail):
  Run face detection (MediaPipe Face Detection) on original
  If faces detected: center crop on the primary face
  If no faces: saliency map (gradient-based attention model) to find most interesting region
  → 150×150 crop is content-aware, not naive center crop

Step 5: Format conversion:
  Input: JPEG, PNG, HEIC, HEIF, WebP, GIF
  Output: WebP for modern browsers (30% smaller than JPEG at same quality)
          JPEG as fallback for older browsers
  Serve correct format based on HTTP Accept header:
    Accept: image/webp → serve .webp
    Otherwise → serve .jpg

Step 6: Compute perceptual hash (pHash):
  DCT-based pHash: 64-bit fingerprint of image content
  Invariant to: resizing, JPEG compression, minor color adjustments, slight crops
  Not invariant to: rotation, flips, heavy filters

Step 7: Content moderation:
  Run lightweight NSFW classifier on thumbnail (fast, < 50ms)
  If high confidence NSFW (score > 0.9): block immediately, return 422 to user
  If borderline (0.3 < score < 0.9): publish, flag for human review queue
  If clearly safe (score < 0.3): proceed without human review
  PhotoDNA hash check: compare pHash against CSAM database → legally mandatory,
    synchronous (must complete before post is marked 'published')

Step 8: Upload all variants to S3 (media-cdn bucket, public read)
  S3 path: media/{user_id}/{media_id}/{variant}.{ext}
  e.g.:    media/u123/m456/medium.webp
           media/u123/m456/medium.jpg  (fallback)

Step 9: Update post_media table with S3 paths
  Set post.status = 'published'
  Publish "post-published" event to Kafka → Fan-out Service
```

**CDN configuration:**
```
CDN (CloudFront) in front of media-cdn S3 bucket:
  Cache-Control: max-age=31536000, immutable
  → 1 year browser cache + CDN cache (media content is immutable once processed)

Cache key: {cdn_domain}/{user_id}/{media_id}/{variant}.{ext}
  → Each variant is separately cacheable

On delete: CloudFront invalidation for /media/{user_id}/{media_id}/*
  → Invalidates all variants for the deleted post
  → Costs $0.005 per 1000 invalidation paths → negligible

Format negotiation:
  CloudFront Lambda@Edge: inspect Accept header → rewrite URL to .webp or .jpg
  Or: use CloudFront's built-in image optimization (Resize + format conversion at edge)
  → Eliminates need for explicit format variants; serve-time conversion
```

**G5:** The pHash deduplication check (Step 6) has a correctness problem under high write load. When two users upload the same viral meme simultaneously, both trigger pHash lookups against the existing pHash index before either has been written. Both pass the dedup check, both get published, both end up in the index. Now there are two copies. The pHash lookup needs a distributed lock or a "upsert-with-check" semantic. The practical fix: store pHash → canonical_media_id in Redis with SETNX semantics: first uploader wins (gets canonical_media_id = their media_id), subsequent uploaders with matching pHash are redirected to the canonical media object. The lock must be held while the canonical media is still being processed — use a two-phase write: reserve the pHash slot before processing, confirm after. Hamming distance comparison for near-duplicates (not just exact matches) requires locality-sensitive hashing (LSH) across the pHash space — this is the harder problem, requiring an approximate nearest-neighbor index over 100B+ pHashes.

## Deep Dive 2: Ranked Feed

**Why ranking instead of pure chronological:**
```
Instagram moved to ranked feed in 2016 → 40% improvement in engagement
Pure chronological problems:
  - User follows 1,000 accounts → misses 90% of content in a normal day
  - Recent posts dominate even if they're less relevant to the viewer
  - Accounts that post frequently drown out infrequent but high-quality posters
```

**Ranking signals:**
```
Relationship signals (strongest):
  - Do you DM this person? (high intimacy)
  - Do you regularly like/comment on their posts? (engagement history)
  - Have you recently viewed their profile? (interest signal)
  - Did they like your recent post? (mutual interest)

Post signals (recency + quality):
  - Post age (newer is better, all else equal)
  - Like velocity in first 30 minutes (early engagement predicts overall performance)
  - Comment-to-like ratio (high comments = polarizing or compelling content)
  - Save rate (saves indicate strong interest, not just passive like)

Interest model (content-based):
  - User's historical engagement by category (travel, food, fitness, art)
  - Visual embedding similarity: CLIP model → 512-dim embedding per image
    Posts similar to past-engaged posts score higher
  - Caption keywords matching user interest graph

Context:
  - Time of day (morning browse = different content preference than evening)
  - Device type (phone vs tablet affects media preference)
```

**Ranking architecture:**
```
Step 1: Candidate pool (200 candidates)
  From Redis feed cache: 150 post_ids (pre-pushed via fan-out)
  From celebrity pull: 30 post_ids (fetched at read time from Cassandra)
  From explore blending (optional): 20 post_ids from accounts not followed
  Total: 200 candidates

Step 2: Feature extraction (parallel, < 20ms total)
  For each candidate post:
    a. Engagement signals: Redis GET post:likes:{post_id} (< 1ms)
    b. Like velocity: Redis ZCOUNT likes_by_hour:{post_id} (recent likes in last 1h)
    c. Relationship score: Feature Store lookup user_affinity:{user_id}:{author_id}
       → pre-computed daily by ML pipeline, served in < 5ms
    d. Content category: post_media.category (stored in PostgreSQL, cached in Redis)
  Total feature vector: ~50 features per candidate

Step 3: Ranking model inference
  Model: LightGBM (gradient boosted trees) — 200 trees, 64 leaves each
  Input: 50-feature vector per candidate
  Output: engagement probability score ∈ [0, 1]
  Inference time: < 2ms for 200 candidates (single-threaded, vectorized)

Step 4: Post-ranking filters
  Diversity: no more than 3 posts from the same user in top 20
  Freshness: at least 30% of top 20 must be from last 24 hours
  Dedup: exclude posts user has already seen (session-level seen-set in Redis)

Step 5: Return top 20 from filtered ranked list
  next_cursor = opaque token encoding last post's rank score + position
  → Stable pagination even as rankings change

Fallback: if ranking service responds in > 50ms → return chronological top 20
```

## Reliability and Operations

- **Upload failure:** Client assigns a `client_upload_id` and retries with the same ID. Server deduplicates by `client_upload_id` — if post was already created, return existing `post_id`. S3 multipart upload tracks progress per chunk; client resumes from last uploaded chunk on network failure.
- **Fan-out failure:** Kafka consumer retries on failure (offset not committed until fan-out batch completes). ZADD is idempotent — adding the same post_id to a sorted set with the same score is a no-op. Safe to replay Kafka messages.
- **Feed cache miss (empty or expired):** Detect empty feed cache (`ZCARD feed:{user_id} == 0`). Serve first page from DB immediately (Cassandra read for each followee's recent posts, merged). Simultaneously trigger async cache rebuild. Show loading placeholder for subsequent pages while cache warms.
- **Ranking service unavailable:** Fall back to chronological order from Redis feed cache. No user-visible error, just slightly less personalized feed.

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| S3 presigned upload | Upload through API servers | Direct to S3: eliminates 4.7GB/sec of upload bandwidth from our fleet; API servers would need massive network capacity just to proxy uploads. Only downside: less control over upload progress (mitigated by multipart + resume). |
| Ranked feed | Chronological | Ranked: 40% higher engagement (Instagram's actual measurement); surfaces best content across all followed accounts. Chronological: simpler, expected by users pre-2016, no ML infrastructure needed. The tradeoff is engineering complexity + user perception of fairness vs engagement. |
| Cassandra for follows (300B edges) | PostgreSQL | 300B rows in PostgreSQL would require extreme sharding (1000+ shards). Cassandra partitions by follower_id natively — fan-out lookup ("who follows user X?") is a single-partition scan. PostgreSQL: richer queries possible, but the follow graph doesn't need rich queries. |
| Async like_count updates | Synchronous update per like | Synchronous: `UPDATE posts SET like_count += 1` on every like → serializable writes, single row = bottleneck at 1M likes/sec on viral posts. Async (via Kafka + batch aggregation): decoupled, scales to any like rate, slight delay (< 30s) in displayed count. |

## G5

**What a G5 engineer adds beyond the above:**

1. **Near-duplicate detection at scale:** Users re-upload viral content constantly. With 100M uploads/day, a single viral meme might be uploaded 10K times in a day. pHash Hamming distance search over 100B+ existing hashes is an approximate nearest-neighbor problem. Build an LSH (Locality-Sensitive Hashing) index: split the 64-bit pHash into 4 × 16-bit bands; for each band, hash it to a bucket. Near-duplicates (Hamming distance < 10) will collide in at least one bucket. This reduces the search from O(10B comparisons) to O(bucket_size × 4) per query. Store the LSH index in Redis sorted sets. On upload: query all 4 bands, union the results, compute exact Hamming distance for candidates, flag if < 10. This enables deduplication at scale without a brute-force scan — the same technique used by Perceptual Image Search at Pinterest.

2. **Stories architecture (ephemeral with hard TTL):** Stories differ from posts in a fundamental way: 24-hour deletion is a hard product contract, not a soft delete. Architecture: separate Cassandra keyspace with `default_time_to_live = 86400` at the row level — Cassandra automatically expires rows, no cron job needed. Stories are pre-fetched for your following list on app open: `SELECT * FROM stories WHERE user_id IN (following_list) AND created_at > now() - 24h`. Story views tracked separately (SET of viewer_ids per story — visible to creator). On expiry: Cassandra TTL removes the row; CDN cache invalidated via a separate cleanup job that runs the CDN purge API for story media URLs before TTL fires. Archive feature: if creator enables "Archive Stories", a pre-expiry Kafka consumer copies the row to permanent storage before TTL fires.

3. **Explore page candidate generation:** Explore (discover content from accounts you don't follow) requires a different candidate generation strategy than home feed. Three sources: (a) Collaborative filtering: "users similar to you liked these posts" — compute user similarity via matrix factorization on the likes matrix (ALS weekly batch job), find 500 similar users, take their recent liked posts. (b) Content-based: CLIP image embeddings for posts you've liked → ANN search (FAISS index) for visually similar posts from your non-following graph. (c) Trending in your network: posts going viral among accounts 2 hops away from you. Merge and re-rank these 1,000 candidates with the same ranking model as home feed but with an "novelty bonus" for accounts you've never interacted with. Cache per-user explore candidates (Redis, 1h TTL) to avoid re-running the expensive candidate generation on every explore page open.

4. **CDN cost optimization at Instagram scale:** 5PB/day of media delivery is the single largest cost line. Optimization layers: (a) AVIF encoding: 50% smaller than JPEG at equivalent quality; serve to Chrome/Android (90% of traffic). WebP for Safari. JPEG as final fallback. Implement at CDN edge (CloudFront image optimization) to avoid storing all three formats. (b) Progressive JPEG: send low-resolution preview immediately; full resolution loads as network delivers. User perceives faster load even if total bytes are the same. (c) Adaptive quality: if viewer is on 3G (detected from connection speed header), serve medium quality even for full-screen view. (d) Lazy loading: only fetch images visible in viewport; cancel fetches for posts that scroll past before loading completes. (e) Cold storage for old media: photos not accessed in 12 months move to S3 Glacier Instant Retrieval (70% cost reduction); CDN miss fetches from Glacier (~100ms additional latency, acceptable for a 1-year-old post).

5. **Moderation at 100M uploads/day:** Fully automated moderation is mandatory — no human team scales to 1,157 uploads/sec. Pipeline: (a) PhotoDNA CSAM check: synchronous, must complete before post publishes. Hash comparison against known CSAM database — legal requirement in most jurisdictions. (b) NSFW classifier: lightweight MobileNet on thumbnail (30ms) — immediate block for obvious violations. Full-resolution EfficientNet for borderline cases (async, may delay publication by 5s). (c) Account reputation gate: accounts created < 24 hours ago get synchronous full-resolution check (higher false positive tolerance for new accounts). (d) Human review queue for confidence 0.3–0.7: prioritized by estimated reach (accounts with 100K+ followers get faster human review). (e) Hash-matching for re-uploads of known-bad content: once content is moderated and removed, its pHash is added to a banned-hash database. Future uploads with matching pHash are blocked instantly without running the classifier.
