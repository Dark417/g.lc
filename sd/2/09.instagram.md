# 09. Instagram — Photo Sharing and Feed

**Type:** social-media
**Difficulty:** L4
**Frequency:** High — photo storage, CDN, feed ranking

---

## Problem Statement

Design Instagram. Users can upload photos and videos, follow other users, and see a feed of posts from people they follow. The system must handle 1B MAU, 100M photos uploaded daily, and serve the home feed in under 200ms with personalized ranking.

---

## Functional Requirements

**Must have:**
- Upload photo/video with caption
- Follow / unfollow users
- Home feed: posts from followed users, reverse-chronological or ranked
- Like and comment on posts
- User profile: grid of posts, follower/following count

**Should have:**
- Stories (24-hour disappearing posts)
- Explore page (discover new content)
- Image filters and basic editing

**Out of scope:**
- DMs (see WhatsApp design)
- Reels algorithm deep-dive
- Live video

---

## Non-Functional Requirements

- **Scale:** 1B MAU, 500M DAU, 100M uploads/day
- **Availability:** 99.99%
- **Latency:** Feed load < 200ms p99
- **Storage:** Photos and videos in object storage; infinite retention
- **Consistency:** Eventual — slight feed delay acceptable

---

## Capacity Estimation

```
Uploads: 100M/day = 1,150/sec; peak ~5,000/sec
Photo size avg: 2MB → 100M × 2MB = 200TB/day
Video avg: 10MB → assume 10% uploads are video → 10M × 10MB = 100TB/day
Total media: ~300TB/day

CDN serving: 500M DAU × 50 photos viewed/day × 200KB (resized) = 5PB/day
→ Need massive CDN footprint

Follow graph: 1B users × avg 500 follows = 500B edges = ~4TB (just IDs)
Feed fan-out: 1,150 uploads/sec × avg 500 followers = 575K fan-out writes/sec to caches
```

## High-Level Architecture

```
Upload:
[Client] ──→ [Upload Service] ──→ [S3 (raw)] ──→ [Kafka] ──→ [Processing Workers]
                                                                      │
                                                      [S3 (resized: thumb, medium, large)]
                                                                      │
                                                              [CDN (CloudFront)]
                                                                      │
                                                              [Metadata DB (PostgreSQL)]

Feed:
[Client] ──→ [Feed API] ──→ [Feed Cache (Redis)] ──→ [Ranking Service] ──→ [Feed DB]
                │
         [Fan-out Service] ──→ pushes post_ids on new upload
```

**Core insight:** Same hybrid push/pull pattern as Twitter. Push for regular users, pull for celebrities (accounts with >1M followers).

```

POST /v1/posts/{id}/like

GET /v1/posts/{id}/comments?cursor=
POST /v1/posts/{id}/comments  Body: { text }
```

## Data Model

```sql
-- PostgreSQL: posts
posts (
  post_id      UUID PRIMARY KEY,
  user_id      UUID NOT NULL,
  caption      TEXT,
  location     TEXT,
  like_count   INT DEFAULT 0,
  created_at   TIMESTAMPTZ
)

-- Media items (a post can have multiple photos)
post_media (
  post_id   UUID,
  media_id  UUID,
  position  INT,  -- order in carousel
  media_type TEXT,  -- 'photo' | 'video'
  s3_paths  JSONB,  -- { thumb: '...', medium: '...', large: '...' }
  PRIMARY KEY (post_id, position)
)

-- Social graph (Cassandra or PostgreSQL with pg_partman)
follows (
  follower_id  UUID,
  followee_id  UUID,
  created_at   TIMESTAMPTZ,
  PRIMARY KEY (follower_id, followee_id)
)

-- Redis: per-user feed (post IDs)
Key: feed:{user_id}
Type: Sorted Set (score=timestamp, member=post_id)
TTL: 7 days
Max size: 1000 post IDs

-- Likes (Cassandra for high write)
likes (
  post_id   UUID PARTITION KEY,
  user_id   UUID CLUSTERING KEY,
  created_at TIMESTAMP
)
```

## Deep Dive 1: Media Processing Pipeline

```
1. Client uploads original photo to S3 (presigned URL)
2. S3 event → Kafka topic "media-uploaded"
3. Processing workers:
   a. Download original from S3
   b. Generate variants:
      - Thumbnail: 150×150 (for profile grid)
      - Medium: 640px wide (feed display)
      - Large: 1080px wide (full view)
      - Original: keep as-is
   c. Apply smart crop: use face detection / saliency maps to center crop
   d. Upload all variants to S3
   e. Generate perceptual hash (pHash) for duplicate detection
   f. Run content moderation (NSFW classifier)
4. Update post_media with S3 paths
5. Set post status to 'published'
6. Fan-out event to feed service
```

**CDN configuration:**
- All media served through CDN (CloudFront)
- Cache-Control: max-age=31536000 (1 year) — media is immutable
- On delete: CDN invalidation for all variants
- Cache key: `{bucket}/{variant}/{media_id}.jpg`

## Deep Dive 2: Ranked Feed

Instagram moved away from pure chronological feed to ranked feed in 2016.

**Ranking signals:**
- Relationship strength (do you DM this person? comment often?)
- Post engagement velocity (how many likes/comments in first 30 min?)
- Interest model (do you usually like travel photos?)
- Recency (newer posts ranked higher, all else equal)
- Story completeness (accounts you regularly view stories from)

**Architecture:**
```
Candidate pool: pre-fetched post_ids from Redis feed cache (200 candidates)
Feature extraction: for each candidate, fetch:
  - engagement signals from Redis (like velocity in last 1h)
  - relationship score from feature store (user A <-> author affinity)
  - content category from post metadata
Ranking model: lightweight gradient boosted model (< 5ms inference)
Post-processing: diversity filter (no more than 3 posts from one person)
Return top 20
```

**Fallback:** If ranking service is slow, return chronological subset.

---

## Reliability and Operations

- **Upload failure:** Client retries with same media_id (idempotent). S3 multipart upload tracks progress.
- **Fan-out failure:** Kafka consumer retries; fan-out is idempotent (Redis ZADD with NX or re-adding same post_id doesn't duplicate).
- **Feed recovery:** If user's feed cache is empty (expired), trigger async rebuild from DB. Show loading state or serve empty feed with "pull to refresh" hint.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| S3 presigned uploads | Upload through our servers | Direct S3: no bandwidth cost on us; indirect: more control, progress tracking |
| Ranked feed | Chronological | Ranked: higher engagement; chronological: simpler, user expects it |
| Cassandra for follows | PostgreSQL | Cassandra: scales follow graph easily; PG: easier joins, harder to shard |

## L5 Additions

1. **Near-duplicate detection:** Users re-upload the same photo. Compute perceptual hash (pHash, dHash) on upload — 64-bit fingerprint. Hamming distance < 10 → near-duplicate. Use locality-sensitive hashing (LSH) to efficiently find near-duplicates among 100B+ existing photos. On detection: link to original, apply same moderation status, deduplicate in CDN (serve same URL). Saves storage and improves copyright enforcement.

2. **Stories architecture:** Stories are ephemeral — 24-hour TTL. Different from posts: no permanent feed entry, stored separately, expiry is hard requirement (not just soft delete). Architecture: separate Cassandra keyspace with TTL column. Stories are pre-fetched for your following list on app open. Story views are tracked but not shown as permanent counts. On expiry, Cassandra TTL auto-deletes. Archive: if user enables "archive stories", move to separate permanent store before TTL fires.

3. **Explore page at scale:** Explore shows personalized content from accounts you don't follow. This requires: (a) candidate generation — collaborative filtering (users like you watch X → show X), content-based (your interest graph → find similar photos), editorial highlights. (b) Real-time signals — what's trending in your network. (c) Deduplication — don't show posts already in your feed. Runs as a separate ML inference pipeline (not the same as home feed ranking). Separate Redis cache key per user for explore candidates.

4. **CDN cost optimization:** Media storage and delivery is the largest cost. Strategies: (a) Encode photos to AVIF/WebP — 30-50% smaller than JPEG at same quality. Serve format based on Accept header. (b) Lazy load images below the fold — browser fetches only when visible. (c) Adaptive quality: if user is on cellular, serve medium quality even for full-screen view. (d) Hot/cold tiering: photos not accessed in 12 months → S3 Glacier Instant Retrieval (70% cost reduction). CDN TTL ensures old photos are evicted from edge.

5. **Moderation at scale:** 100M uploads/day → automated moderation is mandatory. Pipeline: real-time classifier runs on thumbnail immediately (blocks obvious NSFW). Full-resolution classifier runs async (may degrade post quality for borderline). Human review queue for classifier confidence 0.3–0.7. Account reputation score: new accounts with no history get stricter thresholds. Hash-matching against known CSAM database (PhotoDNA or similar) — legally required, must run synchronously before post is published.
