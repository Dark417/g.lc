# 12. AWS S3 — Object Storage

**Type:** storage
**Difficulty:** L5
**Frequency:** High — asked at infra/storage platform interviews

---

## Problem Statement

Design an object storage system like Amazon S3. Users can create buckets, upload objects (files up to 5TB), download them by key, and list objects in a bucket. The system must provide 99.999999999% (11 nines) durability, 99.99% availability, serve exabytes of data, and handle millions of requests per second.

---

## Functional Requirements

**Must have:**
- Create/delete buckets
- PUT object (key, value bytes, metadata)
- GET object by key
- DELETE object
- LIST objects in bucket (with optional prefix filter)
- Multipart upload for objects > 100MB

**Should have:**
- Object versioning (keep all versions of an object)
- Access control (bucket policies, IAM roles)
- Object lifecycle rules (auto-delete or archive after N days)
- Pre-signed URLs (time-limited, no-auth-required access)
- Server-side encryption

**Out of scope:**
- Block storage (EBS)
- File system semantics (EFS)
- CDN (CloudFront sits in front of S3)

---

## Non-Functional Requirements

- **Durability:** 99.999999999% — effectively zero data loss
- **Availability:** 99.99% for GET/PUT; 99.9% for LIST
- **Scale:** Exabytes of storage, millions of objects, millions of RPS
- **Latency:** GET p99 < 100ms for objects < 1MB (excluding transfer time)
- **Consistency:** Strong consistency for GET-after-PUT (Amazon updated this in 2020)

---

## Capacity Estimation

```
Assume 5 billion objects stored, avg 1MB each = 5PB logical data
Replication factor: 3x across AZs (or erasure coding ~1.5x) → ~7.5–15PB physical
  For simplicity: assume 3x replication = 15PB physical

Growth: 1PB/day new data for a large-scale deployment
  15PB / 10TB per disk = 1,500 disks for current data
  Plus overhead, OS, hot spares, future growth headroom: ~5,000 disks
  Spread across 100 storage nodes at 50TB usable each = 100 nodes minimum

Request throughput:
  GET QPS: 1M/sec across cluster
  PUT QPS: 100K/sec
  LIST QPS: 10K/sec (expensive — requires sorted range scan)
  HEAD QPS: 200K/sec (metadata-only, cheap)

Bandwidth:
  Inbound (PUT): 100K/sec × avg 100KB = 10GB/sec inbound
  Outbound (GET): 1M/sec × avg 100KB = 100GB/sec outbound
  Network: high-bandwidth inter-AZ links required for replication
```

## High-Level Architecture

```
[Client]
  │
  ├─ PUT /bucket/key ──→ [Gateway Layer (Stateless)] ──→ [Partition Service]
  │                               │                              │
  │                        [Auth/ACL Service]          [Partition Map (etcd)]
  │                               │                              │
  └─ GET /bucket/key ──────────── │                   [Storage Nodes]
                                  │                              │
                               [Metadata DB]          [Erasure Coding / Replication]
                                  │
                            [DynamoDB-style / Cassandra]
```

**Components:**
- **Gateway:** Stateless request routing. Authenticates requests (SigV4 signature verification), checks quota, enforces bucket policies, routes to correct storage partition. Horizontal scale: add more gateway instances behind load balancer.
- **Partition Service:** Maps (bucket, key) → set of storage node(s). Uses consistent hashing on `hash(bucket_id + key)` to assign partitions. Partition map stored in etcd (strongly consistent). Rebalances partitions when nodes join/leave.
- **Storage Nodes:** Store actual object data on local HDDs/SSDs. Each node manages multiple drives (4–16TB each). Data written using an append-only log format to minimize random I/O. Local SQLite index per drive maps object_id → (byte_offset, size).
- **Metadata DB:** Stores object metadata: key, bucket, size, etag, version_id, storage location, ACL, user-defined metadata. High QPS, must scale horizontally. Cassandra-style wide-column store partitioned by bucket_id.
- **Replication/Erasure Coding Service:** Ensures durability across nodes and Availability Zones. Monitors replication factor per object; initiates re-replication on node failure.

## API Design

```
# Object operations
PUT /bucket/key
Headers: Content-Type, Content-Length, Content-MD5, x-amz-server-side-encryption,
         x-amz-storage-class, x-amz-meta-{custom}
Body: object data (up to 5GB for single-part; up to 5TB via multipart)
Returns: 200 OK, ETag: "\"<md5_hex>\""

GET /bucket/key
Headers: Range: bytes=0-1023 (optional byte-range request for partial reads)
Returns: 200 OK + full object | 206 Partial Content (for Range request)
Headers: ETag, Content-Length, Content-Type, x-amz-version-id

DELETE /bucket/key
Returns: 204 No Content (or 200 OK with delete marker for versioned buckets)

HEAD /bucket/key
Returns: all response headers, zero-byte body — fast metadata check

GET /bucket?prefix=images/&delimiter=/&max-keys=1000&continuation-token=<token>
Returns: XML or JSON:
  { Contents: [{ Key, Size, ETag, LastModified, StorageClass }],
    CommonPrefixes: [{ Prefix }],  # "folders" when delimiter is "/"
    NextContinuationToken, IsTruncated }

# Multipart upload (for objects > 100MB)
POST /bucket/key?uploads
Returns: { UploadId }

PUT /bucket/key?partNumber=1&uploadId=<id>
Body: part data (5MB–5GB per part, except last part can be smaller)
Returns: ETag for this part

POST /bucket/key?uploadId=<id>
Body: <CompleteMultipartUpload> list of { PartNumber, ETag } pairs
Returns: final ETag and key

DELETE /bucket/key?uploadId=<id>  → abort, clean up partial parts
```

## Data Model

**Metadata DB (Cassandra-style, partitioned by bucket):**
```
object_metadata (
  bucket_id     UUID,            -- PARTITION KEY
  key           TEXT,            -- CLUSTERING KEY (sorted for LIST operations)
  version_id    UUID,            -- for versioned buckets; latest version is primary
  size_bytes    BIGINT,
  etag          TEXT,            -- MD5 of object (or multipart ETag)
  content_type  TEXT,
  created_at    TIMESTAMPTZ,
  storage_class TEXT,            -- STANDARD, STANDARD_IA, GLACIER, DEEP_ARCHIVE
  storage_node  TEXT,            -- primary storage node address
  chunk_map     JSONB,           -- multipart: { part_num: { node, offset, size } }
  acl           JSONB,           -- access control entries
  user_metadata JSONB,           -- x-amz-meta-* headers
  delete_marker BOOLEAN,         -- true = object deleted (for versioned buckets)
  PRIMARY KEY (bucket_id, key, version_id)
)

-- Bucket metadata (low volume, authoritative PostgreSQL)
buckets (
  id              UUID PRIMARY KEY,
  name            TEXT UNIQUE NOT NULL,  -- globally unique bucket name
  owner_id        UUID,
  region          TEXT,
  versioning      TEXT DEFAULT 'suspended',  -- 'enabled' | 'suspended'
  lifecycle_rules JSONB,  -- [{ filter: { prefix }, transitions, expiration }]
  encryption      JSONB,  -- { type: 'SSE-S3' | 'SSE-KMS', key_id? }
  replication     JSONB,  -- cross-region replication config
  created_at      TIMESTAMPTZ
)
```

**On-disk format per storage node:**
```
Each node: HDDs organized into "volumes" (one volume per physical disk)
Object data layout on disk:
  /data/volumes/{vol_id}/{object_uuid}  -- raw bytes, no FS metadata per object
  Append-only writes: new version = new file, old version retained until lifecycle rule
  Deletion: tombstone record in local SQLite; actual bytes reclaimed by compaction job

Per-volume SQLite index (rebuilt from disk on node recovery):
  CREATE TABLE object_index (
    object_uuid TEXT PRIMARY KEY,
    byte_offset  INTEGER,
    size_bytes   INTEGER,
    checksum     TEXT
  );

Compaction job (runs nightly):
  1. Scan all volumes for tombstoned objects older than retention period
  2. Build new data file without deleted objects
  3. Update SQLite index atomically
  4. Delete old data file
```

## Deep Dive 1: Durability — Erasure Coding vs Replication

**The durability math:**
```
Target: 99.999999999% durability = expected annual object loss rate < 10^-11

Disk annual failure rate (AFR): ~0.5–2% for enterprise HDDs
Probability of losing data = P(all copies fail before replacement)

With 3x replication across 3 AZs:
  P(all 3 fail in same period) = AFR^3 = 0.01^3 = 10^-6 per disk group
  This does NOT reach 11 nines without additional protections
```

**Replication (simple, used for hot/critical data):**
```
Store N identical copies across N different nodes/AZs (N=3 for S3 Standard)

Write path:
  PUT → primary node → primary writes to disk
        → replicate to secondary_1 (async or sync)
        → replicate to secondary_2 (async or sync)
  Ack to client when: 2 of 3 nodes confirm write (quorum write)

Read path:
  GET from any replica (nearest available node)
  On mismatch between replicas: trigger background re-replication from authoritative copy

Durability:
  Assuming correlated failure probability within AZ = small but non-zero
  Cross-AZ replication provides geo-durability against entire AZ outage
  Storage overhead: 3x (1TB object stored as 3TB physical)
```

**Erasure coding (S3 Standard class implementation):**
```
Reed-Solomon erasure coding: split object into K data chunks + M parity chunks
  Any K of (K+M) chunks can reconstruct the full object
  S3 Standard: approximately 6+3 or similar within an AZ, replicated across AZs

Concrete example (6+2 scheme):
  Object = 1GB
  Split into K=6 data chunks of 170MB each = 1020MB (slight padding)
  Compute M=2 parity chunks of 170MB each via Galois Field arithmetic
  Total: 8 chunks = 1360MB stored vs 1024MB original
  Overhead ratio: 8/6 = 1.33x (vs 3x for replication)
  Tolerance: can lose any 2 of 8 chunks and still reconstruct

Cross-AZ distribution:
  Place 8 chunks across 3 AZs (3 + 3 + 2 distribution)
  Can tolerate complete loss of 1 AZ (lose 3 chunks, still have 5 ≥ K=6? No.)
  So in practice: 3 AZs each with their own erasure-coded copies + cross-AZ erasure
  Real S3: combination of intra-AZ erasure + cross-AZ replication for 11 nines

Durability formula (simplified):
  P(object loss) ≈ C(K+M, M+1) × AFR^(M+1) × (recovery_window)
  With M=2, AFR=0.01, recovery_window=1 day: ≈ 10^-11 per object-year ✓
```

**Erasure coding read path:**
```
GET object:
  1. Fetch K of K+M chunks from storage nodes in parallel
  2. Verify checksum of each chunk
  3. Reconstruct original data (fast matrix multiply over Galois Field)
  4. Return to client

If a chunk is unavailable (node down):
  Fetch remaining K chunks from K+M-1 available nodes
  Reconstruct using parity: adds ~10-20ms decode latency
  Background: reconstruct missing chunk, write to replacement node (degraded read triggers repair)
```

**Tradeoff summary:**
```
Replication:
  Storage overhead: 3x
  Read latency: O(1) — just read from nearest copy
  Write latency: Quorum write, synchronous to 2 nodes
  Repair: simple copy operation

Erasure coding (6+2):
  Storage overhead: 1.33x — saves significant cost at exabyte scale
    At 1EB of logical data: 1EB × 1.33 = 1.33EB physical vs 1EB × 3 = 3EB for replication
    Savings: 1.67EB of storage at $0.023/GB/month = $38M/month saved
  Read latency: slightly higher (read K=6 chunks + decode, even if no failure)
  Write latency: higher (compute M=2 parity chunks before writing)
  Repair: more expensive (reconstruct data to regenerate missing chunk)

S3 uses erasure coding within each AZ to minimize storage cost,
plus replication across AZs for geographic durability
```

**G5:** Would highlight the distinction between erasure coding for storage efficiency and erasure coding for availability. The 1.33x overhead of a 6+2 scheme saves enormous cost at exabyte scale, but reconstruction from K=6 chunks requires reading from 6 nodes in parallel — if network jitter causes even one chunk read to be slow (the "straggler problem"), the GET latency is gated on the slowest of the 6 reads. This is why S3 issues K+1 or K+2 speculative reads ("hedged reads") and cancels the slowest when enough chunks arrive to reconstruct — at the cost of wasted bandwidth. The optimal hedging strategy (how many extra requests to issue, and when) is a non-obvious latency-cost tradeoff that depends on the tail latency distribution of the storage tier. Additionally, for objects below a certain size threshold (typically < 1MB), the overhead of splitting into K=6 chunks and computing parity exceeds the benefit — S3 uses simple replication for small objects and erasure coding only for large objects.

## Deep Dive 2: Object Listing (Difficult Problem)

**Problem:** `GET /bucket?prefix=images/` must return all objects with that prefix, paginated, consistently. A bucket may have 100 billion objects.

**Naive approach:** Full table scan → O(N) with N = 100B rows → completely impractical even with indexes if the result set is large.

**Key-sorted approach using Cassandra:**
```
Object metadata stored sorted by key (Cassandra clustering key):
  Partition: bucket_id
  Clustering: key (lexicographic sort)

LIST query:
  SELECT * FROM object_metadata
  WHERE bucket_id = ? AND key >= 'images/' AND key < 'images0'
  LIMIT 1000

  Note: 'images0' is the lexicographic successor of 'images/' + chr(0xFF)
  This is a range scan on the clustering key — efficient in Cassandra

Pagination:
  continuation_token = base64_encode(last_key_seen)
  Next page: WHERE key > last_key_seen LIMIT 1000
```

**Problem with single-partition Cassandra:** All objects in one bucket live in one Cassandra partition. A bucket with 100B objects = one enormous partition → hot partition, memory pressure, slow scans.

**Solution: B-tree metadata tier with prefix-based sharding:**
```
Shard the key namespace within each bucket:
  shard_id = hash(bucket_id || key[0:2]) % num_shards  (first 2 chars of key)
  Each shard covers a contiguous range of keys

Per-shard: B-tree index engine (custom or RocksDB)
  Optimized for point lookups (GET object metadata) and range scans (LIST)
  B-tree: O(log N) lookup per shard, sequential page reads for range scan

LIST with prefix:
  1. Determine which shards overlap with the prefix
     (usually 1-2 shards for typical prefix lengths)
  2. Range scan each relevant shard in parallel
  3. Merge results (already sorted within each shard)
  4. Return top 1000 with continuation token

For very common prefixes (e.g., "" = list everything):
  Must scatter to all shards → parallel fan-out → merge
  Expensive: LIST entire large bucket is inherently O(N)
  Rate-limit and add advisory warnings in docs
```

**Delimiter handling (virtual directory semantics):**
```
GET /bucket?prefix=photos/&delimiter=/
  → Returns objects whose keys start with "photos/"
  → Groups keys with additional "/" after the prefix as "CommonPrefixes"
  → Example keys: photos/2024/img1.jpg, photos/2024/img2.jpg, photos/2023/img.jpg
  → Result: CommonPrefixes: ["photos/2024/", "photos/2023/"]
            Contents: (none at this level)
  → Simulates a directory listing without a real directory hierarchy
  → Implementation: during range scan, check if key has delimiter after prefix
                    if yes: truncate at first delimiter, add to CommonPrefixes set
                    if no: add to Contents
```

**G5:** Would challenge whether consistent pagination is achievable without significant cost. When a client paginates through a large LIST result across multiple requests, objects can be inserted or deleted between pages — the continuation token (last_key_seen) does not snapshot the state. A key inserted between page 1 and page 2 will appear in page 2 even though it didn't exist at page 1 time; a key deleted between pages will be missing. True snapshot isolation for LIST would require MVCC with a read timestamp, which means keeping all object versions until all active LIST cursors have advanced past them — prohibitively expensive at 100B objects. S3 documents this behavior explicitly: LIST results are "eventually consistent" (pre-2020) or "consistent at the time of the request but not snapshot-isolated across pages" (post-2020). For workloads requiring a consistent snapshot (e.g., S3 Inventory for data lake), S3 generates a manifest file via a background scan with a consistent read timestamp, delivered as a separate object — this is how S3 Inventory works. The manifest approach moves cost from read-time to write-time.

## Deep Dive 3: Strong Consistency (Post-2020)

**Before 2020:** S3 provided eventual consistency for HEAD-after-PUT and LIST-after-PUT. You could PUT an object and immediately HEAD it and receive a 404, or PUT and DELETE and then LIST and still see the deleted object.

**After 2020:** Amazon announced strong read-after-write consistency for all S3 operations.

**How it works — single-object linearizability:**
```
PUT object:
  1. Client sends PUT to Gateway
  2. Gateway routes to primary storage node (via partition map)
  3. Primary writes object data to disk
  4. Primary writes metadata entry to Metadata DB (single-writer per key, leader-based)
  5. Metadata DB confirms write committed (including to replica)
  6. Gateway acks 200 OK to client
  → The metadata commit is the "commit point"

GET object (immediately after PUT):
  1. Client sends GET to Gateway
  2. Gateway reads metadata from Metadata DB primary/leader (NOT from a replica)
     → Reading from the leader guarantees seeing the just-committed write
  3. Metadata specifies which storage node holds the data
  4. Gateway fetches object data from storage node
  5. Returns to client
  → GET always sees the metadata from step 4 of PUT (same commit log)

DELETE then GET:
  If DELETE committed to Metadata DB before GET reads metadata:
    → Metadata returns delete marker → GET returns 404 ✓
  If GET reads metadata before DELETE commits:
    → Metadata returns object → GET returns object ✓ (DELETE hasn't happened yet)
  → Linearizable: operations appear to happen atomically at one point in time
```

**Implementation detail — metadata leader reads:**
```
Metadata DB: Raft or Paxos-based distributed KV store, one leader per partition
  Leader has the committed log; followers may be slightly behind
  Strong reads MUST go to leader to avoid reading stale replica state

Cost:
  All GET/HEAD metadata reads route to leader
  Leader becomes a bottleneck for hot keys (objects with 1M reads/sec)
  Mitigation: read lease mechanism (leader grants read lease to followers;
              during lease validity, follower reads are linearizable)
  Or: metadata caching with read-your-own-writes guarantee
      (cache session affinity: after PUT, route GETs to same server that did PUT)
```

**G5:** Would push on the interaction between strong consistency and versioned buckets. When versioning is enabled, a PUT creates a new version_id atomically. A concurrent GET without a version_id specified returns the "latest" version — but defining "latest" in a distributed system requires the same leader-read guarantee applied not just to a single key but to the version ordering. The tricky case: PUT version v2 commits on shard A, replicates to shard B asynchronously; a LIST versions request hits shard B before replication completes and returns only v1 as the latest. This is why S3's strong consistency guarantee for versioned buckets required redesigning the version-ordering metadata to flow through the same Raft log as object writes, rather than a separate version counter. The lesson for system design: consistency guarantees that seem simple for single objects become substantially harder when ordering relationships span multiple pieces of metadata (object + version list + list results).

## Reliability and Operations

- **Bit rot detection:** Background scrubber reads every stored chunk, computes checksum, compares to stored checksum (in volume SQLite index). On mismatch: log, alert, restore from another replica. Runs continuously on a rolling schedule — all data scrubbed every 90 days. This catches silent disk corruption that RAID would miss (RAID only detects complete disk failure, not bit flip on a live disk).
- **Node failure detection and repair:** Each storage node sends a heartbeat to the cluster controller every 10 seconds. After 3 missed heartbeats (30 seconds): node marked "suspected down." After 5 minutes: mark as down, begin background re-replication. Re-replication throttled to leave bandwidth for normal requests (e.g., max 100MB/sec repair bandwidth per node).
- **Disk failure — proactive evacuation:** Storage nodes monitor SMART (Self-Monitoring Analysis and Reporting Technology) data. On SMART alerts (reallocated sectors, seek errors): proactively migrate all objects off the at-risk disk before it fails. Evacuation complete → disk replaced. This converts surprise failures into planned maintenance.
- **Rack-awareness and AZ placement:** Replicas placed on nodes in different racks (different power circuits and top-of-rack switches). For erasure coding: chunks distributed to ensure no two chunks from the same erasure group are in the same rack. Tolerate: single rack failure without data unavailability.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Erasure coding (6+2) for large objects | 3x replication | Erasure: 1.33x storage overhead (saves ~$38M/month at 1EB scale); 3x replication: simpler reads (no decode), faster GET on small objects |
| Content-addressed storage on disk | Path-addressed storage | CAS: automatic dedup, integrity verification via hash; Path-based: easier human debugging, no hash computation overhead |
| Strong consistency (leader reads) | Eventual consistency | Strong: simpler client code (no retry-until-consistent logic); Eventual: higher read throughput via replica reads, lower leader load |
| Flat key namespace (bucket/key) | Hierarchical filesystem | Flat: infinitely scalable, no directory rename bottleneck; FS semantics: intuitive but rename of a directory with 1B objects is O(N) |
| Append-only on-disk writes | In-place updates | Append-only: no fragmentation, crash-safe (old data intact until compaction); In-place: less storage amplification, no compaction overhead |

## G5

1. **Storage classes and intelligent tiering:** S3 has STANDARD, STANDARD-IA (Infrequent Access, $0.0125/GB/month vs $0.023), GLACIER INSTANT RETRIEVAL ($0.004/GB/month, 1ms restore), GLACIER FLEXIBLE ($0.0036/GB/month, minutes-to-hours restore), and DEEP ARCHIVE ($0.00099/GB/month, 12-hour restore). The intelligent tiering engine cannot use exact access logs for privacy reasons (knowing a user accessed an object reveals sensitive information). Instead, use a per-object sliding window bloom filter: on each access, set a bit corresponding to the current week; check the last N weeks for any set bits. Objects with no recent bits get downgraded. The bloom filter approach is O(1) per access check and privacy-preserving (probabilistic, not exact). When an object is restored from Glacier, S3 proactively promotes it to STANDARD-IA (not full STANDARD) to avoid paying the higher tier cost speculatively; if access continues, an ML model trained on access patterns predicts whether full STANDARD promotion is warranted.

2. **Cross-region replication (CRR):** Objects replicated asynchronously to a second region for disaster recovery and geo-compliance. Write path: PUT to source region → write to source storage → publish replication event to Kafka → replication worker in destination region consumes event → fetches object data from source → writes to destination region's storage. RPO (recovery point objective): typically < 15 minutes; S3 Replication Time Control (RTC) guarantees 99.99% of objects replicated within 15 minutes, backed by SLA. RTO: near-zero (destination bucket is live and readable throughout). Customers must configure CRR per bucket with a replication rule (can filter by prefix or tag). Versioned source + versioned destination: replicate all versions including delete markers, or optionally exclude delete markers (for backup use cases where you don't want to replicate accidental deletions).

3. **Requestor Pays and billing attribution:** Every GET/PUT request is tagged in the access log with: requester identity, requester AWS account ID, bucket owner account ID, bytes transferred, operation type. This log is the source of truth for billing. The pipeline: access logs → Kinesis Data Firehose → S3 → Athena aggregation → billing system, running hourly. For large S3 customers (Netflix, Airbnb), data transfer OUT costs often exceed storage costs. Requestor Pays feature: bucket owner configures the bucket; requester pays the data transfer cost instead of the bucket owner. Cross-account access via resource-based bucket policies allows fine-grained permissions without IAM role assumption. The billing attribution system is critical infrastructure — billing errors have direct revenue impact and are auditable.

4. **Event notifications and ETL triggers:** S3 Event Notifications allow customers to configure triggers on PUT/DELETE/lifecycle events. Design: on successful write to storage, append an event record to a per-bucket event Kafka partition (not the data path — event is appended asynchronously after the PUT ack). Event fan-out service reads the Kafka partition and routes to configured destinations: Lambda function ARN (synchronous invocation), SQS queue (message enqueue), SNS topic (fan-out to multiple subscribers). At-least-once delivery semantics; destinations must be idempotent. The Lambda invocation path handles millions of events/day for image processing pipelines (PUT → Lambda → thumbnail generation), data lake ingestion (PUT CSV → Glue crawler → schema detection → Athena queryable), and ML training data pipelines. The event system must not block the PUT response — it is fully asynchronous with a best-effort delivery SLA of < 1 second for most events.

5. **S3 Select and columnar pushdown:** S3 Select allows executing SQL queries on CSV, JSON, and Parquet objects without downloading the full object. The storage node itself executes the filter predicate and projection, streaming only matching rows to the client. For Parquet: the storage node reads the row group metadata (min/max statistics per column) to skip entire row groups that cannot match the WHERE clause — this is column-oriented predicate pushdown. A 100GB Parquet file with 1000 row groups might have a query that skips 950 row groups, returning only 5GB of data and saving 95% of transfer cost. This is architecturally significant: compute is pushed into the storage tier (disaggregated storage with compute at the storage layer), a pattern also seen in Snowflake's external tables and Databricks Delta Lake over S3. The tradeoff: storage nodes now need CPU resources for query execution, which must be capacity-planned separately from I/O throughput capacity.
