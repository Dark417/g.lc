# 12. AWS S3 — Object Storage

**Type:** storage
**Difficulty:** L5
**Frequency:** High — asked at infra/storage platform interviews

---

## Problem Statement

Design an object storage system like Amazon S3. Users can create buckets, upload objects (files up to 5TB), download them by key, and list objects in a bucket. The system must provide 99.999999999% (11 nines) durability, 99.99% availability, serve exabytes of data, and handle millions of requests per second.

---

## Functional Requirements

**Must have:**
- Create/delete buckets
- PUT object (key, value bytes, metadata)
- GET object by key
- DELETE object
- LIST objects in bucket (with optional prefix filter)
- Multipart upload for objects > 100MB

**Should have:**
- Object versioning (keep all versions of an object)
- Access control (bucket policies, IAM roles)
- Object lifecycle rules (auto-delete or archive after N days)
- Pre-signed URLs (time-limited, no-auth-required access)
- Server-side encryption

**Out of scope:**
- Block storage (EBS)
- File system semantics (EFS)
- CDN (CloudFront sits in front of S3)

---

## Non-Functional Requirements

- **Durability:** 99.999999999% — effectively zero data loss
- **Availability:** 99.99% for GET/PUT; 99.9% for LIST
- **Scale:** Exabytes of storage, millions of objects, millions of RPS
- **Latency:** GET p99 < 100ms for objects < 1MB (excluding transfer time)
- **Consistency:** Strong consistency for GET-after-PUT (Amazon updated this in 2020)

---

## Capacity Estimation

```
Assume 5 billion objects stored, avg 1MB each = 5PB data
Replication factor: 3 across AZs → 15PB physical
Growth: 1PB/day (aggressive estimate for large service)

GET QPS: 1M/sec across cluster
PUT QPS: 100K/sec
LIST QPS: 10K/sec (expensive operation)

Disk: 15PB / 10TB per disk = 1,500 disks minimum
  Plus overhead, hot spares, metadata storage: ~5,000 disks
  Spread across 100 storage nodes (50TB each): 100 nodes minimum
```

---

## High-Level Architecture

```
[Client]
  │
  ├─ PUT /bucket/key ──→ [Gateway Layer (Stateless)] ──→ [Partition Service]
  │                               │                              │
  │                        [Auth/ACL Service]          [Partition Map (etcd)]
  │                               │                              │
  └─ GET /bucket/key ──────────── │                   [Storage Nodes]
                                  │                              │
                               [Metadata DB]          [Erasure Coding / Replication]
                                  │
                            [DynamoDB-style / Cassandra]
```

**Components:**
- **Gateway:** Stateless request routing. Auth, quota check, route to correct partition.
- **Partition Service:** Maps (bucket, key) → storage node(s). Uses consistent hashing.
- **Storage Nodes:** Store actual object data. Each node manages multiple hard drives (4-16TB each). Uses local filesystem (XFS) optimized for large sequential writes.
- **Metadata DB:** Stores object metadata: key, bucket, size, etag, version, storage location, ACL. Separate from object data.
- **Replication/Erasure Coding service:** Ensures durability across nodes and AZs.

## API Design

```
# Object operations
PUT /bucket/key
Headers: Content-Type, Content-Length, Content-MD5, x-amz-server-side-encryption
Body: object data
Returns: 200 OK, ETag: "{md5_hash}"

GET /bucket/key
Headers: Range: bytes=0-1023 (optional, for partial reads)
Returns: 200 OK, object bytes | 206 Partial Content

DELETE /bucket/key
Returns: 204 No Content

HEAD /bucket/key
Returns: headers only (metadata), no body

GET /bucket?prefix=images/&delimiter=/&max-keys=1000&continuation-token=
Returns: XML { Contents: [{ Key, Size, ETag, LastModified }], NextContinuationToken }

# Multipart upload
POST /bucket/key?uploads → { UploadId }
PUT /bucket/key?partNumber=1&uploadId=xxx (upload each part)
POST /bucket/key?uploadId=xxx (complete, list part ETags)
DELETE /bucket/key?uploadId=xxx (abort)
```

## Data Model

**Metadata DB (Cassandra-style, partitioned by bucket):**
```
object_metadata (
  bucket_id     UUID PARTITION KEY,
  key           TEXT CLUSTERING KEY,
  version_id    UUID,           -- for versioned buckets
  size_bytes    BIGINT,
  etag          TEXT,           -- MD5 of object
  content_type  TEXT,
  created_at    TIMESTAMPTZ,
  storage_class TEXT,           -- STANDARD, GLACIER, DEEP_ARCHIVE
  storage_node  TEXT,           -- which node(s) hold the data chunks
  chunk_map     JSONB,          -- for multipart: { chunk_id: [node1, node2] }
  acl           JSONB,
  user_metadata JSONB
)

-- Bucket metadata (low volume, PostgreSQL)
buckets (
  id             UUID PRIMARY KEY,
  name           TEXT UNIQUE NOT NULL,
  owner_id       UUID,
  region         TEXT,
  versioning     BOOLEAN DEFAULT FALSE,
  lifecycle_rules JSONB,
  created_at     TIMESTAMPTZ
)
```

**On-disk format per storage node:**
```
Each node: HDD/SSD organized into "volumes" (one per physical disk)
Object data: stored as flat file on disk: /data/volumes/{vol_id}/{object_id}
  - No filesystem metadata for object (just raw bytes)
  - Append-only writes (no in-place updates)
  - Deletion: mark as deleted in local metadata, actual space reclaimed by compaction job

Disk index (SQLite per volume): object_id → byte_offset, size
  - Fast lookup of objects on a given disk
  - Rebuilt from disk scan on node recovery
```

## Deep Dive 1: Durability — Erasure Coding vs Replication

**Replication (simple, default):**
```
Store 3 copies across 3 different AZs
Write: PUT → primary node → replicate to 2 secondaries → ack when 2 of 3 ack
Read: GET from any replica

Durability: if P(disk failure/year) = 0.01%, P(all 3 fail simultaneously) = (0.0001)^3 = very small
Storage overhead: 3x
```

**Erasure coding (S3's Standard class uses 6+2 per AZ, plus cross-AZ replication):**
```
Reed-Solomon erasure coding:
  Split object into K data chunks + M parity chunks (k=6, m=2 → 8 total)
  Any K of K+M chunks can reconstruct the full object
  Overhead: (K+M)/K = 8/6 = 1.33x (much better than 3x replication)
  Can tolerate M=2 simultaneous chunk losses

AWS S3 Standard:
  Within AZ: erasure code K+M chunks across drives
  Across AZs: 3 AZs, each with its own erasure-coded copy
  Total durability: 11 nines (10^-11 annual object loss probability)
```

**Tradeoff:**
- Erasure coding: ~1.5x overhead vs 3x for replication; slower reads (must read K chunks + decode)
- Replication: higher overhead but O(1) reads (just fetch from nearest replica)
- S3 uses erasure coding within AZ to minimize storage cost, replication across AZs for geo-durability

## Deep Dive 2: Object Listing (Difficult Problem)

**Problem:** LIST `/bucket?prefix=images/` must return all objects with that prefix, paginated, consistently. Bucket may have 100 billion objects.

**Naive approach:** Full table scan → terrible performance

**Key-sorted approach:**
```
Object keys are stored in sorted order in Cassandra (with bucket as partition key)
LIST = range scan: WHERE bucket_id = X AND key >= 'images/' AND key < 'images0'

Pagination via continuation token:
  token = base64_encode(last_key_seen)
  Next page: WHERE key > last_key_seen LIMIT 1000
```

**Problem with this:** Cassandra doesn't efficiently support range scans on the clustering key at massive scale (100B keys per bucket).

**Solution: B-tree index on (bucket_id, key) with sharding:**
```
Shard the key namespace: hash(bucket_id + key_prefix_first_2_chars) → shard
Each shard has a B-tree index for its key range
LIST: broadcast to all shards with prefix, merge results
  → Parallel scatter-gather query
  → O(log N) per shard + merge

Metadata tier (separate from data tier):
  Store metadata in dedicated nodes with B-tree storage engine (not Cassandra)
  Optimized for point lookups and range scans
```

## Deep Dive 3: Strong Consistency (POST-2020)

**Before 2020:** S3 had eventual consistency for HEAD-after-PUT (you might see stale metadata briefly).

**After 2020:** Strong read-after-write consistency.

**How:** Metadata DB provides linearizability for single-object operations:
```
PUT object:
  1. Write object data to storage nodes (quorum)
  2. Write metadata entry with version timestamp (single-writer per key via leader election)
  3. Return success only after metadata committed

GET object:
  1. Read metadata (always from leader/primary for strong consistency)
  2. Read object data from storage node specified in metadata

→ GET after PUT always sees the PUT (metadata committed before ack)
→ DELETE then GET: if DELETE committed, GET returns 404
```

**Cost:** Metadata reads must go to primary (not replica) → more load on primary, higher latency. Mitigated by caching: cache metadata with read-your-own-writes guarantee via session affinity.

---

## Reliability and Operations

- **Bit rot detection:** Background job reads every object, computes MD5, compares to stored ETag. On mismatch: restore from another replica. Runs on a rolling basis (scan all objects every 6 months).
- **Node failure:** Replication/erasure coding controller detects node failure via heartbeat. Initiates background re-replication to restore replication factor. Reads continue from remaining replicas during recovery.
- **Disk failure:** Node detects bad sectors via SMART data. Proactive evacuation: migrate objects off the dying disk before it fails completely.
- **Rack-awareness:** Replicas placed on different racks (different power/network circuits) to tolerate rack-level failures.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Erasure coding | 3x replication | Erasure: ~1.5x overhead, slower reads; Replication: simpler, faster reads |
| Content-addressed storage | Path-addressed | CAS: automatic dedup; Path: easier human navigation |
| Strong consistency | Eventual | Strong: simpler client logic; Eventual: higher write throughput |
| Flat key namespace | Hierarchical FS | Flat: simpler, scales better; FS: more intuitive, complex namespace management |

## L5 Additions

1. **Storage classes and intelligent tiering:** S3 has STANDARD, STANDARD-IA (Infrequent Access), GLACIER, DEEP_ARCHIVE. Design the tiering engine: track access frequency per object (using bloom filters, not exact counts — privacy). Objects not accessed in 30 days → downgrade to IA (50% cost reduction). Not accessed in 90 days → archive to Glacier. Intelligent Tiering: ML model predicts access likelihood based on historical pattern. On restore from Glacier: warm up to IA, pre-cache in STANDARD if access velocity spikes.

2. **Cross-region replication (CRR):** Objects replicated asynchronously to a second region. Architecture: write to primary region → replicated via Kafka to secondary region's storage nodes → confirm to customer when secondary ack received (or async). RPO (recovery point objective): typically < 15 minutes for CRR. RTO: near-zero (secondary is live). Use case: disaster recovery, geo-compliance (keep EU data in EU). Customer configures: bucket → rule → replicate all or prefix-filtered objects to target region.

3. **Requestor pays and cross-account access:** Bucket owner pays for storage; requestor pays for bandwidth. Cross-account access via IAM resource policies. Design the billing attribution system: every GET/PUT request tagged with: requester identity, requester account, bucket owner account, bytes transferred. Billing pipeline: Kafka → aggregate by account-pair → daily billing report. For large S3 users (Netflix), bandwidth costs > storage costs; Requestor Pays + direct connect reduces transfer costs.

4. **Event notifications and ETL triggers:** S3 Event Notifications: PUT/DELETE/lifecycle events → SNS/SQS/Lambda. Design: on successful write to storage node, append event to per-bucket event log (Kafka partition per bucket). Event fan-out service: reads log, routes to configured destinations (Lambda ARN, SQS queue, SNS topic). At-least-once delivery; destination must be idempotent. Used for: image processing pipelines (PUT image → Lambda → resize + thumbnail), data lake ingestion (data lands in S3 → Glue crawler detects schema → Athena queryable).

5. **Metadata index and S3 Select:** S3 Select allows running SQL queries on CSV/JSON/Parquet objects without downloading the full file. Architecture: S3 nodes support columnar pushdown for Parquet (skip row groups that don't match WHERE clause). Results streamed to client. For large queries (multi-object): Athena wraps S3 Select with a distributed query engine. S3 Inventory: daily or weekly manifest file listing all objects in bucket with metadata — generated by background scan, delivered as CSV/ORC/Parquet to another bucket. Replaces expensive LIST API for large-bucket use cases.
