# 04. Unique ID Generator

**Type:** core
**Difficulty:** L3–L4
**Frequency:** High — often asked as warm-up or embedded in larger designs

---

## Problem Statement

Design a distributed unique ID generation system. IDs must be globally unique across all nodes, sortable by generation time (roughly), fit in 64 bits, and be generated at high throughput (hundreds of thousands per second) without coordination bottlenecks.

---

## Functional Requirements

**Must have:**
- Generate globally unique IDs
- IDs are 64-bit integers (fits in standard DB bigint)
- IDs are roughly time-ordered (newer IDs > older IDs)
- System generates at least 100K IDs/sec

**Should have:**
- IDs are k-sortable (IDs generated within the same millisecond may not be perfectly ordered, but IDs from different seconds are ordered)
- Low latency generation: < 1ms per ID
- No single point of failure

**Out of scope:**
- Globally sequential IDs (requires coordination)
- Human-readable IDs

---

## Non-Functional Requirements

- **Throughput:** 100K+ IDs/sec total across all nodes
- **Latency:** < 1ms per ID generation
- **Uniqueness:** Zero collisions — ever
- **Availability:** ID generation must work even if some nodes fail

---

## Capacity Estimation

```
100K IDs/sec → 8.64B IDs/day
10 generator nodes × 10K IDs/sec each = 100K IDs/sec total

64-bit ID space: 2^63 = 9.2 × 10^18 unique values
At 100K/sec: 9.2 × 10^18 / 100,000 = 2.9 billion seconds ≈ 92 years of unique IDs
```

---

## High-Level Architecture

```
Client Service
      │
      ├──→ Generator Node 1 (machine_id=1) → returns 64-bit ID
      ├──→ Generator Node 2 (machine_id=2) → returns 64-bit ID
      └──→ Generator Node N (machine_id=N) → returns 64-bit ID

[ZooKeeper / etcd] ← assigns machine_id to each generator node
```

No coordination between generator nodes after boot-time machine_id assignment.

---

## API Design

```
# Service API
GET /v1/id → { id: int64, id_str: string }
POST /v1/ids  Body: { count: int }  → { ids: int64[] }  # batch generation

# Internal / SDK
generate() → uint64   # in-process generation, no network call
```

---

## Deep Dive: ID Format — Snowflake

**Twitter Snowflake — the standard answer:**

```
 63        22        12      0
  ┌─────────┬─────────┬──────┐
  │ time_ms │ node_id │ seq  │
  │ 41 bits │ 10 bits │12bits│
  └─────────┴─────────┴──────┘
```

- **41 bits timestamp:** milliseconds since custom epoch (e.g., 2020-01-01). 2^41 ms ≈ 69 years.
- **10 bits machine ID:** supports 1024 distinct generator nodes. Assigned at startup from ZooKeeper.
- **12 bits sequence:** 2^12 = 4096 IDs per millisecond per node → 4M IDs/sec per node.

**Generation algorithm:**
```
mutex.lock()
now_ms = current_time_ms()

if now_ms == last_ms:
    seq = (seq + 1) & 0xFFF   // 12-bit mask
    if seq == 0:               // sequence overflow
        wait until next ms
elif now_ms > last_ms:
    seq = 0
    last_ms = now_ms
else:
    panic("clock moved backward")  // NTP adjustment

id = (now_ms - epoch) << 22 | machine_id << 12 | seq
mutex.unlock()
return id
```

**Clock skew problem:** If server clock moves backward (NTP adjustment), IDs could collide.
- Solution: on clock-backward event, wait until clock catches up before generating IDs
- Drift tolerance: refuse to generate IDs if clock is more than 2 seconds behind last generated ID

## Alternative Approaches

**Option A: UUID v4 (random)**
```
128-bit random number, formatted as hex string
Pros: no coordination, trivial to implement
Cons: 128-bit (not 64-bit), not sortable, random (bad for DB indexing)
```

**Option B: UUID v7 (time-ordered, 2023 standard)**
```
48-bit timestamp (ms) | 4-bit version | 12-bit random | 64-bit random
Pros: sortable, standardized, no coordination
Cons: 128-bit, not perfectly sequential within ms
```

**Option C: Database auto-increment**
```
Single DB as counter
Pros: simplest, perfectly sequential
Cons: single point of failure, can't scale writes
```

**Option D: Ticket server (Flickr-style)**
```
Dedicated MySQL with AUTO_INCREMENT
REPLACE INTO tickets64 (stub) VALUES ('a');
SELECT LAST_INSERT_ID();
Pros: simple distributed generation
Cons: SPOF (mitigate with two servers, one for even/odd IDs)
```

**Option E: Segment's KSUID**
```
20 bytes: 4 bytes timestamp + 16 bytes random
Pros: globally unique, sortable, URL-safe base62 encoding
Cons: 160-bit, not a simple integer
```

**Recommendation:** Snowflake for 64-bit integer requirement. UUID v7 if 128-bit is acceptable and you don't want the operational complexity of assigning machine IDs.

---

## Reliability and Operations

- **Node failure:** Client SDK detects failure, round-robins to another generator node. No data loss — generator nodes are stateless (state is just last_ms + seq + machine_id).
- **Machine ID management:** ZooKeeper assigns machine IDs. On node restart, request a new machine ID (don't reuse — avoids potential collision if old node's in-flight requests haven't all been processed).
- **Monitoring:** Track IDs generated/sec, clock drift per node, sequence exhaustion events.
- **Epoch selection:** Pick a recent epoch (not Unix epoch) to maximize the 41-bit timestamp range. E.g., epoch = 2020-01-01 → usable until ~2089.

## L5 Additions

1. **Handling data center awareness:** Extend Snowflake with a datacenter ID: `timestamp(41) | datacenter_id(5) | node_id(5) | sequence(12)`. This provides 32 datacenters × 32 nodes per DC with no cross-DC coordination. IDs are still globally unique.

2. **Monotonic IDs within a process:** Snowflake guarantees time-ordering globally only across milliseconds. Within the same ms, IDs from different nodes are interleaved. For use cases needing strict ordering within a service (e.g., event sourcing), use a local sequence per entity: `entity_version = (last_version + 1)` with optimistic locking. Don't conflate "unique" with "ordered" — they're separate requirements.

3. **ID introspection:** Encode metadata in the ID itself for operational value. `extract_time(id)` → when was this ID created? `extract_node(id)` → which node generated it? Useful for debugging data anomalies, correlating logs with created objects, and auditing data flows. Build a library `IDUtil.parse(id) → {timestamp, datacenter, node, sequence}`.

4. **ID generation in the client SDK (embedded):** For maximum performance, embed generation in the client library. Each client gets a unique machine ID from a lightweight allocation service at startup. The embedded generator produces IDs locally — zero network calls for generation. Drawback: machine ID allocation service must be available at startup. Use cached IDs in local disk as fallback to survive restarts without re-registration.

5. **Predictability attack mitigation:** Snowflake IDs expose node ID and creation time. An attacker can enumerate recently created objects by guessing ID ranges. Mitigations: (a) apply a reversible shuffle (SipHash with secret key) to permute the ID space — still sortable if you know the secret, appears random externally. (b) Use separate external IDs (UUIDs) for public-facing APIs while using Snowflake IDs internally for DB performance.
