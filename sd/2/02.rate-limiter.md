# 02. Rate Limiter

**Type:** core
**Difficulty:** L4
**Frequency:** Very High — especially at infra/platform interviews

---

## Problem Statement

Design a distributed rate limiter that can enforce request limits across a fleet of API servers. The rate limiter should support multiple limit policies (per-user, per-IP, per-endpoint), handle burst traffic gracefully, operate with sub-millisecond latency, and remain correct even when the rate-limit state is spread across multiple nodes.

---

## Functional Requirements

**Must have:**
- Allow or deny an incoming request based on a configured rate limit rule
- Support multiple limit types: per-user, per-IP, per-API-key, per-endpoint
- Return standard HTTP 429 Too Many Requests with `Retry-After` header
- Configure limits without redeploying (dynamic rules)

**Should have:**
- Soft limits (warn) vs hard limits (block)
- Burst allowance: allow short bursts above the steady-state rate
- Distributed: consistent limits across all API server instances
- Graceful degradation: if rate-limit service is down, fail open (allow) not closed

**Out of scope:**
- Request queuing / delayed retry
- Billing and quota management

---

## Non-Functional Requirements

- **Latency:** Rate-limit check must add < 1ms p99 to request path
- **Availability:** Rate limiter must not be a single point of failure
- **Accuracy:** Over-counting by 10% acceptable; significant under-counting is not
- **Scale:** Handle 5M RPS across the fleet
- **Consistency:** Eventually consistent is acceptable — a few extra requests getting through during propagation delay is fine

---

## Capacity Estimation

```
Fleet scale:
  5M RPS total across all API servers
  50 API servers × 100K RPS each = 5M RPS

Redis operations:
  Each request requires 1 atomic Redis call (Lua script)
  5M RPS → 5M Redis ops/sec

Redis Cluster sizing:
  Single Redis node: ~100K–300K ops/sec (depending on op complexity)
  Lua scripts (multi-key): ~100K ops/sec per node conservatively
  Required: 5M / 100K = 50 node-equivalents
  With clustering: 10 primary shards × 3 replicas = 30 nodes total
  → Each primary handles 500K ops/sec (achievable with pipelining)

Memory per counter:
  Active users: 10M
  Limit types per user: ~10 (per-user global, per-endpoint × 5, per-IP, etc.)
  Per counter in Redis: ~64 bytes (key string + integer value + TTL overhead)
  Total: 10M × 10 × 64B = 6.4GB → fits in a 3-shard cluster with 4GB/shard

Bandwidth:
  Each Redis call: ~100 bytes request + ~20 bytes response
  5M/sec × 120B = 600MB/s → well within 10GbE links
```

## High-Level Architecture

```
Client Request
      │
      ▼
[API Gateway / Middleware Layer]
      │
      ├──→ [Rate Limit Middleware] ──→ [Local Rules Cache (30s TTL)]
      │           │
      │     [Redis Cluster]  ← atomic Lua script: check + increment
      │           │
      │    ┌──────┴──────┐
      │   Allow          Deny
      │    │              │
      │    ▼              ▼
      │ [Business]   HTTP 429 + Retry-After header
      │  [Logic]
      │    │
      ▼    ▼
[Response to Client]
Headers: X-RateLimit-Limit: 100
         X-RateLimit-Remaining: 47
         X-RateLimit-Reset: 1735689600

[Rules Service] ──→ [PostgreSQL / etcd] ← source of truth for rules
      ↑
[Admin API]  (update limits without deploy)
```

**Components:**
- **Rate Limit Middleware:** Runs in-process in the API server (no extra network hop to a sidecar). Builds the limit key from request context, calls Redis, returns Allow/Deny in < 1ms.
- **Redis Cluster:** Stores counters and sliding window data. Atomic operations via Lua script prevent TOCTOU races.
- **Rules Service:** Stores limit configurations (per-customer, per-endpoint). API servers cache a local copy in memory, refreshed every 30s from the Rules Service. This means limit changes propagate within 30s without any request needing to hit the Rules Service.
- **Rules Store:** PostgreSQL (or etcd for simpler KV) for persistent rule configuration with audit history.

## API Design

**Internal (called by middleware per request):**
```
check_and_increment(
  key:        string,     -- e.g., "rl:user:u123:/api/posts:minute"
  limit:      int,        -- max requests in window
  window_sec: int         -- rolling window duration
) → {
  allowed:    bool,
  remaining:  int,        -- requests left in current window
  reset_at:   timestamp   -- when window resets (for Retry-After header)
}
```

**Admin API for managing rules:**
```
PUT /v1/rate-limits/{rule_id}
Body: {
  key_pattern: "user:{id}:endpoint:/api/posts",
  limit: 100,
  window_sec: 60,
  burst: 20              -- allow up to 120 in 60s if bucket allows
}

GET /v1/rate-limits/{rule_id}
DELETE /v1/rate-limits/{rule_id}
GET /v1/rate-limits?key_pattern=user:*   -- list rules matching pattern
```

## Data Model

**Redis key schema:**
```
Pattern: rl:{strategy}:{entity}:{endpoint}:{window_bucket}

Examples:
  rl:user:u123:/api/posts:1704067200     -- per-user, per-endpoint, fixed-minute window
  rl:ip:1.2.3.4:global:1704067200       -- per-IP, global, fixed-minute window
  rl:apikey:k_abc:*:1704067200          -- per API key, all endpoints
  rl:user:u123:global:prev              -- previous window (for sliding window counter)
  rl:user:u123:global:curr              -- current window (for sliding window counter)

Key TTL: window_sec × 2  (prev window key expires 2 windows after creation)
```

**Rules store (PostgreSQL):**
```sql
rate_limit_rules (
  rule_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  key_pattern   TEXT NOT NULL,       -- "user:*:endpoint:/api/*"
  limit_count   INT NOT NULL,
  window_sec    INT NOT NULL,
  burst         INT DEFAULT 0,       -- extra allowance above limit_count
  priority      INT DEFAULT 0,       -- higher priority rules take precedence
  enabled       BOOLEAN DEFAULT true,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
)
CREATE INDEX ON rate_limit_rules (key_pattern, priority DESC);
```

## Deep Dive 1: Algorithm Choices

### Fixed Window Counter
```
Key: rl:user:{id}:{endpoint}:{floor(now/60)}
INCR <key>
EXPIRE <key> 60

If counter > limit → reject.
```
- O(1) time, O(1) space per user per endpoint
- **Problem:** "boundary burst" — a user sends 100 req at 12:00:59 and 100 at 12:01:00. Both windows allow them. In a 2-second span, 200 requests slip through against a limit of 100. This can be exploited deliberately.

### Sliding Window Log
```
ZADD rl:user:{id} {now_ms} {request_uuid}
ZREMRANGEBYSCORE rl:user:{id} 0 {now_ms - window_ms}
count = ZCARD rl:user:{id}

If count > limit → reject. Else allow.
EXPIRE rl:user:{id} {window_sec}
```
- Perfectly accurate: counts exactly the right requests in a rolling window
- **Problem:** Memory-heavy — stores a sorted set entry per request. At 100 req/min limit × 10M users = 1B sorted set members. Each entry: ~40 bytes → 40GB just for rate limit logs. At high volume this is untenable.

### Sliding Window Counter (chosen)
```
Uses two adjacent fixed windows + linear interpolation:

  prev_window_count  = GET rl:user:{id}:prev
  curr_window_count  = GET rl:user:{id}:curr
  elapsed_fraction   = (now % window_sec) / window_sec

  estimated_count = prev_window_count × (1 - elapsed_fraction) + curr_window_count

  If estimated_count >= limit → reject
  Else → INCR rl:user:{id}:curr, allow
```
- **Accuracy:** < 1% error at window boundaries (interpolation approximates actual count)
- **Space:** O(1) — only 2 counters per user per limit rule
- **Time:** O(1) — 2 GETs + 1 INCR, all atomic via Lua

**Full Lua implementation:**
```lua
-- KEYS[1] = current window key, KEYS[2] = previous window key
-- ARGV[1] = now_ms, ARGV[2] = window_ms, ARGV[3] = limit
local key_curr = KEYS[1]
local key_prev = KEYS[2]
local now      = tonumber(ARGV[1])
local window   = tonumber(ARGV[2])
local limit    = tonumber(ARGV[3])

local prev_count = tonumber(redis.call('GET', key_prev) or 0)
local curr_count = tonumber(redis.call('GET', key_curr) or 0)

-- What fraction of the current window has elapsed?
local window_start    = math.floor(now / window) * window
local elapsed_frac    = (now - window_start) / window

-- Weighted estimate: older window contributes less as time passes
local estimated = math.floor(prev_count * (1 - elapsed_frac) + curr_count)

if estimated >= limit then
    -- Return: {allowed=false, remaining=0, reset_at=next_window_start}
    return {0, 0, window_start + window}
end

-- Allow: increment current window counter
redis.call('INCR', key_curr)
redis.call('EXPIRE', key_curr, window * 2)  -- keep for 2 windows (prev + curr)
local remaining = limit - estimated - 1
return {1, remaining, window_start + window}
```

**G5:** The sliding window interpolation assumes a uniform request distribution within the previous window — if the previous window had a spike at its very end (e.g., 100 requests in the last 1 second of a 60-second window), the interpolation underestimates the effective rate at the start of the current window. For strict SLA enforcement (financial APIs, safety-critical systems), this 1% error is unacceptable. Use the exact sliding window log for those endpoints, gated by a rule flag `exact: true`, and accept the memory cost for the small number of endpoints that require it. For the 99% of APIs where approximate is fine, the counter approach is correct.

### Token Bucket (good for burst scenarios)
```
State per user: { tokens: float, last_refill_time: timestamp }

On each request:
  elapsed = now - last_refill_time
  tokens  = min(bucket_capacity, tokens + elapsed × refill_rate)
  last_refill_time = now
  if tokens >= 1:
    tokens -= 1
    allow
  else:
    reject

Stored as: HSET rl:bucket:{user} tokens {value} last_refill {timestamp}
Updated atomically via Lua script
```
- **Pros:** Natural burst handling — user can save up tokens over quiet periods and spend them in a burst
- **Cons:** 2 fields to maintain vs 1 counter; floating-point arithmetic in Lua; last_refill_time can drift on clock adjustments
- **When to use:** APIs that explicitly want to allow bursting (e.g., "10 req/sec normally but allow up to 50 in a burst")

**Recommendation:** Sliding window counter for standard rate limiting. Token bucket when explicit burst behavior is a product requirement.

## Deep Dive 2: Distributed Consistency

**The race condition problem:**
```
Scenario: 50 API servers, user at limit=100, current count=99
  Thread A on Server 1: GET count → 99
  Thread B on Server 2: GET count → 99
  Thread A: count < limit → INCR → count=100 → allow
  Thread B: count < limit → INCR → count=101 → allow (WRONG — over limit by 1)
```

**Solution: Atomic Lua script in Redis**
- Redis is single-threaded; Lua scripts execute atomically with no interleaving
- The GET + compare + INCR happens as one indivisible operation
- No WATCH/MULTI/EXEC overhead (optimistic locking with retries)
- Latency: single round trip from API server to Redis → ~0.3ms p50, ~0.8ms p99

**Cross-datacenter rate limiting:**
```
Option 1: Per-region limits (chosen for simplicity)
  global_limit = 1000 req/min
  per_region_limit = 1000 / num_regions (with 20% headroom)

  US region gets limit=417, EU gets limit=417, APAC gets limit=417
  Total allowed: 1,251 (25% over global limit in worst case)
  Acceptable: user would need to split requests perfectly across regions to exploit

Option 2: Global counter with cross-region replication
  Write to local Redis, async replicate to other regions
  Accept up to replication_lag × request_rate overshoot
  At 100 req/sec and 100ms cross-region lag: 10 extra requests max → acceptable

Option 3: CRDTs (PNCounters)
  Increment-only counter with convergent replication
  Each region has its own shard; gossip protocol merges counts
  Reads are approximate (may miss recent increments from other regions)
  Implementation complexity is high — not worth it unless overshoot is truly unacceptable
```

**Hot key mitigation:**
If a single user fires 1M RPS (DDoS or runaway client), their rate-limit key becomes a hot key on one Redis shard, saturating it.
```
Solution: Striped counters
  Instead of one key "rl:user:{id}", use N sharded keys:
    shard = random.int(0, N-1)   -- pick a random shard for this request
    key = f"rl:user:{id}:shard{shard}"
    INCR key (limit = global_limit / N per shard)

  To read total: sum all N shards (GET rl:user:{id}:shard0 ... shardN-1)
  Works because INCR is O(1) and striping spreads load across N Redis nodes
  N=10 reduces per-shard load by 10× for hot keys
```

**G5:** The Lua script atomicity guarantee breaks down at the Redis Cluster level when a rate-limit rule spans multiple Redis keys on different shards — which happens with the sliding window approach (curr and prev window keys may hash to different slots). Redis Cluster requires all keys in a Lua script to be on the same shard. Solution: use hash tags to force co-location: `rl:{user123}:curr` and `rl:{user123}:prev` — the `{user123}` portion is hashed, so both keys land on the same shard. This is a non-obvious correctness requirement that breaks silently if omitted (script executes against wrong shard, returns stale data). Enforce this in the key-building library layer, not left to individual callers.

## Reliability and Operations

- **Redis completely down:** Rate limiter fails open — allow all traffic. Log that rate limiting is disabled and alert on-call. Never block legitimate users because the rate limiter is degraded. The alternative (fail closed) causes a total service outage.
- **Redis partitioned (primary lost):** Redis Sentinel or Cluster promotes replica within 10–30s. During this window, some counters from the failed primary are lost. Counters restart from 0 → slight under-counting for one window. Acceptable — far better than blocking all traffic.
- **Local in-process fallback:** Each API server maintains a local sliding window in-process memory as a fallback if Redis is unreachable (circuit breaker open). This local limiter applies per-server limits (global_limit / num_servers). Protects against total bypass while Redis is down. Syncs back when Redis recovers.
- **Rules cache consistency:** Local rules cache is refreshed every 30s. During the 30s window, some servers may run old limits after a limit change. Acceptable for most use cases. For immediate enforcement (security incident requiring instant block), push a cache-busting signal via a pub/sub channel that all servers subscribe to.
- **Monitoring:** Track per-rule: requests allowed/sec, requests denied/sec, Redis latency p50/p99, rules cache age. Alert if denied rate > 5% (possible misconfiguration) or Redis p99 > 2ms (capacity issue).

## Security and Abuse

- Rate limit by IP as first defense layer, evaluated before authentication (protects auth endpoints from credential stuffing)
- Rate limit by API key for authenticated endpoints (tied to account; revocable)
- Unauthenticated requests get 10× tighter limits than authenticated (10 req/min vs 100 req/min)
- `Retry-After` header: set to exact seconds until reset, not a random value — allows well-behaved clients to back off correctly
- Avoid leaking `X-RateLimit-Remaining` for security-sensitive endpoints (auth, password reset) — attackers use it to probe limits
- Block at CDN/WAF for extreme volumetric attacks (millions of RPS from single IP) — don't let these reach the rate limiter at all

## Tradeoffs

| Approach | Pros | Cons | Why or why not |
|----------|------|------|----------------|
| In-process (local) limiter | Zero latency, no network call | Limits not shared — each server enforces only its local count; user can exceed global limit by routing to many servers | Only viable for single-server setups or as fallback |
| Centralized Redis | Accurate distributed limiting, atomic operations | Extra network hop (~0.5ms p50), SPOF risk mitigated by cluster | Best choice for distributed enforcement at < 5ms budget |
| Leaky bucket | Guarantees smooth output rate (no bursts) | Delays requests rather than rejecting them — requires queuing infrastructure; increases tail latency | Use only when smooth output rate matters more than low latency |
| Token bucket | Handles burst naturally; good UX for bursty APIs | Slightly more state (tokens + refill_time); floating-point arithmetic | Use when product explicitly needs burst allowance |
| Sliding window counter | Accurate, O(1) space, O(1) time | ~1% approximation at window boundaries | Best default for most API rate limiting |
| Fixed window | Simplest implementation | Up to 2× overshoot at window boundary — exploitable | Only use for non-security-critical limits with very loose enforcement |

## G5

**What a G5 engineer adds beyond the above:**

1. **Adaptive rate limiting based on system health:** Static limits assume the system can always handle the allowed rate. In practice, downstream services (DB, cache) have variable capacity. Implement a feedback loop: measure P99 latency of the protected service every 5 seconds → if P99 exceeds SLO threshold, reduce admission rate by 20% → check again → stabilize. This is the core of Netflix's Concurrency Limit library and TCP's AIMD congestion control, applied to API rate limiting. The limit becomes a control variable, not a static config. Requires careful damping (don't oscillate) — use an exponentially weighted moving average of latency, not raw samples.

2. **Priority queues and tiered admission:** Not all requests deserve equal treatment under load. Premium users > free users > unauthenticated. During normal operation, all tiers are admitted freely. Under saturation (queue depth rising), selectively drop lower-priority requests first. Implementation: maintain 3 token buckets per rate-limiter instance with different sizes. High-priority bucket drains from shared capacity first; low-priority gets what remains. Use weighted fair queueing so low-priority traffic is never completely starved — always gets at least 10% of capacity.

3. **Rate limiting as a service (RaaS) with batch check API:** Embedding rate limiting in each API server couples the rate-limit logic to every service. For large orgs with 100+ services, a dedicated RaaS is easier to operate. Use gRPC with strict deadlines: if RaaS call exceeds 2ms, time out and fail open. The key optimization is a batch check endpoint: `check_many([{key, limit, window}, ...]) → [{allowed, remaining}]` — amortizes the gRPC overhead across multiple concurrent limit checks. Useful for APIs that enforce several limits per request (per-user + per-IP + per-endpoint) in a single round trip.

4. **Geo-distributed eventual consistency model:** For APIs serving users across 5+ regions, synchronizing counters globally in real-time is prohibitively expensive (adds 100–200ms cross-region latency). Use an eventually consistent model instead: each PoP tracks local counts in Redis and gossips updates to other PoPs every 1 second. Each PoP interpolates global count as `local_count × num_pops`. Accept a 1–5% overshoot window equal to (replication_lag × global_request_rate / num_pops). This is how Cloudflare's distributed rate limiter works. The overshoot is bounded and acceptable for soft limits; for hard limits (financial, compliance), use synchronous cross-region coordination with the associated latency cost.

5. **Cost-aware rate limiting (resource units instead of request count):** Counting requests treats a 10ms DB read the same as a 5-second analytics aggregation — they are not equivalent in resource consumption. Assign resource cost units: simple read = 1 unit, list query = 5 units, complex join = 20 units, file upload = 100 units. Rate-limit on total cost units per minute, not request count. This provides fairer capacity allocation: a user running expensive queries exhausts their budget faster than a user making simple reads, even at the same request count. Costs are defined in the endpoint config; middleware reads cost from local rules cache and deducts accordingly. Requires careful cost calibration — measure actual resource consumption per endpoint type in production.
