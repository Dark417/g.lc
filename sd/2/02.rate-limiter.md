# 02. Rate Limiter

**Type:** core
**Difficulty:** L4
**Frequency:** Very High — especially at infra/platform interviews

---

## Problem Statement

Design a distributed rate limiter that can enforce request limits across a fleet of API servers. The rate limiter should support multiple limit policies (per-user, per-IP, per-endpoint), handle burst traffic gracefully, operate with sub-millisecond latency, and remain correct even when the rate-limit state is spread across multiple nodes.

---

## Functional Requirements

**Must have:**
- Allow or deny an incoming request based on a configured rate limit rule
- Support multiple limit types: per-user, per-IP, per-API-key, per-endpoint
- Return standard HTTP 429 Too Many Requests with `Retry-After` header
- Configure limits without redeploying (dynamic rules)

**Should have:**
- Soft limits (warn) vs hard limits (block)
- Burst allowance: allow short bursts above the steady-state rate
- Distributed: consistent limits across all API server instances
- Graceful degradation: if rate-limit service is down, fail open (allow) not closed

**Out of scope:**
- Request queuing / delayed retry
- Billing and quota management

---

## Non-Functional Requirements

- **Latency:** Rate-limit check must add < 1ms p99 to request path
- **Availability:** Rate limiter must not be a single point of failure
- **Accuracy:** Over-counting by 10% acceptable; significant under-counting is not
- **Scale:** Handle 5M RPS across the fleet
- **Consistency:** Eventually consistent is acceptable — a few extra requests getting through during propagation delay is fine

---

## Capacity Estimation

```
5M RPS fleet-wide
50 API servers × 100K RPS each
Each request does 1 Redis call (INCR or SCRIPT) = 5M Redis ops/sec
Redis Cluster: 300K ops/sec per node → need ~17 nodes for rate-limit ops alone
  → Use Redis Cluster with 6 shards × 3 replicas each

Memory per counter:
  Active users: 10M
  Counters per user: ~10 (per endpoint limits)
  Per counter: 64 bytes (key + value + TTL overhead)
  Total: 10M × 10 × 64B = 6.4GB — fits in a Redis cluster
```

---

## High-Level Architecture

```
Client Request
      │
      ▼
[API Gateway / Middleware Layer]
      │
      ├──→ [Rate Limit Check] ──→ [Redis Cluster] ──→ Allow / Deny
      │           │
      │    (sub-ms decision)
      │
      ▼ (if allowed)
[Business Logic]
      │
      ▼
[Response to Client]
        + Headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
```

**Components:**
- **Rate Limit Middleware:** Runs in-process in API server or as a sidecar. Builds the limit key, calls Redis, returns decision.
- **Redis Cluster:** Stores counters and sliding window data. Atomic operations via Lua script.
- **Rules Service:** Stores limit configurations (per-customer, per-endpoint). Cached locally in API servers, refreshed every 30s.
- **Rules Store:** PostgreSQL or etcd for persistent rule configuration.

---

## API Design

**Internal (called by middleware):**
```
check_and_increment(key: string, limit: int, window_sec: int) → { allowed: bool, remaining: int, reset_at: timestamp }
```

**Admin API for managing rules:**
```
PUT /v1/rate-limits/{rule_id}
Body: { key_pattern: "user:{id}:endpoint:/api/posts", limit: 100, window_sec: 60, burst: 20 }

GET /v1/rate-limits/{rule_id}
DELETE /v1/rate-limits/{rule_id}
```

## Data Model

**Redis key schema:**
```
rate:{strategy}:{entity_id}:{endpoint}:{window}
e.g., rate:user:u123:/api/posts:2024010112  (for 1-hour window ending at this timestamp)
```

**Rules store (PostgreSQL):**
```sql
rate_limit_rules (
  rule_id      UUID PRIMARY KEY,
  key_pattern  TEXT,        -- "user:*:endpoint:/api/*"
  limit_count  INT,
  window_sec   INT,
  burst        INT,
  created_at   TIMESTAMPTZ
)
```

## Deep Dive 1: Algorithm Choices

### Fixed Window Counter
```
Counter per time window (e.g., per minute).
INCR rate:user:123:2024010112
If counter > limit → reject.
```
- Simple, O(1) time and space
- Problem: "boundary burst" — user sends 100 req at 12:00:59 and 100 at 12:01:00 → 200 req slip through in 2 seconds

### Sliding Window Log
```
Store timestamp of every request in a sorted set.
ZADD rate:user:123 <timestamp> <request_id>
ZREMRANGEBYSCORE rate:user:123 0 (now - window)
Count = ZCARD rate:user:123
```
- Accurate but memory-heavy: stores every request timestamp
- O(n) per check where n = requests in window — bad at high volume

### Sliding Window Counter (chosen)
```
Hybrid: use two fixed windows + interpolation.
current_count = prev_window_count × (1 - elapsed/window) + curr_window_count
```
- Good accuracy (< 1% error at boundary)
- O(1) space: only 2 counters per user per limit
- Implementation via Lua script in Redis

```lua
-- Sliding window Lua script
local key_curr = KEYS[1]  -- current window
local key_prev = KEYS[2]  -- previous window
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

local prev_count = tonumber(redis.call('GET', key_prev) or 0)
local curr_count = tonumber(redis.call('GET', key_curr) or 0)
local window_start = math.floor(now / window) * window
local elapsed = (now - window_start) / window

local estimated = prev_count * (1 - elapsed) + curr_count

if estimated >= limit then
  return 0  -- reject
end

redis.call('INCR', key_curr)
redis.call('EXPIRE', key_curr, window * 2)
return 1  -- allow
```

### Token Bucket (good for burst)
```
Each user has a bucket of tokens, refilled at fixed rate.
Request consumes 1 token. If empty → reject.
Allows bursting up to bucket capacity.
```
- Great for APIs that want to allow short bursts
- Slightly more complex state (tokens + last_refill_time)

**Recommendation:** Sliding window for most APIs. Token bucket when burst behavior is explicitly needed.

---

## Deep Dive 2: Distributed Consistency

**Problem:** 50 API servers each do local increments to Redis. With pipelined calls there's a race:
- Thread A reads count=99 (limit=100)
- Thread B reads count=99
- Both increment → count=101 → 2 requests slip through

**Solution: Lua script for atomic check-and-increment**
- Redis executes Lua atomically (single-threaded Redis engine)
- One round trip per rate-limit check
- No WATCH/MULTI/EXEC overhead

**Cross-datacenter:** Don't try to synchronize counters globally in real-time.
- Each region has its own Redis cluster
- Limit per region = global_limit / num_regions (with some headroom)
- Accept that a user could slightly exceed global limit if requests hit different regions
- Alternative: CRDTs (conflict-free replicated data types) — too complex, not worth it

---

## Reliability and Operations

- **Redis down:** Rate limiter should fail open (allow all traffic). Never block legitimate requests because the rate limiter is down. Log that rate limiting is disabled.
- **Redis partitioned:** If primary fails, promote replica within 30s (Redis Sentinel / Cluster). Some counters may be lost → accept slight under-counting during failover.
- **Local fallback:** API servers cache the last known limit state for 5 seconds as in-memory fallback if Redis is unreachable.
- **Hot keys:** If one user ID gets 1M RPS, their rate-limit key becomes a Redis hot key. Mitigation: shard the counter across N Redis keys and sum them (probabilistic counting).

---

## Security and Abuse

- Rate limit by IP as first line of defense (before auth check)
- Rate limit by API key for authenticated endpoints
- Stricter limits for unauthenticated requests (10x tighter)
- Headers expose rate-limit info — use non-enumerable reset times to prevent gaming
- Block at CDN/WAF for extreme volumetric attacks (millions of RPS from single IP)

---

## Tradeoffs

| Approach | Pros | Cons |
|----------|------|------|
| In-process (local) rate limiter | Zero latency, no network call | Limits not shared across servers |
| Centralized Redis | Accurate distributed limiting | Extra network hop (~0.5ms), SPOF risk |
| Leaky bucket | Smooth output rate | Extra queueing complexity, delays requests |
| Token bucket | Handles burst | More state to maintain |
| Sliding window counter | Accurate, efficient | Slight approximation at window boundary |

---

## L5 Additions

1. **Adaptive rate limiting:** Instead of static limits, adjust dynamically based on system load. If DB latency > 200ms, throttle all writes by 50% automatically. Implemented as a feedback loop: measure P99 latency → if exceeding SLO → reduce admission rate → stabilize. This is what Netflix's Concurrency Limit library does.

2. **Tiered rate limits with priority queues:** Not all requests are equal. Premium users > free users > unauthenticated. Implement priority queuing at the API gateway: high-priority requests bypass the strict counter and consume from a larger bucket. Use weighted fair queueing so low-priority traffic is served but not at cost to high-priority.

3. **Rate limiting as a service (RaaS):** Instead of embedding in each API server, run as a dedicated service with its own SLA. Use gRPC with deadlines — if RaaS call takes > 2ms, time out and fail open. Expose a batch check API: `check_many([{key, limit}, ...]) → [{allowed, remaining}]` to amortize network round trips for APIs with multiple resource checks.

4. **Distributed rate limiting with eventual consistency at edge:** For globally distributed APIs, use an eventually consistent model: each PoP (point of presence) tracks local counts and gossips updates to other PoPs every second. Interpolate global count. Accept a 2–5% overshoot window. This is how Cloudflare's distributed rate limiter works.

5. **Cost-aware rate limiting:** Rate limit not by request count but by resource cost units. A simple read = 1 unit. A complex query = 10 units. A file upload = 100 units. Users are limited by total cost/min, not request/min. Enables fairer capacity allocation and prevents expensive queries from dominating.
