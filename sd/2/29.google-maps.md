# 29. Google Maps — Location and Navigation Service

**Type:** data
**Difficulty:** G5
**Frequency:** Medium-High — geo data, routing algorithms, real-time traffic

---

## Problem Statement

Design a mapping and navigation service like Google Maps. Users can search for places, view maps, get turn-by-turn navigation between two points, and see real-time traffic conditions. The system must handle 1B daily active users, sub-200ms map tile serving, and route computation in < 2 seconds for any two points globally.

---

## Functional Requirements

**Must have:**
- Render map tiles (vector/raster) for any geographic region and zoom level
- Search for places by name or address (geocoding)
- Reverse geocoding: given lat/lng → address
- Route computation: driving/walking/cycling route from A to B with turn-by-turn
- Real-time traffic data overlaid on map and factored into routing

**Should have:**
- ETA estimation with traffic awareness
- Alternative routes (multiple options)
- Offline maps (download region for offline use)
- Reporting: accidents, speed traps, road closures (Waze-style crowdsourcing)

**Out of scope:**
- 3D building rendering
- Indoor mapping
- Transit routing (separate problem)

---

## Non-Functional Requirements

- **Scale:** 1B DAU, 100M route requests/day, 50B map tile serves/day
- **Latency:** Map tiles < 50ms; route < 2s; place search < 200ms
- **Availability:** 99.99% — maps are critical infrastructure
- **Data freshness:** Road closures reflected in routing within 5 minutes; new POIs within 24 hours

## Capacity Estimation

```
Map tiles:
  Web Mercator tile grid:
    Zoom 0: 1 tile (entire world)
    Zoom N: 4^N tiles
    Zoom 15 (street level detail): 4^15 = 1.07 billion tiles
    Active tiles (non-ocean, non-uninhabited): ~500M tiles
    Average tile size: 50KB (vector MVT format)
    Total tile data: 500M × 50KB = 25TB
  With all zoom levels (0–15): ~100TB total tile storage → S3
  Hot tiles (major cities, zoom 10–15): ~10TB → fits in CDN cache

  Tile serving:
    50B tiles/day = 578K/sec average; peak 2M/sec
    CDN cache hit rate: ~95% for urban tiles
    Origin (S3) requests: 5% × 578K = 29K/sec → S3 can handle this

Route computation:
  100M routes/day = 1,160/sec average; peak 10K/sec
  World road graph: ~50M nodes (intersections), ~100M edges (road segments)
  With Contraction Hierarchies preprocessing: query time ~5–10ms per route
  → 100 routing servers handle peak 10K/sec (each handles ~100 queries/sec)

Place search (geocoding/POI):
  10B searches/day = 116K/sec → Elasticsearch cluster (10 shards, 3 replicas)
  Place index: 200M POIs × 300B = 60GB → fits in Elasticsearch with field compression

GPS probe ingest (traffic):
  1B DAU × 10% driving at any time × 1 update/5s = 20M GPS updates/sec
  Event size: 50B → 1GB/sec ingest → Kafka cluster (50 partitions, keyed by segment_id)
  After map-matching and aggregation: ~1M road segment speed updates/min

Storage summary:
  Map tiles (S3): 100TB
  Road graph (in-memory per routing server): 4GB per server × 100 servers = 400GB RAM
  Traffic data (Cassandra, current speeds): 100M segments × 100B = 10GB → hot in RAM
  Historical traffic (Cassandra, partitioned): 1 year × 100M segments × 24×7 time slots = 4TB
  Place index (Elasticsearch): 200M POIs × 300B = 60GB
```

## Data Model

**Road graph (in-memory, custom binary format):**
```
Nodes (intersections): ~50M globally
  node_id: uint32 (4B)
  lat: int32 (fixed-point, 1e-7 degrees precision, 4B)
  lng: int32 (4B)
  edge_start_idx: uint32 (index into edge array, for fast adjacency lookup)
  Total per node: 16B × 50M nodes = 800MB

Edges (road segments): ~100M globally
  edge_id: uint32 (4B)
  from_node: uint32 (4B)
  to_node: uint32 (4B)
  distance_m: uint32 (4B)
  speed_limit_kph: uint8 (1B)
  road_class: uint8 (1B — motorway, trunk, primary, secondary, residential)
  flags: uint8 (1B — is_one_way, is_toll, is_ferry, is_unpaved)
  shortcut_of: uint32 (4B — for Contraction Hierarchy shortcuts; 0 if real edge)
  Total per edge: 23B × 100M edges = 2.3GB
  With shortcuts (~50% overhead from CH): ~3.5GB total edges

Total road graph in RAM per routing server: ~4.3GB
Modern server: 64–128GB RAM → easily holds graph with room for OS page cache + JVM heap
```

**Traffic data (Cassandra + Redis):**
```sql
-- Cassandra: current and historical segment speeds
CREATE TABLE segment_speeds (
  segment_id     bigint,                   -- maps to edge_id in road graph
  timestamp      timestamp,
  speed_kph      float,
  source         text,                     -- 'gps_probe' | 'sensor' | 'user_report'
  probe_count    int,                      -- how many GPS probes contributed
  PRIMARY KEY (segment_id, timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC)
  AND default_time_to_live = 604800;       -- 7-day TTL for individual readings

-- Redis: current speed per segment (hot cache, updated every minute)
Key: speed:{segment_id} → speed_kph (FLOAT, TTL = 300s / 5 minutes)
-- Routing servers read from Redis for current traffic overlay
-- Fallback: read from Cassandra if Redis key expired

-- Historical traffic (for ETA on future departure)
CREATE TABLE historical_traffic (
  segment_id     bigint,
  day_of_week    tinyint,           -- 0=Sunday, 6=Saturday
  hour_of_day    tinyint,           -- 0–23 UTC
  avg_speed_kph  float,
  percentile_85  float,             -- 85th percentile speed (for conservative ETA)
  sample_count   int,
  PRIMARY KEY (segment_id, day_of_week, hour_of_day)
);
-- Pre-computed from 6 months of GPS probe data
-- 100M segments × 7 days × 24 hours × 3 values × 4B = ~200GB
```

**Map tile storage:**
```
Vector tiles (MVT) in S3: 100TB
  s3://maps-tiles/v3/{zoom}/{x}/{y}.mvt
  Generated from: OpenStreetMap data (CC-BY-SA) + commercial data (DigitalGlobe, TomTom)
  Updated: incremental nightly for changed tiles; full rebuild quarterly

Raster tiles (PNG, pre-rendered) in S3: 300TB (larger due to 3 style variants)
  s3://maps-tiles/raster/{style}/{zoom}/{x}/{y}.png
  Styles: roadmap, satellite, terrain
  Only zoom 0–13 pre-rendered (14–20 rendered on-demand by tile service)
```

## Deep Dive 1: Route Computation Algorithm

**Why naive shortest path is too slow:**
```
World road graph: 50M nodes, 100M edges
Plain Dijkstra: O(E log V) = O(100M × log 50M) = O(100M × 26) ≈ 2.6B operations
  → ~2 seconds per query on a single core → not fast enough for 10K queries/sec

A* with geographic heuristic:
  f(n) = g(n) + h(n)
    g(n): actual travel time from origin to node n
    h(n): optimistic estimate from n to destination
           = straight-line distance / max_possible_speed (e.g., 200 kph)
  Better than Dijkstra (expands fewer nodes) but still slow for cross-country routes
  Typically explores 5–10M nodes for a 1,000km route → ~500ms → still too slow
```

**Contraction Hierarchies (CH) — production standard:**
```
Phase 1: Offline preprocessing (~24 hours for world graph, parallelized)
  1. Assign importance rank to each node:
     Rank criteria: edge difference (shortcuts needed if removed),
                    road hierarchy (highway > arterial > residential),
                    number of contracted neighbors
  2. Contract nodes in importance order (lowest importance first):
     For each node v to contract:
       - Find all pairs (u, w) where v is on the only shortest path from u to w
       - Add shortcut edge: u → w with weight = dist(u,v) + dist(v,w)
       - Remove v from graph (logically)
  3. Result: augmented graph with ~50M shortcuts added
     Total edges: 100M real + 50M shortcuts = 150M edges

Phase 2: Online query (~5–10ms for any global route)
  Bidirectional Dijkstra on the contracted graph:
    Forward search: from origin, expanding nodes in ascending importance order
    Backward search: from destination, expanding nodes in ascending importance order
    Both searches only visit nodes with importance >= current frontier importance (upward search)
    Searches meet at a high-importance node (major highway or city center)
    Meeting point: node where forward_dist[v] + backward_dist[v] is minimized

  Why it's fast:
    Each direction explores only O(√N) nodes instead of O(N) (bidirectional CH property)
    50M node graph → each direction explores ~7,000 nodes → ~14K total node relaxations
    At 100ns per relaxation: 14K × 100ns = 1.4ms per query → well within 2s SLA

Unpacking shortcuts (to get turn-by-turn directions):
  Shortcuts represent multiple real road segments
  Recursive unpacking: if edge is a shortcut, find the bypassed node → recurse
  Until all edges are real road segments
  Store shortcut composition in edge table (shortcut_of field)
```

**Traffic-aware routing:**
```
At query time: modify edge weights with current traffic
  For each edge in road graph:
    base_time_sec = distance_m / (speed_limit_kph / 3.6)
    current_speed = redis.get(f"speed:{edge_id}") OR speed_limit_kph
    actual_time_sec = distance_m / (min(current_speed, speed_limit_kph) / 3.6)
    traffic_factor = actual_time_sec / base_time_sec  # 1.0 = free flow; 3.0 = congested

Problem: CH shortcuts are precomputed with base travel times (no traffic)
  CH shortcuts don't account for traffic on the contracted path segments

Solution A: Traffic-aware CH (Customizable Route Planning / CRP):
  Precompute a hierarchy topology that doesn't change when traffic changes
  At query time: customize edge weights with traffic → run CH with customized weights
  Trade-off: more complex preprocessing; support for fast traffic updates
  Used by: Microsoft Bing Maps

Solution B: Simpler — apply traffic only to top edges of CH hierarchy
  Highways (high-rank edges): apply traffic weights
  Local roads (low-rank edges): use base travel time (traffic data less reliable + lower impact)
  Approximation: good enough for major routes; slightly inaccurate for local detours
  Used by: many production routing systems
```

**G5:** Contraction Hierarchies are fast for point-to-point routing, but have a subtle correctness issue when traffic causes a normally-contracted road (e.g., a residential street) to become faster than the main highway. CH shortcuts pre-contracted that highway: the routing query follows the shortcut and never considers the shortcut-bypassed residential street as an alternative — even if traffic made it faster. The fix is Customizable Route Planning (CRP): separate the graph hierarchy topology (computed once, offline) from the edge weights (customized per query with live traffic). CRP allows exact traffic-aware routing with CH-level speed, but requires recomputing "customization cells" every time traffic changes — feasible at per-region granularity (recompute affected metro area every 2 minutes) but expensive to do globally at 1-minute refresh. This is a genuine engineering trade-off that Google Maps and HERE Maps handle differently, and it's worth discussing explicitly.

## Deep Dive 2: Map Tile Generation and Serving

**Tile coordinate system (Web Mercator / XYZ tiles):**
```
Web Mercator projection (EPSG:3857): projects sphere onto flat plane
  Distortion: accurate near equator; extreme distortion near poles
  Tiles are square: 256×256 or 512×512 pixels
  Zoom level N: 2^N × 2^N grid of tiles
  Zoom 15: 32,768 × 32,768 = 1.07B tiles total; ~500M non-empty (land areas)

Tile identification:
  (zoom=15, x=5242, y=12663) → bounding box in lat/lng
  Formula: lng = x / 2^zoom × 360 - 180
           lat_rad = arctan(sinh(π × (1 - 2y/2^zoom)))
           lat = lat_rad × 180 / π

Request URL: /tiles/15/5242/12663.mvt
  → S3 key: tiles/15/5242/12663.mvt → CDN caches with key = URL
```

**Vector tile generation pipeline:**
```
Data sources:
  - OpenStreetMap (CC-BY-SA, free): roads, buildings, POIs, land use, water bodies
  - Commercial data (paid): business hours, addresses, traffic restrictions, 3D buildings
  - Government data (free): official road network, speed limits, turn restrictions
  - Satellite imagery: background raster layer (separate from vector tiles)

Processing pipeline (run weekly for full refresh, nightly for changes):
  1. Download OSM planet.osm.pbf (100GB) or change-set diffs (daily, ~100MB)
  2. Import into PostGIS spatial database:
     - Normalize: address parsing, POI deduplication, language normalization
     - Enrich: add speed limits, turn restrictions from commercial data
  3. Generate vector tiles with Tippecanoe (Mapbox open-source):
     - For each zoom level 0–15:
       * Select appropriate features (don't show buildings at zoom 5)
       * Simplify geometries (fewer points at lower zoom = smaller tiles)
       * Clip to tile boundaries
       * Encode as protocol buffer (MVT format)
  4. Upload tiles to S3 (changed tiles only, via content hash diff)
  5. Invalidate CDN cache for changed tiles (surrogate key: affected tile URLs)

Differential updates (for nightly changes):
  - Compare OSM changeset to previous version
  - Identify affected geographic regions
  - Re-generate only tiles in those regions at zoom 12–15 (detail tiles change most)
  - ~1M tiles regenerated nightly vs 500M total (0.2% daily change rate)
```

**Serving at scale:**
```
CDN architecture:
  Hot tiles (zoom 10–15, urban areas): 95%+ CDN cache hit rate
    Cache TTL: 7 days (road networks change slowly)
  Traffic tiles (speed overlay): separate tile layer, 5-minute TTL (changes frequently)
    Traffic is a separate tile layer: /traffic-tiles/{zoom}/{x}/{y}.png
    Overlaid client-side on top of base map tiles
  Cold tiles (rural zoom 14–15, rare areas): CDN miss → S3 → serve
    S3 with CloudFront regional caching: < 100ms globally

Cost at scale:
  S3 storage: 100TB × $0.023/GB = $2,300/month
  CDN bandwidth: 50B tiles/day × 50KB = 2.5PB/day → at $0.008/GB = $20,000/day ($600K/month)
  → CDN cost is dominant; optimizing tile compression and cache hit rate is critical
  Tile compression: Brotli compression on vector tiles (additional 20% vs gzip)
```

**G5:** Map tile serving at 50B tiles/day has a subtle cache coherence problem during map data updates. When a road is added or modified, the affected tiles at zoom levels 10–15 must be invalidated across all 250 CDN PoPs. A typical road change affects a 10km corridor = ~100 tiles at zoom 15 = ~400 tiles total across zoom levels 10–15. With 250 PoPs, that's 100,000 individual cache invalidations per road change. With 10,000 road changes per day (from OSM and commercial data), that's 1 billion purge operations per day — exceeding most CDN purge rate limits. The scalable solution is version-based cache busting: embed a map data version number in the tile URL (`/tiles/v{version}/{zoom}/{x}/{y}.mvt`). On map update, increment the version → all cached tiles at the old version become "orphaned" (not purged, just never requested again). New requests use the new version URL → CDN cold start for updated areas. Eviction handles orphaned tiles over time. This approach requires zero purge API calls but temporarily doubles CDN storage during the transition.

## Deep Dive 3: Real-Time Traffic Collection

**GPS probe data (primary and highest-coverage source):**
```
Collection:
  Users with Google Maps app (or Android location services) running in background
  GPS update every 3–5 seconds when in a moving vehicle
  1B users × 10% driving at any time × 1 update/4s = 25M GPS updates/sec
  Event: { user_id_hash, lat, lng, timestamp, speed_reported_kph, heading }

Processing pipeline:
  1. Map matching (snap GPS point to nearest road segment):
     - GPS accuracy: ±5m → point could be on either side of road, or nearby street
     - HMM (Hidden Markov Model): probabilistic match considering speed, heading, road geometry
     - Output: (segment_id, speed_kph, direction, timestamp) per GPS point
  2. Speed smoothing per segment:
     - Collect all GPS points on segment_id in last 1 minute
     - Remove outliers (speeds > speed_limit × 1.5 → GPS noise)
     - Median speed (robust to outliers): segment's current speed
  3. Write to traffic store:
     - Redis: SET speed:{segment_id} {speed_kph} EX 300  (5-min TTL)
     - Cassandra: INSERT INTO segment_speeds for historical analysis

Privacy:
  User ID anonymized (hash rotated daily)
  GPS coordinates snapped to segment ID before storage (exact location not stored)
  Never store individual user's route (only aggregate segment speeds)
  GDPR: users can opt out of location sharing; consent required in EU
```

**Incident reports (crowdsourced):**
```
User reports: tap on map → report type (accident, speed camera, road closure, hazard)
  Report event: { user_id, lat, lng, report_type, timestamp }

Confidence pipeline:
  1. Publish to Kafka: incident-reports topic
  2. Flink processing:
     - Geo-cluster reports within 50m radius and 10-minute window
     - Count distinct users reporting same incident
     - Check GPS probe data: are cars slowing down in that area?
       Correlation: if probes show 50% speed reduction → incident confirmed
     - Compute confidence: 0.0–1.0
       confidence = min(1.0, reporter_count × 0.3 + probe_correlation × 0.7)
  3. Publish to map if confidence > 0.7:
     - Write to incidents DB (PostgreSQL, geo-indexed with PostGIS)
     - Update routing edge weights for affected segments: multiply travel_time × delay_factor
     - Expire: incident removed when probes show speed recovering to normal OR after 2h

Accuracy:
  High-confidence incidents (confidence > 0.9): virtually all real
  False positive rate: ~5% for confidence 0.7–0.9 (require confirmation)
  Human review: traffic team monitors high-severity incidents (road closures, major accidents)
```

**G5:** GPS probe data for traffic relies on crowdsourced density, but has a systematic coverage gap: low-traffic roads (residential streets, rural roads) have very few probes → speed estimates are stale or missing. Routing for a local detour through a neighborhood may use 30-minute-old speed data while main highway data is fresh (1-minute-old). The silent impact: when an algorithm routes through "apparently fast" residential streets based on stale data, actual travel time is worse than predicted — and the ETA is wrong. The mitigation has two parts: (a) confidence-weighted speed estimates — low probe count → high variance → use historical traffic model instead of sparse live data; (b) time-since-last-update decay — if last probe was 15 minutes ago, linearly interpolate between last probe speed and historical average for this time-of-day. This requires tracking `last_probe_timestamp` alongside `speed_kph` in the Redis cache, and is the reason why Google Maps ETA can be significantly better than OpenStreetMap-based alternatives in rural areas with different probe density.

## Reliability and Operations

- **Routing service failure:** Routing is stateless (CH graph loaded from S3 on startup). Kubernetes replaces failed pod automatically. Graph loading from S3: ~3 minutes (3.5GB network transfer at 20MB/sec). During loading: requests load-balanced to other healthy instances. CH graph versioned in S3: routing service can load any version — facilitates A/B testing new graph preprocessing.
- **Traffic data staleness:** If traffic pipeline is down (Kafka consumer lag grows), routing uses historical averages for affected segments. Alert on: Redis key `speed:{segment_id}` missing for > 5 minutes for highway segments. ETA is less accurate during outage but routing is still functional.
- **Map data correctness:** Issues reported by users: "turn not possible," "road doesn't exist," "business is closed." Quality pipeline: user reports → review queue → map editors verify → fix in OSM/commercial data → next nightly tile regeneration picks up fix. High-severity issues (wrong direction, missing major highway): emergency fix within 4 hours via on-call map editor.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Contraction Hierarchies | Plain A* | CH: 100–1000× faster queries after 24-hour preprocessing; exact results; A*: simpler, no preprocessing, but 100× slower — cannot meet 2s SLA for long routes |
| Vector tiles (MVT) | Pre-rendered raster PNG | Vector: 4× smaller files, GPU-rendered at any resolution, style-changeable without regeneration, labels rotate with map; Raster: simpler client, no GPU needed, but 4× larger and requires re-rendering per style |
| In-memory CH road graph | DB-backed routing | In-memory: 5–10ms queries; DB (PostGIS): 500ms–5s per route — 100× slower, cannot meet SLA |
| GPS probe traffic | Fixed sensor network (induction loops) | GPS: near-zero infrastructure cost, massive coverage via existing user base; Sensors: precise speed, works without smartphones, but $10K–50K per installation × 100M road segments = infeasible globally |
| HMM map-matching | Nearest-road snap | HMM: probabilistic, handles GPS noise, parallel roads (consider road sequence); Nearest snap: simple but assigns wrong road when GPS drifts, especially near highway junctions |

## G5

1. **Multi-modal routing:** "Get me to the airport" might combine: walk → subway → bus → walk. Each mode has different graph characteristics. Architecture: time-expanded graph where each node is (location, time) — critical because missing a scheduled transit requires waiting for the next one. Transit edges: (stop, departure_time) → (stop, arrival_time) based on GTFS schedule. Waiting is an edge with time cost. Dijkstra on this expanded graph finds the optimal combination of transit, walking, and cycling. Real-time transit data (GTFS-RT) updates arrival times as buses/trains run late. Practical complexity: the time-expanded graph is orders of magnitude larger than the road graph (100M nodes for road vs billions of (location, time) pairs for transit). Production: restrict to transit stop nodes only; use A* with time-of-day heuristics to prune the search.

2. **ETA prediction as an ML regression problem:** Traditional ETA = route_distance / average_segment_speeds. Modern ETA (Google's DeepETA, Apple's NavigationNet): ML regression on rich features. Feature set: route distance, road types along route, current traffic per segment, historical traffic for this day/hour, weather conditions (rain reduces speeds 10–20%), special events (stadium nearby, rush hour), vehicle type. Label: actual travel time from billions of completed navigation sessions. Model: gradient-boosted trees (fast inference, handles missing features) or a neural network with positional encoding for the route sequence. Accuracy: 20–30% lower RMSE on ETA vs formula-based approach. Training data: billions of navigation sessions with ground truth travel time → unique advantage for incumbents.

3. **Offline maps at petabyte scale:** 1B users download regional maps. Architecture: each region packaged as a versioned bundle. Bundle contents: vector tiles + road graph for offline routing + address index for offline geocoding + POI data. Sizes: city = 50MB, country = 500MB–2GB (Japan: ~1.5GB), continent = 5–10GB. S3 hosting: 1B users × 5% with offline download × 1GB avg = 50PB storage. Delta updates: when map data changes, generate binary diff between old bundle version and new (bsdiff algorithm) — typically 1–5% of bundle size changed per month. Client downloads delta (~20MB for country) instead of full bundle (~1GB). Privacy: offline routing runs entirely on-device (no server round-trip) — user's destination is never sent to servers during offline navigation. Addressing: bundle must include a searchable address trie for offline geocoding.

4. **Indoor navigation:** Airports, malls, hospitals need indoor maps. Data collection: surveying teams map floor plans + measure WiFi/BLE fingerprints throughout. Indoor positioning: GPS doesn't work indoors → triangulate from WiFi RSSI (signal strength at each access point), Bluetooth beacon RSSI, pedestrian dead reckoning (accelerometer + compass integration). Architecture: indoor maps stored as vector tiles with a floor dimension added (zoom, x, y, floor). Routing graph includes floor transitions (escalators, elevators, stairs) as edges with transfer times. Positioning service: client scans WiFi/BLE → sends signal signature to positioning service → returns estimated (lat, lng, floor, accuracy_m). Accuracy: WiFi-only: ±5m; BLE beacons: ±2m; sensor fusion: ±1m. Deployment cost: mapping + beacon installation per building limits coverage to major high-traffic venues.

5. **Street View and semantic map understanding:** Street View is not just a visual experience — it's a data collection flywheel. Architecture: Street View cars collect 360° imagery at every street. Image processing pipeline (billions of images): (a) privacy: face detection + license plate detection → Gaussian blur before any image serves publicly; (b) address number extraction: OCR on house number placards → feeds geocoding index; (c) business sign recognition: EAST text detector → business name normalization → POI discovery and verification; (d) road condition analysis: semantic segmentation → detect potholes, faded lane markings, missing signs → feed into map quality scoring. All processing is offline batch ML (GPU cluster). This creates a self-updating map: Street View cars re-drive areas every 1–3 years, and the ML pipeline automatically detects changes (new building, new business, road changes) and flags them for map update. This is the data collection advantage that gives Google Maps consistently better POI freshness than OSM in most regions.
