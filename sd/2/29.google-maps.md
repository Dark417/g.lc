# 29. Google Maps — Location and Navigation Service

**Type:** data
**Difficulty:** L5
**Frequency:** Medium-High — geo data, routing algorithms, real-time traffic

---

## Problem Statement

Design a mapping and navigation service like Google Maps. Users can search for places, view maps, get turn-by-turn navigation between two points, and see real-time traffic conditions. The system must handle 1B daily active users, sub-200ms map tile serving, and route computation in < 2 seconds for any two points globally.

---

## Functional Requirements

**Must have:**
- Render map tiles (vector/raster) for any geographic region and zoom level
- Search for places by name or address (geocoding)
- Reverse geocoding: given lat/lng → address
- Route computation: driving/walking/cycling route from A to B with turn-by-turn
- Real-time traffic data overlaid on map and factored into routing

**Should have:**
- ETA estimation with traffic awareness
- Alternative routes (multiple options)
- Offline maps (download region for offline use)
- Reporting: accidents, speed traps, road closures (Waze-style crowdsourcing)

**Out of scope:**
- 3D building rendering
- Indoor mapping
- Transit routing (separate problem)

---

## Non-Functional Requirements

- **Scale:** 1B DAU, 100M route requests/day, 50B map tile serves/day
- **Latency:** Map tiles < 50ms; route < 2s; place search < 200ms
- **Availability:** 99.99% — maps are critical infrastructure
- **Data freshness:** Road closures reflected in routing within 5 minutes; new POIs within 24 hours

---

## Capacity Estimation

```
Map tiles:
  World at zoom 15 (street level): ~500M tiles, each ~50KB = 25TB of tile data
  With all zoom levels: approximately 100TB total
  Hot tiles (major cities, zoom 10-15): ~10TB → fits in CDN cache

Route computation:
  100M routes/day = 1,160/sec average; peak ~10K/sec
  World road graph: ~50M nodes, ~100M edges
  Per route computation: modified Dijkstra/A* → O(E log V) per query
  Precomputed hierarchical graph → much faster in practice

Place search:
  10B searches/day = 116K/sec
  Place index: 200M POIs × 300B = 60GB → fits in distributed search index
```

## Data Model

**Road graph:**
```
Nodes (intersections):
  node_id (uint32), lat, lng, type (intersection/endpoint)
  ~50M nodes globally

Edges (road segments):
  edge_id, from_node, to_node, distance_m, speed_limit_kph, road_type, is_one_way
  travel_time_sec = distance_m / (speed_kph / 3.6)  ← base time
  traffic_factor = current_speed / speed_limit       ← from traffic data
  ~100M edges globally

Stored in: custom binary format + memory-mapped files for fast in-memory access
  Nodes: 50M × 16B = 800MB in RAM per routing server
  Edges: 100M × 32B = 3.2GB in RAM per routing server
  → Each routing server holds entire world graph in RAM
```

**Traffic data:**
```
Road segment speed updates (Cassandra):
  segment_id (edge_id), timestamp, speed_kph, source (GPS probe, sensor, report)
  Partitioned by segment_id, clustered by timestamp DESC
  Current speed: latest entry per segment (last 5 minutes)

Aggregated speed by time-of-day (historical traffic):
  segment_id, day_of_week, hour, avg_speed_kph
  Used for future ETA estimates ("leave at 5 PM" → uses historical Friday rush hour data)
```

**Map tile storage:**
```
Vector tiles (MVT format) in S3:
  s3://maps/tiles/{zoom}/{x}/{y}.mvt
  Generated offline from OpenStreetMap or proprietary data
  Regenerated when roads/POIs change (incremental updates)

Raster tiles (PNG) pre-rendered from vector data:
  s3://maps/raster-tiles/{style}/{zoom}/{x}/{y}.png
  Styles: roads, satellite, terrain
```

## Deep Dive 1: Route Computation Algorithm

**A* algorithm for single-pair shortest path:**
```
f(n) = g(n) + h(n)
  g(n): actual cost from start to n (travel time)
  h(n): heuristic estimate from n to goal (Euclidean distance / max_speed)

Priority queue: nodes sorted by f(n)
Expand nodes with lowest f(n) first

A* is fast for road graphs because:
  Road graphs have geographic structure → heuristic is tight → few nodes expanded
  1M nodes explored out of 50M for a cross-country route

Real-world challenge: A* is still too slow for very long routes on a 50M-node graph
```

**Contraction Hierarchies (CH) — production approach:**
```
Preprocessing (offline, hours):
  1. "Contract" less important nodes: assign importance rank to each node
     (road hierarchy: highway > arterial > residential)
  2. Add "shortcut edges" that bypass low-importance nodes
  3. Build a hierarchical graph where search stays on important roads

Query (online, milliseconds):
  Bidirectional Dijkstra on the contracted graph:
    Forward search from origin (going up the hierarchy)
    Backward search from destination (going up the hierarchy)
    Meet in the middle at high-importance node (major highway)

Performance:
  Preprocessing: ~24 hours for full world graph
  Query: ~1-10ms for any global query (vs minutes for plain Dijkstra)
  Memory: ~2x overhead for shortcuts

Used by: OSRM (Open Source Routing Machine), Graphhopper
```

**Traffic-aware routing:**
```
At query time: overlay current traffic on the edge weights
  travel_time = distance / min(speed_limit, current_speed)

Current speed from: GPS probe data (anonymized user GPS), sensor loops, crowd reports
Aggregated every 1 minute per road segment
Stored in Redis (hot) + Cassandra (historical)

Integration:
  Load traffic weights into routing graph (in-memory, shared memory across routing instances)
  Update routing weights every 1 minute from traffic service
  Re-query with updated weights → ETA reflects current conditions
```

## Deep Dive 2: Map Tile Generation and Serving

**Tile coordinate system (Web Mercator / XYZ tiles):**
```
Zoom level 0: 1 tile covers entire world
Zoom level 1: 4 tiles (2×2)
Zoom level N: 4^N tiles
Zoom level 15 (street detail): 2^30 = 1 billion tiles (most are ocean/uninhabited → skip)
  Active tiles: ~500M

Tile identification: (zoom, x, y) → bounding box in lat/lng
```

**Vector tile pipeline:**
```
1. Source data: OSM (OpenStreetMap) + commercial data
2. Process with Tippecanoe (open source vector tile generator):
   - Split data into tiles at each zoom level
   - Simplify geometry at lower zooms (fewer points on roads at zoom 5 vs zoom 15)
   - Filter features by zoom (buildings only at zoom > 13, POIs at zoom > 12)
3. Store vector tiles in S3
4. When data changes (road added, building updated):
   - Identify affected tiles at each zoom level
   - Re-generate only those tiles
   - Invalidate CDN cache for affected tiles

Vector tile advantages over raster:
  - Small file (50KB vs 200KB for PNG)
  - Rendered on client (GPU) → crisp at any resolution
  - Style changes without re-generating tiles
  - Labels rotate correctly with map rotation
```

**Serving at scale:**
```
CDN caches tiles:
  Popular areas (NYC, London at zoom 10-15): 95%+ cache hit rate
  Cache TTL: 7 days for tiles (change infrequently); 5min for traffic tiles

Cold areas (rural Bhutan at zoom 15): CDN miss → S3 origin → serve
  S3 direct serve: < 100ms globally via CloudFront

Total tile storage: 100TB in S3 (~$2,300/month)
CDN bandwidth: 50B tiles/day × 50KB = 2.5PB/day → CDN critical
```

## Deep Dive 3: Real-Time Traffic Collection

**GPS probe data (primary source):**
```
Users with Google Maps app (background) → location updates every 3-5 seconds when driving
  1B users × 10% driving at any time × 1 update/4s = 25M GPS updates/sec

Processing pipeline:
  GPS point → map match to road segment (snap to nearest road)
  Compute speed: distance since last point / time elapsed
  Smooth: average of last 10 points for this segment
  Aggregate: median speed of all probes on segment in last 1 minute

Privacy: anonymize before storage; never store identifiable routes
  Just: (segment_id, timestamp, speed_kph, direction)
```

**Traffic incident reports:**
```
User reports: accident, police, speed camera, hazard
  → Appear on map after N confirmations (fraud prevention)
  → Expire after time or when contradicted by speed data returning to normal

Pipeline:
  Report → Kafka → incident processing → confidence scoring → DB
  Confidence: how many reports, are probes slowing down in that area?
  Only publish to map when confidence > 0.8
```

## Reliability and Operations

- **Routing service failure:** Routing is stateless (graph in memory). Auto-healing: K8s replaces failed pod. Graph loaded from S3 on startup (~3 minutes). During recovery: load balance to healthy instances.
- **Traffic data staleness:** If traffic pipeline is down, routing uses historical averages. ETA is less accurate but functional. Alert on traffic data age > 5 minutes.
- **Map data correctness:** Crowd reports + auto-import from government data. Review queue for significant changes (new highway). Reversions if change causes routing errors.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Contraction Hierarchies | Plain A* | CH: 1000x faster queries after preprocessing; A*: simpler, slower |
| Vector tiles | Raster tiles | Vector: smaller, resolution-independent; Raster: simpler rendering |
| In-memory road graph | DB-backed | In-memory: fast queries; DB: accurate but 100x slower |
| GPS probe traffic | Sensor-based | GPS: free, massive coverage; Sensors: precise but expensive infrastructure |

## L5 Additions

1. **Multi-modal routing:** "Get me from my house to the airport" might combine: walk → subway → bus → walk. Each transport mode has different graph edges. Algorithm: time-expanded graph where nodes are (location, time) pairs. Transit schedules become edges with departure times. Waiting is an edge with time cost. Dijkstra finds optimal combined path. Real-time transit data (GTFS-RT) updates travel times. This is significantly harder than driving routing — the state space is (location × time) vs just location.

2. **ETA prediction as ML problem:** Traditional ETA = route length / average speed. Modern ETA (DeepETA, Google Maps ML): treats ETA as a regression problem. Features: route distance, road types, current traffic per segment, historical traffic for this time/day, weather, special events. Label: actual travel time from historical trips. Model: gradient-boosted trees or neural network. Accuracy: 20-30% better ETA vs formula-based. Requires: massive training data (billions of completed trips with actual vs estimated time).

3. **Offline maps at petabyte scale:** 1B users download regional maps for offline use. Architecture: each region pre-packaged as a bundle (100MB for a city, 2GB for a country). Bundles stored in S3, served via CDN. Version management: bundle has a version hash; client checks version daily. Delta updates: only changed tiles are in the update package (binary diff). Privacy: offline routing runs entirely on device; no routing requests sent to server for offline regions. Storage on device: LevelDB or SQLite for tile metadata + files for tile data.

4. **Indoor navigation:** Airports, malls, hospitals need indoor maps. Data collection: surveying teams map floor plans. Positioning: GPS doesn't work indoors → WiFi fingerprinting, Bluetooth beacons, sensor fusion (accelerometer + compass). Architecture: indoor maps stored separately from outdoor maps. Floor plans as vector tiles with floor dimension. Routing: graph changes per floor + staircase/elevator transitions between floors. Positioning: client collects WiFi RSSI scan → sends to server → server returns estimated position (or run ML model on-device).

5. **Street View and spatial computer vision:** Not just navigation — semantic understanding of the world. Architecture: Street View cars collect 360° imagery. Image processing: (a) face and license plate blurring (privacy), (b) address number extraction (OCR for geocoding), (c) business sign detection (POI discovery), (d) road condition analysis (pothole detection for municipalities). ML pipeline: batch processing on collected imagery → extract structured data → feed into map database. This is the data collection flywheel that gives Google Maps its accuracy advantage.
