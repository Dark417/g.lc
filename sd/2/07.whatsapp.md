# 07. WhatsApp — Chat Messaging System

**Type:** social-media
**Difficulty:** L4–L5
**Frequency:** Very High — real-time messaging is a canonical design

---

## Problem Statement

Design a messaging system like WhatsApp or Facebook Messenger. Users can send 1:1 messages and group messages. Messages must be delivered reliably (at-least-once), support offline delivery (messages stored until recipient comes online), and handle 100M DAU sending 60B messages/day with < 100ms delivery latency.

---

## Functional Requirements

**Must have:**
- Send and receive 1:1 messages (text, images, files)
- Group messaging (up to 500 members)
- Offline message delivery: store messages until recipient is online
- Message status: sent, delivered, read (✓, ✓✓, 🔵✓✓)
- Message ordering: messages in a conversation are ordered

**Should have:**
- Real-time delivery when online (WebSocket or long-polling)
- Push notifications when offline
- Message history (last 90 days)
- Last seen and online presence

**Out of scope:**
- End-to-end encryption (mention Signal Protocol as the approach)
- Voice/video calls (separate design)
- Story/status posts

---

## Non-Functional Requirements

- **Scale:** 100M DAU, 60B messages/day, ~700K messages/sec
- **Availability:** 99.99% — messaging is mission-critical
- **Latency:** < 100ms delivery for online recipients
- **Durability:** No message loss — messages must survive server crashes
- **Ordering:** Messages within a conversation maintain order (by server-side timestamp)

---

## Capacity Estimation

```
Messages: 60B/day = 700K/sec average; peak ~2M/sec
Message size avg: 200 bytes (text) → 60B × 200B = 12TB/day
Media: ~30% messages have media → 18B media messages × 50KB avg = ~900TB/day → S3

Online users: 100M DAU × 40% concurrent at peak = 40M concurrent connections
WebSocket connections: 40M connections × ~5KB/connection = 200GB memory for connection state
  → Need hundreds of chat servers; each handles ~100K connections
```

---

## High-Level Architecture

```
[Client App]
    │
    ├─ WebSocket ──→ [Chat Server] ──→ [Message Queue (Kafka)] ──→ [Message Store (Cassandra)]
    │                     │                                               │
    │             [Presence Service]                              [Notification Service]
    │             [Session Store (Redis)]                              │
    │                                                          [APNS / FCM]  (offline)
    │
    └─ REST ──→ [API Server] ──→ [User Service] / [Group Service] / [Media Service]
                                        │
                                 [PostgreSQL / DynamoDB]
```

**Components:**
- **Chat Server:** Maintains WebSocket connections. Routes messages to online recipients or queues for offline delivery. Stateful — each client sticks to one chat server.
- **Message Queue (Kafka):** Decouples message receipt from delivery. Buffer for offline messages. Fan-out for group messages.
- **Message Store (Cassandra):** Persistent message history, partitioned by conversation_id.
- **Presence Service:** Tracks online/offline status, last seen. Redis with 30s heartbeat TTL.
- **Session Store (Redis):** Maps user_id → chat_server_id for routing messages.
- **Notification Service:** Pushes notifications to offline users via APNS/FCM.

## API Design

```
# WebSocket events (bidirectional)
Client → Server:
  { type: "message", to: user_id, conversation_id, content, client_msg_id, timestamp }
  { type: "ack", message_id }
  { type: "heartbeat" }

Server → Client:
  { type: "message", from: user_id, message_id, content, timestamp }
  { type: "delivery_receipt", message_id, status: "delivered"|"read" }
  { type: "presence", user_id, status: "online"|"offline", last_seen }

# REST APIs
GET /v1/conversations/{id}/messages?before=<cursor>&limit=50
POST /v1/media/upload → { media_id, url }
GET /v1/users/{id}/presence
POST /v1/groups (create group)
PUT /v1/groups/{id}/members
```

## Data Model

```sql
-- Cassandra: messages partitioned by conversation_id
messages (
  conversation_id  UUID PARTITION KEY,
  message_id       TIMEUUID CLUSTERING KEY DESC,  -- time-ordered UUID
  sender_id        UUID,
  content          TEXT,
  media_url        TEXT,
  status           TEXT,  -- sent, delivered, read
  created_at       TIMESTAMP
)

-- Conversations table
conversations (
  conversation_id  UUID PRIMARY KEY,
  type             TEXT,  -- "1:1" or "group"
  participants     SET<UUID>,
  last_message_id  UUID,
  updated_at       TIMESTAMP
)

-- User inbox: what conversations does user have, in what order?
user_conversations (
  user_id          UUID PARTITION KEY,
  updated_at       TIMESTAMP CLUSTERING KEY DESC,
  conversation_id  UUID
)

-- Redis: online status
Key: presence:{user_id}
Value: { server_id, connected_at }
TTL: 30 seconds (refreshed by heartbeat)

-- Redis: message delivery queue (offline users)
Key: offline_queue:{user_id}
Type: List
Value: message_id (fetch full message from Cassandra)
```

## Deep Dive 1: Real-Time Message Delivery

**Online path (happy path):**
```
1. Sender sends message via WebSocket to Chat Server A
2. Chat Server A:
   a. Assigns message_id (Snowflake)
   b. Writes to Cassandra (async, but WAL guarantees durability)
   c. Looks up recipient's chat server from Session Store (Redis)
   d. If recipient is on Chat Server B: send via internal message bus (Kafka topic: chat-delivery)
   e. Chat Server B pushes message to recipient's WebSocket
3. Recipient's app sends ACK
4. Server updates message status to "delivered"
5. Sender's app receives delivery receipt via WebSocket
```

**Delivery receipt flow:**
```
sent (✓): message_id written to Cassandra
delivered (✓✓): recipient's device ACKs receipt
read (🔵✓✓): recipient opens the conversation
```

**Message ordering:**
- Use `message_id` as TIMEUUID (time-based UUID v1) — naturally ordered by time
- Within same millisecond: lexicographic comparison of UUID random component
- Sender's client shows optimistic ordering; server corrects if needed

---

## Deep Dive 2: Offline Message Delivery

**When recipient is offline:**
```
1. Chat Server can't find recipient in Session Store (not online)
2. Route message to Offline Queue Service
3. Write message_id to Redis offline queue: LPUSH offline_queue:{user_id} {message_id}
4. Trigger push notification via APNS (iOS) or FCM (Android)
   → "New message from Alice"
5. When user comes online:
   a. Connect to Chat Server, authenticates
   b. Chat Server reads offline queue: LRANGE offline_queue:{user_id} 0 -1
   c. Fetch full messages from Cassandra by message_id
   d. Deliver in order, clear queue
   e. Send delivery receipts back to senders
```

**Push notification reliability:**
- FCM/APNS are best-effort. If device is on airplane mode, queued at FCM.
- Max FCM storage: 4 weeks (collapsed/uncollapsed).
- Don't send message content in push payload (privacy) — just "You have new messages"
- On app open, pull unread messages from server anyway.

---

## Deep Dive 3: Group Message Fan-out

**Group messaging challenge:** Message to 500-person group → 500 delivery operations

**Approach:**
```
1. Sender sends message to Chat Server
2. Chat Server writes message to Cassandra with conversation_id = group_id
3. Publishes event to Kafka: topic "group-messages", key = group_id
4. Fan-out service (sharded by group_id):
   a. Reads group membership from Group Service
   b. For each online member: route to their Chat Server via internal bus
   c. For each offline member: add to offline queue + trigger push notification
   d. Skip sender (they already have the message)
```

**Fan-out optimization:**
- Cache group membership (Redis), refreshed on member add/remove
- Batch delivery: bundle up to 100 member deliveries per Kafka message
- For large groups (500 members): use server-side rendering — each member's client fetches group timeline on open (pull model for large groups)

---

## Reliability and Operations

- **Chat server failure:** Client reconnects to any available Chat Server. Session Store updated. Undelivered in-flight messages are in Kafka — redelivered by fan-out service.
- **Cassandra failure:** Write fails → client retries (idempotent by message_id). At-least-once delivery guarantee.
- **Duplicate detection:** Cassandra upsert by message_id is idempotent. Client assigns client_msg_id; server deduplicates by (sender_id, client_msg_id) within a time window.
- **Message expiry:** For end-to-end encrypted apps, implement message expiry: messages deleted from Cassandra after TTL. For regular apps: archive to cold storage after 90 days.

---

## Security and Abuse

- End-to-end encryption: Signal Protocol (Double Ratchet algorithm) — keys stored only on devices, server never sees plaintext
- Authentication: OAuth2 + device-specific session tokens
- Spam detection: rate limit new accounts; ML classifier on message content (server-side, before encryption)
- Block/report: blocked user's messages dropped at routing layer

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| WebSocket | Long-polling | WebSocket: real-time, bidirectional; Long-poll: simpler, higher latency |
| Cassandra | MySQL | Cassandra: scales writes, flexible schema; MySQL: richer queries, harder to shard |
| Kafka fan-out | Direct DB poll | Kafka: decoupled, scalable; Direct poll: simpler but tight coupling |
| Pull for large groups | Push fan-out | Pull: cheaper writes; Push: lower read latency |

## L5 Additions

1. **Connection draining and hot server migration:** When a chat server is overloaded or needs to restart, it must drain connections gracefully. Send each connected client a "reconnect" frame directing them to a new server address. Connections migrate over 30 seconds. Load balancer health check excludes draining server from new connections. Client SDK has exponential backoff + jitter for reconnects.

2. **Message search:** Full-text search across message history is expensive. Two approaches: (a) Client-side search: messages downloaded to device, search runs locally (WhatsApp's approach for E2E encrypted chats). (b) Server-side search: Elasticsearch index per user, updated async on every message. For large orgs (Slack/Teams), server-side is necessary. Index only non-encrypted messages; for E2E, client must decrypt and index locally.

3. **Multi-device support:** User has phone + tablet + web browser — all should receive the same messages. Session Store maps user_id → [list of chat servers, one per device]. Fan-out sends to all active sessions. Offline queue is per-device, not per-user. Message read state synced across devices via a separate read-receipt event (marking read on one device marks read for all via a read-sync event).

4. **Message sync for chat history:** When user installs app on new device, they need message history. For non-E2E: pull from Cassandra. For E2E (encrypted): history is not available on server in readable form. Options: (a) WhatsApp uses iCloud/Google Drive encrypted backup — key stored by user. (b) Share encryption keys between devices (key agreement protocol). (c) Only sync message IDs; content is lost (simplest, but bad UX).

5. **Geo-distributed architecture:** Users in different regions connect to nearest data center. Chat servers are regional. Message routing: if sender and receiver are in different regions, message travels from sender's region to receiver's region via a private backbone (not public internet). Region routing table maps user_id → home region (stored in global Redis or DNS). Cross-region latency adds ~50ms overhead; acceptable given 100ms SLA.
