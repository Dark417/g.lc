# 07. WhatsApp — Chat Messaging System

**Type:** social-media
**Difficulty:** G4–G5
**Frequency:** Very High — real-time messaging is a canonical design

---

## Problem Statement

Design a messaging system like WhatsApp or Facebook Messenger. Users can send 1:1 messages and group messages. Messages must be delivered reliably (at-least-once), support offline delivery (messages stored until recipient comes online), and handle 100M DAU sending 60B messages/day with < 100ms delivery latency.

---

## Functional Requirements

**Must have:**
- Send and receive 1:1 messages (text, images, files)
- Group messaging (up to 500 members)
- Offline message delivery: store messages until recipient is online
- Message status: sent, delivered, read (one checkmark, two checkmarks, blue checkmarks)
- Message ordering: messages in a conversation are ordered

**Should have:**
- Real-time delivery when online (WebSocket or long-polling)
- Push notifications when offline
- Message history (last 90 days)
- Last seen and online presence

**Out of scope:**
- End-to-end encryption (mention Signal Protocol as the approach)
- Voice/video calls (separate design)
- Story/status posts

---

## Non-Functional Requirements

- **Scale:** 100M DAU, 60B messages/day, ~700K messages/sec
- **Availability:** 99.99% — messaging is mission-critical
- **Latency:** < 100ms delivery for online recipients
- **Durability:** No message loss — messages must survive server crashes
- **Ordering:** Messages within a conversation maintain order (by server-side timestamp)

## Capacity Estimation

```
Message throughput:
  60B messages/day ÷ 86,400 sec = 694,444/sec average
  Peak (3× average) = ~2,083,333/sec ≈ 2M/sec

Message storage:
  Average text message: 200 bytes
  60B × 200B = 12TB/day text messages
  90-day retention: 12TB × 90 = ~1.08PB (just text)
  Media (30% of messages have attachments):
    18B messages × average 50KB media = 900TB/day → stored in S3 separately

WebSocket connections:
  100M DAU × 40% concurrent at peak = 40M simultaneous connections
  Memory per connection: ~5KB (socket buffer + session state)
  40M × 5KB = 200GB of connection state
  Each chat server handles 200K connections (commodity server with 4GB RAM for sessions)
  Fleet size: 40M / 200K = 200 chat servers (plus redundancy)

Kafka throughput:
  2M messages/sec × ~500B/message = 1GB/sec Kafka ingestion
  With 3× replication: 3GB/sec Kafka write bandwidth
  → 5 Kafka brokers × 600MB/s = sufficient (with headroom)
```

## High-Level Architecture

```
[Client App]
    │
    ├─ WebSocket ──→ [Chat Server fleet]
    │                     │              │
    │              [Session Store]  [Message Routing]
    │              (Redis: who's       │
    │               on which server)   ▼
    │                          [Kafka: message-events]
    │                                  │
    │                     ┌────────────┴────────────┐
    │                     ▼                         ▼
    │              [Message Processor]    [Fan-out Service]
    │                     │               (for groups)
    │              [Message Store]
    │              (Cassandra: persistent history)
    │                     │
    │              [Notification Service]
    │                     │
    │              [APNS / FCM] ←── offline users
    │
    └─ REST ──→ [API Server] ──→ [User Service (PostgreSQL)]
                                 [Group Service (PostgreSQL)]
                                 [Media Service (S3)]
```

**Components:**
- **Chat Server:** Maintains WebSocket connections. Stateful — each client is pinned to one server for the duration of a session. Routes messages to online recipients via internal message bus (Kafka). Handles 200K connections each.
- **Session Store (Redis):** Maps `user_id → {chat_server_id, connected_at}`. Used by routing logic to find where a recipient is connected. TTL = 30s, refreshed by heartbeat.
- **Kafka:** Decouples write (from sender) from read delivery (to recipient). Also serves as the fan-out backbone for group messages.
- **Message Store (Cassandra):** All persistent message history, partitioned by `conversation_id`. Write-optimized, append-mostly workload.
- **Presence Service:** Online/offline status with 30-second heartbeat. Redis sorted set of `{user_id: last_heartbeat_timestamp}`.
- **Notification Service:** Pushes notifications to offline users via APNS (iOS) and FCM (Android). Async, decoupled from the delivery path.

## API Design

```
# WebSocket events (bidirectional, MessagePack encoded for efficiency)
Client → Server:
  { type: "msg",       to: user_id, conversation_id, content, client_msg_id, ts }
  { type: "ack",       message_id }          -- delivery acknowledgment
  { type: "read",      conversation_id, up_to_message_id }  -- mark as read
  { type: "heartbeat" }                      -- keeps session alive + updates presence

Server → Client:
  { type: "msg",      from: user_id, message_id, content, ts }
  { type: "receipt",  message_id, status: "delivered"|"read" }
  { type: "presence", user_id, status: "online"|"offline", last_seen }
  { type: "ack",      client_msg_id, server_message_id }  -- confirms msg received by server

# REST APIs (for non-real-time operations)
GET /v1/conversations/{id}/messages?before=<cursor>&limit=50
POST /v1/media/upload → { media_id, upload_url }  -- returns S3 presigned URL
GET /v1/users/{id}/presence
POST /v1/groups               Body: { name, member_ids }
PUT /v1/groups/{id}/members   Body: { add?: [], remove?: [] }
```

## Data Model

```sql
-- Cassandra: message storage, partitioned by conversation_id
-- All messages in a conversation → single partition (efficient range reads)
messages (
  conversation_id  UUID,           -- PARTITION KEY: routes to correct Cassandra node
  message_id       TIMEUUID,       -- CLUSTERING KEY DESC: time-ordered UUID v1
  sender_id        UUID,
  content_type     TEXT,           -- 'text', 'image', 'video', 'file'
  content          TEXT,           -- text content or media_url
  status           TEXT,           -- 'sent', 'delivered', 'read'
  created_at       TIMESTAMP,
  PRIMARY KEY (conversation_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC)
  AND default_time_to_live = 7776000;  -- 90 days in seconds

-- Conversations index
conversations (
  conversation_id  UUID PRIMARY KEY,
  type             TEXT,           -- '1:1' or 'group'
  participants     SET<UUID>,
  last_message_id  UUID,
  updated_at       TIMESTAMP
)

-- User's conversation list (for inbox view)
user_conversations (
  user_id          UUID,           -- PARTITION KEY
  updated_at       TIMESTAMP,      -- CLUSTERING KEY DESC (most recent first)
  conversation_id  UUID,
  PRIMARY KEY (user_id, updated_at, conversation_id)
) WITH CLUSTERING ORDER BY (updated_at DESC)

-- Redis: presence tracking
Key:   presence:{user_id}
Type:  Hash
Value: {server_id: "chat-server-42", connected_at: 1735689600}
TTL:   30 seconds (refreshed every 15s by client heartbeat)

-- Redis: offline message queue (per device for multi-device support)
Key:   offline_queue:{user_id}:{device_id}
Type:  List (LPUSH for new msgs, RPOP when device comes online)
Value: message_id (full content fetched from Cassandra)
TTL:   30 days (after which push notification is all that remains)
```

## Deep Dive 1: Real-Time Message Delivery

**Happy path — both sender and recipient online:**
```
Step 1: Sender's client sends WebSocket frame to Chat Server A
  {
    type: "msg",
    to: "user456",
    conversation_id: "conv789",
    content: "Hello!",
    client_msg_id: "client-local-uuid-123"  ← for deduplication
  }

Step 2: Chat Server A processes message:
  a. Assign server-side message_id (TIMEUUID — time-based UUID v1)
  b. Send server ACK back to sender:
     { type: "ack", client_msg_id: "client-local-uuid-123", server_message_id: "timeuuid-abc" }
     → Sender can now show checkmark (✓ "sent")
  c. Write to Cassandra asynchronously (WAL ensures durability before ack):
     INSERT INTO messages (conversation_id, message_id, sender_id, content, status)
     VALUES ('conv789', timeuuid-abc, 'user123', 'Hello!', 'sent')
  d. Look up recipient's session: Redis GET presence:user456
     → { server_id: "chat-server-17", connected_at: ... }

Step 3: Cross-server delivery:
  If recipient on same server (Chat Server A): direct WebSocket push
  If recipient on different server (Chat Server B):
    Publish to Kafka topic "chat-delivery", key=conversation_id:
    { message_id, from, to, content, server_id: "chat-server-17" }
    Chat Server B consumes from its partition, pushes to user456's WebSocket

Step 4: Recipient's client sends ACK:
  { type: "ack", message_id: "timeuuid-abc" }

Step 5: Chat Server B updates message status:
  UPDATE messages SET status='delivered' WHERE conversation_id='conv789' AND message_id=timeuuid-abc
  Send delivery receipt to sender:
  { type: "receipt", message_id: "timeuuid-abc", status: "delivered" }
  → Sender sees ✓✓ "delivered"

Step 6: Read receipt:
  When user456 opens the conversation:
  { type: "read", conversation_id: "conv789", up_to_message_id: "timeuuid-abc" }
  → Server sends { type: "receipt", message_id: "timeuuid-abc", status: "read" }
  → Sender sees 🔵✓✓ "read"

Total latency (happy path): ~20–50ms (mostly network RTT)
```

**Message ordering guarantee:**
```
TIMEUUID (UUID v1) = 60-bit timestamp (100ns resolution) + clock sequence + node ID
  → Naturally ordered by creation time across nodes
  → No two UUIDs from the same node are equal
  → Within 100ns resolution: lexicographic comparison of random component

Client-side optimistic display:
  Sender displays message immediately in UI (before server ACK)
  Server ACK may reorder if network reordering occurs
  Client corrects display order based on server-assigned message_id
```

**G5:** The WebSocket delivery path has a subtle at-most-once failure mode: if Chat Server B crashes between receiving the Kafka message and pushing it to the recipient's WebSocket, the message is consumed from Kafka (offset committed) but never delivered. The Kafka consumer group offset should not be committed until the WebSocket push is confirmed. However, WebSocket push confirmation is not a synchronous acknowledgment — the server can only know the push was sent, not that the client received it. The correct architecture is: commit Kafka offset only after the message_id is written to Cassandra AND the recipient's offline queue — these are durable. The WebSocket push is best-effort on top of this durable base. If the push fails, the recipient reconnects and pulls from their offline queue or Cassandra history. At-least-once delivery with client-side deduplication by message_id, not at-most-once.

## Deep Dive 2: Offline Message Delivery

**Full offline delivery flow:**
```
Step 1: Sender sends message to Chat Server A
Step 2: Chat Server A checks recipient presence:
  Redis GET presence:user456 → null (TTL expired = offline)

Step 3: Store durably + notify:
  a. Write message to Cassandra (primary durability)
  b. LPUSH offline_queue:user456:* message_id  (offline queue, one per device)
  c. Publish to Kafka "push-notifications" topic

Step 4: Notification Service consumes from Kafka:
  Fetch message content from Cassandra
  Call APNS (iOS): POST https://api.push.apple.com/3/device/{device_token}
  Body: {
    aps: {
      alert: { title: "Alice", body: "Hello!" },
      badge: 3,           -- unread count (fetched from user's unread counter)
      sound: "default"
    },
    message_id: "timeuuid-abc"  -- for client to deduplicate on receive
  }

Step 5: User's device wakes, notification received
Step 6: App opens, user authenticates, WebSocket connects to any Chat Server (C)
Step 7: Chat Server C (new server, user's session migrated):
  a. Check offline queue: LRANGE offline_queue:user456:device1 0 -1
  b. Fetch messages from Cassandra by message_id list
  c. Deliver in order to client WebSocket
  d. Client sends ACK for each message received
  e. Server marks messages as delivered, sends receipts to original senders
  f. Clear offline queue: DEL offline_queue:user456:device1
```

**Push notification design:**
```
APNS vs FCM payload design:
  Don't send full message content in push payload (privacy + 4KB limit)
  Instead: { "new_messages": true, "conversation_id": "conv789" }
  App wakes, connects to Chat Server, pulls messages itself

  Exception: "data messages" (silent push) — wake app in background to sync
  without showing a notification UI (used for low-priority sync)

FCM delivery guarantees:
  "Normal priority": may delay indefinitely if device in Doze mode
  "High priority": wakes device immediately (limited quota per app/day)
  WhatsApp uses high priority for message notifications

Offline queue max size:
  If user is offline for 30 days: queue may have thousands of messages
  Strategy: store only the last 100 message_ids in the queue
  On app open: pull the last 90 days from Cassandra by conversation (full sync)
  → Queue is just a hint that new messages exist, not a complete buffer
```

## Deep Dive 3: Group Message Fan-out

**Group messaging challenge:**
```
500-person group → 1 message → 499 delivery operations

Naive approach: sender's Chat Server handles all 499 deliveries
  → 499 Redis presence lookups → 499 conditional sends
  → Chat Server A is blocked for 499 × 0.3ms = ~150ms just for one group message
  → Blocks all 200K of Chat Server A's other connections

Correct approach: delegate to specialized fan-out service
```

**Fan-out service architecture:**
```
Step 1: Sender's Chat Server A:
  a. Write message to Cassandra
  b. Publish to Kafka "group-messages" topic (key = group_id for ordering)
  c. Return ACK to sender immediately

Step 2: Fan-out workers (partitioned by group_id, N workers):
  Consume from Kafka "group-messages":
  a. Load group membership from Group Service
     (cached in Redis: SET group_members:{group_id} → member_id list, TTL=5min)
  b. For each member (excluding sender):
     - Check presence: Redis GET presence:{member_id}
     - If online: publish to "chat-delivery" Kafka topic with target server_id
     - If offline: LPUSH offline_queue:{member_id} message_id
                   enqueue push notification
  c. Batch deliveries: use Kafka batch producer for efficiency
     (group 100 member deliveries into one Kafka batch)

Step 3: Chat Servers consume "chat-delivery" for their connected users:
  Push to WebSocket (same as 1:1 delivery)

Fan-out latency for 500-member group:
  Kafka publish: 5ms
  Fan-out worker: 500 presence lookups (Redis pipeline) + 500 Kafka messages = ~50ms
  End-to-end for online members: 5ms + 50ms + WebSocket push = ~100ms (within SLA)
```

**Large group pull model:**
```
For very large groups (WhatsApp allows 1024, Slack channels have thousands):
  Fan-out at 1024 members × 700K messages/sec would require 718M Redis ops/sec → untenable

Switch to pull model above 500 members:
  Group messages written to Cassandra only (no fan-out to individual queues)
  Members' apps periodically poll the group conversation (every 30s when app is open)
  → "Group timeline" model: members fetch new messages since last cursor

Hybrid:
  < 200 members: push fan-out (real-time delivery)
  200–500 members: push fan-out but rate-limited (max 1 fan-out per 100ms per group)
  > 500 members: pull model with push notification as hint
```

**G5:** The group fan-out service has a correctness issue with member list caching. If user X is removed from a group, the cached group membership in Redis (TTL=5min) may still include X for up to 5 minutes. During that window, X receives messages they shouldn't. Conversely, if user Y is added, they miss messages for up to 5 minutes. The fix is not to shorten the TTL (that increases Redis pressure) — it's to publish membership change events to Kafka and have fan-out workers invalidate the specific group's cache immediately on receiving the event (`DEL group_members:{group_id}`). The next fan-out for that group reloads fresh membership from the Group Service. This gives eventual consistency bounded by Kafka end-to-end latency (~50ms) rather than cache TTL (5min).

## Reliability and Operations

- **Chat server failure:** Client has exponential backoff + jitter reconnect logic. WebSocket connection drops → client reconnects to any available Chat Server (load balancer picks one). New Chat Server registers the user in Session Store (Redis SET presence:{user_id}). In-flight messages that were pushed to the failed server's WebSocket but not ACK'd by client: client re-requests last N messages from Cassandra on reconnect (pull-on-reconnect ensures no gaps).
- **Cassandra failure:** Write fails → client retries with same client_msg_id (idempotent by message_id). At-least-once delivery. Cassandra is multi-datacenter; coordinator routes to available DC on failure.
- **Duplicate detection:** Client assigns a `client_msg_id` (UUID) to each message. Server deduplicates within a 5-minute window: `SETNX dedup:{sender_id}:{client_msg_id} 1 EX 300`. If key already exists, message is a duplicate — return ACK with existing server message_id, don't re-write.
- **Connection draining for server restarts:** Send each connected client a `{type: "reconnect", new_server: "chat-server-23"}` frame. Client reconnects to specified server (or load balancer if address not provided). New connections are refused by the draining server after sending reconnect signals. Allow 30s for graceful drain; hard-stop after 30s.

## Security and Abuse

- End-to-end encryption: Signal Protocol (Double Ratchet + X3DH key exchange). Keys stored only on devices — server stores encrypted ciphertext only, cannot read messages
- Authentication: OAuth2 access token (short-lived, 1h) + refresh token. Device-specific session tokens — each device has a unique credential
- Spam detection: rate limit new accounts (max 10 new conversations/day for accounts < 7 days old). ML classifier on message velocity (sending 1000 messages in 1 minute = bot behavior)
- Block/report: blocked user's messages are silently dropped at the fan-out routing layer (no delivery, no error to sender — prevents confirmation of block)

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| WebSocket | Long-polling | WebSocket: true bidirectional real-time (< 100ms delivery); Long-polling: higher latency (1–30s depending on poll interval), more HTTP overhead, simpler to implement through proxies. WebSocket is standard for modern chat apps |
| Cassandra | MySQL | Cassandra: partitions by conversation_id → natural horizontal scaling; time-clustered reads are efficient. MySQL: harder to shard conversation data, no native time-series clustering, but richer query support |
| Kafka fan-out | Direct server-to-server RPC | Kafka: durable buffer survives fan-out service restarts; natural load leveling. Direct RPC: lower latency (no Kafka hop) but tight coupling; fan-out service restart = messages lost |
| Pull model for large groups | Push fan-out always | Fan-out at 1000+ members × high message rate creates unsustainable Redis/Kafka load. Pull trades delivery latency (poll interval) for write scalability — correct tradeoff for large groups |

## G5

**What a G5 engineer adds beyond the above:**

1. **Connection draining and hot server migration:** When a chat server is overloaded (CPU > 80% sustained) or scheduled for maintenance, it must shed connections gracefully without dropping messages. Send a directed `{type: "reconnect"}` WebSocket frame to 10% of connections per second (spreading the reconnect storm). The load balancer excludes the draining server from new connections immediately upon receiving a health-check failure signal. During the 30-second drain window, the server continues processing messages for still-connected clients. Client SDK uses exponential backoff with jitter (1–10s) to spread reconnect attempts — without jitter, all 200K clients of a failed server reconnect simultaneously, creating a thundering herd on the load balancer.

2. **Multi-device message synchronization:** A user with phone + laptop + web browser must see the same messages on all devices. Session Store maps `user_id → [list of {server_id, device_id}]`. Fan-out delivers to all active sessions simultaneously. Offline queue is per-device, not per-user — each device tracks its own delivery cursor. Read receipts (blue checkmarks) sync across devices via a dedicated read-sync event: marking a conversation as read on the phone sends `{type: "read_sync", conversation_id, up_to_message_id}` to all other active sessions for the same user. This prevents the annoyance of blue checkmarks only appearing on the device that opened the conversation.

3. **Message search (server-side vs client-side):** Full-text search across message history. WhatsApp's approach for E2E encrypted chats: client-side search only (device downloads all messages, SQLite FTS5 index on device). For non-E2E (Slack-style enterprise): Elasticsearch index per workspace, updated asynchronously via Kafka consumer. Index only message text + sender/conversation metadata; store message_id for retrieval. Shard Elasticsearch index by conversation_id to keep related messages co-located. Re-index delay: 5–30 seconds (Kafka lag). For compliance/legal hold: a separate "content inspection" pipeline can access unencrypted messages before E2E keys are applied — this is the architectural point where E2E and compliance requirements conflict.

4. **Geo-distributed architecture with regional chat servers:** Users in EU connect to EU chat servers (< 20ms intra-region RTT vs 120ms transatlantic). 1:1 messages where sender and receiver are in different regions: message travels sender's region → backbone → receiver's region. Backbone routing via private WAN (AWS Global Accelerator or dedicated peering), not public internet — reduces variance from ~50ms to ~20ms. The session store (who is on which server) must be globally accessible: either a globally replicated Redis (with 50ms sync) or a per-region session store with a global routing table. For cross-region delivery, the sender's region acts as coordinator, looks up receiver's region from the global table, and forwards via backbone — one extra hop but keeps regional autonomy.

5. **Presence system at scale:** 100M DAU with 40M concurrent connections means 40M presence updates every 15 seconds = 2.67M Redis writes/sec just for heartbeats. This saturates a single Redis cluster. Solutions: (a) Only update presence if status changed (exponential backoff for heartbeat if no status change → reduces writes by 80%). (b) Presence is eventually consistent: store last_heartbeat_ms, compute online status as `last_heartbeat_ms > now - 30s`. Accept that "last seen" granularity is 15s, not 1s — users cannot tell the difference. (c) Presence pub/sub: instead of polling presence for each message delivery, subscribe to presence change events. Fan-out service subscribes to `presence_changes` Kafka topic; only updates routing when a user transitions between online/offline, not on every heartbeat.
