# 20. Parking Garage System

**Type:** commerce
**Difficulty:** G3–G4
**Frequency:** Medium-High — OOP + system design, often asked at onsite interviews

---

## Problem Statement

Design a parking garage management system. The system must track available spots, handle vehicles entering and exiting, issue tickets, calculate fees, and process payments. Design both the object model and the distributed backend for a network of 100 garages with 500 spots each.

---

## Functional Requirements

**Must have:**
- Vehicle enters: find and assign an available spot; issue parking ticket
- Vehicle exits: scan ticket, calculate fee, process payment, release spot
- Support multiple vehicle types (motorcycle, compact, large, handicapped)
- Display available spots per floor and type at entrance
- Real-time spot availability tracking

**Should have:**
- Online reservation: reserve a spot in advance
- Monthly pass / subscription support
- License plate recognition (LPR) for entry/exit without physical ticket
- Analytics: occupancy rates, revenue per garage

**Out of scope:**
- Navigation inside the garage (turn-by-turn to spot)
- EV charging management (mentioned in G5)
- Valet parking

---

## Non-Functional Requirements

- **Scale:** 100 garages × 500 spots = 50,000 spots total
- **Availability:** 99.9% — garage gate must work even if central service is down
- **Latency:** Gate opens in < 1 second from ticket scan
- **Consistency:** Strong for spot assignment — can't double-assign a spot
- **Reliability:** Edge-first design: gate controller works offline if central server unreachable

---

## Capacity Estimation

```
Throughput:
  100 garages × 500 spots × 2 events (enter + exit) × avg 4 turns/day = 400K events/day
  400K / 86400 = 4.6 events/sec average — extremely low throughput
  Peak (morning rush at downtown garage): 500 entries/hour = 8/sec per garage
  → This is NOT a scale challenge; it's a design clarity and correctness challenge

Data volumes:
  Spot state:   50K spots × 200B per record = 10MB → trivially fits in Redis
  Tickets:      400K/day × 500B = 200MB/day → 70GB/year in PostgreSQL → no issue
  Revenue:      avg $10 × 400K = $4M/day → financial accuracy is critical
  Analytics:    400K events/day × 200B = 80MB/day raw events → ClickHouse for OLAP
  Long-term:    5 years × 70GB/year = 350GB tickets → easily managed PostgreSQL

Gate controller constraints:
  Must operate offline: local SQLite state, sync on reconnect
  Resilience: gate hardware uptime > 99.9% → most failures are software/network
  Each gate: Raspberry Pi or embedded x86 with 4GB RAM, 64GB eMMC storage
```

## Object Model

```python
from enum import Enum
from decimal import Decimal, ROUND_UP
from datetime import datetime
from typing import Optional, List

class VehicleType(Enum):
    MOTORCYCLE = "motorcycle"
    COMPACT    = "compact"
    LARGE      = "large"
    HANDICAPPED = "handicapped"   # requires handicapped-accessible spot

class SpotType(Enum):
    SMALL       = "small"        # fits motorcycle only
    COMPACT     = "compact"      # fits compact cars + motorcycles
    LARGE       = "large"        # fits all standard vehicles
    HANDICAPPED = "handicapped"  # accessible, fits all vehicle types

# Compatibility matrix: which vehicle types fit in which spot types
SPOT_COMPATIBILITY = {
    SpotType.SMALL:       {VehicleType.MOTORCYCLE},
    SpotType.COMPACT:     {VehicleType.MOTORCYCLE, VehicleType.COMPACT},
    SpotType.LARGE:       {VehicleType.MOTORCYCLE, VehicleType.COMPACT, VehicleType.LARGE},
    SpotType.HANDICAPPED: {VehicleType.MOTORCYCLE, VehicleType.COMPACT,
                           VehicleType.LARGE,      VehicleType.HANDICAPPED},
}

class Vehicle:
    def __init__(self, license_plate: str, vehicle_type: VehicleType):
        self.license_plate = license_plate
        self.vehicle_type = vehicle_type

class ParkingSpot:
    def __init__(self, spot_id: str, garage_id: str, floor: int,
                 spot_number: int, spot_type: SpotType):
        self.spot_id = spot_id       # e.g., "G01-F02-S023"
        self.garage_id = garage_id
        self.floor = floor
        self.spot_number = spot_number
        self.spot_type = spot_type
        self.is_occupied: bool = False
        self.vehicle: Optional[Vehicle] = None

    def can_fit(self, vehicle_type: VehicleType) -> bool:
        return vehicle_type in SPOT_COMPATIBILITY[self.spot_type]

class ParkingTicket:
    def __init__(self, ticket_id: str, license_plate: str, spot_id: str, entry_time: datetime):
        self.ticket_id = ticket_id
        self.license_plate = license_plate
        self.spot_id = spot_id
        self.entry_time = entry_time
        self.exit_time: Optional[datetime] = None
        self.fee: Optional[Decimal] = None
        self.status: str = 'active'  # 'active' | 'paid' | 'lost' | 'voided'

class PricingPolicy:
    def __init__(self,
                 base_rate_per_hour: Decimal,
                 day_maximum: Decimal,
                 motorcycle_discount_rate: float = 0.5,   # 50% of standard rate
                 evening_rate: Optional[Decimal] = None,  # rate after 6pm
                 lost_ticket_fee: Decimal = Decimal("50.00"),
                 monthly_pass_price: Decimal = Decimal("150.00")):
        self.base_rate_per_hour = base_rate_per_hour
        self.day_maximum = day_maximum
        self.motorcycle_discount_rate = motorcycle_discount_rate
        self.evening_rate = evening_rate
        self.lost_ticket_fee = lost_ticket_fee
        self.monthly_pass_price = monthly_pass_price

class Floor:
    def __init__(self, floor_number: int, spots: List[ParkingSpot]):
        self.floor_number = floor_number
        self.spots = spots

    def available_spots(self, vehicle_type: VehicleType) -> List[ParkingSpot]:
        return [s for s in self.spots if not s.is_occupied and s.can_fit(vehicle_type)]

class ParkingGarage:
    def __init__(self, garage_id: str, name: str, address: str,
                 floors: List[Floor], pricing: PricingPolicy):
        self.garage_id = garage_id
        self.name = name
        self.address = address
        self.floors = floors
        self.pricing = pricing

    def get_available_spots(self, vehicle_type: VehicleType) -> List[ParkingSpot]:
        available = []
        for floor in self.floors:
            available.extend(floor.available_spots(vehicle_type))
        return available

    def assign_spot(self, vehicle: Vehicle) -> Optional[ParkingSpot]:
        """Assign smallest compatible available spot (efficiency: save large spots for large vehicles)."""
        # Priority order: prefer exact-fit spot type first, then larger
        type_priority = {
            VehicleType.MOTORCYCLE: [SpotType.SMALL, SpotType.COMPACT, SpotType.LARGE],
            VehicleType.COMPACT:    [SpotType.COMPACT, SpotType.LARGE],
            VehicleType.LARGE:      [SpotType.LARGE],
            VehicleType.HANDICAPPED: [SpotType.HANDICAPPED, SpotType.LARGE],
        }
        for spot_type in type_priority[vehicle.vehicle_type]:
            for floor in self.floors:
                for spot in floor.spots:
                    if not spot.is_occupied and spot.spot_type == spot_type:
                        spot.is_occupied = True
                        spot.vehicle = vehicle
                        return spot
        return None  # no compatible spot available

    def release_spot(self, spot_id: str) -> bool:
        for floor in self.floors:
            for spot in floor.spots:
                if spot.spot_id == spot_id:
                    spot.is_occupied = False
                    spot.vehicle = None
                    return True
        return False  # spot_id not found

    def available_count(self, spot_type: Optional[SpotType] = None) -> int:
        count = 0
        for floor in self.floors:
            for spot in floor.spots:
                if not spot.is_occupied:
                    if spot_type is None or spot.spot_type == spot_type:
                        count += 1
        return count
```

## High-Level Architecture

```
[Entrance Gate Controller]             [Exit Gate Controller]
  │ (edge device: Raspberry Pi)         │ (edge device: Raspberry Pi)
  │ - Local SQLite for spot state        │ - Reads ticket, computes fee
  │ - Works offline up to 4 hours        │ - Processes payment locally if offline
  │ - Syncs via MQTT on reconnect        │
  └───────────────┬──────────────────────┘
                  │ MQTT / HTTPS (TLS)
                  ▼
         [Garage API Server]
           ├──→ [PostgreSQL (tickets, revenue, billing)]
           ├──→ [Redis (real-time spot state, available counts)]
           └──→ [Kafka (gate events → analytics pipeline)]

         [Central Backend] ──→ [Management Dashboard (React)]
           ├──→ [Spot Management Service]
           ├──→ [Pricing Engine]
           ├──→ [Payment Service (Stripe)]
           └──→ [Analytics (ClickHouse)]
```

**Edge-first design rationale:** Gate hardware must open the barrier in < 1 second. Central server round-trip over a flaky 4G connection might add 200–500ms; server downtime would block vehicles. Each gate controller maintains a local SQLite copy of spot state and pricing rules. On entry: local assignment + local ticket write + open gate, all < 200ms. Sync with central server asynchronously. If offline for up to 4 hours: continue operating locally. On reconnect: push local events to central server for reconciliation.

## Deep Dive 1: Spot Assignment

**Spot assignment with closest-spot preference:**
```python
def assign_spot(garage_id: str, vehicle_type: str, preferred_floor: int = 1) -> Optional[str]:
    """
    Assigns the closest available compatible spot.
    Returns spot_id or None if garage is full for this vehicle type.
    """
    # Compatibility priority: assign smallest sufficient spot type
    spot_types_by_vehicle = {
        "motorcycle":   ["small", "compact", "large"],
        "compact":      ["compact", "large"],
        "large":        ["large"],
        "handicapped":  ["handicapped", "large"],
    }

    for spot_type in spot_types_by_vehicle[vehicle_type]:
        # Try preferred floor first, then expand
        for floor in [preferred_floor] + [f for f in range(1, 6) if f != preferred_floor]:
            key = f"available_spots:{garage_id}:{spot_type}:floor:{floor}"

            # ZPOPMIN: atomically removes and returns member with lowest score (= shortest distance)
            result = redis.zpopmin(key, count=1)
            if result:
                spot_id = result[0][0]  # (member, score) tuple

                # Atomically mark spot as occupied
                redis.hset(f"spot:{spot_id}", mapping={
                    "status": "occupied",
                    "assigned_at": int(time.time()),
                    "vehicle_type": vehicle_type
                })

                # Decrement available count (for entrance display)
                redis.decr(f"avail:{garage_id}:{spot_type}")

                return spot_id

    return None  # No compatible spot available

def release_spot(spot_id: str, garage_id: str, spot_type: str, floor: int):
    """Returns a spot to the available pool on exit."""
    # Get spot's distance_to_elevator from spots table (cached in Redis or DB lookup)
    distance = get_distance_to_elevator(spot_id)

    # Return to sorted set
    redis.zadd(f"available_spots:{garage_id}:{spot_type}:floor:{floor}", {spot_id: distance})

    # Clear spot state
    redis.hset(f"spot:{spot_id}", "status", "available")
    redis.hdel(f"spot:{spot_id}", "vehicle_type", "assigned_at")

    # Increment count
    redis.incr(f"avail:{garage_id}:{spot_type}")
```

**Why `ZPOPMIN` is the right Redis primitive:**
```
ZADD available_spots:G01:compact:floor:1 {distance} {spot_id}
  → Sorted by distance_to_elevator (lower = closer = better)

ZPOPMIN available_spots:G01:compact:floor:1 1
  → Atomically removes and returns the closest available compact spot on floor 1
  → O(log N) time complexity
  → Atomic: no race condition between two simultaneous entries both grabbing same spot

Alternative (less optimal):
  SPOP (random removal from Set): O(1) but ignores distance preference
  LINDEX + LREM (list head): O(1) but list must be maintained in sorted order

For garages without distance preference (e.g., small lots): SPOP is sufficient and simpler.
For large garages where walkers appreciate shortest path: ZPOPMIN with distance score.
```

**Garage initialization (on service start):**
```python
def initialize_garage_redis(garage_id: str):
    """Load all active spots into Redis sorted sets. Run once on startup or after Redis flush."""
    spots = db.query("""
        SELECT id, floor, spot_type, distance_to_elevator
        FROM spots
        WHERE garage_id = %s AND is_active = TRUE
          AND id NOT IN (  -- exclude currently occupied spots
              SELECT spot_id FROM tickets
              WHERE garage_id = %s AND status = 'active'
          )
    """, garage_id, garage_id)

    pipe = redis.pipeline()
    for spot in spots:
        key = f"available_spots:{garage_id}:{spot.spot_type}:floor:{spot.floor}"
        pipe.zadd(key, {spot.id: spot.distance_to_elevator or 0})

    # Initialize count keys
    for spot_type in ['small', 'compact', 'large', 'handicapped']:
        count = sum(1 for s in spots if s.spot_type == spot_type)
        pipe.set(f"avail:{garage_id}:{spot_type}", count)

    pipe.execute()
```

**G5:** Would challenge whether the Redis ZPOPMIN approach correctly handles the edge case where Redis and the database diverge. If the gate controller crashes between ZPOPMIN (spot removed from Redis set) and the DB ticket insert, the spot is "lost" — it's not in the Redis available set (so no one can be assigned to it) and not in the DB as occupied (so no ticket exists). On service restart, the `initialize_garage_redis` function re-queries the DB to rebuild Redis state, but this function itself has a race condition: it reads "not in active tickets" but a ticket insert could be in-flight. The correct fix is a two-phase write: (1) Insert a provisional ticket record in the DB first (status='assigning'), (2) then ZPOPMIN from Redis. On failure between steps 1 and 2: a background job clears provisional tickets older than 60 seconds and puts their spots back in the Redis sets. This ensures the DB is the source of truth, and Redis is a derived view that is always re-buildable from the DB.

## Deep Dive 2: Fee Calculation

```python
def calculate_fee(ticket: ParkingTicket, pricing: PricingPolicy, exit_time: datetime) -> Decimal:
    """
    Calculate parking fee with:
    - Per-hour billing (first partial hour charged as full hour)
    - Daily maximum cap
    - Vehicle type discount
    - Lost ticket policy
    - Multi-day stay handling
    """
    if ticket.entry_time >= exit_time:
        return Decimal("0.00")  # zero-duration stay (within grace period)

    # Grace period: first 15 minutes free
    GRACE_MINUTES = 15
    adjusted_entry = ticket.entry_time + timedelta(minutes=GRACE_MINUTES)
    if exit_time <= adjusted_entry:
        return Decimal("0.00")

    duration = exit_time - adjusted_entry
    total_seconds = duration.total_seconds()
    raw_hours = Decimal(total_seconds) / Decimal(3600)

    # Billing rule: round up to nearest hour
    billable_hours = raw_hours.quantize(Decimal('1'), rounding=ROUND_UP)

    # Multi-day calculation: apply daily maximum cap per 24-hour period
    # This prevents runaway fees for forgotten vehicles
    fee = Decimal("0.00")
    remaining_hours = billable_hours
    current_time = adjusted_entry

    while remaining_hours > 0:
        # Determine hours in this day (up to midnight or remaining, whichever is smaller)
        midnight = (current_time + timedelta(days=1)).replace(
            hour=0, minute=0, second=0, microsecond=0
        )
        hours_remaining_in_day = Decimal(
            (midnight - current_time).total_seconds() / 3600
        ).quantize(Decimal('1'), rounding=ROUND_UP)

        hours_this_period = min(remaining_hours, hours_remaining_in_day)

        # Evening rate: lower rate after 6pm (if configured)
        if pricing.evening_rate and current_time.hour >= 18:
            period_rate = pricing.evening_rate
        else:
            period_rate = pricing.base_rate_per_hour

        period_fee = hours_this_period * period_rate

        # Cap at daily maximum
        daily_fee = min(period_fee, pricing.day_maximum)
        fee += daily_fee

        remaining_hours -= hours_this_period
        current_time = midnight

    # Vehicle type discount
    if ticket.vehicle_type == VehicleType.MOTORCYCLE.value:
        fee = fee * Decimal(str(pricing.motorcycle_discount_rate))

    # Lost ticket: charge maximum(computed_fee, lost_ticket_fee)
    if ticket.status == 'lost':
        fee = max(fee, pricing.lost_ticket_fee)

    return fee.quantize(Decimal("0.01"))  # round to cents

# Example calculation:
# Entry: 8:00am Monday, Exit: 6:30pm Tuesday (34.5 hours)
# Rate: $5/hour, day_max: $30, vehicle: compact
#
# Day 1 (Monday 8am → midnight = 16 hours):
#   fee = min(16 × $5, $30) = min($80, $30) = $30.00
# Day 2 (midnight → Tuesday 6:30pm = 18.5h → 19 billable hours):
#   fee = min(19 × $5, $30) = min($95, $30) = $30.00
# Total: $30 + $30 = $60.00
```

**G5:** Would note that naive per-hour billing with a daily cap has a subtle edge case for overnight parkers: a vehicle that enters at 11:55pm and exits at 12:05am is charged for 2 "hours" (1 partial hour before midnight + 1 partial hour after), but the "day" has reset, so the daily maximum applies per partial day rather than per 24-hour rolling window. The legal definition of "day" in parking fee calculation varies by jurisdiction — some require calendar day (midnight-to-midnight), others allow 24-hour rolling window. The rolling window favors the garage (higher revenue for overnight stays). The calendar day favors parkers but creates perverse incentives (enter at 11:58pm, pay for 2 partial days at daily max). The fee calculation implementation should be configurable per garage based on local regulations, with the `day_boundary` being a parameter rather than hardcoded midnight.

## Reliability and Operations

- **Gate controller offline (network failure):** Local SQLite on gate controller. On entry: look up spot state from local SQLite → assign spot → write ticket to local SQLite → open gate. Local operation < 200ms. When network restores: push all locally-created tickets to central server via HTTPS POST. Server reconciles: if ticket_id doesn't exist centrally, create it; if it already exists (duplicate sync), ignore. Conflict resolution: DB ticket is authoritative over local SQLite on any disagreement.
- **Payment processor down:** If Stripe API is unavailable: log the payment attempt in a local retry queue with ticket status = 'payment_pending'. Barrier opens regardless (vehicle cannot be held hostage). Night batch reconciliation: retry all payment_pending tickets. If payment still fails after 24h: send invoice to vehicle owner via email (using license_plate → user lookup). Monthly summary for fleet accounts.
- **Spot count discrepancy:** Run nightly reconciliation script:
  ```sql
  SELECT s.id, s.spot_type,
    redis_occupied.count AS redis_says_occupied,
    db_active.count      AS db_says_active
  FROM spots s
  JOIN LATERAL (
    SELECT COUNT(*) FROM tickets WHERE spot_id = s.id AND status = 'active'
  ) db_active ON TRUE
  -- compare with Redis spot state
  WHERE redis_occupied.count != db_active.count
  ```
  Alert if discrepancy > 2 spots per garage. Automatic recovery: rebuild Redis from DB state.
- **LPR camera failure:** Fall back to manual ticket (physical QR/barcode). Gate defaults to "open on scan" mode with a 10-minute exit window after fee payment. No vehicle is trapped by LPR failure.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis ZPOPMIN for spot assignment | DB `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1` | Redis: atomic O(log N), sub-millisecond; DB: durable immediately but ~5ms, requires a lock, connection overhead at high concurrency |
| Edge-first offline mode | Centralized only (gate waits for server) | Edge: resilient to network/server outages (99.9% availability achievable); Centralized: simpler consistency model, but single outage blocks all gates |
| Sorted set for closest-spot preference | Random SPOP | Sorted: better UX (shorter walk), same atomicity; Random: simpler, O(1), acceptable when garage has uniform layout |
| SQLite on gate controller | Stateless gate (always requires server) | SQLite: offline capability, fast local operations; Stateless: simpler sync, but any network hiccup blocks entry |
| PostgreSQL for tickets | DynamoDB | PostgreSQL: rich queries (revenue reports, complex aggregations), ACID transactions; DynamoDB: better global scale, but overkill for 50K spots and no complex queries needed |

## G5

1. **License plate recognition (LPR) as primary access method:** Replace physical ticket with automatic gate: camera reads license plate → lookup or create ticket. Hardware: industrial IP cameras (Avigilon/Genetec) with embedded LPR GPU module. Software: YOLO-based object detection → license plate region crop → CNN OCR model → character string. End-to-end: < 500ms from vehicle stopping to gate opening. Confidence threshold: if OCR confidence < 85% → fall back to manual ticket issuance (human operator button). Privacy: license plate images are PII in California (CCPA) and EU (GDPR). Retention policy: raw images deleted after 30 days; text records (license plate + timestamp) retained per jurisdiction requirements (often 90 days). Monthly pass vehicles: LPR matches plate against `monthly_passes` table → auto-authorize without any payment interaction. This eliminates the gate transaction from ~30 seconds (manual ticket) to ~3 seconds (LPR) and removes the cost of physical ticket hardware.

2. **Dynamic pricing and revenue optimization:** Implement occupancy-based surge pricing to maximize revenue and distribute demand. Pricing tiers per occupancy bracket: 0-70% full → standard rate; 70-85% → 1.2× rate ("High Demand" shown); 85-95% → 1.5× rate; 95%+ → 2× rate ("Almost Full"). Display real-time pricing on the entrance sign and the mobile app. Regulatory note: some cities (San Francisco, Chicago) regulate parking price changes — check local ordinances. Predictive pricing: ML model trained on 2 years of occupancy data by hour/day/season + event calendar (sports, concerts, conventions) → pre-announce prices on the app 24 hours ahead. A driver seeing "tomorrow's game day rate: $45" can decide whether to drive or take transit, smoothing demand. Revenue impact: dynamic pricing increases revenue 15–25% vs flat-rate pricing without increasing capacity.

3. **EV charging spot management:** EV spots have embedded chargers (Level 2: 7.2kW, or DC Fast Charge: 50–150kW). Spot assignment logic: if vehicle is an EV (detected via LPR + DMV lookup or user app profile), prefer EV-enabled spots. If no EV spots available: assign regular spot (EV can still park, just can't charge). Charging billing: separate from parking fee — billed per kWh or per hour of charging session, not per parking hour. Integration: charger communicates via OCPP (Open Charge Point Protocol) with the central Charging Management System. Charger session events (start, energy delivered, end) → Kafka → billing service. Prevent ICE vehicles from occupying EV spots ("ICE-ing"): LPR at EV spots checks registration type via DMV API; non-EV vehicle in EV spot → alert attendant + email/SMS to vehicle owner after 15 minutes.

4. **Multi-site aggregation and real-time routing:** From a driver's perspective, the question is "which garage near my destination has availability AND affordable pricing?" Aggregation API: `GET /v1/search?lat=37.77&lng=-122.41&radius_km=1&vehicle_type=compact`. Each garage in the network pushes availability updates via MQTT to a central broker on every spot change. Central broker maintains a Redis GEO set: `GEOADD garages:available {lng} {lat} {garage_id}`. Search query: `GEORADIUS garages:available -122.41 37.77 1 km ASC` → up to 20 nearby garages → fetch `avail:{garage_id}` counts and `current_rate:{garage_id}` from Redis. Combine into a ranked response (sort by: walking distance + current rate + predicted availability in 30 min). The predictive availability component: a lightweight ML model using current occupancy trajectory + time of day + historical patterns forecasts whether a garage will be full by the time the driver arrives (typically 5–15 minutes away).

5. **Analytics and business intelligence for operators:** Garage operators want rich operational insights to optimize staffing, pricing, and maintenance schedules. Data pipeline: gate events → Kafka `gate-events` topic → Flink 10-minute tumbling windows → ClickHouse OLAP. Key metrics computed: (a) Occupancy heatmap: % full by hour-of-day × day-of-week × garage → identify peak/trough patterns. (b) Revenue per spot per day: total revenue / active spots / days. (c) Average dwell time by vehicle type, entry time bucket. (d) Turnover rate: how many different vehicles occupied each spot per day. (e) Peak demand forecasting: next 7 days occupancy prediction by garage. Anomaly detection: if occupancy drops > 30% vs same day/hour last week → alert (broken gate? local road closure? competing new garage?). Compare revenue-per-spot across garages in the same city to identify underperformers (candidates for dynamic pricing or layout reconfiguration). Dashboard: Grafana or custom React app. All ClickHouse queries run < 100ms on 5 years of event history (ClickHouse handles 400K events/day × 365 days × 5 years = 730M rows trivially).
