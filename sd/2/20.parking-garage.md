# 20. Parking Garage System

**Type:** commerce
**Difficulty:** L3–L4
**Frequency:** Medium-High — OOP + system design, often asked at onsite interviews

---

## Problem Statement

Design a parking garage management system. The system must track available spots, handle vehicles entering and exiting, issue tickets, calculate fees, and process payments. Design both the object model and the distributed backend for a network of 100 garages with 500 spots each.

---

## Functional Requirements

**Must have:**
- Vehicle enters: find and assign an available spot; issue parking ticket
- Vehicle exits: scan ticket, calculate fee, process payment, release spot
- Support multiple vehicle types (motorcycle, compact, large, handicapped)
- Display available spots per floor and type at entrance
- Real-time spot availability tracking

**Should have:**
- Online reservation: reserve a spot in advance
- Monthly pass / subscription support
- License plate recognition (LPR) for entry/exit without physical ticket
- Analytics: occupancy rates, revenue per garage

**Out of scope:**
- Navigation inside the garage (turn-by-turn to spot)
- EV charging management
- Valet parking

---

## Non-Functional Requirements

- **Scale:** 100 garages × 500 spots = 50,000 spots total
- **Availability:** 99.9% — garage gate must work even if central service is down
- **Latency:** Gate opens in < 1 second from ticket scan
- **Consistency:** Strong for spot assignment — can't double-assign a spot
- **Reliability:** Edge-first design: gate controller works offline if central server is unreachable

---

## Capacity Estimation

```
Events: 100 garages × 500 spots × 2 (enter+exit) × 4 turns/day = 400K events/day = 5/sec
Very low throughput — this is not a scale problem, it's a design clarity problem

Data storage:
  50K spots × 200B = 10MB for spot state (trivially fits in Redis)
  Tickets: 400K/day × 500B = 200MB/day; 1 year = 70GB (fits in PostgreSQL)
  Revenue: $10 avg × 400K events = $4M/day (financial accuracy critical)
```

## Object Model

```python
# Core entities

class VehicleType(Enum):
    MOTORCYCLE = "motorcycle"
    COMPACT = "compact"
    LARGE = "large"
    HANDICAPPED = "handicapped"

class SpotType(Enum):
    SMALL = "small"      # motorcycle fits
    COMPACT = "compact"  # compact + motorcycle
    LARGE = "large"      # all vehicles
    HANDICAPPED = "handicapped"  # accessible

class Vehicle:
    license_plate: str
    vehicle_type: VehicleType

class ParkingSpot:
    spot_id: str        # e.g., "A-1-023" (garage A, floor 1, spot 23)
    garage_id: str
    floor: int
    spot_number: int
    spot_type: SpotType
    is_occupied: bool
    vehicle: Optional[Vehicle]

    def can_fit(self, vehicle_type: VehicleType) -> bool:
        fits = {
            SpotType.SMALL: {VehicleType.MOTORCYCLE},
            SpotType.COMPACT: {VehicleType.MOTORCYCLE, VehicleType.COMPACT},
            SpotType.LARGE: {VehicleType.MOTORCYCLE, VehicleType.COMPACT, VehicleType.LARGE},
            SpotType.HANDICAPPED: {VehicleType.MOTORCYCLE, VehicleType.COMPACT, VehicleType.LARGE, VehicleType.HANDICAPPED},
        }
        return vehicle_type in fits[self.spot_type]

class ParkingTicket:
    ticket_id: str
    license_plate: str
    spot_id: str
    entry_time: datetime
    exit_time: Optional[datetime]
    fee: Optional[Decimal]
    status: str  # 'active', 'paid', 'lost'

class ParkingGarage:
    garage_id: str
    name: str
    address: str
    floors: List[Floor]
    pricing: PricingPolicy

    def get_available_spots(self, vehicle_type: VehicleType) -> List[ParkingSpot]
    def assign_spot(self, vehicle: Vehicle) -> Optional[ParkingSpot]
    def release_spot(self, spot_id: str) -> None
    def available_count(self, spot_type: Optional[SpotType] = None) -> int

class PricingPolicy:
    base_rate_per_hour: Decimal
    day_maximum: Decimal           # never charge more than this per day
    motorcycle_discount: float     # e.g., 0.5 = 50% of standard rate
    evening_rate: Decimal          # discounted rate after 6pm
    lost_ticket_fee: Decimal
    monthly_pass_price: Decimal
```

## High-Level Architecture

```
[Entrance Gate Controller]          [Exit Gate Controller]
  │ (edge device, works offline)      │ (edge device)
  │                                   │
  └───────────────┬───────────────────┘
                  │ MQTT / HTTP
                  ▼
         [Garage API Server] ──→ [PostgreSQL (tickets, revenue)]
                  │          ──→ [Redis (real-time spot state)]
                  │
         [Central Backend]
                  │
      ┌───────────┼───────────┐
[Spot Mgmt]  [Pricing Eng]  [Payment Svc]
                  │
         [Management Dashboard]
         [Analytics (ClickHouse)]
```

**Edge-first design:** Gate controller has a local copy of spot state (SQLite). Can operate autonomously for 4 hours if central server is unreachable. Syncs with central server when reconnected.

## Deep Dive 1: Spot Assignment

**Spot assignment algorithm:**
```python
def assign_spot(garage_id, vehicle_type) -> Optional[str]:
    # Get compatible spot types (smallest first for efficiency)
    compatible = get_compatible_spot_types(vehicle_type)

    for spot_type in compatible:  # motorcycle: [small, compact, large]
        # Get available spot from Redis
        spot_id = redis.spop(f"available_spots:{garage_id}:{spot_type}")
        if spot_id:
            # Atomically update spot state
            redis.set(f"spot:{spot_id}", json.dumps({
                "status": "occupied",
                "ticket_id": ticket_id
            }))
            redis.decr(f"avail:{garage_id}:{spot_type}")
            return spot_id

    return None  # garage full for this vehicle type
```

**Redis SADD/SPOP for O(1) spot assignment:**
```
On garage initialization:
  SADD available_spots:G01:compact  G01-F01-S001 G01-F01-S002 ... G01-F01-S100

On vehicle entry:
  SPOP available_spots:G01:compact → returns one spot atomically

On vehicle exit:
  SADD available_spots:G01:compact G01-F01-S023 (returns the spot to pool)
```

This is O(1) and atomic — no race conditions, no double-assignment.

**Closest spot preference:**
- Randomized SPOP doesn't pick the closest spot
- Enhancement: use Redis sorted set (score = distance from elevator)
  `ZPOPMIN available_spots:G01:compact` → always returns closest available
- Or: assign spots floor by floor (fill floor 1 before floor 2)
  Multiple sets: `available_spots:G01:compact:floor:1`, etc.

## Deep Dive 2: Fee Calculation

```python
def calculate_fee(ticket: ParkingTicket, pricing: PricingPolicy, exit_time: datetime) -> Decimal:
    duration = exit_time - ticket.entry_time
    hours = Decimal(duration.total_seconds() / 3600)

    # First partial hour is charged as full hour
    billable_hours = hours.quantize(Decimal('1'), rounding=ROUND_UP)

    fee = Decimal(0)
    entry = ticket.entry_time

    # Day-by-day calculation (to apply day maximum correctly)
    remaining_hours = billable_hours
    current_time = entry

    while remaining_hours > 0:
        hours_today = min(remaining_hours, 24)
        daily_fee = min(hours_today * pricing.base_rate_per_hour, pricing.day_maximum)
        fee += daily_fee
        remaining_hours -= hours_today

    # Apply vehicle type discount
    if ticket.vehicle_type == VehicleType.MOTORCYCLE:
        fee *= pricing.motorcycle_discount

    # Lost ticket fee
    if ticket.status == 'lost':
        fee = max(fee, pricing.lost_ticket_fee)

    return fee.quantize(Decimal('0.01'))
```

## Reliability and Operations

- **Gate controller offline:** Local SQLite on gate controller. On entry: write locally, issue ticket, open gate. Sync with central server when online. Conflict resolution: server is authoritative; local state is eventually consistent.
- **Payment system down:** If payment processor is down, log the attempt with a queue. Gate still opens (payment deferred). Night batch reconciliation of deferred payments.
- **Spot count discrepancy:** Run nightly reconciliation: count physically occupied spots (via sensor feed or LPR camera scan) vs system state. Alert on mismatch > 2 spots per garage.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis SPOP for spot assignment | DB SELECT + UPDATE | Redis: atomic O(1); DB: durable but slower, needs locking |
| Edge-first offline mode | Centralized only | Edge: resilient to network; Centralized: simpler, consistent |
| Sorted set for closest spot | Random SPOP | Sorted: better UX; Random: simpler, still functional |

## L5 Additions

1. **License plate recognition (LPR) as primary access method:** Cameras at entrance and exit. ML model (ResNet-based) recognizes license plate in < 500ms. On entry: look up if vehicle has a monthly pass or reservation. On exit: match to active ticket. Fallback: manual ticket if LPR fails (camera dirty, obscured plate). LPR reduces average gate time from 30s (manual ticket) to 5s (automatic). Privacy concern: license plate data is PII in some jurisdictions — apply retention policy, encrypt at rest.

2. **Dynamic pricing and revenue optimization:** Implement surge pricing based on real-time occupancy. At 90% full: increase rate 1.5x. At 95%: 2x. This reduces demand and optimizes revenue. Display "high demand" on entrance display. Integration with city-wide parking availability APIs (some cities mandate sharing occupancy data). Predictive pricing: ML model on historical patterns → pre-announce prices for events (concert night = higher rates starting 4pm).

3. **EV charging spot management:** Growing segment. EV spots have associated charger (L2 or DC fast). Spot assignment: if vehicle is EV, prefer EV-enabled spots. If no EV spots: assign regular spot (they can still park). Charger booking: separate booking flow (charge time, expected kWh, billing per kWh vs per hour). Prevent ICE vehicles from occupying EV spots (LPR checks vehicle registration type via DMV API).

4. **Multi-site aggregation and routing:** From a driver's perspective: find any available garage near their destination. Aggregation API: `GET /v1/search?lat=37.77&lng=-122.41&vehicle_type=compact&max_walk_minutes=10`. Response: list of garages within walking distance, sorted by price/availability combination. Real-time: each garage pushes availability updates to central broker (MQTT). Central broker aggregates → serves search. Add reservation from the aggregated search view.

5. **Analytics and business intelligence:** Garage operators want: occupancy heatmaps by hour/day, revenue breakdown by vehicle type, average dwell time, peak hours by day of week. Pipeline: gate events → Kafka → Flink (10-minute windows) → ClickHouse (OLAP). Dashboard: Grafana or custom. Anomaly detection: alert if occupancy drops 30% vs same day last week (broken gate? bad weather?). Benchmark: compare revenue per spot across garages to identify underperformers.
