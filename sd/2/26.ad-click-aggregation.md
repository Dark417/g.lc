# 26. Ad Click Aggregation

**Type:** data
**Difficulty:** L4–G5
**Frequency:** High — stream processing, approximate counting, latency-accuracy tradeoff

---

## Problem Statement

Design an ad click aggregation system. Advertisers want to know how many times their ads were clicked, aggregated by ad ID, campaign, time window, and user demographics. The system must handle 1M click events/sec, provide near-realtime results (< 1 minute lag), support queries on historical aggregates, and accurately attribute clicks without double-counting.

---

## Functional Requirements

**Must have:**
- Ingest click events: ad_id, user_id, timestamp, geo, device_type
- Aggregate click counts by: ad_id + time window (1min, 5min, 1hr, 1day)
- Query: "how many clicks did ad X get in the last hour?"
- Deduplication: same user clicking same ad within 10 minutes = 1 click

**Should have:**
- Real-time streaming aggregation (< 60s lag)
- Fraud detection: remove bot/fraudulent clicks
- Breakdown by: geo, device type, time-of-day
- Reconciliation: ensure streaming and batch results match

**Out of scope:**
- Impression tracking (similar pattern, separate system)
- Attribution modeling (last-click vs multi-touch)
- Billing based on clicks (uses this data as input)

---

## Non-Functional Requirements

- **Scale:** 1M clicks/sec = 86B clicks/day
- **Latency:** Near-realtime results within 60 seconds; historical queries < 1 second
- **Accuracy:** < 0.1% error on deduplication; billing-grade accuracy for daily totals
- **Availability:** 99.99% — lost click data = lost revenue
- **Query performance:** Ad dashboard query < 1s for hourly/daily aggregates

```
After deduplication (assume 5% duplicate clicks from double-taps, retries):
  81.7B unique clicks/day

Aggregate storage (ClickHouse):
  10M distinct ads × 1440 minutes/day = 14.4B aggregate rows/day (if storing per-minute)
  Per row: ~100B → 1.44TB/day → too large; instead aggregate to 5-min buckets
  10M ads × 288 five-min-buckets/day = 2.88B rows/day × 100B = 288GB/day
  Compressed (columnar): ~30GB/day → very manageable

Raw event archive (S3, Parquet):
  17TB/day × 4x Parquet compression = ~4.3TB/day
  7-day retention: 30TB → S3 at $0.023/GB = $690/month
  30-day billing retention: 130TB → $3,000/month (columnar, compressed)

Redis for deduplication:
  At 1M clicks/sec, 1M unique (ad_id, user_id) pairs in 10-minute window
  Per-entry bloom filter approach:
    FPP = 0.001 (0.1% false positive rate)
    m = -n × ln(p) / ln(2)² = 1M × 6.93 / 0.48 = 14.4M bits = 1.8MB per ad per window
  But we have 10M active ads → 10M × 1.8MB = 18TB → impractical
  Better: partition bloom filters by ad popularity tier (hot ads: exact Redis; cold ads: bloom)
```

```sql
CREATE TABLE click_aggregates (
  ad_id           UInt64,
  campaign_id     UInt64,
  time_bucket     DateTime,       -- truncated to window size (5-min buckets for storage)
```

```
  Key: dedup:{ad_id}:{user_id_hash}:{10min_window}
  Type: STRING with TTL = 600s (10 minutes)
  SETNX: returns 1 (new click, count it) or 0 (duplicate, skip)

  Key: bloom:{ad_id}:{10min_window_id}

  FPP = 0.1% (acceptable; over-dedup by 0.1%, under-dedup never)
```

## Deep Dive 1: Deduplication at Scale

**Requirement:** Same user clicking same ad within 10 minutes = 1 billable click. At 1M clicks/sec, naive Redis SET per (ad_id, user_id) is too memory-intensive.

**Bloom filter math:**
```
Window: 10 minutes
Unique (ad_id, user_id) pairs in 10-min window:
  Assume 100K active ads; avg 100 unique users clicking per ad per 10min
  → 100K × 100 = 10M unique pairs per 10-min window

Bloom filter sizing (per ad, per 10-min window):
  n = 100 unique users per ad (much smaller per-ad bloom filter)
  p = 0.001 (0.1% FPP)
  m = -n × ln(p) / ln(2)² = -100 × (-6.9) / 0.48 = 1,440 bits = 180 bytes per bloom filter
  k = ceil(m/n × ln(2)) = ceil(14.4 × 0.693) = 10 hash functions

Total memory: 100K active ads × 180B × 6 windows (10-min buckets in 1 hour) = 108MB → trivial!
Much better than 10M Redis keys × 100B/key = 1GB per 10-min window
```

**Implementation (Redis BITFIELD bloom filter):**
```python
import hashlib
import struct

def bloom_check_and_set(redis, ad_id: str, user_id_hash: str, window_id: int) -> bool:
    """
    Returns True if this is a NEW click (should be counted).
    Returns False if this is a DUPLICATE (should be skipped).
    FPP: ~0.1% false positives (duplicates that are actually new clicks).
    """
    key = f"bloom:{ad_id}:{window_id}"
    m = 1440  # bloom filter bit size
    k = 10    # number of hash functions

    # Generate k bit positions using double-hashing (fast, good distribution)
    h1 = int(hashlib.md5(f"{user_id_hash}:1".encode()).hexdigest(), 16)
    h2 = int(hashlib.md5(f"{user_id_hash}:2".encode()).hexdigest(), 16)
    bit_positions = [(h1 + i * h2) % m for i in range(k)]

    pipe = redis.pipeline()
    # Check all k bits
    for pos in bit_positions:
        pipe.getbit(key, pos)
    bits = pipe.execute()

    if all(bits):  # all bits set → likely a duplicate
        return False

    # New click: set all k bits
    pipe = redis.pipeline()
    for pos in bit_positions:
        pipe.setbit(key, pos, 1)
    pipe.expire(key, 660)  # 10 min + 60s grace
    pipe.execute()
    return True  # count this click
```

**For billing-grade accuracy (batch layer):**
```
Streaming bloom filter gives ~99.9% accuracy — good for real-time dashboards.
For billing, need exact deduplication:
  Spark batch job reads raw Parquet from S3 (7-day retention)
  Exact dedup: GROUP BY ad_id, user_id_hash, floor(timestamp_ms / 600000)
    → Count distinct (ad_id, user_id_hash) pairs per 10-min window
  This is exact (no false positives); runs hourly on yesterday's data
  Output: daily_aggregates table in ClickHouse (authoritative billing numbers)
```

**G5:** Bloom filters give one-sided errors: false positives (declare "duplicate" when it's a new click) reduce billable click counts, directly reducing advertiser spend and platform revenue. The 0.1% FPP means 0.1% of legitimate clicks are silently discarded in the streaming layer. Over 86B daily clicks, that's 86M clicks/day incorrectly dropped — at $0.01/click average CPC, that's $860K/day in undercounting. The streaming layer's undercounting is tolerable because the batch layer corrects it for billing, but the batch correction requires the raw Parquet archive to be complete (no Kafka message loss). The hidden dependency is that the Flink S3 sink must use exactly-once semantics (Flink's S3 sink with checkpointing and epoch-based filenames) — otherwise a Flink restart re-writes duplicate records to S3, causing the batch layer to overcount. The bloom filter FPP and the S3 sink durability guarantee are tightly coupled correctness requirements that must both hold.

## Deep Dive 2: Flink Streaming Aggregation

**Flink job for multi-window click aggregation:**
```java
DataStream<ClickEvent> clicks = env
    .addSource(new KafkaSource<ClickEvent>()
        .setTopics("click-events")
        .setGroupId("flink-click-aggregator")
        .setStartingOffsets(OffsetsInitializer.committedOffsets()))
    .assignTimestampsAndWatermarks(

DataStream<ClickEvent> deduped = clicks
    .keyBy(click -> click.getAdId() + ":" + click.getUserIdHash())

// 1-minute tumbling windows, keyed by ad_id
deduped
    .keyBy(ClickEvent::getAdId)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .allowedLateness(Time.minutes(2))
    .sideOutputLateData(lateClickTag)
    .aggregate(new ClickCountAggregator(), new ClickWindowResultFunction())
    .addSink(new ClickHouseSink("click_aggregates"));

// 5-minute and 1-hour windows (same pipeline, different window sizes)
// ...

// Side output: late events → write to separate S3 path for batch correction
env.fromSideOutput(deduped, lateClickTag)
    .addSink(new S3Sink("s3://ad-clicks/late/..."));
```

**Watermarking for event-time windowing:**
```
Problem: mobile devices are often offline; click events arrive minutes later
Solution: watermark = max(server_timestamp_seen) - 2 minutes
  → Window fires when watermark passes window end time
  → Waits up to 2 minutes for late-arriving events
  → Events > 2 minutes late → side output (batch handles them)

Choosing watermark delay:
  Too low (0): many late events → side output → batch must handle → streaming accuracy poor
  Too high (10 min): windows fire very late → streaming latency is 10 min (violates SLA)
  2 minutes: empirically covers 99% of mobile late events; 1-min window fires at t+3min
```

**Flink keyed state for deduplication:**
```java
public class ClickDeduplicator extends KeyedProcessFunction<String, ClickEvent, ClickEvent> {
    // State: set of (ad_id, user_id_hash) pairs seen in last 10 minutes
    private MapState<String, Long> seenClicks;  // key=hash, value=first_seen_timestamp

    @Override
    public void processElement(ClickEvent event, Context ctx, Collector<ClickEvent> out)
            throws Exception {
        String key = event.getAdId() + ":" + event.getUserIdHash();
        Long firstSeen = seenClicks.get(key);

        if (firstSeen == null || event.getTimestampMs() - firstSeen > 600_000) {
            // New click or outside 10-min dedup window
            seenClicks.put(key, event.getTimestampMs());
            out.collect(event);
            // Schedule state cleanup after 10 min (avoid state unbounded growth)
            ctx.timerService().registerEventTimeTimer(event.getTimestampMs() + 600_000);
        }
        // else: duplicate within 10-min window → drop silently
    }

    @Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<ClickEvent> out) {
        // Clean up expired state entries to prevent memory leak
        // In practice: iterate seenClicks and remove entries older than 10 min
    }
}
```

**G5:** Flink's tumbling event-time windows have a correctness issue specific to the ad click domain: the watermark is derived from the maximum observed event timestamp across ALL ad_ids in the partition. A single cold partition (an ad with very low click volume, maybe 1 click per 5 minutes) can delay the watermark for all ads in that partition, holding all windows open and preventing them from firing. In Kafka-keyed-by-ad_id partitioning, popular ads and dormant ads land on the same partition. Solution: use per-key idleness detection — `WatermarkStrategy.withIdleness(Duration.ofMinutes(5))`. Flink marks a partition's input as idle after 5 minutes of no new events and advances the global watermark based on remaining non-idle inputs. Without this, a stream with 10K active ads and 10K dormant ads will have its windows perpetually blocked by the 10K dormant ones, defeating the entire event-time model.

## Fraud Detection

**Real-time signals (applied in Flink before aggregation):**
```java
// Flink fraud filter: runs before aggregation
clicks
    .keyBy(ClickEvent::getIpHash)
    .window(SlidingEventTimeWindows.of(Time.minutes(1), Time.seconds(10)))
    .aggregate(new ClickRateAggregator())
    .filter(rate -> {
        // Rule 1: > 20 clicks from same IP in 1 minute → bot
        if (rate.getCount() > 20) return true;  // flag as fraud
        // Rule 2: click < 50ms after page load (physically impossible for human)
        if (rate.getMinInterClickMs() < 50) return true;
        return false;
    })
    .getSideOutput(fraudClickTag);

// Other fraud rules:
// - Same user clicks same ad > 5 times in 1 hour
// - Known bot user agents (googlebots, scrapers, headless Chrome)
// - Clicks from datacenter IP ranges (AWS, GCP, Azure IP space → not real users)
```

**Batch fraud scoring (higher accuracy):**
```
Daily Spark ML job:
  Features per (user_id_hash, day):
    - clicks_per_hour: unusually high → fraud
    - unique_ads_clicked: very high → suspicious
    - click_to_conversion_rate: near 0 → click farm
    - ip_reputation_score: from threat intelligence feed
    - device_fingerprint_consistency: same device, different user IDs → shared device fraud

  Model: gradient boosted classifier (LightGBM)
  Output: fraud_probability score per (user, day)
  Threshold: 0.8 → mark user's clicks as fraud for that day
  Retroactively mark S3 events: UPDATE Parquet → rerun batch aggregation
```

## Reliability and Operations

- **Kafka consumer lag:** Flink consumer group offset committed after each checkpoint (every 60s). On Flink restart: resume from last checkpoint offset; reprocess last 60s of events. Dedup logic prevents double-counting from reprocessed events.
- **ClickHouse insert failure:** Flink buffers inserts in local state, retries with exponential backoff. ClickHouse append-only + SummingMergeTree: safe to retry (duplicate inserts for same time_bucket + ad_id are summed — idempotent if written atomically per batch).
- **S3 archive completeness:** Every raw Kafka event written to S3 via Flink side output (before aggregation, not after). S3 is the source of truth for batch reprocessing. Alert: if Flink S3 sink lag > 5 minutes → batch reprocessing will be incomplete.
- **Reconciliation SLA:** Daily batch job at 6 AM UTC processes previous day's data. By 8 AM UTC, billing numbers are finalized. Discrepancy > 0.1% triggers alert and manual review. Common causes: Flink restart reprocessed events without proper checkpointing, Kafka message loss (hardware failure on broker without sufficient ISR).

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Lambda architecture (Flink + Spark) | Kappa (streaming only with reprocessing) | Lambda: separate fast+approximate vs slow+exact layers; Kappa: simpler single code path but reprocessing requires retaining all Kafka events for weeks and re-running the same job (only one answer, but delayed) |
| ClickHouse SummingMergeTree | Cassandra for aggregates | ClickHouse: OLAP vectorized execution, excellent for multi-dimensional range queries; Cassandra: better write throughput but slower analytical queries without pre-aggregation |
| Bloom filter dedup | Exact Redis SET per (ad_id, user_id) | Bloom: constant per-ad memory (180B vs 100B per user); Exact: no false positives but memory scales linearly with unique users |
| Tumbling event-time windows | Processing-time windows | Event-time: correct despite network delays, auditable; Processing-time: simpler but incorrect for delayed mobile clicks |
| Kafka keyed by ad_id | Keyed by user_id | ad_id keying: all clicks for one ad go to same partition (better window state locality); user_id: better dedup locality but fragments ad-level windows across partitions |

## G5

1. **Exactly-once delivery in streaming pipeline — the full stack:** Flink + Kafka + ClickHouse exactly-once requires three interlocking guarantees: (a) Flink checkpointing: Flink periodically snapshots its entire operator state + Kafka consumer offsets to S3. On failure, restore from checkpoint and replay only from the checkpointed Kafka offset. (b) Exactly-once ClickHouse sink: ClickHouse doesn't support 2PC, but Flink can write to ClickHouse using epoch-numbered batch IDs — if the same checkpoint epoch is replayed, ClickHouse uses `ReplacingMergeTree` to replace the old partial result with the identical new one. (c) S3 archive with atomic file commits: Flink writes Parquet files to a staging path, then atomically renames on checkpoint completion — preventing partial files from being visible to Spark batch jobs. Without all three, a single Flink restart causes either data loss (missed events) or double-counting (duplicate aggregates) in billing numbers.

2. **Multi-dimensional aggregation without combinatorial explosion:** Advertisers want arbitrary dimension slices: "clicks by US mobile users between 6–9 PM." Naive: pre-compute all dimension combinations → 2^N rows per ad per time window (N dimensions). With N=5 dimensions (geo, device, hour, publisher, creative), that's 32× storage. Better approach: store fine-grained atomic dimensions in ClickHouse (`per-geo + per-device + per-hour` as separate rows, not cross-joined) and let ClickHouse's vectorized aggregation engine compute cross-dimensional queries at query time. ClickHouse can aggregate 1B rows/sec per core — a dashboard query touching 10M rows runs in < 100ms. For known high-traffic queries: use ClickHouse Materialized Views (auto-updated on insert) to pre-compute the top 20 most-queried dimension combinations only.

3. **Lambda vs Kappa architecture — the real tradeoff:** Lambda architecture's speed and batch layers produce two different answers for the same time range during the reconciliation window: streaming shows 80M clicks, batch will eventually show 82M. Advertisers who query during this window see provisional numbers that will later change — creating trust and support issues. Kappa architecture (stream-only with reprocessing) avoids this by having a single source of truth: if the streaming numbers are wrong, reprocess from Kafka (same code, new job instance with different output topic). The trade-off is not correctness but cost: reprocessing Kafka requires retaining all raw events for the entire reprocessing window (days to weeks), at 17TB/day = 119TB for 7 days. The Kappa approach is cleaner but requires very long Kafka retention or S3-backed replay, making it expensive at this volume. The choice is operational complexity (Lambda) vs storage cost (Kappa).

4. **Geographic distribution and data residency:** Click events from EU users must be processed and stored in EU (GDPR Article 44 — no cross-border transfer without adequacy decision). Architecture: regional Kafka clusters (US, EU, APAC), each processing its own region's clicks. Regional Flink + ClickHouse clusters per region. For cross-regional campaign reports (an advertiser wants global click totals): aggregate anonymized counts (no user_id) at the global analytics tier — only aggregates cross borders, never individual user-linked events. Routing: ad server tags each click event with the user's regulatory jurisdiction; Kafka topic routing ensures EU clicks never enter a US Kafka partition. This requires geographic data residency enforcement at the Kafka producer level, not just at rest.

5. **Real-time bidding feedback loop:** Click and CTR data directly feeds back into real-time bidding decisions. If CTR for an ad placement is trending up → bid higher. If trending down → reduce spend. System: Flink computes 1-minute rolling CTR (clicks / impressions) per (ad_id, placement_id) → writes to Redis pub/sub → SSE push to advertiser dashboard and bidding API. Latency requirement: CTR updates available within 30 seconds of click. Architecture: `ZADD ctr:{ad_id} {ctr_value} {timestamp}` → bidding algorithm pulls latest CTR via `ZRANGEBYSCORE`. The feedback loop creates a system dynamics challenge: if CTR measurement has high variance (small sample size), bidding adjustments amplify noise → oscillations in bid prices. Smooth CTR using exponential moving average: `CTR_t = α × CTR_current + (1 - α) × CTR_t-1`, with α = 0.1 for stability.
