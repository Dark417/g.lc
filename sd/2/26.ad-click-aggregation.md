# 26. Ad Click Aggregation

**Type:** data
**Difficulty:** L4–L5
**Frequency:** High — stream processing, approximate counting, latency-accuracy tradeoff

---

## Problem Statement

Design an ad click aggregation system. Advertisers want to know how many times their ads were clicked, aggregated by ad ID, campaign, time window, and user demographics. The system must handle 1M click events/sec, provide near-realtime results (< 1 minute lag), support queries on historical aggregates, and accurately attribute clicks without double-counting.

---

## Functional Requirements

**Must have:**
- Ingest click events: ad_id, user_id, timestamp, geo, device_type
- Aggregate click counts by: ad_id + time window (1min, 5min, 1hr, 1day)
- Query: "how many clicks did ad X get in the last hour?"
- Deduplication: same user clicking same ad within 10 minutes = 1 click

**Should have:**
- Real-time streaming aggregation (< 60s lag)
- Fraud detection: remove bot/fraudulent clicks
- Breakdown by: geo, device type, time-of-day
- Reconciliation: ensure streaming and batch results match

**Out of scope:**
- Impression tracking (similar pattern, separate system)
- Attribution modeling (last-click vs multi-touch)
- Billing based on clicks (uses this data as input)

---

## Non-Functional Requirements

- **Scale:** 1M clicks/sec = 86B clicks/day
- **Latency:** Near-realtime results within 60 seconds; historical queries < 1 second
- **Accuracy:** < 0.1% error on deduplication; billing-grade accuracy for daily totals
- **Availability:** 99.99% — lost click data = lost revenue
- **Query performance:** Ad dashboard query < 1s for hourly/daily aggregates

---

## Capacity Estimation

```
Clicks: 1M/sec = 86B/day
Event size: 200B (ad_id, user_id, timestamp, geo, device) → 200MB/sec = 17TB/day raw

After dedup (assume 5% duplicates): 81.7B unique clicks/day

Aggregates per ad (hourly): avg 1000 ads × 24 hours = 24K aggregate rows/day
  Per row: 200B → 24K × 200B = trivial

Storage for raw events (7-day retention): 17TB/day × 7 = 119TB
  Use Parquet columnar format (4x compression): ~30TB
  On S3: cheap ($0.023/GB)
```

## High-Level Architecture

```
[Ad Server] ──→ [Kafka: click-events] ──→ [Flink Streaming Job]
                                               │
                                    ┌──────────┼──────────┐
                               [1min agg]  [5min agg]  [1hr agg]
                                    │
                              [ClickHouse / Redis]
                                    │
                              [Query API] ──→ [Ad Dashboard]

[Spark Batch Job] (hourly/daily, billing-grade accuracy)
  ← reads from S3 raw click archive
  ← writes to ClickHouse daily_aggregates
```

**Lambda Architecture:**
- **Speed layer (Flink):** Real-time aggregation, < 1 min lag, acceptable approximation
- **Batch layer (Spark):** Exact computation on full data, runs hourly/daily
- **Serving layer (ClickHouse):** Stores both; serves both

## Data Model

**ClickHouse: Aggregated click counts (OLAP, columnar)**
```sql
CREATE TABLE click_aggregates (
  ad_id         UInt64,
  campaign_id   UInt64,
  time_bucket   DateTime,     -- truncated to window size
  window        String,       -- '1min', '5min', '1hour', '1day'
  geo           String,       -- 'US', 'UK', ...
  device_type   String,       -- 'mobile', 'desktop', 'tablet'
  click_count   UInt64,
  unique_users  UInt64
) ENGINE = SummingMergeTree()
  PARTITION BY toYYYYMMDD(time_bucket)
  ORDER BY (window, ad_id, time_bucket, geo, device_type);

-- SummingMergeTree: automatically sums click_count on merge
-- Very efficient for pre-aggregated data from streaming jobs
```

**Kafka: Raw click events (ephemeral)**
```
Topic: click-events (16 partitions, keyed by ad_id for ordering)
Retention: 7 days
```

**S3: Raw click archive (permanent)**
```
s3://ad-clicks/{date}/{hour}/clicks-{partition}-{offset}.parquet
Schema: { ad_id, user_id, campaign_id, timestamp, geo, device_type, is_fraud }
```

**Redis: Real-time deduplication**
```
Key: dedup:{ad_id}:{user_id}
Type: SET with TTL = 10 minutes
→ SADD dedup:ad123:user456 {click_id}: returns 1 (new) or 0 (duplicate)
Or: bloom filter per (ad_id, hourly_bucket) for high-scale dedup
```

## Deep Dive 1: Deduplication at Scale

**Requirement:** Same user clicking same ad within 10 minutes = 1 billable click.

**Naive approach:** SET dedup:{ad_id}:{user_id} with 10-min TTL
- At 1M clicks/sec: 1M Redis SETEX/sec → too many keys; Redis memory pressure

**Counting Bloom Filter approach:**
```
Time window: 10 minutes
Bloom filter per (ad_id, 10-minute-window):
  4 bits per slot (counting bloom filter for deletion)
  False positive rate: 0.1% (acceptable — 0.1% legitimate clicks counted as duplicates)

At 1M unique (ad_id, user_id) pairs per 10-min window:
  m = -n * ln(p) / ln(2)^2 = 1M * 6.93 / 0.48 = 14.4M bits per ad per window → manageable

Implementation: Redis BITFIELD for per-ad-per-window bloom filters
  Key: bloom:{ad_id}:{window_id}
  On click: check bloom → if bit set → duplicate; if not → set bit + count click
  On window expiry: delete bloom filter key
```

**For billing-grade accuracy:** Batch reprocessing
- Raw events in S3 → Spark job → exact dedup using GROUP BY (ad_id, user_id, hour)
- This is the "batch layer" — takes longer (1 hour) but is exact
- Streaming is approximate; batch is billing-grade

## Deep Dive 2: Flink Streaming Aggregation

**Flink job for 1-minute windowed counts:**
```java
DataStream<ClickEvent> clicks = env
    .addSource(new KafkaSource<>(...));

clicks
    .keyBy(click -> click.adId)  // partition stream by ad_id
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))  // 1-min tumbling windows
    .aggregate(new ClickCountAggregator(), new ClickWindowFunction())
    .addSink(new ClickHouseSink());

// Handle late data (clicks arriving up to 2 min late due to mobile network delays)
.allowedLateness(Time.minutes(2))
.sideOutputLateData(lateOutputTag);  // route late data separately
```

**Watermarking for event time:**
```
Problem: mobile devices may have delayed click events (offline, then sync)
Solution: watermark = max_observed_event_time - 2_minutes
  → Window fires when watermark passes window_end
  → Waits up to 2 minutes for late events
  → Events later than 2 minutes go to side output (batch processing)
```

**Flink state for sliding windows:**
```
Sliding window (last 5 minutes, updated every minute):
  KeyedState: per ad_id, maintain a circular buffer of 5 1-minute counts
  On each 1-minute trigger: update buffer, sum last 5 = 5-minute count
  Emit: {ad_id, 5min_count, window_end}
```

## Fraud Detection

**Real-time signals:**
```
Bot detection rules (applied in Flink before aggregation):
1. Click rate per IP > 10/min → flag as bot
2. Same user clicks same ad > 5 times in 1 hour → flag as click fraud
3. User agent is known bot/scraper → always exclude
4. Time between impression and click < 50ms → physically impossible → exclude

Flink rule:
  clicks.keyBy(ip).window(SlidingWindow.of(1min, 10sec)).count()
  → if count > threshold: emit to fraud stream (excluded from counts)
```

**Batch fraud scoring (Spark ML):**
- User behavior features: click patterns, conversion rate, IP reputation, device fingerprint
- Model scores each user/IP per day → fraud probability
- Low-confidence real-time rules + high-confidence batch model = best accuracy

---

## Reliability and Operations

- **Kafka consumer lag:** Flink consumers track Kafka offset. On Flink restart, resume from last committed offset. May reprocess events within 30s window → dedup handles this.
- **ClickHouse insert failure:** Buffer inserts in Flink state, retry with exponential backoff. ClickHouse is append-only, SummingMergeTree deduplicates on merge.
- **S3 archive:** Every Kafka message written to S3 via Flink side output (before aggregation). S3 is the authoritative source for batch reprocessing.
- **Reconciliation:** Daily batch job compares streaming totals with batch totals. Alert on discrepancy > 0.5%.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Lambda architecture | Kappa (streaming only) | Lambda: batch accuracy + streaming speed; Kappa: simpler but harder billing accuracy |
| ClickHouse | Cassandra for aggregates | ClickHouse: OLAP query performance; Cassandra: better write throughput |
| Bloom filter dedup | Redis SET dedup | Bloom: memory efficient; Redis SET: exact but memory intensive |
| Tumbling windows | Session windows | Tumbling: simple, predictable; Session: user-centric but complex |

## L5 Additions

1. **Exactly-once delivery in streaming pipeline:** Flink + Kafka + ClickHouse exactly-once requires: (a) Flink checkpointing: Flink periodically snapshots its state + Kafka offsets to S3. On failure: restore from checkpoint, re-process from snapshot offset (at-least-once). (b) Exactly-once sink: ClickHouse doesn't support 2-phase commit, but SummingMergeTree with idempotent inserts (using ReplacingMergeTree for dedup by event_id) provides effectively-once semantics. (c) Kafka exactly-once producer: Flink uses Kafka transactional API to write to Kafka exactly-once if needed for downstream.

2. **Multi-dimensional aggregation:** Advertisers want slices: "clicks by US mobile users between 6-9 PM." Naive: pre-compute all combinations → 2^N combinatorial explosion. Better: store fine-grained aggregates (per-geo, per-device, per-hour) and let ClickHouse compute cross-dimensional queries at query time. ClickHouse's vectorized execution handles these queries in < 1s for typical dashboards. For known high-value queries: materialize with Materialized Views in ClickHouse (auto-updated on insert). Pre-compute only the top 20 most-queried dimension combinations.

3. **Attribution tracking (last-click vs multi-touch):** Simple click counting is not enough for sophisticated advertisers. They want: "User saw Ad A, then Ad B, then bought — how do we attribute the conversion?" Last-click: attribute to Ad B (most common). Multi-touch: split attribution across Ad A and Ad B. Architecture: track the full user click journey (session-level event log in Cassandra). Attribution model runs as batch Spark job nightly: reads click journeys, applies model, writes attribution credits to ad system. Multi-touch requires joining click events with conversion events.

4. **Geographic distribution and data residency:** Click events from EU users must be processed and stored in EU (GDPR). Architecture: regional Kafka clusters (US, EU, APAC). Each region processes its own clicks. EU regional ClickHouse + Flink cluster. Global rollup: anonymized, aggregated data (no user_id) flows to global analytics cluster for cross-regional campaign reports. Routing: ad server tag click with user region → Kafka topic routing ensures EU clicks go to EU Kafka only.

5. **Real-time bidding feedback loop:** Advertisers use click data to adjust their bidding strategy in real-time. If CTR (click-through rate) for an ad is trending up → bid higher. System: streaming CTR computation → advertiser real-time API endpoint (WebSocket or SSE push) → bidding algorithm adjusts. Latency requirement: CTR updates within 30 seconds. Architecture: Flink computes 1-minute rolling CTR → writes to Redis (per ad_id) → SSE push to advertiser dashboard. This is the intelligence layer on top of raw click counting.
