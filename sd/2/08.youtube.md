# 08. YouTube — Video Streaming Platform

**Type:** social-media
**Difficulty:** L4–L5
**Frequency:** Very High — video upload, transcoding pipeline, adaptive streaming

---

## Problem Statement

Design a video streaming platform like YouTube or Netflix. Users can upload videos, and other users can stream them. The system must handle 500 hours of video uploaded per minute, 1B hours of video watched daily, support adaptive bitrate streaming (different quality for different connection speeds), and provide sub-2-second startup time.

---

## Functional Requirements

**Must have:**
- Upload video (up to 5GB file)
- Watch video with adaptive bitrate streaming
- Search videos by title/keyword
- Video metadata: title, description, view count, likes

**Should have:**
- Comments and likes
- Recommendations / related videos
- Video chapters / timestamps
- Subtitles / captions

**Out of scope:**
- Live streaming (separate design)
- Content ID / copyright matching
- Ads serving

---

## Non-Functional Requirements

- **Scale:** 500 hours uploaded/min; 1B hours watched/day
- **Latency:** Video start < 2 seconds; buffering < 0.1% of playback time
- **Availability:** 99.99% — downtime = lost revenue
- **Durability:** No video loss (videos stored permanently by default)
- **Bandwidth:** 1B hours/day × 1Mbps avg = ~125TB/s — must use CDN

---

## Capacity Estimation

```
Uploads: 500 hours/min = 30K minutes/min = 500 files/min = ~8 files/sec
  Average upload: 1 hour video × 500MB = 8 × 500MB/sec = 4GB/sec ingest

Storage per video (all resolutions):
  1hr video: 360p(200MB) + 480p(400MB) + 720p(1GB) + 1080p(2GB) + 4K(8GB) = ~12GB
  Daily storage: 500hr/min × 60min × 12GB = ~360TB/day

Watch: 1B hours/day = 42M hours/hour = 700K concurrent streams
  Avg bitrate 2Mbps → 700K × 2Mbps = 1.4Tbps bandwidth
  → CDN must serve this; origin servers serve only cache misses (< 5%)
```

## Deep Dive 1: Video Upload and Transcoding Pipeline

**Upload flow:**
```
1. Client requests upload URL from API → gets S3 presigned URL + video_id
2. Client uploads directly to S3 (multipart for large files; resume on failure)
   - S3 multipart: split into 100MB chunks, upload in parallel, complete when all done
   - Client reports progress; on resume: list already-uploaded parts, skip them
3. S3 triggers event notification → Kafka topic "video-uploaded"
4. Transcoding orchestrator picks up event:
   - Creates transcoding jobs for each target format (360p, 480p, 720p, 1080p, 4K, WebM/VP9, HLS/H264)
   - Submits jobs to transcoding worker pool
5. Workers (GPU-equipped for fast transcoding):
   - Download raw video from S3
   - Transcode using FFmpeg to target resolution/codec
   - Upload segments to S3 in HLS format: video_id/1080p/seg001.ts, seg002.ts, ...
   - Generate manifest file: video_id/manifest.m3u8 (references all resolutions)
6. On all jobs complete: update video status to 'ready', invalidate CDN cache for manifest

Parallel transcoding:
  - 8 resolutions × 2 codecs = 16 jobs
  - Submit all 16 in parallel (different workers)
  - Each worker transcodes one resolution → total time ≈ time for longest single job
```

**HLS (HTTP Live Streaming) format:**
```
master.m3u8:
  #EXTM3U
  #EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=640x360
  360p/index.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=3000000,RESOLUTION=1280x720
  720p/index.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=8000000,RESOLUTION=1920x1080
  1080p/index.m3u8

720p/index.m3u8:
  #EXTM3U
  #EXT-X-VERSION:3
  #EXT-X-TARGETDURATION:10
  seg000.ts
  seg001.ts
  ...
```

Client player downloads master manifest → picks best resolution based on bandwidth → downloads 10-second segments → switches quality as bandwidth changes (ABR algorithm).

## Deep Dive 2: CDN and Video Delivery

**Challenge:** 1.4 Tbps can't come from origin. CDN is mandatory.

**CDN strategy:**
- Push popular videos to CDN before users watch them (pre-warm top 1000 videos)
- Pull model for everything else: first request to CDN edge → cache miss → fetch from S3 → cache at edge
- Segment-level caching: each 10-second segment is a separate cacheable object
- Cache TTL: 7 days for video segments (immutable after transcoding); 60 seconds for manifest (may change status)

**Adaptive bitrate (ABR) client algorithm:**
```
BOLA algorithm (Buffer-Occupancy based Lyapunov Algorithm):
1. Maintain a playback buffer (target: 30 seconds buffered)
2. Measure download speed of last segment
3. If buffer > 30s and speed > 8Mbps → switch to 1080p
4. If buffer < 10s or speed < 3Mbps → switch to 720p or lower
5. Never let buffer drain to 0 (that causes rebuffering)
```

**Startup optimization:**
- First segment: always download lowest quality (360p) for instant start
- While 360p plays: download 720p or 1080p segments for subsequent playback
- CDN edge preloads the first segment for popular videos (proactive)

## Reliability and Operations

- **Transcoding failure:** Worker crashes mid-job → job released back to queue after visibility timeout. Retry with same or different worker. Max 3 retries; then mark video as failed.
- **Storage durability:** S3 provides 99.999999999% (11 nines) durability. Versioning enabled — accidental deletes recoverable.
- **CDN cache invalidation:** On video metadata update (title change) → invalidate manifest URL at CDN. On video deletion → purge all CDN segments (CDN purge API).
- **View count accuracy:** Real-time view count approximate (Redis). Exact count reconciled hourly from Cassandra aggregation. Avoid exact count on every page load — too slow.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| HLS | DASH (Dynamic Adaptive Streaming over HTTP) | HLS: Apple-native, widely supported; DASH: open standard, better codec flexibility |
| GPU transcoding workers | CPU-only | GPU: 10-50x faster, higher cost; CPU: slower, cheaper, more widely available |
| S3 presigned upload | Upload to our server | Direct S3: cheaper, no bandwidth cost on our servers; indirect: more control |
| CDN pull | CDN push | Pull: no upfront cost; Push: better hit rate for popular but wastes space for long-tail |

## L5 Additions

1. **Content-Aware Encoding:** Don't use fixed bitrate per resolution. Analyze each scene: action scenes need higher bitrate; static talking-head scenes can use much lower bitrate at same quality. Netflix uses shot-based encoding (SBE): encode each shot independently at optimal bitrate. Result: 20–40% smaller files at equivalent visual quality (measured by VMAF). Requires a per-title encoding pipeline that analyzes the full video before encoding.

2. **Multi-CDN strategy:** Rely on a single CDN vendor is a risk. Use 2–3 CDN providers simultaneously. Route traffic based on performance probing: send 1% of requests to alternate CDN, measure latency, adjust routing dynamically. Client manifests can point to multiple CDN URLs for the same segment; on failure, client retries with alternate CDN. Reduces dependency on vendor, improves tail latency.

3. **Realtime transcoding for live → VOD:** When a live stream ends, the recorded HLS segments are already in S3. Transcoding job just needs to: (a) generate lower-quality versions (live stream produces one quality), (b) create the master manifest pointing to all qualities, (c) set appropriate metadata. This is much faster than raw video transcoding. Target: live stream VOD available within 2 minutes of stream end.

4. **Thumbnail generation and selection:** Generate 10 thumbnails at evenly-spaced intervals per video. Score each using a visual attractiveness model (no blur, good composition, emotionally engaging faces). Return top 3 candidates for uploader to choose. Auto-select the best if uploader doesn't choose within 24 hours. Thumbnail is served from CDN with 7-day cache. A/B test different thumbnails for new videos to optimize click-through rate.

5. **Video recommendation pipeline:** Watch history, video engagement signals (like rate, watch time, skip rate), and video embeddings (content similarity) are processed in batch (daily) and real-time (streaming) pipelines. Candidate generation (from user's watch history → similar videos via embedding ANN search) → ranking model (XGBoost or neural network scoring candidates) → diversity filter (no more than 3 videos from same channel) → serve top 20. This is a separate ML system; the storage design for recommendation feeds the same data model as Twitter's timeline.
