# 08. YouTube — Video Streaming Platform

**Type:** social-media
**Difficulty:** G4–G5
**Frequency:** Very High — video upload, transcoding pipeline, adaptive streaming

---

## Problem Statement

Design a video streaming platform like YouTube or Netflix. Users can upload videos, and other users can stream them. The system must handle 500 hours of video uploaded per minute, 1B hours of video watched daily, support adaptive bitrate streaming (different quality for different connection speeds), and provide sub-2-second startup time.

---

## Functional Requirements

**Must have:**
- Upload video (up to 5GB file)
- Watch video with adaptive bitrate streaming
- Search videos by title/keyword
- Video metadata: title, description, view count, likes

**Should have:**
- Comments and likes
- Recommendations / related videos
- Video chapters / timestamps
- Subtitles / captions

**Out of scope:**
- Live streaming (separate design)
- Content ID / copyright matching
- Ads serving

---

## Non-Functional Requirements

- **Scale:** 500 hours uploaded/min; 1B hours watched/day
- **Latency:** Video start < 2 seconds; buffering < 0.1% of playback time
- **Availability:** 99.99% — downtime = lost revenue
- **Durability:** No video loss (videos stored permanently by default)
- **Bandwidth:** 1B hours/day × 1Mbps avg = ~125TB/s — must use CDN

## Capacity Estimation

```
Upload throughput:
  500 hours/min = 30,000 minutes/min of content
  Assuming average video length 10 minutes: 3,000 videos/min = 50 videos/sec
  Average raw upload file: 1 hour at 1080p ≈ 2GB (H.264, ~4Mbps)
  Shorter average (10 min): 333MB per upload
  Upload ingest bandwidth: 50 files/sec × 333MB = 16.7GB/sec inbound

Storage per video (all resolutions + codecs):
  1-hour video, HLS segments:
    4K (AV1):   8GB
    1080p (H264): 2GB
    720p (H264):  1GB
    480p (H264):  500MB
    360p (H264):  200MB
    WebM 720p:    1.5GB
  Total per video: ~13GB for all renditions

  Daily storage added:
    500hr/min × 60min × 13GB/hr = 390TB/day
  10-year archive (assuming 30% YoY growth):
    Year 1: 142PB; Year 10: ~4EB (use S3 Intelligent-Tiering)

Watch bandwidth:
  1B hours/day = 41.7M hours/hour = 694,444 concurrent streams at peak
  Average bitrate: 2Mbps (mix of qualities)
  Peak bandwidth: 694K × 2Mbps = 1.39Tbps
  CDN serves 95%+ of this: 1.32Tbps from CDN edge
  Origin serves < 5% (cache misses on long-tail content): ~70Gbps

View counts:
  1B hours/day ÷ avg 10 min/video = 6B video starts/day = 69,444/sec
```

## API Design

```
# Upload flow
POST /v1/videos/upload-url
Body:    { title: string, description: string, filename: string, file_size_bytes: int }
Returns: {
  video_id:   "vid_abc123",
  upload_url: "https://s3.amazonaws.com/raw-uploads/...?X-Amz-Signature=...",
  upload_id:  "s3-multipart-id",     -- for resumable uploads
  expires_at: "2026-03-07T12:00:00Z" -- presigned URL expiry
}

PUT {upload_url}   [multipart upload directly to S3, bypasses our servers]
  Multipart: split file into 100MB chunks, upload in parallel
  Returns ETag per part; client assembles part list

POST /v1/videos/upload-url/{upload_id}/complete
Body: { parts: [{part_number, etag}] }   -- complete S3 multipart
Returns: { status: "processing" }

# Watch flow
GET /v1/videos/{id}
Returns: {
  id, title, description, view_count, like_count, comment_count,
  duration_sec, thumbnail_url, channel: {id, name, subscriber_count},
  stream: { master_manifest_url, available_qualities: ["360p","720p","1080p","4k"] },
  subtitles: [{ language, url }],
  status: "ready" | "processing" | "failed"
}

# HLS manifest (served directly by CDN)
GET https://cdn.example.com/videos/{video_id}/master.m3u8
GET https://cdn.example.com/videos/{video_id}/1080p/index.m3u8
GET https://cdn.example.com/videos/{video_id}/1080p/seg{N}.ts

# Engagement
POST /v1/videos/{id}/view    -- register view event (increments counter async)
POST /v1/videos/{id}/like
GET /v1/videos/{id}/comments?cursor=&limit=20
POST /v1/videos/{id}/comments  Body: { text, reply_to?: comment_id }
```

## Deep Dive 1: Video Upload and Transcoding Pipeline

**Resumable multipart upload to S3:**
```
Problem: 5GB video upload on a mobile connection can fail halfway
Solution: S3 multipart upload with client-side resume

Protocol:
  1. POST /v1/videos/upload-url → server creates S3 multipart upload, returns upload_id
  2. Client splits file into N × 100MB chunks
  3. Client uploads each chunk in parallel (4–8 concurrent connections):
       PUT {presigned_url_part_N} → returns ETag for that part
  4. If connection drops:
       Client calls GET /v1/videos/{upload_id}/progress
       Server calls S3 ListMultipartUploadParts → returns which parts are done
       Client resumes by re-uploading only missing parts
  5. On all parts done: POST /v1/videos/{upload_id}/complete
       Server calls S3 CompleteMultipartUpload → assembles final object
  6. S3 triggers event → SQS → Transcoding Orchestrator

Chunk size tuning:
  Smaller chunks (10MB): more parallel, better resume granularity, more S3 PUT requests
  Larger chunks (500MB): fewer requests, but must restart a large chunk on failure
  100MB is a good default; tune based on typical connection speeds
```

**Transcoding orchestrator and worker pool:**
```
Target renditions per video:
  Quality   Codec    Bitrate    Resolution    Notes
  -------   -----    -------    ----------    -----
  360p      H264     500 Kbps   640×360       Universal compatibility
  480p      H264     1.5 Mbps   854×480
  720p      H264     3 Mbps     1280×720      HD sweet spot
  1080p     H264     6 Mbps     1920×1080     Full HD
  4K        H265     20 Mbps    3840×2160     Premium
  720p      VP9      2.5 Mbps   1280×720      For Chrome/Android
  1080p     AV1      4 Mbps     1920×1080     Next-gen (smaller files)

Segment duration: 10 seconds per .ts segment (standard; 4–6s for live)
Total renditions: 7 jobs submitted in parallel

Worker implementation (FFmpeg-based):
  Each worker is a GPU-equipped EC2 instance (g4dn.xlarge: T4 GPU)
  GPU accelerates: H264/H265 decode + HEVC/H264 encode (NVENC)
  Throughput: T4 GPU encodes 1080p at 60× real-time
    → 10-minute video encoded in 10 seconds

Transcoding pipeline per worker:
  1. Download raw video from S3 to local NVMe (parallel download, 10Gbps NIC)
  2. FFmpeg command:
     ffmpeg -hwaccel cuda -i input.mp4 \
            -vf scale=1280:720 \
            -c:v h264_nvenc -b:v 3M \
            -c:a aac -b:a 128k \
            -f hls -hls_time 10 -hls_playlist_type vod \
            -hls_segment_filename "seg%04d.ts" \
            output/720p/index.m3u8
  3. Upload segments to S3: videos/{video_id}/720p/seg0001.ts, seg0002.ts, ...
  4. Upload manifest: videos/{video_id}/720p/index.m3u8
  5. Report completion to Orchestrator

Orchestrator logic:
  Track which of the 7 renditions are complete
  When all 7 done:
    Generate master manifest (master.m3u8) listing all renditions
    Upload to S3: videos/{video_id}/master.m3u8
    UPDATE videos SET status='ready', published_at=now() WHERE video_id=...
    Publish "video-ready" event to Kafka → triggers CDN pre-warm for popular content
    Delete raw upload from S3 (raw-uploads/ bucket, no longer needed)
```

**HLS manifest format:**
```
# master.m3u8 (served by CDN, allows player to pick quality)
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-STREAM-INF:BANDWIDTH=500000,RESOLUTION=640x360,CODECS="avc1.42c01e,mp4a.40.2"
360p/index.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=3000000,RESOLUTION=1280x720,CODECS="avc1.4d401f,mp4a.40.2"
720p/index.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=6000000,RESOLUTION=1920x1080,CODECS="avc1.640028,mp4a.40.2"
1080p/index.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720,CODECS="vp09.00.31.08"
720p_vp9/index.m3u8

# 720p/index.m3u8 (one per quality level)
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:10
#EXT-X-MEDIA-SEQUENCE:0
#EXT-X-PLAYLIST-TYPE:VOD
#EXTINF:10.0,
seg0001.ts
#EXTINF:10.0,
seg0002.ts
...
#EXT-X-ENDLIST
```

Player behavior: downloads master manifest → picks 720p based on measured bandwidth → downloads 10-second segments sequentially → switches quality up or down as bandwidth changes.

**G5:** The transcoding pipeline treats all videos equally — but a 10-second viral clip and a 4-hour documentary require very different transcoding strategies. Segment duration is fixed at 10 seconds, but a 4-hour documentary with 10-second segments = 1,440 segment files per rendition × 7 renditions = 10,080 S3 objects per video. S3 handles this fine, but manifest generation and CDN cache warming become expensive. For long-form content, use longer segments (60 seconds) to reduce file count by 6×. For short viral clips (< 60 seconds), use a single-file approach: one `.ts` file per quality instead of segmented. The transcoding orchestrator should choose segment strategy based on video duration, not apply a one-size-fits-all 10-second segment duration.

## Deep Dive 2: CDN and Video Delivery

**CDN architecture:**
```
CDN Provider: Cloudfront (or Fastly, Akamai) with 200+ global PoPs

Object types and cache strategies:
  master.m3u8:  Cache-Control: max-age=60, s-maxage=60
                → Short TTL (status can change: video processing, quality additions)
  {quality}/index.m3u8: Cache-Control: max-age=300
                → Medium TTL (quality manifest rarely changes post-transcoding)
  {quality}/seg{N}.ts: Cache-Control: max-age=86400, immutable
                → Long TTL (segments are content-addressed — never change)
                → Cache key: full path including video_id and quality
  Thumbnails:   Cache-Control: max-age=604800
                → 1 week (thumbnails change rarely)

Cache key design:
  Full URL path: /videos/vid_abc123/720p/seg0042.ts
  → Each 10-second segment is a separate cache object
  → 700K concurrent streams × 1 segment/10s = 70K CDN object fetches/sec

Segment-level caching advantage:
  If a video is 60 minutes = 360 segments per quality
  User seeking to 30-minute mark: CDN fetches from seg180 onward
  Previously cached segs 0-179 still benefit others watching from the beginning
  → Cache hit rate for popular videos: > 99%
```

**CDN pre-warming for popular content:**
```
When a video goes viral or is scheduled for release:
  1. Predict traffic: ML model on upload metadata (channel size, title keywords,
     early engagement signals in first 30 minutes post-publish)
  2. Pre-warm top CDN PoPs:
     for each PoP in high_traffic_regions:
       for each quality in ['360p', '720p', '1080p']:
         for each segment in all_segments:
           CDN.prefetch(f"https://cdn.example.com/videos/{id}/{quality}/seg{n}.ts")
  3. Cost: 1 hour video × 360 segments × 3 qualities = 1,080 pre-warm requests/video
     At 100KB/segment avg: 108MB per video per PoP
     For top 1000 videos × 50 PoPs: 5.4TB of pre-warming → expensive but worth it for #1 trending

Long-tail content (< 100 views/day):
  Pure pull caching: first viewer causes cache miss → fetch from S3 → cache at edge
  Cache TTL = 7 days even for long-tail (immutable segments don't need invalidation)
  Storage cost at CDN: minimal (long-tail rarely accessed; evicted by LRU)
```

**Adaptive bitrate algorithm (client-side):**
```
BOLA (Buffer-Occupancy based Lyapunov Algorithm) — used by DASH and HLS players:

State:
  buffer_seconds:  how many seconds of video are buffered locally (target: 30s)
  last_throughput: measured download speed of last 3 segments (exponential moving avg)

Decision logic per segment:
  if buffer_seconds < 5:
    # Emergency: buffer critically low → lowest quality immediately
    quality = '360p'
  elif last_throughput < 1.5 Mbps:
    quality = '360p'
  elif last_throughput < 3 Mbps:
    quality = '480p'
  elif last_throughput < 6 Mbps and buffer_seconds > 15:
    quality = '720p'
  elif last_throughput > 8 Mbps and buffer_seconds > 25:
    quality = '1080p'
  else:
    quality = current_quality  # stay stable

  # Never let buffer drop to 0 — that causes rebuffering (worst user experience)
  # Switch down aggressively; switch up conservatively (hysteresis)

Startup strategy (sub-2-second start):
  1. Always begin with lowest quality (360p): start downloading immediately
  2. Fetch seg001.ts of 360p: ~200KB at any reasonable connection speed = < 1s
  3. While 360p seg001 plays (10 seconds): download 720p or 1080p segs 002+
  4. Seamless quality switch between segments
```

**G5:** The CDN layer assumes video segments are immutable after upload — and they are for VOD. But subtitles and captions are frequently added or corrected after a video publishes, and the subtitle `.vtt` files are served from the same CDN. If the CDN caches a buggy subtitle file with a 7-day TTL, all viewers see incorrect captions for a week. The fix: subtitle files need a shorter TTL (10 minutes) or cache-busting via versioned URLs (`/subtitles/v3/en.vtt` instead of `/subtitles/en.vtt`). More broadly: separate the CDN cache policy by content mutability — video segments (immutable), subtitle files (mutable), manifests (mutable), thumbnails (occasionally mutable). Each category needs its own TTL and invalidation strategy. Mixing them into one policy is a common oversight that leads to either stale user-facing content or unnecessary origin load.

## Reliability and Operations

- **Transcoding failure:** Worker crashes mid-job → SQS visibility timeout (5 min) expires → message becomes visible again → picked up by another worker. Retry max 3 times; after 3 failures → move to dead-letter queue (DLQ) → alert + manual investigation. Worker uses a checkpoint: uploads completed segments to S3 as it goes. On retry: check which segments already exist in S3, resume from last completed segment.
- **Storage durability:** S3 provides 99.999999999% (11 nines) durability (data replicated across 3+ AZs). Versioning enabled on raw-uploads bucket — accidental delete of original file is recoverable. Lifecycle policy: move raw uploads to S3 Glacier after transcoding completes (saves ~70% storage cost for raws).
- **CDN cache invalidation:** On video deletion: call CDN invalidation API for `videos/{video_id}/*` (purges all segments, manifests, thumbnails). On metadata update (title change): only manifest URL changes; segments are unaffected. On thumbnail update: invalidate specific thumbnail URL only.
- **View count accuracy:** Redis INCR on every view → approximate real-time count. Kafka consumer batches view events → writes to Cassandra (authoritative). Every hour: Spark job aggregates Cassandra view events → writes exact count to PostgreSQL `videos.view_count`. Redis reseeded from PostgreSQL every midnight. Result: Redis count is accurate within 1 hour; PostgreSQL count is accurate within 1 hour; Cassandra has full audit history.

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| HLS | MPEG-DASH | HLS: native support on iOS/Safari (required for Apple devices); wider CDN support; simpler tooling (FFmpeg HLS is well-understood). DASH: open standard, better codec flexibility (can mix H264 and AV1 in same manifest), used by Netflix/YouTube. Modern systems often support both. |
| GPU transcoding (NVENC) | CPU-only (x264) | GPU: 10–100× faster encode, critical for meeting the "video ready quickly" SLA; T4 GPU encodes 1080p at 60× real-time vs CPU at 1–2× real-time. CPU: cheaper per machine, more flexible (software codec support), but far too slow for 500 hours/min upload rate. |
| S3 presigned upload | Upload through our API server | Direct S3: client→S3 without touching our servers → saves enormous bandwidth cost (500 files/sec × 333MB = 166GB/sec would require 1,330 × 1Gbps server NICs). Through our servers: enables progress tracking and validation before S3, but is architecturally infeasible at this scale. |
| CDN pull (cache-on-miss) | CDN push (pre-populate all content) | Pull: zero upfront cost; automatic for long-tail content. Push: guarantees cache hits but requires O(videos × renditions × PoPs) storage — impractical for 10M+ videos. Hybrid: push for top 1% of content, pull for the rest. |

## G5

**What a G5 engineer adds beyond the above:**

1. **Content-Aware Encoding (per-title, per-shot):** Netflix's innovation: don't apply a fixed bitrate (e.g., 3Mbps for 720p) to all content. Analyze the video first: a talking-head interview has low motion and fine detail → can be encoded at 1.5Mbps at equivalent quality. An action film with fast motion and complex scenes needs 4Mbps for the same quality. Per-shot encoding: split video at scene boundaries, encode each shot at its optimal bitrate (measured by VMAF score target, e.g., VMAF=93). Result: 20–40% smaller files at identical perceptual quality, or 20–40% better quality at the same file size. Implementation: a pre-analysis pass (fast, CPU-only) → shot detection + complexity scoring → per-shot bitrate assignment → parallel shot encoding → reassemble with concatenation. Adds 2× pipeline complexity but dramatically improves storage cost and viewer quality.

2. **Multi-CDN strategy for resilience and tail latency:** A single CDN provider is a dependency risk (Fastly outage in 2021 took down half the internet). Run 2–3 CDN providers simultaneously: Cloudfront + Fastly + Akamai. Segment the traffic by geography or by URL pattern: Cloudfront for Americas, Akamai for Asia. Alternatively: use a CDN orchestration layer (Cedexis/Citrix Intelligent Traffic Management) that routes each viewer to the best-performing CDN based on real-time measurements. Client manifests can include segment URLs from multiple CDNs: on segment download failure, retry immediately with alternate CDN URL — sub-second failover without buffering.

3. **Thumbnail generation and A/B selection:** Generate 10 candidate thumbnails at evenly-spaced video timestamps. Score each with a visual quality model: blur detection (Laplacian variance), face attractiveness score, composition rules, color vibrancy. Upload top 3 candidates to S3, store in `video_thumbnails` table with scores. Present top 3 to uploader for selection during upload. Auto-select highest-scoring if no selection within 24 hours. For existing videos: A/B test two thumbnails by serving each to 50% of recommendation impressions and measuring click-through rate over 48 hours. Auto-promote the higher CTR thumbnail. Thumbnail changes require CDN cache invalidation: use versioned URLs (`thumb_v2.jpg`) to invalidate cleanly without purging the entire thumbnail edge cache.

4. **Realtime transcoding state machine with observability:** The transcoding pipeline has 7 parallel jobs → 7 possible failure points. A naive orchestrator just waits for all 7 and retries failed ones. A production orchestrator needs: (a) Partial availability: if 360p, 720p are done but 1080p is still transcoding, mark video as "partially ready" and serve available qualities immediately. Users can watch 720p while 1080p finishes. (b) Priority: transcode 360p and 720p first (most viewers use these); 4K last. (c) Cost optimization: for low-viewership videos, don't transcode 4K (98% of users never watch 4K); trigger 4K transcoding only when view count exceeds 10K.

5. **Video recommendation pipeline architecture:** Recommendations drive 70%+ of YouTube watch time — this is as important as the delivery infrastructure. Candidate generation: collaborative filtering via matrix factorization (ALS on user-video watch history) produces 1,000 candidate videos per user. Content-based candidates: video embedding similarity (from visual and audio features) adds 500 more. Total: 1,500 candidates. Ranking model: XGBoost or two-tower neural network scores each candidate on predicted watch time, like probability, and share probability. Diversity filter: no more than 3 videos from the same channel, at most 2 from the same category. The entire inference pipeline runs in < 100ms, served from a dedicated recommendation service with GPU accelerated ranking. Batch pre-computation updates user and video embeddings daily; real-time signals (just liked a video) update the ranking inputs within 30 seconds via Kafka → feature store update.
