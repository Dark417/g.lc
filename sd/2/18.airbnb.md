# 18. Airbnb — Hotel/Home Booking Platform

**Type:** commerce
**Difficulty:** L4
**Frequency:** High — inventory, search, booking transactions, availability

---

## Problem Statement

Design Airbnb. Hosts can list properties with availability calendars and pricing. Guests can search for properties by location, check-in/check-out dates, and book. The system must handle 100M registered users, 5M listings, 1M bookings/day, and prevent double-booking (two guests booking the same property for overlapping dates).

---

## Functional Requirements

**Must have:**
- Host: create/edit listing (photos, description, amenities, price/night)
- Host: manage availability calendar (block/unblock dates, set prices per date)
- Guest: search listings by location, check-in/check-out dates, guests count
- Guest: view listing details + availability
- Guest: book a listing (reserve dates, pay)
- Booking confirmation / cancellation

**Should have:**
- Reviews (after stay completion)
- Messaging between host and guest
- Wishlist / saved properties
- Price recommendation for hosts

**Out of scope:**
- Payment processing details (use Stripe/Braintree)
- Customer support ticketing
- Smart pricing algorithm details

---

## Non-Functional Requirements

- **Scale:** 100M users, 5M listings, 1M bookings/day
- **Availability:** 99.99% — booking failures lose revenue
- **Consistency:** Strong for bookings (no double-booking); eventual for search
- **Latency:** Search results < 300ms; booking confirmation < 2s
- **Durability:** No booking lost

## Capacity Estimation

```

  5M listings × 5KB avg metadata = 25GB → fits in Redis for hot listings
  Photos: avg 20 photos × 500KB = 10MB per listing → S3 for all photos
  Total photo storage: 5M × 10MB = 50TB → S3

  5M listings × 365 days/year = 1.825B availability rows/year
  Per row: listing_id(8B) + date(4B) + status(1B) + price(4B) + booking_id(8B) = 25B
  Total: 1.825B × 25B = 45GB → fits in PostgreSQL, also cacheable in Redis

Bookings:
  1M bookings/day = 11.6/sec average; peak ~100/sec
  Each booking: avg 4 nights → 4 availability rows updated per booking
  Availability updates: 100/sec × 4 = 400 writes/sec at peak

Search:
  100M DAU × 5 searches/day = 500M searches/day = 5,787/sec average
  Peak (summer, holiday seasons): 5× = ~30K search QPS
  Search is read-heavy → Elasticsearch + Redis caching required

Cache efficiency:
  Hot listings (top 1% = 50K listings) get 80% of traffic → cache these
  50K listings × 5KB = 250MB → trivial cache footprint
```

## Deep Dive 1: Preventing Double-Booking

**The critical requirement:** Two guests cannot book the same listing for any overlapping dates.

**Why it's hard:** Search shows a listing as "available" to 10,000 concurrent users. Multiple users can begin the booking flow simultaneously. Between when user A sees "available" and when they submit payment, user B could complete their booking and take the dates.

**Approach A: Pessimistic locking with SELECT FOR UPDATE (recommended)**
```sql
-- Booking transaction (PostgreSQL)
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Step 1: Lock all availability rows for the requested dates
-- FOR UPDATE acquires exclusive row locks; concurrent transactions block here
SELECT date, status, booking_id
FROM availability
WHERE listing_id = $1
  AND date >= $check_in
  AND date < $check_out  -- check_out is exclusive (you don't "use" checkout day)
ORDER BY date
FOR UPDATE;

-- Step 2: Verify all dates are available
-- (if any row has status != 'available', rollback)
-- Application code: check result set, any non-'available' row → ROLLBACK

-- Step 3: Reserve all dates atomically
UPDATE availability
SET
  status = 'booked',
  booking_id = $new_booking_id
WHERE listing_id = $1
  AND date >= $check_in
  AND date < $check_out
  AND status = 'available';   -- defensive: only update if still available

-- Step 4: Verify the update touched exactly the expected number of rows
-- rows_updated != (check_out - check_in) → another transaction modified the rows
-- → ROLLBACK, return "no longer available"

-- Step 5: Create the booking record
INSERT INTO bookings (id, listing_id, guest_id, check_in, check_out, ...)
VALUES ($new_booking_id, $1, $guest_id, $check_in, $check_out, ...);

COMMIT;
```

**Why `SELECT FOR UPDATE` works:**
- Row-level exclusive lock acquired on each availability row in step 1
- A second concurrent transaction trying to lock the same rows for the same date range → BLOCKS (waits for first to commit or rollback)
- First transaction commits → second transaction unblocks, re-reads rows, sees `status='booked'` → returns "unavailable" to the second guest
- This is correct serializable behavior with no race condition

**What happens to the second guest:**
- Their transaction blocks for typically < 1 second (time for payment processing)
- On unblock: reads `status='booked'`, rolls back, returns HTTP 409 Conflict "These dates are no longer available"
- Retry: they see the calendar updated and can choose different dates

**Approach B: Optimistic locking (for lower contention scenarios)**
```sql
-- Add a version column to availability rows
ALTER TABLE availability ADD COLUMN version INT NOT NULL DEFAULT 0;

-- Check without lock (read phase):
SELECT date, status, version FROM availability
WHERE listing_id = $1 AND date BETWEEN $check_in AND $check_out_minus_1;

-- Application: verify all status = 'available'

-- Update with version check (write phase):
UPDATE availability
SET status = 'booked', booking_id = $booking_id, version = version + 1
WHERE listing_id = $1
  AND date BETWEEN $check_in AND $check_out_minus_1
  AND status = 'available'
  AND version = $read_version;  -- fails if version changed between read and write

-- If rows_updated != expected: conflict → retry from the beginning
-- This is a CAS (Compare-And-Swap) operation at the row level
```

**When to use optimistic vs pessimistic:**
- Pessimistic (FOR UPDATE): correct choice for booking, where contention happens in bursts (flash sales) and the cost of retry is high (payment already initiated, user frustrated)
- Optimistic: better for low-contention updates where retries are cheap (e.g., updating a listing's description)

**Distributed lock (for microservices architecture where DB lock is insufficient):**
```
Acquire distributed lock before starting DB transaction:
  SET lock:listing:{listing_id}:dates:{check_in}:{check_out} {booking_id} NX EX 30
  → NX: only set if not exists (atomic acquire)
  → EX 30: auto-release after 30 seconds (safety: if process crashes, lock releases)
  → Returns OK (acquired) or nil (already locked → return "booking in progress, retry")

Then: perform DB transaction as above
Finally: DEL lock:listing:{listing_id}:dates:{check_in}:{check_out}

Rationale for both distributed lock + DB transaction:
  Distributed lock: fast rejection (< 1ms) of concurrent requests at application layer
  DB transaction: correctness guarantee even if distributed lock fails
  Defense in depth: two independent mechanisms for double-booking prevention
```

**G5:** Would flag that SELECT FOR UPDATE on a listing's availability rows creates contention under flash-sale conditions — for example, a popular listing on a holiday weekend (July 4th in a beach town) might receive 200 simultaneous booking attempts. Each concurrent transaction locks the same 4 date rows and queues behind the first. The lock wait queue serializes all 200 attempts, creating a thundering herd: all transactions wait, then all but one fail immediately. This can cause database connection exhaustion and lock timeouts that cascade into application errors. Mitigation: use optimistic locking (version column + conditional UPDATE) instead of SELECT FOR UPDATE. If the UPDATE returns 0 rows (version mismatch), retry at the application layer. With optimistic locking, no transactions block each other — they all proceed concurrently, and only the DB write phase (microseconds) conflicts. This reduces lock contention by 10–100× at the cost of occasional retry overhead, which is acceptable when the p99 contention rate is < 1%. For the top 100 most popular listings during peak season, consider a booking queue: serialize attempts at the application layer with a queue-based UI ("You're #47 in line"), so the DB only sees one write at a time.

## Deep Dive 2: Search — Availability-Filtered Search

**The compound search challenge:**
Search for "2-bedroom apartment in SF, July 1-5" requires simultaneously filtering by:
1. Geographic location (lat/lng within SF bounding box or radius)
2. Property attributes (bedrooms >= 2, type = apartment)
3. Availability (ALL dates July 1-4 must be status='available')
4. Price range
5. Amenities
6. Result ranking (relevance, rating, price competitiveness)

No single database handles all these efficiently at 30K QPS. The solution is a two-service architecture:

**Search architecture (two-service pipeline):**
```
Step 1: Elasticsearch — fast geo + attribute filtering
  Query (structured DSL):
  {
    "query": {
      "bool": {
        "must": [
          { "geo_bounding_box": { "location": { "top_left": SF_NW_CORNER, "bottom_right": SF_SE_CORNER } } }
        ],
        "filter": [
          { "term": { "property_type": "apartment" } },
          { "range": { "bedrooms": { "gte": 2 } } },
          { "range": { "base_price": { "gte": 50, "lte": 300 } } },
          { "terms": { "amenities": ["wifi"] } },
          { "term": { "status": "active" } }
        ]
      }
    },
    "size": 5000  # get top 5K candidates for availability filtering
  }
  Result: 5,000 candidate listing_ids (returned in ~20ms)

Step 2: Availability batch check (parallel Redis lookups)
  For each candidate listing_id, check if ALL requested dates are available:
    cache_key = avail:{listing_id}:{check_in}:{check_out}
    result = Redis.get(cache_key)
    if result is None:
      # Cache miss: query PostgreSQL
      query = """
        SELECT COUNT(*) as available_count
        FROM availability
        WHERE listing_id = %s
          AND date >= %s AND date < %s
          AND status = 'available'
      """
      count = db.query(query, listing_id, check_in, check_out)[0]
      nights = (check_out - check_in).days
      is_available = (count == nights)
      Redis.setex(cache_key, 60, '1' if is_available else '0')
    else:
      is_available = (result == '1')

  Execute in parallel (asyncio / thread pool): 5,000 lookups × ~1ms each = 5ms total
  Filter: keep only available listings
  Result: ~2,000 available listings

Step 3: Rank and paginate
  score(listing) = (
    text_relevance_score            # from Elasticsearch BM25 on title/description
    + avg_rating × log(1 + review_count)  # trust-weighted rating
    + host_response_rate × 0.1     # responsiveness signal
    + price_competitiveness         # how this price compares to similar listings
    - distance_from_center × 0.05  # slight penalty for distance from search center
  )
  Sort by score DESC, paginate with cursor, return top 20
```

**Availability caching strategy:**
```
Redis Hash per listing:
  Key:   avail:{listing_id}
  Type:  Hash (field = date string "2026-07-01", value = status)
  TTL:   60 seconds

On booking completion: DEL avail:{listing_id}
On host blocking date: DEL avail:{listing_id}
On cancellation (dates become available again): DEL avail:{listing_id}

For popular listings with many concurrent searches (e.g., a Superhost cabin getting 1K searches/min):
  Pre-compute "available date ranges" as a sorted set of available (start, end) intervals
  Key: avail_ranges:{listing_id}
  Cache as: [{ start: "2026-06-01", end: "2026-06-28" }, { start: "2026-08-01", end: "2026-12-31" }]
  Availability check: O(log N) interval lookup instead of N individual date checks
  Invalidate on any availability change
```

**G5:** Would note that the 60-second Redis TTL means a listing that just got booked can still appear as "available" in search results for up to 60 seconds. A guest clicks on such a listing, starts the booking flow, and only discovers unavailability when they hit the DB transaction. This is accepted as "eventual consistency for search" — but the UX can be improved by eagerly invalidating the cache key on booking (publish a `booking_completed` Kafka event → cache invalidation consumer). More subtly, the availability batch check in Step 2 introduces a TOCTOU (Time-of-Check-Time-of-Use) race: availability is checked during search, but another user could book the listing in the milliseconds between the search check and the user initiating the booking flow. This is unavoidable without holding a read lock during the entire browse session (prohibitively expensive). The correct design acknowledges this and handles the conflict gracefully in the booking service with a clear error message, rather than trying to eliminate the race window.

## Reliability and Operations

- **Booking with payment failure — two-phase flow:**
  Phase 1: Mark dates as `status='reserved_pending_payment'` (prevents double-booking during payment processing)
  Phase 2: Initiate Stripe PaymentIntent capture (async, up to 10 seconds)
  On success: set status='booked', booking.payment_status='captured'
  On payment failure: revert status back to 'available', delete pending booking record, notify guest
  Timeout: if payment capture exceeds 10 minutes with no response, run background job to release reservation and notify guest
- **Listing sync to Elasticsearch:** Async via Kafka. On listing create/update: write to PostgreSQL → publish to Kafka `listing-events` topic → ES indexer consumer updates Elasticsearch. Lag: typically 5–30 seconds. For listing detail pages: always read from PostgreSQL (authoritative); Elasticsearch is only for search filtering/ranking.
- **Host approval required (non-instant-book):** Guest submits booking request → status='pending'. Host has 24 hours to accept or decline. If host accepts: proceed to payment capture. If host declines or 24h elapses: release availability reservation, notify guest. Implementation: delayed job (Celery/Sidekiq) scheduled 24h after booking creation → auto-decline if still pending.
- **Availability table read replicas:** All search availability checks routed to PostgreSQL read replicas. Booking writes (SELECT FOR UPDATE + UPDATE) routed to primary. Replica lag: < 100ms — acceptable for search staleness SLA but NOT acceptable for booking. Booking service always reads from primary.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| `SELECT FOR UPDATE` (pessimistic lock) | Optimistic locking (version column) | Pessimistic: correct for high-stakes booking, no retries; Optimistic: better throughput for low-contention SKUs, requires retry logic, fails under flash-sale conditions |
| Elasticsearch + PostgreSQL availability check | All in PostgreSQL | ES: fast geo+attribute search with rich faceting; All-PG: simpler, single data store, but full-table scans for geo queries on 5M listings are prohibitively slow |
| Redis availability cache (60s TTL) | No cache — DB-only | Redis: serves 99% of availability browse traffic without DB hits; trade-off: up to 60s stale display, requires cache invalidation on booking |
| Date-granularity availability table (1 row/date/listing) | Range-based booking table | Date-granularity: simple overlap check (`status = 'available'`); Range-based: compact storage, but range overlap queries require `NOT EXISTS (SELECT ... WHERE ranges overlap)` — complex and slower |
| Kafka for listing→ES sync | Synchronous ES write on listing update | Kafka: decoupled, ES indexing can lag without blocking listing creation; Sync ES write: always consistent, but ES write failure blocks listing creation |

## G5

1. **Price recommendation engine for hosts:** Hosts systematically underprice or overprice relative to market. Design: historical booking data per listing (occupancy rate by price point), competitor listings in the same neighborhood (same property type, bedrooms, amenities), local event calendar (concerts, conferences → demand spike). ML model: gradient-boosted tree trained on (listing features, date features, local demand features) → predict optimal price that maximizes revenue (price × expected occupancy rate). Output: nightly price suggestion shown on host calendar UI ("Suggested: $185/night — +18% vs your current price"). A/B tested: hosts who follow price suggestions achieve 22% higher revenue on average. Model retrained weekly on fresh booking data. For real-time signals (sudden local event): a rule-based layer overrides the model's suggestion with a surge coefficient.

2. **Instant booking vs request-to-book flow:** Instant Book (80% of Airbnb supply): guest pays immediately, dates blocked, host auto-confirmed. Request-to-book (20%): host manually approves. Architecture for request-to-book: Stripe `PaymentIntent` with `capture_method: manual` — funds authorized (held on guest's card) but not captured. Host sees the request with a 24-hour response deadline. On host accept: `stripe.paymentIntents.capture(payment_intent_id)` — funds transferred. On host decline/timeout: `stripe.paymentIntents.cancel(payment_intent_id)` — authorization released (no charge to guest). Authorization hold on the guest's card expires after 7 days if not captured — important to confirm or cancel within that window. SAGA compensating transaction: if payment capture succeeds but the booking status update fails, a background reconciliation job detects the discrepancy (captured PaymentIntent with no confirmed booking) and either completes the booking or issues a refund.

3. **Multi-currency pricing and tax compliance:** Guests see prices in their local currency (display conversion at time of search); hosts receive payment in their registered bank currency. Currency conversion: locked at booking time using live exchange rate + small spread — guest sees an estimated amount in local currency, charged the exact equivalent. VAT/GST by jurisdiction: 60+ countries require Airbnb to collect and remit accommodation tax. Architecture: tax calculation service that takes (listing country, listing city, booking dates, guest country) → returns (tax_amount, tax_rate, jurisdiction_code). Result stored per booking in an immutable `tax_records` table for tax authority audit. Annually: generate remittance reports per jurisdiction from `tax_records` → automated filing in supported countries. This is genuinely one of the hardest operational problems in global e-commerce — getting it wrong is a legal liability.

4. **Trust, safety, and Superhost badges:** Trust is the core business model — a stranger is sleeping in your home. Superhost criteria (recomputed quarterly): response rate ≥ 90%, acceptance rate ≥ 90%, < 1% cancellation rate, avg rating ≥ 4.8, minimum 10 stays or 100 nights. Implemented as a nightly batch job: compute metrics per host from bookings and reviews tables, write `superhost_status` to user profile. Guest trust: ID verification (passport/driver's license via Jumio or Stripe Identity, typically < 30 seconds), booking history, account age. Fraudulent booking detection: new account + luxury listing + high-value booking + non-standard payment → hold for manual review. The trust graph (who has stayed with whom, who has reviewed whom) is also used for fraud ring detection: a cluster of accounts with no mutual connections that all left each other 5-star reviews → suspicious.

5. **Availability table under peak load:** During holiday weekends (July 4th, Thanksgiving, New Year's Eve), popular listings in tourist destinations receive thousands of concurrent availability checks and hundreds of simultaneous booking attempts. The availability table is the bottleneck: it's a hot partition (many rows for one listing_id, all being read/written). Mitigations: (a) Read replicas for availability browse checks — route all search-time availability reads to replicas, only booking writes to primary. (b) Connection pooling via PgBouncer in transaction mode: reduces PostgreSQL connection overhead from 1 connection per app thread to 1/10th of that. (c) For SELECT FOR UPDATE contention: implement a listing-level booking queue in Redis — serialize booking attempts so only one enters the DB transaction at a time. The queue provides both double-booking protection at the application layer and a better user experience ("You're next in line"). (d) Pre-scale the availability DB partition to 3× normal capacity two weeks before peak periods, validated via load testing at 10× normal traffic.
