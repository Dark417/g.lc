# 18. Airbnb — Hotel/Home Booking Platform

**Type:** commerce
**Difficulty:** L4
**Frequency:** High — inventory, search, booking transactions, availability

---

## Problem Statement

Design Airbnb. Hosts can list properties with availability calendars and pricing. Guests can search for properties by location/dates, view details, and book. The system must handle 100M registered users, 5M listings, 1M bookings/day, and prevent double-booking (two guests booking the same property for overlapping dates).

---

## Functional Requirements

**Must have:**
- Host: create/edit listing (photos, description, amenities, price/night)
- Host: manage availability calendar (block/unblock dates, set prices per date)
- Guest: search listings by location, check-in/check-out dates, guests count
- Guest: view listing details + availability
- Guest: book a listing (reserve dates, pay)
- Booking confirmation / cancellation

**Should have:**
- Reviews (after stay completion)
- Messaging between host and guest
- Wishlist / saved properties
- Price recommendation for hosts

**Out of scope:**
- Payment processing details (use Stripe/Braintree)
- Customer support ticketing
- Smart pricing algorithm details

---

## Non-Functional Requirements

- **Scale:** 100M users, 5M listings, 1M bookings/day
- **Availability:** 99.99% — booking failures lose revenue
- **Consistency:** Strong for bookings (no double-booking); eventual for search
- **Latency:** Search results < 300ms; booking confirmation < 2s
- **Durability:** No booking lost

---

## Capacity Estimation

```
Listings: 5M × 5KB = 25GB → fits in cache
Availability data: 5M listings × 365 days = 1.8B availability records/year
  Per record: listing_id(8B) + date(4B) + status(1B) + price(4B) = 17B
  Total: 1.8B × 17B = ~30GB → fits in Redis

Bookings: 1M/day = 12/sec average; peak ~100/sec
Search: 100M DAU × 5 searches/day = 500M/day = 5,800/sec; peak ~30K/sec
  Search is read-heavy → needs caching and search index (Elasticsearch)
```

## Data Model

```sql
-- PostgreSQL: listings
listings (
  id            UUID PRIMARY KEY,
  host_id       UUID NOT NULL,
  title         TEXT,
  description   TEXT,
  property_type TEXT,  -- 'apartment', 'house', 'room', ...
  lat           DECIMAL(9,6),
  lng           DECIMAL(9,6),
  city          TEXT,
  country       TEXT,
  base_price    DECIMAL(8,2),  -- per night
  max_guests    INT,
  amenities     TEXT[],
  photos        JSONB,         -- [{ url, caption }]
  avg_rating    DECIMAL(2,1),
  review_count  INT,
  status        TEXT,          -- 'active', 'inactive', 'suspended'
  created_at    TIMESTAMPTZ
)

-- Availability calendar: one row per date per listing
availability (
  listing_id   UUID,
  date         DATE,
  status       TEXT,    -- 'available', 'booked', 'blocked_by_host'
  price        DECIMAL(8,2),  -- override for this specific date (NULL = use base_price)
  booking_id   UUID,    -- which booking reserved this date (NULL if available)

)
-- This is the critical table: enforces no double-booking

  id               UUID PRIMARY KEY,
  listing_id       UUID NOT NULL,
  guest_id         UUID NOT NULL,

  total_price      DECIMAL(8,2),
  status           TEXT,  -- 'pending', 'confirmed', 'cancelled', 'completed'
  cancellation_reason TEXT,
  payment_id       TEXT,  -- Stripe payment intent ID
  created_at       TIMESTAMPTZ,
  confirmed_at     TIMESTAMPTZ
)
```

## Deep Dive 1: Preventing Double-Booking

**Critical requirement:** Two guests cannot book the same listing for overlapping dates.

**Challenge:** High concurrency. Multiple users browsing the same listing simultaneously → both try to book same dates.

**Approach A: Optimistic locking with database constraints**
```sql
-- Unique constraint prevents double-booking at DB level
CREATE UNIQUE INDEX ON availability (listing_id, date)
  WHERE status = 'booked';  -- only one 'booked' entry per listing+date

-- Booking transaction:
BEGIN TRANSACTION;

-- Check availability (SELECT FOR UPDATE locks rows)
SELECT date, status FROM availability
WHERE listing_id = ? AND date BETWEEN check_in AND check_out
FOR UPDATE;  ← row-level lock

-- If any date has status != 'available': ROLLBACK, return conflict

-- Reserve dates
UPDATE availability
SET status = 'booked', booking_id = ?
WHERE listing_id = ? AND date BETWEEN check_in AND check_out AND status = 'available';

-- Check rows updated = expected count (all dates were available)
IF rows_updated != expected_count: ROLLBACK, return conflict;

-- Create booking record
INSERT INTO bookings ...;

COMMIT;
```

**This works** because:
- `SELECT FOR UPDATE` acquires exclusive lock on availability rows
- Concurrent transaction for same dates will block until first commits/rollbacks
- Unique constraint is last-resort protection if logic has a bug

**Approach B: Distributed lock (for microservices)**
```
Acquire Redis lock: SET lock:listing:{id} {booking_id} NX EX 30
  → Only one booking attempt proceeds at a time per listing
  → All others retry or fail fast

Then perform DB transaction as above
Release lock after commit
```

**Recommendation:** DB-level transaction with `FOR UPDATE` is the most correct. Distributed lock adds redundant protection for multi-service architectures.

## Deep Dive 2: Search — Availability-Filtered Search

**Challenge:** Search for "2-bedroom in SF, July 4-8" requires:
1. Geographic filtering (lat/lng within SF bounding box)
2. Amenity filtering (2 bedrooms)
3. Availability filtering (all dates July 4-8 must be available)
4. Price filtering
5. Result ranking (by relevance, rating, price)

**Search architecture:**
```
Step 1: Elasticsearch geo + attribute filtering
  Query: must: { geo_bounding_box: SF bounds }, filter: { min_bedrooms: 2, amenities: [...], price range }
  → Returns listing_ids (fast, Elasticsearch handles geo + filtering)
  Result: 5,000 candidate listings

Step 2: Availability pre-filter (before expensive ranking)
  For each candidate listing_id:
    Check Redis: availability:{listing_id}:{check_in}:{check_out} → cached result
    Cache miss: query availability table for date range
    Filter out listings with any unavailable date
  Result: 2,000 available listings

Step 3: Rank and return top 20
  Score = relevance_score + price_competitiveness + host_response_rate + avg_rating
  Return top 20 with pagination cursor
```

**Availability caching:**
```
Redis hash per listing:
  Key: avail:{listing_id}
  Value: Hash of date → status (e.g., "2024-07-01" → "available", "2024-07-04" → "booked")
  TTL: 60 seconds

On booking: invalidate avail:{listing_id}
On host blocking date: invalidate avail:{listing_id}
```

**For popular listings with many concurrent searches:** Pre-compute "available date ranges" and cache as a set of available intervals. Much faster than per-date lookup for multi-week searches.

## Reliability and Operations

- **Booking with payment failure:** Two-phase: (1) reserve dates optimistically, (2) charge payment. If payment fails: release date reservation, notify guest. Timeout: reserved but unpaid bookings released after 10 minutes.
- **Listing sync with Elasticsearch:** Async sync via Kafka. On listing create/update → Kafka event → ES indexer. Elasticsearch may lag by 10-30s. For search: acceptable. For listing detail: read from PostgreSQL for accuracy.
- **Host goes offline during booking:** Booking confirmation is synchronous from guest perspective. Host has 24 hours to confirm if "Host approval required" is enabled; otherwise, auto-confirm for "Instant Book" listings.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| DB `SELECT FOR UPDATE` | Application-level lock | DB lock: atomic and correct; App lock: may have race conditions |
| Elasticsearch + availability filter | All in DB | ES: fast geo+attribute search; All DB: simpler but slower for complex geo queries |
| Redis availability cache | DB-only | Redis: fast for browse; DB: authoritative for booking |
| Date-level availability rows | Range-based booking table | Date-level: simple overlap check; Range: efficient for long bookings but complex overlap queries |

## L5 Additions

1. **Price recommendation engine:** Hosts often set suboptimal prices. Design: historical booking data per listing (occupancy rate, price, competitor prices in same area). ML model: given {date, location, property type, amenities, local events}, predict optimal price to maximize revenue (price × occupancy). Output: nightly price suggestions. Host can accept or override. Real-time signals: local event (concert, conference) spikes demand → suggest surge pricing. Model retrained weekly.

2. **Instant booking vs request-to-book:** Instant book: payment captured at booking, host auto-confirmed. Request-to-book: host has 24h to accept/decline, payment held but not captured. Architecture: Stripe "payment intent" with `capture_method: manual`. On host decline: cancel payment intent. On host accept: capture payment. Complex state machine: pending → host_accepted → payment_captured → confirmed. SAGA pattern if payment capture fails after host accepts.

3. **Multi-currency and taxes:** Guests see prices in local currency; hosts receive in their bank currency. Currency conversion at search time (display only) and at booking time (actual rate locked). VAT/GST calculation by jurisdiction: some countries require Airbnb to collect and remit tax. Architecture: tax service that takes (listing_id, booking_dates, guest_country) → tax amount and jurisdiction. Stored per booking for audit/remittance. Airbnb runs into different tax laws in 70+ countries.

4. **Superhost and trust signals:** Trust is the core business model. Design: host trust score = composite of: response rate, acceptance rate, cancellation rate, review score, hosting tenure, verified ID. Calculated nightly, stored in a features DB, joined at search time for ranking. Badge and filtering: "Superhost" badge shown in search results. Guest trust: ID verification, booking history. Fraudulent booking detection: new account + premium listing + unusual dates → hold for manual review.

5. **High-availability for peak travel periods:** Holiday weekends (July 4, Thanksgiving) → 10x normal traffic. Capacity planning: provision for 3x normal always-on, auto-scale to 10x within 5 minutes. Availability table is the bottleneck: many concurrent reads and writes. Mitigation: (a) read replicas for search availability checks, (b) leader for booking writes, (c) connection pooling (PgBouncer). Load testing: run chaos experiments 2 weeks before peak — inject latency into payment service, kill DB replica, simulate surge.
