# 11. Google Drive — File Storage and Sync

**Type:** storage
**Difficulty:** L4–L5
**Frequency:** Very High — file sync, chunking, conflict resolution

---

## Problem Statement

Design Google Drive. Users can upload, store, and sync files across devices. Multiple devices should stay in sync automatically. Users can share files with others (view/edit permissions). The system must handle 1B users, petabytes of storage, and file changes propagating to all synced devices within 5 seconds.

---

## Functional Requirements

**Must have:**
- Upload files (up to 50GB per file)
- Download files
- Sync files across multiple devices: file changes on one device appear on others within 5 seconds
- Folder organization
- Share files with others (view-only or edit permission)

**Should have:**
- Version history (last 30 versions)
- Resume interrupted uploads
- Conflict resolution when two devices edit the same file simultaneously
- Offline support: edit locally, sync when connected

**Out of scope:**
- Real-time collaborative editing (Google Docs — see separate design)
- Full-text search within documents
- OCR and image search

---

## Non-Functional Requirements

- **Scale:** 1B users, 15TB avg per user (free tier 15GB; paying users more) — total 15EB storage
- **Availability:** 99.99% for file operations
- **Sync latency:** File change visible on other devices within 5 seconds
- **Durability:** 99.999999999% (11 nines) — no data loss ever
- **Bandwidth optimization:** Only sync changed parts of a file, not the whole file

---

## Capacity Estimation

```
Storage: 1B users × 15GB avg = 15EB raw; with 3x replication = 45EB physical
New uploads: assume 0.1% of users upload 1 file/day = 1M files/day
  Avg file: 1MB → 1TB/day new data
Large file uploads (videos): 1% × 1GB = 10M GB = 10TB/day
Total: ~11TB/day

Sync events: 100M active files modified/day = 1,150/sec change notifications
Metadata queries: 100M DAU × 10 queries/day = 1,150 QPS metadata
```

## High-Level Architecture

```
[Client (Desktop/Mobile)] ──→ [Sync Client Library]
                                      │
                 ┌────────────────────┼────────────────────┐
                 │                    │                    │
          [Upload API]        [Download API]        [Notification Service]
                 │                    │                         │
        [Chunk Store (S3)]   [CDN/S3]              [WebSocket / Long-poll]
                 │
        [Metadata Service]
                 │
        [Metadata DB (PostgreSQL)]
        [File Index (Elasticsearch)]
```

**Components:**
- **Sync Client Library:** Watches local file system. Detects changes, chunks files, uploads delta chunks, applies remote changes.
- **Upload API:** Receives chunks, assembles into complete file, writes to S3. Supports resume via chunk-addressed upload.
- **Metadata Service:** Stores file tree (folders, files, versions, permissions). Single source of truth.
- **Chunk Store (S3):** Content-addressed blob storage. Chunks identified by SHA-256 hash.
- **Notification Service:** Pushes file change events to all client devices of a user.
- **CDN:** Accelerates downloads for popular shared files.

## API Design

```
# Upload (chunked)
POST /v1/files/upload/init
Body: { filename, size, parent_folder_id, client_modified_at }
Returns: { upload_id, chunk_size: 4MB }

PUT /v1/files/upload/{upload_id}/chunk/{chunk_index}
Body: raw bytes (4MB max)
Returns: { received: bool }

POST /v1/files/upload/{upload_id}/complete
Body: { chunk_hashes: string[] }  -- SHA-256 of each chunk
Returns: { file_id, version_id }

# Download
GET /v1/files/{id}/content → redirect to S3 presigned URL or CDN URL
GET /v1/files/{id}/versions → list of version objects
GET /v1/files/{id}/versions/{version_id}/content → download specific version

# Metadata
GET /v1/folders/{id}?include_children=true
POST /v1/folders  Body: { name, parent_id }
PUT /v1/files/{id}  Body: { name?, parent_id?, permissions? }
DELETE /v1/files/{id}  → moves to trash (30-day retention)

# Sync
GET /v1/sync/delta?cursor=<cursor>  → list of changes since cursor
POST /v1/sync/subscribe  → get WebSocket URL for real-time change events
```

## Data Model

-- PostgreSQL: file/folder metadata
nodes (
  id           UUID PRIMARY KEY,
  parent_id    UUID REFERENCES nodes(id),
  owner_id     UUID NOT NULL,
  name         TEXT NOT NULL,
  type         TEXT,  -- 'file' | 'folder'
  size_bytes   BIGINT,

  created_at   TIMESTAMPTZ,
  modified_at  TIMESTAMPTZ,
  trashed_at   TIMESTAMPTZ,

  UNIQUE (parent_id, owner_id, name)  -- no duplicate names in same folder
)

-- Version history
versions (
  id          UUID PRIMARY KEY,
  file_id     UUID NOT NULL,
  version_num INT,
  size_bytes  BIGINT,
  content_hash TEXT,

  granted_at   TIMESTAMPTZ
)

## Deep Dive 1: File Chunking and Delta Sync

**Why chunking?**
- Large files can't be uploaded atomically (network interruptions)
- When a file changes, only upload the changed chunks (delta sync)
- Deduplication: identical chunks across different files stored once

**Chunking algorithm:**
```
Fixed-size chunking: split every 4MB
  Simple, but poor delta performance (insert 1 byte shifts all chunk boundaries)

Content-defined chunking (CDC) — better:
  Use rolling hash (Rabin fingerprint) to find chunk boundaries
  Split when hash(window) % target_size == 0
  Chunk sizes: 4KB–16MB, avg ~4MB
  Insert 1 byte near start: only 1-2 chunks change; rest unchanged
  Used by rsync, LBFS, Dropbox
```

**Delta sync for large file modification:**
```
File: 1GB video, user edits first 5 minutes

1. Client re-chunks modified file using CDC
2. New chunk set: 240 chunks total
3. Most chunks are unchanged (same hash as before)
4. Changed chunks: ~5 (only the edited portion)
5. Client checks which chunk hashes already exist on server (S3 HEAD requests, or diff with stored chunk list)
6. Upload only 5 new chunks (not 1GB)
7. Server updates version record with new chunk list
8. Other devices download only the 5 changed chunks
```

## Deep Dive 2: Conflict Resolution

**Scenario:** User edits file on laptop (offline), edits same file on phone, both come online.

**Conflict detection:**
```
Client tracks:
  - base_version: the last server version this edit was based on
  - client_modified_at: local timestamp of change

On sync:
  Upload client's changes with base_version=V5
  Server: if current version != V5 → conflict detected
```

**Resolution strategies:**

**Strategy A: Last-write-wins (LWW)**
- Simpler: whoever synced last wins
- Loser's changes are lost silently
- Acceptable for casual use; bad for important files

**Strategy B: Fork (Google Drive's approach for documents)**
- Keep both versions
- Rename one: "filename (1).docx"
- User sees both, manually merges
- Simple to implement, preserves all data
- User must manually resolve

**Strategy C: 3-way merge (for text files)**
```
Base (V5) + Local Changes + Remote Changes → attempt auto-merge
Merge algorithm (similar to git merge):
  - Lines changed only in local → take local
  - Lines changed only in remote → take remote
  - Lines changed in both → conflict marker (<<<< / ====  / >>>>)
Auto-merge success: 80% of cases
Conflict: user must edit and resolve
```

**Recommended:** Implement LWW by default for binary files, 3-way merge for text files, always preserve both versions in history.

## Reliability and Operations

- **Upload failure:** Client retries individual failed chunks. Server uses chunk hash to verify integrity. Incomplete upload expires after 24h.
- **Sync loop prevention:** Each client has a unique device_id. When server broadcasts a change notification, include the originating device_id. Each device ignores changes it originated (to prevent echo loop).
- **Quota enforcement:** Check available quota before accepting upload. Quota stored in user profile, decremented transactionally with file creation.
- **Storage tiering:** Files not accessed in 1 year → move to S3 Glacier (1/10th cost). Transparent to user: on access, restore from Glacier (async, 1-12 hours; show "Restoring..." status).

---

## Security and Abuse

- Authentication: OAuth2 per device; device-level token revocation
- Encryption at rest: AES-256 per file (envelope encryption with per-file key encrypted with master key in KMS)
- Zero-knowledge option: client-side encryption before upload (Google doesn't have your key)
- Virus scanning: async scan of uploads using ClamAV + commercial scanner before marking as shareable
- Access log: audit trail of who accessed/modified a file (GDPR-required)
- Sharing controls: link sharing can be disabled by org admin (Workspace customers)

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| CDC chunking | Fixed-size chunking | CDC: better delta sync; Fixed: simpler implementation |
| Content-addressed storage | Path-based storage | CAS: automatic dedup; Path-based: easier to navigate |
| Fork conflict resolution | Auto-merge | Fork: never loses data; Auto-merge: seamless but may silently combine changes |
| S3 for chunks | Custom block store | S3: managed durability; Custom: lower latency, full control |

## L5 Additions

1. **Global deduplication:** SHA-256 content-addressed chunks enable cross-user dedup. If 1M users have the same PDF, store it once. Naive approach: check if chunk hash exists before upload. Privacy concern: reveals whether someone else has the same file ("convergent encryption" attack). Solution: encrypt each chunk with `key = H(chunk_content || user_secret)` — dedup works within a user's files but not across users. For enterprise: org-level dedup with shared namespace.

2. **Selective sync and virtual filesystem:** Client doesn't need all files locally. Virtual filesystem driver (Windows: WinFSP; macOS: FUSE; Linux: kernel module) makes all files appear as local but are actually "placeholders". On access: transparent download. On offline: cached files available; others show "unavailable". This is how OneDrive's "Files On-Demand" and macOS iCloud Drive work. The sync client maintains a local SQLite DB of placeholder metadata for fast `ls` operations without downloading content.

3. **Team Drives / Shared Drives:** Files owned by the team, not an individual. When someone leaves, files remain. Quota charged to team, not individual. Architecture change: `owner_id` in nodes table refers to `entity_id` which can be user or team. Permission inheritance: folder permissions propagate to children (but can be overridden). Efficient permission lookup: materialized permissions cache (Redis) per node per user — rebuilt on permission change via graph traversal.

4. **Real-time sync performance at scale:** 100M sync events/day = 1,150/sec average but bursty. Architecture: per-user notification channel (Redis pub/sub or Kafka topic per user). When file changes: write to metadata DB → publish to user's notification channel → all that user's connected clients receive the event. For users with 10 devices, fan-out = 10. Total fan-out: 100M events × avg 3 devices = 300M deliveries/day = 3,500/sec — manageable.

5. **Cross-region consistency:** User accesses Drive from multiple regions (USA in morning, EU in afternoon via VPN). File metadata must be consistent. Use CockroachDB or Spanner for metadata: globally replicated, strongly consistent. Chunks in S3: replicate to relevant regions on access (S3 replication rules). Files modified in EU region: metadata writes go to global primary (Spanner), chunks stored in EU-region bucket, replicated to US-region on first access from US. Cache TTL = 5 seconds for metadata (matches sync latency SLA).
