# 11. Google Drive — File Storage and Sync

**Type:** storage
**Difficulty:** G4–G5
**Frequency:** Very High — file sync, chunking, conflict resolution

---

## Problem Statement

Design Google Drive. Users can upload, store, and sync files across devices. Multiple devices should stay in sync automatically. Users can share files with others (view/edit permissions). The system must handle 1B users, petabytes of storage, and file changes propagating to all synced devices within 5 seconds.

---

## Functional Requirements

**Must have:**
- Upload files (up to 50GB per file)
- Download files
- Sync files across multiple devices: file changes on one device appear on others within 5 seconds
- Folder organization
- Share files with others (view-only or edit permission)

**Should have:**
- Version history (last 30 versions)
- Resume interrupted uploads
- Conflict resolution when two devices edit the same file simultaneously
- Offline support: edit locally, sync when connected

**Out of scope:**
- Real-time collaborative editing (Google Docs — see separate design)
- Full-text search within documents
- OCR and image search

---

## Non-Functional Requirements

- **Scale:** 1B users, 15TB avg per user (free tier 15GB; paying users more) — total 15EB storage
- **Availability:** 99.99% for file operations
- **Sync latency:** File change visible on other devices within 5 seconds
- **Durability:** 99.999999999% (11 nines) — no data loss ever
- **Bandwidth optimization:** Only sync changed parts of a file, not the whole file

## Capacity Estimation

```
Users: 1B total; 100M DAU
Storage: 1B users × 15GB avg = 15EB raw; with 3x replication = 45EB physical
  Physical storage across ~4.5M drives at 10TB each

New uploads (daily):
  0.1% of users upload 1 file/day = 1M files/day
  Avg file size: 1MB → 1TB/day new data
  Large file uploads (videos, 1% of uploads, avg 1GB): 10TB/day
  Total: ~11TB/day new ingestion

Metadata:
  1B files total × 1KB metadata per file = 1TB metadata
  Metadata QPS: 100M DAU × 10 queries/day = 1B/day = 11,600/sec avg; peak 50K/sec

Sync events:
  100M files modified/day = 1,157/sec change notifications
  Each notification fans out to avg 3 devices per user = 3,500 deliveries/sec

Chunk dedup savings (rough estimate):
  30% of uploaded content is duplicate (popular software, common docs)
  Effective storage reduction: 0.7x on new uploads
```

## High-Level Architecture

```
[Client (Desktop/Mobile)] ──→ [Sync Client Library]
                                      │
                 ┌────────────────────┼────────────────────┐
                 │                    │                    │
          [Upload API]        [Download API]        [Notification Service]
                 │                    │                         │
        [Chunk Store (S3)]   [CDN/S3]              [WebSocket / Long-poll]
                 │
        [Metadata Service]
                 │
        [Metadata DB (PostgreSQL / Spanner)]
        [File Index (Elasticsearch)]
```

**Components:**
- **Sync Client Library:** Watches local file system. Detects changes, chunks files, uploads delta chunks, applies remote changes. Maintains a local SQLite database of file hashes and sync state.
- **Upload API:** Receives chunks, assembles into complete file, writes to S3. Supports resume via chunk-addressed upload. Verifies chunk integrity by comparing SHA-256 hash.
- **Metadata Service:** Stores file tree (folders, files, versions, permissions). Single source of truth for file structure.
- **Chunk Store (S3):** Content-addressed blob storage. Chunks identified by SHA-256 hash — the hash IS the key. Identical chunks across files stored once.
- **Notification Service:** Pushes file change events to all client devices of a user. Uses Redis pub/sub per-user channel; WebSocket for connected clients, APNs/FCM for mobile push.
- **CDN:** Accelerates downloads for popular shared files. Files accessed by many users are automatically promoted to CDN edge nodes.

## API Design

```
# Upload (chunked)
POST /v1/files/upload/init
Body: { filename, size, parent_folder_id, client_modified_at, total_chunks }
Returns: { upload_id, chunk_size: 4MB, existing_chunks: string[] }
  # existing_chunks: server reports which SHA-256 hashes it already has
  # Client skips uploading chunks the server already has (dedup)

PUT /v1/files/upload/{upload_id}/chunk/{chunk_index}
Headers: Content-SHA256: <sha256_of_chunk>
Body: raw bytes (4MB max)
Returns: { received: bool, integrity_ok: bool }

POST /v1/files/upload/{upload_id}/complete
Body: { chunk_hashes: string[] }  -- SHA-256 of each chunk in order
Returns: { file_id, version_id }

# Download
GET /v1/files/{id}/content → redirect to S3 presigned URL or CDN URL
GET /v1/files/{id}/versions → list of version objects
GET /v1/files/{id}/versions/{version_id}/content → download specific version

# Metadata
GET /v1/folders/{id}?include_children=true
POST /v1/folders  Body: { name, parent_id }
PUT /v1/files/{id}  Body: { name?, parent_id?, permissions? }
DELETE /v1/files/{id}  → moves to trash (30-day retention)

# Sync
GET /v1/sync/delta?cursor=<cursor>
  → Returns list of changes since cursor; cursor is a server-assigned opaque token
  → Response: { changes: [{ file_id, event: 'created'|'modified'|'deleted', version_id, ... }], next_cursor }
POST /v1/sync/subscribe  → get WebSocket URL for real-time change events
```

## Data Model

```sql
-- PostgreSQL: file/folder metadata (sharded by owner_id)
nodes (
  id           UUID PRIMARY KEY,
  parent_id    UUID REFERENCES nodes(id),
  owner_id     UUID NOT NULL,
  name         TEXT NOT NULL,
  type         TEXT,  -- 'file' | 'folder'
  size_bytes   BIGINT,
  content_hash TEXT,  -- SHA-256 of full file (for dedup detection)
  created_at   TIMESTAMPTZ,
  modified_at  TIMESTAMPTZ,
  trashed_at   TIMESTAMPTZ,
  version_num  INT DEFAULT 1,

  UNIQUE (parent_id, owner_id, name)  -- no duplicate names in same folder
)

-- Index for sync delta queries: all files modified since a given time
CREATE INDEX nodes_modified ON nodes (owner_id, modified_at DESC);

-- Version history: last 30 versions kept
versions (
  id          UUID PRIMARY KEY,
  file_id     UUID NOT NULL REFERENCES nodes(id),
  version_num INT,
  size_bytes  BIGINT,
  content_hash TEXT,
  chunk_ids   JSONB,   -- ["sha256_1", "sha256_2", ...] ordered list of chunk hashes
  created_at  TIMESTAMPTZ,
  created_by  UUID,    -- user or device that created this version
  device_id   UUID     -- which device uploaded this version (for sync loop prevention)
)

-- Chunks: content-addressed (S3 key = SHA-256 hash)
-- No DB table needed for chunks: S3 key IS the chunk_id
-- Client verifies by calling PUT /upload/{id}/init → server checks S3 for each hash
-- exists? YES → skip upload (dedup across user's files)
-- exists? NO → upload chunk

-- Permissions
permissions (
  resource_id  UUID,  -- file_id or folder_id
  grantee_id   UUID,  -- user_id or group_id
  access_level TEXT,  -- 'viewer' | 'editor' | 'owner'
  granted_at   TIMESTAMPTZ,
  PRIMARY KEY (resource_id, grantee_id)
)

-- Sync cursors: track per-device sync position
sync_cursors (
  user_id    UUID,
  device_id  UUID,
  cursor     BIGINT,  -- monotonically increasing sequence number
  updated_at TIMESTAMPTZ,
  PRIMARY KEY (user_id, device_id)
)
```

## Deep Dive 1: File Chunking and Delta Sync

**Why chunking?**
- Large files cannot be uploaded atomically — network interruptions would require restarting the entire upload
- When a file changes, only upload the changed chunks (delta sync), not the entire file
- Deduplication: identical chunks across different files stored only once, saving storage cost

**Chunking algorithm — two approaches:**

```
Fixed-size chunking: split every 4MB
  Simple to implement
  Poor delta performance: inserting 1 byte at position 0 shifts ALL chunk boundaries
  Every chunk after the insertion point becomes "new" → must re-upload the entire file

Content-defined chunking (CDC) — Rabin fingerprinting:
  Compute a rolling hash over a sliding window of bytes
  Split boundary when: hash(window) % target_chunk_size == 0
  Minimum chunk: 512KB; Maximum chunk: 16MB; Average: ~4MB
  Insert 1 byte near start: 1-2 chunks near the insertion point change
  All chunks after the edit region: same hash as before → NOT re-uploaded
  Result: 99% of a 1GB file edit affecting only a small region = only ~5MB re-uploaded
  Used by: rsync, LBFS, Dropbox, Google Drive
```

**Delta sync walkthrough for a large file modification:**
```
Scenario: 1GB video file, user edits the first 5 minutes
Total chunks at avg 4MB: 1024MB / 4MB = 256 chunks

1. Client re-chunks modified file using CDC
2. New chunk list: 256 chunks total
3. Client computes SHA-256 of each chunk
4. Client sends chunk hashes to server (POST /upload/init → existing_chunks response)
5. Server compares with stored chunk list from previous version
6. Changed chunks: ~5 (only the edited portion's boundaries shifted)
7. Unchanged chunks: 251 — server already has these
8. Client uploads only the 5 new chunks (~20MB, not 1GB)
9. Server creates new version record with updated chunk list
10. Other devices receive delta notification, download only the 5 changed chunks

Bandwidth saved: 1GB → 20MB = 98% reduction
```

**S3 storage layout:**
```
Chunk key: chunks/{sha256_hash[0:2]}/{sha256_hash[2:4]}/{sha256_hash}
  Example: chunks/a3/f7/a3f7b9c12d...
  Path prefix sharding avoids hot-key issues on S3 LIST operations

File assembly: not needed at rest; client assembles chunks on download
  Client downloads only chunks it doesn't already have locally (local cache)
```

**G5:** Would note that content-defined chunking (CDC) boundary selection via Rabin fingerprinting has a tunable min/max chunk size that requires careful calibration for the specific workload. Too small (< 512KB average): too many chunks, metadata overhead per file dominates, and S3 HEAD requests for dedup checks become the bottleneck. Too large (> 16MB average): delta sync effectiveness drops sharply — a 1-byte change in a 16MB chunk forces re-uploading 16MB. Optimal for cloud storage is typically 4–8MB average. Additionally, chunk deduplication across users requires careful security isolation: chunks must be encrypted with per-user keys even if content is identical, or you create a cross-user content inference attack (an attacker can determine whether another user has a specific file by observing whether a chunk upload is skipped as already-present — a form of "convergent encryption" side channel). The correct mitigation is to use per-user S3 key prefixes with separate KMS data keys, accepting storage cost of re-storing identical content for different users.

## Deep Dive 2: Conflict Resolution

**Scenario:** User edits file on laptop (offline), edits same file on phone, both come online simultaneously.

**Conflict detection:**
```
Each sync client tracks:
  - base_version: the version_id this edit was based on (last synced version)
  - client_modified_at: local timestamp of when the change was made
  - device_id: which device made the change

Upload request includes:
  { upload_id, base_version_id: "v5", device_id: "laptop-uuid" }

Server: if current_version_id != base_version_id → conflict detected
  The file was modified by another device since this client last synced
```

**Resolution strategies:**

**Strategy A: Last-write-wins (LWW)**
```
Whoever synced last wins; the other's changes are silently overwritten
Ordering: use client_modified_at timestamp (clock skew risk) or server receive time
Simple to implement; acceptable for casual/non-critical files
Risk: silent data loss — the losing client's changes vanish with no notification
```

**Strategy B: Fork — Google Drive's approach for opaque binary files**
```
Keep both conflicting versions in the user's Drive
Rename the conflicting copy: "filename (Bob's conflicting copy 2024-07-01).docx"
User sees both files, manually merges or discards
Guarantees: no data is ever lost silently
Downside: user must manually resolve; can accumulate many conflict copies over time
Best for: binary files (images, PDFs, videos) where auto-merge is not possible
```

**Strategy C: 3-way merge — for text and structured files**
```
Inputs:
  Base version (v5): the last common ancestor
  Local changes: client A's edits
  Remote changes: client B's edits (already on server as v6)

Algorithm (similar to git merge):
  Lines changed only in local → take local version
  Lines changed only in remote → take remote version
  Lines changed in both → conflict marker inserted:
    <<<< LOCAL
    my version of this line
    ====
    their version of this line
    >>>> REMOTE
  User must open the file and resolve marked conflicts

Auto-merge success rate: ~80% of concurrent edits (edits in different regions of the file)
Conflict requiring manual resolution: ~20% (edits to same line/region)
```

**Recommended implementation:**
- Binary files (images, videos, executables): Fork strategy — never risk corrupting binary data
- Text files (.txt, .md, code): Attempt 3-way merge; fall back to fork if conflict markers exist
- Google Docs/Sheets: OT (operational transformation) — real-time collaborative editing, separate design
- Always preserve all versions in version history regardless of strategy

**G5:** Would push beyond simple 3-way merge and flag the vector clock problem in a multi-device sync system. With more than 2 devices, determining the true "last common ancestor" requires vector clocks or a CRDTs (conflict-free replicated data types) approach rather than a simple base_version comparison. If device A syncs to v5, device B forks from v4 (not v5), and both upload changes simultaneously, the "base version" for 3-way merge is ambiguous. Production systems like Dropbox and Google Drive maintain a per-device logical clock (Lamport timestamp) associated with each version to correctly identify the causal ancestor even in multi-device scenarios. For the subset of files that are JSON or YAML structures (config files, design tokens), a structure-aware merge that operates on the parsed tree rather than raw lines reduces spurious conflicts by 60–70% compared to line-based diff.

## Reliability and Operations

- **Upload failure recovery:** Client retries individual failed chunks. Server uses SHA-256 hash sent in Content-SHA256 header to verify integrity on receipt. Chunk uploads are idempotent — re-uploading the same chunk hash is a no-op. Incomplete uploads (upload_id not completed) expire after 24 hours; a background job reclaims storage.
- **Sync loop prevention:** Each client has a unique device_id registered at login. When the server broadcasts a change notification, the event includes the originating device_id. Each device ignores change events that it originated (to prevent echoing its own upload back as a change to apply). The Notification Service tags each event: `{ file_id, version_id, origin_device_id }`.
- **Quota enforcement:** Check available quota atomically before accepting upload. Quota stored in user profile table, updated with a conditional UPDATE: `SET used_bytes = used_bytes + chunk_size WHERE used_bytes + chunk_size <= quota_bytes`. If constraint fails, return 507 Insufficient Storage. Quota enforcement is rate-limited per user to prevent quota check storms.
- **Storage tiering:** Files not accessed in 12 months → automatically migrated to S3 Glacier Instant Retrieval (cost: ~$0.004/GB/month vs $0.023/GB/month for S3 Standard). On user access: transparent restore. Files in Deep Archive (cold for 3+ years): restore takes 12 hours; show "Restoring..." status in Drive UI. User can proactively "pin" important files to Standard tier.
- **Metadata DB replication:** PostgreSQL with synchronous replication to at least 1 replica before acknowledging writes. Read replicas serve metadata reads (list folder, sync delta queries). For global scale, Spanner provides globally consistent metadata without cross-region read staleness.

---

## Security and Abuse

- **Authentication:** OAuth2 per device with device-level token revocation. Compromise one device? Revoke its token from Drive security settings without affecting other devices.
- **Encryption at rest:** AES-256 per file using envelope encryption. Per-file data encryption key (DEK) stored encrypted with a master key in Google Cloud KMS. DEK is stored in the file metadata row. Rotation: KMS master key rotation triggers re-encryption of all DEKs (background job, no user impact).
- **Zero-knowledge option (Workspace):** Client-side encryption (CSE) — user's encryption keys held by a third-party key management provider, not Google. Google stores only ciphertext. Trade-off: full-text search, thumbnail previews, and collaboration features are unavailable for CSE files.
- **Virus scanning:** Async ClamAV + commercial scanner (e.g., VirusTotal API) triggered after upload. File marked `scan_status: pending` until complete. Sharing is blocked until scan clears. Known-bad content hash → immediate block via SHA-256 blocklist.
- **Access log:** Audit trail of who accessed/modified a file stored in an append-only log (BigTable). Accessible via Admin Console for Workspace customers. GDPR Article 30 requires this for enterprise accounts.
- **Sharing controls:** Link sharing can be globally disabled by org admin. Workspace orgs can restrict external sharing to specific domains.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| CDC chunking | Fixed-size chunking | CDC: superior delta sync (only changed chunks re-uploaded); Fixed: simpler to implement but poor performance for large file edits |
| Content-addressed storage (SHA-256 as key) | Path-based storage | CAS: automatic dedup within user's files, integrity verification built in; Path-based: intuitive navigation, easier debugging |
| Fork conflict resolution for binary files | LWW | Fork: zero data loss, user sees both versions; LWW: seamless UX but silently discards one version |
| S3 for chunks | Custom distributed block store | S3: managed 11-nines durability, operational simplicity; Custom: potentially lower latency and egress cost but years of operational work |
| PostgreSQL for metadata | Cassandra | PostgreSQL: ACID transactions for rename/move operations; Cassandra: better write throughput but no multi-row transactions |

## G5

1. **Global deduplication with security isolation:** SHA-256 content-addressed chunks enable cross-user deduplication. If 1M users upload the same ISO file, storing it once saves 99.9999% of storage cost. The naive implementation — check if chunk hash exists before upload, skip if found — creates a cross-user information leak known as the "convergent encryption" attack: an adversary uploads a file and observes whether the upload is skipped (the chunk already exists), revealing that another user has the identical content. Mitigation: dedup only within a single user's namespace, or for enterprise Workspace accounts, within an organization's namespace. Per-user S3 key prefix with separate KMS encryption keys ensures physical isolation even if two users have identical content. The storage cost trade-off is accepted for the security guarantee.

2. **Selective sync and virtual filesystem (Files On-Demand):** A user with 15GB in Drive and a laptop with a 256GB SSD cannot sync everything locally. Virtual filesystem drivers (Windows: WinFSP or Cloud Files API; macOS: FileProvider framework; Linux: FUSE via libfuse) present all Drive files as if they are local but store only metadata. On file open: transparent background download. On offline access: only files explicitly "pinned" (made available offline) are accessible; others show as unavailable with a cloud icon. The sync client maintains a local SQLite database of all file metadata (filenames, sizes, modification times, cloud status) enabling fast `ls -la` and file search operations without downloading any file content. Placeholder files on disk show the correct file size while being 0 bytes actual storage on the local disk.

3. **Team Drives / Shared Drives:** Files owned by the team entity, not an individual user. When an employee leaves, files remain. Quota charged to team, not individual. Architecture change: `owner_id` in nodes table refers to an `entity_id` which can be either user or team (resolved via an entity type discriminator column). Permission inheritance: folder-level permissions propagate to all children automatically, but individual items can override. Efficient permission check for a deeply nested file: materialized permissions cache in Redis, keyed by `(node_id, user_id)`, value is the effective access_level. Cache is invalidated (or rebuilt via graph traversal) whenever a permission changes anywhere in the ancestor chain. This replaces a recursive SQL query up the folder tree on every file access.

4. **Real-time sync performance at scale:** 100M file modifications/day = 1,157/sec average, but highly bursty (morning when users arrive at work, after large sync events). Architecture: per-user notification channel implemented as a Redis pub/sub channel or a Kafka topic per user partition. On file change: write to metadata DB → publish to `notifications:{user_id}` channel → all of that user's connected WebSocket clients receive the event simultaneously. Fan-out is bounded by the number of devices a user has (~3 on average). Total deliveries: 100M events × avg 3 devices = 300M/day = 3,472/sec — comfortably handled by a cluster of WebSocket gateway servers. Each WebSocket server subscribes to Redis pub/sub for the users currently connected to it.

5. **Cross-region consistency:** A user who works from New York in the morning and accesses Drive via a remote desktop in London in the afternoon expects to see the same files without staleness surprises. Using Google Spanner for metadata provides external consistency (linearizability) across all regions — a read in any region reflects all commits globally. For chunk data: S3 Cross-Region Replication copies chunks to the user's "home region" asynchronously. First access from a new region triggers on-demand replication of frequently accessed chunks to that region's S3 bucket, with subsequent reads served locally. Metadata write latency is ~50ms higher with Spanner's two-phase commit, but this is invisible to the user relative to the 5-second sync SLA.
