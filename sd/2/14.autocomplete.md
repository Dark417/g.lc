# 14. Search Autocomplete (Typeahead)

**Type:** search
**Difficulty:** L4
**Frequency:** Very High — data structures + scale combination

---

## Problem Statement

Design a search autocomplete system like Google's search suggestions. As users type, return the top 5 most relevant completions for the current prefix in real-time (< 100ms). Suggestions should be ranked by popularity (frequency of searches). The system must handle 10M QPS at peak (global Google scale).

---

## Functional Requirements

**Must have:**
- Return top 5 completions for a given prefix string
- Ranked by search frequency (most popular suggestions first)
- Latency < 100ms p99
- Support Unicode / multi-language

**Should have:**
- Personalization: factor in user's own recent searches
- Trending boost: searches spiking in last 24h ranked higher
- Spell correction: "gogle" → "google"

**Out of scope:**
- Full semantic query understanding
- Question answering ("what is...")
- Voice input processing

---

## Non-Functional Requirements

- **Scale:** 10M QPS globally (Google-scale); 1M QPS for typical product
- **Latency:** < 100ms p99 (including network from nearest PoP)
- **Freshness:** New trending queries should appear in suggestions within 1 hour
- **Availability:** 99.99% — degraded autocomplete is acceptable (empty suggestions), not unavailability

---

## Capacity Estimation

```
QPS: 10M queries/sec globally
  Per character typed: one request → user types 5 chars → 5 requests
  Actual search submissions: 10M / 5 = 2M actual searches/sec

Prefix index size:
  Unique queries: 1B (rough estimate for Google)
  Avg query length: 20 chars
  Trie nodes: ~O(chars × unique_queries_sharing_prefix) — hard to estimate directly
  Simpler: top 10M queries × 20 prefixes = 200M prefix entries
  Per entry: prefix(50B) + top5_results(200B) = 250B
  Total: 200M × 250B = 50GB → fits in Redis Cluster

Top-K queries: maintain top 100K queries → index is manageable
```

## High-Level Architecture

```
[Client Browser / App]
  │  (keystroke debounce: 50ms)
  │
  ▼
[CDN Edge] ──→ (cache miss) ──→ [Autocomplete API]
  │                                      │
  └─ (cache hit: top common prefixes)    ├──→ [Prefix Index (Redis)]
                                         │
                                         └──→ [Personalization Service]
                                                      │
                                              [User History (Redis)]

[Background pipeline]:
[Search logs (Kafka)] → [Aggregator (Flink)] → [Frequency Store] → [Trie Builder] → [Redis]
```

**Components:**
- **Autocomplete API:** Takes prefix, returns top 5 results from prefix index, optionally merges with personalization.
- **Prefix Index (Redis):** Pre-computed map of prefix → top 5 completions with scores. Updated periodically.
- **Background pipeline:** Counts query frequencies from real search logs, builds/updates the index.


## Data Model

**Trie (conceptual) vs Prefix Hash (practical):**

**Option A: Trie**
```
Trie stores all prefixes:
  "go" → { children: { "o": ... }, top5: ["google", "go fund me", ...] }
  "goo" → { top5: ["google", "good morning", ...] }

Space-optimized: compressed trie (Patricia/radix tree)
  "google" → one node "google", not 6 separate "g", "o", "o", "g", "l", "e" nodes
```

**Option B: Prefix Hash Table (chosen for distributed systems)**
```
Redis Hash:
  Key: autocomplete:en:{prefix}
  Value: JSON list of top 5 [{text, score}, ...]

For prefix "goo":
  HGET autocomplete:en goo → ["google", "good morning", "goodbye", "google maps", "goodreads"]

Advantages:
  - Simple O(1) lookup
  - Easy to update individual prefixes
  - No complex trie traversal to distribute

Disadvantages:
  - Stores all prefixes explicitly (vs trie sharing nodes)
  - No graceful fallback for unknown prefix (trie can walk up to parent)
```

**Practical approach:** Use Redis sorted sets for per-prefix ranking:
```
Key: ac:{lang}:{prefix}
Type: Sorted Set (score = query frequency, member = query string)
  ZADD ac:en:go 98521 "google"
  ZADD ac:en:go 87432 "google translate"
  ZADD ac:en:goo 98521 "google"
  ...

Lookup:
  ZREVRANGE ac:en:goo 0 4 WITHSCORES → top 5 completions for prefix "goo"

Update (when new frequency data arrives):
  ZADD ac:en:goo 99000 "google"  (update score)
  ZREMRANGEBYRANK ac:en:goo 0 -(6)  (keep only top 5 to save memory)
```

## Deep Dive 1: Frequency Aggregation Pipeline

**How do we know which queries are popular?**

```
1. Every search request → log to Kafka ("query-events" topic)
2. Flink aggregation (1-hour tumbling window):
   - Count occurrences of each query
   - Emit: { query: "google maps", count: 453221, window_end: ... }
3. Frequency store (Redis):
   - Running weighted average: new_score = 0.7 × old_score + 0.3 × hourly_count
   - (Decay old counts, boost recent counts)
4. Prefix index updater:
   - For each updated query, update all its prefix entries
   - "google maps" updates: "g", "go", "goo", "goog", "googl", "google", "google ", "google m", "google ma", "google map", "google maps"
   - 11 prefixes to update per query
   - 10M top queries × 11 prefixes = 110M key updates per batch → run as background job
```

**Trending detection:**
```
Trending score = current_hour_count / avg_count_last_7_days_same_hour
If trending_score > 5x → mark as "trending" + apply 2x score boost
Use bloom filter to check "is this a newly viral query" efficiently
```

## Deep Dive 2: Scale — 10M QPS

**CDN caching:**
```
Observation: top 10K prefixes cover 80% of all autocomplete requests
  "g", "go", "goo", "the", "how", ...

CDN edge caches: GET /v1/autocomplete?q=goo
  Cache-Control: max-age=60 (refresh every minute)
  This alone handles 80% of traffic without hitting origin

For the remaining 20% (long-tail prefixes): hits Autocomplete API clusters
  10M QPS × 20% = 2M QPS to origin → 2000 servers at 1K QPS each
  → Manageable with 100 servers in Redis cluster behind 500 API servers
```

**Regional deployment:**
```
Autocomplete API in every region (US, EU, APAC, etc.)
  Language/locale routing: en → US/EU servers; zh → APAC servers
  Each region has its own Redis prefix cache
  Global frequency data aggregated centrally → pushed to regional Redis hourly
  Accepts slight staleness (regional data lags up to 1 hour vs global)
```

---

## Reliability and Operations

- **Stale suggestions:** If Redis is down, return empty suggestions (fail open). Never block the user's search.
- **Index rebuild:** Scheduled nightly full rebuild (10M queries × all prefixes). Run as batch MapReduce job. Write to staging Redis → atomic swap to production Redis slot.
- **Prefix not in index:** For novel long prefixes, fall back to simple "contains" DB query as rare-path fallback.
- **Personalization failure:** If personalization service is slow, return global suggestions only (timeout = 20ms).

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis hash per prefix | Trie in memory | Redis: distributed, ops-friendly; Trie: memory-efficient, complex to distribute |
| CDN for common prefixes | All requests to origin | CDN: dramatically reduces origin load; but 60s staleness for trending updates |
| Hourly index updates | Real-time updates | Hourly: simple, consistent; Real-time: more complex, avoids lag on viral queries |
| Top-5 only | Full sorted set | Top-5 cache: tiny memory footprint; Full: supports custom N but larger |

## L5 Additions

1. **Personalized suggestions:** Merge global suggestions with user's recent searches. Architecture: user's recent queries cached in Redis (last 100 queries, TTL 30 days). On autocomplete request, fetch user's query history (< 5ms, in-memory). Score: `final_score = 0.7 × global_score + 0.3 × personal_recency_score`. Personal recency: higher boost for queries searched in last 7 days vs last 30 days. Privacy: personalization data never leaves user's region; GDPR-compliant deletion on request.

2. **Spell correction integration:** Build a separate spell-correction layer. When prefix returns 0 or very low-quality suggestions, run spell correction (edit distance 1 or 2, using SymSpell algorithm for performance). Return corrected prefix suggestions with "Did you mean: ..." indicator. SymSpell: pre-compute all word combinations within edit distance 2 → O(1) lookup. Add to pipeline as a post-processing step: if primary suggestions count < 3, run spell check and augment.

3. **Multi-term autocomplete:** "python inter" → suggest "python interview questions", "python interpreter", "python internship". The trailing term needs to be handled: split at last word boundary, look up prefix for last word within context. Implement n-gram trie: key = ("python", "inter") → suggestions include full query strings. Memory grows as O(N^2) for bigrams; prune to top 1M bigrams only. Great for developer/technical search boxes.

4. **Safety filtering:** Autocomplete must not suggest offensive, harmful, or policy-violating queries. Maintain a blocklist of ~100K prohibited terms and patterns. Blocklist is a bloom filter for O(1) membership check. Soft blocklist: demote but don't remove (some sensitive topics need to be searchable). Dynamic blocking: when a query starts trending that matches a news event involving violence/tragedy → emergency block via feature flag. Remove within hours as news context changes.

5. **Query segmentation and intent understanding:** "new york pizza" → user wants pizza restaurants in New York, not restaurants named "New York". Parse prefix into {entity: "New York", category: "pizza"}. Architecture: NER (Named Entity Recognition) model runs on prefix (< 5ms with distilBERT). Tag recognized entities → route to geo-aware or category-aware autocomplete. This is how Google Search Autocomplete works at the ML level — entity recognition shapes the candidate ranking.
