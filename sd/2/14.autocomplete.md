# 14. Search Autocomplete (Typeahead)

**Type:** search
**Difficulty:** L4
**Frequency:** Very High — data structures + scale combination

---

## Problem Statement

Design a search autocomplete system like Google's search suggestions. As users type, return the top 5 most relevant completions for the current prefix in real-time (< 100ms). Suggestions should be ranked by popularity (frequency of searches). The system must handle 10M QPS at peak (global Google scale).

---

## Functional Requirements

**Must have:**
- Return top 5 completions for a given prefix string
- Ranked by search frequency (most popular suggestions first)
- Latency < 100ms p99
- Support Unicode / multi-language

**Should have:**
- Personalization: factor in user's own recent searches
- Trending boost: searches spiking in last 24h ranked higher
- Spell correction: "gogle" → "google"

**Out of scope:**
- Full semantic query understanding
- Question answering ("what is...")
- Voice input processing

---

## Non-Functional Requirements

- **Scale:** 10M QPS globally (Google-scale); 1M QPS for typical product
- **Latency:** < 100ms p99 (including network from nearest PoP)
- **Freshness:** New trending queries should appear in suggestions within 1 hour
- **Availability:** 99.99% — degraded autocomplete is acceptable (empty suggestions), not unavailability

---

## Capacity Estimation

```
QPS analysis:
  10M search autocomplete requests/sec globally
  Assumption: user types avg 5 characters before submitting → 5 autocomplete requests per search
  Actual search submissions: 10M / 5 = 2M searches/sec
  Google processes ~8.5 billion searches/day = ~98K searches/sec → 10M QPS autocomplete is plausible

Prefix index size:
  Unique queries in top tier: 1B (rough Google estimate)
  Only top 10M queries need to be in the autocomplete index (long tail is too rare)
  Avg query length: 20 chars → 20 prefixes per query to index
  Total prefix entries: 10M queries × 20 prefixes = 200M prefix entries
  Per entry: prefix string (avg 10B) + 5 suggestions JSON (avg 200B) = 210B
  Total: 200M × 210B = 42GB → fits in a Redis Cluster (6-node cluster with 8GB RAM each = 48GB)

Per-language index:
  20 supported languages → 20 separate prefix indexes
  English dominates: ~50% of queries → English index ~21GB; others smaller
  Total all languages: ~84GB → 12-node Redis Cluster

CDN cache:
  Top 10K prefixes (high frequency): cache at CDN edge
  Cache-Control: max-age=60 (1-minute freshness)
  These 10K prefixes cover ~80% of all autocomplete traffic
```

## High-Level Architecture

```
[Client Browser / App]
  │  (keystroke debounce: 50ms — don't fire on every keystroke)
  │
  ▼
[CDN Edge (Fastly/CloudFront)] ──→ (cache miss) ──→ [Autocomplete API]
  │                                                          │
  └─ (cache hit: top common prefixes, max-age=60)    [Prefix Index (Redis Cluster)]
                                                             │
                                                    [Personalization Service]
                                                             │
                                                    [User History (Redis)]

[Background data pipeline]:
[Search logs] → [Kafka: query-events] → [Flink Aggregator] → [Frequency Store] → [Index Builder] → [Redis]
  (every search       (raw events)        (1-hour windows)    (per-query scores)  (batch update)
   logged here)
```

**Components:**
- **Autocomplete API:** Stateless service. Looks up prefix in Redis, optionally merges with personalization, returns top 5. Target: < 10ms processing time (80ms+ is network from edge to API).
- **Prefix Index (Redis):** Pre-computed map from prefix → sorted list of top-N completions with scores. Updated periodically by the index builder.
- **Background pipeline:** Counts query frequencies from search logs using a streaming aggregator. Produces frequency updates → index builder updates Redis prefix entries.

## Data Model

**Trie (conceptual) vs Prefix Hash (practical):**

**Option A: In-memory Trie**
```
Trie node structure:
  {
    children: Map<char, TrieNode>,
    top_k: List<{text, score}>,   # stored at every node, not just leaves
    is_end: bool
  }

Compressed (Patricia/Radix) trie:
  "google" stored as one node, not 6 separate single-char nodes
  Space-efficient: shared prefixes stored once

Example:
  "go" → { top_k: ["google", "good morning", ...], children: { "o": ... } }
  "goo" → { top_k: ["google", "goodreads", ...], children: { "g": ... } }

Advantages:
  - Memory-efficient: shared prefix storage
  - Graceful fallback: if prefix not found, walk up the trie to parent
Disadvantages:
  - Hard to distribute across machines (trie is a tree, not a hash)
  - Updates require locking portions of the trie
  - Serialization to distribute is complex
```

**Option B: Prefix Hash Table in Redis — chosen for distributed systems**
```
Redis Sorted Set per prefix:
  Key:   ac:{locale}:{prefix}
  Type:  Sorted Set
  Score: query frequency (higher = more popular)
  Member: query string

Example for prefix "goo" in English:
  ZADD ac:en:goo 98521 "google"
  ZADD ac:en:goo 87432 "google translate"
  ZADD ac:en:goo 72100 "google drive"
  ZADD ac:en:goo 65230 "google flights"
  ZADD ac:en:goo 54100 "google classroom"
  ZADD ac:en:goo 43200 "goodreads"
  ZADD ac:en:goo 38100 "good morning america"
  # Keep top ~20 to allow personalization reranking

Lookup:
  ZREVRANGE ac:en:goo 0 4 WITHSCORES
  → returns top 5 suggestions in O(log N + K) time

Update (when new frequency data from aggregation pipeline arrives):
  ZADD ac:en:goo 99200 "google"             # update score
  ZREMRANGEBYRANK ac:en:goo 0 -21           # keep only top 20, evict bottom entries
  → O(log N) per update

Memory per prefix key:
  avg 20 members × (avg 20B text + 8B score) = 560B per prefix key
  200M prefix keys × 560B = 112GB → fits in Redis Cluster with 14+ nodes at 8GB each
```

## Deep Dive 1: Frequency Aggregation Pipeline

**How do we know which queries are popular?**

```
Step 1: Capture every search submission
  Every search request → append to Kafka topic "query-events"
  Event schema: { query_text, locale, user_id (hashed), timestamp, session_id }
  Volume: 2M searches/sec → 2M events/sec into Kafka
  Kafka: 20 partitions, 100 consumers → 100K events/sec per consumer = manageable

Step 2: Flink streaming aggregation (1-hour tumbling windows)
  Aggregate by (locale, query_text) within each 1-hour window
  Emit at window close: { locale, query: "google maps", count: 453221, window_end }
  Deduplication: count unique sessions, not raw requests (avoids bot inflation)

Step 3: Compute decayed frequency score (smoothed across time)
  Goal: score represents "current popularity" not just raw count
  Formula: new_score = α × old_score + (1-α) × hourly_count
    α = 0.7 → gives 70% weight to accumulated history, 30% to latest hour
  This smooths out spikes while still tracking trends
  Stored in frequency store (Redis hash: query → smoothed_score)

Step 4: Prefix index updater
  For each query whose score changed significantly (threshold: > 5% change):
    Compute all prefixes of the query:
      "google maps" (11 chars) → ["g", "go", "goo", "goog", "googl", "google", "google ",
                                   "google m", "google ma", "google map", "google maps"]
      = 11 prefix updates per query
    For each prefix: ZADD ac:{locale}:{prefix} {new_score} {query}
  Volume: top 10M queries × 11 prefixes avg × daily update = 110M ZADD operations per batch
  Run as background job: 110M / 3600s = 30K ZADD/sec → easily handled by Redis Cluster
```

**Trending detection:**
```
Trending score = (current_hour_count) / (avg_count_same_hour_last_7_days + 1)
If trending_ratio > 5x: mark query as "trending" in the index

Implementation:
  Store per-query hourly counts in Redis time series (TTL 7 days + 1 hour)
  Key: trend:{locale}:{query_hash}:{hour_timestamp}
  Value: count (INCR)
  On each aggregation window: compute trending_ratio, update sorted set member metadata

Bloom filter for novel viral queries:
  New query not in top 10M → check if it's trending before adding to index
  Bloom filter (1B entries, 0.1% FPR, ~1.4GB) → O(1) "is this a known query?"
  If not in bloom filter: novel query → add to index only if trending_ratio > 20x
```

## Deep Dive 2: Scale — 10M QPS

**CDN caching — the key to handling 10M QPS:**
```
Observation: autocomplete request distribution is a power law
  Top 1,000 prefixes ("g", "go", "goo", "the", "how", "wh", "wha", etc.)
    → account for ~50% of all autocomplete traffic
  Top 10,000 prefixes → ~80% of traffic

CDN edge caches: GET /v1/autocomplete?q=goo&locale=en
  Cache key: (q, locale) — exclude user_id (personalization is post-cache)
  Cache-Control: public, max-age=60
  At 10M QPS × 80% CDN hit rate = 8M QPS absorbed by CDN
  Remaining 2M QPS hit Autocomplete API origin

For the 20% (long-tail prefixes) hitting origin:
  10M × 20% = 2M QPS
  API server at 10K RPS each = 200 API server instances
  Each API server: Redis lookup (sub-millisecond) → merge personalization → return
  Redis Cluster handles 2M reads/sec easily (each node: ~100K ops/sec × 20 nodes)
```

**Debouncing on the client:**
```javascript
// Client-side: don't fire on every keystroke
let debounceTimer;
function onKeystroke(prefix) {
  clearTimeout(debounceTimer);
  debounceTimer = setTimeout(() => {
    fetchSuggestions(prefix);
  }, 50);  // wait 50ms after last keystroke before firing request
}
// Benefit: user types "google" (6 chars) in 200ms
//   Without debounce: 6 requests
//   With 50ms debounce: 1-2 requests (only fires after pauses)
// Reduces QPS by ~3x client-side
```

**Regional deployment:**
```
Autocomplete sharded by locale:
  "en" queries → US/EU autocomplete clusters
  "zh" queries → APAC autocomplete cluster
  "ar" queries → MENA autocomplete cluster

Each region has its own Redis prefix cache
  Global frequency aggregation runs centrally (once per hour)
  Push updated frequency data to regional Redis clusters via Kafka cross-region replication
  Regional data lags global by up to 1 hour — acceptable per freshness SLA

Regional Redis Cluster:
  English: 14 nodes (112GB index)
  Other languages: 2–4 nodes each
```

**G5:** Would challenge the assumption that a flat prefix hash table is optimal at 10M QPS. The Redis sorted set approach stores every prefix independently, which means a user typing "google maps" generates 11 separate cache lookups ("g", "go", "goo", etc.). An alternative is to push autocomplete resolution to the CDN edge using a compact trie serialized as a binary blob, fetched once per user session and evaluated client-side. Google's actual implementation includes a JavaScript snippet that evaluates autocomplete in the browser against a locally-cached trie for the most common prefixes, falling back to a server RPC only for long-tail prefixes not covered by the local cache. This reduces server QPS by 10-20x for common prefixes and eliminates the network round-trip entirely (sub-millisecond response). The complexity is in delivering and invalidating the local trie blob — it must be small (< 50KB), cover the top 50K prefixes, and be refreshed hourly. This is a genuine G5-level architectural insight that changes the problem from "how do we serve 10M QPS" to "how do we distribute computation to the client."

## Deep Dive 3: Personalization

**Merging global and personal signals:**
```
On autocomplete request with user_id:

Step 1: Fetch global suggestions (from Redis, already done)
  global = ZREVRANGE ac:en:{prefix} 0 9 WITHSCORES  # top 10

Step 2: Fetch user's recent queries matching prefix (< 5ms target)
  Key: user_history:{user_id}
  Type: Sorted Set (score = recency timestamp)
  ZREVRANGEBYSCORE user_history:{user_id} +inf -inf
    LIMIT 0 100  # last 100 queries
  Filter client-side: keep those starting with {prefix}

Step 3: Merge and re-score
  For each global suggestion:
    personal_boost = 0
    if query in user_recent_searches (last 7 days): personal_boost += 30%
    if query in user_recent_searches (last 24 hours): personal_boost += 50%
    final_score = global_score × (1 + personal_boost)

  Re-rank merged list, take top 5

Latency budget:
  Redis global lookup: 1ms
  Redis user history lookup: 1ms (separate call, same datacenter)
  Merge computation: < 1ms (in-memory)
  Total: < 5ms → well within 100ms budget
```

**G5:** Would flag that serving personalized suggestions correctly requires that the user_history lookup happen in the same data center as the autocomplete API request — a global user_id means their history might be stored in a different region. For a user in Europe who has a history built up from US travels, a personalized suggestion request might require a cross-region call (EU → US) adding 80–100ms of latency, which violates the 100ms total SLA. Solution: replicate user history to all regions (small data: 100 queries × 50B = 5KB per user), with writes going to the user's home region and async replication everywhere else. Personalization can be slightly stale (up to 5 minutes) when the user switches regions, which is an acceptable trade-off for meeting the latency SLA.

## Reliability and Operations

- **Stale suggestions during Redis downtime:** If Redis is unavailable, return an empty suggestions list and let the user's search proceed. Never block the primary search action for autocomplete failure. Circuit breaker on Redis client: after 5 consecutive failures, open circuit for 30 seconds, return empty suggestions without waiting for timeout.
- **Index rebuild — atomic swap:** Nightly full rebuild: run MapReduce/Spark job on full 30 days of query logs → compute frequency scores for all 10M queries → write all prefix entries to a staging Redis Cluster → atomic traffic cutover (update DNS or load balancer to point to new cluster). Old cluster kept warm for 1 hour for fast rollback.
- **Prefix not in index:** For novel prefixes or languages with sparse index coverage, fall back to a simple PostgreSQL `ILIKE 'prefix%'` query against a deduplicated query log table. This covers the rare-path long-tail (< 1% of traffic) without needing to index every possible prefix.
- **Personalization service SLA:** 20ms timeout. If user history lookup exceeds 20ms, return global suggestions without personalization. Log the timeout for capacity planning. Never return a slower response to wait for personalization.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis sorted set per prefix | In-memory trie on API server | Redis: distributed, any API server can serve any prefix; Trie: memory-efficient, no network hop, but hard to distribute across servers |
| CDN for common prefixes (max-age=60) | All requests to origin | CDN: eliminates 80% of traffic to origin; trade-off: 60-second staleness for trending updates (new viral query takes 60s to appear at CDN edge) |
| Hourly frequency index updates | Real-time updates per search | Hourly: simple, consistent batch; Real-time: more complex pipeline, avoids lag on breaking-news viral queries |
| Store only top-20 per prefix | Full sorted set | Top-20 cache: tiny memory, sufficient for personalization reranking; Full set: more flexibility but 50x memory cost |
| Decayed exponential moving average score | Raw count | EMA: smooths out overnight spikes, weights recency; Raw count: biased toward older high-volume queries |

## G5

1. **Personalized suggestions with privacy guarantees:** Merge global suggestions with user's recent searches. Architecture: user's recent query list stored in Redis (last 100 queries, TTL 30 days, key: `user_history:{user_id}`). Scoring: `final_score = 0.7 × global_score + 0.3 × personal_recency_score`, where recency score decays exponentially with time since last search. Privacy: user history never leaves the user's assigned region; GDPR Article 17 (right to erasure) implemented as a `DEL user_history:{user_id}` call. For users who opt out of personalization: skip the user history lookup entirely, no data stored. The personalization weight (0.3) is an A/B-tested parameter — higher weights improve click-through rate for repeat queries but reduce discovery of new queries.

2. **Spell correction integration:** Separate spell-correction layer invoked when the primary prefix index returns fewer than 3 suggestions. Algorithm: SymSpell with edit distance ≤ 2. SymSpell pre-computes a delete-and-transpose neighborhood for every word in the dictionary, enabling O(1) lookup vs O(N) brute-force edit distance. Pre-computed dictionary: top 500K query terms + common English words. On correction: return original prefix suggestions (if any) first, then corrected prefix suggestions labeled "Did you mean: \<corrected>". Integration: add as a parallel call (not sequential) with 20ms timeout; merge results if correction call returns first and primary returns < 3.

3. **Multi-term autocomplete for contextual queries:** "python inter" → suggest "python interview questions", "python interpreter", "python internship" — not just single-word completions. Implementation: split prefix at last word boundary ("python", "inter"). Look up bigram index: `ac:en:bigram:python:inter` → suggestions for the full multi-word query. Bigram trie covers top 1M two-word prefixes (space × 1M × 20 prefixes = 20M additional keys). Memory: 20M × 250B = 5GB additional Redis storage. For developer-focused search products, multi-word autocomplete increases suggestion acceptance rate by 25–40% because users' queries are more specific. Prune aggressively: only index bigrams with count > 10K to keep memory bounded.

4. **Safety filtering and content policy:** Autocomplete must not surface offensive, harmful, or policy-violating queries. Blocklist: ~100K prohibited terms and patterns stored as a bloom filter (1B bits, ~125MB, 0.01% FPR) for O(1) membership check during suggestion serving. Soft-block: sensitive but legitimate topics (mental health, medications) are demoted but not removed — some sensitive searches must be reachable. Dynamic emergency blocking: when a mass-casualty news event breaks, a feature flag can immediately suppress autocomplete suggestions in that topic cluster (without index rebuild). The feature flag stores a set of blocked prefix patterns in Redis with a `SMEMBERS blocked_prefixes` check at serve time. Auto-unblock: editorial team reviews and adjusts within hours as news context evolves.

5. **Query segmentation and intent understanding:** "new york pizza" → entity "New York" (location) + category "pizza" (food). A model that understands this intent can route to geo-aware autocomplete (location-anchored suggestions) rather than lexical prefix matching. Architecture: a lightweight NER (Named Entity Recognition) model runs on each prefix — distilBERT quantized to INT8, inference < 5ms on CPU. If a location entity is recognized: augment suggestions with city-specific queries ("pizza delivery san francisco", "pizza near me"). If a brand entity: surface brand-specific completions. This is how Google's autocomplete distinguishes "apple" (the fruit) from "Apple" (the company) based on capitalization and context of prior tokens. The ML layer runs as a sidecar to the Redis lookup, with a hard fallback to pure lexical suggestions if inference exceeds 10ms.
