# 05. Consistent Hashing

**Type:** core
**Difficulty:** L4
**Frequency:** High — fundamental concept; asked as standalone or embedded in sharding designs

---

## Problem Statement

Design and explain consistent hashing. The system must distribute keys across N servers such that when servers are added or removed, only K/N keys need to be remapped (where K is total keys, N is number of servers). Implement it as a reusable library for a distributed caching or storage system.

---

## Functional Requirements

**Must have:**
- `add_node(node_id)` — add a server to the ring
- `remove_node(node_id)` — remove a server from the ring
- `get_node(key) → node_id` — find which node stores a given key
- Minimize key remapping when nodes change

**Should have:**
- Virtual nodes for uniform distribution
- Weighted nodes (some servers have more capacity)
- Replica placement: get next N distinct nodes for a key

---

## Non-Functional Requirements

- **Uniformity:** Keys should be evenly distributed (deviation < 5% from mean with vnodes)
- **Minimal disruption:** Adding/removing a node remaps exactly K/N keys
- **Performance:** `get_node` in O(log N) time

---

## Core Concept

**Problem with naive modular hashing:**
```
node = hash(key) % N

When N changes to N+1: almost all keys map to different nodes
→ massive cache invalidation, data migration storm
```

**Consistent hashing solution:**
```
1. Map all nodes AND keys onto the same circular "ring" of hash values (0 to 2^32 - 1)
2. Each key goes to the first node clockwise from its hash position
3. Adding a node: only keys between the new node and its predecessor remap
4. Removing a node: only that node's keys remap to its successor
5. Expected remapping: K/N keys (1/N fraction)
```

## Implementation

```python
import hashlib
import bisect

class ConsistentHashRing:
    def __init__(self, nodes=None, virtual_nodes=150):
        self.virtual_nodes = virtual_nodes
        self.ring = {}           # hash_position -> node_id
        self.sorted_keys = []    # sorted list of hash positions

        for node in (nodes or []):
            self.add_node(node)

    def _hash(self, key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node_id: str):
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}#{i}"
            h = self._hash(vnode_key)
            self.ring[h] = node_id
            bisect.insort(self.sorted_keys, h)

    def remove_node(self, node_id: str):
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}#{i}"
            h = self._hash(vnode_key)
            del self.ring[h]
            idx = bisect.bisect_left(self.sorted_keys, h)
            self.sorted_keys.pop(idx)

    def get_node(self, key: str) -> str:
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

    def get_nodes(self, key: str, n: int) -> list[str]:
        """Get n distinct nodes for replication."""
        if not self.ring:
            return []
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)

        nodes = []
        seen = set()
        for i in range(len(self.sorted_keys)):
            pos = (idx + i) % len(self.sorted_keys)
            node = self.ring[self.sorted_keys[pos]]
            if node not in seen:
                seen.add(node)
                nodes.append(node)
            if len(nodes) == n:
                break
        return nodes
```

## Virtual Nodes Deep Dive

**Why virtual nodes?**

Without vnodes: each physical node has 1 position on the ring.
- With 3 nodes, each holds exactly 1/3 of keyspace — but only if the hash function distributes perfectly.
- In practice, variance is high with small N.

With 150 vnodes per node:
- 3 nodes × 150 = 450 positions on ring
- Each physical node holds ~150/450 = 1/3 of keys (much more uniform)
- Adding a new node: 150 new positions scattered around ring → takes keys from many existing nodes proportionally

**Weighted nodes:**
```
Node A (4 CPU, 16GB) → 200 vnodes
Node B (2 CPU, 8GB) → 100 vnodes
Node C (1 CPU, 4GB) → 50 vnodes

→ A gets 200/350 ≈ 57% of keys
  B gets 100/350 ≈ 29% of keys
  C gets  50/350 ≈ 14% of keys
```

---

## Data Rebalancing on Node Change

**Adding a node:**
```
Before: A → B → C → D on ring (clockwise)
Add E between B and C:
  - Keys that were in [B, C) now split:
    - Keys in [B, E) → go to E (new node)
    - Keys in [E, C) → stay at C
  - Only ~K/N keys migrate (K = total keys, N = number of nodes)
  - Migration: E fetches its new keys from C
```

**Removing a node:**
```
Remove B:
  - Keys in B's range [A, B) now belong to C
  - C fetches B's keys (if not already replicated there)
  - If replication factor ≥ 2, C already has copies — just update routing
```

## Real-World Usage

**Cassandra:**
- Uses consistent hashing (Murmur3 hash) for token ring
- vnodes enabled by default (256 vnodes per node)
- Replication factor 3: primary + 2 successors on ring

**Redis Cluster:**
- Uses hash slots (0–16383) instead of full consistent hashing
- Key → CRC16(key) % 16384 → slot → node
- Slots are manually or automatically assigned to nodes
- Conceptually similar to consistent hashing with 16384 virtual slots

**Apache Cassandra coordinator:**
- Client SDK knows the ring → sends request directly to correct node
- No proxy hop needed (unlike Redis Cluster's redirect model)

**Memcached (libmemcached):**
- Client-side consistent hashing
- Each client independently computes which server holds the key
- No server-side awareness of the ring

## Tradeoffs

| Aspect | Consistent Hashing | Modular Hashing | Range Partitioning |
|--------|-------------------|----------------|-------------------|
| Rebalancing cost | K/N keys | Almost all keys | Configurable |
| Hotspot risk | Low (with vnodes) | Low | High (sequential keys) |
| Range queries | Not supported | Not supported | Supported |
| Complexity | Moderate | Trivial | Moderate |
| Node weight support | Yes (vnodes) | No | Yes |

## L5 Additions

1. **Rendezvous hashing (highest random weight):** Alternative to consistent hashing for small cluster sizes or when you need a simpler implementation. For a key, score each node as `hash(key + node_id)` and pick the node with the highest score. Adding/removing a node only remaps keys assigned to that node. No ring, no vnodes — just a comparison. Used in CDN routing (Nginx Plus, HAProxy) and Envoy's subset load balancing.

2. **Jump consistent hashing:** O(1) space, O(ln N) time, extremely uniform distribution. Uses a jump hash function to assign keys to buckets. Only supports adding nodes at the end (no arbitrary removal without renaming). Good for immutable-size clusters or append-only cluster expansion. Used in Google Spanner's tablet routing.

3. **Maglev hashing (Google's approach for load balancing):** Builds a lookup table of size M (prime, larger than N) where each position maps to a server. Fill the table by iterating each server's preference list. Result: near-perfect uniformity, consistent assignment, and O(1) lookup. Used in Google's Maglev load balancer — handles millions of connections with minimal disruption on server changes.

4. **Consistent hashing with bounded loads:** Standard consistent hashing can still create hot spots if keys are non-uniform (some keys get far more requests). Add a per-node load bound: if a node's load exceeds `(1 + ε) × average_load`, overflow to the next node on the ring. This gives at-most `1 + ε` more load on any node versus a perfectly balanced distribution. Used in Google's cluster scheduling.

5. **Hybrid: consistent hashing + gossip:** In large clusters (1000s of nodes), maintaining the ring view in all clients is expensive. Use gossip protocol to propagate ring changes: each node periodically gossips its neighbors' positions. Clients maintain a coarse-grained ring and refine on cache miss. Eventual consistency of ring view means occasional mismatch — handle with redirect: node B receives a request for key that belongs to C → forwards to C and returns result, not an error.
