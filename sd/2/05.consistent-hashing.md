# 05. Consistent Hashing

**Type:** core
**Difficulty:** L4
**Frequency:** High — fundamental concept; asked as standalone or embedded in sharding designs

---

## Problem Statement

Design and explain consistent hashing. The system must distribute keys across N servers such that when servers are added or removed, only K/N keys need to be remapped (where K is total keys, N is number of servers). Implement it as a reusable library for a distributed caching or storage system.

---

## Functional Requirements

**Must have:**
- `add_node(node_id)` — add a server to the ring
- `remove_node(node_id)` — remove a server from the ring
- `get_node(key) → node_id` — find which node stores a given key
- Minimize key remapping when nodes change

**Should have:**
- Virtual nodes for uniform distribution
- Weighted nodes (some servers have more capacity)
- Replica placement: get next N distinct nodes for a key

---

## Non-Functional Requirements

- **Uniformity:** Keys should be evenly distributed (deviation < 5% from mean with vnodes)
- **Minimal disruption:** Adding/removing a node remaps exactly K/N keys
- **Performance:** `get_node` in O(log N) time

---

## Core Concept

**Problem with naive modular hashing:**
```
node = hash(key) % N

Example: 5 nodes, key "user123" hashes to position 47 → node 47 % 5 = 2

Add a 6th node:
  "user123" hashes to 47 → 47 % 6 = 5 → DIFFERENT NODE
  → cache miss, must fetch from DB and re-populate node 5

Result: adding 1 node to a 5-node cluster remaps 5/6 = 83% of all keys
        → massive simultaneous cache miss storm → DB overwhelmed
```

**Consistent hashing solution:**
```
1. Map the hash space onto a conceptual ring [0, 2^32) wrapping around
2. Hash each node to a position on the ring: position(node) = hash(node_id)
3. Hash each key to a position on the ring: position(key) = hash(key)
4. A key belongs to the first node clockwise from its hash position

Properties:
  - Adding node E between B and C: only keys in arc [B, E) migrate to E
    All other keys: unaffected
  - Removing node B: only keys in arc [A, B) migrate to C (B's successor)
    All other keys: unaffected
  - Expected keys remapped on change: K/N (1/N fraction)
    Example: remove 1 of 10 nodes → only 10% of keys migrate
```

## Implementation

```python
import hashlib
import bisect
from typing import Optional

class ConsistentHashRing:
    """
    Consistent hash ring with virtual nodes.

    Time complexity:
      add_node:    O(V log V) where V = virtual_nodes
      remove_node: O(V log V)
      get_node:    O(log(N*V)) — binary search on sorted positions

    Space: O(N * V) for ring + sorted_keys
    """

    def __init__(self, nodes: list[str] = None, virtual_nodes: int = 150):
        self.virtual_nodes = virtual_nodes
        self.ring: dict[int, str] = {}          # hash_position → node_id
        self.sorted_keys: list[int] = []         # sorted hash positions for binary search

        for node in (nodes or []):
            self.add_node(node)

    def _hash(self, key: str) -> int:
        """
        Use xxHash or Murmur3 in production (faster, better distribution).
        MD5 used here for clarity.
        Returns a 128-bit int; we use all bits for better distribution.
        """
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node_id: str) -> None:
        """
        Add a node to the ring by creating V virtual node positions.
        Keys in [predecessor_vnode, new_vnode) migrate to this new node.
        """
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}#{i}"      # e.g., "server-3#47"
            h = self._hash(vnode_key)
            self.ring[h] = node_id
            bisect.insort(self.sorted_keys, h) # O(log N) insertion into sorted list

    def remove_node(self, node_id: str) -> None:
        """
        Remove all virtual nodes for this physical node.
        Keys in the removed arcs migrate to the next successor.
        """
        for i in range(self.virtual_nodes):
            vnode_key = f"{node_id}#{i}"
            h = self._hash(vnode_key)
            if h in self.ring:
                del self.ring[h]
                idx = bisect.bisect_left(self.sorted_keys, h)
                self.sorted_keys.pop(idx)

    def get_node(self, key: str) -> Optional[str]:
        """
        Find the responsible node for a key.
        Returns None if ring is empty.
        Time: O(log(N*V)) — binary search on sorted_keys
        """
        if not self.ring:
            return None
        h = self._hash(key)
        # Find first position >= h (clockwise); wrap around if past end of ring
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

    def get_nodes(self, key: str, n: int) -> list[str]:
        """
        Get n distinct physical nodes for a key (for replication).
        Returns up to min(n, num_physical_nodes) distinct nodes.
        """
        if not self.ring:
            return []
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)

        result: list[str] = []
        seen: set[str] = set()
        total_positions = len(self.sorted_keys)

        for i in range(total_positions):
            pos = (idx + i) % total_positions
            node = self.ring[self.sorted_keys[pos]]
            if node not in seen:
                seen.add(node)
                result.append(node)
            if len(result) == n:
                break

        return result

    def get_load_distribution(self) -> dict[str, float]:
        """
        Return the fraction of the ring owned by each physical node.
        Useful for verifying distribution uniformity.
        """
        if not self.ring:
            return {}

        node_arc_length: dict[str, int] = {}
        positions = self.sorted_keys
        ring_size = 2**128  # total hash space

        for i, pos in enumerate(positions):
            node = self.ring[pos]
            next_pos = positions[(i + 1) % len(positions)]
            arc = (next_pos - pos) % ring_size
            node_arc_length[node] = node_arc_length.get(node, 0) + arc

        return {
            node: arc_len / ring_size
            for node, arc_len in node_arc_length.items()
        }
```

## Virtual Nodes Deep Dive

**Why virtual nodes are necessary:**

```
Without virtual nodes (1 physical node = 1 ring position):

  3 nodes on a 2^32 ring:
    Node A at position: 1,200,000,000  (28% from 0)
    Node B at position: 2,800,000,000  (37% from A to B)
    Node C at position: 3,900,000,000  (26% from B to C; 9% from C to end)

  Key distribution: A=37%, B=26%, C=28%... varies widely based on hash luck
  Variance can be 2-3× between nodes → severe load imbalance

With 150 virtual nodes per physical node (450 total positions on ring):
  Each node owns 150/450 = 33.3% of arc length on average
  By law of large numbers: actual distribution within 1–3% of mean
  Empirically measured: < 2% deviation with 150+ vnodes per node
```

**Adding a node with virtual nodes:**
```
Before: 3 nodes × 150 vnodes = 450 positions
Add node D:
  → 150 new positions inserted at random ring locations
  → Each of A, B, C loses ~50 of their 150 arc segments to D
  → All three nodes contribute keys proportionally to D
  → D ends up with ≈ 25% of total keys (1/4 in 4-node cluster)

Key migration: K/N = 25% of all keys, sourced proportionally from A, B, C
```

**Weighted virtual nodes:**
```
Heterogeneous cluster (mixed hardware):
  Node A: 128GB RAM, 16 cores → 200 vnodes → 40% of keys
  Node B: 64GB RAM, 8 cores  → 100 vnodes → 20% of keys
  Node C: 32GB RAM, 4 cores  →  50 vnodes → 10% of keys
  Node D: 64GB RAM, 8 cores  → 100 vnodes → 20% of keys
  Node E: 32GB RAM, 4 cores  →  50 vnodes → 10% of keys
  Total:  500 vnodes; A gets 200/500 = 40%, matching its proportional capacity

Practical: set vnode count proportional to node capacity when adding to cluster
```

**G5:** The vnode count (150 or 256) is chosen at cluster creation time and is expensive to change later — redistributing from 150 to 256 vnodes per node requires migrating data for every newly added vnode position. More critically, if a subset of nodes are upgraded to higher-capacity hardware while the cluster is running, their vnode count must be increased online without moving all their existing data. The right approach is a "drain and re-add" operation: gradually remove old low-capacity node (150 vnodes), let data migrate to successors, then add back the upgraded node with 300 vnodes. Never change vnode counts in-place. Document this as an operational runbook, not an afterthought.

## Data Rebalancing on Node Change

**Adding a node:**
```
Before: ring positions (clockwise): A(100) → B(300) → C(700) → D(900) → A(100+wrap)
Add E at position 500 (between C=700... wait: E=500 is between B=300 and C=700)

Updated ring: A(100) → B(300) → E(500) → C(700) → D(900)

Keys that change:
  Keys in arc (300, 500]: previously went to C at 700; now go to E at 500
  All other arcs: unchanged

Migration: E fetches keys in (300, 500] from C
  At replication factor 3: C already has a replica copy of these keys (as replica 2 or 3)
  → E just claims them as its primary; update routing table
  → Optionally: stream data from C to E to ensure locality

Rate-limiting migration: don't let E saturate C's I/O
  → Transfer at max 10% of C's disk bandwidth (e.g., 50MB/s if C has 500MB/s)
  → Full migration of E's share: at 10PB cluster / 10K nodes = 1TB/node → 20 minutes at 50MB/s
```

**Removing a node (graceful decommission):**
```
Step 1: Mark node B as "leaving" in ring metadata
Step 2: Stop routing new writes to B
Step 3: Flush B's keys to successor C (stream all of B's data to C)
Step 4: Verify C has received all keys (Merkle tree comparison)
Step 5: Remove B from ring → C takes over B's arc
Step 6: Adjust replication: all keys that had B as replica 2 or 3 need a new replica

Emergency removal (node crashed):
  Skip steps 1-4; C absorbs B's arc immediately from its replica copies
  Anti-entropy kicks in to fill any gaps (keys B had that weren't replicated to C)
```

## Real-World Usage

**Apache Cassandra:**
```
Hash function: Murmur3 (fast, excellent distribution)
Token range: -2^63 to +2^63 - 1 (signed 64-bit)
Default vnodes: 256 per node (configurable: num_tokens)
Replication: RF=3 → primary + 2 successors on ring (distinct racks/DCs preferred)
Ring metadata: stored in each node, propagated via Gossip (no central coordinator)

Example cluster routing (RF=3, 3 nodes):
  Node 1: owns tokens 0 to 5000, replicated on nodes 2 and 3
  Node 2: owns tokens 5001 to 10000, replicated on nodes 3 and 1
  Node 3: owns tokens 10001 to 15000, replicated on nodes 1 and 2
```

**Redis Cluster:**
```
Approach: Fixed hash slots (not continuous ring)
  16,384 slots total (0 to 16383)
  Key → CRC16(key) % 16384 → slot → node assignment

Node assignment:
  3 primary nodes: node1=slots[0,5461], node2=slots[5462,10922], node3=slots[10923,16383]
  Each primary has replicas

Advantages of fixed slots:
  - O(1) slot-to-node mapping (just a lookup table of 16K entries)
  - Slot migration is unit of rebalancing (move one slot at a time)

Disadvantage vs consistent hashing:
  - Adding nodes requires manual or semi-automatic slot rebalancing
  - 1/16384 granularity of rebalancing (not as fine-grained as vnodes)
```

**Memcached (libmemcached):**
```
Client-side implementation: consistent hashing computed in client library
  - No server awareness of ring (servers are stateless)
  - Each client independently computes node for each key
  - Client configuration must be identical across all clients (same server list)

Ketama consistent hashing (libmemcached default):
  - 160 vnodes per server
  - MD5 hash with 4 virtual nodes per hash (160/4=40 hash calls per server)
  - De facto standard for Memcached client-side consistent hashing
```

**G5:** Redis Cluster's 16,384 hash slots feel like consistent hashing but have a subtle difference: slot boundaries are fixed, so you can't place a new node at an arbitrary ring position — you must move whole slots. This means adding a 4th node to a 3-node cluster migrates exactly 1/4 of slots (4,096 slots), which is correct on average, but the specific keys within each migrated slot are co-located — if some slots are hotter than others, the load distribution after migration may not be what you expect. Consistent hashing with 256 vnodes per node avoids this by having a finer-grained, more random partition structure. When designing a system that uses Redis Cluster as its hash layer, monitor per-slot key counts and heat; slot rebalancing should be load-aware (move hot slots last, not first).

## Tradeoffs

| Aspect | Consistent Hashing + vnodes | Modular Hashing | Range Partitioning (HBase-style) |
|--------|---------------------------|----------------|----------------------------------|
| Rebalancing cost | K/N keys migrate — excellent | Almost all keys migrate — catastrophic | Configurable split points — good |
| Hotspot risk | Low with 150+ vnodes | Low with uniform hash | High for sequential keys (monotonic IDs) |
| Range queries | Not supported | Not supported | Supported — tables sorted by key |
| Complexity | Moderate — ring state, gossip | Trivial — single formula | Moderate — split metadata management |
| Node weight support | Yes — proportional vnodes | No | Yes — variable-size ranges |
| Failure impact | K/N affected keys | Almost all | Adjacent range affected |

## G5

**What a G5 engineer adds beyond the above:**

1. **Rendezvous hashing (Highest Random Weight):** An alternative to consistent hashing that avoids the ring data structure entirely. For a key K and N nodes, compute `score(K, node_i) = hash(K + node_i)` for all N nodes; assign K to the node with the highest score. Adding/removing a node only remaps keys whose highest score was on the changed node (1/N fraction — same guarantee as consistent hashing). Space: O(N). Lookup: O(N) — must compute hash for all nodes. Used in CDN routing (Nginx Plus, Envoy's ring hash load balancer), where N is small (< 100 nodes) and the O(N) lookup is acceptable. Advantage: no ring state to maintain; works naturally in stateless load balancers.

2. **Jump consistent hashing (Google):** O(ln N) time, O(1) space, near-perfect uniformity. Uses a pseudo-random jump function to assign a key to a bucket (node). Only supports adding nodes at the end (bucket 0 to N-1); removing a node requires relabeling remaining nodes. Trade-off: cannot remove arbitrary nodes without rehashing — suitable only for append-only cluster growth (common in data warehousing). Used in Google Spanner's tablet routing and BigTable's tablet server assignment. Implementation is a 10-line function: `int JumpConsistentHash(uint64 key, int num_buckets)`.

3. **Maglev hashing (Google's load balancer approach):** Builds a permutation table of size M (prime number, M >> N) where each of M positions maps to one of N servers. Fill the table using each server's permutation list (seeded by server name hash). Lookup: O(1) — just index into the table. Properties: near-perfect uniformity (< 1% deviation), consistent assignment (same key always maps to same server unless that server is removed), minimal disruption on server change. Used in Google's Maglev software load balancer handling millions of connections. Particularly good for connection affinity (session stickiness) where you want to maintain consistent routing to avoid connection re-establishment costs.

4. **Consistent hashing with bounded loads:** Standard consistent hashing can still create hotspots if certain keys receive disproportionate request rates (not data volume, but request frequency). A key that receives 10,000 RPS while the average is 100 RPS will overload its node even if the key distribution is perfectly uniform. Add a per-node load bound: if node X's current load exceeds `(1 + ε) × (total_load / N)`, overflow requests for new keys to X's clockwise successor instead. This bounds any node's load to at most `(1 + ε)` times the average load — even for adversarial key distributions. ε = 0.25 is the theoretical optimum (from Google's 2017 paper "Consistent Hashing with Bounded Loads"). Used in Google's cluster scheduling and Vimeo's cache tier.

5. **Hybrid: consistent hashing + gossip for large clusters:** Maintaining the full ring state in every client is expensive at 10K+ nodes: 10,000 nodes × 256 vnodes × 8 bytes = 20MB of ring state per client, updated on every node change. Use a two-tier approach: clients maintain a coarse-grained ring (1 position per physical node, not vnodes), and route to an approximate coordinator. The coordinator has the full ring and routes to the correct final node, sending a "redirect" response if the client guessed wrong. Over time, clients learn the correct routing from redirects and cache it locally. Gossip propagates ring changes: each node gossips its neighborhood to 3 random peers every second; full ring state converges in O(log N) rounds. Clients refresh their ring view from gossip on a 30-second timer. This reduces ring state in clients from 20MB to 80KB while maintaining correct eventual routing.
