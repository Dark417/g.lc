# 01. TinyURL — URL Shortener

**Type:** core
**Difficulty:** L3–L4
**Frequency:** Very High — asked at almost every company

---

## Problem Statement

Design a URL shortening service like TinyURL or bit.ly. Users can submit a long URL and receive a short alias (e.g., `tinyurl.com/abc123`). Anyone with the short URL is redirected to the original URL. The service must handle hundreds of millions of short links and billions of redirects per day.

---

## Functional Requirements

**Must have:**
- Shorten a long URL → returns a unique short URL (7-char alias)
- Redirect: accessing a short URL redirects to the original long URL
- Short URLs never collide for different long URLs
- Short URLs are immutable once created (no re-assignment)

**Should have:**
- Custom alias: user can specify their own short code
- Expiration: short URL can have a TTL; accessing expired URL returns 410 Gone
- Click analytics: total click count per short URL

**Out of scope:**
- Per-click analytics with geo/device breakdowns (mention as extension)
- User accounts and link management
- QR code generation

---

## Non-Functional Requirements

- **Scale:** 100M new URLs/day (write), 10B redirects/day (read); read:write ≈ 100:1
- **Availability:** 99.99% — redirect failures cost users directly
- **Latency:** Redirect p99 < 10ms; creation p99 < 100ms
- **Durability:** No URL loss — once shortened, always resolvable until expiry
- **Storage:** ~5 years retention; 500B URLs × 500 bytes ≈ 250TB total
- **Security:** No redirect to malicious URLs (malware scanning); no enumeration attacks

---

## Capacity Estimation

```
Writes: 100M new URLs/day = 1,150/sec peak ~3,500/sec
Reads:  10B redirects/day = 115,000/sec peak ~350,000/sec

Storage per URL: short_code(7B) + long_url(200B) + metadata(100B) = ~307B
Daily storage:   100M × 307B = ~30GB/day
5-year storage:  30GB × 365 × 5 ≈ 55TB

Bandwidth (read): 350K req/s × 500B = ~170MB/s outbound
```

Key insight: 350K redirect RPS → can't go to DB every time → **cache is mandatory**.

---

## High-Level Architecture

```
User
 │
 ├─ POST /shorten ──→ [Write API] ──→ [ID Generator] ──→ [PostgreSQL]
 │                                                           │
 └─ GET /:code ─────→ [Redirect API] ──→ [Redis Cache] ──→ [PostgreSQL]
                                                  │
                                           [Cache Miss]
                                                  │
                                          [Analytics Queue] ──→ [Kafka] ──→ [ClickHouse]
```

**Components:**
- **Write API:** Validates URL, calls ID generator, writes to DB, populates cache
- **Redirect API:** Looks up cache → DB, returns 301/302 redirect, fires async analytics event
- **ID Generator:** Produces unique short codes (Base62 encoded)
- **Redis:** Hot-URL cache. 80% of traffic hits 20% of URLs → huge cache hit rate
- **PostgreSQL:** Primary store for URL mappings. Read replicas for overflow
- **Kafka + ClickHouse:** Async click event pipeline for analytics

---

## API Design

```
POST /v1/urls
Body:    { long_url: string, custom_alias?: string, ttl_days?: int }
Returns: { short_url: string, short_code: string, expires_at?: timestamp }
Errors:  400 invalid URL, 409 alias taken, 422 malicious URL detected

GET /:code
Returns: HTTP 302 Location: <long_url>
         HTTP 404 if not found
         HTTP 410 if expired

GET /v1/urls/:code/stats
Returns: { click_count: int, created_at: timestamp, last_clicked_at: timestamp }
```

**301 vs 302:**
- 301 Permanent → browser caches → no analytics on repeat visits but lower server load
- 302 Temporary → every redirect hits server → enables analytics
- Use 302 by default; allow 301 opt-in for performance

---

## Data Model

```sql
urls (
  short_code   VARCHAR(10) PRIMARY KEY,   -- 7-char Base62, indexed
  long_url     TEXT NOT NULL,
  user_id      BIGINT,                    -- nullable for anonymous
  created_at   TIMESTAMPTZ NOT NULL,
  expires_at   TIMESTAMPTZ,               -- NULL = never expires
  click_count  BIGINT DEFAULT 0           -- approximate, updated async
)

-- Index for analytics queries
CREATE INDEX ON urls (user_id, created_at);
-- Index for expiry cleanup job
CREATE INDEX ON urls (expires_at) WHERE expires_at IS NOT NULL;
```

**Redis cache schema:**
```
Key:   url:{short_code}
Value: {long_url, expires_at}  (JSON or MessagePack)
TTL:   min(URL TTL, 24h)
```

## Deep Dive 1: ID / Short Code Generation

**Requirements:** Unique, 7 chars, URL-safe, not guessable (no sequential)

**Option A: Random Base62**
- Generate random 7-char string → check DB for collision → retry if collision
- Collision probability with 7 chars (62^7 ≈ 3.5 trillion): at 100M URLs, p(collision) is negligible
- Problem: requires a DB read on every write

**Option B: Hash-based**
- MD5/SHA-256 of long URL → take first 7 chars → Base62 encode
- Same long URL → same short code (natural deduplication)
- Problem: two different URLs can hash-collide on 7-char prefix; need collision check

**Option C: Counter + Base62 (chosen)**
- Global counter (or Snowflake-like ID) → Base62 encode to 7 chars
- Counter at 100M/day reaches 62^7 in ~35,000 years — plenty of space
- Counters can be pre-allocated to each server in blocks (each server gets 1M IDs)
  → No central counter bottleneck
- Not guessable if encoded with a random shuffle of the Base62 alphabet (per-deployment secret)

**Block allocation:**
```
ID Allocator service: hands out blocks [1M, 2M), [2M, 3M), ... to each write server
Each write server uses its block locally — no coordination per request
```

---

## Deep Dive 2: Redirect at 350K RPS

**Cache strategy:** Cache-aside with Redis Cluster
- On redirect: check Redis first
- Cache hit → 302 → return (sub-millisecond)
- Cache miss → read DB → populate cache → return
- Cache hit rate >99% for popular URLs (top 1M URLs cover 80%+ of traffic)

**Cache sizing:**
- 1M hot URLs × 300B = 300MB — trivially fits in Redis
- Even top 10M URLs = 3GB — one Redis node handles this

**Thundering herd on cache miss:**
- Use Redis SETNX (SET if Not eXists) to acquire a per-key lock
- First request fetches from DB, others wait briefly then re-check cache
- Alternatively: short probabilistic early expiry (PER algorithm)

**Redirect server scaling:**
- Redirect API is stateless → horizontal scale behind load balancer
- 350K RPS across 10 servers = 35K RPS each — easily achievable

---

## Reliability and Operations

- **Redis failure:** If Redis is down, fall through to DB. DB can handle 10–20% of peak traffic as fallback.
- **DB failure:** Active-standby with automatic failover (< 30s RTO). Use read replicas for analytics.
- **Expiry cleanup:** Background job (cron) deletes expired URLs in batches; no impact on read path.
- **URL safety scanning:** Async — create the short URL immediately, scan async. If malicious, deactivate and return 451.
- **Idempotency:** Creating the same long_url with same user returns existing short_code — check before insert.

---

## Security and Abuse

- Rate limit creation: 10 URLs/min per IP, 1000/day per authenticated user
- Validate URL format (must be http/https, valid TLD)
- Safe Browsing API integration (Google) for malware detection — async, don't block creation
- Prevent enumeration: use random shuffle of Base62 alphabet so sequential codes look random
- HTTPS only for the short domain

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Counter + Base62 | Random UUID | Counter is sequential (predictable) but faster and no collision check needed |
| Redis cache | DB read replicas | Cache is faster but adds operational complexity and cache invalidation overhead |
| 302 redirect | 301 redirect | 302 enables analytics but doubles server load for repeat visitors |
| PostgreSQL | DynamoDB | Postgres is familiar and supports ACID; DynamoDB scales more easily but costs more and lacks joins |

**Biggest risk:** Cache stampede if Redis goes down under full load → mitigate with circuit breaker and graceful fallback.

## L5 Additions

**What a staff engineer adds beyond the above:**

1. **Geo-distributed redirection:** Deploy redirect servers in every major region (US, EU, APAC). Use Anycast DNS or GeoDNS to route users to nearest edge. Short code DB is globally replicated (CockroachDB or DynamoDB Global Tables) with tunable consistency — reads from local region (eventual), writes globally coordinated.

2. **URL metadata freshness:** For high-value customers (advertisers), the target URL can change (campaign pivot). Support conditional URL update: new short_code is created, old one permanently redirects to new one. Implement a redirect chain limit (max 2 hops) to prevent loops.

3. **Analytics at scale:** Don't write click counts directly to the URL row — creates hot row problem. Write click events to Kafka → aggregate in Flink → write batch updates to ClickHouse. Provide real-time (30s delay) and historical analytics APIs. Cache aggregated counts in Redis, updated by Flink.

4. **Abuse at scale:** Maintain a bloom filter of known-bad domains (synced from threat intelligence feeds). Block at the redirect server level before even checking the URL store. Track click velocity per short code — if a URL gets 1M clicks in 1 minute it's likely being used in a spam campaign → auto-suspend.

5. **Multi-tenancy:** Different customers get isolated namespaces (`acme.sh/abc` vs `brand.co/abc`). Per-tenant rate limits, SLAs, and analytics dashboards. Custom domains via CNAME pointing to our redirect infrastructure — TLS cert provisioning via Let's Encrypt ACME protocol.
