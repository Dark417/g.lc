# 01. TinyURL — URL Shortener

**Type:** core
**Difficulty:** L3–L4
**Frequency:** Very High — asked at almost every company

---

## Problem Statement

Design a URL shortening service like TinyURL or bit.ly. Users can submit a long URL and receive a short alias (e.g., `tinyurl.com/abc123`). Anyone with the short URL is redirected to the original URL. The service must handle hundreds of millions of short links and billions of redirects per day.

---

## Functional Requirements

**Must have:**
- Shorten a long URL → returns a unique short URL (7-char alias)
- Redirect: accessing a short URL redirects to the original long URL
- Short URLs never collide for different long URLs
- Short URLs are immutable once created (no re-assignment)

**Should have:**
- Custom alias: user can specify their own short code
- Expiration: short URL can have a TTL; accessing expired URL returns 410 Gone
- Click analytics: total click count per short URL

**Out of scope:**
- Per-click analytics with geo/device breakdowns (mention as extension)
- User accounts and link management
- QR code generation

---

## Non-Functional Requirements

- **Scale:** 100M new URLs/day (write), 10B redirects/day (read); read:write ≈ 100:1
- **Availability:** 99.99% — redirect failures cost users directly
- **Latency:** Redirect p99 < 10ms; creation p99 < 100ms
- **Durability:** No URL loss — once shortened, always resolvable until expiry
- **Storage:** ~5 years retention; 500B URLs × 500 bytes ≈ 250TB total
- **Security:** No redirect to malicious URLs (malware scanning); no enumeration attacks

---

## Capacity Estimation

```
Writes:
  100M new URLs/day ÷ 86,400 sec/day = 1,157/sec average
  Peak (3× average)                  = ~3,500/sec

Reads:
  10B redirects/day ÷ 86,400         = 115,741/sec average
  Peak (3× average)                  = ~350,000/sec

Storage per URL record:
  short_code  7 bytes
  long_url    200 bytes (average; URLs can be up to 2KB but median ~200B)
  metadata    100 bytes (user_id, timestamps, click_count)
  Total:      ~307 bytes per URL

Daily new storage:
  100M × 307B = 30.7GB/day

5-year cumulative:
  30.7GB × 365 × 5 = ~56TB (fits comfortably in a sharded RDBMS)

Cache sizing:
  80/20 rule: top 20% of URLs = 80% of traffic
  Top 20% of 500M URLs = 100M URLs × 307B = ~30GB → fits in Redis cluster
  Hot key set (top 1M, 99%+ of traffic) = 300MB → fits on single Redis node

Bandwidth (read):
  350K req/s × 500B response ≈ 175MB/s outbound
```

Key insight: 350K redirect RPS → can't go to DB every time → **cache is mandatory**.

## High-Level Architecture

```
User
 │
 ├─ POST /shorten ──→ [Write API] ──→ [ID Generator] ──→ [PostgreSQL Primary]
 │                         │                                      │
 │                   [URL Validator]                       [Read Replicas]
 │                   [Safe Browsing]
 │
 └─ GET /:code ─────→ [Redirect API] ──→ [Redis Cluster] ──→ [PostgreSQL Read Replica]
                                │              │
                          [Cache Hit]    [Cache Miss]
                                │              │
                       [302 Redirect]   [DB Lookup → Cache Write → 302]
                                │
                       [Async: Kafka] ──→ [Analytics Consumer] ──→ [ClickHouse]
```

**Components:**
- **Write API:** Validates URL, calls ID generator, writes to DB, populates cache
- **Redirect API:** Looks up cache → DB, returns 301/302 redirect, fires async analytics event
- **ID Generator:** Produces unique short codes (Base62 encoded counter)
- **Redis Cluster:** Hot-URL cache. 80% of traffic hits 20% of URLs → huge cache hit rate
- **PostgreSQL:** Primary store for URL mappings. Read replicas for redirect fallback and analytics
- **Kafka + ClickHouse:** Async click event pipeline; decoupled from the critical redirect path

---

## API Design

```
POST /v1/urls
Body:    { long_url: string, custom_alias?: string, ttl_days?: int }
Returns: { short_url: string, short_code: string, expires_at?: timestamp }
Errors:  400 invalid URL
         409 alias already taken
         422 malicious URL detected

GET /:code
Returns: HTTP 302 Location: <long_url>         (redirect)
         HTTP 404 if not found
         HTTP 410 if expired

GET /v1/urls/:code/stats
Returns: { click_count: int, created_at: timestamp, last_clicked_at: timestamp }
```

**301 vs 302:**
- 301 Permanent → browser caches the redirect locally → repeat visits never reach our servers → no analytics, but much lower load
- 302 Temporary → every redirect hits our servers → enables full click analytics
- Use 302 by default; allow 301 opt-in via `cache_redirect: true` parameter for performance-sensitive use cases
- Practical implication: at 350K RPS with 302, assume 30% repeat visitors from browser = 350K full load. With 301, those 30% never hit us = ~245K RPS instead.

## Data Model

```sql
urls (
  short_code   VARCHAR(10) PRIMARY KEY,    -- 7-char Base62, serves as the lookup key
  long_url     TEXT NOT NULL,
  user_id      BIGINT,                     -- nullable for anonymous creation
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at   TIMESTAMPTZ,               -- NULL = never expires
  click_count  BIGINT DEFAULT 0           -- approximate; authoritative count in ClickHouse
)

-- Supports: "show all URLs for a user, newest first"
CREATE INDEX ON urls (user_id, created_at DESC);

-- Supports: background cleanup job (SELECT ... WHERE expires_at < now())
CREATE INDEX ON urls (expires_at) WHERE expires_at IS NOT NULL;

-- Supports: deduplication on creation ("does this long_url already have a code?")
CREATE INDEX ON urls (long_url) WHERE user_id IS NULL;
```

**Redis cache schema:**
```
Key:   url:{short_code}
Value: {"long_url": "https://...", "expires_at": 1735689600}
       Encoded as MessagePack for ~30% size reduction vs JSON
TTL:   min(URL expires_at - now(), 24h)
       If URL never expires: TTL = 24h (re-cache on next miss)
```

**Why not store click_count in Redis?** Redis counters would be accurate but lost on Redis restart. Store in DB async via Kafka pipeline; Redis holds an approximate increment buffer flushed every 30 seconds.

## Deep Dive 1: ID / Short Code Generation

**Requirements:** Unique, 7 chars, URL-safe, not guessable (no sequential)

**Option A: Random Base62**
- Generate random 7-char string from alphabet `[0-9A-Za-z]` (62 chars)
- 62^7 = 3.52 trillion possible codes
- At 100M URLs: probability of any collision ≈ (100M)² / (2 × 3.52T) ≈ 0.14% — not negligible
- Requires a DB read (SELECT) on every write to check for collision → serial bottleneck

**Option B: Hash-based**
- MD5(long_url) → take first 43 bits → Base62 encode to 7 chars
- Natural deduplication: same long URL always maps to same short code
- Collision on different URLs: birthday paradox at 100M URLs → p(collision) ≈ 1.4% on 43-bit space → need collision check + suffix retry
- Problem: deterministic → attacker can precompute codes for known long URLs

**Option C: Counter + Base62 (chosen)**
- Global atomic counter → Base62 encode to 7 chars
- Counter 0 → "0000000", counter 1 → "0000001", ..., 62^7 IDs before overflow
- At 100M/day: 62^7 / 100M = 35,200 years of unique codes
- No collision possible — every counter value is unique by definition
- Not guessable: shuffle the Base62 alphabet with a per-deployment secret using a Feistel cipher over the 41-bit counter space → counter 1 maps to "kT7mQ9x" (looks random, but is reversible)

**Block allocation (avoids central bottleneck):**
```
ID Allocator service (backed by ZooKeeper or a simple DB sequence):
  - Hands out non-overlapping blocks to each Write API server
  - Each server requests a block of 1,000,000 IDs at startup
  - Server burns through block locally (no network call per ID)
  - When 10% of block remains, pre-fetches next block asynchronously
  - If Allocator is down: server uses remaining block; degraded but functional

Example:
  Server A: block [0, 1M)      → uses 0, 1, 2, ...
  Server B: block [1M, 2M)     → uses 1M, 1M+1, ...
  Server C: block [2M, 3M)     → uses 2M, ...
```

**Feistel scrambling (prevents enumeration):**
```python
def scramble(n: int, key: int, rounds: int = 8) -> int:
    """Feistel cipher over 42-bit space (covers 62^7 ≈ 2^41.8)"""
    half = 21
    mask = (1 << half) - 1
    L, R = (n >> half) & mask, n & mask
    for i in range(rounds):
        f = hash((R, key, i)) & mask
        L, R = R, L ^ f
    return (L << half) | R

# counter 1000000 → scramble(1000000, SECRET_KEY) → some opaque integer
# → Base62 encode → "kT7mQ9x"
```

**G5:** The block allocation approach has a subtle failure mode: if a Write API server crashes mid-block, those IDs are permanently wasted. At 1M IDs/block and 100M IDs/day, a fleet of 30 servers each wasting up to 1M IDs on restart could burn through ~3% of the daily budget — harmless at this scale, but worth tracking as a metric (wasted_ids_per_day). The real risk is the Feistel key: if it leaks or needs rotation, you cannot re-scramble existing codes because their external short URLs are already published. Treat the Feistel key as a KMS-managed secret with a documented rotation freeze policy: "the scrambling key is permanent for the lifetime of the service."

## Deep Dive 2: Redirect at 350K RPS

**Cache strategy:** Cache-aside with Redis Cluster

```
On redirect request for /:code:
  1. Redis GET url:{code}           ← sub-millisecond
  2. Cache hit:
       Check expires_at in cached value
       If expired → return 410, delete from cache + DB async
       If valid   → return 302 Location: long_url
  3. Cache miss:
       DB SELECT WHERE short_code = {code}
       If not found  → return 404
       If found      → Redis SET url:{code} {value} EX {ttl}
                     → return 302 Location: long_url
```

**Cache sizing:**
```
Hot set analysis (Zipf distribution of URL access):
  Top 1,000 URLs:    accounts for ~40% of traffic = 140K RPS → 300KB in Redis
  Top 10,000 URLs:   accounts for ~60% of traffic = 210K RPS → 3MB
  Top 1,000,000 URLs: accounts for ~99% of traffic = 346K RPS → 300MB

Conclusion: a single 1GB Redis instance handles 99%+ of all redirect traffic.
Redis Cluster with 3 shards × 2 replicas provides HA with no hot-key issues.
```

**Thundering herd on cache miss:**
```
Scenario: popular URL's cache entry expires → 10,000 simultaneous requests all miss cache
→ 10,000 concurrent DB reads for the same key → DB overwhelmed

Solution 1: Redis SETNX lock
  lock_acquired = SETNX lock:{code} 1 EX 1s
  if lock_acquired:
    fetch from DB, SET cache, release lock
  else:
    sleep 10ms, retry Redis GET (lock holder populated cache)

Solution 2: Probabilistic Early Recomputation (PER algorithm)
  beta = 1.0  # tuning parameter
  if (now - beta * exp_time * log(random())) > cache_expiry:
    # probabilistically recompute before TTL expires
    fetch from DB, extend TTL
  # hot URLs are refreshed proactively, never expire at 10K RPS
```

**Redirect server scaling:**
```
350K RPS at p99 < 10ms:
  Each redirect server: 35K RPS (with Redis Cluster doing the heavy lifting)
  CPU: ~5% per server at 35K RPS (mostly network I/O, not compute)
  10 servers gives 3.5× headroom for traffic spikes
  Auto-scaling target: add server when any existing server exceeds 25K RPS sustained
```

**Analytics event pipeline (off critical path):**
```python
# Redirect handler — total latency budget: 10ms
async def redirect(code: str):
    url = await redis.get(f"url:{code}")           # ~0.3ms
    if not url:
        url = await db.fetch(code)                 # ~2ms on miss
        await redis.set(f"url:{code}", url, ex=86400)

    # Fire-and-forget to Kafka — does NOT block the response
    asyncio.create_task(kafka.produce("clicks", {
        "code": code, "ts": time.time(), "ip": request.ip
    }))

    return Response(302, headers={"Location": url["long_url"]})  # ~0.1ms
```

**G5:** The Redis cache TTL of 24h is a blunt instrument. A URL created by a marketing campaign that runs for 4 hours will be cached for 24h — wasting Redis memory. But a URL for a permanent corporate homepage benefits from a much longer TTL (or LRU-based eviction, never proactive expiry). The right model is write-frequency-aware TTL: `TTL = min(24h, max(5min, 24h / sqrt(writes_per_hour + 1)))`. For a URL that gets zero writes after creation, this collapses to 24h. For a URL whose `long_url` target is updated frequently (campaign redirect), the TTL shortens proportionally. Instrument cache stale-hit rate per URL bucket (by age and write frequency) to tune the formula constants.

## Reliability and Operations

- **Redis failure:** If Redis is completely unreachable, fall through to DB read replicas. DB read replicas can handle 20–30K RPS (vs 350K total) — enough for degraded operation at reduced capacity. Circuit breaker opens after 5 consecutive Redis timeouts; closes after Redis health check passes.
- **DB failure:** Active-standby PostgreSQL with Patroni. Automatic failover in < 30s (RTO). WAL streaming to standby ensures RPO ≈ 0 (no data loss). Read replicas for analytics queries.
- **Expiry cleanup:** Cron job runs every 5 minutes: `DELETE FROM urls WHERE expires_at < now() - interval '1 hour'` (1 hour grace period in case of clock skew). Runs in batches of 10K rows to avoid table lock contention.
- **URL safety scanning:** Never block the creation path for scanning — that adds 200–500ms. Create the short URL immediately, publish to Kafka "url-created" topic. Safety scanner consumes this, calls Google Safe Browsing API, sets `status='malicious'` if flagged. Redirect API checks status field (cached alongside long_url); returns 451 Unavailable For Legal Reasons if malicious.
- **Idempotency:** On creation, if `(user_id, long_url)` already exists and is not expired, return existing short_code. Use `INSERT ... ON CONFLICT DO NOTHING ... RETURNING` pattern. Prevents duplicate codes for same user/URL pair.

## Security and Abuse

- Rate limit creation: 10 URLs/min per unauthenticated IP, 1000/day per authenticated user (enforced via Redis sliding window counter)
- Validate URL format: must start with `http://` or `https://`, valid TLD, non-empty host
- Safe Browsing API (Google) for malware/phishing — async post-creation, doesn't block user
- Prevent enumeration: Feistel scrambling makes sequential counter codes appear random externally
- Custom aliases: must match `^[a-zA-Z0-9_-]{4,20}$`; reserved words (`api`, `admin`, `static`, etc.) blocked
- HTTPS only for short domain; redirect any HTTP request to HTTPS first

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| Counter + Base62 | Random UUID | Counter guarantees no collisions (no DB check needed on write); random UUID requires a collision check read, adding ~2ms latency to every creation |
| Feistel scrambling | Plain sequential counter | Prevents URL enumeration attacks (scanning abc0001 → abc0002) without adding any per-request work; alternatives like random Base62 require a DB uniqueness check |
| Redis cache | DB read replicas only | Redis serves 350K RPS at <1ms; DB replicas can handle maybe 50K RPS at 2–5ms; at this scale Redis is not optional |
| 302 redirect (default) | 301 redirect | 301 caches in browser forever — kills analytics for all repeat visits; 302 enables complete click attribution at the cost of every request hitting our servers |
| PostgreSQL | DynamoDB | PostgreSQL gives ACID transactions (important for idempotent creation) and rich indexing; DynamoDB scales more easily but costs 5–10× more at this data volume and lacks the `ON CONFLICT` semantics we rely on |

**Biggest risk:** Cache stampede if Redis goes down under full load. Mitigate with circuit breaker (fail open to DB), pre-warming on Redis restart, and probabilistic early expiry on high-traffic URLs.

## G5

**What a G5 engineer adds beyond the above:**

1. **Geo-distributed redirection:** Deploy redirect servers in every major region (US-East, US-West, EU-West, APAC). Use Anycast DNS to route users to nearest PoP — no GeoDNS propagation delay. The URL DB is globally replicated using CockroachDB or DynamoDB Global Tables with tunable consistency: reads from local region (eventual, 50ms replication lag), writes globally coordinated. For the redirect path, eventual consistency is fine — a 50ms window where a new URL isn't visible in a remote region is acceptable. For creation idempotency checks, route all writes to a single coordinating region to avoid split-brain.

2. **URL metadata freshness and redirect chains:** For enterprise customers (advertisers), the target URL changes (campaign pivots, A/B test rotation). Support `PATCH /v1/urls/:code` to update `long_url`. This invalidates the Redis cache entry immediately (DEL then DB write, not the reverse — avoids a race where a request reads stale cache between write and invalidation). Implement redirect chain detection: if long_url itself is a `tinyurl.com/` URL, unwrap it immediately at creation time (store the final destination) — prevents compounding latency and chain loops. Enforce max redirect chain depth of 1.

3. **Analytics at scale without hot rows:** Never UPDATE `click_count` in the URL row synchronously — every redirect becomes a write, creating a hot row under lock. Instead: Kafka click events → Flink 30-second tumbling window → batch UPSERT to ClickHouse for historical data and Redis INCR for real-time approximate counts. The URL row's `click_count` is updated by a background job every 5 minutes (best-effort). This gives near-real-time analytics without any write contention on the redirect hot path.

4. **Abuse at scale — bloom filter for bad domains:** Maintain a Bloom filter of known-bad domains (updated from threat intelligence feeds like Google Safe Browsing, URLhaus, PhishTank). The filter lives in-memory on every Redirect API server (~500MB for 100M blocked domains at 1% FPR). Check the filter before even reading from Redis — if the domain is flagged, return 451 immediately with zero DB load. For click velocity abuse: track click counts per short_code in a Redis sliding window. If a code receives >10K clicks in 60 seconds (likely a spam campaign), auto-suspend and queue for human review.

5. **Multi-tenancy and custom domains:** Enterprise customers want `acme.co/promo` not `tinyurl.com/abc123`. Each customer has an isolated namespace with their own domain (CNAME to our infrastructure). Per-tenant rate limits, analytics dashboards, and SLAs. TLS cert provisioning via Let's Encrypt ACME protocol with cert-manager. The redirect lookup key becomes `(domain, short_code)` instead of just `short_code` — update cache key to `url:{domain}:{code}` to avoid namespace collisions across tenants.
