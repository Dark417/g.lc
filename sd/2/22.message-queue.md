# 22. Distributed Message Queue (Kafka-like)

**Type:** infra
**Difficulty:** G5
**Frequency:** High — storage internals, distributed log, consumer groups

---

## Problem Statement

Design a distributed message queue like Apache Kafka. Producers publish messages to topics. Consumers subscribe to topics and process messages. The system must provide high throughput (millions of messages/sec), durability (no message loss), ordered delivery within a partition, and support multiple independent consumer groups replaying the same messages.

---

## Functional Requirements

**Must have:**
- Producers publish messages to a named topic
- Topics are divided into partitions for parallelism
- Messages within a partition are ordered and persistent
- Consumers read messages with an offset pointer (can replay from any offset)
- Multiple consumer groups can read the same topic independently (no interference)
- At-least-once delivery guarantee

**Should have:**
- Message retention: configurable by time (7 days) or size (1TB per partition)
- Consumer group rebalancing: when consumers join/leave, partitions are redistributed
- Exactly-once semantics (producer idempotency + transactional API)
- Compression (gzip, snappy, lz4) per message batch

**Out of scope:**
- Message routing / filtering by content (Kafka doesn't do this — see RabbitMQ)
- Dead letter queues
- FIFO across multiple partitions

---

## Non-Functional Requirements

- **Throughput:** 1M messages/sec write; 5M messages/sec read (multiple consumer groups)
- **Latency:** Write p99 < 5ms (after batch flush); Read p99 < 10ms
- **Durability:** Message persisted to disk before ACK
- **Availability:** 99.99% — message loss is catastrophic
- **Scale:** 1PB total storage across the cluster; horizontal scaling

## Capacity Estimation

```
Write throughput:
  1M messages/sec × 1KB avg = 1GB/sec ingest
  With 3x replication = 3GB/sec total write across cluster
  Each broker: 1Gbps NIC → handles ~125MB/sec = ~125K messages/sec (1KB msgs)
  1M messages/sec / 125K per broker = 8 brokers minimum for write bandwidth
  → Deploy 12 brokers (50% headroom for replication + rebalancing overhead)

Read throughput:
  5M messages/sec × 1KB = 5GB/sec read
  Kafka uses OS page cache heavily → hot data served from RAM (effectively free I/O)
  Hot data (last 24h): 1GB/sec × 86,400s = 86TB → across 12 brokers = 7TB/broker
  Typical broker RAM: 128–256GB → hot data fits in page cache for < 24h retention
  Consumer reads effectively at memory bandwidth, not disk bandwidth

Storage:
  1M msg/sec / 1000 partitions = 1000 msg/sec per partition
  Per partition per day: 1000 msg/sec × 1KB × 86,400s = 86GB/day
  7-day retention per partition: 86GB × 7 = 604GB per partition replica
  1000 partitions × 604GB × 3 replicas = 1.8PB physical storage
  → 100 storage nodes at 20TB NVMe each (NVMe for sequential I/O performance)
  With snappy compression (~3x): actual data stored ≈ 600TB

Network:
  Write: 1GB/sec producer + 2GB/sec replication = 3GB/sec → 24Gbps across 12 brokers = 2Gbps/broker (manageable with 10GbE)
  Read: 5GB/sec reads → 40Gbps across 12 brokers; each broker: ~3.3Gbps read → two 10GbE bonded NICs per broker
```

## High-Level Architecture

```
[Producers] ──→ [Broker Cluster (N brokers)]
                      │
               [Topic: "payments"]
                  Partition 0: [Leader: Broker1] ← [Follower: Broker2, Broker3]
                  Partition 1: [Leader: Broker2] ← [Follower: Broker1, Broker3]
                  Partition 2: [Leader: Broker3] ← [Follower: Broker1, Broker2]
                      │
[ZooKeeper / KRaft] ──┘ (cluster metadata, leader election, controller)
                      │
[Consumers (Group A)] ──→ [Consumer Group Coordinator (broker)]
[Consumers (Group B)] ──→ [Consumer Group Coordinator (broker)]
```

**Key insight:** Each partition is an append-only sequential log. Producers write to the leader. Followers pull and replicate. Consumer groups each track their own committed offsets independently — two groups reading the same topic have zero interference. The log is the API.

## API Design

```python
# Producer
producer = KafkaProducer(
    bootstrap_servers=['broker1:9092', 'broker2:9092'],
    acks='all',                   # wait for all ISR replicas
    compression_type='snappy',    # compress batches
    batch_size=65536,             # 64KB batch before sending
    linger_ms=5,                  # wait up to 5ms to fill batch
    retries=3,
    enable_idempotence=True,      # producer-side dedup
)

# Send a message (async, batched internally)
future = producer.send(
    topic="payments",
    key=b"user-123",             # key determines partition assignment
    value=b'{"amount": 5000, "currency": "USD"}',
    headers=[("trace_id", b"abc-123")]
)
producer.flush()  # block until all pending batches are sent

# Consumer
consumer = KafkaConsumer(
    "payments",
    bootstrap_servers=['broker1:9092'],
    group_id="fraud-detection-service",
    auto_offset_reset="earliest",    # start from beginning if no committed offset
    enable_auto_commit=False,        # manual commit after processing
    max_poll_records=500,            # fetch up to 500 records per poll
    fetch_max_bytes=52428800,        # 50MB max fetch response
)

for message in consumer:
    process(message.value)
    consumer.commit()  # commit offset after successful processing
```

**Admin API:**
```
# Create topic with 16 partitions, replication factor 3
kafka-topics.sh --create --topic payments --partitions 16 --replication-factor 3 \
  --config retention.ms=604800000  # 7 days

# Describe topic: shows leaders, ISR, offsets
kafka-topics.sh --describe --topic payments

# Consumer group lag
kafka-consumer-groups.sh --describe --group fraud-detection-service
# → Shows: TOPIC | PARTITION | CURRENT-OFFSET | LOG-END-OFFSET | LAG | CONSUMER-ID
```

## Data Model

**On-disk format: The Append-Only Log**
```
Each partition is stored as a series of immutable segment files on each broker:

/kafka-data/payments-0/              ← partition 0 data directory
  00000000000000000000.log           ← segment starting at offset 0
  00000000000000000000.index         ← sparse offset→file_position index
  00000000000000000000.timeindex     ← timestamp→offset index (for time-based seek)
  00000000000015000000.log           ← segment starting at offset 15,000,000
  00000000000015000000.index
  ...
  00000000000029999999.log           ← active (current) segment

Message format in .log file (record batch, packed binary):
  [baseOffset: 8B]         ← first offset in batch
  [batchLength: 4B]        ← total bytes in batch
  [leaderEpoch: 4B]        ← for leader validation
  [magic: 1B]              ← record format version (currently 2)
  [crc: 4B]                ← CRC32C of everything following
  [attributes: 2B]         ← compression codec, timestamp type, transactional
  [lastOffsetDelta: 4B]
  [firstTimestamp: 8B]
  [maxTimestamp: 8B]
  [producerId: 8B]         ← for idempotent/transactional producers
  [producerEpoch: 2B]
  [baseSequence: 4B]
  [records: N]             ← variable-length array of individual records
    Each record:
      [length: varint][attributes: 1B][timestampDelta: varint]
      [offsetDelta: varint][key: bytes][value: bytes][headers: array]

Sparse index (.index):
  One entry per 4KB of log data (configurable via index.interval.bytes)
  Entry: [relativeOffset: 4B][filePosition: 4B]
  To find offset N: binary search index → get file position → seek → scan forward
  → O(log(index_size)) + O(entries_per_4KB) ≈ O(1) in practice
```

**Consumer offset storage:**
```
Consumer group offsets stored in internal Kafka topic: __consumer_offsets
  Key: (group_id, topic, partition)  → serialized binary
  Value: offset_metadata { committed_offset, metadata, commit_timestamp }
  Compacted topic: keeps only the latest offset per (group, topic, partition)
  Replicated 3x by default

Periodically committed by consumer after processing each batch.
On consumer restart: read committed offset from __consumer_offsets → resume from there.
```

## Deep Dive 1: Write Path and Durability

**Producer write path (step by step):**
```
1. Producer SDK batches messages in memory:
   - Batch accumulates until batch.size (64KB) OR linger.ms (5ms) is reached
   - Multiple partitions batched into one TCP request (pipelining)

2. Partition selection:
   - With key: partition = murmur2(key) % num_partitions
     → same key always goes to same partition → ordering guarantee
   - No key: sticky partition assignment (round-robin across batches, not messages)
     → avoids small batches from continuous round-robin

3. Send batch to partition leader:
   - TCP connection maintained per broker (connection pool)
   - Request: ProduceRequest { topic, partition, records, acks, timeout }

4. Leader receives batch:
   a. Validates CRC, magic byte, record format
   b. Checks producer sequence numbers (idempotent producer: detect duplicates)
   c. Appends to in-memory write buffer (OS page cache — not yet durable)
   d. Appends to active segment .log file

5. ACK behavior (acks config):
   acks=0:   Producer does not wait; no durability guarantee; highest throughput
   acks=1:   Leader writes to local log → sends ACK
             Risk: leader fails before follower replicates → message lost
   acks=all: Leader waits for all ISR (in-sync replicas) to ACK
             Highest durability; message survives any single broker failure

6. Follower replication:
   - Followers continuously issue FetchRequest to leader for new data
   - Leader responds with new records; follower appends to local log
   - Follower sends ACK to leader → leader advances high watermark
   - High watermark = highest offset acknowledged by all ISR members
   - Consumers only see messages up to the high watermark (committed messages)
```

**Durability guarantee with acks=all:**
```
ISR = {Broker1 (leader), Broker2, Broker3}
min.insync.replicas = 2  ← require at least 2 ISR members to ACK

Write flow:
  1. Leader: append to local log
  2. Broker2 fetches → appends → ACKs leader
  3. Broker3 fetches → appends → ACKs leader
  4. Leader: 2+ ISR have ACKed → high watermark advances → ACK producer
  5. Producer: gets success response

If Broker3 is slow (lag > replica.lag.time.max.ms = 10s):
  → Broker3 removed from ISR
  → ISR = {Broker1, Broker2}; still satisfies min.insync.replicas=2
  → Write succeeds at full speed; when Broker3 catches up → re-added to ISR

If leader fails AFTER high watermark advanced:
  → Broker2 or Broker3 elected as new leader
  → Both have the committed message → no data loss
```

**fsync and the OS page cache trade-off:**
```
Kafka does NOT call fsync() after every message write by default.
  → OS page cache holds data in RAM; written to disk asynchronously
  → Very fast writes (RAM bandwidth, not disk bandwidth)
  → If the OS crashes (not just Kafka): data in page cache can be lost

Why this is safe:
  Replication provides durability, not per-message fsync.
  If a broker crashes, leader election picks a replica that has the data.
  → A single-node failure loses nothing (with acks=all and ISR >= 2)
  → Only simultaneous failure of ALL ISR members causes data loss

For strict durability (e.g., financial logs):
  flush.messages=1 (fsync every message)  → 10–100× write throughput reduction
  flush.ms=1000 (fsync every second)      → acceptable for most use cases
  Default: rely on replication instead of fsync — correct design for Kafka's architecture
```

**G5:** The page cache durability argument has a subtle edge case: unclean leader election. If `unclean.leader.election.enable=true` (the default in older Kafka versions), a broker that fell out of the ISR (and therefore missed some messages) can be elected leader when all ISR members are unavailable. This broker would be missing messages that were committed — the high watermark on the new leader rolls back, and those messages are silently lost. With `unclean.leader.election.enable=false`, Kafka refuses to elect an out-of-sync broker and becomes unavailable instead. The correct production setting for financial systems is `unclean.leader.election.enable=false` + `min.insync.replicas=2` + `acks=all`. These three settings together give a true durability guarantee; any one of them alone does not.

## Deep Dive 2: Consumer Group and Partition Rebalancing

**Consumer group protocol:**
```
Consumer group: a set of consumers that collectively consume all partitions of a topic
  Group "fraud-detection" has 4 consumers → each gets 4 of 16 partitions
  No two consumers in the same group read the same partition simultaneously

Group coordinator: one broker per consumer group (determined by: hash(group_id) % num_brokers)
  Tracks membership, heartbeats, offset commits for this group

Join protocol (triggered on group formation or membership change):
  1. Each consumer sends JoinGroupRequest to coordinator
  2. Coordinator selects one consumer as the "leader" (first to join, or arbitrary)
  3. Coordinator sends JoinGroupResponse to all members with full member list
  4. Leader computes partition assignment (using configured AssignorStrategy)
  5. Leader sends SyncGroupRequest with assignment to coordinator
  6. Coordinator distributes assignments to all members via SyncGroupResponse

Assignment strategies:
  RangeAssignor: partitions 0-3 → consumer 0, 4-7 → consumer 1 (default, simple)
  RoundRobinAssignor: interleaved assignment across all topics+partitions
  StickyAssignor: minimize partition movement on rebalance (preserves locality)
  CooperativeStickyAssignor: incremental cooperative rebalancing (Kafka 2.4+)

Rebalancing triggered by:
  - New consumer joins group (sends JoinGroupRequest)
  - Consumer leaves gracefully (LeaveGroupRequest)
  - Consumer crashes (heartbeat.interval.ms default 3s; session.timeout.ms default 45s)
  - New partitions added to topic

Classic (eager) rebalance problem:
  ALL consumers stop consuming → revoke ALL partition assignments → rejoin → re-assign
  "Stop the world" → can cause processing pause of 5–30 seconds
  → Consumer lag spikes during rebalance; downstream latency increases

Incremental cooperative rebalancing (CooperativeStickyAssignor):
  Phase 1: revoke only partitions that need to move
  Phase 2: reassign only those partitions to new owners
  Consumers NOT losing partitions continue consuming throughout
  → Dramatically reduces pause time; most consumers unaffected by one consumer joining
```

**Offset management:**
```
Consumer commits offsets to __consumer_offsets after processing:

Options:
  1. Auto-commit (enable.auto.commit=true, auto.commit.interval.ms=5000):
     - Simple but may reprocess up to 5 seconds of messages on crash
     - Not suitable when processing has side effects (DB writes, API calls)

  2. Manual commit after each record (slowest, safest):
     consumer.commitSync(offset)
     - One commit per message → slowest; at-most-once if committed before processing

  3. Manual commit after batch (recommended):
     for record in batch:
         process(record)
     consumer.commitSync()   # commit after entire batch processed
     - At-least-once: batch reprocessed on crash
     - Good balance of safety and throughput

  4. Async commit:
     consumer.commitAsync(callback)
     - Non-blocking; callback handles commit failure
     - Issue: if committed offset A, then B fails, retry of B could commit B > A
     → Only safe if commits are idempotent or you don't retry async commits

On consumer restart:
  - Read committed offset from __consumer_offsets
  - If no committed offset: auto_offset_reset = "earliest" (replay all) or "latest" (skip to now)
```

**G5:** Exactly-once in Kafka is scoped to the broker layer — end-to-end exactly-once (including consumer-side processing) requires idempotent consumers. A consumer that writes to a database on each message must use the Kafka offset as an idempotency key: before processing each record, execute `INSERT INTO processed_offsets (group_id, topic, partition, offset) VALUES (?, ?, ?, ?) ON CONFLICT DO NOTHING`, and only proceed with the business logic if the insert succeeds, wrapping both the insert and the business operation in the same database transaction. Without this, consumer restarts cause double-processing that Kafka's producer-level idempotence (which deduplicates at the broker, within a single producer session) cannot prevent. This is a different layer of the stack.

## Deep Dive 3: High Read Throughput (Zero-Copy)

**How Kafka achieves 5M messages/sec read:**

```
Naive file serving (standard read + send):
  disk → kernel buffer (DMA copy)
       → user space buffer (copy 1: kernel → user)
       → socket send buffer (copy 2: user → kernel)
       → NIC → network
  Total: 2 copies across kernel/user boundary; 2 context switches per I/O

Kafka zero-copy (sendfile syscall on Linux / TransferTo on Java):
  disk → kernel buffer (DMA copy)
       → NIC (copy via DMA, no CPU involvement) → network
  Total: 0 CPU copies; 1 context switch
  → 2–3× throughput improvement for read-heavy workloads
  → Enables Kafka to saturate a 10GbE NIC serving consumers

OS page cache leverage:
  Linux kernel caches file data in RAM (page cache) automatically
  Kafka consumers reading recent messages (hot data) hit the page cache
  → Disk I/O completely bypassed for hot reads
  → Read latency: ~0.1ms (RAM access) vs ~1ms+ (SSD) vs ~10ms (HDD)

Configuration to maximize page cache usage:
  Kafka process uses very little heap (256MB–1GB typical JVM heap)
  OS gets all remaining RAM for page cache
  Do NOT give Kafka large JVM heap; give it to the OS
  Example: 128GB server → JVM heap: 4GB, page cache: ~120GB
```

**Batching and compression:**
```
Producer batching:
  50 messages × 1KB each → 50KB batch → 1 network round trip
  Without batching: 50 round trips → 50× more syscalls

Compression (applied per batch):
  Snappy: 2–4× compression ratio; fast compression/decompression (good default)
  LZ4:    Similar ratio to snappy; faster, lower CPU → best for high-throughput
  Gzip:   4–6× compression ratio; higher CPU → use for storage-cost-sensitive topics
  Zstd:   Best compression ratio; tunable CPU/ratio tradeoff → Kafka 2.1+

Broker decompresses nothing (for snappy/lz4/zstd):
  Broker stores compressed batches as-is (pass-through)
  Consumer decompresses → same compression preserved end-to-end
  Exception: broker decompresses to validate and assign offsets → re-compresses

Effect: 1M msg/sec × 1KB × 3× compression = 333MB/sec stored instead of 1GB/sec
```

**G5:** Zero-copy only works when the compression codec on disk matches what the consumer requests. If the producer uses gzip and the consumer uses the same setting, the broker passes the compressed batch directly to the NIC via sendfile — zero CPU copies. But if the broker needs to recompress (e.g., broker-level compression override, or message format conversion between magic versions), it must decompress + recompress in user space, negating zero-copy entirely. In practice this happens silently when upgrading Kafka cluster versions with mixed message format epochs. Monitor `kafka.server:type=BrokerTopicMetrics,name=ReassignmentBytesInPerSec` and `RemoteFetchRequestRate` — a sudden increase in broker CPU with no producer/consumer change is the signal that zero-copy is being bypassed.

## Reliability and Operations

- **Leader failure:** ZooKeeper/KRaft detects leader failure via session expiry (ZooKeeper default: 6s session timeout). Controller elects new leader from ISR immediately. Producers and consumers receive `LeaderNotAvailableException`, retry automatically with backoff. Total failover time: typically 10–30 seconds (ZooKeeper path + leader registration + client retry). KRaft mode (Kafka 3.3+) reduces this to < 1 second (no ZooKeeper round-trip).
- **Under-replicated partitions:** If a partition's ISR shrinks below `min.insync.replicas`, Kafka rejects further writes (throws `NotEnoughReplicasException`). Alert aggressively on under-replicated partitions: it means you are one broker failure away from data loss. Root causes: disk full, network partition, slow broker (GC pause, I/O saturation).
- **Log compaction:** For changelog-style topics (user profile updates, config changes) where only the latest value per key matters. Background cleaner thread merges old segments, keeping only the latest record per key. Compacted topics can grow forever (no time-based expiry) — bounded by unique key cardinality. Used by: Kafka Connect, Kafka Streams KTables, service config replication.
- **Consumer lag monitoring:** Lag = leader_end_offset − consumer_committed_offset. Alert on absolute lag (> 100K messages) AND on lag growth rate (> 0, sustained). Growing lag means consumer can't keep up with producer rate. Action: scale up consumer group (add more consumers, up to num_partitions limit). Track lag on all consumer groups, not just your own — a forgotten consumer group accumulating lag can cause topic retention to extend indefinitely if retention is managed by consumer progress (not time).

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Append-only log | In-place update (B-tree) | Log: sequential I/O (1GB/sec SSD writes); B-tree: random I/O (slow at scale, 10–100× worse throughput per disk) |
| Pull-based consumers | Push-based | Pull: consumers control their own pace, no risk of being overwhelmed, can batch; Push: lower latency but risks overwhelming slow consumers |
| Per-partition ordering | Global ordering | Per-partition: linear horizontal scaling; Global: requires single partition = single-threaded write bottleneck |
| OS page cache | Explicit buffer management | OS cache: zero-copy optimization works; zero JVM GC pressure; Explicit: more predictable but complex and gives up sendfile |
| acks=all | acks=1 | acks=all: survives any single broker failure; 10–20% latency overhead; acks=1: faster, but message lost if leader fails before followers replicate |
| KRaft (no ZooKeeper) | ZooKeeper-based | KRaft: simpler ops, faster failover (< 1s vs 30s), single security boundary; ZooKeeper: battle-tested but separate cluster to operate |

## G5

1. **Exactly-once semantics (EOS) — full stack:** Kafka achieves EOS via three interlocking mechanisms: (a) Idempotent producer: broker assigns each producer a PID (producer ID) + epoch. Each message batch has a monotonically increasing sequence number per partition. Broker deduplicates retried batches within the same producer session. (b) Transactional API: producer calls `initTransactions()`, begins a transaction spanning multiple topic-partitions, and either commits or aborts atomically. Consumers with `isolation.level=read_committed` skip aborted records and uncommitted records in-progress. (c) Fenced producers: on producer restart, the new producer gets a higher epoch, and the broker rejects any in-flight writes from the old (lower-epoch) producer — prevents zombie producers from committing stale data after failover. The performance overhead is approximately 10–15% vs at-least-once, primarily from the two-phase commit to the transaction coordinator. Use EOS for billing pipelines; at-least-once + idempotent consumers for everything else.

2. **Tiered storage (Kafka 3.6+):** Local NVMe SSD on brokers is expensive — required for hot data (last 7 days) but wasteful for cold retention (30+ days for compliance). Tiered storage: Kafka automatically offloads segments older than `local.retention.ms` to a remote tier (S3 or GCS). Brokers retain only hot segments locally; consumers reading cold data are transparently served from S3 (broker fetches and caches). The local broker becomes a caching layer rather than the source of truth. Cost impact: NVMe SSD costs ~$0.10/GB/month; S3 costs ~$0.023/GB/month — 77% cost reduction for cold data. Trade-off: cold reads have 50–200ms latency (S3 round-trip) vs < 1ms (page cache). Design implication: topic retention policies (retention.ms) can be set to weeks or months without corresponding broker disk expansion.

3. **Multi-datacenter replication (MirrorMaker 2):** Replicate topics across Kafka clusters for DR or cross-region processing. MirrorMaker 2 (MM2) is built on Kafka Connect: a source connector reads from the upstream cluster and a sink connector writes to the downstream cluster. Key feature: offset translation. MM2 maintains a mapping of upstream offsets → downstream offsets, stored in an internal topic. When a consumer group fails over to the DR cluster, it can look up its upstream committed offset, translate it to the downstream equivalent, and resume near where it left off — typically losing only the last few seconds of processing. This is what enables near-zero RPO (recovery point objective) for Kafka-based pipelines in DR scenarios.

4. **Kafka Streams and table-stream duality:** Beyond simple message passing, Kafka enables stream processing without a separate cluster. Kafka Streams is an embedded library that reads from topics, processes (filter, map, join, aggregate), and writes to output topics — deployed as part of your application. KTable represents a topic as a changelog stream interpreted as a materialized table (latest value per key, backed by local RocksDB). Stream-KTable join: enrich payment events with user profile data from a KTable → personalized fraud signals, joined in milliseconds with no external DB call. The table is kept locally on each stream processing instance, partitioned by the same key scheme as the source topic. This dual nature (stream = log of events; table = snapshot of latest state) is Kafka's most powerful abstraction for building stateful pipelines with no external state store dependency.

5. **Quotas and multi-tenancy:** In a shared Kafka cluster (platform team's infrastructure serving 50 microservices), one misbehaving producer can saturate network and starve other services. Kafka quotas: per-client-id and per-user byte-rate limits, enforced by the broker using token buckets. If a client exceeds its quota, the broker delays the response by the calculated throttle time (rather than rejecting) — this applies back-pressure without breaking the client. Design a quota management service: operators configure quotas per team/service via an admin API, stored in ZooKeeper/KRaft, and synced to all brokers dynamically. Chargeback: Kafka JMX exposes `kafka.server:type=ClientMetrics,name=BytesInPerSec` per client-id — aggregate to monthly cost per team. Alert on quota violations (sustained throttling indicates a team needs a quota increase or a bug fix).
