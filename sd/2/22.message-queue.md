# 22. Distributed Message Queue (Kafka-like)

**Type:** infra
**Difficulty:** L5
**Frequency:** High — storage internals, distributed log, consumer groups

---

## Problem Statement

Design a distributed message queue like Apache Kafka. Producers publish messages to topics. Consumers subscribe to topics and process messages. The system must provide high throughput (millions of messages/sec), durability (no message loss), ordered delivery within a partition, and support multiple independent consumer groups replaying the same messages.

---

## Functional Requirements

**Must have:**
- Producers publish messages to a named topic
- Topics are divided into partitions for parallelism
- Messages within a partition are ordered and persistent
- Consumers read messages with an offset pointer (can replay from any offset)
- Multiple consumer groups can read the same topic independently (no interference)
- At-least-once delivery guarantee

**Should have:**
- Message retention: configurable by time (7 days) or size (1TB per partition)
- Consumer group rebalancing: when consumers join/leave, partitions are redistributed
- Exactly-once semantics (producer idempotency + transactional API)
- Compression (gzip, snappy, lz4) per message batch

**Out of scope:**
- Message routing / filtering by content (Kafka doesn't do this — see RabbitMQ)
- Dead letter queues
- FIFO across multiple partitions

---

## Non-Functional Requirements

- **Throughput:** 1M messages/sec write; 5M messages/sec read (multiple consumer groups)
- **Latency:** Write p99 < 5ms (after batch flush); Read p99 < 10ms
- **Durability:** Message persisted to disk before ACK
- **Availability:** 99.99% — message loss is catastrophic
- **Scale:** 1PB total storage across the cluster; horizontal scaling

---

## Capacity Estimation

```
Write: 1M messages/sec × 1KB avg = 1GB/sec ingest
  With 3x replication = 3GB/sec write throughput across cluster
  Each broker: 1Gbps network → handles 125MB/sec = 125K 1KB messages/sec
  1M messages/sec → need at least 8 brokers for write bandwidth

Read: 5M messages/sec × 1KB = 5GB/sec read throughput
  Kafka uses OS page cache heavily → read from memory when hot
  Hot data (last 24h): 1M msg/sec × 86400s × 1KB = 86GB/day → fits in RAM across cluster

Storage per partition:
  Assume 1000 partitions, 7-day retention
  1M msg/sec / 1000 partitions = 1000 msg/sec per partition
  Per partition: 1000 msg/sec × 1KB × 604800s = 604GB per partition replica → 604TB total
  With 3x replication: 1.8PB physical storage → 100+ storage nodes at 20TB each
```

## High-Level Architecture

```
[Producers] ──→ [Broker Cluster (N brokers)]
                      │
               [Topic: "payments"]
                  Partition 0: [Leader: Broker1] ← [Follower: Broker2, Broker3]
                  Partition 1: [Leader: Broker2] ← [Follower: Broker1, Broker3]
                  Partition 2: [Leader: Broker3] ← [Follower: Broker1, Broker2]
                      │
[ZooKeeper / KRaft] ──┘ (cluster metadata, leader election)
                      │
[Consumers (Group A)] ──→ [Consumer Group Coordinator (broker)]
[Consumers (Group B)] ──→ [Consumer Group Coordinator (broker)]
```

**Key insight:** Each partition is a sequential log. Producers write to the leader. Followers replicate. Consumers read from leader (strong consistency) or follower (relaxed). Consumer groups each track their own offsets independently.

## API Design

```python
# Producer
producer = KafkaProducer(bootstrap_servers=['broker1:9092', 'broker2:9092'])

# Send a message (async, batched)
producer.send(topic="payments", key=b"user-123", value=b'{"amount": 50}')
producer.flush()  # wait for all pending messages to be sent

# Consumer
consumer = KafkaConsumer(
    "payments",
    bootstrap_servers=['broker1:9092'],
    group_id="fraud-detection-service",
    auto_offset_reset="earliest"  # start from beginning if no committed offset
)

for message in consumer:
    process(message.value)
    consumer.commit()  # commit offset after processing
```

**Admin API:**
```
# Create topic with 16 partitions, replication factor 3
kafka-topics.sh --create --topic payments --partitions 16 --replication-factor 3

# Describe topic
kafka-topics.sh --describe --topic payments
→ shows leaders, ISR (in-sync replicas), offsets per partition
```

## Data Model

**On-disk format: The Log**
```
Each partition is stored as a series of immutable segment files:
  /kafka-data/payments-0/
    00000000000000000000.log   ← segment starting at offset 0
    00000000000000000000.index ← sparse index: offset → file_offset
    00000000000015000000.log   ← segment starting at offset 15M
    00000000000015000000.index
    ...

Message format in .log file (packed, sequential):
  [offset: 8B][message_size: 4B][crc: 4B][magic: 1B][attributes: 1B]
  [timestamp: 8B][key_size: 4B][key: N bytes][value_size: 4B][value: M bytes]

Index file (.index):
  [relative_offset: 4B][file_position: 4B]  ← every 4096 messages
  Binary search to find file position for a given offset
```

**Consumer offset storage:**
```
Consumer group offsets stored in internal Kafka topic: __consumer_offsets
  Key: (group_id, topic, partition)
  Value: offset + metadata
Periodically committed by consumer after processing
```

## Deep Dive 1: Write Path and Durability

**Producer write:**
```
1. Producer SDK batches messages in memory (batch.size, linger.ms configurable)
2. Producer hashes key to determine partition: partition = hash(key) % num_partitions
   (No key → round-robin across partitions)
3. Sends batch to partition leader broker
4. Leader writes to local disk:
   a. Appends to MemTable (OS write buffer / page cache — not guaranteed durable yet)
   b. Appends to partition log file
5. ACK modes (acks config):
   acks=0: fire and forget (no durability guarantee)
   acks=1: leader wrote to local disk → ACK (leader failure → message lost)
   acks=all: all ISR (in-sync replicas) acked → ACK (highest durability)
6. Followers pull from leader and replicate
7. Once follower replicates, it's in ISR (in-sync replica set)
```

**Durability guarantee with acks=all:**
```
ISR = {Broker1 (leader), Broker2, Broker3}
Write:
  1. Leader writes to log
  2. Followers pull message from leader
  3. Followers write to their logs and send ACK to leader
  4. Leader adds message to committed log (high watermark advances)
  5. Leader sends ACK to producer

If a broker in ISR is slow: remove from ISR after replica.lag.time.max.ms
  → don't wait for slow replica → write performance not impacted
  → min.insync.replicas=2 ensures at least 2 replicas before ack
```

**fsync and performance:**
```
Kafka relies on OS page cache (not calling fsync after every write)
  → Very fast writes (memory bandwidth vs disk bandwidth)
  → If OS crashes: messages in page cache can be lost

Recovery: broker recovers from leader or other replicas (not from disk fsync)
→ Replication provides durability, not per-message fsync
→ This is a key Kafka design decision: availability over strict persistence

For strict durability: flush.messages=1 (fsync every message) → 10-100x slower
```

## Deep Dive 2: Consumer Group and Partition Rebalancing

**Consumer group protocol:**
```
Consumer group: a set of consumers that together consume all partitions of a topic
  Group "fraud-detection" has 4 consumers → each gets 4 of the 16 partitions

Assignment:
  Group coordinator (one broker per group): tracks membership
  On first consumer join:
    → Coordinator selects a group leader (first consumer to join)
    → Leader receives list of all members + subscriptions
    → Leader computes partition assignment (round-robin or sticky)
    → Leader sends assignment to coordinator
    → Coordinator distributes to all members

Rebalancing triggered by:
  - New consumer joins
  - Consumer leaves (heartbeat timeout → 10 seconds default)
  - New partition added to topic

During rebalance: ALL consumers in the group pause consumption (stop-the-world)
→ This is a known Kafka limitation

Mitigation: incremental cooperative rebalancing (Kafka 2.4+)
  → Only reassign the partitions that need to move, not all partitions
  → Most consumers keep their partitions, no full pause
```

**Offset management:**
```
Consumer commits offsets to __consumer_offsets after processing
  Commit frequency tradeoff:
    Auto-commit every 5 seconds: simple but may reprocess 5 seconds of messages on crash
    Manual commit after each message: exactly-once (with idempotent processing) but slower
    Manual commit after each batch: good balance

On consumer restart: read committed offset → resume from there
If no offset: auto_offset_reset = "earliest" (from beginning) or "latest" (from now)
```

## Deep Dive 3: High Read Throughput (Zero-Copy)

**How Kafka achieves 5M messages/sec read:**

```
Naive file read:
  disk → kernel buffer (DMA) → user buffer (copy 1) → socket buffer (copy 2) → NIC → network
  2 copies across kernel/user boundary

Kafka zero-copy (sendfile syscall):
  disk → kernel buffer (DMA) → NIC (copy 1) → network
  → Only 1 copy; kernel handles transfer without going to user space
  → 2-3x throughput improvement for read-heavy workloads

OS page cache:
  Hot data (recent messages) lives in OS page cache (RAM)
  Read hits page cache → no disk I/O at all
  → Consumers reading recent data: effectively reading from RAM (< 0.1ms latency)
  → This is why Kafka allocates all available RAM to OS, not JVM heap
```

## Reliability and Operations

- **Leader failure:** ZooKeeper/KRaft detects leader failure (session timeout ~6s). Elects new leader from ISR. Consumers and producers reconnect automatically. Failover time: 10-30 seconds.
- **Under-replicated partitions:** Alert if partition's ISR < min.insync.replicas. This means durability is degraded — potential message loss on leader failure.
- **Log compaction:** For "event sourcing" use cases where you only need latest value per key (e.g., user profile updates). Background process removes superseded messages for the same key. Non-compacted topics: time/size-based retention. Compacted topics: keep latest per key forever.
- **Consumer lag monitoring:** Track lag = leader_end_offset - consumer_committed_offset. Alert on growing lag (consumer not keeping up with production rate). This is a leading indicator of data pipeline health.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Append-only log | In-place update | Log: sequential I/O (fast); In-place: random I/O (slow at scale) |
| Pull-based consumers | Push-based | Pull: consumers control pace; Push: lower latency, risk overwhelming slow consumer |
| Per-partition ordering | Global ordering | Per-partition: scales horizontally; Global: requires single partition = throughput bottleneck |
| OS page cache | Explicit buffer management | OS cache: zero-copy optimization; Explicit: more predictable but complex |
| acks=all | acks=1 | acks=all: safest; acks=1: faster, risk of message loss on leader failure |

## L5 Additions

1. **Exactly-once semantics (EOS):** Kafka achieves exactly-once via: (a) Idempotent producer: broker deduplicates retried messages using sequence numbers per partition. (b) Transactional API: producer opens transaction spanning multiple topics — either all messages committed or none. (c) Consumer reads committed messages only (isolation.level=read_committed). Use case: billing pipeline — charge exactly once. Implementation: transaction coordinator per broker manages in-flight transactions. Two-phase commit across partition leaders. Performance overhead: ~10% vs at-least-once.

2. **Tiered storage (Kafka 3.6+):** Local disk is expensive (NVMe SSDs for performance). Long-term retention (6 months+) is stored in S3. Architecture: hot data (< 7 days) on local brokers. Cold data automatically offloaded to S3. Consumers reading cold data: broker fetches from S3 transparently. Read cache for recently fetched S3 segments. Reduces broker disk cost by 80% for long-retention use cases. Trade: cold reads have higher latency (100ms vs 1ms).

3. **Multi-datacenter replication (MirrorMaker):** Replicate topics from one Kafka cluster to another for DR or cross-region processing. MirrorMaker 2: built on Kafka Connect framework. Topic replication with offset translation (consumer offsets mapped between clusters). Allows consumer groups to failover to replica cluster and resume near their original offset. Used for: global event streaming pipelines, disaster recovery, multi-region analytics.

4. **Kafka Streams and table-stream duality:** Beyond simple message passing, Kafka enables stream processing. Kafka Streams: embedded library that reads from topics, processes (transform, join, aggregate), writes to output topics. KTable: changelog stream interpreted as a table (latest value per key). Stream-table join: enrich stream events with reference data from a table. Example: enrich payment events with user profile data (KTable) → personalized fraud detection. No separate cluster needed — runs inside your application.

5. **Quotas and multi-tenancy:** In shared Kafka clusters (platform team's Kafka), one misbehaving producer can saturate the network and starve other services. Kafka quotas: per-client-id or per-user byte rate limits. Broker enforces with token bucket: if client exceeds quota, broker delays response (throttling). Design a quota management service: operators configure quotas per team/service in a central store, quotas sync to all brokers periodically. Chargeback: meter bytes produced/consumed per client → monthly cost allocation.
