# 24. API Gateway

**Type:** infra
**Difficulty:** L4
**Frequency:** High — reverse proxy, auth, rate limiting, routing

---

## Problem Statement

Design an API gateway that sits in front of a fleet of microservices. The gateway handles authentication, authorization, rate limiting, routing to backend services, request/response transformation, SSL termination, and observability. It must handle 500K requests/sec with < 5ms added latency.

---

## Functional Requirements

**Must have:**
- Route incoming requests to the appropriate backend service
- Authentication: validate API keys, JWT tokens, OAuth2
- Rate limiting: per-client, per-endpoint
- SSL/TLS termination at gateway
- Request/response logging

**Should have:**
- Request transformation: header injection, URL rewriting, request body transformation
- Circuit breaker: stop forwarding to unhealthy backends
- Canary routing: route X% of traffic to new service version
- Plugin/middleware architecture: extensible without core changes

**Out of scope:**
- Service discovery (separate component — gateway reads from service registry)
- API documentation (Swagger/OpenAPI)
- Developer portal

---

## Non-Functional Requirements

- **Throughput:** 500K requests/sec
- **Latency:** Added latency < 5ms p99 (gateway overhead only)
- **Availability:** 99.999% — gateway is the single entry point; its downtime is total outage
- **Scalability:** Horizontally scalable; no shared state between gateway instances
- **Security:** No auth bypass; secure by default

---

## Capacity Estimation

```
500K req/sec
Each request: auth check (~0.5ms Redis lookup), rate limit check (~0.5ms), routing (~0.1ms), backend call (not gateway's time)
Gateway CPU: 500K × 5ms / 1000ms = 2,500 req/ms/core; modern server ~10 cores → needs ~250 servers
  (Heavy use of async/event loop — Nginx/Envoy handle ~100K req/sec per core)
  → 5 gateway servers for throughput

Network: 500K × 1KB (avg request) = 500MB/sec inbound; same for outbound = 1GB/sec
  → 10Gbps NIC per server × 5 servers = 50Gbps capacity → sufficient
```

## API Design

**Gateway configuration (declarative):**
```yaml
routes:
  - path: /api/v1/users
    service: user-service
    methods: [GET, POST]
    auth: required
    rate_limit:
      per_user: 100/min
      per_ip: 1000/min
    plugins:
      - name: request-id-injector
      - name: cors
        config: { allow_origins: ["*"] }

  - path: /api/v1/internal/*
    service: internal-service
    auth: service-to-service  # mTLS only, no user tokens
    rate_limit:
      per_service: 10000/min

  - path: /api/v2/products
    service: product-service
    canary:
      service: product-service-v2
      percentage: 10
```

**Admin API:**
```
GET  /admin/routes → list all routes and configurations
POST /admin/routes → add a new route
PUT  /admin/routes/{id} → update route
DELETE /admin/routes/{id} → remove route

GET  /admin/health → gateway health + backend health checks
GET  /admin/metrics → prometheus-compatible metrics endpoint
```

## Data Model

**Gateway state (minimal — stateless design):**
```
Routes:    stored in etcd / Consul; cached in-process, refreshed every 30s
Auth keys: validated against Auth Service or verified locally (JWT with public key)
Rate limit counters: Redis Cluster (shared across all gateway instances)
Circuit state: in-process per gateway instance (not shared — eventual consistency acceptable)
```

**Redis rate limit schema:**
```
Key: ratelimit:{user_id}:{endpoint}:{window}
Type: Integer (INCR)
TTL: window_seconds × 2

Key: ratelimit:{ip}:{window_minute}
Type: Integer
TTL: 120s
```

## Deep Dive 1: Request Processing Pipeline

**Per-request processing steps (in-process, async):**

```
1. TCP/TLS handshake (< 1ms for resumed sessions)
2. HTTP/2 demux (connection → stream → request)
3. Request parsing (headers, method, path)
4. Route matching (trie lookup by path prefix, O(log N))
5. Plugin pipeline execution (in order):
   a. Request ID injection (add X-Request-ID header if missing)
   b. Auth middleware:
      - Extract Bearer token from Authorization header
      - JWT: verify signature locally with cached public key (no network call)
      - API key: lookup in local hash table (synced from Auth Service)
      - Service-to-service: verify mTLS client certificate
      - If invalid: return 401 immediately (< 0.5ms, no backend call)
   c. Rate limit check:
      - Redis INCR + EXPIRE (Lua script, atomic, ~0.5ms)
      - If exceeded: return 429 with Retry-After header
   d. Request transformation (add/remove/modify headers)
6. Backend selection (service registry lookup, load balancing)
7. Forward request to backend (non-blocking async I/O)
8. Receive response from backend
9. Response transformation
10. Log request+response (async, fire-and-forget to Kafka)
11. Return response to client
```

**Critical path optimization:**
- JWT verification is local (no network call) — use public key cached in memory
- Route lookup uses radix trie — O(path_length) time
- Rate limit: single Redis round trip with Lua script
- Everything async/non-blocking (no thread blocking on network I/O)

## Deep Dive 2: Circuit Breaker

**Problem:** If backend service is failing, continuing to forward requests wastes time and makes things worse.

**Circuit breaker states:**
```
CLOSED (normal):
  - Forward requests to backend
  - Track error rate in sliding window (last 100 requests)
  - If error rate > 50%: transition to OPEN

OPEN (failure):
  - Immediately return 503 (no backend call)
  - After 30s timeout: transition to HALF-OPEN

HALF-OPEN (testing recovery):
  - Let 1 request through
  - If success: transition to CLOSED
  - If failure: back to OPEN
```

**Implementation:**
```python
class CircuitBreaker:
    def __init__(self, threshold=0.5, window=100, timeout=30):
        self.state = "CLOSED"
        self.failure_window = deque(maxlen=window)  # sliding window of success/fail
        self.last_state_change = time.time()
        self.threshold = threshold
        self.timeout = timeout

    def before_call(self):
        if self.state == "OPEN":
            if time.time() - self.last_state_change > self.timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitOpenError()

    def after_call(self, success):
        self.failure_window.append(success)
        error_rate = self.failure_window.count(False) / len(self.failure_window)
        if error_rate > self.threshold and self.state == "CLOSED":
            self.state = "OPEN"
            self.last_state_change = time.time()
        elif success and self.state == "HALF_OPEN":
            self.state = "CLOSED"
        elif not success and self.state == "HALF_OPEN":
            self.state = "OPEN"
            self.last_state_change = time.time()
```

## Deep Dive 3: Canary Deployments

**Routing N% of traffic to a new version:**

```python
def select_backend(route, request):
    if route.canary and random.random() < route.canary.percentage / 100:
        # Sticky canary: hash user_id to ensure consistent routing
        if route.canary.sticky:
            user_hash = hash(request.user_id) % 100
            if user_hash < route.canary.percentage:
                return route.canary.service
        else:
            return route.canary.service
    return route.service
```

**Metrics comparison:**
- Track error rate, latency, and custom business metrics separately per version
- Canary dashboard: side-by-side comparison of v1 vs v2
- Auto-rollback: if canary error rate > 5% or p99 latency > 500ms → revert canary % to 0

## Reliability and Operations

- **Gateway instance failure:** Stateless design → any instance can handle any request. Load balancer detects failure (health check timeout) and stops routing. Recovery: replace instance in < 60s.
- **Redis failure:** Rate limiting fails open (allow all requests). Log that rate limiting is disabled. Auth still works (JWT is verified locally).
- **Thundering herd on deploy:** Config reload via hot reload (SIGHUP in Nginx). New routes take effect without connection drops. In-flight requests complete normally.
- **Observability:** Every request generates a structured log entry (async). Prometheus metrics: request rate, error rate, latency histograms per route. Distributed tracing: add X-Request-ID header, trace ID, span ID to all backend calls.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Stateless gateway | Stateful (session affinity) | Stateless: horizontal scale, no SPOF; Stateful: simplifies some auth patterns |
| Local JWT verification | Remote auth service call | Local: fast (no network); Remote: can revoke tokens instantly |
| In-process circuit breaker | Distributed circuit state | In-process: fast, no coordination; Distributed: consistent state across instances |
| Redis rate limiting | In-memory per-instance | Redis: accurate distributed; In-memory: inaccurate but zero latency |

## L5 Additions

1. **GraphQL gateway:** REST gateway is well-understood. GraphQL adds complexity: a single POST /graphql request can touch many backends. Gateway must: parse GraphQL query → build execution plan (which subgraph/service handles which type) → dispatch sub-queries in parallel → merge results. This is Apollo Federation or AWS AppSync. Key challenge: schema stitching (combine multiple service schemas into one). Performance: parallel sub-query dispatch with timeout and partial result on failure. Complexity: schema changes require coordination across teams.

2. **gRPC transcoding:** Many internal services use gRPC. External clients often use REST. Gateway performs HTTP/JSON → gRPC transcoding: parse REST request → map to proto message → call gRPC service → convert proto response → return JSON. Mapping defined in .proto service annotations (google.api.http option). This is what Google's Cloud Endpoints and Envoy's gRPC-JSON transcoding do. Reduces external surface area while keeping internal services gRPC-native.

3. **Request signing and replay protection:** For financial or critical APIs, require HMAC-signed requests. Client signs request body with a shared secret. Gateway verifies signature before forwarding. Replay protection: include a nonce (one-time value) and timestamp. Gateway: check nonce not seen before (Redis SET NX with 5-minute TTL), check timestamp within 5-minute window. This prevents man-in-the-middle attacks from replaying captured requests. Used by AWS APIs, Stripe API, Twilio.

4. **Dynamic routing and A/B testing at scale:** Beyond simple canary: route based on arbitrary request attributes. Customer segment → feature flag → service version. Architecture: gateway reads feature flag state from cache (synced from LaunchDarkly or internal flag service). Request comes in with user_id → look up feature flag for that user → route to corresponding service version. Latency: feature flag lookup must be < 1ms (in-memory cache with 10s TTL). This enables instant rollout/rollback per customer segment without code deploy.

5. **Service mesh vs API gateway:** Clarify the difference. API gateway: north-south traffic (external clients → internal services). Service mesh (Istio, Linkerd): east-west traffic (service-to-service inside the cluster). The gateway handles external auth, rate limiting, and routing. The mesh handles service discovery, mTLS between services, retries, and observability for internal calls. In mature architectures, you have both. Design implication: gateway doesn't need to handle internal service-to-service communication details; that's the mesh's job.
