# 24. API Gateway

**Type:** infra
**Difficulty:** L4
**Frequency:** High — reverse proxy, auth, rate limiting, routing

---

## Problem Statement

Design an API gateway that sits in front of a fleet of microservices. The gateway handles authentication, authorization, rate limiting, routing to backend services, request/response transformation, SSL termination, and observability. It must handle 500K requests/sec with < 5ms added latency.

---

## Functional Requirements

**Must have:**
- Route incoming requests to the appropriate backend service
- Authentication: validate API keys, JWT tokens, OAuth2
- Rate limiting: per-client, per-endpoint
- SSL/TLS termination at gateway
- Request/response logging

**Should have:**
- Request transformation: header injection, URL rewriting, request body transformation
- Circuit breaker: stop forwarding to unhealthy backends
- Canary routing: route X% of traffic to new service version
- Plugin/middleware architecture: extensible without core changes

**Out of scope:**
- Service discovery (separate component — gateway reads from service registry)
- API documentation (Swagger/OpenAPI)
- Developer portal

---

## Non-Functional Requirements

- **Throughput:** 500K requests/sec
- **Latency:** Added latency < 5ms p99 (gateway overhead only)
- **Availability:** 99.999% — gateway is the single entry point; its downtime is total outage
- **Scalability:** Horizontally scalable; no shared state between gateway instances
- **Security:** No auth bypass; secure by default

---

## Capacity Estimation

```
Throughput: 500K req/sec

Gateway overhead per request (tight budget):
  TLS handshake:           0ms (resumed session via session tickets)
  Route matching:          0.1ms (radix trie, O(path_length))
  JWT verification:        0.2ms (RSA/ECDSA verify, cached public key)
  Rate limit check:        0.5ms (Redis INCR round trip at < 0.3ms + margin)
  Header transformation:   0.05ms
  Total gateway overhead:  ~1ms average; < 5ms p99

Server sizing:
  Nginx / Envoy: ~100K req/sec per CPU core (async event loop, no thread blocking)
  Target: < 5ms latency with bursty traffic → need headroom
  500K req/sec / 100K per core = 5 cores needed
  Production sizing: 3 servers × 4 cores each = 12 cores (2.4× headroom for bursts)
  With autoscaling: min 3 servers, max 10 servers (Kubernetes HPA on CPU/req-rate)

Network:
  500K req/sec × 2KB avg (req + response headers) = 1GB/sec = 8Gbps
  3 servers × 10GbE NIC = 30Gbps capacity → sufficient with headroom

Redis (rate limiting):
  500K rate limit checks/sec × 1 INCR per check = 500K INCR/sec
  Redis single-thread: ~1M ops/sec → 2-node Redis cluster is sufficient
  Each INCR + EXPIRE = 2 commands → use Lua script for atomicity (1 round trip)

Memory:
  JWT public keys: 1–10 keys in memory → negligible
  Route table: 10K routes × 500B each = 5MB → fits in L3 cache
  Circuit breaker state: 1K services × 100B each = 100KB → in-process
```

## API Design

**Gateway configuration (declarative, version-controlled):**
```yaml
routes:
  - path: /api/v1/users
    service: user-service
    methods: [GET, POST]
    auth: required               # JWT or API key required
    rate_limit:
      per_user: 100/min          # per authenticated user
      per_ip: 1000/min           # per IP (for unauthenticated)
      burst: 20                  # allow short bursts above rate
    plugins:
      - name: request-id-injector
      - name: cors
        config: { allow_origins: ["https://app.example.com"] }
      - name: request-logging
        config: { log_body: false, log_headers: ["X-Request-ID"] }

  - path: /api/v1/internal/*
    service: internal-service
    auth: service-to-service     # mTLS client certificate only; no user tokens
    rate_limit:
      per_service: 10000/min     # per calling service identity

  - path: /api/v2/products       # canary deployment: 10% to new version
    service: product-service
    canary:
      service: product-service-v2
      percentage: 10
      sticky: true               # same user always gets same version

  - path: /health                # public health check endpoint
    service: null
    auth: none
    handler: static
    response: { status: 200, body: "ok" }
```

**Admin API (for dynamic reconfiguration without restart):**
```
GET    /admin/routes              → list all routes + their current config
POST   /admin/routes              → add new route (takes effect within 5s)
PUT    /admin/routes/{id}         → update route config
DELETE /admin/routes/{id}         → remove route

GET    /admin/health              → gateway cluster health + per-backend health
GET    /admin/metrics             → Prometheus-format metrics endpoint
POST   /admin/circuit-breakers/{service}/reset → manually reset a tripped circuit breaker
```

## Data Model

**Gateway state (designed to be minimal — stateless by default):**
```
Routes:          etcd / Consul; in-process copy refreshed every 30s + watch for changes
JWT public keys: JWKS endpoint fetched on startup + refreshed every 1h (or on 401 from backend)
                 Stored in-process: map[kid] → *rsa.PublicKey or *ecdsa.PublicKey
API keys:        Synced from Auth Service to local in-memory hash map (refresh every 60s)
                 Allows auth without a network call per request
Rate limit data: Redis Cluster (shared across all gateway instances for accuracy)
Circuit state:   In-process per gateway instance (not synchronized — acceptable)
```

**Redis rate limit schema (sliding window log):**
```
Key:   ratelimit:{user_id}:{endpoint_hash}:{window_minute}
Type:  Integer (INCR counter)
TTL:   120 seconds (2× window for safety; allows counting across window boundaries)
Value: count of requests in this window

Key:   ratelimit:{ip}:{endpoint_hash}:{window_minute}
Type:  Integer
TTL:   120 seconds

Lua script (atomic check + increment, 1 round trip):
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])
  local current = tonumber(redis.call("INCR", key))
  if current == 1 then
    redis.call("EXPIRE", key, window * 2)
  end
  if current > limit then
    return -1  -- rate limited
  end
  return current
```

## Deep Dive 1: Request Processing Pipeline

**Per-request processing steps (in-process, fully async):**

```
1. TCP accept + TLS handshake (< 1ms for resumed sessions via TLS 1.3 0-RTT)
   Session ticket: client presents session ticket → server decrypts → no full handshake
   OCSP stapling: certificate validity proof served with TLS handshake, no client CA lookup

2. HTTP/2 demux: single TCP connection → multiple concurrent streams (requests)
   h2: multiplexed streams; h3 (QUIC): UDP-based, no head-of-line blocking

3. Request parsing: HTTP method, URL, headers (< 0.05ms for typical headers)

4. Route matching: radix trie lookup by URL path prefix → O(path_length)
   Route table: loaded from etcd at startup, hot-reloaded on change notification
   Unknown route → 404 immediately (no backend call)

5. Plugin pipeline (sequential middleware, short-circuit on error):

   a. Request ID injection (always first):
      Check X-Request-ID header → if absent, generate UUID4
      Add to request headers forwarded to backend
      Add to response headers for client correlation

   b. Authentication middleware:
      Extract credential from request:
        - Authorization: Bearer {jwt}    → JWT path
        - X-API-Key: {key}              → API key path
        - mTLS client cert              → service-to-service path
      JWT verification (no network call):
        1. Decode header → extract kid (key ID)
        2. Look up public key by kid in in-process key map
        3. Verify signature: ECDSA P-256 verify → ~0.2ms
        4. Check claims: exp, nbf, iss, aud
        5. If valid → extract user_id, scopes; inject as X-User-ID, X-Scopes headers
      API key verification:
        Local hash map lookup: key_hash → { user_id, scopes, is_active }
        O(1) lookup → ~0.01ms
      Invalid credential: return 401 immediately, no backend call, log attempt

   c. Authorization:
      Check: does this user's scope allow this method on this path?
      Example: scope "orders:read" allows GET /api/v1/orders but not POST
      Policy: loaded from route config; evaluated in-process
      Unauthorized: return 403 immediately

   d. Rate limit check (Redis round-trip):
      Lua script: INCR counter + check against limit → single round-trip (~0.3ms)
      Exceeded: return 429 with headers:
        X-RateLimit-Limit: 100
        X-RateLimit-Remaining: 0
        X-RateLimit-Reset: {unix_timestamp}
        Retry-After: {seconds}

   e. Request transformation:
      Add headers: X-User-ID, X-Request-ID, X-Forwarded-For, X-Forwarded-Proto
      Remove sensitive headers: Authorization (don't forward JWT to backend)
      URL rewriting: /api/v1/users/{id} → /users/{id} (strip version prefix)

6. Backend selection:
   Service registry lookup: service name → list of healthy instances + weights
   Load balancing: round-robin, least-connections, or consistent hashing (sticky sessions)

7. Forward request to backend: async HTTP/gRPC call (non-blocking I/O)
   Connection pool: maintain persistent connections to each backend pod (avoid TCP handshake per request)
   Timeout: 30s default; configurable per route

8. Receive response; handle circuit breaker update

9. Response transformation: add CORS headers, security headers (X-Frame-Options, etc.)

10. Async logging (fire-and-forget to Kafka — does not block response):
    Structured log: {request_id, user_id, method, path, status, latency_ms, backend_latency_ms, cache_status}

11. Return response to client
```

**Critical path latency breakdown:**
```
Step 1 (TLS resumed):   < 1ms
Step 4 (route match):   0.05ms
Step 5b (JWT verify):   0.2ms
Step 5d (rate limit):   0.3ms
Step 6–8 (proxy):       backend latency (not gateway's budget)
Total gateway overhead: ~0.6ms average, < 5ms p99
```

**G5:** JWT verification without a network call is fast and correct under normal conditions, but has a revocation blindspot: a revoked JWT (user logged out, password changed, account suspended) remains valid until expiry. Short-lived tokens (15-minute expiry) minimize this window. For immediate revocation (compromised token), use a Redis blocklist: `SISMEMBER jwt_blocklist {jti}` adds ~0.3ms per request. The design choice is: (a) short-lived tokens (15min) + no blocklist → simple, fast, but 15-minute revocation window; (b) long-lived tokens (24h) + Redis blocklist → immediate revocation but requires Redis to be on the critical path. A middle ground: use Redis blocklist only for the high-security subset of endpoints (account settings, payment initiation), and rely on short token expiry for everything else — reducing Redis calls to < 5% of requests while covering the highest-risk revocation scenarios.

## Deep Dive 2: Circuit Breaker

**Problem:** If a backend service is failing (returning 5xx, timing out), continuing to forward requests wastes latency budget and makes the failure worse (keeps the service under load instead of letting it recover).

**Circuit breaker state machine:**
```
CLOSED (normal operation):
  Forward all requests to backend
  Track success/failure in a sliding window (last 100 requests, or last 10 seconds)
  If failure rate > 50% (or error count > threshold): transition to OPEN
  Error types that count: 5xx responses, connection refused, timeout

OPEN (backend considered down):
  Immediately return 503 Service Unavailable — no backend call
  Latency: ~0.1ms (no network I/O)
  After timeout (default 30s): transition to HALF-OPEN

HALF-OPEN (testing recovery):
  Allow exactly 1 probe request through to backend
  If probe succeeds: transition to CLOSED (full traffic restored)
  If probe fails: back to OPEN (reset timeout, try again in 30s)
  All other requests during HALF-OPEN: return 503 (not the probe request)
```

**Implementation:**
```python
from collections import deque
import threading
import time

class CircuitBreaker:
    def __init__(self, service: str, threshold: float = 0.5,
                 window: int = 100, timeout_s: float = 30.0):
        self.service = service
        self.state = "CLOSED"
        self.failure_window = deque(maxlen=window)  # sliding window: True=success, False=failure
        self.last_state_change = time.monotonic()
        self.threshold = threshold       # failure rate threshold to open
        self.timeout_s = timeout_s       # seconds before trying HALF-OPEN
        self._lock = threading.Lock()
        self._probe_in_flight = False

    def before_call(self) -> None:
        """Called before making backend request. Raises CircuitOpenError if open."""
        with self._lock:
            if self.state == "OPEN":
                if time.monotonic() - self.last_state_change > self.timeout_s:
                    if not self._probe_in_flight:
                        self.state = "HALF_OPEN"
                        self._probe_in_flight = True
                        return  # allow this probe through
                raise CircuitOpenError(f"Circuit open for {self.service}")
            # CLOSED or HALF_OPEN probe: allow through

    def after_call(self, success: bool) -> None:
        """Called after backend responds (success=True) or fails (success=False)."""
        with self._lock:
            if self.state == "HALF_OPEN":
                self._probe_in_flight = False
                if success:
                    self.state = "CLOSED"
                    self.failure_window.clear()
                else:
                    self.state = "OPEN"
                    self.last_state_change = time.monotonic()
                return

            self.failure_window.append(success)
            if len(self.failure_window) >= 10:  # minimum window before evaluating
                failure_rate = self.failure_window.count(False) / len(self.failure_window)
                if failure_rate > self.threshold and self.state == "CLOSED":
                    self.state = "OPEN"
                    self.last_state_change = time.monotonic()
```

**Per-instance vs distributed circuit state:**
```
Per-instance (current design):
  Each gateway instance has its own CircuitBreaker per backend service
  Instance 1 may see backend as OPEN while Instance 2 sees CLOSED
  → Different users get different behavior; inconsistency lasts at most timeout_s

Distributed circuit state (Redis-backed):
  All instances share one state per service
  More consistent; but Redis adds ~0.3ms to every request (on the critical path)

Recommendation: per-instance is correct for most cases
  - Gateway instances are behind LB; users randomly distributed
  - All instances will open roughly simultaneously if backend is failing
  - Only matters if backend is "flaky for some callers" (split brain scenario)
  - Distributed adds latency for a problem that rarely matters in practice
```

**G5:** A circuit breaker in the gateway protects backends from overload, but it doesn't solve the fundamental problem of cascading failures when multiple backends share a resource (a database, downstream API). A more robust approach is bulkhead isolation: assign separate connection pools (and separate circuit breakers) per backend, so that a slow payment-service doesn't exhaust the shared HTTP client pool and prevent requests to user-service from getting connections. Envoy implements this as per-cluster circuit breaking: `max_connections`, `max_pending_requests`, `max_requests`, `max_retries` — hard limits per upstream cluster. Exceeding `max_pending_requests` immediately returns 503 rather than queuing indefinitely. Without bulkheads, a slow backend causes latency to accumulate in the gateway's connection pool, eventually stalling all backends regardless of their individual health.

## Deep Dive 3: Canary Deployments

**Routing N% of traffic to a new service version:**

```python
def select_backend(route: Route, request: Request) -> ServiceInstance:
    """Selects which version of a service to route to, with optional canary."""
    if not route.canary:
        return load_balance(route.service, request)

    if route.canary.sticky:
        # Hash-based sticky routing: same user always gets same version
        # Deterministic: doesn't require session state
        user_bucket = xxhash(request.user_id) % 100
        if user_bucket < route.canary.percentage:
            return load_balance(route.canary.service, request)
        return load_balance(route.service, request)
    else:
        # Random routing: pure percentage, no stickiness
        # Each request independently random
        if random.randint(0, 99) < route.canary.percentage:
            return load_balance(route.canary.service, request)
        return load_balance(route.service, request)
```

**Why sticky matters:**
```
Without sticky: user A hits v1 on request 1, v2 on request 2, v1 on request 3
  → Inconsistent UI state (user sees different results for same query)
  → Makes debugging hard (error on v2, but next request goes to v1 → "it works now")
  → Cannot A/B test behavioral changes that span multiple requests

With sticky (hash(user_id) % 100 < canary_pct):
  → Same user always in same bucket → consistent experience
  → Can track engagement metrics (conversion rate) per version per user cohort
  → Easy rollback: set percentage = 0 → all users route to stable version
  → Deterministic: no session state needed; works across all gateway instances
```

**Automated rollback:**
```
Canary monitoring (runs every 30 seconds):
  Compare metrics between stable (v1) and canary (v2):
    - Error rate (5xx responses / total): alert if canary error_rate > stable_error_rate + 2%
    - p99 latency: alert if canary p99 > stable p99 × 1.5
    - Business metric: conversion rate, add-to-cart rate (if available)

Auto-rollback trigger:
  if canary.error_rate > threshold OR canary.p99_latency > threshold:
      route.canary.percentage = 0  # immediately stops canary traffic
      alert("Canary auto-rolled back for {route.service}: error_rate={canary.error_rate}")

Progression (if canary is healthy):
  Gradually increase percentage: 1% → 5% → 10% → 25% → 50% → 100%
  Minimum soak time at each stage: 30 minutes (gather statistical significance)
```

**G5:** Canary routing at the gateway layer is only effective if the canary service can handle the traffic it receives without side effects leaking back to stable users. A common failure mode: canary service v2 writes to a new database schema or new Kafka topic format that v1 cannot read. If 10% of orders are processed by v2 and written in the new schema, v1 instances querying the order DB see malformed data for those 10% and start failing for ALL users (100%), not just canary users. The fix is to enforce a "dual-write" or backward-compatible schema deployment before routing any canary traffic: v2 must be able to read v1's data and v1 must be able to read v2's data. Only after verifying schema compatibility across both versions is canary percentage increased. This is the key insight from Google's progressive delivery practices — the routing change is the easy part; the data compatibility contract is the hard part.

## Reliability and Operations

- **Gateway instance failure:** Stateless design → any instance handles any request. Load balancer detects failure via health check (HTTP GET /health) and stops routing within 10–30 seconds (health check interval). New instance starts and joins the cluster; no warm-up needed (state loaded from etcd/Redis on init).
- **Redis failure (rate limiting store):** Rate limiting fails open — all requests allowed through. Log the degradation as an alert. JWT auth continues working (local). API key auth continues working (local cache). Redis failure does not cause outage; it degrades rate limiting accuracy temporarily.
- **etcd failure (config store):** Gateway uses in-memory cached config (loaded at startup, refreshed every 30s). Config changes cannot be pushed until etcd recovers, but existing routes continue serving. Only new route additions or changes are blocked.
- **Thundering herd on config reload:** Route config changes propagate via etcd watch (push notification, not polling). Each gateway instance applies the new config atomically (swap in-process route table pointer). In-flight requests complete with old config; new requests use new config. No connection drops.
- **Observability:** Every request: structured log (async Kafka, < 0.1ms). Prometheus metrics: `gateway_requests_total{method, path, status}`, `gateway_request_duration_seconds{route}`, `gateway_circuit_state{service}`, `gateway_rate_limited_total{user_id_bucket}`. Distributed trace: inject trace-id + span-id into backend request headers; collect with OpenTelemetry.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Stateless gateway (no session affinity) | Stateful (session-pinned) | Stateless: horizontal scale, no SPOF, any instance handles any request; Stateful: simplifies auth token caching but creates hot instances and failover complexity |
| Local JWT verification (public key in memory) | Remote auth service call per request | Local: ~0.2ms, no network dependency; Remote: can revoke tokens instantly but adds ~2ms latency and makes auth service a critical dependency |
| Per-instance circuit breaker | Distributed circuit state (Redis) | Per-instance: zero latency overhead, eventually consistent across instances; Distributed: consistent but ~0.3ms per request on critical path |
| Fixed-window rate limiting (Redis INCR) | Sliding window log (Redis ZSET) | Fixed-window: 1 Redis op, O(1) memory; Sliding: 2-3 ops, more memory, no edge-of-window burst — worth it for strict limits (SMS, payments) |
| Sticky canary (hash-based) | Random canary | Sticky: consistent per-user, A/B testable, debuggable; Random: simpler but inconsistent UX for multi-request flows |

## G5

1. **GraphQL gateway:** REST gateway is well-understood. GraphQL adds complexity: a single `POST /graphql` request can touch many backends. Gateway must: parse GraphQL query → build execution plan (which subgraph handles which type) → dispatch sub-queries in parallel → merge results (Apollo Federation, AWS AppSync). Key challenge: schema stitching across team boundaries. Performance: parallel sub-query dispatch with timeout + partial result on subgraph failure (graceful degradation vs hard failure). N+1 problem: a query for 100 users each requesting their orders triggers 100 separate order lookups — DataLoader pattern (batch + deduplicate within a request cycle) is mandatory. Schema evolution: adding fields is safe; removing or renaming requires versioned deprecation with active client monitoring.

2. **gRPC transcoding:** Many internal services use gRPC for efficiency. External clients (browsers, mobile) often use REST. Gateway performs HTTP/JSON → gRPC transcoding: parse REST request → map to Protobuf message → call gRPC service → convert proto response → serialize to JSON. Mapping defined in `.proto` service annotations (`google.api.http` option). This is what Google's Cloud Endpoints and Envoy's gRPC-JSON transcoding do natively. Reduces the external API surface to REST while keeping internal services gRPC-native (efficient binary protocol, type safety, streaming). Key consideration: Protobuf field names (snake_case) vs JSON field names (camelCase) — need explicit mapping via `json_name` field option.

3. **Request signing and replay protection:** For financial or critical APIs, require HMAC-signed requests. Client signs: `HMAC-SHA256(secret, method + path + body_hash + timestamp + nonce)`. Gateway verifies signature before forwarding. Replay protection: nonce (UUID, one-time use) stored in Redis SET NX with 5-minute TTL; timestamp must be within ±5 minutes of server time. This prevents MITM attacks from replaying captured requests. Used by AWS Signature V4, Stripe API (idempotency keys serve a similar purpose), Twilio. Performance: HMAC-SHA256 is fast (~0.05ms) and no network call needed — the secret is pre-shared. Key rotation: support key versions (send `X-API-Key-Version` header) so old keys can be rotated without downtime.

4. **Dynamic routing and feature flags at gateway:** Beyond simple canary (all users / N% of users): route based on arbitrary request attributes. `user_segment=enterprise` → route to `product-service-enterprise`. `feature_flag:dark_mode=true` → route to `frontend-v2`. Architecture: gateway reads feature flag state from in-process cache (synced from LaunchDarkly or internal flag service every 10s). Request arrives with user_id → look up feature flags for that user → select target service version. Latency: feature flag lookup must be < 1ms (in-process map lookup, not a network call). This enables instant percentage-based rollout or rollback per customer segment without code deploy or config change — just a flag flip.

5. **Service mesh vs API gateway:** The distinction matters for architecture decisions. API gateway: north-south traffic (external clients → internal services). Handles external auth, rate limiting, SSL termination, API versioning. Service mesh (Istio, Linkerd): east-west traffic (service-to-service communication inside the cluster). Handles mutual TLS between services, retries, circuit breaking, load balancing, and observability for internal calls — without application code changes (sidecar proxy model). In mature architectures, you need both: gateway for external edge concerns; mesh for internal reliability and security. Design implication: don't replicate mesh functionality in the gateway (no service-to-service auth in gateway if mesh handles mTLS) — each layer has its own responsibility boundary.
