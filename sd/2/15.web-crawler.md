# 15. Web Crawler

**Type:** search
**Difficulty:** L4
**Frequency:** High — async processing, deduplication, distributed system at scale

---

## Problem Statement

Design a web crawler that discovers and indexes web pages for a search engine. The crawler starts from seed URLs, fetches pages, extracts links, and adds them to the crawl queue. It must crawl 1 billion pages per day, handle failures gracefully, avoid crawling the same URL twice, and respect robots.txt.

---

## Functional Requirements

**Must have:**
- Accept seed URLs and crawl the web breadth-first
- Fetch page HTML content
- Extract all hyperlinks from the page
- Store page content and metadata (URL, title, crawl timestamp, status code)
- Avoid crawling the same URL twice (deduplication)
- Respect robots.txt (don't crawl disallowed paths)

**Should have:**
- Politeness: don't overwhelm a single domain (limit requests/sec per domain)
- Priority: crawl important pages (high PageRank) more frequently
- Recrawl: re-visit pages periodically to detect updates

**Out of scope:**
- Full search index building (downstream of crawler)
- JavaScript rendering (JS-heavy SPAs need headless browser — mention as extension)
- Content deduplication (different URLs, same content — SimHash detection)

---

## Non-Functional Requirements

- **Scale:** 1B pages/day = ~11,600 pages/sec
- **Performance:** Crawl 1B pages/day on reasonable hardware
- **Availability:** Crawler is a background system; 95% uptime acceptable
- **Deduplication:** 99.99% accuracy — missing 0.01% of duplicates is acceptable
- **Politeness:** Max 1 request/sec per domain by default

---

## Capacity Estimation

```
Pages: 1B/day = 11,600/sec
Avg page size: 500KB HTML → 1B × 500KB = 500TB/day raw storage
  After compression (gzip ~4x): 125TB/day
  After dedup + filtering (keep 30%): ~37TB/day useful content

URLs seen: over time, 10B unique URLs
  Bloom filter at 0.01% false positive rate: ~200 bytes per URL → 2TB
  Better: use URL fingerprint (MD5) → 10B × 16B = 160GB → fits in distributed Redis

Bandwidth: 11,600 pages/sec × 500KB = 5.8GB/sec inbound bandwidth
  → Need ~100 crawler instances with 1Gbps each (100Gbps total inbound)

Crawl rate per domain: avg 100 pages per domain → 1B pages / 100 = 10M domains
  At 1 req/sec per domain = 10M reqs/sec possible in parallel → easily handled
```

## Deep Dive 1: URL Deduplication

**Problem:** The web is full of duplicate and near-duplicate URLs. Must avoid crawling same URL twice.

**URL normalization (before dedup):**
```
1. Lowercase scheme and host: HTTP://Example.COM → http://example.com
2. Remove default ports: http://example.com:80/page → http://example.com/page
3. Remove trailing slash inconsistencies: /page/ → /page
4. Sort query parameters: ?z=1&a=2 → ?a=2&z=1
5. Remove fragment: /page#section → /page (fragment not sent to server)
6. Decode percent-encoding where safe: /page%20title → /page title
7. Strip common tracking parameters: ?utm_source=, ?fbclid=, ?ref=
```

**Deduplication approaches:**

**Option A: Hash set in distributed DB**
```
After normalizing URL:
  hash = MD5(url)
  Check: SELECT 1 FROM crawled_urls WHERE url_hash = hash
  If not found: add to frontier + insert into crawled_urls
  If found: skip
Cost: DB round trip per URL
```

**Option B: Bloom filter (chosen)**
```
Bloom filter: probabilistic data structure
  INSERT: hash URL with k hash functions → set k bits
  LOOKUP: all k bits set → "probably seen" (may false positive)
           any bit unset → "definitely not seen"

Parameters for 10B URLs, 0.01% false positive rate:
  Bits: m = -n × ln(p) / (ln 2)^2 = 10B × 6.93 × 20 ≈ 1.4 trillion bits = 175GB
  Hash functions: k = (m/n) × ln(2) ≈ 13

Too large for single machine → use distributed bloom filter:
  Shard across 10 Redis nodes: each holds 17.5GB → feasible (8GB RAM per shard compressed)

False positive rate: 0.01% → 1M unnecessary skips per 1B URLs → acceptable
False negative rate: 0% — never misses a seen URL
```

**Option C: URL hash table in Redis**
```
SETNX url_seen:{md5_hash} 1 EX 86400
  → Returns 1 (new URL) or 0 (already seen)
  → Atomic check-and-set in O(1)
  → TTL handles old URLs (recrawl eligibility)
Memory: 10B × 30B (key + value overhead) = 300GB → multiple Redis nodes
```

## Deep Dive 2: URL Frontier and Politeness

**Politeness requirements:**
- Don't send more than 1 request/sec to any domain
- Respect robots.txt crawl-delay directive
- Don't overwhelm small sites even if robots.txt allows

**URL Frontier architecture:**
```
Multi-level queue:
  Level 1: Priority queues (priority 1-10)
  Level 2: Per-domain FIFO queues (politeness enforcement)

Scheduler:
  1. Dequeue from priority queue → get next URL to crawl
  2. Check domain's last_crawled time:
     if now - last_crawled >= crawl_delay:
         assign to available crawler worker
     else:
         postpone (add back with scheduled_at = last_crawled + crawl_delay)
  3. Track in-flight requests per domain (cap at max_concurrent=5)

Implementation:
  Priority queue: Redis sorted set (score = priority)
  Per-domain queues: Redis list per domain key
  Last crawl times: Redis hash domain → timestamp
```

**Dynamic priority:**
```
URL priority = base_priority × freshness_factor × importance_score
  base_priority: set by submitter
  freshness_factor: 1.0 for new URLs; increases with time since last crawl
  importance_score: estimated PageRank (count of backlinks seen so far)

High-priority: news sites, major domains, frequent content updates
Low-priority: deep link pages, rarely-linked content, old archives
```

## Deep Dive 3: Failure Handling and Recrawl

**Fetch failures:**
```
Strategy per status code:
  200: process and store
  301/302: follow redirect (max 5 hops), deduplicate final URL
  404: mark as not found, don't recrawl for 30 days
  429 (rate limited): backoff exponentially, respect Retry-After header
  5xx: retry up to 3 times with exponential backoff, then mark as temporarily unavailable
  Connection timeout: retry up to 3 times, then skip (mark for recrawl in 1 day)
```

**Recrawl scheduling:**
```
Recrawl frequency based on content change rate:
  News sites: recrawl every 1 hour
  Blogs: recrawl every 1 week
  Static pages: recrawl every 1 month

Change detection:
  Compare content_hash on recrawl
  If hash unchanged: extend recrawl interval × 1.5 (less frequent)
  If hash changed: reset to base interval, re-index content
```

---

## Reliability and Operations

- **Crawler worker failure:** Workers are stateless. In-progress URLs returned to frontier queue after visibility timeout (5 min). No work lost.
- **Frontier DB failure:** Use Kafka as durable frontier buffer. URL frontier events → Kafka topic → crawl workers consume. Redis is a fast preview layer for scheduling, Kafka provides durability.
- **DNS poisoning / malicious redirects:** Validate redirect destinations (same-domain or allowlisted cross-domain). Detect redirect loops (URL already in redirect chain).
- **Spider traps:** Pages that generate infinite unique URLs (e.g., calendars with infinite date navigation). Detection: URL depth limit (max 20 hops from seed), URL length limit (max 500 chars), seen-domain URL count limit (max 1M URLs per domain).

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Bloom filter dedup | DB hash lookup | Bloom: O(1) and memory-efficient; DB: exact (no false positives) but slower |
| Kafka frontier | Redis sorted set | Kafka: durable; Redis: faster, easier to prioritize |
| Fixed crawl delay | Adaptive delay | Fixed: simple, respectful; Adaptive: more aggressive but harder to tune |

## L5 Additions

1. **JavaScript rendering:** ~30% of modern web is JS-rendered (SPAs, React apps). For these: run headless Chrome (Puppeteer/Playwright) in a separate worker pool. Trigger JS rendering for URLs where: HTTP response content-length < 1KB (suspiciously small for a page), URL pattern matches SPA patterns, or explicit flag from URL metadata. JS rendering is 10-20x slower than raw HTTP fetch → reserve for high-priority URLs. Scale: a pool of 100 headless Chrome instances handles ~500 JS renders/sec.

2. **Distributed link graph:** During crawling, build the link graph (which pages link to which). Stored in Neo4j or a distributed graph database. Use for PageRank computation (Hadoop/Spark batch job, daily). PageRank score feeds back into crawl priority: high-PageRank pages crawled more frequently. Link graph also useful for: spam detection (pages with only links to one domain), content quality scoring, social network analysis of the web.

3. **Content deduplication (SimHash):** Different URLs may have identical or near-identical content (syndicated articles, mirror sites). SimHash: locality-sensitive hash of document content → 64-bit fingerprint. Documents with Hamming distance < 3 are near-duplicates. Use Hamming distance search: split 64-bit hash into 4 × 16-bit chunks, index each chunk → find documents differing in ≤ 1 chunk → verify Hamming distance. This is Google's approach for detecting duplicate content in search results.

4. **Multi-region crawling:** Deploy crawl workers in every major region. Crawl servers close to the target host = lower latency + respecting local regulations. Route crawls: US URLs → US workers, EU URLs → EU workers, Asian URLs → APAC workers. This also distributes the inbound bandwidth cost across regions. Coordinate via global frontier service: central scheduler assigns URL batches to regional worker pools. Cross-region coordination latency is fine (scheduling is not in the critical path).

5. **Real-time freshness signals:** Don't wait for scheduled recrawl. Subscribe to: RSS/Atom feeds (sites publish their own change notifications), sitemaps with lastmod timestamps (crawl only if lastmod > last_crawl), Google Search Console's Push API (sites can notify crawlers via PubSubHubbub), CDN log sharing (some CDNs share access logs with search engines). Integrate these signals into the URL frontier: externally notified URLs get priority boost and bypass normal recrawl scheduling.
