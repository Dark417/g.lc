# 15. Web Crawler

**Type:** search
**Difficulty:** L4
**Frequency:** High — async processing, deduplication, distributed system at scale

---

## Problem Statement

Design a web crawler that discovers and indexes web pages for a search engine. The crawler starts from seed URLs, fetches pages, extracts links, and adds them to the crawl queue. It must crawl 1 billion pages per day, handle failures gracefully, avoid crawling the same URL twice, and respect robots.txt.

---

## Functional Requirements

**Must have:**
- Accept seed URLs and crawl the web breadth-first
- Fetch page HTML content
- Extract all hyperlinks from the page
- Store page content and metadata (URL, title, crawl timestamp, status code)
- Avoid crawling the same URL twice (deduplication)
- Respect robots.txt (don't crawl disallowed paths)

**Should have:**
- Politeness: don't overwhelm a single domain (limit requests/sec per domain)
- Priority: crawl important pages (high PageRank) more frequently
- Recrawl: re-visit pages periodically to detect updates

**Out of scope:**
- Full search index building (downstream of crawler)
- JavaScript rendering (JS-heavy SPAs need headless browser — mention as extension)
- Content deduplication (different URLs, same content — SimHash detection)

---

## Non-Functional Requirements

- **Scale:** 1B pages/day = ~11,600 pages/sec
- **Performance:** Crawl 1B pages/day on reasonable hardware
- **Availability:** Crawler is a background system; 95% uptime acceptable
- **Deduplication:** 99.99% accuracy — missing 0.01% of duplicates is acceptable
- **Politeness:** Max 1 request/sec per domain by default

```
  Bandwidth: 5.8GB/sec inbound → 100 crawler instances × 1Gbps NIC each

Domain politeness:
  10B pages / avg 100 pages per domain = 100M unique domains

  In practice: 10M active domains at any time (others already crawled)
```

## Deep Dive 1: URL Deduplication

**Problem:** The web has massive duplication — tracking parameters, session IDs, minor URL variants all produce unique URLs pointing to identical content.

**Step 1 — URL normalization (before any dedup check):**
```python
def normalize_url(raw_url: str) -> str:
    parsed = urlparse(raw_url.strip())

    # 1. Lowercase scheme and host
    scheme = parsed.scheme.lower()   # HTTP → http
    host = parsed.netloc.lower()     # Example.COM → example.com

    # 2. Remove default ports
    host = host.replace(':80', '') if scheme == 'http' else host
    host = host.replace(':443', '') if scheme == 'https' else host

    # 3. Decode safe percent-encoding
    path = unquote_plus(parsed.path)

    # 4. Remove trailing slash on non-root paths
    if path != '/' and path.endswith('/'):
        path = path.rstrip('/')

    # 5. Sort query parameters (deterministic ordering)
    params = sorted(parse_qsl(parsed.query))

    # 6. Strip known tracking parameters (common ones that don't affect content)
    TRACKING_PARAMS = {'utm_source', 'utm_medium', 'utm_campaign', 'utm_content',
                       'utm_term', 'fbclid', 'gclid', 'ref', 'source', '_ga'}
    params = [(k, v) for k, v in params if k not in TRACKING_PARAMS]

    # 7. Remove URL fragment (never sent to server)
    fragment = ''

    return urlunparse((scheme, host, path, '', urlencode(params), fragment))

# Example:
# Input:  "HTTP://Example.COM:80/Page/?z=1&utm_source=google&a=2#section"
# Output: "http://example.com/Page/?a=2&z=1"
```

**Dedup approaches — comparison:**

**Option A: Hash set in distributed DB (exact)**
```sql
-- After normalizing:
url_hash = MD5(normalized_url)  -- 16 bytes

-- Check:
SELECT 1 FROM crawled_pages WHERE url_hash = $1 LIMIT 1;
-- If found: skip. If not found: insert and add to frontier.

Cost: 1 DB round trip per extracted URL
At 11,600 pages/sec × avg 50 links per page = 580,000 URL checks/sec
PostgreSQL with proper indexing: ~50K queries/sec per server
→ Need 12 PostgreSQL read replicas for URL dedup checks alone
→ Expensive but exact (0% false positives, 0% false negatives)
```

**Option B: Bloom filter — chosen for primary dedup layer**
```python
# Bloom filter math for 10B URLs at 0.01% false positive rate (FPR):
# n = 10,000,000,000 elements
# p = 0.0001 (0.01% FPR)
# m = -n × ln(p) / (ln(2))^2 = -10B × ln(0.0001) / 0.480 = 10B × 9.21 / 0.480 ≈ 192 billion bits
#   = 192 billion bits / 8 = 24GB
# k = (m/n) × ln(2) = 19.2 × 0.693 ≈ 13 hash functions

# Distributed across 3 Redis nodes, each holding 8GB = 24GB total
# Redis SETBIT / GETBIT operations: O(1)

class DistributedBloomFilter:
    def __init__(self, redis_cluster, num_bits=192_000_000_000, num_hashes=13):
        self.nodes = redis_cluster
        self.num_bits = num_bits
        self.num_hashes = num_hashes

    def _get_bit_positions(self, url: str) -> List[int]:
        positions = []
        for seed in range(self.num_hashes):
            h = mmh3.hash(url, seed, signed=False)
            positions.append(h % self.num_bits)
        return positions

    def add(self, url: str):
        for pos in self._get_bit_positions(url):
            node = self.nodes[pos % len(self.nodes)]
            node.setbit('bloom', pos, 1)

    def might_contain(self, url: str) -> bool:
        for pos in self._get_bit_positions(url):
            node = self.nodes[pos % len(self.nodes)]
            if not node.getbit('bloom', pos):
                return False  # definitely NOT seen
        return True  # probably seen (may be false positive)

# False positive consequence: 0.01% of new URLs skipped → ~116 valid URLs/sec not crawled
# False negative: impossible — bloom filter never forgets a seen URL
# Acceptable trade-off for 99.99% dedup accuracy requirement
```

**Option C: URL hash set in Redis (exact, memory-bounded)**
```
For recently-seen URLs (last 30 days):
  SETNX url_seen:{md5_hash_first_8_chars} 1 EX 2592000  # 30-day TTL
  → Returns 1 if new (not seen), 0 if already seen
  → TTL enables recrawl eligibility after 30 days
  Memory: 10B URLs × 30B (key overhead in Redis) = 300GB
  → Too large for hot Redis; use for recent URLs only, bloom filter for long-term

Combined approach (two-tier):
  1. Check bloom filter first (O(1), 24GB distributed) — if "definitely not seen": add and crawl
  2. If "probably seen": verify with Redis exact set (last 30 days) or DB lookup
  → Bloom filter handles 99.99% of dedup checks
  → Redis/DB only consulted for the 0.01% false positives
```

**G5:** Would note that the bloom filter approach has a subtle operational problem: the filter can never shrink or have entries removed. As the crawl expands from 1B to 10B to 100B URLs, the false positive rate grows unless the filter is rebuilt. Rebuilding a 24GB distributed bloom filter requires a full scan of the crawled_pages table, computing hashes for all URLs, and re-inserting — this takes hours for a 100B-URL database and the filter must be in read-only mode during rebuild. An alternative data structure is the Cuckoo filter: it supports deletion (unlike bloom), has better constant factors for lookup, and can be rebuilt incrementally. For a production crawler, consider partitioning the URL space by domain hash into time-bounded shards — each shard is a separate bloom filter covering URLs first seen within a specific month. To check "have we seen this URL?", query the relevant shard(s). Old shards can be archived when all their URLs are past recrawl eligibility. This turns a single monolithic 24GB filter into a manageable set of smaller, rotatable filters.

## Deep Dive 2: URL Frontier and Politeness

**Politeness requirements:**
- Default: max 1 request/sec per domain (crawl-delay from robots.txt overrides)
- Some sites specify Crawl-delay: 10 → 1 request per 10 seconds
- Small sites: even 1 req/sec may be too aggressive — monitor 5xx rate, back off if errors
- Never send concurrent requests to the same domain (even if they'd be under rate limit)

**URL Frontier multi-level architecture:**
```
Level 1: Priority Queue (10 priority buckets, 1=low, 10=high)
  Implementation: Redis sorted set per priority, score = enqueue timestamp (FIFO within priority)
  Keys: frontier:priority:1 ... frontier:priority:10
  ZPOPMIN frontier:priority:10 → get highest priority, earliest enqueued URL

Level 2: Per-domain FIFO queues (politeness enforcement)
  Each domain gets its own Redis list: frontier:domain:{domain_hash}
  When a URL is dequeued from Level 1, it's placed into its domain's Level 2 queue

Scheduler logic (runs every 100ms):
  For each domain with items in Level 2:
    1. Check domain_crawl_state[domain].last_crawled_at
    2. If now() - last_crawled_at >= crawl_delay_ms:
       a. Pop URL from frontier:domain:{domain}
       b. Assign to available crawler worker
       c. Update domain_crawl_state[domain].last_crawled_at = now()
    3. Else: skip (this domain needs to wait)
  Result: at most 1 in-flight request per domain at any time
```

**robots.txt compliance:**
```python
def is_allowed(url: str, user_agent: str = "MyCrawler/1.0") -> bool:
    domain = extract_domain(url)

    # Check cache first (TTL 24h)
    robots = redis.get(f"robots:{domain}")

            robots = resp.text if resp.status_code == 200 else ""
        except:

        redis.setex(f"robots:{domain}", 86400, robots)

    # Parse and check
    parser = RobotFileParser()
    parser.parse(robots.splitlines())
    return parser.can_fetch(user_agent, url)

#   Crawl-delay: <seconds>
#   Sitemap: <sitemap_url>  ← add sitemap URLs to frontier as high-priority seeds
```

**Dynamic priority scoring:**

## Deep Dive 3: Failure Handling and Recrawl

**Fetch failures — per HTTP status strategy:**
```python
def handle_fetch_result(url: str, resp: Optional[Response]) -> CrawlAction:
    if resp is None:  # connection timeout or DNS failure
        return CrawlAction(
            action="retry",
            delay_seconds=exponential_backoff(domain_error_count, base=60, max=3600),
            increment_error_count=True
        )

    status = resp.status_code
    if status == 200:
        return CrawlAction(action="process", store=True)

    elif status in (301, 302, 303, 307, 308):  # redirect
        # Follow redirect (max 5 hops to prevent infinite redirect loops)
        final_url = follow_redirects(url, max_hops=5)
        if final_url and not bloom_filter.might_contain(normalize_url(final_url)):
            return CrawlAction(action="enqueue_redirect", redirect_to=final_url)
        return CrawlAction(action="skip")

    elif status == 404:
        return CrawlAction(action="mark_dead", next_crawl_after_days=30)

    elif status == 410:  # Gone (permanent)
        return CrawlAction(action="mark_dead", next_crawl_after_days=365)

    elif status == 429:  # Rate limited
        retry_after = int(resp.headers.get('Retry-After', 60))
        return CrawlAction(action="retry", delay_seconds=retry_after, update_crawl_delay=True)

    elif status in range(500, 600):  # Server error
        errors = increment_domain_error_count(url_domain)
        if errors >= 5:
            return CrawlAction(action="skip_domain_temporarily", hours=1)
        return CrawlAction(
            action="retry",
            delay_seconds=exponential_backoff(errors, base=30, max=600)
        )
```

**Recrawl scheduling:**
```python
def compute_next_crawl(url_hash: str, current_content_hash: str, prev_content_hash: str,
                       current_interval_hours: int) -> datetime:
    if current_content_hash != prev_content_hash:
        # Content changed: reset to base interval (content is active)
        return now() + timedelta(hours=BASE_RECRAWL_INTERVAL[url_type])
    else:
        # Content unchanged: slow down recrawl (exponential backoff)
        new_interval = min(current_interval_hours * 1.5, MAX_INTERVAL_HOURS)
        return now() + timedelta(hours=new_interval)

# Base recrawl intervals by domain type:
BASE_RECRAWL_INTERVAL = {
    "news":    1,      # nytimes.com, bbc.com: every hour
    "blog":    168,    # weekly
    "ecommerce": 24,   # product pages: daily (prices change)
    "static":  720,    # monthly
}
MAX_INTERVAL_HOURS = 8760  # 1 year maximum recrawl interval
```

**G5:** Would question whether a single change-detection hash (MD5 of full content) is the right signal for recrawl scheduling. Full-page content hash changes whenever an ad banner rotates, a "you might also like" widget updates, or a comment is posted — even if the main article content is unchanged. This causes news sites to appear "always changing" and get over-crawled, while the actual article text is static. A smarter approach: extract the main content body (using readability algorithms like Mozilla Readability or Diffbot's content extraction), hash only that, and use the full-page hash only for detecting sidebar/structural changes separately. This two-signal recrawl system reduces unnecessary recrawl of unchanged article content by ~40% on major news sites while still catching real content updates quickly.

## Reliability and Operations

- **Crawler worker failure:** Workers are fully stateless — they consume a URL from the frontier queue, fetch it, and write results. If a worker crashes mid-fetch: the URL visibility timeout (5 minutes in Kafka) expires, and the URL is re-delivered to another worker. No work is lost. Workers have no local state beyond the current in-flight URL.
- **Frontier durability:** Kafka as the durable URL frontier. Redis serves as a fast scheduling cache (priority queues, per-domain rate limiting state), but Kafka holds the durable log of URLs to crawl. On complete Redis failure: rebuild scheduling state from Kafka consumer group offsets — Redis is reconstructable.
- **DNS poisoning and malicious redirects:** Validate that the final redirect destination is a legitimate HTTP/HTTPS URL. Block: `file://`, `ftp://`, `javascript:` schemes. Detect redirect loops: track redirect chain, abort if any URL appears twice. Validate destination IP is not in RFC1918 private ranges (SSRF prevention — a malicious site could redirect crawler to `http://169.254.169.254/` AWS metadata endpoint).
- **Spider traps:** Infinite unique URL generation (calendars, infinite pagination, session-parameterized URLs). Defenses: (a) URL depth limit (max 20 hops from any seed). (b) URL length limit (max 500 characters). (c) Per-domain URL count limit: max 1M URLs queued per domain (prevents one domain from monopolizing the frontier). (d) Repeated path segment detection: `/a/a/a/a/a/` → depth limit triggers.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Bloom filter dedup (0.01% FPR) | Exact DB hash lookup | Bloom: O(1), 24GB total memory for 10B URLs; DB: exact (0% FPR) but 580K queries/sec requires 12+ DB read replicas |
| Kafka for durable frontier | Redis sorted set only | Kafka: survives Redis cluster failure, replay from offset; Redis: faster scheduling, no disk overhead, but lost on cluster wipe |
| Fixed crawl delay (robots.txt) | Adaptive delay based on server response times | Fixed: deterministic, respectful, easy to reason about; Adaptive: more throughput but complex to tune, risk of over-crawling slow servers |
| MD5 for URL hashing | SHA-256 | MD5: 16 bytes (compact), fast, collision probability negligible for URL dedup; SHA-256: cryptographically secure but unnecessary overhead for non-security use case |

## G5

1. **JavaScript rendering for SPAs:** ~30% of modern web content requires JavaScript execution to render meaningful HTML (React, Angular, Vue SPAs). For these pages, raw HTTP fetch returns a near-empty `<div id="root"></div>`. Solution: a separate JS rendering worker pool running headless Chromium (via Puppeteer or Playwright). Trigger JS rendering when: (a) raw HTML content is < 2KB (suspiciously small for a real page), (b) URL matches SPA patterns (`/app/`, `/#/`, React Router paths), (c) response headers include `X-Rendered-By: React`. JS rendering is 15–20x slower than raw HTTP fetch (1–3 seconds per page vs 50–200ms). Scale: 100 headless Chrome instances × 500ms avg render time = 200 renders/sec. Reserve for high-priority URLs only. Memory: each headless Chrome uses ~200MB RAM → 100 instances = 20GB RAM dedicated to JS rendering.

2. **Distributed link graph for PageRank:** During crawling, build a link graph: (source_url_hash, destination_url_hash) edges. Volume: 1B pages × avg 50 links = 50B edges/day. Storage: 50B × 16B (source hash + dest hash) = 800GB/day → partition into daily snapshots on S3. Weekly Spark GraphX job: load full link graph, run Pregel-style iterative PageRank (50 iterations to convergence), output page_rank scores per URL hash. Feed PageRank scores back into the frontier scheduler (high-PageRank pages get priority 8–10). PageRank also powers: spam link farm detection (thousands of outbound links to one domain from thin pages), content quality signals, and domain authority scoring.

3. **Content deduplication with SimHash:** Different URLs with nearly identical content (syndicated articles, mirror sites, scraped copies) should be deduplicated before indexing. SimHash: compute a 64-bit locality-sensitive hash of the document's term frequency vector. Documents with Hamming distance ≤ 3 bits are near-duplicates (empirically validated threshold). Lookup: split the 64-bit SimHash into 4 × 16-bit chunks, store in 4 hash tables keyed by each chunk — find documents differing in at most 1 chunk, then verify Hamming distance. Storage for 10B pages: 10B × 8B SimHash = 80GB → fits in Redis cluster. Finding near-duplicates for a new page: 4 Redis hash lookups in parallel → O(1) per lookup → < 5ms total. Dedup rate on web crawl: ~20–30% of pages are near-duplicates. This is the algorithm Google originally described in their 2007 "Detecting Near-Duplicates for Web Crawling" paper.

4. **Multi-region crawling for latency and compliance:** Deploy crawl workers in every major cloud region. Route crawls by target host geography: `.co.uk` domains → UK workers (lower RTT, respects local regulations); `.cn` domains → APAC workers; EU domains → EU workers (GDPR: crawled content of EU residents stays in EU until search index processing). Coordinate via a global frontier service: central scheduler assigns URL batches to regional worker pools based on domain geolocation (MaxMind GeoIP database). Cross-region coordination adds ~50ms scheduling latency, which is acceptable since fetching is the bottleneck (100ms–2s per page). Benefit: average crawl latency reduced by 30–40% for non-US domains (300ms RTT from US → 30ms RTT within region for European sites).

5. **Real-time freshness signals beyond scheduled recrawl:** Subscribing to external change notification protocols dramatically improves freshness for high-priority content. (a) PubSubHubbub / WebSub: major blogging platforms (WordPress.com, Blogger) push `hub.mode: publish` pings when content updates — receiver processes the push and queues the URL at priority 10 for immediate recrawl. (b) Sitemap with `<lastmod>`: crawl sitemaps.xml every hour; queue any URL with `lastmod > last_crawl_time`. (c) RSS/Atom feeds: subscribe to feeds for news sites, detect new entries, immediately queue. (d) CDN access log sharing: Cloudflare's "Edge Cache" logs can reveal which pages are experiencing sudden traffic spikes — a traffic spike on a URL that hasn't changed for months is a signal it was just linked from a viral source. These signals bypass the recrawl scheduler entirely and insert URLs directly into the priority-10 queue, achieving sub-minute freshness for major news events vs the 1-hour scheduled recrawl.
