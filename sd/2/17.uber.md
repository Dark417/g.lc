# 17. Uber — Ride Sharing Platform

**Type:** commerce
**Difficulty:** L4–L5
**Frequency:** Very High — geo-matching, real-time location, supply-demand

---

## Problem Statement

Design a ride-sharing platform like Uber. Riders request rides from their location, drivers broadcast their location in real-time, the system matches riders with nearby drivers, and tracks the trip until completion. Must handle 10M rides/day globally, match riders within 2 seconds, and handle real-time GPS updates from 1M+ active drivers.

---

## Functional Requirements

**Must have:**
- Driver: update location every 4 seconds while active
- Rider: request a ride from current location to destination
- System: find and offer ride to nearest available driver within 2 sec
- Driver: accept or decline ride
- Track trip in real-time (rider sees driver location, ETA updates)
- Complete ride: fare calculation, payment, receipt

**Should have:**
- Upfront pricing (fare estimate before requesting)
- Dynamic pricing (surge pricing)
- Driver and rider ratings
- Ride cancellation

**Out of scope:**
- Payment processing (use Stripe/Braintree)
- Navigation turn-by-turn (use Google Maps API)
- Fraud detection (mention as extension)

---

## Non-Functional Requirements

- **Scale:** 10M rides/day; 1M active drivers at peak sending location every 4s = 250K location updates/sec
- **Matching latency:** Find and offer ride to driver within 2 seconds
- **Location freshness:** Driver location data < 4 seconds old
- **Availability:** 99.99% — any downtime loses revenue
- **Consistency:** Eventual for location; strong for ride assignment (no double-booking a driver)

---

## Capacity Estimation

```
Driver location updates: 1M drivers × 1 update/4s = 250K writes/sec
  Each update: driver_id(8B) + lat(8B) + lng(8B) + timestamp(8B) = 32B
  Bandwidth: 250K × 32B = 8MB/sec → trivial

Ride requests: 10M/day = 115/sec average; peak ~1,000/sec
  Each request triggers: geo-search (find nearby drivers) + matching

Active driver geo index:
  1M drivers on map → each occupying a geohash cell
  Redis GEOADD: 1M entries × 50B = 50MB → trivially fits in Redis

Trip tracking: 1M concurrent trips at peak × 1 location update/4s = 250K GPS writes/sec
  Per trip: 2 participants (rider + driver) watching location
```

## High-Level Architecture

```
[Driver App] ──→ [Location Service] ──→ [Redis Geo Index]
    │                                        │
    │                                [Nearby Driver Search]
    │                                        │
[Rider App] ──→ [Ride Request Service] ──→ [Matching Engine]
                                                │
                               [Driver Offer + Accept/Decline]
                                                │
                                      [Trip Management Service]
                                                │
                               ┌────────────────┼────────────────┐
                        [Driver App]    [Rider App]     [Payment Service]
                        (live tracking)  (ETA updates)
```

**Components:**
- **Location Service:** Receives driver GPS updates, writes to Redis Geo index. Ephemeral — location data is only useful for current state.
- **Matching Engine:** On ride request: geo-query nearby drivers → filter available → score candidates → dispatch offer.
- **Trip Management Service:** Manages trip state machine (requested → matched → en route → arrived → in progress → completed). Source of truth for trip state.
- **Real-time Tracking:** WebSocket connection per active trip. Pushes driver location updates to rider; ETA updates.
- **Surge Pricing Service:** Computes demand/supply ratio per geohash cell → pricing multiplier.

## API Design

```

Body: { pickup, dropoff, ride_type, payment_method_id }

POST /v1/rides/{id}/cancel  Body: { reason }

# WebSocket (trip tracking)
ws://api.uber.com/v1/rides/{id}/track
  → Server pushes: { driver_lat, driver_lng, eta_sec, status_update? }
```

## Data Model

```sql
-- PostgreSQL: rides (strong consistency for financial data)
rides (
  ride_id       UUID PRIMARY KEY,
  rider_id      UUID NOT NULL,
  driver_id     UUID,               -- NULL until matched
  status        TEXT,               -- 'searching','matched','enroute','arrived','in_progress','completed','cancelled'
  pickup_lat    DECIMAL(9,6),
  pickup_lng    DECIMAL(9,6),
  dropoff_lat   DECIMAL(9,6),
  dropoff_lng   DECIMAL(9,6),
  fare_estimate DECIMAL(8,2),
  final_fare    DECIMAL(8,2),
  surge_multiplier DECIMAL(3,2),
  requested_at  TIMESTAMPTZ,
  matched_at    TIMESTAMPTZ,
  completed_at  TIMESTAMPTZ,
  payment_status TEXT
)

-- Driver availability (in Redis for speed, not DB)
Redis:
  GEOADD drivers:available {lng} {lat} {driver_id}
  GEORADIUS drivers:available {rider_lng} {rider_lat} 5 km ASC COUNT 20

-- Driver state (Redis, fast-changing)
Key: driver:{driver_id}
Value: { status: "available"|"busy"|"offline", ride_id?, last_updated }
TTL: 30s (driver must heartbeat to stay "available")

-- Trip events (append-only, Cassandra)
trip_events (
  ride_id    UUID PARTITION KEY,
  event_time TIMESTAMP CLUSTERING KEY,
  event_type TEXT,  -- 'location_update', 'status_change', 'message'
  data       JSONB
)
```

## Deep Dive 1: Matching Algorithm

**Matching steps:**
```
1. Rider requests ride from (lat=37.77, lng=-122.41)

2. Geo-query: find nearby available drivers
   GEORADIUS drivers:available -122.41 37.77 5 km ASC COUNT 20
   → Returns 20 nearest drivers with distances

3. Filter: only drivers with correct vehicle type (standard/XL/premium)
   Check driver:{driver_id}.status == "available"
   Remove drivers who recently declined (30s cooldown)

4. Score remaining candidates:
   score = w1 × (1/distance) + w2 × driver_rating + w3 × eta_accuracy
   → Rank by score; top 3 are primary candidates

5. Dispatch offer to best driver:
   a. Send push notification to Driver app + WebSocket message
   b. Set timeout: driver has 15 seconds to accept
   c. If accepted: create ride assignment, update state, notify rider
   d. If declined or timeout: offer to 2nd candidate
   e. If all 20 decline: expand radius to 10km, repeat

6. On acceptance:
   UPDATE rides SET driver_id=X, status='matched' WHERE ride_id=Y
   SREM drivers:available {driver_id}  ← atomically remove from available pool
   driver:{driver_id}.status = "busy"
```

**Preventing double-booking:**
```
Use Redis Lua script for atomic check-and-assign:
  local status = redis.call('HGET', 'driver:'..driver_id, 'status')
  if status == 'available' then
    redis.call('HSET', 'driver:'..driver_id, 'status', 'busy', 'ride_id', ride_id)
    redis.call('ZREM', 'drivers:available', driver_id)
    return 1  -- success
  else
    return 0  -- driver already taken
  end
```

## Deep Dive 2: Real-Time Location and Tracking

**Driver location pipeline:**
```
Driver app → HTTP POST every 4s → [Location API] → [Kafka: driver-locations]
  │                                                         │
  │                                              [Location Consumer]
  │                                                         │
  │                                         Redis: GEOADD + HSET driver state
  │
  └─ If driver is on a trip:
     → Kafka: trip-tracking topic
     → Trip tracker consumes → pushes via WebSocket to rider + driver apps
```

**Trip tracking via WebSocket:**
```
Rider's app maintains WebSocket connection to Trip Tracker server
Trip Tracker subscribed to Kafka topic "trip-locations:{ride_id}"
On each driver location update:
  → Compute ETA using Google Maps API (or cached road graph)
  → Push to rider: { lat, lng, eta_sec }
Driver's app also receives: { rider_lat, rider_lng } (for pickup navigation)
```

**Handling 250K location writes/sec:**
```
Kafka partitioned by driver_id → 250K events/sec total
  20 partitions × 50 consumers = 12,500 events/sec per consumer → manageable

Consumer writes to Redis:
  GEOADD drivers:available: only for available drivers
  HSET driver:{id}: always (trip tracking needs this)

Redis cluster: 1M active drivers × 50B = 50MB → easily fits on a single Redis node
With replication: 5 nodes (1 primary, 4 replicas) → 250K writes/sec on primary → fine
```

## Deep Dive 3: Surge Pricing

**Dynamic pricing algorithm:**
```
Every 5 minutes per geohash cell (precision 5 = ~5km × 5km):
  1. Count requests in last 10 minutes: demand
  2. Count available drivers in cell: supply
  3. demand_supply_ratio = demand / (supply × avg_capacity)
  4. surge_multiplier = f(demand_supply_ratio):
     ratio < 1.0: 1.0x (no surge)
     1.0-1.5: 1.2x
     1.5-2.0: 1.5x
     2.0-3.0: 2.0x
     > 3.0: up to 4.0x

Stored in Redis per geohash cell:
  Key: surge:{geohash5_prefix}
  Value: { multiplier, updated_at }
  TTL: 5 minutes
```

## Reliability and Operations

- **Location service down:** Driver location data in Redis expires (TTL 30s). On restart, drivers must re-send location. No permanent data loss — location is transient.
- **Matching service down:** Ride requests queue in Kafka. On recovery, process queued requests. Rider sees "searching" for longer but doesn't lose their request.
- **Trip in progress during outage:** Trip state in PostgreSQL is durable. Driver and rider reconnect — they retrieve current trip state from PostgreSQL.
- **Partial match (driver accepted, DB write failed):** Use idempotent write with `ON CONFLICT DO NOTHING`. If DB write fails, driver sees "assignment failed" and goes back to available.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis Geo for driver index | PostGIS | Redis: in-memory, faster; PostGIS: persistent, richer geo queries |
| WebSocket for trip tracking | Polling | WebSocket: real-time, low latency; Polling: simpler, higher latency |
| Kafka for location pipeline | Direct Redis write | Kafka: durability, replay; Direct: simpler, faster |
| Per-cell surge pricing | Global surge | Per-cell: more accurate; Global: simpler, less granular |

## L5 Additions

1. **Driver pre-positioning:** Instead of reactive dispatch, predict where demand will spike. ML model: historical ride data + events (concerts, airport arrivals) + time of day → demand heatmap for next 30 minutes. Push "suggested destinations" to available drivers in low-demand areas. Incentivize drivers to move to predicted high-demand areas before demand materializes. Reduces average ETW (estimated time to wait) for riders during surge events.

2. **Pool / shared rides matching:** Uber Pool requires matching multiple riders going in the same direction. Algorithm: cluster ride requests by spatial proximity (geohash) and temporal window (within 5 minutes). For each cluster, run route optimization (TSP-variant) to find optimal pickup/dropoff order. Constraint: no rider's ETA increases by more than 10 minutes due to detour. Optimization runs in < 2 seconds. Backend: separate "pool matching" microservice; Pool driver can accept multiple rides simultaneously.

3. **Two-sided marketplace dynamics:** At scale, dispatch becomes a multi-objective optimization problem. Objective: minimize total wait time × fare completion rate × driver efficiency. Formulate as an assignment problem: N riders, M drivers, cost matrix based on ETA + driver preference + rider preference. Hungarian algorithm or auction-based matching. For 1000 simultaneous requests in a city, compute globally optimal assignment in < 500ms. Batch matching every 3 seconds (vs immediate dispatch) improves overall efficiency by 15-20%.

4. **Fraud detection on trips:** GPS spoofing (drivers fake their location to avoid long pickups), fare manipulation (driver takes longer route), account takeover (stolen payment). Detection: GPS trajectory anomaly detection (speed > 200mph → fraud). Route deviation: compare actual GPS path to expected route — deviations > 20% trigger review. ML model on trip features: time of day, location, fare, driver history → fraud score. Automatic refund trigger if fraud confirmed.

5. **Global architecture and localization:** Uber operates in 70+ countries, each with different: payment methods (cash in India), phone number formats, map providers (Baidu in China), currency and regulations. Architecture: core platform is global; country-specific adapters for payments, maps, and compliance. Data residency: trip data for Indian users stored in India (RBI regulations). GDPR: EU data in EU region. Multi-region active-active with regulatory data boundaries enforced at the routing layer.
