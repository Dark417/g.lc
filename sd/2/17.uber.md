# 17. Uber — Ride Sharing Platform

**Type:** commerce
**Difficulty:** L4–L5
**Frequency:** Very High — geo-matching, real-time location, supply-demand

---

## Problem Statement

Design a ride-sharing platform like Uber. Riders request rides from their location, drivers broadcast their location in real-time, the system matches riders with nearby drivers, and tracks the trip until completion. Must handle 10M rides/day globally, match riders within 2 seconds, and handle real-time GPS updates from 1M+ active drivers.

---

## Functional Requirements

**Must have:**
- Driver: update location every 4 seconds while active
- Rider: request a ride from current location to destination
- System: find and offer ride to nearest available driver within 2 sec
- Driver: accept or decline ride
- Track trip in real-time (rider sees driver location, ETA updates)
- Complete ride: fare calculation, payment, receipt

**Should have:**
- Upfront pricing (fare estimate before requesting)
- Dynamic pricing (surge pricing)
- Driver and rider ratings
- Ride cancellation

**Out of scope:**
- Payment processing (use Stripe/Braintree)
- Navigation turn-by-turn (use Google Maps API)
- Fraud detection (mention as extension)

---

## Non-Functional Requirements

- **Scale:** 10M rides/day; 1M active drivers at peak sending location every 4s = 250K location updates/sec
- **Matching latency:** Find and offer ride to driver within 2 seconds
- **Location freshness:** Driver location data < 4 seconds old
- **Availability:** 99.99% — any downtime loses revenue
- **Consistency:** Eventual for location; strong for ride assignment (no double-booking a driver)

## Capacity Estimation

```
Driver location updates:
  1M active drivers × 1 update/4 sec = 250,000 location writes/sec
  Per update: driver_id(8B) + lat(8B) + lng(8B) + heading(4B) + speed(4B) + timestamp(8B) = 40B
  Inbound bandwidth: 250K × 40B = 10MB/sec → trivial

Ride requests:
  10M rides/day = 115.7/sec average
  Peak: concert/event end or airport rush → assume 10× = 1,157/sec
  Each request: 1 geo-search + 1 matching computation

Active driver geo index:
  1M drivers × 50B per entry in Redis GEO = 50MB → trivially fits in one Redis node
  Redis GEOADD/GEORADIUS: O(N+log M) where M = results returned

Trip tracking (concurrent trips):
  Peak concurrent trips: 10M rides/day × 20 min avg trip / 1440 min/day ≈ 140K concurrent trips
  Each trip: 2 WebSocket connections (rider + driver) × 250 location pushes per trip
  Location push: 140K × 0.25/sec (1 push/4sec) = 35K WebSocket writes/sec

Surge pricing computation:
  Geohash precision 5 cells (~5km × 5km): ~1M cells globally
  Active cells during peak: ~10K
  Compute every 5 minutes: 10K cells × 5 min = 50K calculations/5min = trivial
```

## High-Level Architecture

```
[Driver App] ──→ [Location Service] ──→ [Redis Geo Index]
    │                                        │
    │                                [Nearby Driver Search]
    │                                        │
[Rider App] ──→ [Ride Request Service] ──→ [Matching Engine]
                                                │
                               [Driver Offer + Accept/Decline]
                                                │
                                      [Trip Management Service]
                                                │
                               ┌────────────────┼────────────────┐
                        [Driver App]    [Rider App]     [Payment Service]
                        (live tracking)  (ETA updates)

[Kafka pipelines]:
  driver-locations: 250K events/sec → Location Consumer → Redis + Trip Tracker
  ride-requests: 1K events/sec → Matching Engine
  trip-events: audit log for all state changes
```

**Components:**
- **Location Service:** Receives driver GPS updates (HTTP or WebSocket from driver app). Publishes to Kafka `driver-locations` topic. Consumer updates Redis GEO index. Stateless and horizontally scalable.
- **Matching Engine:** On ride request: geo-query nearby drivers → filter → score → dispatch offer sequentially to candidates. Owns the assignment state machine.
- **Trip Management Service:** Manages trip state machine. Source of truth for all trip state in PostgreSQL. Coordinating service for payment, notification, rating triggers.
- **Real-time Tracking:** One WebSocket server per trip (or subscribed via Kafka topic per trip). Pushes driver location updates to rider app; ETA recomputed each push.
- **Surge Pricing Service:** Reads demand (ride requests per cell) and supply (available drivers per cell) from Redis every 5 minutes. Computes surge multiplier. Stores in Redis with 5-minute TTL.

## Deep Dive 1: Matching Algorithm

**Full matching flow with state machine:**
```
1. Rider submits POST /v1/rides/request → ride_id created, status='searching'

2. Matching Engine picks up the request (via Kafka or direct call)

3. Geo-query nearby available drivers:
   GEORADIUS drivers:available:standard -122.4194 37.7749 5 km
     ASC             -- sorted by distance (closest first)
     COUNT 20        -- get top 20 candidates
     WITHCOORD       -- include coordinates for ETA calculation
   Returns: [(driver_id, distance_km, lat, lng), ...]

4. Filter candidates:
   - Minimum rating: driver_rating >= 4.5 (configurable per market)
   - Exclude drivers who declined this ride in the last 60 seconds (30s cooldown)
   - Exclude drivers with pending decline from a different ride request

5. Score remaining candidates:
   score(driver) = w1 / (1 + distance_km)        # proximity (dominant factor)
                 + w2 × driver_rating / 5.0        # quality signal
                 + w3 × acceptance_rate            # reliability signal
                 + w4 × (1 - recent_cancellations) # penalize cancellers
   weights: w1=0.6, w2=0.2, w3=0.1, w4=0.1

6. Dispatch offer to best-scoring available driver:
   - Send push notification (APNs/FCM) + WebSocket message to driver app
   - Message: { ride_id, pickup_distance_km, pickup_eta_sec, rider_rating, fare_estimate }
   - Start 15-second countdown timer

7. Wait for acceptance (timeout = 15 seconds):
   IF accepted within 15s:
     → Execute atomic assignment (see below)
     → Notify rider: "Driver found! {driver_name} arriving in {eta} min"
     → Status = 'matched'

   IF declined or no response within 15s:
     → Mark driver as "skipped" for this request (30s cooldown)
     → Move to next candidate in ranked list
     → If all 20 candidates exhausted: expand radius to 10km, repeat from step 3
     → If still no match after 10km search: notify rider "searching for driver..."

8. If no driver found after 5 minutes: suggest broader radius or different ride type
```

**Atomic driver assignment — preventing double-booking:**
```lua
-- Redis Lua script (executes atomically on Redis server)
local driver_key = 'driver:' .. KEYS[1]  -- KEYS[1] = driver_id
local ride_id = ARGV[1]

-- Check current driver status atomically
local status = redis.call('HGET', driver_key, 'status')

if status ~= 'available' then
  return {0, 'already_assigned'}  -- driver grabbed by another ride request
end

-- Atomically transition driver to 'busy'
redis.call('HMSET', driver_key,
  'status', 'busy',
  'ride_id', ride_id,
  'assigned_at', ARGV[2]  -- ARGV[2] = current timestamp
)

-- Remove from available geo set
redis.call('ZREM', 'drivers:available:' .. ARGV[3], KEYS[1])  -- ARGV[3] = ride_type

return {1, 'assigned'}

-- Python caller:
result = redis.eval(LUA_SCRIPT, 1, driver_id, ride_id, timestamp, ride_type)
success = result[0] == 1
```

**Why atomic Lua script?** Redis processes one command at a time within a script. A race condition between two ride requests offering the same driver is impossible — the second EVAL sees `status == 'busy'` and returns 0.

**G5:** Would identify a subtle liveness issue in the cascading offer model: if 20 drivers all decline sequentially (each taking up to 15 seconds to time out), the rider waits up to 300 seconds before the 10km expansion kicks in. Real Uber uses a smarter dispatch: batch matching. Instead of sequential dispatch (offer to driver 1, wait, offer to driver 2, wait...), simultaneously offer to the top 3 drivers with a "first to accept wins" model. When driver 1 accepts, send a cancellation notification to drivers 2 and 3. This reduces worst-case matching time from O(N × timeout) to O(timeout) regardless of how many drivers decline. The complexity is that two drivers might click "accept" at nearly the same time — the Lua atomic script ensures only one assignment succeeds, and the losing driver gets a "ride taken" notification with no penalty to their acceptance rate.

## Deep Dive 2: Real-Time Location and Tracking

**Driver location pipeline:**
```
Driver app (iOS/Android):
  GPS reading every 1 second (OS level)
  Send update every 4 seconds (battery/bandwidth optimization):
    PUT /v1/driver/location HTTP/2 with body { lat, lng, heading, speed, accuracy }
  If on trip: update frequency may increase to every 2 seconds for smoother tracking

Location API (stateless):
  Authenticate request (JWT verify, O(1))
  Publish to Kafka topic "driver-locations" (fire-and-forget, < 1ms)
  Return 200 OK (no waiting for Kafka ack in the critical path)
  Note: if driver has a pending assignment, piggyback it in the response

Location Consumer (Kafka consumer group):
  Reads from "driver-locations" topic
  For each event:
    1. GEOADD drivers:available:{ride_type} {lng} {lat} {driver_id}  # update geo index if available
    2. HMSET driver:{driver_id} last_lat {lat} last_lng {lng} last_seen {ts}  # always update state
    3. EXPIRE driver:{driver_id} 30  # reset TTL (auto-expire if driver goes offline)
    4. If driver is on a trip: publish to "trip-tracking:{ride_id}" Kafka topic
```

**Trip tracking via WebSocket:**
```
Architecture:
  Rider's app → WebSocket connection to Trip Tracker server
  Trip Tracker server → Kafka consumer for "trip-tracking:{ride_id}" topic

On each driver location update (every 4 seconds):
  1. Kafka consumer receives event: { driver_id, lat, lng, heading, speed, ride_id }
  2. Call ETA service: GET /eta?from={driver_lat,lng}&to={pickup_lat,lng}&via=road_graph
     (uses pre-computed road graph, not Google Maps API, for speed < 50ms)
  3. Push to rider via WebSocket: { driver_lat, driver_lng, driver_heading, eta_seconds, timestamp }
  4. If status = 'in_progress': also update dropoff ETA and push to both rider and driver

ETA computation:
  Use Dijkstra or A* on a pre-loaded road graph (map data from OpenStreetMap + traffic)
  Pre-computed shortest path trees for major city centers (fast lookup)
  Real-time traffic adjustment: weight road segment by current speed = historical_speed × traffic_factor
  Traffic factor updated every 5 minutes from aggregated GPS data of all active drivers
```

**Handling 250K location writes/sec:**
```
Kafka:
  250K events/sec → 50 partitions (5K events/sec per partition)
  200 consumers across 20 consumer instances → each handles ~1,250 events/sec

Redis Cluster (for geo index):
  Geo sets partitioned by ride_type: drivers:available:standard, :xl, :premium
  Each ride_type set: ~700K drivers × 50B = 35MB per set → trivially small
  Write throughput: 250K GEOADD/sec → single Redis primary handles ~100K writes/sec
  → Shard by geohash region: drivers:available:standard:us_east, :us_west, :eu, etc.
     Matching engine queries only the relevant regional shard
     Dramatically reduces per-shard write rate

Redis cluster sizing:
  5 primary nodes × 5 regions = 25 nodes
  + 1 replica per primary = 50 nodes total
  250K writes/sec / 25 primaries = 10K writes/sec per primary → comfortable
```

**G5:** Would note that the current model stores ALL available drivers in a single Redis GEO set per ride_type per region. GEORADIUS queries scan the geo index and return the N nearest — but if you have 500K drivers in a dense metro, a 5km radius might return 10K candidates, and the scoring/filtering logic must process all 10K before selecting the top 3 to offer. Uber's H3 hexagonal hierarchy (an open-source library Uber published) is a better primitive: H3 cells at resolution 7 (~1km² each) allow O(1) lookup of drivers in a specific cell and its neighbors without a full radius scan. For a 5km match radius, you only inspect ~49 H3 cells (a center cell + 2 rings of neighbors), each cell containing O(10) drivers in normal density. This reduces the matching search space from O(10K) candidates to O(500) candidates — a 20x reduction that matters at peak when matching latency is tight.

## Deep Dive 3: Surge Pricing

**Dynamic pricing algorithm:**
```
Every 5 minutes, for each active geohash precision-5 cell (~5km × 5km):

1. Count demand:
   demand = COUNT of ride_requests in this cell in last 10 minutes
   Source: read from Kafka "ride-requests" topic aggregated by geohash5

2. Count supply:
   supply = COUNT of available drivers in this cell
   Source: GEORADIUS on Redis geo index, count mode

3. Compute demand-supply ratio:
   ratio = demand / (supply × avg_vehicle_capacity)  # avg capacity ≈ 1.2 for standard

4. Map ratio to surge multiplier:
   if ratio < 0.8:  multiplier = 1.0x   # undersupply of demand, no surge
   elif ratio < 1.2: multiplier = 1.0x  # balanced
   elif ratio < 1.8: multiplier = 1.2x  # mild demand excess
   elif ratio < 2.5: multiplier = 1.5x  # moderate surge
   elif ratio < 4.0: multiplier = 2.0x  # high surge
   else:             multiplier = min(4.0, 1 + ratio × 0.6)  # capped at 4x per regulations

5. Store in Redis:
   SETEX surge:{geohash5_prefix} 300 "{\"multiplier\":1.5,\"updated_at\":\"2026-03-06T...\"}"
   300 = 5-minute TTL (auto-expires, forcing fresh recompute)

On ride estimate request:
   GEORADIUS to find geohash5 of pickup location
   GET surge:{geohash5} → multiplier (default 1.0 if key missing)
   Apply to base fare: estimated_fare = base_fare × multiplier
```

**Regulatory considerations:**
```
Some cities (NYC, etc.) cap surge at specific multipliers via regulation
Configuration: per-city max_surge_multiplier stored in city configuration table
  GET surge cap: SELECT max_surge FROM city_config WHERE city = 'NYC'
  Applied as: multiplier = min(computed_multiplier, max_surge)

Price transparency:
  Show surge multiplier clearly before rider confirms: "1.5x surge pricing in effect"
  Require explicit confirmation for surge > 2x (UX requirement)
  Store surge_multiplier at time of request in rides table (immutable after booking)
```

## Reliability and Operations

- **Location service down:** Driver location data in Redis expires via 30-second TTL. On service restart: drivers must re-send a location update (happens automatically at next 4-second GPS cycle). Maximum data loss: 30 seconds of location staleness — acceptable since location updates every 4s anyway.
- **Matching service failure:** Ride requests queued in Kafka `ride-requests` topic (default retention: 7 days). On matching service recovery: process queued requests in order. Riders see "searching..." state for longer during the outage. No requests are lost; FIFO ordering maintained per partition.
- **Trip in progress during central outage:** Trip state persisted in PostgreSQL (durable). Driver app caches the current trip state locally (pickup/dropoff addresses, rider info). Trip continues even if server is unreachable for minutes — driver can complete the trip offline. On reconnect: sync final state (completion, GPS trail) to server.
- **Partial assignment (driver accepted, DB write failed):** Use idempotent upsert: `INSERT INTO rides (ride_id, driver_id, ...) ON CONFLICT (ride_id) DO UPDATE SET driver_id = EXCLUDED.driver_id`. If the DB write fails after Redis assignment: matching engine retries with exponential backoff. Redis Lua script ensures driver is still marked 'busy' during retry. If retry fails after 3 attempts: compensate by releasing the driver in Redis (reverse the Lua assignment) and re-offer to next candidate.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis GEO for driver index | PostGIS (PostgreSQL extension) | Redis: in-memory, sub-millisecond GEORADIUS queries, simple TTL-based expiry; PostGIS: persistent, richer spatial queries (routing-aware), but ~10ms query latency vs <1ms |
| WebSocket for trip tracking | HTTP long-polling | WebSocket: true real-time (< 100ms delivery), efficient (no repeated HTTP headers); Long-polling: simpler to implement, works behind restrictive proxies, but 1–4s latency |
| Kafka for location pipeline (async) | Synchronous Redis write from app | Kafka: durable, replay on consumer failure, decoupled; Direct Redis: simpler, ~5ms faster, but no durability or replay if consumer crashes |
| Per-cell geohash surge pricing | Global surge multiplier | Per-cell: accurate (airport is 5x, downtown 1.2x simultaneously); Global: simpler, but blunt instrument that either under-charges in hot zones or over-charges in quiet areas |

## G5

1. **Driver pre-positioning with demand prediction:** Reactive dispatch (respond when a ride is requested) leaves riders waiting for a driver to travel to their area. Proactive approach: ML model trained on 2+ years of historical ride data, real-time event calendar (concerts, airports, sports), weather, and time of day predicts demand heatmaps for the next 30 minutes at geohash precision-5 resolution. Output: a heatmap of expected demand per cell. Drivers in low-demand cells adjacent to predicted high-demand areas receive in-app "suggested destination" prompts with an earnings incentive (guaranteed surge rate if they arrive before demand materializes). This is Uber's "Expected Demand" and "Heat Map" features. Reduces average rider ETA by 20–30% during predictable surge events compared to purely reactive dispatch.

2. **Pool / shared rides matching:** Uber Pool requires matching multiple riders with similar routes within a time window. Algorithm: (1) Collect ride requests within the same geohash5 cell and a 5-minute window. (2) Cluster by destination proximity (k-means on dropoff coordinates). (3) For each cluster, run a route optimization (Traveling Salesman Problem variant with constraints): maximum 2 riders per vehicle, no rider's detour can increase their ETA by more than 10 minutes. (4) Solve via heuristic (nearest neighbor + 2-opt improvement) in < 2 seconds. (5) Assign to the same driver atomically (same Lua script, extended for pool assignment). The TSP variant with time window constraints is NP-hard in general, but with 2 riders and practical geographies the heuristic finds near-optimal solutions 95% of the time.

3. **Two-sided marketplace global optimization:** At scale, per-request greedy dispatch is suboptimal. Consider: 5 riders and 5 drivers, all nearby, but each greedy first-choice assignment creates 3 drivers sitting idle while 2 riders wait longer than necessary. Batch matching: accumulate ride requests over a 3-second window, then solve a global assignment problem (minimize total weighted wait time across all matched pairs). Formulation: bipartite matching with cost matrix C[rider, driver] = expected_wait_time. Hungarian algorithm solves this in O(N³); for N=1000 simultaneous requests (a large city during peak), N³ = 10⁹ operations — too slow. Practical alternative: auction-based matching (each driver bids on their preferred rider, conflicts resolved by re-bidding). Uber's own research shows batch matching improves market efficiency by 15–20% on overall wait time compared to greedy sequential dispatch.

4. **Fraud detection on trips:** Common fraud patterns: (a) GPS spoofing — driver fakes location to avoid long pickups. (b) Fare manipulation — driver takes a circuitous route to inflate mileage-based fare. (c) Account takeover — stolen payment method used to request rides. Detection mechanisms: (a) GPS trajectory validation: speed > 150 mph, or position jumps > 1km in 4 seconds → physics violation → flag trip for review. (b) Route deviation score: compute the ratio of actual driven distance to optimal route distance via road graph; deviation > 30% → automatic refund trigger. (c) Payment anomaly: new account + premium listing + payment velocity > 3 rides/hour → hold for manual review. ML model (XGBoost) on 50+ trip features produces a fraud probability score; > 0.8 → auto-refund + driver deactivation review.

5. **Global architecture with regulatory data residency:** Uber operates in 70+ countries with conflicting regulatory requirements: India (RBI mandates trip data stored in India), EU (GDPR — EU resident data in EU), China (operates as DiDi under different brand with separate infrastructure). Core platform services are global; data residency is enforced at the API gateway layer via a "data jurisdiction" header derived from the rider's country at account creation. PostgreSQL for ride records: deployed per regulatory region (India cluster, EU cluster, US cluster). Trip data is written only to the appropriate regional cluster. Global analytics: anonymized aggregate data flows to a central data warehouse; PII stays regional. This is implemented as a routing decision at the Trip Management Service: `SELECT cluster FROM data_jurisdiction WHERE country = rider.country` determines which PostgreSQL cluster receives the write.
