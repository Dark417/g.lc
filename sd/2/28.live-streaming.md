# 28. Live Streaming Platform (Twitch-like)

**Type:** data
**Difficulty:** G5
**Frequency:** Medium-High — real-time data pipelines, WebRTC, fan-out

---

## Problem Statement

Design a live streaming platform like Twitch. Streamers broadcast live video, which is watched by thousands to millions of concurrent viewers with < 5-second latency. The platform must handle 100K concurrent live streams, 10M concurrent viewers, real-time chat, and stream recording/VOD.

---

## Functional Requirements

**Must have:**
- Stream ingest: broadcaster uploads live video (RTMP or WebRTC)
- Transcode live stream to multiple qualities (360p, 720p, 1080p)
- Deliver live stream to viewers via HLS or DASH (< 5s latency)
- Real-time chat: viewers can send messages to the stream (< 1s delivery)
- Viewer count display (near-real-time)

**Should have:**
- DVR rewind: viewers can pause and rewind up to 60 seconds
- Stream recording → auto-publish as VOD after stream ends
- Clips: viewers can clip 30-second highlights
- Stream key management: broadcaster gets a unique stream key

**Out of scope:**
- Monetization (subscriptions, bits, ad revenue sharing)
- Stream scheduling and notifications
- Channel points and game integrations

---

## Non-Functional Requirements

- **Scale:** 100K concurrent live streams; 10M concurrent viewers; 1M concurrent chat messages/sec
- **Latency:** End-to-end < 5 seconds (broadcast → viewer screen)
- **Availability:** 99.99% — streamers lose income if their stream goes down
- **Scalability:** Viral stream goes from 0 to 1M viewers in minutes
- **Quality:** Adaptive bitrate; smooth playback even on variable networks

## Capacity Estimation

```
Ingest bandwidth:
  100K concurrent streams × 5 Mbps avg ingest = 500 Gbps ingest
  Each ingest PoP handles 10Gbps → need 50 ingest PoPs globally
  Regional distribution: 40% North America, 30% Europe, 20% Asia, 10% other

Delivery bandwidth:
  10M concurrent viewers × 2 Mbps avg (720p) = 20 Tbps delivery
  → Must use CDN; impossible to serve 20 Tbps from origin infrastructure
  CDN cost at $0.008/GB: 20Tbps × 3600s/hr × 0.008/8 = $72,000/hour at peak

Transcoding:
  100K streams × 3 quality levels (360p, 720p, 1080p) = 300K encoding sessions
  Each GPU encoder handles ~20 simultaneous NVENC hardware encodes
  → 300K / 20 = 15,000 GPU instances (spot instances, scale with stream count)
  At $0.90/GPU-hour: 15,000 × $0.90 = $13,500/hour transcoding cost

Chat:
  1M messages/sec (across all streams; peak single stream: 100K msg/sec for 1M-viewer stream)
  Message size: 200B → 200MB/sec total chat traffic
  Fan-out: 1 message → 100K WebSocket pushes (for a viral stream) → 20GB/sec of chat delivery

Storage:
  Recordings: 100K streams × 3hr avg × 5 Mbps = 225TB/day
  With compression to storage quality (2 Mbps): 90TB/day
  S3 at $0.023/GB: $2,070/day for raw VOD storage
  Lifecycle: hot (< 90 days) on S3 Standard; warm (> 90 days) on S3 Glacier ($0.004/GB)

Segments per stream:
  LLHLS: 2-second segments × 3 qualities × 100K streams = 300K segment writes/sec
  Each segment ≈ 500KB → 150GB/sec S3 write throughput → use multi-region S3 with Transfer Acceleration
```

## High-Level Architecture

```
[Broadcaster]
    │ RTMP (port 1935) or SRT (port 9000)
    ▼
[Ingest PoP — nearest to broadcaster]
    │  raw H.264/H.265 stream
    ▼
[Transcoding Cluster — GPU workers]
    │  3 quality levels in parallel
    ▼
[Segment Writer] ──→ [S3: HLS segments + manifests]
                           │
                      [CDN: caches segments]
                           │
                      [Viewers pull segments]

[Stream Metadata DB (PostgreSQL)]
[Stream State (Redis)] ── live/offline, viewer count, current segment index

[Chat Service]
    [Viewer] ──WebSocket──→ [Chat Server (stateful, per-stream)]
    [Chat Servers] ←── [Kafka: chat-{stream_id}] ←── [Message from any viewer]
    [Chat archive → Cassandra]

[Recording Service]
    Concatenate segments → MP4 → publish as VOD
    [VOD Service → S3 + CDN]

[Viewer Count Service]
    Heartbeats from CDN → Redis HyperLogLog per stream → approximate viewer count
```

```sql
CREATE TABLE streams (
  stream_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  channel_id         UUID NOT NULL,
  title              TEXT NOT NULL,
  category           TEXT,
  tags               TEXT[],
  started_at         TIMESTAMPTZ,
  ended_at           TIMESTAMPTZ,
  peak_viewers       INT,
  total_views        BIGINT,
  status             TEXT NOT NULL DEFAULT 'created',
    -- 'created' | 'live' | 'ended' | 'processing_vod' | 'vod'
  stream_key_hash    TEXT NOT NULL UNIQUE,    -- hashed; original key not stored
  ingest_server      TEXT,                   -- which ingest PoP received the stream
  transcode_job_id   TEXT,                   -- reference to transcoding job
  recording_s3_path  TEXT,                   -- S3 path for VOD after processing
  created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_streams_live ON streams (category, peak_viewers DESC)
  WHERE status = 'live';
```

```
-- Cassandra: chat messages
-- Partition key = stream_id (all messages for a stream on same nodes)
-- Clustering key = message_id TIMEUUID DESC (newest first)
CREATE TABLE chat_messages (
  stream_id    UUID,
  message_id   TIMEUUID,

```

## Deep Dive 1: Low-Latency Video Pipeline

**Why standard HLS has high latency:**
```
Standard HLS (RFC 8216):
  Segment duration: 6–10 seconds
  Player buffers 3 segments before playing: 18–30 seconds
  Origin response time + CDN cache: + 5 seconds
  Total: 25–40 second latency (glass-to-glass)
  → Used for sports on cable; fine for asynchronous viewing

Low-Latency HLS (LLHLS, Apple WWDC 2019):
  Segment duration: 2 seconds
  Partial segments: 0.2-second "parts" (TS chunks within each segment)
  Player buffers: 2 segments ahead
  Total: 2–5 second latency
  → What Twitch uses today

WebRTC (for sub-1-second):
  Uses SFU (Selective Forwarding Unit), not CDN
  Works for < 10K viewers; CDN-unfriendly at 1M+ viewers
  Latency: 100–500ms (glass-to-glass)
```

**Complete LLHLS pipeline:**
```
Step 1: Broadcaster → Ingest PoP
  RTMP connection: TCP, reliable, broadcaster's OBS/Streamlabs pushes to nearest ingest PoP
  SRT (Secure Reliable Transport): better for high packet loss connections (cellular, wifi)
    SRT: UDP-based, ARQ (automatic repeat request), ~150ms overhead, 30% less dropped frames

Step 2: Ingest → Transcoder (GPU cluster)
  Ingest PoP receives raw H.264 stream → forwards to transcoding worker via gRPC
  Transcoder: NVENC hardware encoding on GPU (10× faster than CPU x264)
  Simultaneous outputs:
    360p @ 500 Kbps (mobile low bandwidth)
    720p @ 2.5 Mbps (standard)
    1080p @ 5 Mbps (high quality)
  GPU instance: handles 15–20 simultaneous stream encodes
  Encoding latency: < 100ms per segment (hardware encode)

Step 3: Segment generation → S3
  Every 2 seconds: write new segment files
    stream-abc/360p/seg-00001.ts  (500KB)
    stream-abc/720p/seg-00001.ts  (625KB)
    stream-abc/1080p/seg-00001.ts (1.25MB)
  Every 200ms: write partial segment (LL-HLS "part")
    stream-abc/720p/seg-00001-part-003.ts  (125KB)
  Update manifest file (m3u8): add new segment + partial segment references
  CDN invalidation: surrogate-key purge for this stream's manifest URL

Step 4: Viewer segment download
  Player requests master manifest → selects quality based on bandwidth estimate
  Player polls segment manifest every 2 seconds (new segments available)
  For LLHLS: player uses HTTP/2 server push or long-polling on manifest endpoint
    → Manifest server pushes update when new partial segment available → < 200ms notification
  Player downloads segment → decodes → renders

End-to-end latency breakdown:
  Encode: 100ms
  Segment write to S3: 200ms
  CDN propagation: 50ms
  Player buffer: 2 segments = 4 seconds
  Total: ~4.5 seconds (within 5s SLA)
```

**Ultra-low latency (< 1 second) with WebRTC SFU:**
```
Architecture: Broadcaster → WebRTC SFU → Viewer WebRTC clients
  SFU (Selective Forwarding Unit): server that receives one stream and forwards to N clients
  Latency: < 500ms (only 1 server hop + jitter buffer)
  Scalability: 1 SFU handles 1,000–10,000 viewers
  For 1M viewers: cascade SFUs (SFU tree, 3 levels, 1000-fan-out per level)

Limitations:
  - Expensive: dedicated WebRTC infrastructure vs CDN edge caching
  - Each viewer needs unique UDP flow (can't share cached segments like HLS)
  - Firewall traversal: UDP blocked by many corporate firewalls (fallback to TCP)
  - No DVR rewind (no cached segments)

Use cases: casino live dealers, telemedicine, interactive live auctions
Not suitable: 1M+ concurrent viewers (HLS CDN is 100× cheaper at scale)
```

**G5:** LLHLS reduces viewer latency to 3–5 seconds, but introduces a subtle viewer synchronization problem for streams with interactive elements (polls, predictions, game events). Different viewers are at different points in the stream: a viewer with fast internet is 2 seconds behind live; a viewer with poor internet (buffering) is 8 seconds behind. When a game event fires at second T, the fast viewer sees it at T+2, the slow viewer at T+8 — a 6-second difference. For chat (text), this creates the "chat is ahead of my stream" experience that every Twitch viewer knows. True synchronization requires either: (a) reference player position to server time and clip manifest URLs to a known offset (complex client implementation), or (b) accept the latency variance and design interactive features with a delay window that covers the maximum expected viewer lag (e.g., polls close 10 seconds after stream event, not 5 seconds). The second approach is what Twitch currently does for predictions and polls.

## Deep Dive 2: Chat Fan-out at Scale (1M viewers)

**The problem:** A chat message from one viewer must reach 1M other viewers within 1 second.

**Tiered fan-out architecture:**
```
Message flow:
  Viewer types message → WebSocket → Chat Server A
      → Kafka: topic "chat-{stream_id}" (single partition; ordered messages)
           → 10 Regional Aggregator Services (one per geographic region)
                 → Each aggregator pushes to 100 Chat Servers in its region
                       → Each Chat Server pushes to ~1,000 local WebSocket connections

Math:
  Kafka → 10 aggregators: 10 Kafka consumers (fast, parallel)
  Each aggregator → 100 chat servers: 100 TCP connections per aggregator (small fan-out)
  Each chat server → 1,000 viewers: WebSocket write (in-process loop)
  Total: 1M viewer deliveries in 3 hops
  Latency budget: Kafka (20ms) + aggregator→server (10ms) + server→viewer (5ms) = 35ms
  Well within 1-second SLA

Without tiered fan-out (naive: all chat servers consume Kafka directly):
  1,000 chat servers × 1,000 viewers/server = 1M viewers
  Each chat server is a Kafka consumer → 1,000 Kafka consumers per stream
  Kafka can handle this for 100K streams × 1 consumer/stream, but:
  For 1M-viewer streams with high message rate, having 1,000 consumers per topic partition
  creates coordination overhead and replay contention → use tiered aggregation instead
```

**Per-chat-server state:**
```
Each chat server maintains:
  connections: Map<stream_id, Set<WebSocket>>   // viewers watching each stream
  On WebSocket connect: register (stream_id, ws) → subscribe to Kafka topic for stream_id
  On WebSocket disconnect: deregister
  On Kafka message: for each ws in connections[stream_id]: ws.send(message)

  Max connections per server: 10,000–100,000 WebSockets (depends on memory, OS limits)
  Each WebSocket: ~32KB RAM (socket buffer + metadata) → 10K × 32KB = 320MB
  → 64GB server supports ~2M WebSocket connections (with tuned OS settings)
```

**Message persistence and scroll-back:**
```
Chat messages persisted to Cassandra (async, after fan-out):
  Kafka → Flink consumer → Cassandra write (partitioned by stream_id)
  Does not block real-time delivery
  Scroll-back: SELECT FROM chat_messages WHERE stream_id = ? LIMIT 50 → last 50 messages
  New viewer joining mid-stream sees last 50 messages (context)
```

**G5:** Kafka-based chat fan-out serializes message order within a single partition (stream_id → consistent partition). This is correct for ordering, but creates a hot partition problem: a viral stream with 100K messages/min maps to one Kafka partition, which has a max throughput of ~100MB/sec (plenty for chat text). However, if the chat server fleet for this stream also Kafka-consumer-fan-out to each viewer, you'd need 1,000 separate Kafka consumer group members reading the same single partition — but Kafka only allows one consumer per partition per consumer group. The tiered aggregation architecture above solves this, but it means the 10 regional aggregators are now a single point of ordered delivery per stream. If an aggregator fails, its 100K viewers in that region get chat latency until the aggregator restarts and catches up from its Kafka offset. The fix: make aggregators stateless with a 10-second Kafka retention window — on restart, they re-consume the last 10 seconds and de-duplicate already-delivered messages via a per-viewer sequence number in the WebSocket protocol.

## Deep Dive 3: Viewer Count and Engagement Metrics

**Real-time viewer count at 10M concurrent viewers:**
```
Approach 1: HyperLogLog (chosen approach)
  On each segment request (every 2 seconds per viewer):
    PFADD stream:{stream_id}:viewer_hll:{minute} {viewer_id}
  Query viewer count: PFCOUNT stream:{stream_id}:viewer_hll:{minute}
  Accuracy: ±0.81% error rate (hardcoded in HyperLogLog algorithm)
  Memory: 12KB per stream per minute, regardless of viewer count
  10M viewers on 1 stream: 12KB (!)

  Decay: keep last 5 minutes of HLL keys; viewer count = PFCOUNT of merged last 3 minutes
  This smooths out the 2-second segment download timing jitter

Approach 2: Heartbeat counter (simpler)
  On segment fetch: INCR viewer_count:{stream_id}; EXPIRE 90 seconds
  On new segment (2s): refresh expiry → counter stays alive while viewer is active
  On disconnect: counter naturally decrements when heartbeat stops (90s expiry)
  Accuracy: ±30 seconds (a departed viewer's count persists up to 90s)
  Memory: 1 integer per stream → negligible

Approach 3: Sampled counting (for massive scale)
  Route 10% of viewer heartbeats to count service → multiply by 10
  Error: ±10% → only acceptable for rough public display

Production: use HyperLogLog (Approach 1) — exact enough (±1%), memory-efficient,
  and provides distinct user count (not just request count)
```

## Reliability and Operations

- **Ingest server failure:** Broadcaster's RTMP client detects connection drop → automatic reconnect (within 5 seconds). On reconnect: same or new ingest PoP (DNS load balance). Transcoder detects stream timeout → pauses segment writing → viewer sees "stream offline" → when broadcaster reconnects, transcoder resumes. CDN serves last cached segments during brief reconnection gap (< 10s → most viewers don't notice).
- **Viral stream scale-out (0 → 1M viewers in 5 minutes):** CDN handles delivery auto-scaling (stateless cache). Chat servers must scale: Kubernetes HPA on WebSocket connection count. Target: spin up new chat server pod in < 60 seconds. Initial pod count: 1 per active stream; scale out as viewers join. Transcoding: already running for this stream; no change needed (encoding is 1:1 with stream count, not viewer count).
- **Stream recording reliability:** Transcoder writes segments to S3 concurrently with CDN upload (two S3 PUT operations per segment). On stream end: a background job lists all segment files from S3 `s3://recordings/stream-abc/` → invokes ffmpeg concat → creates MP4 → uploads to VOD bucket → marks stream status as 'vod'. Segment completeness check: verify segment sequence is contiguous (no gaps) before transcoding to MP4; fill gaps with black frames if needed.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| LLHLS (2s segments, CDN delivery) | WebRTC SFU | LLHLS: scales to millions via CDN, DVR-compatible, robust; WebRTC: < 500ms latency but 100× more expensive at 1M+ viewers, complex infrastructure |
| Kafka for chat fan-out | Redis pub/sub | Kafka: durable, replayable, ordered per stream; Redis pub/sub: lower latency (sub-ms), no persistence — lost messages on server restart |
| HyperLogLog for viewer count | Exact counter (Cassandra counter column) | HLL: 12KB regardless of viewer count, ±0.8% error; Exact: 8B per viewer in set = 80MB at 10M viewers, but exact |
| CDN for video delivery | P2P (WebTorrent / Livepeer) | CDN: reliable, low latency, mature; P2P: dramatically cheaper at scale but unreliable (peers leave), inconsistent latency, low user adoption |
| Tiered aggregator for chat fan-out | Direct all-servers-consume-Kafka | Tiered: manageable consumer count, regional isolation; Direct: simpler but requires N_servers consumers per popular stream's partition |

## G5

1. **Stream health monitoring and adaptive encoding:** Broadcaster's network often degrades before the stream drops entirely. Monitor ingest stream metrics in real-time at the ingest PoP: bitrate (Kbps), frame rate (fps), keyframe interval (seconds), packet loss rate. Thresholds: bitrate drops > 30% for 10 seconds → push "low bitrate" alert to broadcaster overlay (OBS plugin). Keyframe interval > 4s → viewers can't seek in DVR, HLS segment boundaries become irregular → alert. Adaptive response: if broadcaster's available bandwidth drops → automatically switch transcoder to lower output resolution set (720p + 360p instead of 1080p + 720p + 360p) → frees 40% of bitrate → more stable stream. Send stream health score (0–100) to broadcaster in real-time via WebSocket to the dashboard, so they can troubleshoot without relying on viewer complaints.

2. **Interactive features — polls, predictions, channel points:** Modern streams are interactive: viewers predict game outcomes, vote in polls, redeem channel points. Architecture: a real-time event bus per stream (separate from chat: lower volume, higher priority). Event types: `poll_created`, `poll_vote`, `prediction_outcome`, `celebration_trigger`. Fan-out: same tiered aggregation as chat, but delivered as WebSocket events with `type: "overlay"`. Client renders an interactive overlay UI on top of the video player. Viewer actions (vote, predict): client → REST API → DB write + event publish. Aggregation: Flink computes real-time vote tallies; pushes result back to stream event bus within 200ms. Round-trip < 1 second for any interactive action. Challenge: synchronizing interactive event timing with video playback position (viewer at different latency offsets sees different video frames when poll fires).

3. **Content moderation at stream scale:** 100K live streams × 1M chat messages/sec = 1B messages/hour — impossible to manually moderate. Architecture: tiered moderation pipeline. Tier 1 (< 1ms, in Flink): regex/keyword filter for hard rules (slurs, spam URLs, phone numbers). Block immediately before fan-out. Tier 2 (< 10ms, ML microservice): toxicity classifier (DistilBERT fine-tuned on Twitch data), hate speech detection. Score > 0.9 → block; 0.5–0.9 → hold for human review (shown to moderators' queue). Tier 3 (human moderation): channel moderators + Twitch staff review flagged messages; retroactively delete + ban. AutoMod: combine all tiers; channel-configurable strictness levels (1–4). At 1M msg/sec: Tier 2 model must run < 1ms/message → use ONNX runtime with INT8 quantization; horizontal scale ML servers.

4. **Global ingest PoPs and broadcaster quality of experience:** A broadcaster in Tokyo streaming to US-WEST datacenter adds 150ms round-trip latency, causing RTMP connection instability and dropped frames. Solution: ingest PoPs in every major city (Tokyo, Singapore, Sydney, Mumbai, São Paulo, Lagos, etc.). Broadcaster's streaming software connects to nearest PoP via DNS round-robin + health check redirect. PoP relays stream to centralized transcoding via CDN backbone (Cloudflare/Akamai backbone — 10–40Gbps, 5–30ms cross-continent). SRT protocol (UDP-based, ARQ): for broadcasters on cellular or poor WiFi, SRT recovers from 10–15% packet loss with < 200ms added latency, compared to RTMP which drops frames on > 2% packet loss. Adoption: provide OBS plugin or SDK that auto-selects RTMP vs SRT based on measured packet loss to the ingest PoP.

5. **DVR and time-shift at scale:** Viewers can pause and rewind up to 60 seconds behind live. The implementation: CDN segment retention per stream. CDN edge: keep last 30 LLHLS segments (60 seconds at 2s per segment) per quality tier, even as new segments arrive. Segment TTL rule: `min(60s from live_edge, cache_capacity)`. "Time-shifted" viewer requests an older segment URL → CDN cache hit (segment not yet evicted) → serves it. When stream ends: retain last 60s of segments for 5 minutes (so viewers watching DVR during ending can finish playback). Storage cost: 100K streams × 30 segments × 3 qualities × 500KB = 4.5TB of "hot" DVR data in CDN simultaneously. The non-obvious issue: if a viewer is paused and then seeks back more than 60s, they've exceeded the DVR window. The graceful degradation: serve an error frame with "Live content no longer available — click to go to live" — do not allow the player to request segments beyond the DVR window, as the CDN will 404 on them, causing confusing errors.
