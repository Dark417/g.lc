# 28. Live Streaming Platform (Twitch-like)

**Type:** data
**Difficulty:** L5
**Frequency:** Medium-High — real-time data pipelines, WebRTC, fan-out

---

## Problem Statement

Design a live streaming platform like Twitch. Streamers broadcast live video, which is watched by thousands to millions of concurrent viewers with < 5-second latency. The platform must handle 100K concurrent live streams, 10M concurrent viewers, real-time chat, and stream recording/VOD.

---

## Functional Requirements

**Must have:**
- Stream ingest: broadcaster uploads live video (RTMP or WebRTC)
- Transcode live stream to multiple qualities (360p, 720p, 1080p)
- Deliver live stream to viewers via HLS or DASH (< 5s latency)
- Real-time chat: viewers can send messages to the stream (< 1s delivery)
- Viewer count display (near-real-time)

**Should have:**
- DVR rewind: viewers can pause and rewind up to 60 seconds
- Stream recording → auto-publish as VOD after stream ends
- Clips: viewers can clip 30-second highlights
- Stream key management: broadcaster gets a unique stream key

**Out of scope:**
- Monetization (subscriptions, bits, ad revenue sharing)
- Stream scheduling and notifications
- Channel points and game integrations

---

## Non-Functional Requirements

- **Scale:** 100K concurrent live streams; 10M concurrent viewers; 1M concurrent chat messages/sec
- **Latency:** End-to-end < 5 seconds (broadcast → viewer screen)
- **Availability:** 99.99% — streamers lose income if their stream goes down
- **Scalability:** Viral stream goes from 0 to 1M viewers in minutes
- **Quality:** Adaptive bitrate; smooth playback even on variable networks

---

## Capacity Estimation

```
Streams: 100K concurrent × 5 Mbps avg = 500 Gbps ingest
Viewers: 10M concurrent × 2 Mbps avg (720p) = 20 Tbps delivery
  → Must use CDN; origin can't serve 20 Tbps directly

Chat: 1M messages/sec (peak, viral stream)
  Each message: 200B → 200 MB/sec for chat alone
  Fan-out: message to 100K viewers of one stream = 100K WebSocket pushes/sec per message

Storage (recordings): 100K streams × 3hr avg × 5 Mbps = 675 TB/day
  With compression and lower bitrate for storage: ~200 TB/day
  S3: cheap long-term storage
```

## Deep Dive 1: Low-Latency Video Pipeline

**Traditional HLS latency problem:**
```
Standard HLS: 10-second segments → 30-second end-to-end latency
Low-Latency HLS (LLHLS): 0.5-2 second segments → 3-7 second latency
```

**End-to-end pipeline:**
```
Broadcaster → Ingest Server:
  RTMP (Real-Time Messaging Protocol): industry standard, < 100ms
  SRT (Secure Reliable Transport): better for unreliable networks

Ingest Server → Transcoder:
  Relay raw stream to transcoding cluster (GPU machines)
  Transcoder generates multiple quality outputs simultaneously:
    360p @ 500Kbps, 720p @ 2Mbps, 1080p @ 5Mbps

Transcoder → CDN:
  Output: LLHLS segments (2 seconds each) + partial segments (every 200ms for live edge)
  Upload to S3 + invalidate CDN origin cache
  CDN edge caches segments: each segment is small (400KB for 2s at 720p) → easy to cache

Viewer experience:
  Download manifest every 2 seconds (to discover new segments)
  Download 2-second video segments
  Buffer: maintain 3 segments (6 seconds) ahead of playback
  On bandwidth drop: switch to lower quality segment stream

Timeline:
  Broadcast → encode → upload to S3 → CDN fetch → viewer download = ~3-5 seconds
```

**Ultra-low latency (< 1 second) with WebRTC:**
```
WebRTC: peer-to-peer or SFU-based (Selective Forwarding Unit)
  Broadcaster → WebRTC ingestion server (SFU)
  SFU → fans out to up to 10K viewers via WebRTC data channel
  For > 10K viewers: cascade to another SFU layer

Latency: < 500ms (glass-to-glass)
Cons: complex, expensive, doesn't work well with CDN
Used by: Twitch co-streams, Discord Go Live, Zoom
Not suitable for: 1M+ viewer streams (CDN HLS is better at that scale)
```

## Deep Dive 2: Chat Fan-out at Scale (1M viewers)

**The problem:** 1 chat message → must reach 1M viewers within 1 second.

**Architecture:**
```
[Viewer] ──WebSocket──→ [Chat Server A]
[Viewer] ──WebSocket──→ [Chat Server B]
[Viewer] ──WebSocket──→ [Chat Server C]
              ...
[Broadcaster types message] ──→ [Chat Server X]
              │
              ▼
      [Kafka: chat-{stream_id}]
              │
    [All Chat Servers consume]
              │
    [Push to all connected WebSocket clients]
```

**Each chat server:**
- Maintains 10K–100K WebSocket connections
- Subscribed to Kafka topic for the stream
- On message: push to all local connections

**Challenge:** 10K viewers per server × 1,000 servers = 10M connections. 1 message → 1,000 Kafka consumers → each pushes to 10K connections.

**Math:**
```
1M viewer stream → 10K chat messages/sec
Each message: 1K servers × delivery to their 1K local viewers = 1K deliveries per message
Total: 10K messages/sec × 1K servers = 10M deliveries/sec

Per server: 10K messages/sec × delivery to 1K viewers = 10M WebSocket writes/sec on each server
→ Too much! Need different approach for 1M viewer streams
```

**Solution for mega-streams:**
```
Hierarchical fan-out:
  1. Message written to Kafka
  2. 10 "region aggregators" subscribe to Kafka
  3. Each region aggregator fans out to 100 chat servers in their region
  4. Each chat server pushes to its 1K local connections

Result:
  Kafka → 10 aggregators (fast, 10 consumers)
  Each aggregator → 100 chat servers (100 local connections per aggregator)
  Each chat server → 1K viewers
  Total: 1M viewers, 3 hops, < 500ms

For 10M viewer streams: add another hierarchy layer
```

## Deep Dive 3: Viewer Count and Engagement Metrics

**Real-time viewer count challenge:**
```
10M concurrent viewers — can't count exact connections at scale

Approach 1: HyperLogLog in Redis
  On each viewer segment request: PFADD viewer_hll:{stream_id}:{minute} {viewer_id}
  Approximate distinct viewer count with 0.81% error
  Very memory efficient: 12KB per stream regardless of viewer count

Approach 2: Heartbeat counter
  Each viewer sends heartbeat every 30 seconds (or on segment fetch)
  Server: INCR viewer_count:{stream_id} on heartbeat; DECR on disconnect
  Approximate: ±30 seconds lag
  Memory: 1 integer per stream

Approach 3: Sampling
  Count viewers on 10% of chat servers → extrapolate × 10
  Fast, low overhead, acceptable approximation (±10%)
```

## Reliability and Operations

- **Ingest server failure:** Broadcaster reconnects automatically (RTMP has reconnection logic). Transcoder notices stream ended → marks stream as offline. Broadcaster reconnects to any available ingest server. CDN serves cached segments during reconnection gap.
- **Viral stream scale-out:** Stream gets picked by Twitter → goes from 10K to 1M viewers in 5 minutes. CDN handles this automatically (horizontal scaling). Chat servers need to scale: auto-scale based on active connections count. Viewer count service: Redis counters handle any scale.
- **Stream recording reliability:** Transcoder also writes segments to S3 concurrently with CDN upload. On stream end: compile segments into MP4 (ffmpeg concat) → publish as VOD.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| LLHLS for delivery | WebRTC | LLHLS: scales to millions, CDN-friendly; WebRTC: < 1s latency but complex at scale |
| Kafka for chat fan-out | Direct pub/sub (Redis) | Kafka: durable, replay; Redis pub/sub: lower latency, no persistence |
| HyperLogLog viewer count | Exact counting | HLL: constant memory; Exact: correct but 10M counters = 80MB RAM |
| CDN for video | P2P (WebTorrent) | CDN: reliable, fast; P2P: cheap but unreliable, low adoption |

## L5 Additions

1. **Stream health monitoring and auto-recovery:** Broadcaster's stream health often degrades (bad network, CPU spike) before dropping completely. Monitor ingest stream bitrate, frame rate, keyframe interval in real-time. Thresholds: if bitrate drops > 30% → alert broadcaster. If keyframe interval > 5s → viewers can't seek correctly. Auto-recovery: if ingest is unstable, pre-warm a backup ingest server and prompt broadcaster to reconnect. CDN serves last good segment during brief dropout → seamless for most viewers.

2. **Interactive features — polls, predictions, channel points:** Modern streams are interactive. Architecture: a realtime event bus per stream (Redis pub/sub or WebSocket broadcast). Stream overlay API: GET /streams/{id}/overlays → current active overlay type (poll, prediction, celebration animation). Client listens on WebSocket for overlay events. Viewer participation events (vote, predict) → written to fast event store → aggregated by stream engine → result broadcast back to all viewers. All must happen in < 1 second round-trip.

3. **Content moderation at stream scale:** 100K live streams × 1M chat messages/sec = massive moderation challenge. Automated: ML model on each chat message (hate speech, spam, links). At 1M/sec: model must run in < 1ms → use a tiny lightweight model (< 10M parameters) or rule-based for high-confidence cases. ML model for borderline cases. Human moderators: each channel can have moderators who get a "moderation queue" of flagged messages. AutoMod: automated word filter + ML score → hide by default, require mod approval.

4. **Global ingest PoPs and broadcaster quality:** Broadcaster in Tokyo streaming to a US datacenter adds 150ms round-trip latency. Solution: ingest PoPs in every major region. Broadcaster's streaming software connects to nearest PoP (DNS routing). PoP relays stream to primary transcoding region via backbone (high-bandwidth, low-latency private network). Transcoding happens centrally or at regional transcoding nodes. Broadcasters in Southeast Asia, Africa, South America often have poor upstream bandwidth → offer SRT protocol (better than RTMP for unstable connections).

5. **DVR and time-shift at scale:** Every viewer can pause and rewind up to 60 seconds behind live. This seems simple but requires: (a) CDN must keep the last 30 HLS segments (60s worth) per stream for each viewer quality. CDN segment eviction: segment older than 60s from live edge is removed from DVR window. (b) "Time-shifted" viewers get served stale segments — CDN handles this via segment retention rules per stream. (c) Stream end + DVR: when stream ends, the last 60s of DVR window must remain accessible briefly. (d) 100K concurrent streams × 30 segments × 3 qualities × 400KB = 10TB of "hot" DVR segments in CDN — manageable.
