# 19. Amazon — E-Commerce Platform

**Type:** commerce
**Difficulty:** L4–L5
**Frequency:** High — product catalog, inventory, orders, recommendations

---

## Problem Statement

Design Amazon's e-commerce platform. Customers can search for products, view details, add to cart, and place orders. The system must handle 300M active customers, 350M products, 1M orders/day, and ensure inventory is never oversold.

---

## Functional Requirements

**Must have:**
- Product catalog: browse, search, view product details with price and stock
- Shopping cart: add/remove/update items
- Order placement: checkout with shipping address + payment
- Inventory tracking: decrement stock on purchase; never oversell
- Order status tracking

**Should have:**
- Product reviews and ratings
- Personalized recommendations
- Seller/marketplace: third-party sellers list products
- Price history and alerts

**Out of scope:**
- Fulfillment and logistics (warehouse management)
- Returns processing
- Advertising (Sponsored Products)

---

## Non-Functional Requirements

- **Scale:** 300M customers, 350M products, 1M orders/day
- **Availability:** 99.99% — Prime Day is a $12B event; every second down = significant loss
- **Latency:** Product page < 100ms; search < 200ms; checkout < 1s
- **Inventory consistency:** Strong — no overselling
- **Eventual:** Search index, recommendations can lag by minutes

---

## Capacity Estimation

```
Products: 350M × 5KB avg = 1.75TB product data
  Cache top 1M products × 5KB = 5GB → fits in Redis

Orders: 1M/day = 12/sec avg; peak (Prime Day): 100× = 1,200/sec
  Each order has 3 items avg → 3M inventory decrements/day = 35/sec avg; 3,500/sec peak

Search: 300M users × 3 searches/day = 900M/day = 10K/sec
  Peak: 100K/sec (Prime Day)

Product catalog reads: heavy caching needed
  Product page view: 300M users × 20 pages/day = 6B/day = 70K/sec
```

## Deep Dive 1: Inventory Management (No Overselling)

**The critical problem:** Multiple customers simultaneously buy the last item.

**Option A: Pessimistic locking (SELECT FOR UPDATE)**
```sql
BEGIN;
SELECT available_qty FROM inventory WHERE sku = 'SKU-123' FOR UPDATE;
-- Lock acquired, others wait

IF available_qty >= quantity THEN
  UPDATE inventory SET available_qty = available_qty - quantity WHERE sku = 'SKU-123';
  INSERT INTO order_items ...;
  COMMIT;
ELSE
  ROLLBACK; -- out of stock
END IF;
```
- Simple and correct
- Problem: locks rows under high contention → serializes all purchases of hot SKUs

**Option B: Optimistic locking (version counter)**
```sql
-- Read without lock
SELECT available_qty, version FROM inventory WHERE sku = 'SKU-123';
-- qty=5, version=42

-- Update with version check (fails if someone else updated)
UPDATE inventory
SET available_qty = available_qty - 1, version = version + 1
WHERE sku = 'SKU-123' AND version = 42 AND available_qty >= 1;

IF rows_affected == 0: retry (someone else bought first)
ELSE: success
```
- Better for low-contention items (most SKUs)
- High contention (hot SKUs on Prime Day): many retries

**Option C: Reserve + Confirm (two-phase, recommended for high scale)**
```
Phase 1 (at add-to-cart or checkout initiation):
  reserve(sku, quantity, reservation_id, ttl=15min)
    → UPDATE inventory SET reserved_qty = reserved_qty + quantity
         WHERE available_qty - reserved_qty >= quantity
  → Store reservation with TTL

Phase 2 (at payment confirmation):
  confirm(reservation_id)
    → UPDATE inventory SET available_qty = available_qty - quantity,
                           reserved_qty = reserved_qty - quantity
    → Create order record
    → If payment failed: release(reservation_id) → decrement reserved_qty

Background cleanup:
  Expire reservations older than TTL → release reserved_qty
```

**Hot SKU optimization (Prime Day flash sale):**
```
Problem: 10K customers/sec trying to buy 1000 units of a hot item

Solution: inventory sharding + in-memory batching
1. Shard inventory: split 1000 units across 10 Redis counters (100 units each)
   Client hashes to one shard → most decrement operations don't conflict

2. Redis atomic decrement:
   SET inventory:SKU-123:shard:3 100
   DECR inventory:SKU-123:shard:3
   → Returns new value; if < 0: INCR back + return "out of stock for this shard"

3. Reconcile shards to DB async (every 1 second)

4. On shard exhaustion: rebalance from other shards

Accept slight inaccuracy: may oversell by ε (epsilon) then compensate with "rain check"
```

## Deep Dive 2: Product Search

**Why Elasticsearch, not just a DB?**
- Full-text relevance ranking (BM25 over title, description, brand)
- Faceted search (category, brand, price range counts)
- Geo-search for "available in your zip code"
- Synonyms: "headphones" = "earphones" = "earbuds"
- Spell correction and fuzzy matching

**Index structure:**
```
Elasticsearch product document:
{
  product_id, sku, title, description (truncated), brand, category_path, price, avg_rating,
  review_count, in_stock (bool), seller_id, attributes: { color, size, material, ... }
}

Boost signals in ranking:
  + title_match (BM25)
  + avg_rating × log(review_count)  -- more reviews = more confident rating
  + recency (newer products slight boost)
  + in_stock priority
  - penalize out-of-stock by 50% in ranking

Facets (aggregate counts):
  categories, brands, price ranges, avg_rating buckets, Prime-eligible
  → Pre-aggregated in Elasticsearch aggregation pipeline
```

**Search → Product page transition:**
```
Search returns product_ids + basic display data (from Elasticsearch)
Product page: fetch full product data from Product Service (PostgreSQL + Redis)
  Price, stock, variants, photos come from PostgreSQL/Redis (authoritative)
  NOT from Elasticsearch (Elasticsearch is eventually consistent)
```

## Reliability and Operations

- **Cart recovery:** Cart stored in Redis (TTL 30 days) AND synced to DynamoDB (persistent). If Redis is down, cart loaded from DynamoDB. On login (returning user), merge local cart with server cart.
- **Order service down:** Order requests queue in Kafka. Consumer processes in order. Customer sees "your order is being processed" — acceptable.
- **Inventory overcount:** Run daily reconciliation job: compare sum of (order_items.quantity) for fulfilled orders vs (initial_inventory - current_available). Alert on discrepancy > 1%.
- **Flash sale readiness:** Pre-scale inventory service to 10x 2 hours before sale. Pre-warm caches. Simulate peak load in staging 1 week before.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Reserve + Confirm inventory | DB FOR UPDATE | Reserve: non-blocking; FOR UPDATE: serializes writes |
| Redis cart | DB-only cart | Redis: fast; DB: consistent across devices without sync logic |
| Elasticsearch for search | PostgreSQL full-text | ES: better relevance, facets, scale; PG: simpler, fresher data |
| Separate inventory service | Inline in order service | Separate: scalable, isolated; Inline: simpler, fewer network hops |

## L5 Additions

1. **Recommendation engine integration:** "Customers who bought X also bought Y" is a billion-dollar feature. Architecture: batch collaborative filtering (Spark ALS on purchase history, daily) → item-item similarity matrix stored in DynamoDB. Real-time: user's session context (viewed items) → ANN lookup in feature store (Faiss index in memory). Blend: offline similarity + real-time session = final recommendation list. A/B test continuously. Personalization model served from a dedicated recommendation service with a < 50ms SLA.

2. **Pricing service complexity:** Amazon changes prices millions of times per day. Price is not stored on the product — it's a separate dynamic service. Price factors: competitor scraping, demand signals, inventory level, seller rules, promotional schedules. Architecture: event-driven price update pipeline → Kafka → price consumers → update price in both PostgreSQL and Elasticsearch index. Cart locks price at time of "add to cart" (stored with item); if price changes before checkout, warn user. Final price locked at order placement.

3. **Distributed checkout saga:** Checkout touches multiple services: inventory (reserve), payment (charge), fulfillment (create shipment), notification (send email). If payment succeeds but fulfillment service is down: inventory is reserved but order not shipped. SAGA pattern with compensating transactions: inventory release on fulfillment failure. Orchestrator-based SAGA: a dedicated checkout orchestrator service manages the state machine. All steps are idempotent and can be retried. State persisted in a saga_state DB table for recovery.

4. **Catalog at scale — product variations:** A T-shirt with 5 colors × 3 sizes = 15 variants (SKUs). At 350M products with avg 5 variants = 1.75B SKUs. Product tree model: parent product (display, search, reviews) → child variants (inventory, shipping, specific price). Search on parent; buy a specific variant. Challenges: filtering by variant attribute (filter red T-shirts → aggregate across all parent products with a red variant). Elasticsearch nested document model for variants within parent document.

5. **Global supply chain integration:** Amazon stores inventory across 200+ fulfillment centers globally. Inventory service must know: not just how many units exist, but WHERE. Customer's shipping address → fulfillment center routing: nearest FC with stock in preferred shipping time. If nearest FC is out of stock: check next nearest, estimate cross-FC shipping time. Reservation made at the FC level, not globally. Rebalancing: automated inventory transfer between FCs based on demand prediction. This is the real complexity of Amazon's inventory at scale.
