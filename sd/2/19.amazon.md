# 19. Amazon — E-Commerce Platform

**Type:** commerce
**Difficulty:** G4–G5
**Frequency:** High — product catalog, inventory, orders, recommendations

---

## Problem Statement

Design Amazon's e-commerce platform. Customers can search for products, view details, add to cart, and place orders. The system must handle 300M active customers, 350M products, 1M orders/day, and ensure inventory is never oversold.

---

## Functional Requirements

**Must have:**
- Product catalog: browse, search, view product details with price and stock
- Shopping cart: add/remove/update items
- Order placement: checkout with shipping address + payment
- Inventory tracking: decrement stock on purchase; never oversell
- Order status tracking

**Should have:**
- Product reviews and ratings
- Personalized recommendations
- Seller/marketplace: third-party sellers list products
- Price history and alerts

**Out of scope:**
- Fulfillment and logistics (warehouse management)
- Returns processing
- Advertising (Sponsored Products)

---

## Non-Functional Requirements

- **Scale:** 300M customers, 350M products, 1M orders/day
- **Availability:** 99.99% — Prime Day is a $12B event; every second down = significant revenue loss
- **Latency:** Product page < 100ms; search < 200ms; checkout < 1s
- **Inventory consistency:** Strong — no overselling under any load
- **Eventual:** Search index, recommendations can lag by minutes

## Deep Dive 1: Inventory Management (No Overselling)

**The critical challenge:** During a Prime Day flash sale, 100,000 customers per second compete for 500 units of a hot item. The inventory table row for that SKU becomes the hottest row in the entire database.

**Option A: Pessimistic locking (SELECT FOR UPDATE)**
```sql
BEGIN;
SELECT available_qty, reserved_qty
FROM inventory
WHERE sku = 'B08FBVS4DD'
FOR UPDATE;  -- exclusive row lock; all other transactions wait

-- Check availability
IF available_qty - reserved_qty >= requested_quantity THEN
  -- Decrement available
  UPDATE inventory
  SET available_qty = available_qty - requested_quantity,
      version = version + 1
  WHERE sku = 'B08FBVS4DD';

  -- Create order
  INSERT INTO orders ...;
  INSERT INTO order_items ...;
  COMMIT;
ELSE
  ROLLBACK;  -- out of stock
END IF;
```

Result: correct, but serializes all purchases through a single row lock.
At 100K competing transactions: only 1 proceeds at a time, others queue at the lock.
DB connection exhaustion occurs before the queue is processed.

**Option B: Optimistic locking (recommended for most SKUs)**
```sql
-- Step 1: Read without lock
SELECT available_qty, version FROM inventory WHERE sku = 'B08FBVS4DD';
-- Example result: available_qty=10, version=42

-- Step 2: Conditional update (CAS - Compare And Swap)
UPDATE inventory
SET available_qty = available_qty - $qty,
    version = version + 1
WHERE sku = 'B08FBVS4DD'
  AND version = 42              -- must match what we read
  AND available_qty >= $qty;    -- must still have stock

-- Check rows_affected:
-- 1 → success, stock decremented atomically
-- 0 → either version changed (someone else bought) or out of stock → retry or fail
```

Performance: no blocking between transactions; each attempt takes < 5ms.
High contention: many retries, but no queue buildup.
Acceptable when p99 retry count < 3 (low contention items).

**Option C: Reserve + Confirm (recommended for high-scale, two-phase checkout)**
```
Phase 1: Reserve inventory (at checkout initiation, BEFORE payment):
  reserve(sku, qty, reservation_id, ttl=600s):
    UPDATE inventory
    SET reserved_qty = reserved_qty + qty
    WHERE sku = ? AND (available_qty - reserved_qty) >= qty
    RETURNING available_qty - reserved_qty AS remaining;
    -- If 0 rows: out of stock, reject immediately
    -- Store reservation: reservations(id, sku, qty, expires_at)

Phase 2: Confirm inventory (AFTER successful payment):
  confirm(reservation_id):
    -- Convert reservation to actual deduction:
    UPDATE inventory
    SET available_qty = available_qty - qty,
        reserved_qty = reserved_qty - qty
    WHERE sku = ?;
    DELETE FROM reservations WHERE id = reservation_id;
    -- Create order record

Failure handling:
  Payment fails: release(reservation_id) → reserved_qty -= qty
  Server crash: background job cleans up reservations past expires_at
```

**Hot SKU optimization for flash sales (Redis sharding):**
```python
# Problem: 100K customers/sec competing for 500 units
# Solution: distribute inventory across multiple Redis counters (shard by customer hash)

NUM_SHARDS = 10
UNITS_PER_SHARD = 50  # 500 total / 10 shards

def atomic_decrement_hot_sku(sku: str, customer_id: str, qty: int) -> bool:
    # Hash customer to shard (consistent, not random, to prevent double-buy per customer)
    shard = hash(customer_id) % NUM_SHARDS
    redis_key = f"inventory:{sku}:shard:{shard}"

    # Lua script: atomic decrement
    lua_script = """
    local current = tonumber(redis.call('GET', KEYS[1]))
    if current == nil or current < tonumber(ARGV[1]) then
        return -1  -- insufficient inventory in this shard
    end
    return redis.call('DECRBY', KEYS[1], ARGV[1])
    """
    result = redis.eval(lua_script, 1, redis_key, qty)

    if result == -1:
        # Try adjacent shards (round-robin failover)
        for i in range(1, NUM_SHARDS):
            alt_shard = (shard + i) % NUM_SHARDS
            alt_key = f"inventory:{sku}:shard:{alt_shard}"
            result = redis.eval(lua_script, 1, alt_key, qty)
            if result != -1:
                break

    if result == -1:
        return False  # truly out of stock across all shards

    # Async sync to DB (every 1 second)
    publish_to_kafka("inventory-decrments", { sku, qty, shard, timestamp })
    return True

# Background sync job (runs every 1 second):
#   1. Aggregate all decrements from Kafka for each SKU
#   2. UPDATE inventory SET available_qty = available_qty - total_decrements WHERE sku = ?
#   3. If available_qty goes negative: oversell detected → issue "rain check" to last buyer
```

**G5:** Would push on a non-obvious consistency issue with the Redis sharding approach: the total in-flight inventory across all Redis shards may not match the DB `available_qty` at any given moment, because the Kafka-based async sync introduces a lag. If the inventory service crashes between Redis decrements and the DB sync, Redis was decremented but the DB was not — the DB shows more inventory than actually exists. On crash recovery, the Kafka topic replay reconciles this, but during the outage window, the discrepancy exists. Production mitigation: write the Redis decrement AND the Kafka event atomically using Redis Lua script + Kafka transactional API (kafka-transactions). If either fails, both are rolled back. This ensures exactly-once semantics between Redis and Kafka, though "exactly-once" with Kafka transactions adds ~5ms latency per operation — an acceptable cost for a flash sale where correctness is paramount.

## Deep Dive 2: Product Search

**Why Elasticsearch, not just PostgreSQL?**

PostgreSQL full-text search (`tsvector`, `GIN` index) works for small catalogs (< 10M products) but breaks down at Amazon scale:
- Faceted aggregation (count products per brand, per price range) requires expensive GROUP BY over 350M rows
- Geo-search ("available in your zip code") requires PostGIS, not standard Postgres
- Spelling correction and synonym handling are not built-in
- Relevance ranking is a black box (pg's `ts_rank` is rudimentary)

Elasticsearch provides all of the above as first-class features.

**Elasticsearch index design:**
```json
{
  "mappings": {
    "properties": {
      "product_id":      { "type": "keyword" },
      "title":           { "type": "text", "analyzer": "english", "boost": 3 },
      "description":     { "type": "text", "analyzer": "english" },
      "brand":           { "type": "keyword" },
      "category_path":   { "type": "keyword" },
      "price":           { "type": "float" },
      "avg_rating":      { "type": "float" },
      "review_count":    { "type": "integer" },
      "in_stock":        { "type": "boolean" },
      "prime_eligible":  { "type": "boolean" },
      "seller_id":       { "type": "keyword" },
      "attributes":      { "type": "object" },
      "created_at":      { "type": "date" },
      "sales_rank":      { "type": "integer" }
    }
  }
}
```

**Ranking function:**
```
BM25 text relevance score (from Elasticsearch)
  + avg_rating × log(1 + review_count) × 0.2   # trust-weighted quality
  + (1.0 if in_stock else -0.5)                  # strong penalty for out-of-stock
  + (1.0 if prime_eligible else 0)               # Prime boost
  + log(1 + sales_rank_inverse) × 0.3           # bestseller signal
  + recency_score × 0.05                         # slight boost for new products

Faceted aggregations (Elasticsearch aggs):
  { "aggs": {
      "by_brand":       { "terms": { "field": "brand", "size": 20 } },
      "by_price_range": { "range": { "field": "price",
                           "ranges": [{"to":25},{"from":25,"to":50},{"from":50,"to":100},{"from":100}] } },
      "by_rating":      { "range": { "field": "avg_rating",
                           "ranges": [{"from":4},{"from":3,"to":4},{"from":2,"to":3}] } }
    }
  }
  Facet aggregations run in the same query as the result — one Elasticsearch request
```

**Search → product page data flow:**
```
Search result: contains Elasticsearch-stored fields (title, price, rating, in_stock, photo_url)
  → Displayed immediately from ES response (fast path)

Product detail page: ALWAYS fetch from Product Service (PostgreSQL + Redis)
  → Reason: Elasticsearch is eventually consistent (may lag 30s behind DB)
  → Price, stock, variant details must be authoritative
  → Product Service: Redis cache lookup (99% hit) → PostgreSQL fallback

Consistency model:
  Search index: eventual consistency (30s lag acceptable)
  Product page: strong read from authoritative source
```

**G5:** Would challenge whether a single Elasticsearch cluster is the right architecture for 350M products at 100K search QPS on Prime Day. Elasticsearch scales horizontally — each shard handles ~10M documents efficiently — but at 100K QPS, the scatter-gather overhead (each query fans out to all shards) creates p99 latency spikes. The production mitigation is multi-level: (a) Elasticsearch behind an API cache (Redis) with 10-second TTL for identical query strings — identical searches within 10s share a cached result. (b) Bloom filter pre-check: if the query string has never been seen before, go to Elasticsearch; if it has been seen recently and the result is in cache, return cached result. (c) Request coalescing: if 50 simultaneous identical requests arrive, only one is sent to Elasticsearch; all 50 wait for that one result. This reduces Elasticsearch QPS by 60–80% during flash sales when many users search the same terms.

## Reliability and Operations

- **Cart recovery across sessions:** Cart stored in Redis (TTL 30 days) AND asynchronously synced to DynamoDB (persistent, globally replicated). On Redis node failure: cart reloaded from DynamoDB on next request (< 200ms). On user login (returning user with browser cart): server-side merge: `merged = server_cart.merge(client_cart)`, preferring higher quantity per SKU for conflicts. DynamoDB as source of truth for persistence; Redis as hot cache.
- **Order service failure during checkout:** Checkout uses SAGA pattern (see G5 section). Intermediate state persisted in `saga_state` table. On recovery: saga orchestrator resumes from last committed step. Customer is notified of delay but order is not lost.
- **Inventory overcount reconciliation:** Daily batch job: compare `sum(order_items.quantity)` for all orders with status='confirmed' or later, against initial inventory levels minus current available_qty. Discrepancy > 1% → page on-call engineer for investigation. Likely causes: uncleared reservations (reservation TTL bug), Redis shard sync failure, returns not properly re-credited.
- **Flash sale (Prime Day) readiness:** Pre-scale inventory service to 10x normal capacity starting 2 hours before sale. Pre-warm Redis with inventory shard data. Pre-generate product pages as static HTML for the top 100 sale items (serve from CDN, 0 origin load). Simulate peak load in staging environment 1 week before: inject 10x traffic, verify no component falls over, measure p99 latencies under load.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Reserve + Confirm inventory pattern | `SELECT FOR UPDATE` (pessimistic) | Reserve+Confirm: non-blocking, scales to 100K/sec with Redis sharding; FOR UPDATE: simpler, correct, but serializes all writes through one DB lock — fails at flash sale scale |
| Redis cart | DB-only cart | Redis: sub-millisecond reads, handles 700K page views/sec; DB: always consistent across devices without sync logic, but too slow for cart reads at this QPS |
| Elasticsearch for search | PostgreSQL full-text search | ES: rich relevance ranking, facets, synonyms, spell correction; PG full-text: simpler ops, fresher data, but lacks faceted aggs and relevance at 350M products |
| Separate inventory service | Inventory inline in order service | Separate: can scale inventory writes independently of order processing; inline: simpler code, fewer network hops, harder to hot-scale |
| DynamoDB for cart persistence | PostgreSQL | DynamoDB: globally distributed, sub-millisecond writes at any scale; PG: ACID, but harder to replicate globally with < 10ms latency |

## G5

1. **Recommendation engine architecture:** "Customers who bought X also bought Y" generates an estimated 35% of Amazon's revenue. Architecture: (a) Offline batch — Spark ALS (Alternating Least Squares) collaborative filtering on the full purchase history matrix (300M users × 350M products, sparse) runs nightly, producing an item-item similarity matrix. Top 20 similar items per product stored in DynamoDB: `similar_items_{product_id}` → list of (product_id, similarity_score). (b) Real-time session — user's current session (viewed/carted items) processed by a streaming pipeline → ANN (Approximate Nearest Neighbor) lookup in a Faiss index served in memory on recommendation servers. Session signals override offline scores for products viewed in the last 30 minutes. (c) Blend: `final_score = 0.6 × offline_similarity + 0.4 × session_recency_boost`. Served from a dedicated recommendation service with < 50ms SLA; failing that, fall back to editorial curated lists.

2. **Dynamic pricing at Amazon scale:** Amazon's pricing team changes prices millions of times per day in response to competitor prices, inventory levels, and demand signals. Price is not stored on the product record — it's a separate dynamic data entity. Architecture: (a) Price computation pipeline: competitor price feeds (scraped/API) + internal demand signals + inventory level → rule engine or ML model → new price for a SKU. (b) Price event published to Kafka `price-updates` topic. (c) Consumer: update `product_variants.price` in PostgreSQL + update Elasticsearch document + invalidate Redis cache. (d) Cart behavior: price is locked at add-to-cart time (stored in cart item). If price decreases before checkout: show "price dropped" — give the lower price. If price increases: warn user ("price may have changed"), let them confirm. Final price is locked at order placement — stored immutably in `order_items.unit_price` regardless of subsequent price changes.

3. **Checkout SAGA for distributed transactions:** Checkout touches multiple services: Inventory (reserve stock), Payment (charge card), Fulfillment (create shipment), Notification (send confirmation email). If payment succeeds but fulfillment service is down: inventory is reserved but no shipment created — an inconsistent state. SAGA pattern with compensating transactions: each step has a forward action and a compensating action. Orchestrator-based SAGA: a dedicated `CheckoutOrchestrator` service manages the state machine:
   ```
   State: [inventory_reserved → payment_captured → fulfillment_created → notification_sent]
   On inventory failure: nothing to compensate (no previous step)
   On payment failure: compensate → release_inventory_reservation
   On fulfillment failure: compensate → refund_payment, release_inventory
   On notification failure: ignore (best-effort, non-critical)
   ```
   All steps are idempotent: retry-safe with the same `order_id`. Orchestrator state persisted in `saga_state` table. On orchestrator crash: resume from last committed step using Kafka topic replay.

4. **Product catalog at scale — variants and filtering:** A T-shirt with 5 colors × 3 sizes = 15 SKUs. At 350M products with avg 5 variants = 1.75B SKUs. Data model challenge: search on parent (find all T-shirts by brand), buy a specific variant (SKU-level inventory). Elasticsearch: parent document contains variant attributes as a `nested` object. Filter query: "find products with a variant where color=red AND size=M AND in_stock=true" uses Elasticsearch nested query with inner_hits. Challenge: updating a single variant's `in_stock` field requires re-indexing the entire parent document (all 15 variants). At 1M order/day × 3 items × avg 5 variants = 15M ES re-index operations/day → partition ES index updates to off-peak hours or use partial document updates (`_update` with `doc` fragment).

5. **Global fulfillment center routing:** Amazon stores inventory across 200+ fulfillment centers (FCs) globally. When a customer orders a SKU, the system must answer: which FC has stock AND can deliver by the estimated delivery date? Architecture: inventory is tracked at the FC level (`inventory.warehouse_id`). Order routing: (a) Customer shipping address → zip code → lookup nearest FC clusters from `fc_routing_table`. (b) For each candidate FC: check local inventory level, compute shipping time estimate (distance × carrier SLA). (c) Select FC that minimizes delivery time while having sufficient stock. (d) Reserve inventory at the selected FC specifically, not globally. Rebalancing: ML model predicts regional demand by SKU for the next 2 weeks → automated inter-FC transfer requests when inventory imbalance is detected. This is the "distributed inventory allocation" problem and it's the core operational complexity that Amazon's fulfillment engineering solves — it's qualitatively different from a simple global inventory count.
