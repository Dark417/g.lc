# 23. CDN — Content Delivery Network

**Type:** infra
**Difficulty:** L4–G5
**Frequency:** Medium-High — caching layers, routing, edge computing

---

## Problem Statement

Design a Content Delivery Network (CDN) like Cloudflare or Akamai. The CDN should cache static content (images, JS, CSS, video) at geographically distributed edge nodes, serve users from the nearest edge (minimizing latency), and handle 10M requests/sec globally with 95%+ cache hit rate.

---

## Functional Requirements

**Must have:**
- Cache static content at edge nodes (PoPs) worldwide
- Route user to nearest edge node
- On cache miss: fetch from origin, cache at edge, serve
- Support cache invalidation (purge a URL globally)
- Respect origin cache headers (Cache-Control, Expires)

**Should have:**
- DDoS protection (absorb volumetric attacks at edge)
- TLS termination at edge
- HTTP/2 and HTTP/3 support
- Edge computing (run JavaScript/WASM at edge for custom logic)
- Analytics: cache hit rate, origin traffic, error rates per PoP

**Out of scope:**
- Load balancing between origin servers (that's a separate problem)
- WAF (Web Application Firewall) — mention as extension

---

## Non-Functional Requirements

- **Scale:** 10M requests/sec globally; 100Tbps peak bandwidth
- **Latency:** Static content served in < 50ms from edge to user (typically < 10ms for cached)
- **Cache hit rate:** > 95% for static assets
- **Availability:** 99.999% — CDN downtime can take down major websites
- **Purge SLA:** Cache invalidation propagated globally within 5 seconds

---

## Capacity Estimation

```
Requests:
  10M req/sec globally, 250 PoPs = 40K req/sec per PoP (average)
  Peak PoPs (NYC, London, Tokyo): up to 500K req/sec per PoP
  Peak PoP sizing: 500K req/sec × 1KB avg response = 500MB/sec per PoP
  NIC requirement: 500MB/sec = 4Gbps; use multiple 10GbE links (bonded)

Bandwidth:
  10M req/sec × 100KB avg object size = 1TB/sec = 8Tbps globally (cached traffic)
  Origin bandwidth (5% miss rate): 10M × 5% × 100KB = 50GB/sec = 400Gbps
  CDN must absorb origin traffic at regional tier, not forward all misses directly

Cache storage per PoP:
  "Hot" tier (SSD): top-cached content, ~1TB per PoP
    10M objects globally × 5% resident per PoP = 500K objects × 100KB avg = 50GB → easily fits
  "Warm" tier (HDD/NVMe JBOD): 10–20TB per PoP for long-tail content
  Memory (page cache): 64–128GB for in-flight data and metadata index

Metadata overhead:
  1TB SSD / 100KB avg = 10M cached objects per PoP
  Per-object metadata: ~200 bytes (URL hash, disk path, TTL, size, hit count)
  10M × 200B = 2GB metadata → fits in RAM → O(1) cache lookup with in-memory hashmap

Global tile storage:
  100Tbps CDN bandwidth requires distributed peering at IXPs (Internet Exchange Points)
  Cloudflare peers at 200+ IXPs; costs $0 for settlement-free peering vs $0.01/GB transit
```

## High-Level Architecture

```
[User in Tokyo]
    │
    ▼ (DNS resolves to nearest PoP via Anycast)
[Edge PoP — Tokyo]
    │
    ├─ Cache HIT  → serve from local SSD/memory (< 5ms)
    │
    └─ Cache MISS → [Regional Hub — Singapore] (second cache tier)
                         │
                         ├─ Cache HIT  → serve + back-fill edge cache
                         │
                         └─ Cache MISS → [Origin Shield — one PoP per origin region]
                                              │
                                              └─ Cache MISS → [Customer Origin Server]
                                                              fetch + propagate up the tiers

[DNS / Routing Layer]
  Anycast BGP: same IP announced from all PoPs → ISP routes to nearest
  (alternative: GeoDNS for customers who need fine-grained routing control)

[Control Plane — globally distributed]
  - Cache invalidation event stream (Kafka-backed)
  - Configuration updates (version-controlled, pushed via gossip or pub/sub)
  - Analytics aggregation pipeline

[Monitoring — per PoP]
  - Request rate, cache HIT/MISS ratio, origin error rate
  - Synthetic probes every 30s from 50 locations
  - Alert: cache hit rate < 90% for > 5 min → investigate cache key design
```

**Three-tier caching rationale:**
- **Edge PoP (250 sites):** 1–2TB SSD per node. Serves the user directly. Optimized for p99 latency.
- **Regional Hub (10–20 sites):** 50–200TB. Fallback for edge misses. Collapses many edge requests into one origin fetch (request coalescing).
- **Origin Shield (1–3 sites):** Single point of contact with the customer's origin. Completely eliminates the thundering herd on origin during cache invalidation.

## API Design

```
# CDN serves content at customer's domain (via CNAME to CDN)
GET https://images.shopify.com/products/abc.jpg
  → DNS CNAME → cdn.shopify.com → CDN Anycast IP
  → PoP checks cache → HIT: serve directly | MISS: fetch from regional hub or origin

# Customer management API (Cloudflare-style)
POST /v1/zones/{zone_id}/purge-cache
Body:
  Specific URLs:
    { "files": ["https://example.com/logo.png", "https://example.com/style.css"] }
  By tag (surrogate keys):
    { "tags": ["product:123", "category:electronics"] }
  Purge everything:
    { "purge_everything": true }  // use carefully — causes thundering herd
Returns: { "id": "purge-abc-123", "status": "success" }

POST /v1/zones/{zone_id}/rules    # cache rules, bypass rules, redirect rules
GET  /v1/zones/{zone_id}/analytics?since=1h&metrics=requests,bandwidth,hit_rate
Returns: {
  requests: { total: 1234567, cached: 1197543, uncached: 37024 },
  bandwidth: { cached_bytes: 12345678, uncached_bytes: 370240 },
  hit_rate: 0.970,
  by_country: { US: { requests: 500000, hit_rate: 0.975 }, ... }
}
```

## Data Model

**Per-edge cache storage (SSD tier):**
```
Key:   {zone_id}:{url_normalized}:{vary_bucket}
Value: { response_headers, body_on_disk_path, cached_at, expires_at, etag, size_bytes }

In-memory metadata index (fast lookup):
  Hash map: url_hash → { disk_path, ttl, hit_count, last_accessed, size }
  Implemented as: custom LRU hash map in C (not Redis — Redis has too much overhead per key)
  Typical implementations: Nginx proxy_cache, Varnish, Squid, or custom C/C++ daemon

Filesystem layout (hash-based to avoid hot inodes):
  /cache/ssd/
    a5/b3/   ← first 2 hex chars of md5(url_hash) for directory distribution
      a5b3f8c2d1e9.cache   ← cache object file
      ...

Eviction: LRU on SSD tier; items with remaining TTL but low hit count evicted first
  Size-aware: prefer evicting large objects with low hit-count (free more space per eviction)

HDD tier (warm cache, 10–20TB):
  Same structure; items promoted from HDD → SSD on second access within TTL
  Items from SSD → HDD when evicted but TTL not yet expired (demote vs delete)
```

## Deep Dive 1: Routing — How Users Get to the Nearest PoP

**Anycast routing (BGP-based):**
```
All PoPs announce the SAME IP address range via BGP:
  e.g., 104.16.0.0/20 announced by every Cloudflare PoP worldwide

User DNS query:
  → Returns 104.16.0.0 (CDN's anycast IP, same for every user)
  → User's ISP routes packet toward nearest BGP advertiser for 104.16.0.0
  → Packet reaches the nearest PoP (typically within 10–20ms)
  → No configuration change needed; routing is automatic

Failover: if a PoP goes down
  → BGP WITHDRAW for 104.16.0.0 from that PoP
  → Within 30–60 seconds, ISPs re-route traffic to next-nearest PoP
  → No DNS TTL expiry to wait for; routing happens at L3 (network layer)

Advantages:
  ✓ Instant failover (BGP withdrawal, no DNS change)
  ✓ DDoS absorption: attack traffic is spread across all PoPs globally
  ✓ No client-side change needed (CDN can add new PoPs transparently)

Used by: Cloudflare (all traffic), Google (8.8.8.8), Akamai
```

**GeoDNS (DNS-based routing):**
```
CDN operates authoritative DNS servers per region:
  User in Japan queries → authoritative DNS server returns Tokyo PoP IP
  User in UK queries → authoritative DNS returns London PoP IP
  (Determined by the user's resolver IP geolocation)

Advantages: explicit control over traffic distribution; easy to redirect by geo
Disadvantages:
  - DNS TTL means failover takes 60s–5min (until cached records expire)
  - Resolver IP ≠ user IP (VPNs, corporate resolvers in different country)
  - Requires managing many A/AAAA record sets per zone

Used by: AWS CloudFront, smaller CDNs without BGP infrastructure
```

**Request coalescing at edge (preventing thundering herd on cache miss):**
```
Problem: 10,000 concurrent requests arrive for /logo.png which just expired
  → All 10,000 would race to fetch from origin → origin gets 10,000 requests

Solution: request coalescing (also called request collapsing)
  First request for /logo.png: start origin fetch, hold the request
  Requests 2–10,000: detect in-flight fetch in progress → wait
  Origin responds → cache it → release all 10,000 waiting requests simultaneously
  → Origin receives exactly 1 request instead of 10,000

Implementation:
  In-process mutex per cache key:
  if (mutex.tryLock(key)):
      response = fetch_from_origin(url)
      cache.put(key, response)
      mutex.unlock(key)
      return response
  else:
      mutex.waitForUnlock(key)
      return cache.get(key)  # now populated by the winning goroutine
```

**G5:** Anycast routes to the topologically nearest PoP per BGP path, which is not always the lowest-latency PoP for the user. BGP shortest path is measured in AS hops, not milliseconds — a PoP three AS hops away may have 5ms latency, while a PoP one AS hop away (with a congested or indirect peering path) may have 30ms. Production CDNs solve this with latency-based Anycast steering: measure actual RTT from each PoP to the user's ISP prefix using ICMP probes, build a latency map of (prefix → optimal PoP), and use BGP communities to de-prefer specific PoPs for specific prefixes. Cloudflare's "Argo Smart Routing" does exactly this — routes requests over the lowest-latency path rather than the topologically shortest BGP path, delivering 30% latency improvement for users in regions with poor BGP topology.

## Deep Dive 2: Cache Invalidation

**Challenge:** 250 PoPs need to purge a URL within 5 seconds globally.

**Architecture: Fan-out via event stream + PoP pull**
```
1. Customer POSTs purge request to CDN control plane API
2. Control plane validates request, writes purge event to distributed log (Kafka):
   Topic: purge-events
   Key: {zone_id}:{url_hash}
   Value: { zone_id, url, purge_type, timestamp, purge_id }
   Replication: 3; retention: 24 hours (PoPs can catch up after transient disconnect)

3. Every PoP runs a lightweight Kafka consumer:
   Consumer group: "pop-purge-{pop_id}"
   Each PoP has its own consumer group → independent progress tracking
   All 250 PoPs consume the same topic in parallel

4. On purge event received at PoP:
   Hard purge: delete cache object from disk + remove from metadata index
   Soft purge: mark object as stale (TTL=0) → next request revalidates with If-None-Match

5. Timeline:
   Customer API call → Kafka write: < 10ms
   Kafka consumer lag at PoP: ~50–200ms (well-tuned consumers)
   All 250 PoPs invalidate within 1–3 seconds → well within 5-second SLA
```

**Soft vs hard invalidation:**
```
Soft invalidation (stale-while-revalidate):
  Mark cached item as stale, but keep it on disk
  Next incoming request: serve stale immediately (instant response) + async revalidate
    → PoP sends GET /logo.png with If-None-Match: "{etag}" to origin
    → 304 Not Modified: still serve cached; 200 New Content: update cache
  Pros: no thundering herd (origin sees conditional requests, cheap); zero user-visible latency
  Cons: first request after purge gets stale content momentarily (acceptable for most)
  Use for: content updates (CSS deployment, product image update)

Hard invalidation:
  Delete from disk cache immediately
  Next request: full cache miss → origin fetch → potential thundering herd
  Combine with request coalescing to contain thundering herd
  Use for: GDPR right-to-erasure (must not serve deleted content), security incidents, critical errors
```

**Surrogate keys (tag-based invalidation):**
```
Problem: "purge by URL" doesn't scale for content with complex dependencies.
  A product page for item #123 might be cached at 50 different URLs
  (different sort orders, locales, A/B variants)
  Purging all of them requires enumerating 50 URLs — fragile and slow.

Solution: surrogate keys (Cloudflare Cache-Tags, Fastly Surrogate-Key):
  Origin adds header: Surrogate-Key: product:123 category:electronics
  CDN stores tags alongside cached objects (in metadata index)
  On purge: POST /purge { tags: ["product:123"] }
    → CDN finds ALL cached objects tagged with "product:123"
    → Purges all in one operation, regardless of URL

Implementation at PoP:
  Inverted index: tag → {url_hash_1, url_hash_2, ...}
  ZSET in Redis (local to PoP) or custom in-memory trie
  On tag purge: enumerate all url_hashes for tag → mark stale or delete

Purge-all-for-zone:
  Can't enumerate all cached URLs efficiently.
  Solution: zone version counter in metadata index.
  On purge_everything: increment zone's epoch counter (stored in Redis)
  Cache lookup: include zone_epoch in cache key check; old epoch = stale
  Old objects expire via LRU naturally over time
```

**G5:** Would question whether cache invalidation by URL is sufficient — for a CDN serving a SPA, a single deployment changes dozens of hashed asset filenames. A smarter approach is surrogate keys (Cloudflare Cache Tags, Fastly surrogate-keys): tag each cached object with logical identifiers (e.g., `product:123`) and invalidate by tag on any product update, regardless of URL. This reduces both the number of purge API calls and the risk of stale objects from missed invalidations. The second non-obvious concern: tag-based purge at 250 PoPs requires each PoP to maintain an inverted index (tag → set of URL hashes). This index must be updated atomically on every cache write (add tags) and every purge (remove entries). At peak (500K req/sec per PoP with 95% miss rate = 25K writes/sec), the inverted index becomes a write bottleneck unless implemented as a lock-free concurrent data structure or sharded by tag prefix.

## Deep Dive 3: Cache Efficiency Optimization

**Vary header normalization:**
```
HTTP Vary header: "Vary: Accept-Encoding, Accept-Language"
Means: cache a separate copy per unique combination of these headers
Naive: cache key = URL + raw Accept-Encoding + raw Accept-Language
  User sends: Accept-Encoding: gzip, deflate, br, identity
  Different browsers send this differently → thousands of permutations per URL

Solution: Vary normalization
  Accept-Encoding: normalize to "gzip" | "br" | "identity" (pick best supported)
  Accept-Language: normalize to language code only ("en" not "en-US,en;q=0.9")
  User-Agent: normalize to device class ("mobile" | "desktop" | "tablet")

Cache key: URL + normalized_vary_string
  Reduces permutations: 3 encodings × 50 languages × 3 devices = 450 max (vs millions)
  In practice: 2 encodings × 2 languages × 2 devices = 8 variants for most content

Serve-Vary header sent to downstream:
  The response must still include Vary: Accept-Encoding
  Browser uses raw Vary to decide whether to reuse its local cache
```

**Cache warming strategies:**
```
Problem: cold PoP (new startup, maintenance restart) = 100% cache miss → origin hammered

Strategy 1 — Top-N pre-warming:
  After purge event or PoP startup: fetch top 1000 most-accessed URLs for each zone
  Popularity data: from analytics pipeline (top URLs by request count, last 24h)
  Pro-actively GET each URL from origin → populate cache before users arrive
  Target: pre-warm 80% of traffic volume in < 5 minutes

Strategy 2 — Peer-to-peer (P2P) warming:
  New PoP or restarted PoP: pull popular content from neighboring PoPs, not origin
  Neighboring PoPs are on CDN backbone (10–100Gbps, < 5ms latency)
  Origin bandwidth saved; faster than pulling from customer origin

Strategy 3 — Predictive warming:
  Before major events (Super Bowl, product launch, scheduled traffic spike):
  Pre-cache specific content to ALL edge PoPs proactively
  Event known in advance → fetch and distribute before the spike
  Required for live event CDN performance
```

**G5:** Cache hit rate alone is an incomplete metric for CDN efficiency. A more useful measure is "origin offload ratio" — what fraction of bytes that would have gone to origin were served from cache. A CDN can have 95% request hit rate but only 50% byte-offload ratio if the 5% of misses are all large video files. Conversely, serving thousands of tiny JS/CSS files from cache gives high request hit rate with negligible byte savings. Capacity planning and cost modeling should be done on bytes, not requests. For a CDN serving mixed content, instrument separately: (a) small static assets (JS/CSS/images) — optimize for request hit rate; (b) large video/downloads — optimize for byte hit rate using range-request aware caching and segment pre-fetch. The CDN's business model (charging per GB transferred) aligns with byte offload, not request offload.

## Reliability and Operations

- **PoP failure:** Anycast BGP automatically withdraws the PoP's announcement → traffic re-routes to next-nearest PoP within 30–60 seconds. Capacity planning: each PoP is sized to absorb 2× its normal traffic, so adjacent PoPs can take over. For active-active PoPs in the same city (e.g., two NYC PoPs), failover is instant via anycast re-routing within the city.
- **Origin overload:** Circuit breaker per origin zone: if origin error rate > 10% for 30 seconds → serve stale content instead of propagating origin errors to users. `Cache-Control: stale-if-error=86400` (serve stale for up to 24h if origin is down). Better to serve slightly stale content than a 502 error.
- **DDoS absorption:** CDN absorbs volumetric attacks at the edge: 100Tbps global capacity can absorb most attacks. Rate limiting per IP at edge (token bucket, Redis on PoP). For large attacks (1Tbps+) exceeding a single PoP: BGP blackhole routing for the attacking prefix, or send to scrubbing center. Challenge: distinguishing volumetric attack from viral traffic spike (both look like rapid traffic increase).
- **TLS certificate management:** Certificates (wildcard + customer SNI certs) managed centrally; deployed to all PoPs via secure distribution pipeline (not manual). Certificate rotation: deploy new cert with overlap window (old cert still valid); atomically switch active cert. ACME protocol (Let's Encrypt) for automatic renewal. All PoPs support TLS 1.3; TLS 1.0/1.1 disabled globally.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Anycast routing | GeoDNS | Anycast: instant failover, L3 routing, no DNS TTL delay; GeoDNS: more routing control, easier to implement but 60s+ failover time |
| Three-tier caching (edge + regional + shield) | Flat PoP network | Three-tier: dramatically lower origin load, protects origin from thundering herd; Flat: simpler, one fewer cache hop but 20× more origin traffic |
| Soft invalidation (stale-while-revalidate) | Hard delete on purge | Soft: no thundering herd, immediate response; Hard: guaranteed freshness but potential origin overload |
| Surrogate key (tag-based) invalidation | URL-based purge | Surrogate: efficient for complex content graphs, one API call for N URLs; URL: simpler to implement, sufficient for simple sites |
| Request coalescing | No coalescing | Coalescing: collapses N concurrent misses to 1 origin request; No coalescing: simpler but origin sees full concurrent miss storm |

## G5

1. **Edge computing (Cloudflare Workers, Lambda@Edge):** Instead of just caching, run code at the edge. Use cases: A/B testing (50% of requests get variant B without a round-trip to origin), authentication at edge (validate JWT without hitting auth server — public key verification is stateless), request transformation (rewrite URLs, inject tracking scripts, modify response headers based on geo), personalization (serve different HTML based on user segment). Architecture: V8 isolates per request (not containers — 0ms cold start vs 50–500ms for containers). 5ms CPU time limit per request; 128MB memory limit. Access to edge KV store (Cloudflare KV: eventually consistent, 1–2s propagation globally). This is the future of CDN — not just caching but programmable infrastructure at the edge.

2. **HTTP/3 and QUIC at scale:** HTTP/3 runs over QUIC (UDP-based transport). Benefits over HTTP/2: (a) no head-of-line blocking at transport layer — a dropped UDP packet only stalls that one stream, not all multiplexed streams; (b) 0-RTT connection resumption for returning clients (saves one full round-trip on reconnect); (c) connection migration (phone switches WiFi → 4G without dropping connection — QUIC connection ID persists across IP changes). CDN implementation: QUIC requires custom UDP socket handling (not standard TCP). Gradually roll out: negotiate via Alt-Svc header. 15–20% latency improvement for mobile users on lossy networks (cellular, WiFi with packet loss > 1%). Challenge: UDP is frequently rate-limited or blocked by enterprise firewalls — must maintain HTTP/2 fallback indefinitely.

3. **Real-time analytics pipeline:** CDN generates massive structured log streams. Each PoP logs every request: timestamp, URL, status code, cache_status (HIT/MISS/EXPIRED), bytes, country, PoP ID, protocol. Architecture: compressed log batches sent from each PoP to regional aggregators every 1 second → Kafka → Flink for real-time aggregation (5-second windows) → ClickHouse for storage and query. Customer dashboard shows: requests/sec by PoP, cache hit rate over time, error rates by status code, bandwidth by content type, top URLs by traffic, origin response time. Log pipeline volume: 10M req/sec × 200 bytes per log line = 2GB/sec = 172TB/day. Partition ClickHouse by zone_id + date; tiered storage (30 days hot, 1 year cold on S3-backed ClickHouse).

4. **Streaming video optimization:** Live streams and large VOD files require specialized caching. Range-request handling: `GET /video.mp4 Range: bytes=X-Y` — CDN caches only the requested byte range, not the full file. For live HLS/DASH: CDN must handle high-frequency (every 2s) invalidations for the live manifest (.m3u8), while caching individual video segments with longer TTLs (10–30s). Segment-level deduplication: CDN detects when two customers are streaming the same content (e.g., a live sports event on two different media companies' CDN zones) and serves from a single cached copy — "canonical URL" matching by content hash. Slice-and-cache: split large files into 1MB slices internally; different PoPs can independently cache different slices, reducing per-PoP storage requirements for long videos.

5. **Multi-CDN routing for resilience:** Enterprise customers can't depend on a single CDN vendor. Build a CDN orchestration layer: maintain contracts with multiple CDN vendors (Cloudflare, Akamai, Fastly). Synthetic monitoring: probe each CDN from 50 locations every 30 seconds measuring p50/p99 latency and error rate. If a CDN performs worse than threshold for a region → route users from that region to a different CDN via DNS (sub-5s change via low-TTL DNS). Customer interface: single CNAME target; orchestration layer handles CDN selection transparently. Failover example: Cloudflare has an incident in Asia-Pacific → within 60 seconds, orchestration routes all APAC DNS queries to Akamai. This achieves > 99.999% combined availability even if each individual CDN has 99.99% SLA.
