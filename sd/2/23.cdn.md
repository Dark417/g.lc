# 23. CDN — Content Delivery Network

**Type:** infra
**Difficulty:** L4–L5
**Frequency:** Medium-High — caching layers, routing, edge computing

---

## Problem Statement

Design a Content Delivery Network (CDN) like Cloudflare or Akamai. The CDN should cache static content (images, JS, CSS, video) at geographically distributed edge nodes, serve users from the nearest edge (minimizing latency), and handle 10M requests/sec globally with 95%+ cache hit rate.

---

## Functional Requirements

**Must have:**
- Cache static content at edge nodes (PoPs) worldwide
- Route user to nearest edge node
- On cache miss: fetch from origin, cache at edge, serve
- Support cache invalidation (purge a URL globally)
- Respect origin cache headers (Cache-Control, Expires)

**Should have:**
- DDoS protection (absorb volumetric attacks at edge)
- TLS termination at edge
- HTTP/2 and HTTP/3 support
- Edge computing (run JavaScript/WASM at edge for custom logic)
- Analytics: cache hit rate, origin traffic, error rates per PoP

**Out of scope:**
- Load balancing between origin servers (that's a separate problem)
- WAF (Web Application Firewall) — mention as extension

---

## Non-Functional Requirements

- **Scale:** 10M requests/sec globally; 100Tbps peak bandwidth
- **Latency:** Static content served in < 50ms from edge to user (typically < 10ms for cached)
- **Cache hit rate:** > 95% for static assets
- **Availability:** 99.999% — CDN downtime can take down major websites
- **Purge SLA:** Cache invalidation propagated globally within 5 seconds

---

## Capacity Estimation

```
10M req/sec globally, 250 PoPs worldwide = 40K req/sec per PoP average
Peak PoP (NYC, London): 500K req/sec

Cache storage per PoP:
  Top cached content: ~1TB per PoP (SSD)
  Long-tail: HDD storage (10TB per PoP)

Bandwidth:
  10M req/sec × 100KB avg = 1TB/sec total → 100Tbps
  Per PoP: 1TB/s / 250 = 4Gbps average; peak PoPs: 400Gbps (NY, London, Tokyo)
  → Large PoPs need 400Gbps network capacity (100+ 10GbE links)

Origin traffic (5% miss rate):
  10M × 5% = 500K req/sec to origin → CDN must handle without overwhelming origin
```

---

## High-Level Architecture

```
[User in Tokyo]
    │
    ▼ (DNS resolves to nearest PoP)
[Edge PoP - Tokyo]
    │
    ├─ Cache HIT → serve from local SSD (< 5ms)
    │
    └─ Cache MISS → [Regional Hub - Singapore] (second layer)
                         │
                         ├─ Cache HIT → serve from regional SSD + cache at edge
                         │
                         └─ Cache MISS → [Origin Server]
                                              │
                                         fetch + cache at regional + edge

[DNS / Routing Layer]
  Anycast IP or GeoDNS → routes user to nearest PoP

[Control Plane (global)]
  - Cache invalidation propagation
  - Configuration updates
  - Analytics aggregation
```

**Two-tier caching:**
- **Edge PoP (250 sites):** Small cache (1-2TB SSD), low latency. Serves users directly.
- **Regional Hub (10-20 sites):** Larger cache (50TB+). Serves as fallback for edge misses. Reduces origin load dramatically.

## API Design

```
# CDN serves content at customer's domain (via CNAME to CDN):
GET https://images.shopify.com/products/abc.jpg
  → DNS resolves to CDN PoP's IP
  → PoP checks cache → HIT: serve directly | MISS: fetch origin → cache → serve

# Customer management API (Cloudflare-style)
POST /v1/zones/{zone_id}/purge-cache
Body: { files: ["https://example.com/logo.png"] }  → purge specific URLs
   OR { everything: true }  → purge all (use carefully)
Returns: { id, status: "success", message: "Cache purged" }

POST /v1/zones/{zone_id}/rules  → cache rules (cache everything, bypass auth, ...)
GET  /v1/zones/{zone_id}/analytics  → traffic, hit rate, bandwidth stats
```

## Data Model

**Per-edge cache storage:**
```
SSD cache (hot, 1TB):
  Key: {zone_id}:{url_normalized}:{vary_headers}
  Value: { response_headers, body, cached_at, ttl, etag }

  Implementation: custom LRU cache engine (not Redis — too much overhead for CDN scale)
  Typical: Nginx proxy_cache or Varnish or custom C implementation

  Directory structure:
  /cache/
    a5/b3/  ← first 2 hex chars of md5(url) for filesystem distribution
      a5b3f8c2d1e9.cache  ← full cache file

HDD cache (warm, 10TB):
  Same structure but slower tier
  LRU eviction at full capacity
  Items promoted from HDD to SSD on second access (warming)
```

**Cache metadata (in-memory hashmap on each PoP):**
```
For each cached URL:
  { url_hash, disk_path, size, cached_at, ttl, hit_count, last_accessed }
  Kept in RAM for fast lookup (~200 bytes per entry)
  1TB cache / 100KB avg object = 10M objects → 2GB metadata RAM → feasible
```

## Deep Dive 1: Routing — How Users Get to the Nearest PoP

**Anycast routing:**
```
All PoPs announce the SAME IP address via BGP (Border Gateway Protocol)
  e.g., 104.16.0.0 is announced by 250 PoPs worldwide

User DNS query:
  → Returns Cloudflare's anycast IP (104.16.0.0)
  → User's ISP routes to NEAREST BGP neighbor advertising that IP
  → Packet reaches nearest PoP (usually within 10ms)

Advantage: No DNS TTL issues; routing is at network layer; instant failover
  (If a PoP goes down, BGP withdraws the announcement → traffic routes to next nearest)

Used by: Cloudflare, Google's 8.8.8.8, Akamai
```

**GeoDNS (alternative):**
```
DNS resolver returns different IPs based on querying user's location:
  User in Japan → returns Tokyo PoP IP
  User in UK → returns London PoP IP

Advantage: easier to implement; allows sending traffic to specific clusters
Disadvantage: relies on DNS TTL (if TTL=60s, failover takes up to 60s)
Used by: AWS CloudFront, many smaller CDNs
```

## Deep Dive 2: Cache Invalidation

**Challenge:** 250 PoPs need to purge a URL within 5 seconds globally.

**Approach: Fan-out via message bus + pull**
```
1. Customer POSTs purge request to CDN control plane
2. Control plane writes purge event to distributed log (Kafka or proprietary)
   Key: {zone_id}:{url_hash}
3. Each PoP subscribes to the purge event stream (their own consumer group)
4. On purge event: PoP marks cached object as invalidated
   → Next request: serve stale content with revalidation (conditional GET to origin)
   OR: immediately delete from disk cache

Timeline:
  Kafka consumer lag: ~100ms to process event
  250 PoPs all consuming from same Kafka → all invalidate within 2-3 seconds
  → Meets 5-second SLA
```

**Soft invalidation vs hard invalidation:**
```
Soft: Mark cached item as stale → serve to next request with If-None-Match/If-Modified-Since
  → Origin responds 304 (not modified) → still serve cached; or 200 with new content
  → Pros: no thundering herd on origin (conditional requests are cheap)
  → Used for: content updates where slightly stale is acceptable

Hard: Delete from cache immediately → next request → full cache miss → origin fetch
  → Pros: immediately fresh; Cons: thundering herd if popular content
  → Used for: security-sensitive content, GDPR deletion, critical errors
```

**Purge-all performance:**
```
"Purge everything" for a zone (all customer URLs) is expensive:
  Can't enumerate all cached URLs efficiently
  Solution: each cached object has a version tag per zone
  On purge-all: increment zone's version counter
  Cache lookup: include version in key → old versions are effectively stale
  No explicit deletion needed; eviction cleans up over time
```

## Deep Dive 3: Cache Efficiency Optimization

**Vary header problem:**
```
Content may vary by: Accept-Encoding (gzip vs br), Accept-Language, User-Agent (device)
If cached by URL only: wrong content served to different clients

Solution: Vary normalization
  Accept-Encoding: normalize to "gzip" or "br" (ignore specific version)
  User-Agent: normalize to device class ("mobile", "desktop")
  Cache key: URL + normalized_vary_headers
  → Reduces cache permutations from millions to ~4 (desktop-gzip, desktop-br, mobile-gzip, mobile-br)
```

**Cache warming:**
```
On PoP startup or after eviction: cold cache = all requests miss → origin hammered

Warm-up strategy:
1. Top-N pre-warming: after purge event, pro-actively fetch top 1000 URLs for this zone from origin
2. Peer-to-peer warming: new PoP pulls popular content from neighboring PoPs, not origin
   (Saves origin bandwidth; neighboring PoPs are fast via backbone)
3. Predictive warming: before a major event (Super Bowl live stream), pre-cache video segments to all edge PoPs
```

## Reliability and Operations

- **PoP failure:** Anycast BGP automatically routes to next nearest PoP. No configuration change needed. Capacity planning: each PoP can absorb 2x its normal traffic as overflow.
- **Origin overload:** Circuit breaker per origin: if origin error rate > 10%, serve stale cache instead of propagating errors. Better to serve slightly stale content than show 500 errors.
- **DDoS absorption:** CDN absorbs volumetric attacks at the edge (bandwidth capacity) + rate limiting per IP at edge. For large attacks (1Tbps+): scrubbing centers with specialized DDoS mitigation hardware.
- **TLS certificate management:** Wildcard certs and customer custom certs managed centrally; deployed to all PoPs via secure key distribution. Zero-downtime rotation: new cert deployed while old cert remains valid; atomically switch.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Anycast routing | GeoDNS | Anycast: instant failover, layer 3; GeoDNS: DNS TTL delay, more control |
| Two-tier caching | Flat PoP network | Two-tier: lower origin load; Flat: simpler, lower latency on miss |
| Soft invalidation | Hard delete | Soft: no thundering herd; Hard: immediate freshness |
| Origin shield | All PoPs hit origin | Shield: dramatically reduces origin load; adds 1 hop latency |

## L5 Additions

1. **Edge computing (Cloudflare Workers, Lambda@Edge):** Instead of just caching, run code at the edge. Use cases: A/B testing (50% of requests get variant B — no round trip to origin), authentication at edge (validate JWT without hitting auth server), request transformation (rewrite URLs, modify headers, inject scripts), personalization (serve different content based on geo, device). Architecture: V8 isolates per request (not containers — much faster cold start). 5ms CPU time limit per request. Access to KV store at edge (Cloudflare KV: eventually consistent, globally replicated). This is the future of CDN — not just caching but compute.

2. **HTTP/3 and QUIC:** HTTP/3 uses QUIC (UDP-based transport). Benefits over HTTP/2: (a) no head-of-line blocking at transport layer, (b) 0-RTT connection resumption for returning clients, (c) connection migration (phone switches from WiFi to 4G without dropping connection). CDN implementation: QUIC support requires custom UDP socket handling (can't use standard HTTP servers). Gradually roll out: negotiate via Alt-Svc header. 15-20% latency improvement for mobile users on lossy networks.

3. **Real-time analytics pipeline:** CDN generates massive log streams. Each PoP logs every request. Architecture: compressed log batches sent to regional aggregators every 1 second → Kafka → Flink for real-time aggregation → ClickHouse for storage. Customer dashboard shows: requests/sec by PoP, error rates by status code, cache hit rate over time, bandwidth by content type, top URLs by traffic. Anomaly detection: sudden spike in 5xx errors → page on-call. This log pipeline itself is 100TB/day — requires careful partitioning and retention policies.

4. **Streaming video optimization:** Live streams and large VOD files can't be fully cached before serving. Range-request handling: GET with Range: bytes=X-Y → CDN fetches only that byte range from origin. For live HLS: CDN must handle high-frequency (every 2s) cache invalidations for live segment manifests (.m3u8), while caching individual segments (10s TTL). Slice-and-cache: for large files, split into 1MB slices — different PoPs can cache different slices, reducing per-PoP storage requirements.

5. **Multi-CDN routing for resilience:** Enterprise customers can't depend on a single CDN vendor. Build a CDN orchestration layer: maintain multiple CDN vendors (Cloudflare, Akamai, Fastly). Synthetic monitoring: probe each CDN from 50 locations every 30 seconds. If a CDN performs worse than threshold for a region → route users from that region to a different CDN via DNS. Customer sees: single CNAMing target; we handle CDN selection. This reduces dependency on vendor SLAs and achieves > 99.999% availability.
