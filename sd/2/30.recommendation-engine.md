# 30. Recommendation Engine

**Type:** data
**Difficulty:** L5
**Frequency:** High — ML systems, feature stores, AB testing, retrieval + ranking

---

## Problem Statement

Design a recommendation engine for an e-commerce or content platform. Given a user, return a personalized list of N items they are likely to engage with (purchase, watch, click). The system must handle 1B users, 100M items, serve recommendations in < 100ms, and continuously improve through user feedback.

---

## Functional Requirements

**Must have:**
- Return top-N recommended items for a user
- Handle new users with no history (cold start)
- Factor in recent behavior (session context)
- Support multiple recommendation contexts: homepage, product page (similar items), email

**Should have:**
- Near-real-time updates: new user interactions affect recommendations within minutes
- A/B testing support: different model variants served to different user segments
- Business rule injection: boost promoted items, exclude out-of-stock, respect category diversity
- Explanation: "Because you watched X" or "Similar to X"

**Out of scope:**
- Full ML training infrastructure (assume models are pre-trained externally)
- Causal inference for attribution
- Multi-armed bandit optimization

---

## Non-Functional Requirements

- **Scale:** 1B users, 100M items, 1M recommendation requests/sec
- **Latency:** < 100ms p99 for recommendation response
- **Freshness:** Interaction signals influence recommendations within 5 minutes
- **Availability:** 99.99% — fallback to popular items if ML fails
- **Accuracy:** Optimize for click-through rate (CTR) or purchase conversion rate

---

## Capacity Estimation

```
Recommendation requests: 1M/sec
  Each request: retrieve ~1000 candidates + score + rank → return top 20
  Latency budget: 100ms total
    - Candidate retrieval: 20ms
    - Feature lookup: 30ms
    - Scoring/ranking: 40ms
    - Response: 10ms

Feature store size:
  User features: 1B users × 1KB = 1TB → Redis Cluster or DynamoDB
  Item features: 100M items × 2KB = 200GB → fits in Redis
  Embeddings (200-dim float): 100M items × 200 × 4B = 80GB → Faiss in-memory ANN index
```

## High-Level Architecture

```
[Client Request: user_id, context, page_size]
    │
    ▼
[Recommendation API]
    │
    ├──→ [Candidate Generator] ──→ [ANN Index (Faiss/HNSW)]
    │         │                         │ fast embedding search
    │    [Candidate sources]:           │ ~100K candidates → filter to 1000
    │    - User embedding → similar items
    │    - Session context → session-based candidates
    │    - Popular items → fallback for cold start
    │    - Editor's picks / trending
    │
    ├──→ [Feature Store] ──→ user features, item features, context features
    │
    ├──→ [Ranking Model] ──→ score each of 1000 candidates
    │         (XGBoost or 2-tower neural network)
    │
    ├──→ [Business Rules Layer] ──→ boost, demote, filter, diversity
    │
    └──→ [Top-N results with explanation metadata]

[Background pipelines]:
  [User interaction events (Kafka)] → [Feature Pipeline (Flink)] → [Feature Store update]
  [Training data generation] → [Model training (Spark/TF)] → [Model registry] → [serving]
```

## Data Model

**Feature Store:**
```
User features (Redis / DynamoDB, served online):
  user_{id}: {
    embedding: float[200],     // user interest embedding (updated daily)
    recent_categories: [str],  // top 5 categories in last 7 days
    avg_price_range: float,
    preferred_brands: [str],
    location_geo: str,
    tenure_days: int
  }

Item features (Redis, served online):
  item_{id}: {
    embedding: float[200],     // item content embedding
    category: str,
    price: float,
    avg_rating: float,
    purchase_count_7d: int,    // popularity signal
    is_new: bool,              // launched in last 30 days
    inventory_level: int
  }

Session context features (Redis, ephemeral):
  session_{session_id}: {
    viewed_items: [item_id],   // last 20 viewed in session
    searched_queries: [str],
    last_action: timestamp
  }
  TTL: 30 minutes
```

**Embedding index (Faiss, in-memory):**
```
Item embeddings: 100M × 200 dim float32 = 80GB
Index type: HNSW (Hierarchical Navigable Small World)
  - Approximate nearest neighbor search
  - Build time: ~24 hours for 100M items
  - Query time: ~5ms for top-100 nearest neighbors
  - Recall@100: > 95% (vs exact search)

User query: user_embedding → HNSW.search(embedding, k=1000) → 1000 candidate items
```

## Deep Dive 1: Two-Stage Architecture (Retrieval + Ranking)

**Why two stages?**

```
Stage 1 — Candidate Retrieval (speed, recall):
  Goal: from 100M items, efficiently find ~1000 relevant candidates
  Methods:
    A. ANN on user embedding → items with similar embeddings (collaborative filtering)
    B. ANN on session embedding → items similar to what user just viewed
    C. Popular items in user's top categories (coverage for cold start)
    D. Trending items (recency signal)
    E. Recently purchased category (replenishment)

  Each method contributes 200-300 candidates → union → deduplicate → 1000 total
  Speed: ~20ms total (parallel fetches)

Stage 2 — Ranking (quality):
  Goal: from 1000 candidates, predict engagement probability for each
  Use richer features + more expensive model
  Methods:
    A. Feature lookup from feature store (user + item + context features)
    B. Scoring: gradient boosted model (XGBoost, millisecond inference) or
               Two-tower neural network (dot product of pre-computed embeddings)
    C. Output: score for each candidate → sort → top 20

  Speed: ~40ms (1000 items × 40µs per item with XGBoost)
```

**Two-tower neural network:**
```
Architecture:
  User tower: user features → 200-dim embedding
  Item tower: item features → 200-dim embedding
  Score: dot_product(user_embedding, item_embedding)

Training:
  Positive pairs: (user, item they purchased/clicked)
  Negative pairs: random items (negative sampling)
  Loss: InfoNCE / in-batch softmax

Serving:
  Pre-compute item embeddings offline → store in Faiss
  At request time: compute user embedding → ANN search → retrieve candidates
  This IS both retrieval and coarse ranking in one model
  For fine ranking: use full cross-attention or XGBoost on richer features
```

## Deep Dive 2: Real-Time Feature Updates

**Problem:** User buys something → recommendation should immediately reflect it.

**Event-driven feature update pipeline:**
```
User clicks item → Kafka: "item-clicked" event
    │
    ▼
Flink streaming job (1-minute window):
    - Update user's recent_categories (sliding window top-5)
    - Update user's interaction vector (increment category counts)
    - Update item's click_count_7d (decrement old, increment new)
    ↓
Write to feature store (Redis):
    - user_{user_id}: update recent_categories + interaction_history
    - item_{item_id}: update popularity signal

End-to-end: event → feature store update in < 2 minutes
Recommendations at t+5 minutes reflect the interaction
```

**Daily batch update (full feature recomputation):**
```
Spark job:
    - Recompute user embeddings from full interaction history (matrix factorization)
    - Recompute item embeddings from content + interaction data
    - Update all user/item features in feature store
    - Rebuild Faiss ANN index with new embeddings
Time: ~4 hours for 1B users
Deployed at: 3 AM UTC (off-peak)
```

## Deep Dive 3: Cold Start Problem

**New user (no history):**
```
Strategy: use context and demographics
1. If user is signed up: use registration data (age, location, stated interests)
2. Session context: what have they clicked in this session? → session-based collab filtering
3. Fallback: popular items in their geo/device type segment
4. First purchase/click: immediately update user features → next recommendation is personalized

Onboarding flow:
  Ask user to select 3-5 interest categories on signup
  → Seed initial user embedding from category averages
  → Within 5 interactions: full personalization kicks in
```

**New item (no interactions):**
```
Content-based embedding: use item's title, description, category, price to generate embedding
  (sentence transformer on text features → item embedding)
  → Can be retrieved via ANN before any interactions
  → No popularity signal initially → rank below established items
  → Add "new" boost to ensure some exposure → collect early signals
  → After 100 clicks: blend content embedding + collaborative embedding
```

## Reliability and Operations

- **Model serving failure:** Fall back to rule-based recommendations (top-selling items by category). Never return empty recommendations.
- **Feature store unavailability:** Fall back to item popularity ranking (no personalization) with cached last-known user features (staleness acceptable).
- **Faiss index reload:** Takes 30+ minutes for 100M items. Use index versioning: keep two copies (A/B swap). Update new copy offline → atomic swap → remove old. Zero downtime.
- **A/B testing infrastructure:** Recommendation API reads experiment assignment from feature flag service (by user_id). Different experiment groups use different model versions. Measure: CTR, purchase rate, revenue per impression. Statistical significance check before graduating winner.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Two-stage retrieval+ranking | Single-stage | Two-stage: scales to 100M items; Single-stage: simpler, worse recall at scale |
| HNSW approximate ANN | Exact nearest neighbor | HNSW: fast, slight recall loss; Exact: perfect but O(N) per query |
| Pre-computed embeddings | Real-time embedding computation | Pre-computed: fast serving; Real-time: fresher but high latency |
| XGBoost for ranking | Deep neural network | XGBoost: fast, interpretable; NN: better accuracy, slower inference |

## L5 Additions

1. **Cross-domain recommendations:** User browses music → recommend podcasts. User reads tech articles → recommend developer tools. Single model trained across all content types with unified item embedding space. Challenge: items from different domains have incomparable signals. Solution: domain adaptation layers in the two-tower model. Domain-specific features in item tower; domain indicator in embedding. Requires careful negative sampling to prevent domain collapse (music items never co-engaged with podcast items → model learns they're different, not that they're not related).

2. **Exploration vs exploitation (bandit approach):** A pure exploitation model shows users only items it's confident about → filter bubble. Add exploration: 10-15% of recommendations are "exploratory" — items outside user's usual preferences but with high potential. Contextual bandit: for exploratory slot, pick from items with high uncertainty (Thompson sampling) or high potential in similar users. Track feedback → update bandit policy. This both fights filter bubbles and helps discover new successful items (important for longtail content).

3. **Causal recommendations (deconfounding):** User likes action movies and always watches on Friday nights. Model may learn that "Friday night + device" predicts action movie engagement — but this is correlation, not causation. Showing a comedy on Friday night might still work. Counterfactual debiasing: inverse propensity scoring in training (downweight common patterns, upweight rare exposures). Doubly robust estimation: combine propensity model with direct reward model. Causally unbiased model generalizes better to new item exposures.

4. **Real-time session-based recommendations (without user profile):** Guest users or anonymous sessions. No user history → use session context only. Model: transformer-based sequential model (BERT4Rec or SASRec). Input: sequence of item IDs viewed in session → predict next item. Architecture: item sequence → positional encoding → multi-head attention → item embedding prediction → ANN search for nearest items. 50ms inference for 20-item sequences. Trains on historical session data (billions of sessions). Outperforms collaborative filtering for short sessions.

5. **Recommendation diversity and fairness:** If recommendations are purely optimized for CTR, popular items dominate. Long-tail items (90% of catalog) get near-zero exposure. Fairness interventions: (a) categorical diversity: at most K% of recommendations from same category. (b) Provider diversity: no single seller dominates. (c) Long-tail boost: occasionally inject long-tail items to collect signals. (d) Fairness metrics: track exposure distribution across item popularity quintiles; alert if bottom quintile gets < 5% of impressions. This is both an ethical requirement and a business requirement (long-tail suppliers leave the platform without sufficient exposure).
