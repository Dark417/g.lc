# 30. Recommendation Engine

**Type:** data
**Difficulty:** G5
**Frequency:** High — ML systems, feature stores, AB testing, retrieval + ranking

---

## Problem Statement

Design a recommendation engine for an e-commerce or content platform. Given a user, return a personalized list of N items they are likely to engage with (purchase, watch, click). The system must handle 1B users, 100M items, serve recommendations in < 100ms, and continuously improve through user feedback.

---

## Functional Requirements

**Must have:**
- Return top-N recommended items for a user
- Handle new users with no history (cold start)
- Factor in recent behavior (session context)
- Support multiple recommendation contexts: homepage, product page (similar items), email

**Should have:**
- Near-real-time updates: new user interactions affect recommendations within minutes
- A/B testing support: different model variants served to different user segments
- Business rule injection: boost promoted items, exclude out-of-stock, respect category diversity
- Explanation: "Because you watched X" or "Similar to X"

**Out of scope:**
- Full ML training infrastructure (assume models are pre-trained externally)
- Causal inference for attribution
- Multi-armed bandit optimization

---

## Non-Functional Requirements

- **Scale:** 1B users, 100M items, 1M recommendation requests/sec
- **Latency:** < 100ms p99 for recommendation response
- **Freshness:** Interaction signals influence recommendations within 5 minutes
- **Availability:** 99.99% — fallback to popular items if ML fails
- **Accuracy:** Optimize for click-through rate (CTR) or purchase conversion rate

## Capacity Estimation

```
Recommendation requests: 1M/sec
  Typical: 1B DAU × 10 page views/day / 86,400 = ~1.16M requests/sec ≈ 1M/sec

Latency budget per request (total: 100ms):
  Candidate retrieval (ANN search):   20ms
  Feature lookup (user + item):       30ms
  Scoring/ranking (model inference):  40ms
  Business rules + serialization:     10ms
  Total:                             100ms

Feature store sizing:
  User features: 1B users × 1KB per user = 1TB
    → Redis Cluster (sharded, ~100 nodes × 10GB = 1TB) or DynamoDB (managed)
    → Access pattern: 1M reads/sec → Redis at 100K reads/sec per node → 10 nodes minimum; use 30 for headroom
  Item features: 100M items × 2KB = 200GB
    → Single large Redis cluster (10 nodes × 20GB), read-heavy, refreshed daily

Item embedding index:
  100M items × 200 dimensions × 4 bytes (float32) = 80GB
  Faiss HNSW index: ~1.5× overhead for graph structure = ~120GB total
  → One server: 256GB RAM can hold full index with room for OS + JVM
  → Replicate across 3 servers (for availability and read throughput)
  → Rebuild index: 24 hours for 100M items; blue-green swap for zero downtime

Serving cluster:
  1M requests/sec × 100ms latency = 100,000 concurrent requests
  Each server: ~10K concurrent requests (async I/O)
  → 10 recommendation API servers (with autoscaling to 30 for safety)
  Each server: makes parallel calls to ANN index + feature store + ranking model
```

**Embedding index (Faiss HNSW, in-memory):**
```python
# 100M × 200-dim float32 = 80GB
# HNSW (Hierarchical Navigable Small World) graph:
#   Build: O(N × log(N)) ~ 24 hours for 100M items
#   Query: O(log(N)) ~ 5ms for top-100 nearest neighbors
#   Memory: ~1.5× raw embedding size = 120GB
#   Recall@100: > 95% vs exact brute force search

import faiss

# Build index (offline, once per day)
index = faiss.IndexHNSWFlat(200, 32)  # 200 dims, 32 neighbors per layer
index.hnsw.efConstruction = 200       # higher = better recall, slower build
embeddings = load_item_embeddings()   # (100M, 200) float32 numpy array
index.add(embeddings)
faiss.write_index(index, "item_hnsw.index")

# Query (online, < 5ms)
index.hnsw.efSearch = 50  # higher = better recall, slower search
user_embedding = get_user_embedding(user_id)  # (1, 200) float32
scores, item_ids = index.search(user_embedding, k=1000)  # top-1000 candidates
```

## Deep Dive 1: Two-Stage Architecture (Retrieval + Ranking)

**Why two stages?**

```
Stage 1 — Candidate Retrieval (speed, recall):
  Goal: from 100M items, efficiently find ~800 relevant candidates
  Methods (run in parallel):
    A. ANN on user embedding → items with similar embeddings (collaborative filtering)
       → 300 candidates; works best for established users with rich interaction history
    B. ANN on session embedding (mean of last 5 viewed items' embeddings)
       → 200 candidates; responsive to in-session intent shift; works for new users
    C. Popular items in user's top categories (high-recall fallback)
       → 200 candidates; critical for cold start; safety net
    D. Trending items globally + within user's categories
       → 150 candidates; discovery and freshness signal
    E. Recently interacted category top-sellers (replenishment)
       → 150 candidates for consumable categories

  Union + deduplicate: ~800 unique candidates
  Speed: ~20ms total (parallel fetches; ANN search is the bottleneck at ~5ms)

Stage 2 — Ranking (quality):
  Goal: from 800 candidates, predict engagement probability for each
  Use richer features + more expensive model than Stage 1

  Feature assembly:
    - User features: from Redis (single HMGET call)
    - Item features: batch HMGET for all 800 items (~30ms for 800 × 200B items)
    - Cross-features computed in-process:
        dot_product(user.embedding, item.embedding)  # semantic similarity
        item.category in user.recent_categories       # category match boolean
        abs(user.avg_order_value - item.price) / user.avg_order_value  # price fit

  Scoring: XGBoost gradient boosted model
    800 items × 50µs per item = 40ms total
    Output: P(click) or P(purchase) per candidate → sort descending → take top 20
```

**Two-tower neural network (alternative for retrieval + coarse ranking):**
```python
# Architecture: two independent towers, combined with dot product
# User tower: user features → 200-dim user embedding
# Item tower: item features → 200-dim item embedding
# Score: dot_product(user_emb, item_emb) = cosine similarity × magnitude

# Training:
#   Positive pairs: (user, item they purchased/clicked)
#   Negative pairs: random items (negative sampling, 100 negatives per positive)
#   Loss: InfoNCE (in-batch softmax), softmax temperature = 0.07
#   Optimizer: Adam, learning rate 1e-4

# Serving:
#   Pre-compute item_tower(item_features) for all 100M items → store in Faiss
#   Online: compute user_tower(user_features) → ANN search in Faiss
#   This IS both retrieval (candidates) and coarse ranking in one step

# Fine ranking (top-50 from two-tower):
#   Cross-feature model: shared bottom layer + cross-attention between user and item embeddings
#   More expensive but higher accuracy for the final top-20 selection
```

**G5:** The two-tower model's retrieval and ranking outputs are semantically inconsistent: the ANN retrieves items similar to the user embedding, but the XGBoost ranker may re-rank based on features orthogonal to embedding similarity (inventory level, sponsored status, recent popularity). This creates a position bias problem: if high-ranked items in training data received more clicks due to their position (not relevance), the ranking model learns "high-position = good" rather than "relevant item = good." The fix is inverse propensity scoring (IPS) in training: weight each (user, item, click) example by 1/P(position_shown), where P(position) is estimated from a randomization experiment (shuffle positions for 5% of impressions). IPS-weighted training teaches true relevance; without it, the model perpetuates existing position bias, disadvantaging newly added items that only appear in low positions initially.

## Deep Dive 2: Real-Time Feature Updates

**Problem:** User buys Sony headphones at 2:00 PM → recommendations should stop showing more Sony headphones within minutes.

**Event-driven feature update pipeline:**
```
Event: user purchases item-123 (Sony WH-1000XM5, category: Headphones)
    │
    ▼
Kafka: "user-events" topic → record:
  { user_id, event_type: "purchase", item_id, category, timestamp, session_id }
    │
    ▼
Flink streaming job (real-time feature computation):
  KeyedState per user_id

  On each event (within 1-minute tumbling window):
    1. Update user.recent_categories (sliding top-5, last 7 days):
       → Increment score for "Headphones" category; decay older categories proportionally
    2. Update user.last_purchase_category, user.last_purchase_days_ago = 0
    3. Recompute session_embedding = mean(embeddings of last 5 session items)
    4. Update item.purchase_count_7d (increment for purchased item)

  Write to Redis (pipeline, atomic batch):
    HSET user:{user_id} recent_categories ["Headphones", "Electronics", ...]
    HSET user:{user_id} last_purchase_category "Headphones"
    SADD exclude_items:{user_id} item-123  # don't re-recommend just-purchased item
    EXPIRE exclude_items:{user_id} 86400  # 24-hour exclusion window

  End-to-end latency: event → Kafka (< 5s) → Flink (< 30s) → Redis write → API reads update
  Total: < 1 minute from purchase to updated recommendation
```

**Immediate post-purchase deduplication (bypasses Flink pipeline):**
```python
# At purchase API completion (before returning response to client):
def on_purchase_complete(user_id: str, item_id: str):
    # Immediate Redis write — does not wait for Flink pipeline
    redis.sadd(f"exclude_items:{user_id}", item_id)
    redis.expire(f"exclude_items:{user_id}", 86400)
    # Recommendation API checks this set in Business Rules layer
    # → item excluded from next recommendation request immediately
```

**Daily batch update (full feature recomputation):**
```
Spark job (nightly, 3 AM UTC):
  1. Recompute user embeddings via ALS matrix factorization on full interaction history
     Scale: 4 hours on 1000-node Spark cluster for 1B users × 100M items
  2. Recompute item embeddings: blend(content_embedding, collaborative_embedding)
  3. Update all user/item features in Redis via bulk pipeline
  4. Rebuild Faiss HNSW index from new embeddings → blue-green swap (atomic pointer swap)
     Build time: 24 hours; keep old index serving until new one is ready
```

**G5:** Real-time feature updates via Flink have a 30–60 second pipeline latency before the feature store reflects the interaction. During this window, a user who just purchased an item can receive a recommendation showing that same item again. The standard mitigation is the immediate `exclude_items` Redis set (written at purchase API time, bypassing Flink). However, this pattern has a subtle gap: the recommendation API must check this set in the Business Rules layer, not in the Feature Store layer — which means the exclusion list must be an explicit filter applied after ranking, not a feature fed into the model. If the exclusion check is forgotten during a model version upgrade, the filter silently stops working with no test coverage catching it. The correct engineering practice: write a separate integration test that verifies `recently-purchased items do not appear in recommendations` as part of the CI/CD gate for any recommendation service deployment.

## Deep Dive 3: Cold Start Problem

**New user (no interaction history):**
```
Layer 1 (0 interactions): Context and demographics
  Geo-based popularity: user in Japan → trending Japanese content first
  Traffic source: came from a "gaming laptops" Google ad → show gaming laptops
  Time-based: 9 PM Friday → entertainment, not work tools
  Device: mobile → mobile-friendly products (lower price, portable)

Layer 2 (1–5 interactions): Session-based collaborative filtering
  session_embedding = mean(embeddings of clicked/viewed items)
  ANN search on session_embedding → items viewed by similar session patterns
  Works for ALL users; personalizes within minutes of first click

Layer 3 (5–20 interactions): Sparse collaborative filtering bootstrap
  user_embedding ≈ mean(item_embeddings of interacted items)
  Fast approximation; improves as history grows
  Full matrix factorization runs nightly → better embeddings by next day

Onboarding flow (optional explicit mitigation):
  "Select your interests" on signup: 3–5 categories
  Seed user_embedding = mean(category_centroid_embeddings for selected categories)
  Eliminates cold-start; user gets personalized recs from first page view
  Tradeoff: adds signup friction; reduces cold-start quality problem
```

**New item (no interactions):**
```
Content-based bootstrapping:
  Generate item embedding from text: run sentence-transformer on title + description + category
    → 768-dim text embedding → project to 200-dim item embedding space
  Store in Faiss index → immediately retrievable via ANN before any interactions
  Items similar to existing popular items will have nearby embeddings

Exposure seeding:
  No popularity signal initially → ranks below established items
  "New item boost": inject into top-30 results for 5% of users in item's category
  After 100 interactions: blend(0.3 × content_embedding + 0.7 × collaborative_embedding)
  Collaborative embedding becomes dominant as interaction count grows

Cold-start risk: content embedding ≠ collaborative embedding
  A new product may have misleading text (marketing copy) but actual purchase behavior
  differs from what the text implies (e.g., "premium" headphones that are actually budget)
  After enough interactions, collaborative embedding corrects the mismatch
```

## Reliability and Operations

- **Model serving failure:** Fall back to rule-based recommendations (top-selling items by category in user's recent_categories). Never return empty recommendations — always serve something. Alert on model error rate > 0.1%. Monitor: recommendation API latency p99 (spikes indicate model or feature store issues).
- **Feature store unavailability (Redis down):** Fall back to item popularity ranking using a local in-process LRU cache of top-1000 popular items (updated every 5 minutes, pre-loaded on server startup). Zero personalization during outage but zero user-visible errors.
- **Faiss index reload (24-hour build):** Blue-green deployment: maintain two Faiss index copies (A and B) on each serving node. Update index B offline → when ready, swap serving pointer atomically (in-process atomic pointer swap). Both copies require 120GB × 2 = 240GB RAM → spec servers with 256GB+ RAM. After swap, gradually release index A memory.
- **A/B testing:** Experiment assignment: `hash(user_id + experiment_id) % 100 < pct` → deterministic, no session state. Log `model_version` + `experiment_group` on every recommendation request. Measure: CTR, purchase rate, revenue-per-user. Minimum 7-day soak (capture weekly seasonality before declaring significance).

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Two-stage retrieval + ranking | Single-stage (score all items) | Two-stage: scales to 100M items with < 100ms; Single-stage: simpler but 100M × 40µs = 4,000s per request — not feasible |
| HNSW approximate ANN | Exact nearest neighbor (brute force) | HNSW: 5ms, 95%+ recall, 120GB for 100M items; Exact: perfect recall but O(N) per query = 20 seconds at 100M items |
| Pre-computed item embeddings | Real-time item embedding computation | Pre-computed: 0ms retrieval overhead, 200GB RAM; Real-time: fresher for new items but 50ms+ latency per request |
| XGBoost for ranking | Deep cross-attention neural network | XGBoost: 40ms for 800 items, interpretable, no GPU; NN: 5–10% better CTR, 5–10× slower inference, GPU required |
| Redis for feature store | DynamoDB (NoSQL) | Redis: sub-ms latency, in-memory; DynamoDB: managed, unlimited scale, but 5–10ms per read × 800 items = exceeds latency budget for ranking step |

## G5

1. **Cross-domain recommendations:** User browses music → recommend podcasts → recommend developer tools. Single model trained across all content types with a unified item embedding space. Challenge: collaborative signals don't naturally bridge domains (users who buy music rarely co-buy developer tools in the same session). Solution: domain-bridging architecture in the two-tower model — domain indicator as an input to the item tower; domain-specific transformation layers; shared 200-dim output space. Negative sampling must be domain-stratified: include hard negatives within the same domain (similar items the user didn't click) AND cross-domain negatives (items from adjacent domains the user hasn't engaged with). Without careful negative sampling, the model learns to perfectly separate domains (music embeddings cluster far from developer tools) — making cross-domain retrieval impossible. The design goal is controlled proximity: different domains are near-ish in embedding space if user behavioral signals connect them.

2. **Exploration vs exploitation (contextual bandit approach):** A pure exploitation model shows users only items it's highly confident about → filter bubble. User who bought one horror movie gets only horror movies forever. Add structured exploration: 10–15% of recommendations are "exploratory" — items outside user's typical preferences but with high potential based on contextual signals. Contextual bandit with Thompson sampling: for each exploratory slot, sample a CTR estimate from each candidate item's Beta(alpha, beta) distribution (alpha = historical clicks, beta = historical non-clicks). Select item with highest sample. Update Beta parameters with observed click/no-click feedback. This both fights filter bubbles (users discover new interests) and accelerates platform discovery of successful new item-user pairings (critical for long-tail content monetization). Measure category diversity as a separate metric alongside CTR to verify exploration budget is actually expanding user interest space.

3. **Causal recommendations (deconfounding position bias):** User sees item in position 1 → clicks it. Was it clicked because it was relevant (relevance) or because it was in position 1 (position bias)? Training without debiasing: items shown in position 1 have 3× more positive training signal than position 5 due to higher click-through probability — not higher relevance. The model learns "position 1 items are good" rather than "good items should be in position 1." Counterfactual debiasing via inverse propensity scoring (IPS): weight each training example by 1/P(position_shown), where P(position) is estimated from a controlled randomization experiment (shuffle recommendation positions randomly for 5% of impressions). IPS-weighted training learns true item-user relevance, generalizing better to new items that haven't accumulated high-position exposure. Doubly-robust estimator: combine IPS with a direct reward model for variance reduction. This is a correctness issue that silently degrades model quality on every training iteration without debiasing.

4. **Real-time session-based recommendations for anonymous users:** Guest users and fresh sessions have no user history. Session-based model (SASRec or BERT4Rec): input is sequence of item_ids viewed in current session (up to 20 items). Architecture: item_id → 64-dim embedding lookup (100M × 64 = 25GB) → positional encoding → 4-layer multi-head self-attention (6 heads) → 64-dim session embedding → ANN search for top-100 nearest items. Training: next-item prediction from billions of historical sessions using masked item prediction (BERT4Rec) or causal language modeling (SASRec). Inference: 20-item session → 50ms on CPU with ONNX INT8 quantization. Outperforms collaborative filtering for sessions with < 10 items because it captures sequential intent ("user viewed keyboard, then mouse, then monitor stand → building a workstation setup") rather than long-term taste profile. Deploy as a dedicated retrieval source in the candidate generator: weighted 40% for sessions with < 5 items, 10% for sessions with > 20 items.

5. **Recommendation diversity and catalog fairness:** Pure CTR optimization causes rich-get-richer dynamics: popular items accumulate clicks → higher CTR → more recommendations → more clicks. Long-tail items (90% of catalog) receive near-zero impressions → small sellers and independent creators leave the platform. Fairness interventions applied in the Business Rules layer: (a) categorical diversity constraint: at most 3 of top-20 recommendations from the same category (MMR — Maximal Marginal Relevance reranking: greedily select each item to maximize relevance minus redundancy); (b) provider fairness: no single brand or seller occupies > 30% of recommendations; (c) long-tail injection: one long-tail item per 10 recommendations in the user's interest category, selected by content embedding similarity (ensures exposure without irrelevance); (d) fairness monitoring: track impression share by item popularity decile; alert if bottom 50% of items receive < 5% of impressions. This is simultaneously an ethical requirement (fair platform access for small suppliers) and a business requirement (long-tail diversity increases catalog depth → higher customer retention → platform defensibility).
