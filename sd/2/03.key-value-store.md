# 03. Key-Value Store (Distributed Cache)

**Type:** core
**Difficulty:** L4–L5
**Frequency:** Very High — Memcached/Redis internals, DynamoDB-style KV stores

---

## Problem Statement

Design a distributed key-value store that supports GET and PUT operations on arbitrary binary values, scales to petabytes of data across thousands of nodes, handles node failures transparently, and provides tunable consistency. Think DynamoDB, Cassandra's KV model, or a large-scale Memcached cluster.

---

## Functional Requirements

**Must have:**
- `put(key, value)` — store a key-value pair
- `get(key) → value` — retrieve by key
- `delete(key)` — remove a key
- Automatic data distribution across nodes
- No single point of failure

**Should have:**
- TTL per key
- Tunable consistency: eventual vs strong
- Automatic rebalancing when nodes join/leave
- Versioning to detect conflicts (vector clocks or timestamp)

**Out of scope:**
- Range queries or secondary indexes (this is pure KV)
- Transactions across keys

---

## Non-Functional Requirements

- **Scale:** 10PB data, 10K nodes, 1M ops/sec
- **Latency:** GET p99 < 1ms for hot data; PUT p99 < 5ms
- **Availability:** 99.999% — must serve reads even during partial failures
- **Durability:** No data loss (persistence + replication)
- **Consistency:** Tunable — eventual by default, strong available as option

---

## Capacity Estimation

```
Data: 10PB total
Replication factor 3 → 30PB physical storage
Average value size: 1MB → 10B keys total
Keys per node: 10B / 10K nodes = 1M keys/node

Write QPS: 100K/sec × replication_factor(3) = 300K write ops/sec across cluster
Read QPS: 900K/sec (assuming 90% reads)

Network per write: 1MB × 3 replicas = 3MB × 100K writes/sec = 300GB/sec cluster-wide
→ Need high-bandwidth interconnect (25Gbps+ per node)
```

---

## High-Level Architecture

```
Client SDK (consistent hashing → picks coordinator node)
      │
      ▼
[Coordinator Node]  ← receives the request
      │
      ├──→ Replica 1 (primary)
      ├──→ Replica 2
      └──→ Replica 3   ← write to W of N, read from R of N
```

**Components:**
- **Client SDK:** Knows the cluster topology. Uses consistent hashing to find the coordinator. Retries on failure.
- **Coordinator:** Receives client request, routes to correct replica set, handles quorum.
- **Storage nodes:** Each node stores a partition of the keyspace. Has a local persistent store (LSM tree or B-tree).
- **Gossip protocol:** Nodes share health and topology info via gossip (like Cassandra/Dynamo).
- **Coordinator election:** Any node can be coordinator for a given key — no special "master" node.

---

## API Design

```
# External (client SDK)
put(key: bytes, value: bytes, ttl?: int, consistency?: "eventual" | "strong") → { version: VectorClock }
get(key: bytes, consistency?: "eventual" | "strong") → { value: bytes, version: VectorClock } | null
delete(key: bytes) → void

# Internal node-to-node
replicate(key, value, version, source_node) → ack
sync_request(key_range) → key_value_list   # for anti-entropy
gossip(node_state) → cluster_state
```

---

## Data Model

**On each storage node:**
```
Storage engine: LSM-tree (RocksDB)
  - Optimized for write-heavy workloads
  - Compaction handles deleted keys
  - Bloom filter per SSTable for fast negative lookups

Key format:  {partition_key}:{sort_key}   (for range queries within a partition)
Value:       opaque bytes (up to configured max, e.g., 1MB)
Metadata:    { version: VectorClock, ttl: timestamp, tombstone: bool }

Memory:
  - MemTable: recent writes (in-memory, also in WAL)
  - Block cache: hot SSTables cached in RAM (LRU)
  - Bloom filter: check if key exists before disk read
```

## Deep Dive 1: Consistent Hashing and Data Distribution

**Problem:** How to distribute keys across 10K nodes and handle node changes without rehashing everything?

**Naive approach:** `node = hash(key) % N`
- Adding/removing a node remaps almost all keys → massive data migration

**Consistent Hashing:**
- Map both nodes and keys onto a ring (0 to 2^64)
- Each key goes to the first node clockwise from its hash position
- Adding a node: only keys between new node and its predecessor migrate
- Removing a node: only that node's keys migrate to its successor

**Virtual nodes (vnodes):**
- Each physical node occupies multiple positions on the ring (150–200 vnodes)
- More uniform distribution than single position
- Allows heterogeneous nodes: powerful node = more vnodes

```
Physical node A → vnodes at positions [3, 47, 91, 134, ...]
Physical node B → vnodes at positions [12, 56, 103, 167, ...]

Key "foo" hashes to 50 → goes to node at position 56 → Node B
```

**Replication:** Primary partition + 2 replicas at next 2 distinct nodes clockwise.

---

## Deep Dive 2: Consistency Model (CAP tradeoffs)

**Quorum reads/writes (Dynamo-style):**
```
N = total replicas (e.g., 3)
W = replicas that must ack a write before returning success
R = replicas that must respond to a read before returning

Strong consistency:  R + W > N  (e.g., W=2, R=2 with N=3)
High availability:   W=1, R=1 (eventual consistency — any replica serves)
Balanced:            W=2, R=1 (durable writes, fast reads, eventual consistency)
```

**Read repair:**
- On a quorum read, if replicas have different versions → return newest, async write newest to lagging replicas

**Hinted handoff:**
- If a replica node is down during a write, the coordinator stores the write as a "hint"
- When the node comes back, hint is replayed → eventual consistency restored

**Vector clocks for conflict detection:**
```
Each write carries a vector clock: { node_A: 3, node_B: 1 }
If two versions have concurrent clocks → conflict
Conflict resolution: last-write-wins (LWW), merge, or surface to client
Amazon's shopping cart used client-side merge (union of cart items)
```

---

## Deep Dive 3: Storage Engine (LSM Tree vs B-Tree)

**LSM Tree (Log-Structured Merge Tree) — used by Cassandra, RocksDB:**
```
Writes: append to MemTable (in memory) + WAL
When MemTable full: flush to SSTable (immutable sorted file on disk)
Reads: check MemTable → Level 0 SSTables → Level 1 → ... → Level N
Compaction: periodically merge SSTables to remove duplicates and tombstones
```
- Write amplification: 1x (just append)
- Read amplification: high without bloom filters
- Space amplification: moderate (until compaction)
- **Best for write-heavy workloads**

**B-Tree — used by PostgreSQL, MySQL:**
```
In-place updates on a balanced tree
Reads: O(log N) with predictable path
Writes: random I/O for updates
```
- Read amplification: low (predictable path)
- Write amplification: high (random write, page split, journaling)
- **Best for read-heavy workloads**

**Choice for KV store:** LSM tree — we expect write-heavy + high-scale use cases.

---

## Reliability and Operations

- **Node failure detection:** Gossip protocol with phi accrual failure detector (adaptive threshold)
- **Data repair:** Merkle tree per node for efficient anti-entropy — compare tree hashes, sync only differing subtrees
- **Compaction:** Background, rate-throttled to avoid impacting read/write performance
- **Backup:** Periodic snapshots to object storage (S3). Point-in-time recovery via WAL replay.
- **Monitoring:** Per-node: reads/writes QPS, p99 latency, compaction lag, replication lag.

---

## Security

- Client authentication via mTLS (service-to-service)
- Encryption at rest per-shard using envelope encryption
- Keyspace-level access control (client X can only read namespace Y)
- Audit log for all writes to sensitive keyspaces

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| LSM tree | B-tree | LSM wins on write throughput; B-tree better for point reads |
| Consistent hashing | Range-based sharding | Consistent hashing: uniform distribution; Range: enables range scans |
| Eventual consistency | Strong consistency | Eventual: higher availability; Strong: easier to reason about correctness |
| Vector clocks | LWW timestamps | Vector clocks: accurate causality; LWW: simpler but can lose writes |

## L5 Additions

1. **Hot key mitigation:** If a single key gets 100K reads/sec (think: a popular feature flag or OAuth token), it becomes a "hot key" — one node gets crushed. Solutions: (a) Read replicas: 10 replicas for this key, client round-robins. (b) Local read cache in client SDK with short TTL. (c) Key sharding: split "hotkey" into "hotkey#0" through "hotkey#9", scatter across nodes, aggregate on read (for counters only).

2. **Multi-model storage:** Don't force everything into one LSM engine. Tiered storage: hot data (< 1 week) on NVMe SSDs; warm data (1 week–3 months) on HDDs with in-memory index; cold data (> 3 months) on object storage with lazy loading. Transparent to clients — coordinator routes to appropriate tier based on key metadata.

3. **Geo-replication:** Multi-region active-active. Each region has a full copy. Writes go to local region (W=1 or W=2 locally) then async replicate globally. Last-write-wins based on Hybrid Logical Clock (HLC) — handles clock skew better than wall clock. Conflicts from concurrent cross-region writes resolved by HLC timestamp or CRDTs for specific data types (counters, sets).

4. **Compaction optimization:** Tiered compaction (like Cassandra's STCS) accumulates write amplification. Switch to Leveled Compaction (LCS) for read-heavy keyspaces: more CPU for compaction but dramatically better read performance. Automatically switch compaction strategy based on read/write ratio per keyspace.

5. **Zero-downtime rolling upgrades:** Cluster topology changes (version upgrades, node type changes) must be transparent. Protocol: (1) add new nodes, (2) migrate data (streaming rebalance with rate limiting), (3) verify consistency via Merkle tree comparison, (4) decommission old nodes. Never take > 5% capacity offline simultaneously. Track cluster health score — pause migration if health degrades.
