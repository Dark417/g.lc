# 03. Key-Value Store (Distributed Cache)

**Type:** core
**Difficulty:** G4–G5
**Frequency:** Very High — Memcached/Redis internals, DynamoDB-style KV stores

---

## Problem Statement

Design a distributed key-value store that supports GET and PUT operations on arbitrary binary values, scales to petabytes of data across thousands of nodes, handles node failures transparently, and provides tunable consistency. Think DynamoDB, Cassandra's KV model, or a large-scale Memcached cluster.

---

## Functional Requirements

**Must have:**
- `put(key, value)` — store a key-value pair
- `get(key) → value` — retrieve by key
- `delete(key)` — remove a key
- Automatic data distribution across nodes
- No single point of failure

**Should have:**
- TTL per key
- Tunable consistency: eventual vs strong
- Automatic rebalancing when nodes join/leave
- Versioning to detect conflicts (vector clocks or timestamp)

**Out of scope:**
- Range queries or secondary indexes (this is pure KV)
- Transactions across keys

---

## Non-Functional Requirements

- **Scale:** 10PB data, 10K nodes, 1M ops/sec
- **Latency:** GET p99 < 1ms for hot data; PUT p99 < 5ms
- **Availability:** 99.999% — must serve reads even during partial failures
- **Durability:** No data loss (persistence + replication)
- **Consistency:** Tunable — eventual by default, strong available as option

## Capacity Estimation

```
Data volume:
  10PB total user data
  Replication factor 3 → 30PB physical storage
  Average key size: 100 bytes; average value size: 1MB
  Total keys: 10PB / 1MB = 10B keys
  Keys per node: 10B / 10K nodes = 1M keys/node

Throughput:
  Total ops/sec: 1M
  Read:write ratio = 9:1 (typical KV workload)
  → 900K reads/sec, 100K writes/sec
  With replication factor 3: each write fan-outs to 3 nodes
  → 300K write ops internally

Network bandwidth:
  Write bandwidth: 100K writes/sec × 1MB avg value × 3 replicas = 300GB/s cluster-wide
  → 300GB/s ÷ 10K nodes = 30MB/s per node → need 1Gbps+ NICs per node (25Gbps recommended)
  Read bandwidth: 900K reads/sec × 1MB = 900GB/s from cache; most served from RAM

Memory sizing (hot data cache per node):
  Target: serve top 20% of keys from RAM (Zipf distribution → 80% of reads)
  Hot keys per node: 1M × 20% = 200K keys × 1MB avg = 200GB per node
  → High-memory nodes (256GB+ RAM) or accept more cache misses
```

## High-Level Architecture

```
Client SDK
  │  (consistent hashing → selects coordinator node directly)
  │
  ▼
[Coordinator Node]   ← any node can be coordinator for any key
  │
  ├──→ Replica 1 (primary for this key range)
  ├──→ Replica 2
  └──→ Replica 3
       W of 3 must ack write / R of 3 must respond to read

[Gossip Protocol mesh] ← all nodes exchange heartbeats + ring state
[Seed nodes]          ← new nodes bootstrap from seed list
```

**Components:**
- **Client SDK:** Embeds consistent hashing ring state. Routes requests directly to coordinator — no proxy hop. Refreshes ring state on gossip events. Retries on failure.
- **Coordinator Node:** Receives client request, fans out to W or R replicas for this key's range, waits for quorum response, returns to client. Any node can be coordinator for any key.
- **Storage Nodes:** Each stores its assigned partition range. Local persistent store is RocksDB (LSM-tree). In-memory MemTable + block cache for hot data.
- **Gossip Protocol:** Periodic neighbor exchange of node state (alive/suspect/dead), ring membership, and load statistics. Convergence in O(log N) rounds. Based on SWIM protocol.
- **Failure Detector:** Phi accrual failure detector — adaptive threshold based on recent heartbeat inter-arrival times. More accurate than fixed-timeout detectors.

## API Design

```
# External client SDK API
put(
  key:          bytes,
  value:        bytes,
  ttl?:         int,             -- seconds; null = no expiry
  consistency?: "eventual" | "strong"  -- default: eventual
) → { version: VectorClock }

get(
  key:          bytes,
  consistency?: "eventual" | "strong"
) → { value: bytes, version: VectorClock } | null

delete(key: bytes) → void       -- internally a tombstone write

# Internal node-to-node protocol (gRPC)
replicate(key, value, version, ttl, source_node_id) → ack
read_repair(key, version) → { value, version }     -- for lagging replicas
sync_request(key_range: [start, end]) → stream<key, value, version>  -- anti-entropy
gossip(my_state: NodeState) → cluster_state        -- heartbeat + ring update
```

## Data Model

**On each storage node (RocksDB):**
```
Storage layout:
  Column Family "data":
    Key:   {partition_key}:{sort_key}
    Value: {payload: bytes, ttl: uint32, tombstone: bool}

  Column Family "metadata":
    Key:   {partition_key}:{sort_key}
    Value: {version: VectorClock, created_at: uint64, last_updated: uint64}

RocksDB configuration:
  MemTable size:      128MB (write buffer; flushed to L0 SSTable when full)
  Block cache:        50% of RAM (LRU; serves hot reads without disk I/O)
  Bloom filter:       10 bits/key per SSTable (reduces false disk reads by ~99%)
  Compression:        LZ4 for L0-L2 (fast); Zstd for L3+ (better ratio)
  Compaction:         Leveled (predictable read performance) for read-heavy keyspaces
                      FIFO for time-series append-only keyspaces

WAL:
  Every write appended to WAL before MemTable → durability without fsync per write
  Group commit: batch WAL writes every 2ms → 500x fewer fsyncs
```

**Vector clock format:**
```
VectorClock: { node_id: uint16 → sequence: uint64 }
Stored as: sorted list of (node_id, seq) pairs, varint-encoded
Size: typically 3–5 entries × 10 bytes = 30–50 bytes overhead per key

Comparison:
  VC_A dominates VC_B: all entries in A ≥ corresponding entries in B
  Concurrent: neither dominates → conflict
  Resolution: last-write-wins (LWW) by wall clock, or surface to client
```

## Deep Dive 1: Consistent Hashing and Data Distribution

**Problem:** Distribute 10B keys across 10K nodes. When nodes change, minimize key remapping.

**Naive modular hashing fails:**
```
node = hash(key) % N

Adding node 10,001: almost every key's (hash % N) changes → 99.99% of keys migrate
→ 10PB of data movement → takes days, saturates network
```

**Consistent hashing with virtual nodes:**
```
Ring: hash space [0, 2^64)
  Each physical node → V virtual node positions on ring
  V = 150–256 vnodes per node (Cassandra default: 256)

Key routing:
  key_hash = murmur3_64(key)   -- fast, uniform hash
  Find first vnode position ≥ key_hash (clockwise)
  That vnode's physical node is the coordinator

Replication:
  Next 2 distinct physical nodes clockwise = replicas 2 and 3
  "Distinct physical" ensures replicas aren't on the same machine
```

**Why virtual nodes improve uniformity:**
```
With 3 physical nodes, 1 vnode each:
  Variance in key fraction per node: high (depends on hash luck)
  Example: Node A: 43%, Node B: 29%, Node C: 28%

With 3 physical nodes, 150 vnodes each:
  Each node has 150/450 = 33.3% of ring positions
  Law of large numbers → actual key distribution is very close to 33.3%
  Empirically: < 3% deviation from mean with 150+ vnodes

With heterogeneous capacity:
  Node A (128GB RAM): 200 vnodes → ~44% of keys
  Node B (64GB RAM):  100 vnodes → ~22% of keys
  Node C (32GB RAM):   50 vnodes → ~11% of keys
  Remaining nodes fill proportionally
```

**Rebalancing on node change:**
```python
# Adding a node: only keys in [predecessor, new_node) migrate
# Expected migration: K/N keys (1/N fraction)
# Example: adding 1 node to 10K node cluster → 0.01% of keys migrate

# At 10PB × 0.01% = 1TB migration → at 1Gbps/node: ~8 seconds of migration
# Rate-limit migration to 100MB/s per node to avoid impacting live traffic
# → 10TB/100MBs = 100 seconds background migration with minimal impact
```

**G5:** Virtual nodes solve load distribution but create a subtle operational hazard: when a node fails, its vnodes are scattered across the ring, so its keys are absorbed by many different successor nodes simultaneously. This is usually fine, but during a simultaneous multi-node failure (e.g., a rack loses power), overlapping vnode ranges from multiple failed nodes may all fail over to the same successor. With 150 vnodes per node and 3 nodes failing, 450 vnode ranges suddenly need coverage — some successors absorb 2–3× their normal load. Monitor per-node vnode count and set an automated circuit breaker: if a node would absorb more than 1.5× its normal key range after a failure event, pause the failure-driven rebalancing and alert, rather than cascading the overload.

## Deep Dive 2: Consistency Model (CAP Tradeoffs)

**Quorum reads and writes (Dynamo-style):**
```
N = replication factor (3 in our design)
W = replicas that must acknowledge a write
R = replicas that must respond to a read

Consistency guarantees:
  Strong (linearizable): R + W > N → W=2, R=2 (any 2 of 3)
    - Guaranteed to see the latest write
    - Higher latency: must wait for 2 replicas; slower replica sets tail latency
    - Lower availability: if 2+ replicas fail, writes block

  High availability:  W=1, R=1
    - Write returns after 1 ack; read from any 1 replica
    - May read stale data (eventual consistency)
    - Maximizes availability; handles 2 of 3 replicas failing

  Balanced (typical choice): W=2, R=1
    - Write is durable (2 replicas) before returning
    - Read may be slightly stale (1 replica, not guaranteed to have latest)
    - Good durability + high read availability
```

**Read repair:**
```
On a quorum read with R=2:
  Fetch from replica 1: version = {A:5, B:2}
  Fetch from replica 2: version = {A:4, B:2}   ← lagging replica

  Coordinator compares versions:
    Replica 1 has higher version → send async write to replica 2
    Return replica 1's value to client

  Result: replica 2 catches up without explicit anti-entropy
  Benefit: frequently-read keys stay consistent automatically
  Cost: extra write per read (acceptable for hot data)
```

**Hinted handoff (write durability during node failures):**
```
Scenario: replica 3 is temporarily unavailable (network partition, restart)
  Write arrives, needs W=2 acks
  Replica 1 acks. Replica 2 acks. Replica 3 unreachable.
  → W=2 achieved → write returns success to client

  Coordinator stores a "hint" for replica 3:
    Key: hint:{replica3_id}:{key} = value
    TTL: 1 hour (configurable)

  When replica 3 recovers:
    Coordinator notices (via gossip) that replica 3 is back
    Replays stored hints to replica 3
    Replica 3 is now fully consistent

  Without hinted handoff: must wait for Merkle tree anti-entropy sync (slower)
```

**Vector clocks for conflict detection:**
```
Scenario: two clients write to same key while network is partitioned

  Client A writes to replica 1: value="foo", VC={node1: 1}
  Client B writes to replica 2: value="bar", VC={node2: 1}
  Partition heals → replicas exchange values

  Comparing VCs: {node1:1} and {node2:1} are concurrent (neither dominates)
  → Conflict detected

Resolution strategies:
  1. Last-write-wins (LWW): use wall clock timestamp; node2's write at T=1003 wins
     Pros: simple. Cons: clock skew can cause wrong winner; silently drops data
  2. Client-side merge: surface both versions to client; client decides
     Used by Amazon DynamoDB (and shopping cart in the original Dynamo paper)
     Pros: no data loss. Cons: application must handle conflicts
  3. CRDT: data type that converges automatically (set union, counter max)
     Pros: automatic. Cons: only works for specific data types
```

## Deep Dive 3: Storage Engine (LSM Tree vs B-Tree)

**LSM Tree (Log-Structured Merge Tree):**
```
Write path:
  1. Write to WAL (disk, sequential → fast fsync)
  2. Write to MemTable (in-memory sorted map)
  3. When MemTable full (e.g., 128MB): flush to L0 SSTable (immutable, sorted file on disk)
  4. Background compaction: merge L0 SSTables → L1 → L2 → ... (sorted, deduplicated)

Read path:
  1. Check MemTable (in-memory, O(log N))
  2. Check L0 SSTables in reverse order (newest first)
  3. Check L1, L2, ... (bloom filter reduces unnecessary disk reads)
  Each level is 10× larger than the previous; most data is in deep levels

Compaction removes:
  - Older versions of the same key (keep only newest)
  - Tombstoned (deleted) keys
  - Expired TTL keys
```

**Write amplification analysis:**
```
LSM write amplification:
  Original write: 1× to MemTable
  Flush to L0: 1× disk write
  Compaction L0→L1: reads and rewrites data ~10× (level size ratio)
  Total write amplification: ~10–30× depending on levels

B-tree write amplification:
  In-place update: reads page, modifies, writes back: 2× per update
  Page splits: can cascade → higher in worst case
  Journal/WAL: 2× additional
  Total: ~6–10× for random writes, but with random I/O (vs sequential for LSM)
```

**Performance comparison:**
```
                     LSM Tree          B-Tree
Write throughput:    Very high         Moderate
Write I/O pattern:   Sequential        Random
Read throughput:     High (with cache) Very high (predictable path)
Read I/O (cold):     Multiple files    Single path O(log N)
Space efficiency:    Moderate (until compaction) Good
Delete performance:  Fast (tombstone)  Moderate (in-place)
Best for:            Write-heavy       Read-heavy
Used by:             RocksDB, Cassandra, LevelDB  PostgreSQL, MySQL InnoDB, SQLite
```

**Bloom filter effectiveness:**
```
Per SSTable: one Bloom filter at 10 bits/key
False positive rate: ~1% (probability of unnecessary disk read)
Memory cost: 10M keys × 10 bits = 12.5MB per SSTable's filter

Without bloom filter: every GET that misses MemTable → disk read on every SSTable
With bloom filter: only 1% of misses → unnecessary disk read → ~100× fewer disk reads
```

**G5:** RocksDB's default leveled compaction runs background compaction threads that compete with read/write I/O. In write-heavy bursts, compaction falls behind and L0 accumulates too many SSTables — read performance degrades because more files must be checked. The standard mitigation is to tune `max_write_buffer_number` and `level0_slowdown_writes_trigger`, but these are static. A better approach is rate-adaptive compaction: monitor the L0 file count in real-time; when it exceeds a threshold, temporarily reduce write admission (slow down incoming puts) rather than letting L0 grow unbounded. This is the same feedback loop principle as TCP congestion control — prefer controlled slowdown over uncontrolled degradation. Expose `compaction_lag_bytes` as a SLO metric; page if it exceeds 10GB sustained for more than 5 minutes.

## Reliability and Operations

- **Node failure detection:** Gossip protocol with phi accrual failure detector (adaptive heartbeat threshold). Marks node "suspect" after missed heartbeats, "failed" after confirmation from 2+ other nodes. Avoids false positives during GC pauses or slow network.
- **Anti-entropy with Merkle trees:** Each node maintains a Merkle tree over its key ranges. Periodic anti-entropy: exchange tree roots with replicas. If roots differ, binary search down the tree to find differing subtrees. Sync only the divergent key ranges. Far more efficient than full key exchange — at 1M keys/node, finds a 1000-key divergence with ~20 comparisons.
- **Compaction rate-limiting:** Background compaction is throttled to a max I/O rate (e.g., 50MB/s per node) to avoid saturating disk bandwidth needed for reads and writes. Compaction backpressure: if L0 SSTable count exceeds threshold, slow down incoming writes (back-pressure signal to clients).
- **Backup and restore:** Periodic RocksDB checkpoints (hard-links to current SSTable files) to S3. Checkpoint creation is O(1) — just links, no copy. S3 upload runs in background at rate-limited speed. Point-in-time recovery: restore from checkpoint + replay WAL entries since checkpoint.
- **Monitoring:** Per-node metrics: GET/PUT p50/p99/p999, compaction lag, bloom filter miss rate, replica lag, disk I/O utilization, memory pressure (block cache hit rate).

## Security

- mTLS between all nodes and between clients and nodes (service mesh with cert rotation)
- Encryption at rest: per-shard AES-256 encryption with keys managed by KMS; envelope encryption pattern (data key per shard, encrypted by KMS master key)
- Keyspace-level ACLs: client service X gets read/write to namespace "user-sessions", read-only to "feature-flags", no access to "payment-data"
- Audit log: all writes to sensitive keyspaces logged with client identity, timestamp, key (not value) → append-only audit trail in separate storage

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| LSM tree (RocksDB) | B-tree (PostgreSQL page store) | LSM excels at high write throughput with sequential I/O; B-tree has better cold read latency but degrades under write-heavy load due to random I/O and lock contention |
| Consistent hashing + vnodes | Range-based sharding (like HBase) | Consistent hashing: automatic rebalancing with minimal key movement; Range sharding: enables range queries but requires manual split management and creates hot-spot risk on sequential keys |
| Eventual consistency (default) | Strong consistency | Eventual: any replica responds immediately, maximizes availability even during partitions; Strong: needs quorum (R+W>N), adds latency and reduces availability |
| Vector clocks | Last-write-wins (LWW) timestamps | Vector clocks: detect concurrent writes and surface conflicts without data loss; LWW: simpler but silently drops writes if clock skew or network reordering occurs — unacceptable for inventory or financial data |
| Gossip protocol | Centralized configuration service | Gossip: O(log N) convergence, no SPOF, scales to 10K nodes; Centralized (ZooKeeper): consistent but can become bottleneck, harder to scale past ~100 nodes |

## G5

**What a G5 engineer adds beyond the above:**

1. **Hot key mitigation with read replicas and client-side cache:** A single key receiving 100K reads/sec (OAuth token cache, popular feature flag, viral post metadata) maps to one primary node — that node saturates while others are idle. Solution layers: (a) Replicate hot keys to additional "overflow" replicas beyond the base replication factor — detected automatically when a key's read rate exceeds `total_cluster_reads / num_nodes × 10`. (b) Client SDK maintains a local in-process LRU cache (e.g., 10MB, 5s TTL) for reads — a 5s stale read is acceptable for many use cases and eliminates 99% of traffic for ultra-hot keys. (c) For counters: shard "hotkey" into "hotkey#0" through "hotkey#N-1", scatter-write, gather-read (sum). This turns a single hot key into N cool keys at the cost of read aggregation.

2. **Multi-model tiered storage:** Forcing all data into one RocksDB instance ignores access patterns. Hot data (written in last 7 days, accessed frequently): NVMe SSDs, block cache in RAM, target < 1ms GET. Warm data (1 week – 3 months): spinning HDD, no in-memory cache, target < 50ms GET. Cold data (3+ months): S3 Glacier Instant Retrieval, 128MB chunk reads, target < 5 seconds. The coordinator's routing layer checks key metadata (last_accessed_at, written_at) and routes to the appropriate tier transparently. Cold reads trigger a background promotion to warm tier if access frequency picks up. Storage cost reduction: 10PB of data → ~2PB hot (NVMe at $0.20/GB) + 5PB warm (HDD at $0.04/GB) + 3PB cold (Glacier at $0.004/GB) = $600M vs $2B if all on NVMe.

3. **Geo-replication with Hybrid Logical Clocks:** Multi-region active-active: each region has a full replica, writes go to local region (W=1 or W=2 locally) and async replicate globally. The conflict resolution challenge: wall clock timestamps across regions can disagree by 10–500ms (NTP drift, leap seconds). Last-write-wins based on wall clock silently drops the correct write if clocks disagree. Use Hybrid Logical Clock (HLC) instead: `HLC = max(wall_clock, last_known_HLC) + logical_increment`. HLC is always ≥ wall clock and captures causality across nodes. Concurrent cross-region writes detected when HLC timestamps are equal → use CRDTs for commutative data types (counters, sets), surface conflict to client for non-commutative types.

4. **Compaction strategy selection per keyspace:** Using the same compaction strategy for all data is a false economy. Time-series keyspaces (logs, metrics, events) have predictable write patterns: old data is never updated, only new data is appended. Use FIFO compaction for these — just drop oldest SSTables when space is needed, no merge work needed. Feature-flag keyspaces (read-heavy, rarely updated) benefit from Leveled Compaction (LCS): higher compaction CPU cost, but guaranteed single-file reads per key at each level. Auto-select compaction strategy based on observed keyspace characteristics: if write rate > read rate × 10, use Tiered; if read rate > write rate × 5, use Leveled; if TTL covers 90%+ of keys, use FIFO.

5. **Zero-downtime rolling upgrades:** In a 10K-node cluster, upgrading all nodes simultaneously is impossible. Protocol: (1) Stage 1: deploy new version to 5% of nodes, run in parallel with old version, monitor error rates. (2) Stage 2: migrate 50% if error rates are clean. (3) Stage 3: complete migration. During version coexistence, the gossip protocol must handle mixed node versions — use protocol versioning in gossip messages. Data format changes (e.g., new VectorClock serialization) require a two-phase migration: Phase A adds new format alongside old (both written), Phase B removes old format. Never remove old format support in the same release that adds new format — always leave at least one version gap.
