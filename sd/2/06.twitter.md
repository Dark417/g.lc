# 06. Twitter — News Feed and Timeline

**Type:** social-media
**Difficulty:** G4–G5
**Frequency:** Very High — canonical social media design question

---

## Problem Statement

Design Twitter. Users can post tweets, follow other users, and see a personalized timeline of tweets from people they follow, reverse-chronological. The system must handle 300M active users, 500M tweets/day, and serve timelines with sub-100ms latency at peak.

---

## Functional Requirements

**Must have:**
- Post a tweet (text, optional media)
- Follow / unfollow a user
- View home timeline: tweets from people you follow, reverse-chronological
- User profile page: that user's tweets, follower/following counts

**Should have:**
- Like, retweet, reply
- Trending topics (mention as extension)
- Tweet search (mention as extension)

**Out of scope:**
- DMs (separate design — see WhatsApp)
- Ads injection
- Recommendation of who to follow

---

## Non-Functional Requirements

- **Scale:** 300M DAU, 500M tweets/day, 500K tweet/sec peak reads
- **Availability:** 99.99% for timeline reads
- **Latency:** Timeline load < 100ms p99
- **Consistency:** Eventual — slight delay in seeing new tweets is acceptable
- **Durability:** No tweet loss

---

## Capacity Estimation

```
Tweet writes:
  500M tweets/day ÷ 86,400 sec = 5,787/sec average
  Peak (3.5× average) = ~20,000/sec

Timeline reads:
  300M DAU × 10 reads/day = 3B reads/day
  3B ÷ 86,400 = 34,722/sec average; peak ~150,000/sec

Read:write ratio = 150K : 20K ≈ 8:1 (read-heavy but not as extreme as URL shortener)

Tweet storage:
  Average tweet: 140 chars × 2 bytes (UTF-16) + 100 bytes metadata = ~380 bytes
  500M/day × 380 bytes = 190GB/day
  10-year archive: 190GB × 3,650 = ~695TB (2 petabytes with 3× replication)
  Media (images/videos): separate in S3; ~5× raw tweet volume = ~3.5PB/day

Fan-out writes (the critical capacity driver):
  Average user followers: 200
  5,787 tweets/sec × 200 followers = 1,157,400 writes/sec to timeline caches
  Celebrity account (100M followers) × 1 tweet = 100M cache writes in seconds → problem

Timeline cache sizing:
  300M active users × 800 tweet IDs × 8 bytes/ID = 1.92TB
  → Redis Cluster with ~20 shards handles this comfortably (100GB/shard)
```

## High-Level Architecture

```
Client
  │
  ├─ POST /tweet ──→ [Write API] ──→ [Tweet DB (Cassandra)]
  │                        │
  │                 [Kafka: tweet-events]
  │                        │
  │                 [Fan-out Service]
  │                        │
  │               ┌────────┴────────┐
  │         [Regular users]    [Skip celebrities]
  │          ZADD to Redis          │
  │         timeline cache     (pulled at read time)
  │
  └─ GET /timeline ──→ [Timeline API] ──→ [Redis: timeline:{user_id}]
                              │                    │
                       [User is celebrity  (cache miss: rebuild
                        followed? → fetch           from DB)
                        their recent tweets
                        separately + merge]
                              │
                        [Tweet Service] → [Cassandra: tweets by user_id]
                              │
                        [User Service]  → [PostgreSQL: user profiles]
```

**Components:**
- **Write API:** Validates tweet, saves to Tweet DB (Cassandra), publishes event to Kafka
- **Fan-out Service:** Consumes tweet events from Kafka, pushes tweet_id to all followers' Redis timeline caches (for non-celebrities)
- **Timeline API:** Reads pre-computed cache from Redis, merges in celebrity tweets at read time, hydrates tweet content
- **Tweet DB (Cassandra):** All tweets, partitioned by `user_id` with `created_at` as clustering key (efficient for "get recent tweets by user")
- **Social Graph Service:** Follows relationships stored in Cassandra (adjacency list: follower_id → list of followee_ids)
- **Timeline Cache (Redis):** Per-user sorted set of tweet IDs, scored by creation timestamp. 7-day TTL.
- **User Service:** Profile data, follower counts in PostgreSQL (strong consistency for counts)

## Data Model

```sql
-- Cassandra: tweets partitioned by author for efficient profile page reads
-- and fan-out service lookups ("get user X's last 20 tweets")
tweets (
  user_id     UUID,           -- PARTITION KEY: all of user's tweets in one partition
  created_at  TIMESTAMP,      -- CLUSTERING KEY DESC: newest first
  tweet_id    UUID,           -- globally unique (Snowflake ID encoded as UUID)
  text        TEXT,
  media_ids   LIST<UUID>,
  reply_to    UUID,           -- null if not a reply
  PRIMARY KEY (user_id, created_at, tweet_id)
)

-- Social graph: "who does follower_id follow?"
follows (
  follower_id  UUID,          -- PARTITION KEY
  followee_id  UUID,          -- CLUSTERING KEY
  created_at   TIMESTAMP,
  PRIMARY KEY (follower_id, followee_id)
)
-- Also need reverse: "who follows followee_id?" (for fan-out)
followers_of (
  followee_id  UUID,          -- PARTITION KEY
  follower_id  UUID,          -- CLUSTERING KEY
  PRIMARY KEY (followee_id, follower_id)
)

-- Redis: pre-built timeline cache per user
-- Key:  timeline:{user_id}
-- Type: Sorted Set
-- Score: tweet creation timestamp (Unix ms) → enables time-ordered iteration
-- Member: tweet_id
-- TTL: 7 days (inactive users' caches expire; rebuilt on re-login)
-- Max size: 800 tweet IDs (enough for ~40 pages of 20 tweets each)
ZADD timeline:{user_id} {timestamp} {tweet_id}
ZREMRANGEBYRANK timeline:{user_id} 0 -801   -- trim to 800 most recent
EXPIRE timeline:{user_id} 604800            -- 7 days in seconds
```

## Deep Dive 1: Fan-out Strategy

**Pure push model (fan-out on write):**
```
When user A (200 followers) tweets:
  Fan-out service → 200 ZADD operations to Redis
  200 × 0.3ms Redis write = 60ms total (parallelizable → ~5ms actual)

Problem at celebrity scale:
  User with 100M followers tweets:
  Fan-out service → 100M ZADD operations
  At 1M Redis writes/sec: 100 seconds to complete fan-out
  Meanwhile: followers' timelines are stale for up to 100 seconds
```

**Pure pull model (fan-out on read):**
```
Timeline = merge N sorted streams from all followees:
  Fetch last 20 tweets from each of N users you follow
  Merge by timestamp, return top 20

Problem at high fan-in:
  User follows 5,000 accounts → 5,000 Cassandra reads per timeline load
  5,000 × 2ms DB read = 10 seconds → far over 100ms latency budget
  Even with parallelism: still 50 concurrent DB reads → high DB load
```

**Hybrid model (Twitter's approach):**
```
Classification at tweet-creation time:
  Regular users (followers < 1,000,000): push model
    → Fan-out writes tweet_id to all followers' Redis sorted sets
    → Timeline read is O(1) Redis range query + tweet hydration

  Celebrity users (followers ≥ 1,000,000): skip fan-out
    → Their tweets NOT pre-written to follower caches
    → At timeline read time: fetch their recent tweets separately and merge

Timeline read path (hybrid):
  1. redis.zrevrangebyscore(f"timeline:{user_id}", cursor, 0, limit=200)
     → pre-pushed tweet IDs from non-celebrities you follow

  2. for each celebrity C you follow:
       cassandra.query("SELECT * FROM tweets WHERE user_id=C ORDER BY created_at DESC LIMIT 20")
     → their recent tweets (not pre-pushed)

  3. merge_sorted(push_tweet_ids, celebrity_tweets, key=lambda t: t.created_at)[:20]

  4. batch_fetch tweet content for the top 20 tweet IDs
     → Redis cache for hot tweets (< 1h old); Cassandra for older

Cost analysis:
  Follows 100 celebrities? 100 Cassandra reads at read time.
  Most users follow 0-5 celebrities → acceptable overhead (< 20ms)
```

**Fan-out service architecture:**
```python
def process_tweet_event(tweet: TweetEvent):
    # Only fan-out for non-celebrities
    if tweet.author_follower_count >= 1_000_000:
        return  # celebrities handled at read time

    # Fetch follower list (paginated for large but not celebrity-scale accounts)
    follower_ids = social_graph.get_followers(tweet.user_id, limit=1_000_000)

    # Fan-out in batches to avoid overwhelming Redis
    batch_size = 1000
    for i in range(0, len(follower_ids), batch_size):
        batch = follower_ids[i:i+batch_size]
        pipeline = redis.pipeline()
        for follower_id in batch:
            # Only write to active users (logged in within 7 days)
            if user_service.is_active(follower_id):
                pipeline.zadd(f"timeline:{follower_id}",
                              {tweet.tweet_id: tweet.created_at_ms})
                pipeline.zremrangebyrank(f"timeline:{follower_id}", 0, -801)
                pipeline.expire(f"timeline:{follower_id}", 604800)
        pipeline.execute()
```

**G5:** The 1M follower threshold for celebrity vs regular fan-out is a tunable parameter — and it needs active monitoring, not a static setting. Set it too low (100K followers) and you're doing expensive DB pull-merges for thousands of semi-famous accounts on every timeline read. Set it too high (10M followers) and you're fan-outing 10M Redis writes for accounts like Beyoncé or Elon Musk on every tweet, causing write storms. Instrument the fan-out service to track `p99_fan_out_time_ms` per follower-count tier. If accounts with 500K–1M followers are causing p99 fan-out times exceeding 2 seconds (indicating Redis write pressure), lower the threshold. If timeline read latency is creeping up due to too many pull-merge operations, raise the threshold. Automate this as a feedback control loop: tune the threshold to keep both fan-out p99 < 2s and timeline read p99 < 50ms simultaneously.

## Deep Dive 2: Timeline Read Path

**Cache-first read with cursor-based pagination:**
```
Request: GET /v1/timeline?cursor=<encoded_timestamp>&limit=20

1. Decode cursor → last_seen_timestamp (Unix ms)
2. Redis ZREVRANGEBYSCORE timeline:{user_id}
                           max=(last_seen_timestamp - 1)
                           min=0
                           LIMIT 0 200    ← fetch 200 candidates for celebrity merge
   → returns list of tweet_ids ordered by timestamp DESC

3. For each celebrity C in user's follow list:
   Recent tweets from Cassandra (or Redis hot-tweet cache if < 1h old)

4. merge_sorted(redis_tweet_ids, celebrity_tweets)
   → combined stream sorted by created_at DESC

5. Take first 20 from merged stream
   → next_cursor = base64(last_tweet_in_page.created_at_ms)

6. Batch-hydrate tweet content:
   cache_hits  = redis.mget([f"tweet:{id}" for id in top_20_ids])
   cache_misses = [id for id, val in zip(ids, cache_hits) if val is None]
   db_results   = cassandra.get_tweets_by_ids(cache_misses)  -- IN query
   merge and return
```

**Cache warm-up on inactive user re-login:**
```
User hasn't logged in for 10 days → their Redis cache expired

On login:
  Detect: TTLKEY timeline:{user_id} returns -2 (key doesn't exist)
  Response strategy:
    a. Return first page from DB immediately (cold read ~50ms)
    b. Background: spawn async rebuild task:
         for each non-celebrity followee:
           fetch last 20 tweets from Cassandra
         merge all, write top 800 to Redis
         set TTL = 7 days
    c. Subsequent page loads use warm cache

Don't block login/first page on cache rebuild — that would add seconds of latency
```

**Tweet content caching:**
```
Hot tweet cache (in Redis, separate from timeline sorted set):
  Key:   tweet:{tweet_id}
  Value: full tweet JSON (text, author_summary, counts)
  TTL:   60 minutes (tweets go viral fast; counts change frequently)

Cache hit rate by tweet age:
  < 1 hour: ~95% cache hit (high write rate from likes/retweets invalidating counts)
  1–24 hours: ~80% cache hit
  > 24 hours: ~20% cache hit (most old tweets not in hot cache; pulled from Cassandra)
```

**G5:** Cassandra's eventual consistency means a timeline read immediately after a tweet is posted may return an older result from a lagging replica, even for the tweet's own author viewing their own profile. This breaks the "read your own writes" contract users expect. Solve it with client-side session tokens: when user A posts a tweet, the Write API returns a `write_token = {user_id, tweet_id, created_at}`. The client includes this token in the next timeline read. The Timeline API checks: if `write_token.created_at > last_seen_for_user`, route the Cassandra read for that user's tweets to the coordinator (not a replica) to guarantee strong consistency for that specific query. This adds ~2ms overhead for the first read after a write, affects only the author's own reads, and doesn't degrade availability for everyone else.

## Reliability and Operations

- **Fan-out failure:** If a fan-out worker crashes mid-fan-out, the Kafka consumer group offset is not committed → replay from last committed offset on restart. Fan-out ZADD operations are idempotent (adding the same tweet_id to a sorted set is a no-op if already present) → safe to replay.
- **Cache miss storm on cold start:** Pre-warm caches for the top 10M most active users (by login recency) before any traffic hits. Fan-out workers run a "warming" pass on startup. Staged traffic ramp (10% → 25% → 50% → 100%) gives warming time.
- **DB read fallback:** If Redis Cluster is completely down, fall back to Cassandra for all timeline reads. Performance will degrade (150ms+ per read vs < 10ms with cache) but service remains functional. Circuit breaker on Redis: open after 3 consecutive timeouts; try half-open after 30 seconds.
- **Tweet deletion:** Immediate: set `deleted=true` in Cassandra tweet record (atomic CAS). Asynchronous: fan-out a "delete" event to all timeline caches (best-effort ZREM). At read time: filter out deleted tweet_ids before returning. TTL-based cleanup: `deleted` tweets purged from Cassandra after 30 days.
- **Observability:** Track per-tweet-creation: fan-out lag histogram (time from tweet creation to last follower's cache updated). Alert if median fan-out lag exceeds 5 seconds or p99 exceeds 30 seconds. Track per-timeline-read: cache hit rate, celebrity pull-merge time, total latency breakdown.

## Security and Abuse

- Rate limit: 300 tweets/day per account, 50 follows/hour (anti-spam)
- Spam detection: ML model on tweet text at write time (blocking), with async model for edge cases
- Block/mute: filtered at fan-out time (don't write blocked user's tweets to blocker's cache) and at read time as backup
- Private accounts: fan-out only to approved followers; check privacy at read time as second enforcement layer
- DDoS protection: CDN (Fastly/Cloudflare) in front of all public read endpoints

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| Hybrid push/pull for fan-out | Pure push | Pure push fails for celebrities (100M writes per tweet); pure pull fails for high-follow-count users (5000 DB reads per timeline). Hybrid handles both edge cases at the cost of read-path complexity (celebrity merge logic) |
| Redis sorted sets for timeline | Cassandra timeline table | Redis: O(log N) insert, O(log N + K) range scan, sub-millisecond for both; Cassandra: cheaper for cold/large timelines but 2–5ms p99 even cached — too slow for 100ms budget |
| Eventual consistency | Strong consistency | Strong consistency for timelines (linearizable reads) would require quorum reads from Cassandra on every page load — adds 5–20ms and reduces availability during partial failures. Users tolerating a 1–2 second delay in seeing new tweets is an acceptable product trade-off |
| Cassandra for tweets | MySQL sharded | Cassandra: designed for this exact access pattern (partition by user_id, sort by time); writes scale horizontally with no hotspot. MySQL: requires manual sharding, difficult re-sharding, and doesn't handle wide rows as efficiently |

## G5

**What a G5 engineer adds beyond the above:**

1. **Ranked timeline (ML-driven):** Pure reverse-chronological order is easy to implement but Twitter's actual product uses engagement-maximizing ML ranking. The architectural change: fan-out stores raw tweet IDs with timestamps (same as today). At read time, fetch 200 candidates from Redis, fetch engagement signals from a feature store (tweet like velocity in last 1h, user's historical engagement with this author, content category affinity), run a lightweight gradient-boosted model (< 5ms inference in a feature store sidecar), and return top-ranked 20. The feature store must serve with < 10ms p99 for hundreds of signals per tweet. This pipeline is separate from the core timeline service — degrade gracefully to chronological if ranking service is slow.

2. **Real-time push delivery via WebSocket/SSE:** When a user has the Twitter tab open, they expect new tweets to appear without manually refreshing. Architecture: persistent SSE or WebSocket connection per active browser tab → a connection manager service (C10M problem — use Erlang or Go with epoll). When fan-out writes a tweet to user X's Redis timeline, also publish to a per-user Redis pub/sub channel. Connection manager subscribes to `pubsub:timeline:{user_id}` and pushes new tweet IDs to the connected client. Client fetches tweet content for the new IDs. Offline users buffer up to 100 undelivered events in Redis; on reconnect, receive the buffered "N new tweets" count.

3. **Multi-region active-active with CRDT social graph:** Each region serves its local users with regional Redis and Cassandra clusters. Tweet writes go to local region first (W=2 local replicas), then async replicate to other regions (< 200ms). The hard problem is the social graph (follows): it must be globally consistent — if user in EU follows user in US, the EU fan-out service needs to know about it. Solution: treat the follows table as an eventually consistent CRDT (add-wins set for follow edges). Concurrent follow/unfollow from different regions converge correctly. Use HLC (Hybrid Logical Clock) for cross-region ordering. Accept that a follow may take up to 1 second to appear in all regions.

4. **Trending topics pipeline:** Word frequency counting at tweet-stream scale. Kafka → Flink with 10-second tumbling windows → top-K topics by mention velocity (not raw count — velocity detects emerging trends faster). Store results as Redis sorted set `trending:{location}:{language}`. Personalize: trending for NYC English-speaking users differs from global trending. Spam injection defense: if 1,000 accounts all tweet the same hashtag within 60 seconds and those accounts were created within the last 24 hours → flag as coordinated inauthentic behavior, exclude from trending until reviewed. Deduplicate retweets (only count original tweet content).

5. **Celebrity write problem at 50M+ followers:** Even at the 1M fan-out threshold, accounts with exactly 1M followers trigger 1M Redis writes per tweet. But for accounts with 50M+ followers who are excluded from fan-out, the pull-merge at read time fetches 1 row from Cassandra (their last 20 tweets). That's cheap. The real problem is the 1M–5M range: too large for fast fan-out, too small to justify the pull-merge complexity. Solution: pre-computed fan-out groups. For accounts with 1M–10M followers, pre-shard their follower list into groups of 100K. On tweet, dispatch 10–100 fan-out group tasks in parallel rather than processing all followers serially. Target: complete fan-out for a 5M-follower account in < 10 seconds by parallelizing across fan-out workers.
