# 06. Twitter — News Feed and Timeline

**Type:** social-media
**Difficulty:** L4–L5
**Frequency:** Very High — canonical social media design question

---

## Problem Statement

Design Twitter. Users can post tweets, follow other users, and see a personalized timeline of tweets from people they follow, reverse-chronological. The system must handle 300M active users, 500M tweets/day, and serve timelines with sub-100ms latency at peak.

---

## Functional Requirements

**Must have:**
- Post a tweet (text, optional media)
- Follow / unfollow a user
- View home timeline: tweets from people you follow, reverse-chronological
- User profile page: that user's tweets, follower/following counts

**Should have:**
- Like, retweet, reply
- Trending topics (mention as extension)
- Tweet search (mention as extension)

**Out of scope:**
- DMs (separate design — see WhatsApp)
- Ads injection
- Recommendation of who to follow

---

## Non-Functional Requirements

- **Scale:** 300M DAU, 500M tweets/day, 500K tweet/sec peak reads
- **Availability:** 99.99% for timeline reads
- **Latency:** Timeline load < 100ms p99
- **Consistency:** Eventual — slight delay in seeing new tweets is acceptable
- **Durability:** No tweet loss

---

## Capacity Estimation

```
Tweets written: 500M/day = 5,800/sec average; peak ~20K/sec
Timeline reads: 300M DAU × 10 reads/day = 3B/day = 35K reads/sec; peak ~150K/sec
Read:write ratio = 100:1 → read-heavy system

Tweet storage: 500M × 300 bytes = 150GB/day
Media (images/videos): separate in object storage (S3)

Fan-out writes: avg 200 followers/user → 5,800 × 200 = 1.16M writes/sec to timeline caches
Celebrity problem: 100M followers × 1 tweet = 100M fan-out writes in seconds → push model breaks
```

---

## High-Level Architecture

```
Client
  │
  ├─ POST /tweet ──→ [Write API] ──→ [Kafka: tweet-created] ──→ [Fan-out Service]
  │                         │                                          │
  │                    [Tweet DB]                              [Timeline Cache (Redis)]
  │
  └─ GET /timeline ──→ [Timeline API] ──→ [Timeline Cache] ──→ [Tweet Service]
                                                    │
                                             (cache miss)
                                                    │
                                             [Timeline DB]
```

**Components:**
- **Write API:** Validates tweet, saves to Tweet DB, publishes to Kafka
- **Fan-out Service:** Consumes tweet events, pushes tweet ID to followers' timeline caches
- **Timeline API:** Reads pre-computed timeline from Redis cache, hydrates tweet data
- **Tweet DB:** Stores all tweets (Cassandra — wide-column, partitioned by user_id)
- **Social Graph Service:** Stores follow relationships (graph DB or adjacency list in Redis)
- **Timeline Cache (Redis):** Pre-built per-user sorted set of tweet IDs
- **User Service:** Profile data, follower counts

## API Design

```
POST /v1/tweets
Headers: Authorization: Bearer <token>

GET /v1/timeline?cursor=<cursor>&limit=20
Returns: { tweets: Tweet[], next_cursor: string, has_more: bool }

Tweet: { id, text, author: User, created_at, likes, retweets, replies, media: Media[] }

POST /v1/users/:id/follow
DELETE /v1/users/:id/follow
GET /v1/users/:id/tweets?cursor=&limit=
```

## Data Model

```sql
-- Cassandra: tweets by user_id (partition key), created_at (clustering key)
tweets (
  user_id     UUID PARTITION KEY,
  created_at  TIMESTAMP CLUSTERING KEY DESC,
  tweet_id    UUID,
  text        TEXT,
  media_ids   LIST<UUID>
)

-- Social graph: who does user A follow?
follows (
  follower_id  UUID PARTITION KEY,
  followee_id  UUID CLUSTERING KEY,
  created_at   TIMESTAMP
)

-- Redis: pre-built timeline per user
Key:   timeline:{user_id}
Type:  Sorted Set  (score = created_at timestamp, member = tweet_id)
TTL:   7 days (inactive users' timelines expire)
Size:  Keep latest 800 tweet IDs per user (enough for ~40 pages)
```

---

## Deep Dive 1: Fan-out Strategy

**Push model (fan-out on write):**
- When user A tweets, immediately copy tweet ID to all followers' timeline caches
- Timeline read = O(1) cache read
- Problem: celebrity with 100M followers → 100M writes per tweet → fan-out storm

**Pull model (fan-out on read):**
- Timeline = merge sorted streams from all followees' tweet lists
- No fan-out writes
- Problem: user follows 5000 people → 5000 DB reads per timeline load → slow

**Hybrid model (Twitter's approach):**
```
Regular users (followers < 1M): push fan-out
  → tweet ID written to each follower's timeline cache

Celebrity users (followers > 1M): pull on read
  → not pushed; timeline API fetches celebrity tweets separately and merges

At timeline read time:
  timeline_ids = redis.zrange(f"timeline:{user_id}", 0, 20)
  celebrity_tweets = [fetch recent tweets for each celebrity followed]
  merged = merge_sorted(timeline_ids, celebrity_tweets)[:20]
  tweets = batch_fetch(merged)  // hydrate from tweet service
```

**Fan-out service architecture:**
```
Tweet event → Kafka
  → Fan-out workers (N workers, each handles a partition)
  → For each follower:
      if follower is active (logged in last 7 days):
          redis.zadd(f"timeline:{follower_id}", tweet_id, timestamp)
      else:
          skip (will pull from DB when they log in)
```

---

## Deep Dive 2: Timeline Read Path

**Cache-first read:**
```
1. Read 800 tweet IDs from Redis sorted set for user
2. For each ID, fetch tweet data from Tweet Service (also cached in Redis with 1h TTL)
3. Hydrate author info from User Service (cached)
4. Return page of 20 tweets, cursor = last tweet's timestamp
```

**Cache warm-up on login:**
```
User logs in after being inactive:
  → Their timeline cache has expired (7-day TTL)
  → Build timeline from scratch:
      for each followee: fetch last 20 tweets from Cassandra
      merge and sort by timestamp
      write top 800 to Redis
  → Background operation; first page rendered from DB while cache populates
```

**Cursor-based pagination:**
```
next_cursor = base64_encode(last_tweet_timestamp)
On next request: fetch tweet IDs with score < cursor_timestamp
Prevents page drift as new tweets come in (unlike offset-based)
```

---

## Reliability and Operations

- **Fan-out failure:** If fan-out worker crashes mid-fan-out, Kafka consumer group offset is not committed → replay from last offset. Fan-out writes are idempotent (ZADD NX).
- **Cache miss storm on cold start:** Pre-warm caches for top 10M active users. Fan-out workers run continuously, not just on tweet creation.
- **DB read fallback:** If Redis is down, fall back to DB read. Will be slower but correct.
- **Tweet deletion:** Set `deleted=true` in Tweet DB, asynchronously remove from all timeline caches (best effort). Hide at read time as immediate fallback.
- **Observability:** Track fan-out lag (time from tweet creation to all followers' caches updated). Alert if lag > 5 seconds.

---

## Security and Abuse

- Rate limit: 300 tweets/day per user, 50 follows/hour
- Spam detection: ML model on tweet content at write time; async for edge cases
- Block/mute: filter blocked users' tweets at read time from timeline
- Private accounts: fan-out only to approved followers

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Hybrid push/pull | Pure push | Handles celebrities but adds read-path complexity |
| Redis sorted sets | Cassandra timeline | Redis is faster; Cassandra is cheaper for large cold timelines |
| Eventual consistency | Strong | Slight delay in seeing tweets is fine; strong consistency kills performance |
| Cassandra for tweets | MySQL | Cassandra: scales writes; MySQL: richer queries but harder to shard |

## L5 Additions

1. **Ranked timeline:** Reverse-chronological is simple but engagement-maximizing timelines require ML ranking. Architecture change: fan-out stores raw tweet IDs → ranking service scores each tweet (engagement probability, recency, relationship strength) → return top-scored tweets. Ranking runs at read time on the pre-fetched 200 candidates. Requires feature store for real-time signals (user affinity scores, tweet engagement velocity) with < 10ms lookup.

2. **Real-time delivery (push to client):** Twitter uses a persistent connection (SSE or WebSocket) per active browser tab. When a new tweet appears in a user's timeline, the fan-out service also writes to a "live feed" Redis pub/sub channel. A connection management service subscribes and pushes to the client. Offline users buffer in Redis (max 100 undelivered events); on reconnect, they receive the buffered items and a "N new tweets" indicator.

3. **Trending topics pipeline:** Word frequency counting on tweet stream. Kafka → Flink (10-second tumbling windows) → top-K topics by mention velocity. Store in Redis as sorted set. Personalize by user's location and language. Trending is relative to recent baseline (something trending must be significantly above normal volume). Deduplication of retweets. Spam topic injection: if 1000 accounts all tweet the same hashtag in 1 minute, flag for spam review before promoting.

4. **Multi-region active-active:** Each region serves its local users (write locally, read locally). Tweet writes replicate to all regions asynchronously (< 200ms). Fan-out happens per-region. Timeline cache in each region is eventually consistent — user moving between regions may see slightly different timelines. Social graph (follows) must be globally consistent — use CRDT-based eventually consistent graph with convergence guarantees, or use a single global follow DB with read replicas.

5. **The celebrity write problem at scale:** For accounts with >50M followers, even 1 write per tweet = 50M Redis operations = tens of minutes to complete. Solutions: (a) batch fan-out: write to a "pending fan-out" queue, workers drain it, client sees tweet immediately via pull. (b) Use pre-computed fan-out lists (followers are pre-sharded into fan-out groups of 100K each). (c) Hierarchical fan-out: tweet goes to top-level list → each follower fetches from there on next read, with in-memory deduplication.
