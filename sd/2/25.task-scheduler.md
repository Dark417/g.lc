# 25. Distributed Task Scheduler

**Type:** infra
**Difficulty:** G4–G5
**Frequency:** High — cron jobs, delayed tasks, job queues at scale

---

## Problem Statement

Design a distributed task scheduler that can schedule and execute tasks at specific times (cron-style) or with delays (e.g., "send email in 5 minutes"). The system must handle millions of tasks per day, guarantee at-least-once execution, prevent duplicate execution, and support task priorities and retries.

---

## Functional Requirements

**Must have:**
- Schedule a task at a specific time (one-shot: "run at 2:00 PM")
- Schedule a recurring task (cron: "run every day at midnight")
- Execute tasks via callback URL (webhook) or message queue publish
- Retry failed tasks with configurable backoff
- Cancel a scheduled task before it fires

**Should have:**
- Task priorities (high-priority tasks run before low-priority)
- Dependency chains: "Task B runs only after Task A succeeds"
- Task timeout: if task takes > N seconds, mark as failed and retry
- Distributed execution: no single machine processes all tasks

**Out of scope:**
- Building a full DAG workflow engine (Airflow)
- Real-time streaming tasks (< 1 second delay)
- Task result storage / data passing between tasks

---

## Non-Functional Requirements

- **Scale:** 10M tasks/day scheduled; peak 10K tasks firing/sec
- **Accuracy:** Task fires within 1 second of scheduled time
- **Availability:** 99.99% — missed tasks can cost real money
- **At-least-once:** Tasks must execute; duplicates must be handled by the task handler
- **Throughput:** 10K task executions/sec

## Capacity Estimation

```
Tasks per day: 10M scheduled tasks/day
Average rate:  10M / 86,400 = 116/sec steady-state
Peak rate:     10K/sec (e.g., 10K users all schedule daily reports at 9:00 AM UTC)

Burst handling:
  10K tasks fire simultaneously at 9:00 AM
  Each task: HTTP callback or Kafka publish → ~10ms to dispatch
  Workers needed: 10K/sec × 10ms = 100 concurrent workers minimum
  → Worker pool: 200 workers (2× headroom)

Storage:
  Per-task row: 500B (id, payload, schedule, status, retries, metadata)
  10M tasks/day × 500B = 5GB/day new task records
  Retention: 30 days active + history → 150GB total → PostgreSQL handles this well
  Task index size: 10M active tasks → index on (scheduled_at, status) = ~1GB index

Scheduler polling:
  Poll interval: every 1 second
  Query: tasks WHERE scheduled_at <= NOW() AND status = 'scheduled' LIMIT 1000
  At 1 second poll, 116 tasks/sec average → query returns < 200 rows → fast index scan
  Peak: 10K tasks/sec → spread across scheduler instances via FOR UPDATE SKIP LOCKED

Redis for short-term queue (optional, sub-second precision):
  Sorted set score = scheduled_timestamp_ms
  100ms poll: ZRANGEBYSCORE tasks:due -inf {now_ms} LIMIT 0 500
  10K tasks/sec × 200B per entry = 2MB/sec → Redis handles this easily
  Trade-off: Redis is less durable than PostgreSQL (data loss on crash without AOF/RDB)
```

## High-Level Architecture

```
[Client API]
  │  POST /v1/tasks (schedule task)
  ▼
[Task DB (PostgreSQL)]   ←──── [Scheduler (poller, multiple instances)]
                                    │  polls every 1s for due tasks
                                    │  uses FOR UPDATE SKIP LOCKED (no double-dispatch)
                                    ▼
                         [Kafka: task-ready]    ← [Redis: short-term sorted-set queue]
                                    │
                                    ▼
                         [Worker Pool]  ──→  [Callback HTTP / Kafka publish]
                                    │
                         [Retry Handler]   (re-schedules failed tasks)
                                    │
                         [Task DB: update status]

[Cron Manager]: separate service; generates one-shot tasks from cron job definitions
  - Reads cron_jobs table; computes next_run_at for each
  - Creates task records ahead of time (up to 1 hour in advance)
  - Handles DST and timezone-aware cron expressions
```

**Alternative: Redis-only for small scale (< 1M tasks/day):**
```python
# Enqueue with score = fire time
redis.zadd("tasks:pending", {task_id: fire_timestamp_ms})

# Scheduler polls every 100ms
def poll_loop():
    while True:
        now_ms = int(time.time() * 1000)
        # Atomically dequeue up to 100 due tasks
        pipe = redis.pipeline()
        pipe.zrangebyscore("tasks:pending", "-inf", now_ms, start=0, num=100)
        pipe.zremrangebyscore("tasks:pending", "-inf", now_ms)
        results = pipe.execute()
        task_ids = results[0]
        for task_id in task_ids:
            dispatch(task_id)
        time.sleep(0.1)
```

## Deep Dive 1: Scheduler Polling and Exact Timing

**Problem:** Multiple scheduler instances polling the same DB → two instances might pick up and dispatch the same task simultaneously.

**Solution A: FOR UPDATE SKIP LOCKED (recommended)**
```sql
-- Multiple scheduler instances run this concurrently; PostgreSQL serializes their access
-- SKIP LOCKED: instance 1 locks rows it claims; instance 2 skips those rows and claims others
-- Result: each task claimed by exactly one scheduler instance

BEGIN;

SELECT task_id, payload, callback_url, priority
FROM tasks
WHERE status = 'scheduled'
  AND scheduled_at <= NOW()
ORDER BY priority ASC, scheduled_at ASC
LIMIT 100
FOR UPDATE SKIP LOCKED;          -- ← key: skip rows locked by other scheduler instances

-- Atomically transition claimed tasks to 'queued'
UPDATE tasks
SET status = 'queued',
    worker_id = 'scheduler-instance-1',
    updated_at = NOW()
WHERE task_id IN (... IDs from SELECT ...);

COMMIT;
-- Publish task_ids to Kafka: task-ready topic
```

**Why FOR UPDATE SKIP LOCKED works:**
```
Instance 1 and Instance 2 run SELECT ... FOR UPDATE SKIP LOCKED simultaneously
  Instance 1: locks tasks 1–100 (claims them)
  Instance 2: skips tasks 1–100 (locked) → claims tasks 101–200
  → No overlap; no double-dispatch; no explicit coordination needed
  PostgreSQL handles all serialization internally via row-level locks
  SKIP LOCKED: if a row is already locked → skip it (don't wait) → get next available
```

**Solution B: Leader election (alternative, lower DB load)**
```
One scheduler instance is elected as leader (via ZooKeeper/etcd TTL lease):
  Only leader polls the DB → no concurrent polling → no locking needed
  Leader publishes due tasks to Kafka
  Workers pull from Kafka

Leader election via etcd:
  scheduler-1: PUT /scheduler/leader value="scheduler-1" lease=10s
  scheduler-2: tries same key → conflict → watches for leader failure
  On leader crash: lease expires → scheduler-2 acquires key → becomes leader

Failover time: lease TTL (10s) → maximum 10s gap in polling
→ Acceptable for 1-second accuracy requirement (will catch up immediately on recovery)

Trade-off: simpler DB access vs SPOF risk and 10s failover gap
```

**Timing accuracy:**
```
1-second accuracy (most use cases):
  Scheduler polls every 1 second
  Query: scheduled_at <= NOW() → tasks within last 1 second are picked up
  Latency from fire time to execution: poll_interval(1s) + queue_time + worker_time
  Worst case: 2 seconds (task just missed a poll, waits for next)

Sub-100ms accuracy (financial triggers, auctions):
  Scheduler polls every 100ms
  OR: use Redis sorted set as precision trigger
    ZADD tasks:due {fire_timestamp_ms} {task_id}
    Poll: ZRANGEBYSCORE tasks:due -inf {now_ms}
    100ms loop → worst case: 100ms from scheduled time to dispatch

Very high precision (< 10ms):
  Hierarchical Timing Wheel (hardware-clock-based):
  Time divided into 1ms slots in a circular buffer
  Task stored in slot matching fire_time % wheel_size
  On each 1ms tick: fire all tasks in current slot
  O(1) insertion, O(tasks_per_slot) firing
  Used by: Kafka's DelayQueue, Netty's HashedWheelTimer
```

**G5:** The `FOR UPDATE SKIP LOCKED` pattern is correct but creates a subtle clock skew problem in a distributed deployment: each scheduler instance uses its local `NOW()` to determine which tasks are due. If instance 1's clock is 3 seconds ahead of instance 2's, it will fire tasks up to 3 seconds early while instance 2 delays them. In practice, NTP keeps drift under 1ms on well-managed hosts, but under NTP failure or during clock adjustments (leap seconds, stepped corrections), tasks can fire significantly off-schedule. The robust fix is to use a single authoritative time source for scheduled_at comparisons: query `SELECT NOW()` from the PostgreSQL primary as part of the scheduler's poll loop instead of using the host's local time. This shifts timing authority to the database clock — a single source of truth — and avoids instance-level drift causing silent under- or over-firing.

## Deep Dive 2: At-Least-Once Execution with Lease

**Problem:** Worker picks up task, starts executing, then crashes. Without crash detection, the task never completes — silently dropped.

**Lease-based execution (the heartbeat approach):**
```
1. Scheduler claims task:
   UPDATE tasks
   SET status = 'running',
       worker_id = 'worker-pod-7',
       lease_expires_at = NOW() + INTERVAL '5 minutes',
       started_at = NOW()
   WHERE task_id = ? AND status = 'queued';

2. Worker executes the task (calls callback_url or publishes to Kafka)

3a. On success:
    UPDATE tasks SET status = 'succeeded', completed_at = NOW() WHERE task_id = ?;

3b. On failure (callback returns 4xx/5xx or times out):
    UPDATE tasks
    SET status = 'failed',
        retry_count = retry_count + 1,
        next_retry_at = NOW() + (60 * power(2, retry_count)) * INTERVAL '1 second'
    WHERE task_id = ?;
    -- Retry scheduler picks it up at next_retry_at

4. If worker crashes: lease expires (lease_expires_at < NOW())
   Stale lease detector (background job, runs every 60s):
   SELECT task_id FROM tasks
   WHERE status = 'running' AND lease_expires_at < NOW();
   → Reset to 'scheduled' (retry from the beginning)

5. Long-running tasks: worker sends heartbeat every 30 seconds
   UPDATE tasks SET lease_expires_at = NOW() + INTERVAL '5 minutes'
   WHERE task_id = ? AND worker_id = ?;
   → Extends lease; if worker hangs (heartbeat stops), lease expires and task is re-queued
```

**Idempotency requirements for task handlers:**
```
Since at-least-once is guaranteed, handlers MUST be idempotent.
Task IDs serve as idempotency keys:

Email handler:
  Check: SELECT 1 FROM sent_emails WHERE task_id = ?
  If found: skip (already sent); return 200
  If not found: send email; INSERT INTO sent_emails (task_id) VALUES (?)

Payment retry:
  Use task_id as payment idempotency key to PSP:
  stripe.charge(idempotency_key=task_id, ...)
  → PSP deduplicates by idempotency_key; safe to retry

DB update:
  UPDATE orders SET status='confirmed' WHERE id=? AND status='pending'
  AND NOT EXISTS (SELECT 1 FROM task_completions WHERE task_id=?)
  → CAS (compare-and-swap) makes the update naturally idempotent
```

**G5:** The lease approach prevents tasks from being lost on worker crash, but introduces a subtle double-execution window: a worker calls the callback URL successfully, but before it can write `status='succeeded'` to the database, it crashes. The lease expires, the task is re-queued, and another worker calls the callback again. This is unavoidable at-least-once semantics. The non-obvious case is that this window can be made arbitrarily small but never zero without 2PC between the worker DB write and the external callback. The practical mitigation is to make the lease extension atomic with the callback outcome: use the idempotency key in the callback endpoint to guarantee the downstream system handles duplicate deliveries gracefully, and reduce the lease-to-callback timeout ratio so that a failed DB write is detected within seconds, not minutes.

## Deep Dive 3: Cron Job Scheduling

**Generating next run time with DST awareness:**
```python
import croniter
import pytz
from datetime import datetime

def compute_next_run(cron_expr: str, timezone: str, after: datetime = None) -> datetime:
    """
    Compute the next UTC run time for a cron expression in a given timezone.
    Correctly handles DST: "0 9 * * *" in "America/New_York" fires at
      14:00 UTC during EST (UTC-5) and 13:00 UTC during EDT (UTC-4).
    """
    tz = pytz.timezone(timezone)
    # Use timezone-aware datetime as starting point
    after_local = (after or datetime.now(tz)).astimezone(tz)
    cron = croniter.croniter(cron_expr, after_local)
    next_run_local = cron.get_next(datetime)
    # Localize (croniter returns naive datetime)
    next_run_aware = tz.localize(next_run_local, is_dst=None)  # is_dst=None raises if ambiguous
    return next_run_aware.astimezone(pytz.UTC)

# Always recompute from scratch on each tick — never just add a fixed interval
# Why: adding 24 hours to a DST transition results in wrong time
# "0 9 * * *" at 2024-03-10 09:00 EST → next should be 2024-03-11 09:00 EDT = 13:00 UTC
# Adding 86400s to 14:00 UTC gives 14:00 UTC next day → wrong by 1 hour
```

**Preventing double-execution for distributed cron:**
```sql
-- Multiple cron manager instances compete; only one should fire each cron job

-- Claim a cron job for execution (atomic compare-and-swap)
WITH eligible AS (
  SELECT job_id, cron_expr, timezone, callback_url, payload
  FROM cron_jobs
  WHERE is_active = TRUE
    AND next_run_at <= NOW()
    AND (last_run_at IS NULL OR last_run_at < next_run_at)  -- not already run for this tick
  ORDER BY next_run_at ASC
  LIMIT 10
  FOR UPDATE SKIP LOCKED
)
UPDATE cron_jobs
SET last_run_at = NOW(),
    next_run_at = compute_next_run(cron_expr, timezone, NOW())  -- advance schedule
FROM eligible
WHERE cron_jobs.job_id = eligible.job_id
RETURNING eligible.*;
-- Only one cron manager gets each row; others skip locked rows
```

**Catch-up behavior on scheduler downtime:**
```
System down for 2 hours; cron "every 15 minutes" missed 8 runs.

catch_up = FALSE (most use cases):
  Fire once immediately on recovery (the most recent missed run)
  Skip older missed runs — they're stale, firing them now would be wrong
  Example: "generate daily report" — don't run it 8 times if missed

catch_up = TRUE (specific ETL use cases):
  Fire all missed runs in order, as fast as possible
  Example: "process new orders every 5 min" — need all batches processed
  Risk: thundering herd on recovery; use catch_up=TRUE carefully

Implementation for catch_up=FALSE:
  On recovery: compute next_run_at from NOW() (not from last_run_at)
  SELECT jobs WHERE next_run_at <= NOW() → fire once → advance schedule
```

**G5:** Cron jobs with `max_concurrent=1` (prevent overlapping runs) have a subtle failure mode: if the previous run is still "running" (in-flight or stale lease), the cron manager skips firing the next scheduled run. If the running task has a stale lease (worker crashed), it blocks all future runs of that cron job until the lease expiry cleanup job detects and resets it. With a 5-minute lease and a 60-second cleanup interval, a cron job that runs every minute will miss up to 6 consecutive runs before the stale lease is cleared. The fix: in the cron manager's eligibility query, also allow scheduling the next run if `status='running' AND lease_expires_at < NOW()` — treat an expired lease as implicitly failed, and mark the stale run as failed atomically with scheduling the new run. This closes a gap that's invisible in normal operation but surfaces immediately on any worker crash.

## Reliability and Operations

- **Missed tasks during downtime:** On scheduler restart, immediately scan for tasks with `scheduled_at < NOW() AND status = 'scheduled'`. Fire them immediately (late execution is better than no execution). For cron jobs: `catch_up=FALSE` → fire once for the most recently missed window. Alert on any task that was scheduled more than 5 minutes in the past and is still in 'scheduled' state — indicates a scheduler health issue.
- **Task stuck in 'running':** Background stale-lease detector runs every 60 seconds. Query: `WHERE status='running' AND lease_expires_at < NOW()`. Reset to 'scheduled' (if retry_count < max_retries) or 'failed'. Alert on any task in 'running' for > 3× its timeout_sec.
- **Priority starvation:** Low-priority tasks may never run if high-priority tasks keep arriving. Aging: every 10 minutes, boost priority of tasks waiting > 30 minutes. `UPDATE tasks SET priority = GREATEST(1, priority - 1) WHERE status='scheduled' AND created_at < NOW() - INTERVAL '30 minutes'`.
- **Worker pool sizing:** Monitor queue depth (Kafka consumer lag on `task-ready` topic). Auto-scale workers: `desired_workers = ceil(queue_depth / target_messages_per_worker)`. Alert on queue depth growing for > 5 minutes (consumers can't keep up with producers).

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| FOR UPDATE SKIP LOCKED | Leader election + single poller | SKIP LOCKED: scales horizontally, no leader SPOF; Leader: simpler, fewer DB locks, but 10s failover gap |
| Lease-based execution | Optimistic locking (version field) | Lease: handles worker crash automatically; Optimistic: lower DB load for non-crash path but doesn't detect hangs |
| PostgreSQL for tasks | Redis sorted set | PG: durable by default, rich query (priority + status + schedule), easy backfill; Redis: sub-millisecond polling, less durable without AOF |
| Pull workers via Kafka | Push (scheduler calls workers directly) | Pull: workers control their own pace, backpressure natural; Push: lower latency but overwhelms workers on burst |
| Recompute next_run_at each tick | Increment next_run_at by period | Recompute: handles DST, calendar quirks, leap years correctly; Increment: simpler but wrong across DST transitions |

## G5

1. **DAG-based workflow orchestration:** Beyond single tasks, support workflows: Task A → [Task B, Task C in parallel] → Task D (fan-out + fan-in). Architecture: workflow definition stored as a DAG (directed acyclic graph) in DB. Workflow engine: on each task completion, evaluate which tasks are newly unblocked (all dependencies satisfied) → schedule them immediately. Fan-in: Task D waits for both B and C; counter per join node, `dependencies_remaining` decremented atomically on each predecessor completion; fires when counter reaches 0. Error handling: `on_failure` policy per task — `cancel_workflow`, `continue`, `retry_parallel`. This is what Apache Airflow, Temporal, and AWS Step Functions provide. Key engineering challenge: efficiently evaluating partial completion state across millions of concurrent workflow instances without a full graph scan.

2. **Task routing and data locality:** For tasks that process sharded data (e.g., generate monthly invoice for EU customer), route the task to a worker in the same region to avoid cross-region data transfer and satisfy data residency requirements. Architecture: task record includes an `affinity` tag (e.g., `region:eu`, `shard:user-1234`). Worker pools are tagged with matching labels. Scheduler routes tasks to pools that match affinity tags — implemented as separate Kafka topics per affinity group, with workers only consuming from their topic. This reduces cross-region bandwidth (expensive in cloud), latency, and GDPR compliance risk (EU user data never leaves EU region for processing).

3. **High-precision timing with a hierarchical timing wheel:** For use cases requiring < 10ms accuracy (financial settlement, live auction ending), polling-based scheduler is too imprecise. The hierarchical timing wheel: a circular array of N slots representing 1ms each, with overflow buckets for tasks scheduled > N ms in the future. On each 1ms tick (wall clock + system timer), the wheel advances one slot and fires all tasks in the current slot. O(1) insertion (compute slot = fire_time_ms % wheel_size) and O(tasks_in_slot) firing. For tasks > N ms out: store in overflow buckets, cascade them into the wheel as their time approaches. Netty's `HashedWheelTimer` and Kafka's internal timer use this design. Memory: N slots × avg_tasks_per_slot × task_pointer_size; for N=1000 slots (1 second span) with 10K tasks/sec average = 10 tasks/slot = feasible.

4. **Back-pressure and burst handling:** A burst of 100K tasks firing at midnight (end-of-month batch jobs) is predictable. Without back-pressure, the worker queue grows unboundedly and workers may OOM or starve other task types. Solutions: (a) Priority-aware dispatch: critical tasks always pass through; low-priority tasks are admitted only when queue depth < threshold; (b) Worker auto-scaling: HPA on Kubernetes keyed to Kafka consumer group lag — target lag < 10K messages; (c) Task admission control: if queue depth > 1M tasks, reject new low-priority task submissions with 429, require retry. Drain strategy: process oldest tasks first (FIFO within priority tier) to prevent perpetually stale tasks at the head of the queue.

5. **Cost-aware scheduling for non-urgent workloads:** Cloud compute costs vary by time of day — AWS Spot prices for GPU/CPU instances fluctuate 4–10× between peak and off-peak. Non-urgent tasks (ML training, batch reports, data exports) have an `urgency` tag: `immediate`, `best-effort`, `whenever-cheap`. A cost oracle service queries AWS Spot price APIs every 5 minutes. Scheduler: if `urgency=whenever-cheap` AND spot price < $0.10/hr → dispatch; if price > threshold → defer and re-evaluate at next poll. During cheap windows (2–6 AM UTC, typically 60–80% cheaper), drain the `whenever-cheap` queue aggressively. This approach reduces batch compute costs 40–60% with no change to business outcomes (the reports are ready by morning regardless). Design consideration: the cost oracle must be a non-critical dependency — its failure causes tasks to fall back to `best-effort` (always dispatch), not to block indefinitely.
