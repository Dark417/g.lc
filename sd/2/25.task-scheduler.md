# 25. Distributed Task Scheduler

**Type:** infra
**Difficulty:** L4–L5
**Frequency:** High — cron jobs, delayed tasks, job queues at scale

---

## Problem Statement

Design a distributed task scheduler that can schedule and execute tasks at specific times (cron-style) or with delays (e.g., "send email in 5 minutes"). The system must handle millions of tasks per day, guarantee at-least-once execution, prevent duplicate execution, and support task priorities and retries.

---

## Functional Requirements

**Must have:**
- Schedule a task at a specific time (one-shot: "run at 2:00 PM")
- Schedule a recurring task (cron: "run every day at midnight")
- Execute tasks via callback URL (webhook) or message queue publish
- Retry failed tasks with configurable backoff
- Cancel a scheduled task before it fires

**Should have:**
- Task priorities (high-priority tasks run before low-priority)
- Dependency chains: "Task B runs only after Task A succeeds"
- Task timeout: if task takes > N seconds, mark as failed and retry
- Distributed execution: no single machine processes all tasks

**Out of scope:**
- Building a full DAG workflow engine (Airflow)
- Real-time streaming tasks (< 1 second delay)
- Task result storage / data passing between tasks

---

## Non-Functional Requirements

- **Scale:** 10M tasks/day scheduled; peak 10K tasks firing/sec
- **Accuracy:** Task fires within 1 second of scheduled time
- **Availability:** 99.99% — missed tasks can cost real money
- **At-least-once:** Tasks must execute; duplicates must be handled by the task handler
- **Throughput:** 10K task executions/sec

---

## Capacity Estimation

```
Tasks: 10M scheduled/day; 10K/sec firing rate (tasks can be batched to fire at same time)
  e.g., 10K users set daily email reports at 9:00 AM → 10K tasks fire simultaneously

Storage:
  Each task: 500B (id, payload, schedule, retries) → 10M × 500B = 5GB/day
  Retention: keep 30 days of task history → 150GB total

Timing accuracy:
  Database polling: every 1 second → 10K tasks queried per second → manageable
  Better: sorted set in Redis → O(log N) per check
```

## High-Level Architecture

```
[Client API]
  │  schedule task
  ▼
[Task DB (PostgreSQL)] ──→ [Kafka: task-ready] ──→ [Worker Pool] ──→ [Task Handler]
       │                                                    │             (webhook or queue)
[Scheduler (poller)]    [Redis: short-term queue]          │
  (polls every 1s for                                [Retry Handler]
   tasks due to fire)
       │
  publishes to Kafka when task is due
```

**Alternative: Redis-only for small scale:**
```
Sorted set in Redis (score = fire_time):
  ZADD tasks:pending {fire_timestamp} {task_id}

Scheduler polls (every 100ms):
  ZRANGEBYSCORE tasks:pending -inf {now} LIMIT 0 100
  → Atomically dequeue and process
```

## Deep Dive 1: Scheduler Polling and Exact Timing

**Problem:** Multiple scheduler instances polling the same DB → multiple workers process the same task.

**Solution A: Leader election**
```
Only one scheduler instance is active (elected via ZooKeeper/etcd with lease)
Leader polls DB every 1 second:
  SELECT * FROM tasks WHERE scheduled_at <= NOW() AND status = 'scheduled' LIMIT 1000

Atomically claim tasks:
  UPDATE tasks SET status = 'queued', worker_id = ? WHERE task_id IN (?) AND status = 'scheduled'
  (Only tasks still 'scheduled' get claimed)

Publish claimed task_ids to Kafka → workers consume and execute

Pros: No coordination per task; single query
Cons: Leader is SPOF (mitigate with fast failover < 10s)
```

**Solution B: Database-level row locking (chosen for simplicity)**
```
Multiple scheduler instances compete for tasks:

SELECT * FROM tasks
WHERE scheduled_at <= NOW() AND status = 'scheduled'
ORDER BY priority, scheduled_at
LIMIT 100
FOR UPDATE SKIP LOCKED;  ← PostgreSQL 9.5+; skip rows already locked by other instances

UPDATE tasks SET status = 'queued', worker_id = ?
WHERE task_id IN (... selected IDs ...);

FOR UPDATE SKIP LOCKED: each scheduler instance gets a different batch → no duplicates
```

**Timing accuracy:**
```
Tasks scheduled_at: stored as UTC timestamp
Scheduler polls every 1 second: all tasks with scheduled_at <= NOW() are fired
Accuracy: within 1 second of scheduled time (acceptable for most use cases)

For < 100ms accuracy: use Redis sorted set:
  ZADD tasks:due {scheduled_timestamp} {task_id}
  Scheduler: loop every 100ms:
    ZRANGEBYSCORE tasks:due -inf {now_ms} LIMIT 0 100
    Process + remove: ZREM tasks:due {task_id}
  → More accurate but Redis is not as durable as PostgreSQL
```

## Deep Dive 2: At-Least-Once Execution with Lease

**Problem:** Worker picks up task, starts executing, then crashes. Task must be retried.

**Lease-based execution:**
```
1. Scheduler claims task: SET status = 'running', lock_expires_at = NOW() + 5min
2. Worker executes task
3. On success: SET status = 'succeeded', completed_at = NOW()
4. On failure: SET status = 'failed', retry_count += 1, next_retry_at = NOW() + backoff(retry_count)
5. If worker crashes: lease expires
   → Scheduler re-picks the task (scheduled_at check includes expired leases):
     WHERE (status = 'scheduled' AND scheduled_at <= NOW())
        OR (status = 'running' AND lock_expires_at < NOW())
```

**Heartbeat to extend lease:**
```
Long-running tasks: worker sends heartbeat every 30s:
  UPDATE tasks SET lock_expires_at = NOW() + 5min WHERE task_id = ? AND worker_id = ?

If task has valid worker_id and lock_expires_at in future → still running
If lock expires → re-queued by scheduler
```

**Exactly-once at application level:**
```
Scheduler guarantees at-least-once (task may run twice if worker crashes)
Application must handle idempotency:
  - Email: idempotency_key = (task_id, user_id) → don't send twice
  - Payment: idempotency_key in payment processor → won't charge twice
  - DB updates: use task_id as idempotency key; check if already applied
```

## Deep Dive 3: Cron Job Scheduling

**Generating next run time:**
```python
import croniter

def compute_next_run(cron_expr, timezone, after=None):
    tz = pytz.timezone(timezone)
    after_dt = after or datetime.now(tz)
    cron = croniter.croniter(cron_expr, after_dt)
    next_run = cron.get_next(datetime)
    return next_run.astimezone(pytz.UTC)

# On cron job creation: compute first next_run_at
# When cron job fires: create a task for this run, then compute new next_run_at
```

**Distributed cron: preventing double execution:**
```
Multiple scheduler instances: which one fires a given cron job?

Option: ownership via consistent hashing
  job_id hashed to scheduler instance → only that instance fires this job
  If instance fails: re-hash to another instance (within 30s)

Option: database compare-and-swap
  SELECT cron_jobs WHERE next_run_at <= NOW() AND last_run_at < next_run_at FOR UPDATE SKIP LOCKED;
  UPDATE cron_jobs SET last_run_at = NOW(), next_run_at = compute_next(cron_expr, NOW())
  → Only one instance claims each cron job execution
```

**Timezone handling:**
```
"Every day at 9 AM" in New York means different UTC times during DST:
  EST (UTC-5): 9 AM = 14:00 UTC
  EDT (UTC-4): 9 AM = 13:00 UTC

Always store next_run_at as UTC
Recompute next_run_at from scratch each time (don't just add 24h)
→ Handles DST transitions correctly
```

## Reliability and Operations

- **Missed tasks during downtime:** On scheduler restart, immediately scan for tasks with scheduled_at in the past. Fire them immediately (late execution is better than no execution). For cron jobs: fire the most recent missed execution only (skip intermediate ones unless configured for catch-up).
- **Task stuck in 'running':** Stale lease expiry (5 min default). Background job scans for expired leases every minute and re-queues them.
- **Priority starvation:** Low-priority tasks may starve if high-priority tasks keep arriving. Add aging: boost priority of tasks that have been waiting > 10 minutes.
- **Monitoring:** Alert on: tasks with next_retry_at in past (stuck retries), scheduler-to-worker lag (time from queue to execution), tasks in 'running' with expired lease.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| FOR UPDATE SKIP LOCKED | Leader election | DB locking: simpler; Leader election: lower DB load |
| Lease-based execution | Optimistic locking | Lease: handles crashes; Optimistic: faster for non-crash path |
| PostgreSQL for tasks | Redis sorted set | PG: durable; Redis: faster, less durable |
| Pull-based workers | Push (Kafka) | Pull: workers control pace; Push: lower latency |

## L5 Additions

1. **DAG-based workflow orchestration:** Beyond single tasks, support workflows: Task A → [Task B, Task C in parallel] → Task D. Architecture: workflow definition as a DAG stored in DB. On task completion, workflow engine evaluates which tasks are now unblocked (all dependencies satisfied) → schedule them. Fan-out on completion. Error handling: if Task B fails, should Task C be cancelled? Continue? Depends on policy. This is what Airflow, Temporal, and AWS Step Functions provide at scale. Key challenge: efficiently evaluating dependency graphs across millions of workflows.

2. **Task routing and locality:** For tasks that process data from a specific shard (e.g., generate monthly invoice for customer in EU region), route the task to a worker in the same region. Architecture: task has an affinity tag (region, customer_shard). Worker pools are tagged. Scheduler routes tasks to pools matching affinity. This reduces cross-region data transfer and compliance concerns (EU data processed in EU). Implementation: task affinity stored as DB field; scheduler matches affinity tags when claiming tasks.

3. **High-precision timing (sub-second):** For use cases requiring < 100ms accuracy (e.g., financial settlement triggers, live auction ending), polling-based scheduler is insufficient. Architecture: use a timer wheel or hierarchical timing wheel (Kafka's implementation). Time divided into slots; each task stored in the slot corresponding to its fire time. Wheel advances every 1ms; on advance, fire all tasks in current slot. Memory efficient: circular buffer of N slots × pointers to task lists. Handles millions of tasks with O(1) insertion and O(1) firing.

4. **Back-pressure and rate limiting of task execution:** A burst of 100K tasks firing at midnight is common (daily reports, batch billing). Workers can't handle all at once. Back-pressure: workers report their current load; scheduler throttles task dispatch if workers are busy. Priority-aware throttling: critical tasks always pass through; low-priority tasks are buffered. Worker auto-scaling: detect queue backlog → spin up additional workers (Kubernetes HPA based on queue depth). Target: drain backlog within 5 minutes.

5. **Cost-aware task scheduling:** Cloud costs vary by time of day (AWS spot prices, GCP preemptible VMs). Schedule non-urgent tasks (batch reports, ML training jobs) during off-peak hours (2-6 AM) when spot instances are cheapest. Architecture: task has urgency tag: "immediate", "best-effort", "whenever-cheap". Scheduler has a cost oracle (queries AWS spot price API). If spot price < threshold: run best-effort tasks. If price spikes: pause them, save spot instances for critical tasks. Can reduce compute costs 40-60% for batch workloads.
