# 10. Yelp — Proximity Search / Local Business Discovery

**Type:** social-media
**Difficulty:** L4
**Frequency:** High — geo-search, proximity indexing, read-heavy local search

---

## Problem Statement

Design Yelp. Users can search for businesses near their location (restaurants, shops, services), view ratings and reviews, and add reviews themselves. The system must handle 100M DAU, support efficient geospatial queries ("find restaurants within 2km"), and return results in < 200ms.

---

## Functional Requirements

**Must have:**
- Search businesses by location (lat/lng radius) + optional filters (category, rating)
- View business details: name, address, phone, hours, rating, review count
- Read reviews for a business (paginated)
- Write a review (text + star rating 1-5)

**Should have:**
- Photos attached to reviews or business
- Business owner can respond to reviews
- User profile: reviews written, photos uploaded

**Out of scope:**
- Reservations / booking (OpenTable integration)
- Ads / promoted listings
- Real-time availability (hours are stored, not live)

---

## Non-Functional Requirements

- **Scale:** 100M DAU, 10M businesses, 500M reviews, 50K search QPS
- **Latency:** Search results < 200ms p99; business detail < 50ms p99
- **Availability:** 99.99%
- **Consistency:** Eventual — new reviews appear within 30 seconds is fine
- **Read:write:** 100:1 — heavily read-biased

---

## Capacity Estimation

```
Businesses: 10M records × 1KB = 10GB (trivially fits in memory)
Reviews: 500M × 500B = 250GB; growing at 5M/day = 2.5GB/day
Search QPS: 50K/sec
Review writes: 5M/day = 58/sec (low, not a bottleneck)

Geo index: 10M points with 2D coordinates
  Geohash index: 10M × 20B = 200MB in memory → entire index fits in Redis
```

## API Design

```
GET /v1/search
Params: lat, lng, radius_km (default 5), category?, min_rating?, sort_by=distance|rating|review_count, cursor, limit=20
Returns: { businesses: Business[], total, next_cursor }

Business: { id, name, address, lat, lng, category, avg_rating, review_count, phone, hours }

GET /v1/businesses/{id}
Returns: Business + { photos: Photo[], top_reviews: Review[] }

GET /v1/businesses/{id}/reviews?cursor=&limit=20&sort=recent|relevant
Returns: { reviews: Review[], next_cursor }

Review: { id, author: User, rating: int, text, photos, created_at, owner_response? }

POST /v1/businesses/{id}/reviews
Body: { rating: int, text: string, photo_ids?: string[] }
Returns: { review_id, created_at }
```

## Data Model

```sql
-- PostgreSQL: businesses
businesses (
  id           UUID PRIMARY KEY,
  name         TEXT NOT NULL,
  category     TEXT,
  address      TEXT,
  city         TEXT,
  lat          DECIMAL(9,6),
  lng          DECIMAL(9,6),
  phone        TEXT,
  hours        JSONB,  -- { "mon": "9:00-21:00", ... }
  avg_rating   DECIMAL(2,1),
  review_count INT,
  created_at   TIMESTAMPTZ,

  -- PostGIS geo index
  location     GEOGRAPHY(POINT, 4326)
)
CREATE INDEX ON businesses USING GIST (location);
CREATE INDEX ON businesses (category, avg_rating DESC);

-- Cassandra: reviews (write-heavy, high volume)
reviews (
  business_id  UUID PARTITION KEY,
  review_id    TIMEUUID CLUSTERING KEY DESC,
  user_id      UUID,
  rating       TINYINT,
  text         TEXT,
  photo_ids    LIST<UUID>
)

-- PostgreSQL: review aggregates (denormalized for fast reads)
-- Updated async via Kafka consumer
business_ratings (
  business_id  UUID PRIMARY KEY,
  avg_rating   DECIMAL(2,1),
  review_count INT,
  rating_dist  JSONB  -- { "1": 10, "2": 20, "3": 50, "4": 200, "5": 300 }
)
```

## Deep Dive: Geospatial Indexing

**Problem:** "Find all businesses within 5km of (37.77, -122.41)"
- Naive: scan all 10M businesses and compute distance → O(N), too slow
- Need a spatial index

**Option A: PostGIS (R-tree index)**
```sql
SELECT * FROM businesses
WHERE ST_DWithin(location, ST_MakePoint(-122.41, 37.77)::geography, 5000)
  AND category = 'restaurant'
ORDER BY ST_Distance(location, ST_MakePoint(-122.41, 37.77)::geography)
LIMIT 20;
```
- Excellent for accuracy, powerful filter options
- R-tree index enables O(log N) spatial queries
- Works best in single-region DB; hard to distribute

**Option B: Geohash**
```
Geohash: encode (lat, lng) as a base32 string
  Precision 5 (5 chars): ~5km × 5km cells
  Precision 6 (6 chars): ~1.2km × 1.2km cells

Index: CREATE INDEX ON businesses (geohash_6);

Query:
  1. Compute geohash prefix of search center at appropriate precision
  2. Find adjacent geohash cells (8 neighbors + center = 9 cells)
  3. SELECT * FROM businesses WHERE geohash_6 IN ('abc12', 'abc13', ..., 'abc1e')
  4. Filter by actual distance (haversine formula) to handle cell boundary effects
  5. Sort by distance

Stored in Redis:
  Key: geohash:{precision}:{cell_id}
  Value: Set of business IDs
  → Range search: fetch 9 cells from Redis, union, sort by distance
```

**Option C: S2 Geometry Library (Google's approach)**
```
S2 cells: hierarchical subdivision of sphere into cells
S2CellId: 64-bit integer encoding of cell position + level
  Level 12 = ~0.5km; Level 15 = ~60m

Advantages over geohash:
  - Better geometric properties (cells are nearly square at all levels)
  - No distortion at poles
  - Exact coverage using S2Region.GetCovering()
  - Used in Google Maps, Foursquare, MongoDB

Redis implementation:
  GEOADD businesses {lng} {lat} {business_id}
  GEORADIUS businesses -122.41 37.77 5 km ASC COUNT 50
  → Redis uses sorted set with geohash score → O(log N + M) query
```

**Recommended:** Redis GEODATA for simplicity + PostGIS for complex polygon queries.

## Deep Dive 2: Rating Aggregation

**Problem:** Updating avg_rating on every review write creates a hot row.
- 100 reviews/sec on a popular restaurant → 100 UPDATE business SET avg_rating = ... per sec

**Solution: Async aggregation**
```
1. Review written → Kafka topic "review-created"
2. Rating aggregator consumes events:
   a. UPDATE business_ratings SET review_count += 1, sum_rating += review.rating
   b. Recompute avg = sum_rating / review_count
   c. Write back to businesses table (or business_ratings)
3. Business detail API reads avg_rating from cache (updated every 30s from DB)
4. User sees updated rating within ~30 seconds
```

**Approximate average (streaming):**
```
Running average: new_avg = old_avg + (new_rating - old_avg) / new_count
No need to store sum_rating — saves a column
Works correctly for concurrent updates with optimistic locking
```

## Reliability and Operations

- **Search cache:** Cache popular city/category search results (NYC restaurants sorted by rating) with 60s TTL. 95% cache hit rate for common searches.
- **Review fraud:** Rate limit reviews per user per business (1/day). Flag reviews from newly created accounts. ML model for fake review detection (linguistic patterns, IP clustering).
- **Business data freshness:** Business hours, phone, address change → owner can edit via dashboard. Changes async sync to Redis cache (TTL-based invalidation).

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Redis GEO | PostGIS | Redis: fast, in-memory, simple API; PostGIS: richer spatial queries, more scalable for complex filters |
| Geohash | S2 cells | Geohash: simple, widely supported; S2: geometrically better, more complex |
| Cassandra for reviews | PostgreSQL | Cassandra: scales writes; PG: richer queries, text search support |
| Async rating aggregation | Synchronous update | Async: avoids hot rows; sync: always accurate but creates bottleneck |

## L5 Additions

1. **Personalized ranking:** Beyond distance and rating, rank results by personal relevance. Signals: user's cuisine history, price sensitivity, reviews from people they follow. Architecture: candidate generation from geo query (100 nearby businesses) → scoring: distance score × rating score × personalization score → return top 20. Personalization model runs in < 20ms using pre-computed affinity scores from feature store.

2. **Geospatial sharding:** At global scale, geo data doesn't fit in a single DB. Shard by geo region: US-West, US-East, EU, APAC. Route requests by user's location to the nearest regional shard. Cross-region businesses (chains with locations everywhere) → write to all relevant regional shards. Global search for a chain ("Starbucks in NYC") → route to US-East shard only.

3. **Real-time event detection:** "What's popular right now near me?" Detect anomalies in foot traffic (check-ins, reviews) to surface trending venues. Kafka → Flink → sliding window count of reviews/check-ins per business per 30 minutes. Compare to historical baseline. If 5x above normal → "Popular right now" badge. Alert for negative events (spike in 1-star reviews → investigate).

4. **Offline-first mobile app:** For mobile users, cache last 1000 nearby businesses in local SQLite DB. Allow search and business details to work offline. Sync strategy: on app open, refresh geo index for user's current city in background. If offline, serve from local cache with staleness indicator. Upload queued reviews when connection restores.

5. **Review quality scoring:** Surface the most helpful reviews, not just most recent. Signals: review length, media attached, number of helpful votes, reviewer reputation (verified purchase, review history quality). Score = recency_decay × quality_signal × helpfulness_votes. Pre-compute scores async; sort by score for "Most Relevant" view. Re-rank periodically as helpful votes accumulate.
