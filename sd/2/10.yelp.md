# 10. Yelp — Proximity Search / Local Business Discovery

**Type:** social-media
**Difficulty:** G4
**Frequency:** High — geo-search, proximity indexing, read-heavy local search

---

## Problem Statement

Design Yelp. Users can search for businesses near their location (restaurants, shops, services), view ratings and reviews, and add reviews themselves. The system must handle 100M DAU, support efficient geospatial queries ("find restaurants within 2km"), and return results in < 200ms.

---

## Functional Requirements

**Must have:**
- Search businesses by location (lat/lng radius) + optional filters (category, rating)
- View business details: name, address, phone, hours, rating, review count
- Read reviews for a business (paginated)
- Write a review (text + star rating 1-5)

**Should have:**
- Photos attached to reviews or business
- Business owner can respond to reviews
- User profile: reviews written, photos uploaded

**Out of scope:**
- Reservations / booking (OpenTable integration)
- Ads / promoted listings
- Real-time availability (hours are stored, not live)

---

## Non-Functional Requirements

- **Scale:** 100M DAU, 10M businesses, 500M reviews, 50K search QPS
- **Latency:** Search results < 200ms p99; business detail < 50ms p99
- **Availability:** 99.99%
- **Consistency:** Eventual — new reviews appear within 30 seconds is fine
- **Read:write:** 100:1 — heavily read-biased

---

## Capacity Estimation

```
Business data:
  10M businesses × 1KB per record = 10GB
  → Fits entirely in memory on a single high-RAM server (trivially)
  → Certainly fits in Redis GEO structure for geo indexing

Review volume:
  500M existing reviews × 500 bytes avg = 250GB
  New reviews: 5M/day × 500B = 2.5GB/day (slow growth)
  → PostgreSQL handles 250GB with proper partitioning; Cassandra for scale

Search QPS:
  50K search queries/sec
  Typical search: lat/lng + radius (2–10km) + category filter + sort
  Result set: 20–100 businesses

Geospatial index:
  10M business points with (lat, lng) coordinates
  Geohash index: 10M × ~20 bytes per entry = 200MB total
  → Entire geo index fits in Redis on a single node (with replicas for HA)
  Redis GEORADIUS command: O(N+log(M)) where N = matches, M = total entries
  → At 50K QPS × O(log 10M) ≈ 50K × 23 ops = 1.15M comparisons/sec → trivial

Cache sizing:
  Popular search queries (city + category combos): 100K distinct query signatures
  Each result set: 20 businesses × 1KB = 20KB per cached result
  Total: 100K × 20KB = 2GB → fits in a single Redis instance

Review write rate:
  5M/day ÷ 86,400 = 58/sec → never a bottleneck
```

## Data Model

```sql
-- PostgreSQL: businesses (primary metadata store, 10GB total — fits comfortably)
businesses (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name         TEXT NOT NULL,
  category     TEXT NOT NULL,       -- 'restaurant', 'coffee', 'bar', 'gym', etc.
  address      TEXT,
  city         TEXT NOT NULL,
  state        TEXT,
  country      TEXT DEFAULT 'US',
  lat          DECIMAL(9,6) NOT NULL,
  lng          DECIMAL(9,6) NOT NULL,
  phone        TEXT,
  website      TEXT,
  price_range  CHAR(4),             -- '$', '$$', '$$$', '$$$$'
  hours        JSONB,               -- { "mon": "9:00-21:00", "tue": "9:00-21:00", ... }
  avg_rating   DECIMAL(2,1),        -- cached here for fast search results
  review_count INT DEFAULT 0,
  is_claimed   BOOLEAN DEFAULT false,  -- owner has claimed/verified the business
  created_at   TIMESTAMPTZ DEFAULT now(),

  -- PostGIS geography type for accurate distance calculations on Earth's surface
  location     GEOGRAPHY(POINT, 4326)  -- (lng, lat) in WGS84 coordinate system
)

-- Spatial index for radius queries
CREATE INDEX ON businesses USING GIST (location);
-- Composite index for filtered searches
CREATE INDEX ON businesses (category, avg_rating DESC) WHERE avg_rating IS NOT NULL;
-- Index for city-level searches (common pre-filter)
CREATE INDEX ON businesses (city, category, avg_rating DESC);

-- Cassandra: reviews (high read volume, partitioned for efficient business-level queries)
reviews (
  business_id   UUID PARTITION KEY,     -- all reviews for a business in one partition
  review_id     TIMEUUID CLUSTERING KEY DESC,  -- time-ordered, newest first
  user_id       UUID NOT NULL,
  rating        TINYINT NOT NULL,        -- 1–5 stars
  text          TEXT,
  photo_ids     LIST<UUID>,
  helpful_votes INT DEFAULT 0,
  owner_response_text TEXT,
  owner_response_at   TIMESTAMP,
  created_at    TIMESTAMP,
  PRIMARY KEY (business_id, review_id)
) WITH CLUSTERING ORDER BY (review_id DESC)

-- PostgreSQL: review aggregates (denormalized — avoids scanning Cassandra for counts)
-- Updated asynchronously by the rating aggregator service
business_ratings (
  business_id   UUID PRIMARY KEY REFERENCES businesses(id),
  avg_rating    DECIMAL(2,1),
  review_count  INT,
  rating_dist   JSONB,   -- { "1": 10, "2": 25, "3": 80, "4": 200, "5": 350 }
  sum_rating    INT,     -- for running average calculation
  last_updated  TIMESTAMPTZ
)

-- Redis: geo index (entire 10M business dataset in 200MB — in-memory for ultra-fast queries)
-- GEOADD stores (lng, lat, member) in a sorted set with geohash score
GEOADD businesses {lng} {lat} {business_id}

-- Redis: business detail cache (for fast /businesses/:id responses)
Key:   biz:{business_id}
Type:  JSON string (serialized Business object)
TTL:   5 minutes (business hours, phone, avg_rating change occasionally)

-- Redis: search result cache (for repeated searches in popular areas)
Key:   search:{city}:{category}:{sort_by}:{radius_km}  -- coarse-grained key
Type:  JSON string (serialized search result list of business_ids)
TTL:   60 seconds (new reviews affect rankings)
```

## Deep Dive 1: Geospatial Indexing

**Problem:** "Find all restaurants within 5km of (37.77, -122.41)"

**Option A: Naive linear scan**
```sql
SELECT *, ST_Distance(location, target) AS dist
FROM businesses
WHERE category = 'restaurant'
ORDER BY dist
LIMIT 20;
```
- Scans all 10M businesses for every query → O(10M) per query
- At 50K QPS: 500B operations/sec → catastrophically slow
- No index helps with distance ordering

**Option B: PostGIS with R-tree index (GIST)**
```sql
SELECT id, name, avg_rating,
       ST_Distance(location, ST_MakePoint(-122.41, 37.77)::geography) AS dist_m
FROM businesses
WHERE ST_DWithin(location, ST_MakePoint(-122.41, 37.77)::geography, 5000)  -- 5000m = 5km
  AND category = 'restaurant'
  AND avg_rating >= 4.0
ORDER BY dist_m
LIMIT 20;
```
- R-tree index (GIST) enables O(log N + K) spatial queries
- `ST_DWithin` uses bounding box pre-filter → R-tree traversal → exact distance computation
- Handles complex shapes (polygons, not just circles)
- **Best for:** Complex spatial queries with many filter dimensions
- **Limitation:** Single-node PostgreSQL; hard to distribute across shards while maintaining spatial locality

**Option C: Geohash**
```
Geohash: encode (lat, lng) as a base32 string
  Precision 4 (4 chars): ~40km × 20km cells
  Precision 5 (5 chars): ~5km × 5km cells   ← appropriate for 5km radius search
  Precision 6 (6 chars): ~1.2km × 1.2km cells

Algorithm for 5km radius search:
  1. Compute geohash of search center at precision 5: e.g., "9q8yy"
  2. Find 8 neighboring geohash cells:
     (all adjacent cells at same precision — 3×3 grid = 9 cells including center)
     Neighbors: "9q8yx", "9q8yz", "9q8yv", "9q8yw", "9q8yu", etc.
  3. SELECT * FROM businesses WHERE geohash_5 IN (...9 geohash values...)
  4. Post-filter: for each result, compute exact Haversine distance
     (cell boundaries don't align with circle → some results outside radius)
  5. Sort by exact distance, return top 20

Stored in Redis as hash-to-business-id mapping:
  Key:   geohash5:{cell_id}
  Value: Set of business_ids in that cell
  SMEMBERS geohash5:9q8yy → {business_id_1, business_id_2, ...}
  → Union 9 cells, then filter by exact distance
```

**Option D: Redis GEOADD / GEORADIUS (chosen for primary implementation)**
```
Redis Geo commands use a sorted set with geohash score encoding (lat, lng) as a 52-bit integer:
  GEOADD businesses {lng} {lat} {business_id}   -- O(log N)
  GEORADIUS businesses -122.41 37.77 5 km ASC COUNT 100 WITHCOORD WITHDIST
  → O(N+log(M)) where N = results returned, M = total members
  → Returns business_ids sorted by distance, with exact distances

Advantages:
  - In-memory → sub-millisecond queries (0.1–0.5ms for 10M dataset)
  - GEORADIUS already does exact distance calculation (no post-filter needed)
  - Simple to maintain: GEOADD when new business added, ZREM when deleted
  - 10M businesses × 20 bytes/member ≈ 200MB → fits in Redis easily

Limitations:
  - No native category/rating filter: GEORADIUS returns all businesses in radius
  → Must post-filter by category/rating in application layer
  → At 5km radius in dense city: might return 10,000 businesses → filter down to 20 restaurants
  → Solution: use category-specific geo indexes:
    GEOADD businesses:restaurant {lng} {lat} {business_id}
    GEOADD businesses:coffee {lng} {lat} {business_id}
    → One sorted set per category → GEORADIUS on category-specific set
    → 10 categories × 200MB = 2GB total → still fits in Redis cluster

Recommendation: Redis GEO for primary fast lookup (simple radius + category).
               PostGIS for complex queries (polygon search, multi-criteria spatial join).
               Elasticsearch for full-text search on business name + geo.
```

**Option E: S2 Geometry Library (Google's approach)**
```
S2: hierarchical decomposition of the Earth's surface into cells
S2CellId: 64-bit integer encoding position and level
  Level 12: ~0.5km cell diameter
  Level 15: ~60m cell diameter

Advantages over Geohash:
  - No distortion at poles (geohash cells distort near poles)
  - Cells are nearly square at all levels (geohash cells are rectangular)
  - Exact coverage: S2Region.GetCovering(circle) returns minimal set of cells
    covering the circle exactly (not a bounding square of cells)
  - Used by Google Maps, Foursquare, MongoDB geo indexes

Implementation in Redis:
  Each S2 cell at level 12 mapped to a sorted set of business_ids
  Query: compute covering cells for search circle, SUNIONSTORE 9 cells, sort by distance
  Same concept as geohash but with better geometric properties
```

**G5:** All geospatial index approaches have the same boundary problem: cells at the edge of the search radius may contain businesses that are actually outside the radius. The post-filter step (compute exact Haversine distance) handles this correctly. But there is a less obvious problem with the "8 neighbors" approach for Geohash: when the search center is near a cell corner, the relevant businesses may be in cells that are diagonal neighbors, not edge neighbors. The standard 3×3 grid (9 cells) handles this correctly. However, for very large search radii (50km), a single precision-level cell is too small — the search spans many more cells, making the 9-cell approach inaccurate. The correct solution is to choose the geohash precision based on the search radius at query time: `precision = max(1, 5 - floor(log2(radius_km)))`, and expand the neighbor set accordingly. Build this radius-to-precision mapping into the geospatial query library so callers don't need to manually choose precision.

## Deep Dive 2: Rating Aggregation

**Problem:** Updating `avg_rating` on every review write creates contention.
```
Scenario: popular restaurant gets 100 new reviews/minute
  100 concurrent: UPDATE businesses SET avg_rating = ... WHERE id = ...
  → Row-level lock contention in PostgreSQL
  → Serialized writes → each waits for the previous → p99 write latency balloons

More precisely: 58 reviews/sec globally, but viral/controversial businesses
  can get 10-50 reviews/sec momentarily (one-star bombing after bad press)
```

**Async aggregation pipeline:**
```
Step 1: Review written to Cassandra (append-only, no update → no lock)

Step 2: Kafka event published:
  { business_id, review_id, rating, user_id, created_at }

Step 3: Rating aggregator service (single consumer per business partition):
  Consume from Kafka topic "review-created" partitioned by business_id
  → same business always processed by same consumer → no race condition

  Per business, maintain running state:
    running_sum += new_rating
    running_count += 1
    new_avg = running_sum / running_count

  UPDATE business_ratings
  SET avg_rating = {new_avg},
      review_count = {running_count},
      sum_rating = {running_sum},
      last_updated = now()
  WHERE business_id = {business_id}

  Redis SET biz:rating:{business_id} {new_avg} EX 60
  → Business detail API reads from Redis cache (< 1ms) instead of PostgreSQL
  → Redis updated synchronously by aggregator; PostgreSQL updated in same transaction

Step 4: Propagate to businesses table (denormalized copy, for search results):
  Every 30 seconds: batch UPDATE businesses SET avg_rating = ... for changed businesses
  → Geo index and search cache will reflect new rating within 30s
```

**Streaming running average (Welford's algorithm):**
```python
def update_average(old_avg: float, new_count: int, new_rating: int) -> float:
    """
    Welford's online algorithm: computes running average without storing sum.
    Numerically stable (no catastrophic cancellation) even for large counts.

    new_avg = old_avg + (new_rating - old_avg) / new_count
    """
    return old_avg + (new_rating - old_avg) / new_count

# Example:
# current: avg=4.2, count=100
# new review: rating=2
# new_avg = 4.2 + (2 - 4.2) / 101 = 4.2 - 0.0218 = 4.178
```

Why Welford's instead of `sum / count`?
- No need to store the running sum (saves a column)
- Numerically stable for large counts where sum could overflow float precision
- Works correctly with concurrent updates + optimistic locking (retry on conflict)

**G5:** The async aggregation pipeline handles steady-state review rates well, but fails under the "Yelp bomb" attack: a business receives 10,000 1-star reviews in 10 minutes from coordinated fake accounts (a real phenomenon targeting controversial businesses). The Kafka consumer processes these sequentially, and `avg_rating` drops dramatically in near-real-time, visible to all users. The platform should rate-limit rating updates to the aggregated display: debounce the rating update to at most once per minute per business (not once per review). More importantly, flag anomalous review velocity (10K reviews in 10 min vs a baseline of 10/day → 1000× spike) and pause displaying rating changes for that business until human review. The displayed `avg_rating` should lag the aggregate during suspected bombing events — not hide the bad reviews, but delay the visual score impact until legitimacy is verified. This is a product policy decision, not just a systems one, but the system must be able to implement it.

## Reliability and Operations

- **Search cache:** Cache popular search queries (city="San Francisco", category="restaurant", sort="rating") with 60-second TTL. At 50K QPS, even a 50% cache hit rate reduces geo index load by 25K QPS. Cache key must include all filter parameters — a missed parameter causes stale results.
- **Review fraud:** Rate limit: 1 review per user per business (enforced by unique constraint on `(user_id, business_id)` in a separate table). New accounts (< 7 days old): review is hidden from public display until ML model clears it. Velocity detection: if same user reviews > 5 businesses in 1 hour, flag for review. IP clustering: multiple accounts from same IP creating reviews for same business → suspect coordinated review manipulation.
- **Business data freshness:** Business owners can edit hours, phone, address via dashboard. Changes: UPDATE PostgreSQL → invalidate Redis cache (DEL biz:{business_id}) → Redis rebuilt on next request. No geo index update needed (lat/lng never changes for a business). Batch staleness check: nightly cron compares scraped data (Google Maps API) vs stored data for top 10K businesses → alert if discrepancy.

## Tradeoffs

| Decision | Alternative | Why this choice |
|----------|-------------|-----------------|
| Redis GEO (primary) | PostGIS | Redis: in-memory → sub-millisecond queries; entire 10M business dataset fits in 200MB. PostGIS: richer spatial query language (polygons, within, touches) but requires a DB round trip (2–5ms). Use Redis for the 95% case; PostGIS for complex shapes and polygon-based searches. |
| Category-specific geo indexes | Single index + post-filter | Single index returns all categories then filters in app: in dense NYC, a 5km radius might return 100K businesses → post-filtering 100K records per query is expensive. Category-specific index: each GEORADIUS result is already in the right category → typically 1–5K results → manageable post-filter for rating/price. |
| Cassandra for reviews | PostgreSQL | Cassandra: partition by business_id gives efficient "get reviews for business X" queries; write-optimized for the 5M/day review volume. PostgreSQL: richer queries (full-text search via `pg_trgm`, complex joins) but requires careful indexing; horizontal sharding is harder without a framework. Choose Cassandra when you know your access pattern is partition-key-based reads/writes. |
| Async rating aggregation | Synchronous `UPDATE` per review | Synchronous: always accurate; simpler code. Async: avoids lock contention on hot rows; 30-second eventual consistency is product-acceptable. The synchronous approach would serialize at 10 reviews/sec for popular businesses — manageable, but at 100/sec (review bombing) it degrades. Design for the adversarial case. |

## G5

**What a G5 engineer adds beyond the above:**

1. **Personalized ranking beyond distance and rating:** The naive search result ordering (distance × rating) ignores the individual user. Two signals lift results significantly: (a) Social proof — if 3 friends have reviewed a restaurant, it ranks higher for you (friends' affinity scores, stored in a feature store). (b) Cuisine/price affinity — if a user consistently rates Italian food 5 stars and burgers 3 stars, Italian restaurants rank higher in their results, all else equal. Architecture: candidate generation from Redis GEO (100 businesses in radius) → affinity scoring from feature store (user × category affinity, pre-computed nightly) → final ranking model (LightGBM, < 5ms) → return top 20. Degrade gracefully to rating × distance if ranking service is slow.

2. **Geospatial sharding for global scale:** At 10M businesses in one country, everything fits in a single Redis node. At 1B businesses globally, a single Redis instance is insufficient (200GB just for the geo index). Shard by region: US-West, US-East, EU, APAC, LATAM. Route requests by user's geographic region to the correct shard. Cross-shard queries (user in London searching for Starbucks globally) are a special case — route to all relevant shards, merge results. Business chains (multiple locations globally): write to all relevant regional shards. Global search for a chain by name → Elasticsearch full-text index (geographically independent).

3. **Real-time "Popular Right Now" signal:** Detecting trending venues from foot traffic signals. Kafka → Flink with 30-minute sliding windows → count reviews, check-ins, and "was here" signals per business. Compare to the same business's 30-day historical baseline at this time of day (same hour, same day of week). If current velocity is 5× above baseline → surface "Popular Right Now" badge in search results. This requires a per-business time-series of historical activity — store in Redis sorted sets by day-of-week + hour-of-day bucket. The badge also helps detect negative events: if 1-star review velocity spikes 10× above baseline → alert content team and potentially surface a "Recently flagged reviews" notice.

4. **Offline-first mobile experience:** Users searching for restaurants in low-connectivity areas (subway, rural travel) expect the app to work. Solution: progressive local caching strategy. On app open: fetch and cache the top 500 businesses for the user's current city (most-reviewed, highest-rated) into local SQLite DB. Geo data is compact: 500 businesses × 2KB = 1MB download. Allow full search/detail browsing from local cache when offline, with a "Last synced: 2 hours ago" indicator. Uploads queued reviews when connection restores (draft stored locally with full review text and rating). On reconnect: sync queue → server, then refresh local cache. This architecture also speeds up the common case (cached results appear instantly; server response updates them in the background).

5. **Review quality scoring and "Most Relevant" sort:** Chronological review order surfaces the most recent reviews, which may be low-effort one-liners from new accounts. The "Most Relevant" sort (default on Yelp) uses a quality score: `score = recency_weight × quality_weight × helpfulness_weight`. Recency weight decays exponentially with age (half-life = 6 months). Quality weight: review length (longer = more weight, up to 500 chars), media attached (photos = 2× weight), reviewer reputation (verified Yelp Elite = 3× weight, first review ever = 0.5× weight). Helpfulness weight: helpful_votes count (logarithmic scale). Pre-compute scores async after each new helpful vote or after each daily reputation update. Store pre-computed score in Cassandra review row — avoid computing at read time. Re-rank all reviews for a business when reputation scores change (nightly batch job).
