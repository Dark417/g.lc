# 27. Payment System

**Type:** data
**Difficulty:** L5
**Frequency:** High — financial correctness, idempotency, distributed transactions

---

## Problem Statement

Design a payment processing system for an e-commerce platform. Users can add payment methods, initiate payments for orders, and receive refunds. The system must be financially accurate (never lose money or double-charge), handle payment processor failures gracefully, and maintain complete audit trails for compliance.

---

## Functional Requirements

**Must have:**
- Add/remove payment methods (card tokenization via Stripe/Braintree)
- Charge a payment method for an order amount
- Issue full or partial refunds
- Payment status lifecycle: pending → processing → succeeded/failed
- Idempotent retries: same payment request retried multiple times → charged exactly once

**Should have:**
- Multiple payment methods: credit card, ACH, wallets
- Currency conversion for international payments
- Automatic retry with exponential backoff for transient failures
- Fraud scoring before charging

**Out of scope:**
- Implementing a card network (Visa/Mastercard protocols)
- Building own PSP (payment service provider)
- Cryptocurrency payments

---

## Non-Functional Requirements

- **Scale:** 1M payments/day; 100M$/day processed
- **Availability:** 99.999% — even 5 min downtime = significant revenue loss
- **Financial accuracy:** Exactly-once charging; zero double charges
- **Durability:** Zero payment data loss; 7-year audit retention
- **Latency:** Payment confirmation < 2 seconds
- **Compliance:** PCI-DSS Level 1, SOX audit trail

---

## Capacity Estimation

```
Payments: 1M/day = 12/sec average; peak (Black Friday): 500/sec
Refunds: 5% of payments = 50K/day

Storage:
  Payments: 1M × 1KB = 1GB/day; 7 years = 2.5TB (manageable)
  Audit events: 10 events per payment × 1M = 10M events/day × 500B = 5GB/day

For $100M/day: average transaction = $100
→ At peak 500/sec: $50K/sec flowing through the system
→ Financial correctness is non-negotiable
```

```
# Create payment
POST /v1/payments
Headers: Idempotency-Key: {client_generated_uuid}
```

## Deep Dive 1: Idempotency — Exactly Once Charging

**The critical problem:** Network is unreliable. Client may retry a payment request if the response is lost. Without idempotency: card charged twice.

**Idempotency key protocol:**
```
Client generates UUID: idempotency_key = UUID()
Client sends: POST /payments with Idempotency-Key header

Server:
1. Check: SELECT * FROM payments WHERE idempotency_key = ?
   - If found AND status = 'succeeded': return the existing payment (don't charge again)
   - If found AND status = 'processing': wait and return current status
   - If not found: proceed to charge

2. Insert payment record (pending):
   INSERT INTO payments (idempotency_key, ..., status='pending')
   ON CONFLICT (idempotency_key) DO NOTHING  -- handles concurrent duplicate requests

3. Call PSP
4. Update payment status based on PSP response
5. Return payment to client

Key properties:
  - Idempotency key scoped to: customer_id + key (not just key alone)
  - Idempotency key stored permanently (7 years) — never reused
  - If PSP charges but our DB write fails: reconciliation detects and resolves
```

**Two-phase commit with PSP (not possible — PSPs don't support 2PC):**
```
The fundamental problem:
  PSP charges card → our DB write fails → we think payment failed, but money was taken!

Solution: reconciliation-based recovery
  1. PSP charge succeeds → write to DB (with retry)
  2. If DB write still fails after retries: alert ops
  3. Daily reconciliation: compare our DB vs PSP statements
  4. Discrepancy → investigate and manually reconcile
  5. For customers: timeout → customer sees "payment status unknown" → check order status
```

## Deep Dive 2: PSP Integration and Fallback

**Multi-PSP architecture:**
```
Primary: Stripe (primary for card payments)
Fallback: Braintree (fallback if Stripe is down)
Backup: Adyen (geographic fallback; best for EU/APAC)

Routing logic:
  1. Default: try Stripe
  2. Stripe timeout (> 3s) or error: retry on Braintree
  3. Both fail: queue payment for retry (transient failure) or reject (persistent failure)

PSP selection also based on:
  - Currency (some PSPs are better for certain currencies)
  - Card type (AmEx is better on some PSPs)
  - Geography (EU cards prefer EU-based PSP for lower fees)
```

**PSP adapter pattern:**
```python
class PSPAdapter:
    def charge(self, amount: int, currency: str, token: str, idempotency_key: str) -> PSPResponse:
        raise NotImplementedError

class StripeAdapter(PSPAdapter):
    def charge(self, amount, currency, token, idempotency_key):
        return stripe.PaymentIntent.create(
            amount=amount,
            currency=currency,
            payment_method=token,
            idempotency_key=idempotency_key,
            confirm=True
        )

class BraintreeAdapter(PSPAdapter):
    def charge(self, amount, currency, token, idempotency_key):
        return braintree.Transaction.sale({
            "amount": amount / 100,
            "payment_method_nonce": token,
            # Braintree uses its own idempotency via duplicate checking
        })
```

## Deep Dive 3: Reconciliation

**Why reconciliation is critical:**
- PSP reports every transaction in their settlement file (downloaded daily)
- Our DB must match exactly
- Discrepancies indicate: bugs, failed DB writes, PSP errors, fraud

**Daily reconciliation process:**
```
1. Download PSP settlement file (CSV with all transactions for yesterday)
2. Parse and load into reconciliation table
3. JOIN with our payments table on psp_transaction_id
4. Classify discrepancies:
   - In PSP but not in our DB (psp_only): possible data loss — investigate
   - In our DB but not in PSP (db_only): charge recorded but not settled — may be void
   - Amount mismatch: refund not reflected, currency conversion error
5. Alert on any psp_only entries > $0
6. Auto-resolve db_only if payment is pending and > 7 days old → mark as failed
```

---

## Reliability and Operations

- **DB failure during payment:** Payment in 'pending' status. On DB recovery, system re-checks PSP for status. Reconciliation catches any discrepancies.
- **PSP timeouts:** Treat timeout as "unknown" (not failure). Retry with same idempotency_key. PSP will return existing result if charge already went through.
- **Stuck payments:** Background job: payments in 'processing' for > 5 minutes → query PSP for status → update our DB accordingly.
- **PCI compliance:** Card numbers never touch our servers (Stripe.js collects and tokenizes directly). We store only PSP tokens. All payment data encrypted at rest. Access logs required for all reads of payment data.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Idempotency via DB unique key | Application-level check | DB unique: atomic and correct; App-level: race condition possible |
| Reconciliation-based recovery | 2PC | Reconciliation: practical with PSPs; 2PC: PSPs don't support it |
| Amount in cents (integer) | Decimal/float | Integer: no floating point errors; Decimal: more natural but requires careful handling |
| Append-only audit log | Updateable status table | Append-only: complete history, tamper-evident; Updateable: simpler but lossy |

## L5 Additions

1. **Distributed saga for payment + fulfillment:** Payment success must trigger fulfillment (ship the item). Both must succeed or both must roll back. Saga orchestration: (1) charge payment → (2) create shipment → (3) send confirmation. If step 2 fails: execute compensating transaction (refund). Architecture: a Saga orchestrator service manages state machine. Each step is a separate service call with retry. State stored in DB — survives restarts. This is more reliable than 2PC and supports long-running transactions.

2. **Smart retry logic:** Not all PSP failures are retryable. Permanent failures: "insufficient funds", "card blocked", "fraud detected" — don't retry. Retryable: "connection timeout", "service unavailable", "rate limited". Architecture: PSP error code classification map. Retry with exponential backoff only for retryable errors. User-facing retry: for "insufficient funds", prompt user to update payment method rather than auto-retrying.

3. **Real-time fraud scoring:** Before every charge, score the transaction. Signals: device fingerprint, IP reputation, velocity (how many charges from this IP in last hour), transaction amount vs historical pattern, card BIN (first 6 digits, indicates card type and bank). ML model scores in < 50ms. If score > threshold: decline. If medium risk: 3D Secure challenge (bank SMS OTP). This reduces chargebacks (fraudulent transactions disputed by cardholder) from ~1% to < 0.1%.

4. **Global payment routing and currency:** International payments add complexity: FX conversion, local payment methods (SEPA in EU, UPI in India, Alipay in China). Architecture: currency service determines settlement currency. FX rate locked at time of charge (mid-market rate + small spread for hedging). Payment method routing: geo-detect user → show relevant payment methods (Alipay only for CN, UPI only for IN). PSP selection by geography: Stripe is best for US/EU card; Razorpay for India UPI; WePay for Alipay. PSP router abstracts this complexity.

5. **Double-entry ledger for financial integrity:** Every payment movement is recorded as a double-entry ledger entry: debit customer's liability account, credit merchant's receivable account. Balance equation: sum of all debits = sum of all credits. Any discrepancy in the ledger balance is a bug. Architecture: ledger service is strictly append-only. Reconciliation: periodically compute ledger balance per account → alert if not zero. This is how banks maintain financial integrity and how Stripe / Square / Brex audit their books.
