# 27. Payment System

**Type:** data
**Difficulty:** G5
**Frequency:** High — financial correctness, idempotency, distributed transactions

---

## Problem Statement

Design a payment processing system for an e-commerce platform. Users can add payment methods, initiate payments for orders, and receive refunds. The system must be financially accurate (never lose money or double-charge), handle payment processor failures gracefully, and maintain complete audit trails for compliance.

---

## Functional Requirements

**Must have:**
- Add/remove payment methods (card tokenization via Stripe/Braintree)
- Charge a payment method for an order amount
- Issue full or partial refunds
- Payment status lifecycle: pending → processing → succeeded/failed
- Idempotent retries: same payment request retried multiple times → charged exactly once

**Should have:**
- Multiple payment methods: credit card, ACH, wallets
- Currency conversion for international payments
- Automatic retry with exponential backoff for transient failures
- Fraud scoring before charging

**Out of scope:**
- Implementing a card network (Visa/Mastercard protocols)
- Building own PSP (payment service provider)
- Cryptocurrency payments

---

## Non-Functional Requirements

- **Scale:** 1M payments/day; $100M/day processed
- **Availability:** 99.999% — even 5 min downtime = significant revenue loss
- **Financial accuracy:** Exactly-once charging; zero double charges
- **Durability:** Zero payment data loss; 7-year audit retention
- **Latency:** Payment confirmation < 2 seconds
- **Compliance:** PCI-DSS Level 1, SOX audit trail

```
  Payments: 1M/day × 2KB per record = 2GB/day

  Average: 11.6 payments/sec × 15 DB events/payment = 174 writes/sec → trivial for PostgreSQL
  Peak: 500/sec × 15 = 7,500 writes/sec → may need read replicas for reporting queries
  Use single primary writer (ACID required); read replicas for dashboards only
```

```
  ▼
  [Payment Orchestrator]
  │

  │

  [Payment DB (PostgreSQL, ACID transactions)]

  ├──→ [Event Publisher (Kafka)] ──→ [Order Service, Notification Service]

  [Ledger Balancer]         — periodic sum verification: credits = debits
```

## Deep Dive 1: Idempotency — Exactly Once Charging

**The critical problem:** Networks are unreliable. A client sends a payment request; our server calls the PSP; the PSP charges the card; but the network drops before the PSP's success response reaches our server. Our server times out and returns an error. The client retries. Without idempotency: card charged twice.

**Full idempotency protocol:**
```
Client generates UUID before the request: idempotency_key = UUID4()
Client sends: POST /v1/payments with Idempotency-Key: {key} header

Server (complete protocol):

Step 1: Check for existing record
  SELECT payment_id, status, response_json
  FROM payments
  WHERE idempotency_key = ?

  Case A: found, status = 'succeeded' or 'failed'
    → Return cached response immediately (same HTTP status code as original)
    → Do NOT call PSP again
    → This is the key guarantee: idempotent replay returns identical result

  Case B: found, status = 'processing'
    → Payment is in-flight (original request still processing, or crashed mid-flight)
    → Return 409 Conflict with Retry-After: 5
    → Client retries after 5 seconds

  Case C: not found → proceed to Step 2

Step 2: Create payment record (atomic lock-in)
  INSERT INTO payments (idempotency_key, order_id, amount, status='pending', ...)
  ON CONFLICT (idempotency_key) DO NOTHING
  RETURNING payment_id;
  -- ON CONFLICT handles race condition: two concurrent requests with same key
  -- Only one INSERT wins; the other gets NULL from RETURNING → re-queries and returns cached response

Step 3: Transition to processing
  UPDATE payments SET status = 'processing' WHERE payment_id = ?;

Step 4: Call PSP (outside DB transaction)
  response = psp_adapter.charge(
      amount=payment.amount,
      currency=payment.currency,
      token=payment.payment_method.psp_token,
      idempotency_key=payment.payment_id.str()  // use our payment_id as PSP idempotency key
  )
  -- PSP also stores our idempotency key; if we call twice with same key → same result

Step 5: Update DB with PSP outcome (atomic)
  BEGIN;
    UPDATE payments
    SET status = (response.success ? 'succeeded' : 'failed'),
        psp_payment_id = response.transaction_id,
        failure_reason = response.error_message,
        updated_at = NOW();
    INSERT INTO payment_events (payment_id, event_type, event_data)
    VALUES (?, 'psp_response_received', response.to_json());
    INSERT INTO ledger_entries (...);  -- if succeeded
  COMMIT;
```

**The unsolvable problem — PSP charges but DB write fails:**
```
Failure scenario:
  PSP successfully charges card → response = success
  Network drops between PSP response and our DB write
  Our server crashes; DB never updated
  Status: payment is 'processing' in our DB; card is charged; customer's money is gone

This is NOT solvable with idempotency alone (requires distributed 2PC, which PSPs don't support).

Mitigation strategy:
  1. Retry DB write with exponential backoff (handles transient network failure)
  2. "Stuck payment" detector: scan for payments in 'processing' > 5 minutes
     For each stuck payment: query PSP with our idempotency_key → PSP returns the result
     Update our DB with PSP's authoritative status
  3. Daily reconciliation (see Deep Dive 3): catches any remaining discrepancies
  4. SLA: resolve stuck payments within 10 minutes → charge never > 10 min "in limbo"
```

**G5:** Idempotency at the API layer (same request key → same response, no double-charge) is only half of the problem. The other half is idempotency at the PSP layer: PSP charges can fail after the HTTP call succeeds — the charge went through on the PSP's side but the HTTP response was lost (our server never received it). In this case our DB still shows 'processing', but the card has already been charged. When we retry with the SAME idempotency key to the PSP, Stripe/Braintree returns the result of the original charge (not a new charge) — this is the PSP idempotency guarantee. But if our retry uses a DIFFERENT idempotency key (e.g., system generated a new one on restart), the PSP creates a second charge. The fix: use the payment_id (a stable DB identifier that survives server restarts) as the PSP idempotency key, not a request-level session ID. This requires the payment_id to be generated in Step 2 (before the PSP call) and stored durably, which is why the two-step INSERT then UPDATE pattern above is critical.

## Deep Dive 2: PSP Integration and Fallback

**Multi-PSP architecture for resilience:**
```
Primary PSP:  Stripe (best for US/EU card payments; excellent API)
Fallback PSP: Braintree (PayPal's gateway; good geographic coverage)
Regional PSP: Adyen (best for EU/APAC; lower fees for international cards)

Routing rules:
  1. Default: try Stripe
  2. Stripe timeout (> 3s) or HTTP 5xx: immediately retry on Braintree
     Use same payment_id as Braintree idempotency key → safe retry
  3. Both fail with transient errors: queue for async retry (wait 1 min, retry up to 5×)
  4. Both fail with permanent errors (insufficient funds, fraud block): return failure to client

PSP selection also considers:
  - Currency: EUR payments → prefer Adyen (better conversion rates, lower FX fees)
  - Card type: Amex → Braintree (better Amex acceptance rates)
  - Geography: IN/MY/SG transactions → Adyen (local acquiring, lower fees)
  - Real-time PSP health: if Stripe error rate > 2% → temporarily route all traffic to Braintree
    (checked every 30s via health oracle service)
```

**PSP Adapter Pattern:**
```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class PSPChargeRequest:
    amount_cents: int
    currency: str
    payment_token: str           # PSP-specific tokenized card reference
    idempotency_key: str         # MUST be stable across retries
    description: str
    customer_email: Optional[str] = None
    metadata: dict = None

@dataclass
class PSPChargeResponse:
    success: bool
    psp_transaction_id: Optional[str] = None
    psp_charge_id: Optional[str] = None
    error_code: Optional[str] = None
    error_message: Optional[str] = None
    is_retryable: bool = False   # True: transient; False: permanent failure

class PSPAdapter(ABC):
    @abstractmethod
    def charge(self, req: PSPChargeRequest) -> PSPChargeResponse:
        pass

class StripeAdapter(PSPAdapter):
    # Non-retryable codes (card declined, fraud blocked, etc.)
    PERMANENT_ERROR_CODES = {
        'card_declined', 'insufficient_funds', 'expired_card',
        'incorrect_cvc', 'card_not_supported', 'do_not_honor',
        'fraudulent', 'pickup_card', 'lost_card', 'stolen_card'
    }

    def charge(self, req: PSPChargeRequest) -> PSPChargeResponse:
        try:
            intent = stripe.PaymentIntent.create(
                amount=req.amount_cents,
                currency=req.currency,
                payment_method=req.payment_token,
                confirm=True,
                idempotency_key=req.idempotency_key,
                metadata=req.metadata or {},
            )
            return PSPChargeResponse(
                success=True,
                psp_transaction_id=intent.id,
                psp_charge_id=intent.latest_charge,
            )
        except stripe.error.CardError as e:
            return PSPChargeResponse(
                success=False,
                error_code=e.code,
                error_message=e.user_message,
                is_retryable=(e.code not in self.PERMANENT_ERROR_CODES),
            )
        except (stripe.error.APIConnectionError, stripe.error.APIError) as e:
            return PSPChargeResponse(
                success=False, error_message=str(e), is_retryable=True
            )
```

**G5:** Distinguishing between idempotency at the API layer (same request key → same response, no double-charge) and idempotency at the PSP layer (PSP charges can fail after the HTTP call succeeds — the charge went through but the response was lost) is critical. The latter requires a reconciliation loop: periodically query PSP for charges in the last N hours, match against your DB on `psp_payment_id`, and surface any PSP charges without a corresponding confirmed payment record. This is how Stripe's own payment infrastructure works — their reconciliation API (`/v1/charges?created[gte]=...`) is specifically designed for this use case. Without a reconciliation loop, a PSP call that succeeds but whose response is lost will result in a charged card with no record in your system — the customer is charged but the order is not fulfilled, leading to chargebacks and revenue loss.

## Deep Dive 3: Reconciliation

**Why reconciliation is non-negotiable:**
- PSPs send daily settlement files listing every transaction they processed
- Our DB must match exactly — every charged amount, every refund
- Discrepancies are bugs: data loss, double-charging, or PSP errors
- Required for SOX (Sarbanes-Oxley) compliance and financial audits

**Daily reconciliation process:**
```python
def daily_reconciliation(date: date, psp_name: str) -> ReconciliationReport:
    """
    Compare our payment DB records against PSP settlement file for the given date.
    Runs at 8 AM UTC for previous day's transactions (settlement file available by 6 AM).
    """

    # 1. Download PSP settlement file
    settlement_file = psp_connector.download_settlement(date=date, psp=psp_name)
    # CSV format: psp_transaction_id, amount_cents, currency, status, created_at, ...

    # 2. Load into temporary reconciliation table
    db.bulk_insert("reconciliation_staging", settlement_file.parse_rows())

    # 3. Three-way join analysis
    result = db.query("""
        -- Transactions in our DB (within date range)
        WITH our_payments AS (
          SELECT psp_payment_id, amount, currency, status
          FROM payments
          WHERE psp_name = :psp AND DATE(created_at) = :date
        ),
        -- Transactions in PSP settlement file
        psp_transactions AS (
          SELECT psp_transaction_id, amount, currency, status
          FROM reconciliation_staging
          WHERE import_date = :date
        )
        SELECT
          COALESCE(p.psp_payment_id, s.psp_transaction_id) AS transaction_id,
          p.amount AS our_amount,
          s.amount AS psp_amount,
          p.status AS our_status,
          s.status AS psp_status,
          CASE
            WHEN p.psp_payment_id IS NULL THEN 'psp_only'     -- CRITICAL: charged but no DB record
            WHEN s.psp_transaction_id IS NULL THEN 'db_only'  -- in DB but PSP didn't process
            WHEN p.amount != s.amount THEN 'amount_mismatch'
            WHEN p.status != s.status THEN 'status_mismatch'
            ELSE 'matched'
          END AS reconciliation_status
        FROM our_payments p
        FULL OUTER JOIN psp_transactions s
          ON p.psp_payment_id = s.psp_transaction_id
    """, psp=psp_name, date=date)

    # 4. Alert on critical discrepancies
    psp_only = [r for r in result if r.reconciliation_status == 'psp_only']
    if psp_only:
        alert_critical(f"PSP has {len(psp_only)} charges with no DB record! "
                       f"Total: ${sum(r.psp_amount for r in psp_only) / 100:.2f}")
        # Customer was charged but order not marked as paid → manual investigation + refund

    db_only = [r for r in result if r.reconciliation_status == 'db_only']
    # May be legitimate (void/void-on-timeout cases) or data loss — investigate each

    return ReconciliationReport(
        date=date, psp=psp_name,
        total_matched=len([r for r in result if r.reconciliation_status == 'matched']),
        psp_only_count=len(psp_only),
        db_only_count=len(db_only),
        discrepancy_pct=len(psp_only) / len(result) * 100
    )
```

## Reliability and Operations

- **DB failure during payment (mid-flight):** Payment in 'processing' state. Stuck payment detector scans every 5 minutes for `status='processing' AND updated_at < NOW() - INTERVAL '5 minutes'`. For each stuck payment: query PSP with our payment_id as idempotency key → PSP returns definitive status → update our DB. This resolves stuck payments within 10 minutes.
- **PSP timeout:** Do NOT classify a timeout as a failed charge. The charge may have gone through. Treat timeout as "unknown" status → stuck payment detector will query PSP for definitive answer. Never refund a timeout immediately — always verify with PSP first.
- **PCI-DSS compliance:** Card numbers NEVER touch our servers. Stripe.js (client-side JavaScript) collects card details and tokenizes directly with Stripe — only a PSP token arrives at our backend. We store only the PSP token (a non-sensitive reference). All payment DB tables encrypted at rest (AES-256). All access to payment records logged. Annual PCI audit: penetration testing, access control review, network segmentation verification.
- **Chargebacks:** Customer disputes charge with their bank → bank issues chargeback → PSP deducts funds from our account. Handling: fraud team reviews chargeback + collects evidence (order confirmation, delivery proof, IP logs) → submits dispute to PSP within 7 days. Win rate: ~60–70% for legitimate charges with good evidence. Chargeback rate > 1% → Visa/Mastercard place merchant on monitoring (risk of losing payment processing).

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Idempotency via DB UNIQUE constraint | Application-level check (SELECT then INSERT) | DB UNIQUE: atomic, race-condition-proof; App-level: SELECT-then-INSERT has TOCTOU race: two concurrent requests both see "not exists" and both attempt to INSERT |
| Reconciliation-based recovery from PSP/DB inconsistency | Two-phase commit | Reconciliation: practical with all real PSPs (none support 2PC); 2PC: theoretically correct but PSPs would need to implement it — they don't |
| Amount in cents (integer) | DECIMAL(19,4) or float | Integer: no floating point representation errors ($9.99 = 999 cents, exact); DECIMAL: correct but requires application discipline; float: NEVER use for money (0.1 + 0.2 ≠ 0.3 in float arithmetic) |
| Append-only audit log (payment_events) | Update status fields in payments table only | Append-only: complete history, tamper-evident, supports forensics, satisfies SOX; Update-only: simpler but no history of intermediate states or PSP request/response details |
| Multi-PSP (Stripe primary, Braintree fallback) | Single PSP | Multi-PSP: resilience against PSP outage, geographic optimization, fee negotiation leverage; Single: simpler integration, one reconciliation process, but hard dependency on vendor SLA |

## G5

1. **Distributed saga for payment + fulfillment coordination:** Payment success must atomically trigger fulfillment (reserve inventory, create shipment). Both must succeed or both must roll back — but distributed 2PC is not viable across service boundaries. Saga pattern (orchestration-based): a Saga Orchestrator service manages the state machine: (1) charge payment → (2) reserve inventory → (3) create shipment → (4) send confirmation email. If step 2 fails: execute compensating transaction (refund the charge). If step 3 fails: release inventory reservation AND refund. Each step is a separate service call with idempotency keys. The saga state machine is persisted in DB at every step — survives restarts. Temporal.io or AWS Step Functions implement this pattern. Key correctness requirement: compensating transactions must themselves be idempotent and always-succeed (a refund that fails is a severe problem — handle with dead letter queue and ops escalation).

2. **Smart PSP error classification:** Not all PSP failures are retryable. Permanent failures: "insufficient_funds", "card_blocked", "fraud_detected", "do_not_honor" — retrying will fail again and waste time. Retryable failures: "connection_timeout", "service_unavailable", "rate_limited", "processing_error". Architecture: maintain a PSP error code classification map per PSP (each PSP has different error codes for the same conditions). Classify each error code as: `permanent | retryable | check_psp_before_retry`. For "check before retry": the charge may or may not have gone through — don't retry until the stuck payment detector has queried the PSP for the definitive status. User-facing communication: for permanent failures, prompt user to update payment method immediately (don't retry in background silently — the customer needs to take action).

3. **Real-time fraud scoring pipeline:** Before every charge, score the transaction to detect fraud. Signals (assembled within 50ms): device fingerprint (browser/app signature), IP reputation (known VPN/proxy/TOR exit node), velocity (charges from this IP in last hour, charges to this card in last hour), transaction amount vs customer historical average, card BIN analysis (first 6 digits indicate bank and card type), geographic inconsistency (IP in Germany, billing address in US, shipping to China). ML model: gradient boosted tree (XGBoost), inference < 10ms. If score > 0.8 → reject. If 0.5–0.8 → trigger 3D Secure (bank sends OTP SMS to cardholder). 3D Secure reduces chargeback liability to the bank even if fraud occurs. Expected outcome: reduces fraud chargebacks from ~0.5% to < 0.05% of transaction volume, saving millions at scale.

4. **Global payment routing and multi-currency:** International payments require local payment methods, FX handling, and local acquiring. Architecture: user's locale + payment method availability matrix. Detected country: India → show UPI and cards (no IBAN). China → show Alipay/WeChat Pay (cards rarely used). EU → show SEPA Direct Debit + cards. FX: lock exchange rate at time of charge (mid-market rate + spread for FX risk hedging). Store both original currency (customer_amount, customer_currency) and settled currency (settlement_amount, settlement_currency) in the payments table. Reconciliation must handle multi-currency: compare amounts in settlement currency, not customer currency (FX rates change daily). PSP selection: Stripe for US/EU cards; Razorpay for India UPI; Adyen for Asia-Pacific local methods. PSP router abstracts which PSP handles each payment method.

5. **Double-entry ledger for financial integrity:** Every money movement recorded as a double-entry ledger entry: debit one account, credit another, always equal amounts. For a payment: debit `customer_receivable:customer-123` ($99.99), credit `merchant_payable:merchant-456` ($96.74) + credit `platform_fee_income` ($3.25). For a refund: reverse all entries. Balance equation: sum of all debit amounts = sum of all credit amounts across all entries. Any imbalance is a bug — possibly data corruption or missed ledger entry. Architecture: ledger service is strictly append-only (no UPDATEs, no DELETEs). Periodic balance check: `SELECT SUM(amount) FROM ledger_entries WHERE entry_side='debit'` must equal `SELECT SUM(amount) WHERE entry_side='credit'`. Alert on any imbalance > $0. This is the financial integrity guarantee that Stripe, Square, and every regulated financial institution use to ensure their books are correct.
