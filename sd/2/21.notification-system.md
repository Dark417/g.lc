# 21. Notification System

**Type:** infra
**Difficulty:** G4
**Frequency:** Very High — fan-out, multiple channels, reliability

---

## Problem Statement

Design a notification system that sends push notifications (iOS/Android), SMS, and email to users based on system events. The system must handle 100M notifications/day, support multiple channels with fallback, guarantee at-least-once delivery, and allow users to manage notification preferences.

---

## Functional Requirements

**Must have:**
- Send push notification (iOS APNS, Android FCM)
- Send SMS (Twilio/SNS)
- Send email (SendGrid/SES)
- At-least-once delivery guarantee per channel
- User preference management (opt-in/out per notification type and channel)

**Should have:**
- Priority tiers: critical (OTP, alerts), standard (activity), marketing (promotions)
- Scheduled / delayed notifications
- Notification history per user
- Throttling: don't spam users with too many notifications in a window

**Out of scope:**
- In-app notification inbox (see notification inbox design)
- Two-way messaging / replies
- Rich media in push notifications

---

## Non-Functional Requirements

- **Scale:** 100M notifications/day across all channels; peak 10M/hour
- **Latency:** Critical notifications delivered in < 5 seconds; standard < 30 seconds
- **Availability:** 99.9% — if a channel is down, fall back to another
- **Delivery guarantee:** At-least-once — some duplicates acceptable; no losses
- **Compliance:** GDPR (consent required for marketing), CAN-SPAM (email unsubscribe), TCPA (SMS opt-in)

---

## Capacity Estimation

```
Notifications: 100M/day
  = 100,000,000 / 86,400 = ~1,160/sec average
  Peak (flash sale, major event): ~10,000/sec

Channel breakdown:
  Push:  60M/day = 60% → 700/sec avg
  Email: 30M/day = 30% → 350/sec avg
  SMS:   10M/day = 10% → 115/sec avg

Push (FCM/APNS):
  FCM supports ~250K messages/sec per project → no bottleneck here
  iOS APNS: similar throughput

Email (SES):
  SES sending rate: configurable up to 1M/day on basic, much more on higher tiers
  Cost: $0.10 per 1,000 emails → 30M × $0.10/1000 = $3,000/day

SMS (Twilio):
  Twilio US SMS rate: ~$0.0075/msg → 10M × $0.0075 = $75,000/day
  → SMS is 25× more expensive than email; intelligent channel routing is critical

Storage:
  Notification log: 100M/day × 1KB per record = 100GB/day
  Retention: 30 days → 3TB; use columnar storage (Parquet on S3) for analytics
  Active delivery state: only in-flight notifications need fast access → ~1M records at any time in Redis

Worker sizing:
  Push workers: 700 msgs/sec → with FCM batching (up to 500 per call) = 1.4 FCM calls/sec → trivial
  SMS workers: 115/sec → Twilio rate limit per number; use multiple sender IDs
  Email workers: 350/sec → SES SMTP connection pool, 50 workers sufficient
```

## High-Level Architecture

```
[Source Services (Order, Social, Security, Marketing)]
    │  publish events
    ▼
[Kafka: notification-events]
    │  (partitioned by user_id for ordering)
    ▼
[Notification Orchestrator]
    │
    ├──→ [Preference Service] → check user opt-in, channel preference, quiet hours
    ├──→ [Throttle Service]   → check rate limits per user per type
    │
    ├──→ [Push Queue (Kafka: push-pending)]   ──→ [Push Worker]  ──→ [FCM / APNS]
    ├──→ [Email Queue (Kafka: email-pending)] ──→ [Email Worker] ──→ [SendGrid / SES]
    └──→ [SMS Queue (Kafka: sms-pending)]     ──→ [SMS Worker]   ──→ [Twilio / SNS]
                                                        │
                                               [Delivery Status DB (PostgreSQL)]
                                               [Analytics sink (ClickHouse)]
```

**Key design decisions:**
- Separate Kafka topics per channel: channel-specific failures don't block other channels
- Orchestrator is stateless: reads from Kafka, checks preferences, routes to per-channel topic
- Workers are stateless: pull from Kafka topic, attempt delivery, write result to DB
- Kafka consumer groups allow independent scaling of each channel's worker fleet

## Data Model

```sql
-- PostgreSQL: notifications log
notifications (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          UUID NOT NULL,
  event_type       TEXT NOT NULL,
  channel          TEXT NOT NULL,    -- 'push', 'email', 'sms'
  status           TEXT NOT NULL,    -- 'queued', 'sent', 'delivered', 'failed', 'bounced'
  payload          JSONB NOT NULL,   -- rendered content (title, body, etc.)
  idempotency_key  TEXT UNIQUE NOT NULL,  -- global dedup key
  scheduled_at     TIMESTAMPTZ,
  sent_at          TIMESTAMPTZ,
  delivered_at     TIMESTAMPTZ,
  failed_at        TIMESTAMPTZ,
  error_message    TEXT,
  retry_count      INT DEFAULT 0,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_notifications_user ON notifications (user_id, created_at DESC);
CREATE INDEX idx_notifications_status ON notifications (status, scheduled_at) WHERE status IN ('queued', 'failed');

-- User preferences (read on every notification)
notification_preferences (
  user_id          UUID NOT NULL,
  event_type       TEXT NOT NULL,
  channel          TEXT NOT NULL,   -- 'push', 'email', 'sms'
  enabled          BOOLEAN NOT NULL DEFAULT TRUE,
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, event_type, channel)
);

-- User channel settings (device tokens, contact info)
user_channels (
  user_id          UUID PRIMARY KEY,
  device_tokens    JSONB,    -- [{ token, platform: "ios"|"android", updated_at }]
  email            TEXT,
  phone            TEXT,
  quiet_start      TIME,
  quiet_end        TIME,
  timezone         TEXT DEFAULT 'UTC'
);
```

## Deep Dive 1: Reliable Delivery with Retry

**Problem:** FCM/APNS/Twilio fail transiently. Network timeouts, rate limits, and provider outages must be handled without message loss or excessive duplication.

**Retry strategy per channel:**
```
Push (FCM/APNS):
  Max retries: 3
  Backoff: 1s, 5s, 30s (exponential, jittered)
  Permanent failure conditions:
    - APNS error 410 (device token no longer active) → remove token, do NOT retry
    - FCM error UNREGISTERED → remove token, do NOT retry
    - FCM error SENDER_ID_MISMATCH → configuration error, alert engineer
  Transient failure conditions:
    - 5xx from FCM/APNS → retry with backoff
    - 429 (rate limited) → honor Retry-After header exactly

Email (SES/SendGrid):
  Max retries: 5
  Backoff: 1m, 5m, 15m, 1h, 24h (very conservative — email retries can span days)
  Soft bounce (temporary): mailbox full, server unreachable → retry
  Hard bounce (permanent): bad email address, domain doesn't exist → mark 'bounced', disable email for this address in user_channels
  Unsubscribe: SendGrid webhook fires on unsubscribe → update preferences

SMS (Twilio):
  Max retries: 3
  Backoff: 5s, 30s, 2m
  Invalid number (Twilio error 21211) → do not retry; log as failed
  Carrier filtering (Twilio error 21610) → user has opted out at carrier level → disable SMS
```

**Implementation in worker:**
```python
def process_push_notification(notification_id: str) -> None:
    notification = db.get_notification(notification_id)

    # Idempotency check: already delivered?
    if notification.status == 'delivered':
        return

    try:
        token = user_channels.get_device_token(notification.user_id)
        if not token:
            db.mark_failed(notification_id, reason="no_device_token")
            return

        response = fcm_client.send(
            token=token.token,
            title=notification.payload['title'],
            body=notification.payload['body'],
            data=notification.payload.get('data', {}),
        )
        db.mark_sent(notification_id, provider_msg_id=response.message_id)

    except InvalidTokenError as e:
        user_channels.remove_token(token.token)
        db.mark_failed(notification_id, reason="invalid_token")

    except RateLimitError as e:
        delay = e.retry_after_seconds or exponential_backoff(notification.retry_count)
        requeue_with_delay(notification_id, delay_sec=delay)

    except TransientError as e:
        if notification.retry_count < MAX_PUSH_RETRIES:
            delay = exponential_backoff(notification.retry_count, base=1, multiplier=5)
            db.increment_retry(notification_id)
            requeue_with_delay(notification_id, delay_sec=delay)
        else:
            db.mark_failed(notification_id, reason="max_retries_exceeded")
```

**G5:** The retry strategy above handles provider-side transience, but misses a subtler failure mode: the worker writes "sent" to the DB, but the DB write itself fails (network partition between worker and DB). The notification was delivered to the user's device, but our record says "queued" — so the retry loop sends it again. The correct fix is to write the provider's message ID to the DB *before* considering the send complete, and check for existing provider message IDs at the top of the worker. More precisely: `UPDATE notifications SET status='sent', provider_msg_id=? WHERE id=? AND status='queued'` — the `AND status='queued'` predicate makes the update a no-op if another worker already completed it, giving safe exactly-once semantics under concurrent retries.

## Deep Dive 2: Preference Check and Throttling

**Preference service — fast path:**
```
On each notification entering the Orchestrator:
1. Fetch user preferences (Redis cache, TTL 5 min):
   Key: prefs:{user_id} → serialized preferences JSONB
   On cache miss: load from PostgreSQL, populate cache
   On preference update: invalidate cache key (write-through)

2. For each candidate channel:
   a. Check: is this channel enabled globally? (user.push_enabled)
   b. Check: is this event_type enabled for this channel?
      (notification_preferences WHERE user_id=? AND event_type=? AND channel=?)
   c. Check quiet hours (compare current time in user's timezone):
      - Critical notifications: BYPASS quiet hours always
      - Standard/marketing: if in quiet hours → re-schedule for quiet_end time
   d. Check GDPR consent (for marketing): consent_grants table

3. Check global opt-out list (CAN-SPAM marketing email unsubscribes, TCPA SMS opt-outs):
   Maintain a Redis SET per channel of opted-out users:
   SISMEMBER email_optouts {user_id} → O(1) check
```

**Throttling (sliding window, per user per channel):**
```
Rules:
  - Non-critical push: max 5 per user per hour
  - SMS (any type): max 3 per user per day
  - Marketing email: max 2 per user per week
  - Critical (OTP, security alert): no throttle limit

Redis implementation (Lua script for atomicity):
  local key = "throttle:" .. user_id .. ":push:" .. current_hour
  local count = redis.call("INCR", key)
  redis.call("EXPIRE", key, 7200)  -- 2x window for safety
  if count > 5 then
    return "throttled"
  end
  return "allowed"

On throttle:
  - Marketing: silently drop (record as 'throttled' in notification log)
  - Standard: delay to next available slot (not drop — user wants this)
  - Critical: always deliver regardless
```

**G5:** The per-hour Redis throttle key rolls over sharply at the hour boundary, creating a burst window: a user can receive 5 notifications at 11:59 and 5 more at 12:00 — 10 in two minutes. A true sliding window (track individual timestamps per user in a Redis sorted set, ZRANGEBYSCORE to count within the last 3600 seconds) is 10× more accurate but uses more memory: `ZADD throttle:{user_id}:push {now} {notification_id}` + `ZREMRANGEBYSCORE ... -inf {now-3600}` + `ZCARD`. For push at 1,160/sec across 100M users the sorted set approach is justified; for SMS (115/sec, strict regulatory limit) it is essential — regulators care about the actual 24-hour window, not a calendar-day boundary.

## Deep Dive 3: Template and Content Rendering

**Template engine:**
```
Event type → template selection:
  event_type "order_confirmed" maps to:
    push:  templates/order_confirmed/push.txt
    email: templates/order_confirmed/email.html
    sms:   templates/order_confirmed/sms.txt

Templates stored in S3 (or DB for fast lookup), with variable placeholders:
  push_title: "Your order #{{order_id}} is confirmed!"
  push_body:  "Estimated delivery: {{delivery_date}}. Tap to track."
  email_subject: "Order Confirmation — {{product_name}}"
  email_body: (HTML with product image, order table, tracking link)
  sms_body: "Order #{{order_id}} confirmed. Track: {{short_url}}"

Rendering example (Python / Jinja2):
  payload = {
      "order_id": "ORD-12345",
      "delivery_date": "March 10",
      "product_name": "Sony WH-1000XM5",
      "short_url": "https://ex.co/t/abc"
  }
  rendered_title = Template("Your order #{{order_id}} is confirmed!").render(**payload)
  # → "Your order #ORD-12345 is confirmed!"

Template cache: keep rendered templates in memory (LRU, 1000 templates)
  → template fetching from S3/DB only on cold start or update

i18n: template variants per language code
  Template key: order_confirmed/{channel}/{locale}
  Locale from user profile; fallback to 'en' if locale template missing

SMS length: enforce < 160 chars for single segment; warn if multi-segment (higher cost)
```

**G5:** Template rendering should be idempotent and happen exactly once per notification ID, not once per retry attempt. If the worker re-renders on retry, a time-sensitive payload field (like `{{delivery_date}}`) could produce different content on the second attempt than on the first — or more subtly, a price that changed between attempts. Render the template at orchestration time (before enqueueing), serialize the fully-rendered content into the notification record's `payload` JSONB, and have workers send the pre-rendered content verbatim. This also means retry workers don't need access to the template service at all, reducing the blast radius of a template service outage.

## Reliability and Operations

- **Kafka consumer lag:** Monitor consumer group lag per topic (Kafka JMX or Prometheus kafka-exporter). If lag on `push-pending` grows → scale up push workers. Alert threshold: lag > 10,000 messages or lag growth rate > 1,000/sec.
- **Third-party service SLA:** FCM SLA is "best effort" — no guaranteed delivery. For critical notifications (OTP, security alerts): send via push AND SMS simultaneously. First delivery wins; deduplicate on client using `notification_id` in the payload. Client suppresses the duplicate display.
- **Delivery receipts:** FCM provides async delivery receipts (via Pub/Sub subscription). APNS returns synchronous success/failure; invalid tokens via feedback service. Collect delivery confirmations asynchronously into ClickHouse for analytics. Track: sent vs delivered ratio per channel per notification type.
- **Deduplication:** `idempotency_key` enforced as UNIQUE in PostgreSQL. Workers check before sending: `SELECT id FROM notifications WHERE idempotency_key = ? AND status = 'delivered'`. Also use `INSERT ... ON CONFLICT DO NOTHING` to safely handle concurrent retry races.
- **Monitoring:** Alert on: delivery failure rate > 1% per channel per 5-min window; SMS cost spike (runaway notification loop); Kafka consumer lag > 30s for critical channel; any notification stuck in 'queued' for > 5 minutes.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Separate Kafka topic per channel | Single shared topic | Per-channel: channel failure is isolated, independent scaling, different retention; Single: simpler routing but SMS slowness blocks email |
| At-least-once delivery | Exactly-once | At-least-once: simpler workers, retry on crash; Exactly-once: requires 2PC or idempotent consumers — worth it only for SMS (cost) and OTP (UX) |
| Redis for preferences | DB lookup per notification | Redis: < 1ms with TTL-based invalidation; DB: always fresh but 5–10ms per lookup × 1,160/sec = unsustainable |
| Async orchestrator via Kafka | Synchronous (API call to workers) | Async: decoupled, durable, retry on crash; Sync: lower latency but loses notifications on worker crash |
| Render templates at orchestration time | Render at worker send time | Orchestration-time: content frozen at event time, simpler workers; Worker-time: content fresher but non-deterministic retries |

## G5

1. **Intelligent channel selection:** Don't route to every channel blindly. Learn which channel each user actually engages with using historical engagement signals: push open rate, email click rate, SMS response rate. Build a lightweight logistic regression model per user per event_type: `P(engage | channel, time_of_day, notification_type)`. Route to the channel with the highest predicted engagement that also satisfies cost constraints (push = $0; SMS = $0.0075 — prefer push unless user's push open rate < 5%). A/B test the model: compare engaged notification rates between rule-based routing and ML routing. Expected outcome: 30% SMS cost reduction with equal or better engagement, because many users who receive SMS would have opened push if tried first.

2. **Real-time personalization:** Notification content should be personalized at the payload level. "Hey Alice, your favorite brand Nike just dropped a deal on the Air Max you viewed last week" vs. generic "New deal available." Architecture: a personalization service receives event + user_id → fetches user context from feature store (recently viewed, stated preferences, price range) → fills a richer template. Latency budget: personalization must complete within 200ms of the orchestrator receiving the event (within the critical notification path). Cache user preference profiles in Redis with 10-minute TTL; pre-warm on user login. For marketing emails (non-latency-sensitive), personalization can be heavier — run offline and inject into the scheduled notification payload at render time.

3. **Batch notifications for low-priority:** Social interaction notifications (likes, follows, comments) should not individually push every event. Digest batching: accumulate social notifications for a user in a Redis list per notification_type per hour window. At window flush time, collapse: "3 people liked your post" instead of 3 separate pushes. Rules: if > 3 events of the same type in 60 minutes → batch. Critical events (OTP, security) always bypass batching. Implementation: Flink or a background worker scans Redis lists every 60 seconds, groups by user, renders digests, and enqueues to the push topic. Outcome: reduces push volume 40–60% for social products, significantly improving user engagement scores (users don't disable push when it isn't noisy).

4. **Compliance and consent management:** GDPR requires explicit, auditable consent for marketing notifications. TCPA (US) requires written consent for SMS marketing. Design a `consent_grants` table: `(user_id, channel, notification_category, jurisdiction, granted_at, expires_at, source)`. Before any marketing notification: verify consent exists, is not expired, and user has not subsequently opted out. Maintain an append-only audit log of all consent changes — this is the legal record. Data export: user can request all notifications sent to them (GDPR Article 15) — ClickHouse historical log makes this a simple query. Right to erasure: delete all notification history and preferences; mark user as deleted; all future orchestrator lookups short-circuit to "no channels enabled."

5. **Multi-tenant notification platform:** Build as an internal platform serving multiple products (order service, social service, marketing service). Each producer gets a quota: "order service: 50M notifications/day; marketing: 10M/day." Enforce quotas in the orchestrator using per-producer token buckets in Redis. Producer SLA visibility: each team has a dashboard showing their notification volume, delivery rates, and queue depth. Abuse prevention: if order service suddenly sends 10× normal volume (runaway loop bug), automatically throttle and alert the on-call. Priority queue by producer priority — order confirmation (critical producer) pre-empts marketing batch (low-priority producer) for worker capacity. This is building an internal equivalent of OneSignal or PagerDuty on top of your own infrastructure.
