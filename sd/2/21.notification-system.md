# 21. Notification System

**Type:** infra
**Difficulty:** L4
**Frequency:** Very High — fan-out, multiple channels, reliability

---

## Problem Statement

Design a notification system that sends push notifications (iOS/Android), SMS, and email to users based on system events. The system must handle 100M notifications/day, support multiple channels with fallback, guarantee at-least-once delivery, and allow users to manage notification preferences.

---

## Functional Requirements

**Must have:**
- Send push notification (iOS APNS, Android FCM)
- Send SMS (Twilio/SNS)
- Send email (SendGrid/SES)
- At-least-once delivery guarantee per channel
- User preference management (opt-in/out per notification type and channel)

**Should have:**
- Priority tiers: critical (OTP, alerts), standard (activity), marketing (promotions)
- Scheduled / delayed notifications
- Notification history per user
- Throttling: don't spam users with too many notifications in a window

**Out of scope:**
- In-app notification inbox (see notification inbox design)
- Two-way messaging / replies
- Rich media in push notifications

---

## Non-Functional Requirements

- **Scale:** 100M notifications/day across all channels; peak 10M/hour
- **Latency:** Critical notifications delivered in < 5 seconds; standard < 30 seconds
- **Availability:** 99.9% — if a channel is down, fall back to another
- **Delivery guarantee:** At-least-once — some duplicates acceptable; no losses
- **Compliance:** GDPR (consent required for marketing), CAN-SPAM (email unsubscribe), TCPA (SMS opt-in)

---

## Capacity Estimation

```
Notifications: 100M/day = 1,160/sec average; peak ~10K/sec
Breakdown: push 60%, email 30%, SMS 10%

Push: 60M/day = 700/sec → FCM/APNS handle thousands/sec easily
Email: 30M/day = 350/sec → SES handles this at $0.10/1000
SMS: 10M/day = 115/sec → Twilio at ~$0.0075/SMS

Events triggering notifications: upstream services publish events
  → Our system handles routing, preference check, delivery
```

## Deep Dive 1: Reliable Delivery with Retry

**Problem:** FCM/APNS/Twilio can fail transiently. Need retry with backoff.

**Retry strategy per channel:**
```
Push (FCM/APNS):
  Retry up to 3 times: 1s, 5s, 30s delays
  If device token invalid (APNS: 410, FCM: UNREGISTERED): remove token, don't retry
  If service down (5xx): retry; if throttled (429): respect Retry-After

Email:
  Retry up to 5 times: 1m, 5m, 15m, 1h, 24h
  Soft bounces (temporary): retry
  Hard bounces (bad address): mark as bounced, disable email for this address

SMS:
  Retry up to 3 times: 5s, 30s, 2m
  If invalid number: don't retry, log as failed
```

**Implementation in worker:**
```python
def process_push_notification(notification_id):
    notification = db.get_notification(notification_id)
    if notification.idempotency_key and already_sent(notification.idempotency_key):
        return  # dedup check

    try:
        response = fcm.send(notification.device_token, notification.payload)
        mark_sent(notification_id)
    except InvalidTokenError:
        remove_device_token(notification.device_token)
        mark_failed(notification_id, "invalid_token")
    except TransientError as e:
        if notification.retry_count < MAX_RETRIES:
            delay = exponential_backoff(notification.retry_count)
            requeue_with_delay(notification_id, delay)
        else:
            mark_failed(notification_id, "max_retries_exceeded")
```

## Deep Dive 2: Preference Check and Throttling

**Preference service:**
```
On each notification:
1. Look up user preferences (Redis cache, TTL 5 min)
   → Is push/email/sms enabled for this event_type for this user?
2. Check quiet hours (user's timezone)
   → If critical: override quiet hours
   → If standard/marketing: queue for delivery after quiet hours end
3. Check global opt-out (GDPR / marketing unsubscribe)
```

**Throttling:**
```
Don't send more than:
  - 5 push notifications per user per hour (non-critical)
  - 3 SMS per user per day (any type)
  - No limit on critical (security alerts, OTPs)

Implementation:
  Redis sliding window per user per channel per day:
  INCR throttle:{user_id}:push:{hour}
  EXPIRE throttle:{user_id}:push:{hour} 3600
  If count > limit: drop (marketing) or delay (standard)
```

## Reliability and Operations

- **Kafka consumer lag:** Monitor consumer group lag per topic. If lag grows → scale up workers.
- **Third-party service SLA:** FCM SLA is "best effort". For critical notifications: send via multiple channels simultaneously (push + SMS). First delivery wins; deduplicate on client.
- **Delivery receipts:** FCM delivers receipts async. APNS has feedback service for invalid tokens. Collect delivery confirmations in ClickHouse for analytics.
- **Deduplication:** `idempotency_key` in notifications table (UNIQUE). Workers check before sending. Prevents duplicate sends on retry.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Kafka per-channel queues | Single queue | Per-channel: isolates failures; Single: simpler routing |
| At-least-once delivery | Exactly-once | At-least-once: simpler; Exactly-once: complex, needs dedup on client |
| Redis for preferences | DB lookup | Redis: fast (< 1ms); DB: always fresh but adds latency |
| Async orchestrator | Synchronous | Async: decoupled, resilient; Sync: harder to fail gracefully |

## L5 Additions

1. **Intelligent channel selection:** Don't send to every channel blindly. Learn which channel each user engages with most. Historical engagement: does user tap push notifications? Open emails? Cost optimization: push = free; SMS = $0.0075. Prefer push over SMS unless user has shown SMS preference. ML model: predict engagement probability per user × channel × time_of_day. Route to channel with highest expected engagement. This reduces SMS costs by 30% while maintaining or improving engagement.

2. **Real-time personalization:** Notification content should be personalized. "Hey Alice, your favorite brand Nike just posted a deal on Air Max" vs generic "New deal available". Architecture: personalization service receives event + user context → generates personalized copy using template + ML (rule-based personalization or LLM-generated for marketing). Latency budget: personalization must complete within 200ms of orchestrator receiving event. Cache user preference profiles.

3. **Batch notifications for low-priority:** Don't send individual push for every social interaction. Digest batching: queue social notifications for a user in a Redis list. Flush with 1-hour window: "3 people liked your post" instead of 3 separate pushes. Rules: if > 3 events in 1 hour → batch. Critical events always bypass batching. This reduces notification fatigue and improves engagement rates.

4. **Compliance and consent management:** GDPR requires explicit consent for marketing notifications. TCPA (US) requires written consent for SMS marketing. System design: consent_grants table per user × channel × notification_category × jurisdiction. Before any marketing notification: verify consent exists + not expired + user hasn't opted out. Audit log of all consent changes. Data export: user can request all notifications sent to them (GDPR Article 15). Right to be forgotten: delete all notification history + preferences on request.

5. **Multi-tenant notification platform:** Build as a platform that multiple internal products use. Rate limiting per producer service: "order service can send 50M notifications/day; marketing service can send 10M/day". Priority queue by producer priority. Producer SLA visibility: each producer sees their own delivery metrics, failure rates, queue depth. Abuse prevention: if order service suddenly sends 10x normal volume (bug?), alert and throttle. This is essentially building an internal PagerDuty/OneSignal.
