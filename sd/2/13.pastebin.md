# 13. Pastebin — Text Snippet Sharing

**Type:** storage
**Difficulty:** L3
**Frequency:** Medium — often used as a warm-up or architecture basics question

---

## Problem Statement

Design Pastebin. Users can paste text (code, notes, logs) and receive a short URL to share the content. Pastes can be public or private, have an expiry time, and support syntax highlighting. The system must handle 1M pastes/day and 100M paste reads/day at low latency.

---

## Functional Requirements

**Must have:**
- Create a paste (text up to 10MB) → get short URL
- Read a paste by URL
- Optional expiry (1 hour, 1 day, 1 week, never)
- Syntax highlighting by language

**Should have:**
- Private pastes (accessible only via URL, not listed publicly)
- Password-protected pastes
- Edit paste (create new version, keep old URL working)

**Out of scope:**
- User accounts / paste management dashboard
- Real-time collaborative editing
- File attachments (text only)

---

## Non-Functional Requirements

- **Scale:** 1M writes/day, 100M reads/day; read:write = 100:1
- **Latency:** Read p99 < 50ms
- **Availability:** 99.9%
- **Durability:** Pastes never lost before expiry
- **Storage:** 1M × 10KB avg = 10GB/day; 1 year = 3.6TB

---

## Capacity Estimation

```
Writes:
  1M pastes/day = 11.6/sec average
  Peak: assume 5x average = ~60/sec
  Avg paste size: 10KB → 10GB/day new data
  Large pastes (1% at 1MB each): 1M × 0.01 × 1MB = 10GB/day
  Total write bandwidth: ~20GB/day = ~240KB/sec → trivial

Reads:
  100M reads/day = 1,157/sec average
  Peak: assume 4x = ~5,000/sec read QPS
  Cache hit rate: 99%+ for popular pastes (power-law distribution)
  → ~50 reads/sec actually hit the database

Short URL namespace:
  8 chars, Base62 alphabet (a-z, A-Z, 0-9) = 62^8 = 218 trillion combinations
  At 1M/day, takes 218 trillion / 1M = 218 billion days to exhaust → infinite for practical purposes

Storage:
  5 years × 365 days × 10GB/day = 18.25TB total
  Only non-expired pastes kept: assume 50% expire within 1 week → ~9TB long-term storage
  S3 at $0.023/GB: $207/month for 9TB → dirt cheap
```

## High-Level Architecture

```
[Client]
  │
  ├─ POST /pastes ──→ [Write API] ──→ [PostgreSQL] ──→ [S3 (for large pastes >64KB)]
  │                         │
  │                   [ID Generator]
  │
  └─ GET /:code ──→ [Read API] ──→ [Redis Cache] ──→ [PostgreSQL / S3]
                                          │
                                    [CDN (CloudFront/Fastly for viral pastes)]
```

**Component details:**
- **Write API:** Generates short ID, stores paste content (inline if small, S3 if large), sets Redis TTL. Stateless — horizontally scalable.
- **Read API:** Checks Redis first (99% hit rate for popular pastes); falls back to PostgreSQL. For S3-stored large pastes: redirect to presigned S3 URL or stream through API.
- **ID Generator:** Counter-based (monotonically increasing integer → Base62 encode) or UUID-based (then encode). Counter-based is simpler and avoids collision detection. Use a centralized sequence in PostgreSQL (`SELECT nextval('paste_id_seq')`) or a distributed Snowflake-style ID.
- **CDN:** Popular pastes (viral posts on HN/Reddit) get cached at edge nodes. Cache-Control headers set based on expiry.

## API Design

```
POST /v1/pastes
Body: {
  content: string,       # up to 10MB
  language?: string,     # "python", "javascript", "bash", etc. for syntax highlight
  expiry?: "1h"|"1d"|"1w"|"never",
  private?: bool,        # private = accessible only via direct URL, not listed
  password?: string      # if set, require password to view
}
Returns: { paste_id: "ab3x9k2m", url: "https://paste.co/ab3x9k2m", expires_at?: ISO8601 }

GET /v1/pastes/{id}
Headers: X-Password: <password>  (required if password-protected)
Returns: {
  id: "ab3x9k2m",
  content: string,
  language: string,
  created_at: ISO8601,
  expires_at: ISO8601 | null,
  view_count: int
}
Errors: 404 Not Found | 410 Gone (expired) | 401 Unauthorized (wrong/missing password)

PUT /v1/pastes/{id}
# Creates a new version; old URL still serves old content
# Only creator (by IP or auth token) can edit
Body: { content: string, language?: string }
Returns: { paste_id: "ab3x9k2m", version: 2 }

DELETE /v1/pastes/{id}
# Marks as deleted; future reads return 410 Gone
Returns: 204 No Content
```

## Data Model

```sql
-- PostgreSQL: paste metadata and small content
pastes (
  paste_id       VARCHAR(8) PRIMARY KEY,   -- Base62 short code
  content        TEXT,                     -- NULL if stored in S3 (large pastes)
  s3_key         TEXT,                     -- NULL if content fits in DB
  language       TEXT DEFAULT 'plaintext',
  is_private     BOOLEAN DEFAULT FALSE,
  password_hash  TEXT,                     -- bcrypt hash; NULL if no password
  view_count     INT DEFAULT 0,            -- approximate (Redis INCR, flushed periodically)
  version        INT DEFAULT 1,
  creator_ip     INET,                     -- for abuse tracking / edit permissions
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at     TIMESTAMPTZ,              -- NULL means never expires
  deleted_at     TIMESTAMPTZ,             -- soft delete

  -- Partial index for expiry cleanup (only index rows that have an expiry)
  INDEX idx_pastes_expires ON pastes (expires_at) WHERE expires_at IS NOT NULL
)

-- For edit history (optional v2 feature)
paste_versions (
  paste_id    VARCHAR(8) NOT NULL REFERENCES pastes(paste_id),
  version     INT NOT NULL,
  content     TEXT,
  s3_key      TEXT,
  created_at  TIMESTAMPTZ,
  PRIMARY KEY (paste_id, version)
)
```

**Storage strategy:**
- Content ≤ 64KB: store inline in DB as TEXT column. Rationale: a PostgreSQL row with 64KB TEXT fits in a TOAST table; query returns content in a single round trip. Optimal for p99 read latency.
- Content > 64KB: store in S3 with key `pastes/{paste_id}/v{version}`. DB row has `s3_key` set and `content` null. Read path: fetch DB row → redirect client to presigned S3 URL (valid 5 minutes).
- Threshold choice: 64KB is a practical inflection point where inline storage stops being cheaper than the S3 API call overhead plus the S3 request cost ($0.0004 per 1000 GET requests).

**Redis caching:**
```
Key:   paste:{paste_id}
Value: JSON { content, language, expires_at, is_private, password_hash }
TTL:   min(remaining_seconds_until_expiry, 86400)
       # Never cache past the expiry time
       # 86400 = 24 hours max for non-expiring pastes

Key:   viewcount:{paste_id}
Value: integer (INCR on each read)
TTL:   3600 seconds (flush to DB every hour)
       # Redis INCR is atomic, fast, avoids DB hot-row contention
```

## Deep Dive: Expiry and Cleanup

**Lazy expiry (on every read):**
```python
def get_paste(paste_id: str) -> Paste:
    paste = cache.get(f"paste:{paste_id}")
    if paste is None:
        paste = db.query("SELECT * FROM pastes WHERE paste_id = %s", paste_id)
        if paste:
            ttl = compute_cache_ttl(paste.expires_at)
            cache.set(f"paste:{paste_id}", serialize(paste), ttl)

    if paste is None:
        raise NotFoundError()

    # Lazy expiry check: serve 410 Gone without DB write
    if paste.expires_at and paste.expires_at < datetime.now(utc):
        cache.delete(f"paste:{paste_id}")
        raise ExpiredError()  # → HTTP 410 Gone

    if paste.deleted_at:
        raise DeletedError()  # → HTTP 410 Gone

    return paste
```

**Background cleanup (prevents unbounded storage growth):**
```sql
-- Run every hour as a cron job
-- LIMIT + loop avoids long table locks and replication lag spikes

DO $$
DECLARE
  deleted_count INT;
BEGIN
  LOOP
    DELETE FROM pastes
    WHERE paste_id IN (
      SELECT paste_id FROM pastes
      WHERE expires_at < NOW() - INTERVAL '1 hour'
        AND expires_at IS NOT NULL
      LIMIT 1000
    );
    GET DIAGNOSTICS deleted_count = ROW_COUNT;
    EXIT WHEN deleted_count = 0;
    PERFORM pg_sleep(0.1);  -- throttle: 10 batches/sec max
  END LOOP;
END $$;
```

**S3 lifecycle rule (supplement to DB cleanup):**
```json
{
  "Rules": [{
    "ID": "expire-pastes",
    "Filter": { "Prefix": "pastes/" },
    "Status": "Enabled",
    "Expiration": { "Days": 365 }
  }]
}
```
S3 lifecycle rules delete objects server-side without API calls — efficient for bulk cleanup of expired large pastes.

**Redis TTL alignment:**
- Redis TTL is set to `remaining_seconds_until_expiry` at cache write time
- Expired pastes auto-evict from Redis without any explicit invalidation
- No cache-DB inconsistency window: if paste is in cache and TTL hasn't expired, it's not expired in DB either (TTL was set from the DB's expires_at value)
- Edge case: if TTL is set, paste expires in DB, Redis TTL hasn't fired yet → lazy check on read handles this

## Reliability and Operations

- **ID uniqueness:** Use PostgreSQL sequence `nextval('paste_id_seq')` to get a monotonically increasing integer, then Base62-encode it. Sequence is guaranteed unique per call; no collision checking needed. At 1M/day, sequence grows by ~1M/day; 62^8 = 218 trillion IDs available; no exhaustion risk in any realistic timeframe.
- **View count accuracy:** Don't increment the DB `view_count` column on every read — this creates a hot-row problem at 5,000 reads/sec for popular pastes. Instead: `INCR viewcount:{paste_id}` in Redis (atomic, fast). A background job every 5 minutes reads all `viewcount:*` keys, updates the DB with `UPDATE pastes SET view_count = view_count + $delta WHERE paste_id = $id`, then resets the Redis counter. Result: approximate (within 5 minutes) but not losing counts.
- **Large paste abuse prevention:** Maximum content size: 10MB, enforced at the API gateway before any DB/S3 write. Rate limiting: 3 pastes/minute per IP for anonymous users (checked via Redis sliding window counter), 100/minute for authenticated users. Over-limit: 429 Too Many Requests.
- **Password hashing:** Use bcrypt with work factor 12. Never store the password in plaintext. Verify on read: `bcrypt.checkpw(submitted_password, stored_hash)`. bcrypt is slow by design (< 100ms per check) — acceptable for occasional paste access, unacceptable if called in a hot loop.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| PostgreSQL TEXT for small pastes, S3 for large | S3 for all pastes | DB inline: single round-trip for small content, lower read latency; S3 for all: simpler code, consistent path but adds S3 API call latency and cost per read |
| Counter-based IDs (Base62 integer) | Random IDs (UUIDs) | Counter: predictable, monotonic, no collision check; Random: not guessable (obscurity), but requires uniqueness check on creation |
| Lazy expiry on read | Proactive TTL enforcement | Lazy: simple, handles race conditions cleanly; Proactive: cleaner storage, but requires accurate background job and careful Redis TTL management |
| Redis for view counts | Approximate counts via HyperLogLog | Redis INCR: exact count per paste; HLL: approximate (would only help if counting unique viewers, not total views) |

## G5

1. **Syntax highlighting: server-side vs client-side trade-off at scale:** Client-side highlighting (Prism.js, highlight.js): zero server cost, runs in the user's browser, lazy-loaded JS bundle. Appropriate for low-traffic or content-focused pastes. Server-side (Pygments, tree-sitter): consistent rendering across browsers, works without JavaScript, allows pre-rendering HTML stored as a separate field or S3 object. For a high-read system with a 100:1 read/write ratio, pre-rendering highlighted HTML at paste creation time (one write-time CPU cost) eliminates per-read rendering overhead entirely. Store: `rendered_html` column or S3 key alongside raw content. Cache the rendered HTML in Redis with the same TTL as the raw content. The trade-off: cached HTML values are ~2–5x larger than raw text (HTML markup overhead), but this is irrelevant at 10KB average paste size.

2. **Full-text search for public pastes:** Index public pastes in Elasticsearch for code snippet search. On paste creation (is_private=false): fan-out to Kafka `paste-created` topic → async Elasticsearch indexer consumer. ES index fields: paste_id, content (analyzed), language, created_at. Search API: `GET /v1/search?q=postgres+upsert&language=sql`. ES tokenizes content with a code-aware analyzer (preserves underscores, camelCase splitting). Privacy enforcement: only index public pastes at write time, never index private or password-protected pastes. On paste deletion: publish `paste-deleted` event → ES consumer issues delete-by-id. Eventual consistency: newly created pastes appear in search within ~1 second (Kafka consumer lag).

3. **CDN for popular pastes (viral traffic spikes):** Pastebin pastes go viral frequently (Hacker News, Reddit, Twitter). A single paste can receive millions of reads within an hour. CDN (CloudFront/Fastly) in front of the Read API: `Cache-Control: public, max-age=3600` for non-expiring public pastes; `Cache-Control: private, no-store` for private/password-protected pastes. The CDN will absorb ~99.9% of traffic for a viral paste, reducing origin load to near-zero. Edge cache invalidation on paste deletion: send a cache purge API call to CDN when a paste is deleted or marked as containing abuse content. Estimated cache hit rate for top-1000 pastes: > 99.9%; for the long tail of rarely-accessed pastes: 0% (served from origin or Redis).

4. **Abuse and content moderation at scale:** Pastebin is chronically abused for phishing pages, credit card dumps, malware hosting, and doxing. Multi-layered defense: (a) Rate limiting at the IP and account level — 3 pastes/min anonymous, 100/min authenticated. (b) Pattern detection at write time: scan content for credit card regex (4×4-digit patterns), SSN patterns (US: XXX-XX-XXXX), known phishing kit markers. Reject or quarantine on match. (c) SHA-256 content hash against a blocklist of known-bad content (NCMEC hashes for CSAM, VirusTotal for malware). (d) Abuse report endpoint with < 24-hour takedown SLA for DMCA/GDPR requests. (e) IP reputation scoring: integrate IPQualityScore or similar to score anonymous paste creators; high-fraud-risk IPs get lower rate limits or CAPTCHA.

5. **Bring-your-own domain (BYOD) for enterprise:** Premium users configure `snippets.company.com` to point to Pastebin via DNS CNAME to `custom.paste.co`. TLS certificate: provisioned automatically via Let's Encrypt ACME DNS-01 challenge when the domain is first registered. Tenant mapping: database table `custom_domains (domain TEXT PRIMARY KEY, tenant_id UUID, verified_at TIMESTAMPTZ)`. On request to `snippets.company.com/abc123`: the API gateway reads the `Host` header, looks up `custom_domains` (Redis cache, TTL 60s), resolves to `tenant_id`, then scopes all paste lookups to that tenant. Rate limiting per tenant. A tenant's pastes are only accessible under their custom domain or the main domain depending on privacy settings. BYOD requires SSL offload at the gateway layer that can dynamically serve different certificates per `Host` header (SNI-based TLS).
