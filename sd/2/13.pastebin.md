# 13. Pastebin — Text Snippet Sharing

**Type:** storage
**Difficulty:** L3
**Frequency:** Medium — often used as a warm-up or architecture basics question

---

## Problem Statement

Design Pastebin. Users can paste text (code, notes, logs) and receive a short URL to share the content. Pastes can be public or private, have an expiry time, and support syntax highlighting. The system must handle 1M pastes/day and 100M paste reads/day at low latency.

---

## Functional Requirements

**Must have:**
- Create a paste (text up to 10MB) → get short URL
- Read a paste by URL
- Optional expiry (1 hour, 1 day, 1 week, never)
- Syntax highlighting by language

**Should have:**
- Private pastes (accessible only via URL, not listed publicly)
- Password-protected pastes
- Edit paste (create new version, keep old URL working)

**Out of scope:**
- User accounts / paste management dashboard
- Real-time collaborative editing
- File attachments (text only)

---

## Non-Functional Requirements

- **Scale:** 1M writes/day, 100M reads/day; read:write = 100:1
- **Latency:** Read p99 < 50ms
- **Availability:** 99.9%
- **Durability:** Pastes never lost before expiry
- **Storage:** 1M × 10KB avg = 10GB/day; 1 year = 3.6TB

---

## Capacity Estimation

```
Writes: 1M/day = 12/sec average; peak ~50/sec
Reads:  100M/day = 1,150/sec; peak ~5,000/sec
Storage: 1M × 10KB = 10GB/day; 5 years = 18TB
Short URL namespace: 8 chars Base62 = 62^8 = 218 trillion combos → never runs out
```

---

## High-Level Architecture

```
[Client]
  │
  ├─ POST /pastes ──→ [Write API] ──→ [PostgreSQL] ──→ [S3 (for large pastes >64KB)]
  │                         │
  │                   [ID Generator]
  │
  └─ GET /:code ──→ [Read API] ──→ [Redis Cache] ──→ [PostgreSQL / S3]
                                          │
                                    [CDN (optional for static content)]
```

---

## API Design

```
POST /v1/pastes
Body: { content: string, language?: string, expiry?: "1h"|"1d"|"1w"|"never", private?: bool, password?: string }
Returns: { paste_id, url: "https://paste.co/abc12345", expires_at? }

GET /v1/pastes/{id}
Headers: X-Password (if password-protected)
Returns: { id, content, language, created_at, expires_at, view_count }

PUT /v1/pastes/{id}   (creates new version, old URL still works)
DELETE /v1/pastes/{id}  (marks as deleted, returns 410 Gone on future reads)
```

---

## Data Model

```sql
pastes (
  paste_id    VARCHAR(8) PRIMARY KEY,  -- Base62 short code
  content     TEXT,                    -- NULL if stored in S3 (large pastes)
  s3_key      TEXT,                    -- NULL if content fits in DB
  language    TEXT,
  is_private  BOOLEAN DEFAULT FALSE,
  password_hash TEXT,
  view_count  INT DEFAULT 0,
  created_at  TIMESTAMPTZ,
  expires_at  TIMESTAMPTZ,

  -- Cleanup index
  INDEX (expires_at) WHERE expires_at IS NOT NULL
)
```

**Storage strategy:**
- Content ≤ 64KB: store inline in DB (TEXT column)
- Content > 64KB: store in S3, save S3 key in DB
- Threshold: 64KB fits in a single 8KB PostgreSQL page with room to spare; larger → I/O overhead

**Redis caching:**
```
Key: paste:{paste_id}
Value: JSON { content, language, expires_at }
TTL: min(remaining_expiry, 24h) — never cache past expiry
```

---

## Deep Dive: Expiry and Cleanup

**Lazy expiry (on read):**
```
When serving a paste:
  if paste.expires_at < now():
    return 410 Gone
    # optionally delete from DB
  else:
    return paste
```

**Background cleanup:**
```
Cron job every hour:
  DELETE FROM pastes WHERE expires_at < NOW() - INTERVAL '1 hour'
  # Run in batches of 1000 to avoid long locks

For S3:
  S3 lifecycle rules: delete objects with key prefix "pastes/" after X days
  (based on metadata tag set at upload time)
```

**Redis TTL alignment:**
- Set Redis TTL = remaining seconds until expiry
- Expired pastes automatically evict from cache
- No explicit cache invalidation needed

---

## Reliability and Operations

- **ID uniqueness:** Use counter-based ID (similar to TinyURL) with Base62 encoding. No collision check needed.
- **View count:** Approximate view count using Redis INCR (1s TTL flush to DB). Don't increment DB counter on every read — hot row problem.
- **Large paste abuse:** Max size 10MB; rate limit creation (10 pastes/hour for anonymous, 100/hour for authenticated).

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| PostgreSQL TEXT | S3 for all pastes | DB: fast for small content; S3: cheaper for large |
| Counter-based IDs | Random IDs | Counter: no collision check; Random: not guessable |
| Lazy expiry | Proactive TTL | Lazy: simple; Proactive: cleaner storage but extra complexity |

## L5 Additions

1. **Syntax highlighting server-side vs client-side:** Client-side (Prism.js/highlight.js): zero server cost, runs in browser. Server-side (Pygments): consistent rendering, works without JS, can pre-render to HTML stored alongside raw content. For a high-read system, pre-render HTML on paste creation, cache the rendered HTML. Trade: slightly larger cache values; eliminates per-request rendering cost.

2. **Searching pastes:** Full-text search on public pastes. Elasticsearch index: paste content + metadata. On paste creation, fan-out to index (async via Kafka). Search API: `GET /search?q=keyword&language=python`. Elasticsearch tokenizes and indexes. Privacy: only index public pastes (is_private=false). For private pastes: never indexed, only accessible by direct URL.

3. **CDN for popular pastes:** Some pastes go viral (shared on Reddit/HN). Put a CDN in front of GET /pastes/:id. Cache at edge with Cache-Control: max-age=3600 (1 hour). On expiry, CDN edge fetches fresh copy. Edge cache invalidation on paste deletion. Reduces origin load to near zero for popular pastes.

4. **Abuse and content moderation:** Pastebin is frequently used for phishing pages, malware hosting, data dumps. Rate limit: 3 pastes/min per IP for anonymous users. Pattern detection: scan for credit card patterns, SSN patterns, known malware signatures. Hash-based blocking: blocklist of SHA-256 hashes of known-bad content. Report and takedown API for DMCA/abuse reports with < 24h SLA.

5. **Bring-your-own domain (BYOD):** Premium users can use their own domain (e.g., `snippets.company.com`). Client configures CNAME at their DNS → points to our CNAME target. TLS: we provision cert via Let's Encrypt ACME protocol (on first request to that domain). Map: custom domain → tenant_id in our DB. All paste lookups under that domain resolve to that tenant's pastes. Rate limiting per tenant.
