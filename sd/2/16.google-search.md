# 16. Google Search — Web Search Engine

**Type:** search
**Difficulty:** G5
**Frequency:** Medium-High — asked at search companies, broader infra interviews

---

## Problem Statement

Design a web search engine like Google. Users enter a query and receive the top 10 most relevant web pages in < 500ms. The system must index 100B web pages, handle 100K search queries/sec, and return ranked results incorporating relevance, PageRank, and freshness.

---

## Functional Requirements

**Must have:**
- Accept a text query, return top 10 relevant URLs with title + snippet
- Results ranked by relevance (text match) + importance (PageRank) + freshness
- Handle multi-word queries (AND semantics by default)
- Crawl web pages continuously and keep index fresh (see web crawler design)

**Should have:**
- Spell correction
- Featured snippets (direct answer in result)
- Query understanding: synonyms, intent classification
- Safe search filtering

**Out of scope:**
- Knowledge graph / entity search
- Image/video search
- Ads serving

---

## Non-Functional Requirements

- **Scale:** 100B indexed pages; 100K queries/sec
- **Latency:** Results returned in < 500ms (including network)
- **Freshness:** Major news pages indexed within minutes; normal pages within hours
- **Availability:** 99.999% — Google Search is critical infrastructure
- **Relevance:** Top result is relevant for > 95% of navigational queries

---

## Capacity Estimation

```
Index size:
  100B documents × avg 100 unique terms per document = 10 trillion term-document pairs
  Per pair: doc_id(8B) + term_freq(2B) + position_list_offset(4B) = 14B
  Compressed with delta encoding + VByte: ~4–6B per pair
  Total compressed index: 10T × 5B = 50TB of index data
  Across 200 index shard servers: 50TB / 200 = 250GB per server (fits on SSD)

Query processing:
  100K queries/sec × avg 3.5 terms per query = 350K posting list lookups/sec
  Each lookup: deserialize posting list (500K–100M entries for common terms)
  Intersection of 3-4 posting lists → top 1000 candidates per shard
  200 shards × 100K QPS = 20M shard-queries/sec (each shard handles 100 QPS)

  Scoring: 1000 candidates × 50 ranking features × 4 bytes = 200KB per query
  → At 100K QPS: 20GB/sec of scoring data flowing through ranking servers

Document store:
  100B documents × 300B avg (url, title, 160-char snippet, PageRank) = 30TB
  Sharded across 200 document store servers: 150GB per server
```

## High-Level Architecture

```
[User Query]
     │
     ▼
[Query Processor]
     │
     ├──→ [Spell Corrector] ──→ corrected query terms
     ├──→ [Query Parser] ──→ terms, phrases, operators (site:, filetype:, etc.)
     ├──→ [Query Expander] ──→ synonyms, related entities, intent classification
     │
     ▼
[Index Server Coordinator (Fan-out)]
     │
     ├──→ [Doc Index Shard 1/200] ──→ posting list lookups → top 1000 doc IDs
     ├──→ [Doc Index Shard 2/200] ──→ ...
     ├──→ ... (parallel scatter to all 200 shards)
     └──→ [Doc Index Shard 200/200] ──→ ...
                   │
                   ▼ (gather phase)
            [Score Aggregator]
                   │
                   ├──→ [Ranking Service (ML model, GPU-accelerated)]
                   ├──→ [PageRank Store (read-only Redis)]
                   ├──→ [Freshness Signal Service]
                   │
                   ▼
             [Top 10 results (global)]
                   │
                   ▼
            [Snippet Generator]
                   │
                   ▼
            [Results Page (< 500ms total)]

[Background processes]:
[Crawler] → [Page Processor] → [Indexer] → [Index Shards]
              │
              └──→ [PageRank Computation (Spark, weekly)]
                   [Link Graph Store (distributed)]
```

## Data Model

**Inverted Index — core of any search engine:**
```
Structure:
  Maps each term → posting list (sorted list of documents containing that term)

Term:    "pizza"
Posting list: [
  { doc_id: 100234, tf: 5, positions: [3, 17, 45, 89, 120], doc_len: 300 },
  { doc_id: 200891, tf: 12, positions: [1, 2, 8, 11, 14, ...], doc_len: 150 },
  { doc_id: 300002, tf: 2, positions: [50, 200], doc_len: 800 },
  ...  (millions of entries for common terms)
]

Sorted by doc_id (monotonically increasing) for merge/intersection efficiency

Metadata per term:
  term_df:  document frequency (# docs containing the term)
  idf:      log(N / df)  — precomputed IDF weight

Storage format (Lucene-style segment files):
  .tim file: term → posting list file offset
  .doc file: doc_id delta-encoded (VByte compression)
  .freq file: term frequency per doc
  .pos file:  position lists (for phrase queries)
  Typical compression ratio: 4:1 on posting lists using delta + VByte encoding
```

**Index segmentation:**
```
New content indexed into small "real-time segments" (< 100K docs, in-memory)
Every hour: flush real-time segments to SSD as immutable segment files
Nightly: merge small segments into larger segments (fewer files = faster queries)
Segment merging: log-structured merge (LSM) style — reduces # segments by 10x per merge

Two-tier index:
  Base index: all 100B pages (updated daily via full rebuild or incremental merge)
  Real-time index: last 24 hours of crawled/updated pages (updated every minute)
  Query time: scatter to both tiers, merge results (real-time results get freshness boost)
```

**Document store:**
```sql
-- Distributed key-value store (BigTable/Cassandra style)
docs (
  doc_id       BIGINT PRIMARY KEY,   -- 8-byte hash of URL
  url          TEXT,
  title        TEXT,
  snippet      TEXT,                  -- pre-generated 160-char summary
  body_text    TEXT,                  -- full cleaned text (for on-the-fly snippets)
  content_hash TEXT,
  pagerank     FLOAT,                 -- from weekly PageRank computation
  indexed_at   TIMESTAMP,
  language     TEXT,
  word_count   INT
)

-- PageRank store (read-heavy, Redis for hot docs)
pagerank (
  doc_id    BIGINT PRIMARY KEY,
  rank      FLOAT,              -- 0.0 to 1.0 normalized
  in_links  INT,                -- number of pages linking here
  computed_at TIMESTAMP
)
```

## Deep Dive 1: Inverted Index and Query Processing

**Building the index:**
```
1. Crawler sends raw HTML to Document Processor

2. Document Processor pipeline:
   a. HTML extraction: strip tags, decode entities, extract text content
      - Keep <title>, <h1>-<h6>, <p>, <li> — these signal term importance
      - Boost terms in title (3×), H1 (2×), URL path (1.5×) vs body text (1×)
   b. Language detection (langdetect library): route to language-specific pipeline
   c. Tokenization: "New York pizza" → ["new", "york", "pizza"]
   d. Normalization:
      - Lowercase: "Pizza" → "pizza"
      - Stemming (Porter stemmer): "pizzas" → "pizza", "running" → "run"
      - Stop word removal: "the", "a", "is" (too common, in every posting list)
   e. TF computation: count occurrences per document, per field (title vs body)
   f. Generate posting list entries: (term, doc_id, tf, positions, field_boost)

3. Index Builder:
   a. Sort posting list entries by term (external sort for TB-scale data)
   b. Group entries by term → posting list
   c. Delta-encode doc_ids: [100, 105, 200] → [100, 5, 95] (smaller numbers = better compression)
   d. VByte compress delta-encoded numbers
   e. Write to immutable segment file (.tim, .doc, .freq, .pos)
   f. Merge new segments with existing base index periodically
```

**Query processing — detailed walkthrough:**
```
Query: "pizza delivery new york"
Step 1: Term parsing
  Terms: ["pizza", "delivery", "new", "york"]
  (after stop word removal, stemming, lowercase)

Step 2: Fan-out to all 200 index shards (parallel)
  Each shard: look up posting list for each term
  pizza:    10M docs on this shard
  delivery: 8M docs
  new:      50M docs (very common, "new" → huge posting list, needs optimization)
  york:     20M docs

Step 3: Posting list intersection (AND semantics)
  Strategy: start with smallest list (delivery: 8M), intersect with next smallest (pizza: 10M)
  Merged: docs containing both "pizza" AND "delivery": ~2M docs
  Intersect with "york": ~500K docs
  Intersect with "new": ~400K docs

Step 4: Skip list acceleration (for large lists)
  Problem: intersecting 50M-entry list with 8M-entry list naively is O(50M) comparisons
  Skip list: every 256th entry in posting list has an "index pointer" (skip pointer)
  Binary search with skip pointers: O(log N) to advance to any doc_id
  DAAT (Document-At-A-Time): advance pointers simultaneously across all lists
  → Intersection of 50M × 8M → ~5M comparisons (not 400M naive)

Step 5: Score 400K candidate documents (per shard)
  BM25 score (state of the art TF-IDF variant):
    score(D,Q) = Σ IDF(qi) × [TF(qi,D) × (k1+1)] / [TF(qi,D) + k1×(1 - b + b×|D|/avgdl)]
    k1 = 1.5 (term saturation), b = 0.75 (length normalization)
    avgdl = average document length across all indexed docs

  Additional signals combined with BM25:
    + PageRank: normalized log(1 + in_link_count)
    + Freshness: sigmoid(age_decay) — newer pages get boost for news queries
    + URL quality: root domain vs subdomain, URL depth penalty
    + Click-through rate (CTR): historical click rates from search logs
    + Domain authority: aggregate trust score

  Take top 1000 candidates per shard by BM25+signals score

Step 6: Gather phase — merge 200 shards × 1000 candidates = 200K candidates
  Global merge: top 5000 globally by initial score
  → Send to ML ranking server

Step 7: ML re-ranking (BERT-based cross-encoder, GPU-accelerated)
  Input: (query, candidate_title, candidate_snippet) triplets
  Model: cross-encoder computes relevance score (not just BM25)
  Output: relevance_score per candidate
  Blend: final_score = 0.6 × ML_score + 0.4 × BM25_PageRank_score
  Sort by final_score, take top 10

Step 8: Snippet generation for top 10
  Find sentence in body_text with highest query term density
  Highlight matching terms with <em> tags
  Truncate to 160 characters at sentence boundary
```

**G5:** Would push on the interaction between index freshness and shard consistency. The two-tier index (base + real-time) creates a query-time merge problem: a document updated in the last 24 hours may appear in BOTH the base index (old version) and the real-time index (new version). Without deduplication at query time, the same URL could appear twice in results. The production fix is a "tombstone" mechanism: when a document is updated in the real-time index, its doc_id is added to a tombstone set in the base index shard. During intersection, any doc_id in the tombstone set is skipped in the base index. The real-time index version is used instead. The challenge: tombstone sets grow without bound and must be periodically flushed during index merges. This is non-trivial to implement correctly in a 200-shard distributed system, and getting it wrong produces duplicate results — a visible quality regression that erodes user trust.

## Deep Dive 2: PageRank Computation

**PageRank — the mathematical foundation:**
```
Random surfer model:
  A user browsing the web either:
    - Follows a link on the current page with probability d (damping factor = 0.85)
    - Jumps to a random page with probability (1-d) = 0.15

PageRank formula:
  PR(A) = (1-d)/N + d × Σ [PR(B) / OutLinks(B)]
           (teleportation)    (for all B linking to A)

Where:
  N = total pages in graph
  d = 0.85
  OutLinks(B) = number of outbound links from page B
  PR(B) / OutLinks(B) = each link distributes B's rank equally to all targets

Intuition:
  A page linked by many high-PR pages gets a high PR score
  Links from high-PR pages are worth more than links from low-PR pages
  A page's PR score = its expected fraction of time the random surfer spends on it
```

**Computation at scale:**
```
Link graph:
  100B nodes (pages), estimated 1T+ edges (links)
  Storage: 1T edges × 16B (src_hash + dst_hash) = 16TB edge data
  Stored as adjacency lists in Parquet files on S3, partitioned by src_hash prefix

Distributed computation (Spark GraphX / Apache Giraph / Pregel):
  Input: link graph as adjacency list
  Initialize: all PR = 1/N ≈ 10^-11

  Per iteration (Pregel BSP — bulk synchronous parallel):
    1. Each vertex sends its current PR / out_degree to all neighbors
    2. Each vertex receives messages from all in-neighbors
    3. Each vertex updates: new_PR = (1-d)/N + d × sum(received_messages)
    4. Barrier: wait for all vertices to complete step 3

  Convergence check: stop when max |PR_new - PR_old| < epsilon (typically < 50 iterations)
  Runtime: ~2-4 hours for 100B nodes on 1000-node Spark cluster

Incremental PageRank (daily):
  Only recompute PR for pages near the changed edges (BFS-limited neighborhood)
  Saves 90% compute vs full weekly recomputation
  Full recomputation: weekly (for convergence guarantee)
  Incremental: daily (fast, approximation)

Output:
  Write PR scores to PageRank store (Redis for top 10M high-PR pages, Cassandra for all)
  Feed into index serving via offline join: index builder includes PR score in each posting list entry
```

## Reliability and Operations

- **Index shard failure:** Every index shard has 3 replicas across different racks. Query coordinator automatically routes to a healthy replica. If all replicas of a shard fail: coordinator returns results from remaining shards (partial results, still useful). Alert on > 5% of shards missing (degrades recall noticeably).
- **Query latency budget:** 500ms total. Breakdown: query parsing 10ms, spell check 20ms, scatter to shards 150ms (parallel), gather + merge 30ms, ML ranking 100ms (GPU), snippet generation 50ms, network (user to PoP) 100ms. Any component exceeding its budget → skip it and return degraded results (e.g., no ML ranking → return BM25-only results).
- **Fresh index vs stale:** Real-time index (< 24h content) served in parallel with base index. For breaking news: real-time index results promoted to top 3 positions for news-intent queries (detected via query classifier). For general queries: real-time index mixed in with freshness boost signals.
- **Availability — 99.999%:** 5 nines = 5.26 minutes downtime/year. Achieved via: (a) no single point of failure — all components replicated. (b) Regional failover: if US-EAST region degrades, route queries to US-WEST. (c) Circuit breakers at every service boundary: slow ML ranking → skip ML, use BM25. (d) Read-only degraded mode: index can serve queries even if crawling pipeline is down.

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Distributed inverted index (200 shards) | Single-machine Lucene | Distributed: scales to 100B docs; Single-machine: simpler, lower latency (no scatter-gather), max ~10M docs at practical SSD capacity |
| Two-phase ranking (BM25 → ML) | Pure ML ranking on all candidates | Two-phase: BM25 cheaply reduces 100M → 1000 candidates, ML only on top 1K (affordable); Pure ML: better quality ceiling but 100K× more expensive per query |
| DAAT (document-at-a-time) intersection | TAAT (term-at-a-time) | DAAT: memory-efficient, supports early termination when top-K candidates are certain; TAAT: simpler, but must score all docs for all terms before merging |
| Pre-computed snippets | Query-aware on-the-fly generation | Pre-computed: fast (no text processing at query time), but generic; On-the-fly: query-specific highlighting, fresh content, but adds 50–100ms latency |
| Weekly full PageRank recomputation | Real-time PageRank updates | Weekly full: guaranteed convergence, stable scores; Real-time: faster reflection of new links but never converges, susceptible to link spam attacks |

## G5

1. **Two-phase ranking at scale with GPU acceleration:** Phase 1 — lightweight BM25 + PageRank + 10 fast signals — runs on CPU-based index servers and reduces the candidate pool from billions to top 1000 documents. Phase 2 — BERT-based cross-encoder or a custom ML ranking model (LambdaRank/XGBoost on 500 features) — runs on GPU-accelerated ranking servers. The cross-encoder attends jointly over the query and document, capturing semantic relationships that bag-of-words BM25 misses entirely ("how to lose weight" should match "weight loss tips" even with zero exact term overlap). Latency budget: Phase 1 targets < 100ms, Phase 2 targets < 100ms on GPU batch inference. The GPU batch size is critical: batching 200 (query, doc) pairs per GPU kernel call amortizes the kernel launch overhead, reducing per-pair cost from ~5ms to < 0.5ms.

2. **Real-time index for freshness:** News articles must appear in search results within minutes of publication. Architecture: dedicated real-time crawling pipeline (higher frequency, subscribed to RSS/PubSubHubbub). New or updated page → streaming indexer (Apache Flink) → in-memory posting lists per term, stored in Redis sorted sets. At query time: coordinator sends query to both the base (disk-based Lucene) index AND the in-memory real-time index, merging results with freshness boost. Real-time index capacity: last 24 hours of crawled content (~10M pages/day × 100KB metadata = 1TB — partitioned across 100 Redis nodes). Compaction: every hour, the oldest hour's worth of real-time index data is flushed to the base index via a mini-merge segment.

3. **Personalized search with privacy guarantees:** Signed-in users receive personalized results incorporating: geographic location (city or region), preferred language, past click-through history on results for similar queries, SafeSearch preference, content type preferences. Architecture: personalization signals are computed as a sparse feature vector per user (< 1KB), encrypted at rest with user-specific keys, and transmitted to the ranking server only for that user's query. The ML ranking model accepts these features as additional inputs that modulate the base ranking. Privacy constraint: raw query history is never stored as plaintext on servers — only derived features (e.g., "user has clicked 3 times on cooking-related results in the past month" → a scalar feature). Differential privacy noise is added to aggregate statistics derived from query history to prevent re-identification.

4. **Query understanding beyond keyword matching:** Modern search understands user intent rather than just matching keywords. (a) Named Entity Recognition (NER): "paris" in "paris hilton" vs "paris france" — recognized via surrounding context words. (b) Query classification: navigational (user wants `facebook.com`), informational (wants knowledge), transactional (wants to buy), local (wants nearby business). Classification drives result type: local queries trigger Map Pack results; transactional triggers Shopping results. (c) Semantic equivalence via query expansion: "buy", "purchase", "order" treated as query synonyms; expanding query terms increases recall by ~15% without precision loss. (d) Temporal intent: "weather" → current conditions widget; "super bowl" near the event date → live score widget — served by specialized result handlers that pre-empt organic web results.

5. **Anti-spam and content quality signals:** An estimated 30–40% of crawled web content is spam, SEO manipulation, or thin content designed to exploit search algorithms rather than serve users. Multi-layered quality defenses: (a) Content quality classifier (ML model trained on human-rated "needs met" data from Google's Search Quality Rater Guidelines): predicts document quality on a 5-point scale. (b) Link spam detection: PageRank algorithm naturally discounts link farms (pages with many outbound links to one domain pattern), but adversarial SEO constantly evolves new patterns — adversarial detection is an ML arms race. (c) Cloaking detection: crawler uses both Googlebot user-agent AND disguised browser user-agent for the same URL; if content differs significantly (SimHash distance > threshold), penalize the domain for cloaking. (d) Core Web Vitals: real user performance data (page load time, interactivity, layout stability) from Chrome UX Report feeds into ranking — slow, janky pages rank lower, incentivizing web performance investment.
