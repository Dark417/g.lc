# 16. Google Search — Web Search Engine

**Type:** search
**Difficulty:** L5
**Frequency:** Medium-High — asked at search companies, broader infra interviews

---

## Problem Statement

Design a web search engine like Google. Users enter a query and receive the top 10 most relevant web pages in < 500ms. The system must index 100B web pages, handle 100K search queries/sec, and return ranked results incorporating relevance, PageRank, and freshness.

---

## Functional Requirements

**Must have:**
- Accept a text query, return top 10 relevant URLs with title + snippet
- Results ranked by relevance (text match) + importance (PageRank) + freshness
- Handle multi-word queries (AND semantics by default)
- Crawl web pages continuously and keep index fresh (see web crawler design)

**Should have:**
- Spell correction
- Featured snippets (direct answer in result)
- Query understanding: synonyms, intent classification
- Safe search filtering

**Out of scope:**
- Knowledge graph / entity search
- Image/video search
- Ads serving

---

## Non-Functional Requirements

- **Scale:** 100B indexed pages; 100K queries/sec
- **Latency:** Results returned in < 500ms (including network)
- **Freshness:** Major news pages indexed within minutes; normal pages within hours
- **Availability:** 99.999% — Google Search is critical infrastructure
- **Relevance:** Top result is relevant for > 95% of navigational queries

---

## Capacity Estimation

```
Index size: 100B documents
  Inverted index: avg 100 terms per doc × 100B docs = 10 trillion term-doc pairs
  Per pair: term_id(8B) + doc_id(8B) + position(4B) + weight(4B) = 24B
  Total: 10T × 24B = 240TB just for term-doc pairs
  → Need distributed index, many machines

Queries: 100K/sec
  Each query: look up 3-5 terms → 3-5 index lookups → intersect posting lists
  Intersection of large posting lists (100M documents for "the"): slow
  → Must precompute/optimize; common terms pruned or stored separately

Query per doc score: compute TF-IDF + PageRank + 50 other signals for top 1000 candidates
  → 1000 × 50 features × float = 200KB per query → feasible in < 100ms
```

## Data Model

**Inverted Index — core of a search engine:**
```
Inverted index: maps each term → list of documents containing that term

Term:    "pizza"
Posting list: [
  { doc_id: 100234, tf: 5, positions: [3, 17, 45], doc_len: 300 },
  { doc_id: 200891, tf: 12, positions: [1, 2, 8, ...], doc_len: 150 },
  ...
]
Sorted by doc_id (for merge/intersection efficiency)

Per-term data also stored:
  - term_df (document frequency): how many docs contain this term
  - idf = log(N / df): inverse document frequency

Storage:
  - Index segments: immutable sorted files (like Lucene segments)
  - Merge periodically (compaction)
  - New content → new small segment → periodic merge into larger segments
```

**Document store:**
```
docs (
  doc_id      BIGINT PRIMARY KEY,   -- 8-byte ID
  url         TEXT,
  title       TEXT,
  snippet     TEXT,                  -- pre-generated 160-char summary
  content_hash TEXT,
  pagerank    FLOAT,
  indexed_at  TIMESTAMP,
  language    TEXT
)
```

**PageRank store:**
```
pagerank (
  doc_id    BIGINT PRIMARY KEY,
  rank      FLOAT,              -- normalized 0-1
  computed_at TIMESTAMP
)
→ Recomputed weekly via Spark/MapReduce on the full link graph
```

## Deep Dive 1: Inverted Index and Query Processing

**Building the index:**
```
1. Crawler sends raw HTML to Document Processor
2. Document Processor:
   a. Extract text (strip HTML tags)
   b. Tokenize: "Pizza near me" → ["pizza", "near", "me"]
   c. Normalize: lowercase, stemming (pizza/pizzas → pizz), stop word removal
   d. TF computation: count term frequency per document
   e. Generate posting list entries: (term, doc_id, tf, positions)
3. Index Builder:
   a. Sort posting list entries by term
   b. Group by term → posting list
   c. Write to immutable segment file (sorted)
   d. Merge with existing segments periodically
```

**Query processing:**
```
Query: "pizza delivery new york"
Terms: ["pizza", "delivery", "new", "york"]

1. Look up posting list for each term (parallel):
   pizza: [100K doc_ids with scores]
   delivery: [80K doc_ids with scores]
   new: [500K doc_ids — very common]  ← skip index acceleration needed
   york: [200K doc_ids]

2. Intersect posting lists (AND semantics):
   Start with smallest list (delivery: 80K)
   Merge with pizza: 20K docs containing both
   Merge with york: 8K docs containing all three
   Merge with new: 7K docs containing all four

3. Score remaining 7K docs:
   TF-IDF score per term per doc
   + PageRank
   + Freshness (recent pages get boost)
   + URL structure (homepage > subpage)
   + Click-through rate (popular results get boost)

4. Sort by score, take top 1000 for final ranking

5. ML re-ranking on top 1000 (BERT-based model)

6. Return top 10 with snippet
```

**Posting list intersection optimization:**
```
Skip lists: within posting list, every k-th entry is an "index" entry
  Binary search within the list → O(log N) vs O(N) for finding a doc_id
  Essential for intersecting 100M-entry lists efficiently

DAAT (Document-At-A-Time) processing:
  Process one document at a time across all term lists
  Use heap to advance the slowest-moving list
  Stop when we have top-K candidates with confidence
```

## Deep Dive 2: PageRank Computation

**PageRank (simplified):**
```
PR(A) = (1-d) + d × Σ [PR(B) / OutLinks(B)] for all B linking to A

d = 0.85 (damping factor — probability of following a link)
Random surfer model: user either follows a link (d) or jumps to random page (1-d)

Algorithm:
1. Build link graph: 100B nodes, estimated 1T edges
2. Initialize all PR = 1/N
3. Iterate: for each page, distribute PR to outbound links
4. Converge after ~50 iterations

Scale:
  Graph: 1T edges × 16B per edge = 16TB — distributed across 100+ machines
  Computation: Spark GraphX or Pregel (BSP model)
  Runtime: ~hours per full computation
  Frequency: weekly full recompute; daily incremental for new pages
```

## Reliability and Operations

- **Index shard failure:** Index shards are replicated (3 replicas). Query coordinator skips failed shards gracefully (returns partial results, still useful).
- **Query serving degradation:** If scoring system is slow, return results based on just TF-IDF (no ML re-ranking). Latency budget: 500ms total. Breakdown: query parsing 10ms, index lookup 150ms, scoring 100ms, ML ranking 100ms, snippet 50ms, network 90ms.
- **Fresh index vs stale:** Two tiers: realtime index (last 24h, small, high priority) + base index (full 100B pages). Query coordinator merges results from both.

---

## Tradeoffs

| Decision | Alternative | Tradeoff |
|----------|-------------|----------|
| Distributed inverted index | Single-machine | Distributed: scales to 100B docs; single-machine: simpler, faster for small index |
| TF-IDF + PageRank | Pure ML ranking | Hybrid: interpretable baseline; Pure ML: higher relevance ceiling, black box |
| DAAT intersection | TAAT (term-at-a-time) | DAAT: better early termination; TAAT: simpler implementation |
| Pre-computed snippets | On-the-fly generation | Pre-computed: faster; On-the-fly: always fresh, query-aware highlight |

## L5 Additions

1. **Two-phase ranking at scale:** Phase 1 — lightweight scoring (BM25 + PageRank) to reduce 10B candidate docs to top 1000. Phase 2 — expensive ML ranking (BERT-based cross-encoder) on top 1000. Phase 1 runs on index servers (close to data). Phase 2 runs on dedicated ranking servers (GPU accelerated). This pipeline produces Google-quality relevance while keeping per-query compute cost manageable.

2. **Real-time index for freshness:** News articles must appear in results within minutes. Separate real-time indexing pipeline: crawler detects new/updated page → streaming indexer (Apache Storm/Flink) → in-memory inverted index → merged with base index at query time. Real-time index: smaller (last 24h) + in-memory (Redis sorted sets per term) → fast lookup. Base index: full 100B pages on disk. Query coordinator union of both.

3. **Personalized search:** Signed-in users get personalization signal. Personalization factors: location (city/country), language, past click history (what results you clicked for similar queries), SafeSearch preference. Architecture: personalization profile stored in user data store. At ranking time: for each candidate doc, compute a personalization adjustment (user clicked pages from this domain often → slight boost). Privacy: process personalization in a sandboxed per-request context, never store raw query history on user profile in raw form.

4. **Query understanding — beyond keyword matching:** Modern search understands intent, not just words. (a) Named Entity Recognition: "paris" → city or person? Context determines. (b) Query classification: navigational (user wants specific site), informational (user wants knowledge), transactional (user wants to do something). (c) Semantic equivalence: "buy", "purchase", "order" treated as synonyms. (d) Temporal intent: "weather" → show current weather widget, not web pages. These signals route queries to specialized result handlers (local pack, knowledge panel, shopping results) before standard web results.

5. **Anti-spam and content quality signals:** 30% of web is spam, SEO manipulation, or low-quality. Spam signals: excessive keyword stuffing, link farms, cloaking (serving different content to crawler vs users), thin content (< 300 words). Defense: (a) content quality classifier (ML model trained on human-rated quality data). (b) Link spam detection: links from known spam farms discounted in PageRank. (c) Cloaking detection: re-crawl pages periodically using a real browser user-agent and compare to crawl-agent response. (d) Domain authority: newly registered domains penalized until they accumulate trust signals.
