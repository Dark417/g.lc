# 07. Notification System

## 0) Alex Xu-style discovery path (how to approach in interview)
1. **Clarify scope first**: single region vs global, B2C vs enterprise, consistency expectation, and legal/compliance boundaries.
2. **Back-of-envelope**: QPS, storage/day, peak multiplier, and which component saturates first.
3. **Draw V1 quickly**: API gateway -> stateless service -> cache -> durable storage -> async pipeline.
4. **Pick deepest risks**: usually correctness invariant, hot partition, and failure recovery.
5. **Iterate to V2**: add multi-region, observability, and cost controls.

## 1) Problem framing
- Problem: Design multi-channel notifications with user preferences, retries, and de-duplication.
- Primary users: end users, internal operators, external integrators.
- Success criteria: p99 latency target, availability target, correctness invariant, and cost/QPS objective.
- Non-goals for first round: avoid over-designing rare/edge features until baseline is solid.

## 2) Functional requirements
- Define core APIs with request/response schema and idempotency behavior.
- Support admin operations: policy update, replay/reconciliation, incident controls.
- Support audit trails for critical state transitions.
- Provide backfill/migration flows for schema evolution.

## 3) Non-functional requirements
- Availability target: typically 99.9%-99.99% depending on business criticality.
- Latency target: p50/p95/p99 SLO and max tolerated tail for critical paths.
- Throughput target: sustained + burst traffic with admission control strategy.
- Durability/consistency target: define exactly what must be strongly consistent.
- Security target: AuthN/AuthZ, encryption in transit/at rest, key rotation, and auditability.

## 4) Capacity estimation (interview math)
- Estimate DAU/MAU, requests per user/day, peak factor (often 5x-20x).
- Estimate storage growth: writes/day * payload size * retention + indexes.
- Estimate cache footprint from hotset ratio and TTL strategy.
- Estimate network egress (often hidden cost driver) and cross-region replication cost.

## 5) APIs and contracts
- `POST /v1/requests` for write/create style operations with idempotency key.
- `GET /v1/resources/{id}` for read path and debug metadata.
- `POST /v1/admin/replay` and `POST /v1/admin/reconcile` for operations.
- Error model includes retriable vs non-retriable codes and retry-after hints.

## 6) Data model and storage design
- Core entities defined with immutable event log + mutable projection tables.
- Partition key chosen by dominant access path; secondary index for alternate queries.
- Keep write path append-friendly; derive read models asynchronously where possible.
- TTL/archive strategy for cold data, but preserve compliance/audit retention where required.

## 7) Baseline architecture (V1)
- API gateway enforces authn, quota, and request validation.
- Stateless service layer executes business logic and emits domain events.
- Cache (Redis/memcache) for low-latency hot reads + stampede controls.
- Durable store (SQL/NoSQL/object) for source of truth depending on access patterns.
- Message bus for async fanout, retries, and side effects.

## 8) Mini algorithms / core mechanics
- Outbox pattern from producer DB to MQ ensures at-least-once delivery.
- Idempotency key per notification intent to dedupe retries.
- Channel fallback policy: push->SMS/email depending on urgency + user preference.

## 9) Failure handling and correctness
- Retry with bounded exponential backoff + jitter; include request id for dedupe.
- Timeout budgets propagated hop-by-hop to avoid retry storms.
- Circuit breakers and load shedding to preserve core functionality under stress.
- Dead-letter queue with replay tooling and poison-message quarantine.
- Correctness invariant explicitly tested in reconciliation jobs.

## 10) Scaling strategy
- Horizontal scale stateless tier; shard stateful tier with tenant-aware keys.
- Separate hot/cold paths and isolate noisy neighbors.
- Add asynchronous precomputation for expensive read patterns.
- Use adaptive throttling and per-tenant fairness controls.

## 11) Multi-region and disaster recovery
- Start with active-passive if invariant is strict; move to active-active with conflict policy.
- Replicate logs asynchronously for durability; synchronous only for strict invariants.
- Define RPO/RTO and test disaster game-days quarterly.
- Keep deterministic replay process from immutable logs.

## 12) Security, privacy, and abuse controls
- Principle of least privilege and scoped service identities.
- PII/token handling strategy (tokenization + field-level encryption as needed).
- Abuse pipeline: anomaly detection, rule engine, and manual review hooks.
- Immutable audit log for privileged actions.

## 13) Observability and operations
- Golden signals: latency, error, saturation, traffic.
- Domain KPIs tied to product outcome (conversion, delivery success, correctness rate).
- Tracing across sync + async boundaries using correlation IDs.
- Runbooks: dependency outage, partial data corruption, lag spike, and hot-partition incident.

## 14) Cost and trade-offs
- Cache hit-rate vs memory cost; replication factor vs durability budget.
- Strong consistency scope minimized to invariant-critical writes only.
- Compression/downsampling on analytics paths to reduce storage and query cost.
- Keep a clear “what to cut” list if cost budget is exceeded.

## 15) Interview extension prompts (what interviewer may ask next)
- What changes at 10x traffic and which bottleneck appears first?
- Which invariant can never be violated and how do you prove it?
- How would you support strict compliance region boundaries?
- How do you migrate schema without downtime?

## 16) 5-minute answer script
1. Restate scope and hard constraints.
2. Share scale math and dominant bottleneck.
3. Draw V1 architecture and critical data flow.
4. Deep-dive invariant + failure modes + mitigation.
5. Close with scaling roadmap and tradeoffs.

## 17) Question-specific deep notes

### Concrete API examples
- `POST /notify` {user_id,template_id,channels,priority}
- `PUT /preferences/{user}` channel opt-in/out
- `GET /delivery/{notification_id}` status timeline

### Core data entities
- NotificationIntent(id,user,template,payload_hash,priority)
- DeliveryAttempt(intent_id,channel,provider,status,error)
- Preference(user,channel,quiet_hours,locale)

### Critical path walkthrough
- Producer writes intent to outbox; dispatcher fans out by channel policy.
- Channel workers call provider APIs with idempotency keys.
- Receipts update delivery state machine and analytics.

### Failure scenarios and mitigations
- Provider outage -> failover provider pool per region.
- Duplicate sends -> dedupe table keyed by intent+channel.
- Rate limits by carriers -> adaptive queue pacing.

### Capacity checkpoint
- If 50M/day notifications with 1.4 avg channels, pipeline handles 70M channel-delivery tasks/day.
