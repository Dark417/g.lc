# SDE Knowledge Graph (Interview Order)

## Why this order
Start from **problem framing** (FR/NFR), then quantify with **back-of-envelope**, then choose a **high-level architecture**, then lock down **API** and **data model**. Only then deep-dive subsystems.

## Step-by-step map
1. `01.fr-nfr.md` — Functional + non-functional requirements
2. `02.back-of-envelope-estimation.md` — Capacity, QPS, storage, bandwidth, cost sanity
3. `03.high-level-architecture.md` — Architecture patterns and when to use each
4. `04.api-design.md` — Contract-first API design
5. `05.data-modeling.md` — Entities, access patterns, schema strategy
6. `06.database.md` — SQL/NoSQL engines, indexing, replication, operations
7. `07.caching.md` — Cache patterns, invalidation, consistency edges
8. `08.messaging.md` — Queues, streams, delivery semantics
9. `09.consistency.md` — Transactions, isolation, eventual consistency
10. `10.partitioning.md` — Sharding, rebalancing, hotspot control
11. `11.reliability.md` — Availability, fault tolerance, DR
12. `12.observability.md` — Metrics, logs, traces, SLOs
13. `13.security.md` — AuthN/AuthZ, secrets, abuse resistance
14. `14.performance-capacity.md` — Latency engineering + scaling plan
15. `15.interview-execution.md` — How to communicate and drive the interview

## How to study
- **Pass 1:** 1→5 quickly to build structure.
- **Pass 2:** 6→14 with 2–3 deep dives per week.
- **Pass 3:** Run mock interviews using `15.interview-execution.md`.


## Depth expectations per module
- For each module, explicitly cover: **what it does**, **which problem it solves**, **implementation options**, **component-level differences**, **when to use which**, **popular tools/frameworks**, and **tradeoffs/failure modes**.
- Use interview language: tie each concept to common question types and follow-up probes.
