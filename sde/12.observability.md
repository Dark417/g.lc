## Core Concepts: The Three Pillars

**Why observability matters:** You can't fix what you can't see. A system without observability is a black box — incidents take hours to diagnose instead of minutes.

**Three pillars:**
1. **Metrics:** Numeric measurements over time. Aggregatable. Useful for alerts and dashboards.
2. **Logs:** Discrete events with context. Useful for debugging specific incidents.
3. **Traces:** End-to-end request journeys across services. Useful for latency attribution.

**The difference:** Metrics tell you something is wrong. Logs tell you what happened. Traces tell you where it happened.

## Metrics

**What to measure:**
- **RED method (for services):** Request rate, Error rate, Duration (latency)
- **USE method (for resources):** Utilization, Saturation, Errors (CPU, memory, disk, network)
- **Four golden signals:** Latency, Traffic, Errors, Saturation (Google SRE)

**Metric types:**
- **Counter:** Monotonically increasing (total requests, total errors). Always increasing. Rate of change is useful.
- **Gauge:** Point-in-time value (current memory usage, queue depth, active connections). Can go up or down.
- **Histogram:** Distribution of values (request durations). Records counts in predefined buckets. Enables percentile calculation (p50, p99).
- **Summary:** Pre-calculated percentiles. Cannot aggregate across instances (unlike histogram).

**Cardinality problem:** Adding high-cardinality labels (user_id, request_id) to metrics creates millions of unique time series → explodes storage and query cost. Keep metric labels low-cardinality (endpoint, status_code, region — not user_id).

**Key metrics per component:**
```
HTTP Service: requests/sec, error_rate (5xx%), p50/p99 latency, active connections
Database: query latency, queries/sec, connections in pool, replication lag
Cache: hit_rate, miss_rate, eviction_rate, memory_usage_pct
Queue/Kafka: consumer_lag, messages/sec, partition_count, produce_latency
Disk: read/write IOPS, throughput, saturation (util %), free space
```

## Logging

**Structured logging:** Logs as JSON objects, not free-form text. Machine-parseable.
```json
{
  "timestamp": "2024-07-01T15:30:00Z",
  "level": "ERROR",
  "service": "payment-service",
  "request_id": "req-abc123",
  "user_id": "user-456",
  "message": "Payment failed",
  "psp": "stripe",
  "error_code": "card_declined",
  "duration_ms": 245
}
```

**Log levels:**
- DEBUG: detailed diagnostic info (disable in production unless debugging)
- INFO: normal business events (request received, payment succeeded)
- WARN: unexpected but handled (retry attempt, fallback triggered)
- ERROR: failed operation requiring attention (payment failed, DB connection error)
- FATAL/CRITICAL: system cannot continue (should be rare)

**Log aggregation:** Don't log to local disk in microservices. Ship to centralized log system:
- **Structured log → Fluentd/Filebeat → Elasticsearch → Kibana** (ELK Stack)
- **→ Loki → Grafana** (Grafana stack, cheaper)
- **→ Splunk** (enterprise, expensive)
- **→ AWS CloudWatch Logs** (managed, AWS-native)

**Log retention:** Hot tier (last 7 days): fast, expensive. Cold tier (7-30 days): slower, cheaper. Archive (30+ days): S3/Glacier for compliance.

**Sampling:** High-traffic services can't log every request (too expensive). Log 100% of errors, 10% of slow requests (> 200ms), 1% of normal requests.

## Distributed Tracing

**Why traces:** In microservices, a single user request touches 10+ services. If latency is high, which service is slow? Traces answer this.

**Concepts:**
- **Trace:** End-to-end journey of a single request through all services
- **Span:** One operation within a trace (one service call, one DB query)
- **Context propagation:** Pass trace_id + span_id via HTTP headers (`traceparent` in OpenTelemetry, `X-B3-TraceId` in Zipkin)

**Trace anatomy:**
```
Trace ID: abc123

Span 1: [API Gateway    ] 0ms ─────────────────────────── 250ms (total)
Span 2:   [Auth Service ] 5ms ──── 25ms
Span 3:   [Product Serv ] 30ms ─────────────────── 200ms
Span 4:     [DB Query   ] 35ms ──── 100ms  ← bottleneck
Span 5:     [Cache Fetch] 105ms ─── 15ms
```

**Sampling:** Can't trace every request at high QPS (too much storage). Strategies:
- Head-based: sample at trace start (random %)
- Tail-based: record all; after trace completes, keep only traces with errors or high latency
- Head sampling is simpler; tail sampling captures rare slow paths better

**OpenTelemetry:** Industry standard. Vendor-neutral instrumentation for traces, metrics, logs. Backends: Jaeger, Zipkin, Tempo, AWS X-Ray, DataDog, Honeycomb.

## Alerting and On-Call

**Alert types:**
- **Symptom-based:** Alert on what users experience (error rate, latency). These are direct.
- **Cause-based:** Alert on underlying resources (CPU, memory). Less direct; often noisy.

**Good alerts:**
- Every alert is actionable (has a runbook)
- Alert fires before users notice (or quickly after)
- Low false positive rate (alert fatigue kills on-call culture)
- Clear severity: P1 (wake me up), P2 (fix within 4 hours), P3 (fix this sprint)

**SLO-based alerting:**
```
SLO: 99.9% requests succeed over rolling 30-day window
Error budget: 43 minutes/month

Alert: if error_budget_burn_rate > 14x over last 1 hour → P1 (burn at this rate = budget exhausted in 2 days)
Alert: if error_budget_burn_rate > 6x over last 6 hours → P2
→ Burn rate alerts are more sensitive than threshold alerts
```

**Runbooks:** Every alert should have a runbook: what does this alert mean? What are likely causes? What steps to take? How to verify resolution?

## Dashboards and SLOs

**Dashboard principles:**
- One screen for system health overview (don't bury the lead)
- Separate dashboards: service overview, per-endpoint, infrastructure, business metrics
- Always show error rate, latency (p50, p99), request rate together on same chart (USE+RED)
- Anomaly detection annotations: mark deploys, incidents, config changes on graphs

**SLO tracking:**
- Define SLOs per user journey (not per microservice)
- Track error budget remaining in real-time
- Monthly reviews: is the SLO achievable? Too strict = all budget spent on maintenance. Too loose = users suffering.

## Popular Frameworks

**Prometheus:**
- Pull-based metrics collection (scrapes targets every 15s)
- PromQL: powerful query language for metrics
- Alertmanager: route alerts to PagerDuty, Slack, email
- Best for: Kubernetes-native monitoring
- Client libraries: prometheus-client (Python), client_golang, micrometer (Java)

**Grafana:**
- Visualization layer. Connects to Prometheus, Loki, Tempo, ClickHouse, Elasticsearch.
- Loki: log aggregation (like Prometheus but for logs). Low-cost alternative to ELK.
- Tempo: distributed tracing backend. Integrates with Grafana dashboards.
- Grafana stack (LGTM: Loki + Grafana + Tempo + Mimir) is a common open-source observability stack.

**Datadog:**
- Commercial, SaaS. Metrics, logs, traces, APM in one platform.
- Agent-based: install datadog-agent on each host.
- APM (Application Performance Monitoring): auto-instrumentation for popular frameworks.
- Best for: teams wanting one integrated platform, willing to pay.

**AWS CloudWatch:**
- Managed AWS service. Metrics, logs, dashboards, alarms.
- Container Insights for ECS/EKS. Lambda metrics auto-reported.
- X-Ray: distributed tracing for AWS services.

**ELK Stack (Elasticsearch + Logstash + Kibana):**
- Elasticsearch: search and analytics engine (also used for log storage)
- Logstash / Fluentd / Filebeat: log shippers
- Kibana: visualization and log search UI
- Heavy resource usage. OpenSearch (AWS) is a fork.

**Jaeger / Zipkin:**
- Open-source distributed tracing. OpenTelemetry emits to these.
- Jaeger (CNCF project): Kubernetes-native, Cassandra/Elasticsearch backend.
- Zipkin: older, simpler. B3 propagation format.

**OpenTelemetry:**
- Vendor-neutral SDK for metrics, logs, traces. The new standard.
- Instrument once → send to any backend (Prometheus, Jaeger, DataDog, etc.)
- Collector: receives telemetry, processes, exports to multiple destinations.

## Interview Pitfalls

- **"I'll just add logs"** without mentioning structured logging, aggregation, or log levels.
- **Not mentioning traces.** In microservices, "how do you debug a slow request?" must involve distributed tracing.
- **Alerting on causes, not symptoms.** Alerting on "CPU > 80%" without knowing what user impact that has. Alert on error rate and latency first.
- **No mention of SLOs.** What's your availability target? How do you know when you're violating it?
- **Dashboard overload.** "I'll add 50 metrics to the dashboard" — information overload is as bad as no monitoring. Know the 4 golden signals and start there.
- **Not knowing cardinality.** Adding user_id as a metric label at 1M users = 1M time series. This breaks most metrics systems.
