## Core Concepts

**Why async messaging:** Decouple producers and consumers so they don't need to be up simultaneously. Enable fan-out (one event → many consumers). Buffer against traffic spikes. Enable retry without blocking the caller.

**Message queue vs event stream:**

| Aspect | Queue (RabbitMQ, SQS) | Stream (Kafka, Kinesis) |
|--------|----------------------|------------------------|
| Model | Messages consumed and deleted | Log; consumers track offset |
| Delivery | Each message to one consumer | Same message to many consumer groups |
| Replay | No (once consumed, gone) | Yes (any consumer can re-read from any offset) |
| Retention | Until consumed or TTL | Time-based or size-based (days to weeks) |
| Ordering | FIFO within queue | Ordered within partition |
| Primary use | Task queues, work distribution | Event sourcing, audit log, real-time pipelines |

## Delivery Semantics

**At-most-once:** Message may be lost but never delivered twice.
- Implementation: fire-and-forget. Write to log, don't wait for ACK.
- Use for: log data where some loss is acceptable. Never for financial events.

**At-least-once:** Message may be delivered multiple times but never lost.
- Implementation: producer retries until ACK received. Consumer must be idempotent.
- Use for: almost everything. This is the default for Kafka, SQS, RabbitMQ.

**Exactly-once:** Delivered precisely once. Hardest to achieve.
- Implementation: distributed transaction or idempotency token at consumer.
- Kafka: idempotent producer (sequence numbers) + transactional API (atomic multi-partition writes).
- SQS: no built-in; use application-level idempotency (DB check before processing).
- Use for: payment processing, billing, inventory updates.

**Practical "effectively exactly-once" pattern:**
```
Producer: include a unique message_id in payload
Consumer:
  1. Check if message_id already processed (DB lookup)
  2. If yes: skip (log for monitoring)
  3. If no: process + insert message_id into processed_ids table (atomic transaction)
```

## Ordering and Partitioning

**Why ordering matters:** Updating user balance: must process events in order (deposit, then withdrawal). Processing out of order = wrong balance.

**Kafka ordering guarantee:** Ordered within a partition, not globally.
- Partition key = entity key (user_id, order_id): all events for same entity → same partition → same consumer → processed in order.
- Trade-off: all events for hot entity go to one partition → hot partition problem.

**SQS FIFO queue:** Ordered within a message group. FIFO queues are slower (3K TPS vs 30K TPS for standard queues). Use only when order truly required.

**At-least-once + out-of-order:** With retries, messages can arrive out of order. Solutions:
- Sequence numbers: consumer processes only messages with sequence > last seen.
- Timestamps + dedup window: events within a window can be re-ordered before processing.
- Optimistic concurrency: DB check `WHERE version = expected_version`.

## Consumer Patterns

### Competing Consumers (Work Queue)
```
Producer → Queue → [Consumer 1, Consumer 2, Consumer 3]
Each message processed by exactly one consumer
Use for: distributing work across workers, parallel processing
```

### Fan-out (Publish/Subscribe)
```
Producer → Topic → Consumer Group A → Consumer A1, A2, A3
                 → Consumer Group B → Consumer B1, B2
Same message delivered to all subscriber groups independently
Use for: event distribution (order.created → billing service, email service, analytics)
```

### Event Sourcing
```
All state changes stored as immutable events in append-only log
Current state = replay of all events from beginning
Benefits: audit trail, replay, time-travel debugging
Costs: read complexity, eventual consistency, storage growth
```

### Saga Pattern (for distributed transactions)
```
Choreography: services emit events and react to others' events
  OrderService publishes OrderCreated →
  InventoryService publishes InventoryReserved OR InventoryFailed →
  PaymentService publishes PaymentCharged OR PaymentFailed →
  if any failure: compensating events (InventoryReleased, OrderCancelled)

Orchestration: central coordinator sends commands to services
  SagaOrchestrator: execute(reserve_inventory); on_success: execute(charge_payment); on_failure: execute(release_inventory)
```

## Failure Handling

### Dead Letter Queue (DLQ)
```
Message processing fails repeatedly → after N retries → moved to DLQ
DLQ is a separate queue for manual inspection/replay
Never silently discard failed messages
Monitor DLQ depth as a critical alert
```

### Poison Pill
A message that repeatedly causes a consumer to crash. Eventually hits retry limit → goes to DLQ. DLQ should have monitoring + alerting. Common cause: production data that doesn't match expected format.

### Retry with Backoff
```python
def process_with_retry(message, max_retries=5):
    for attempt in range(max_retries):
        try:
            process(message)
            return
        except TransientError:
            wait = min(2 ** attempt + random.uniform(0, 1), 60)
            time.sleep(wait)
    send_to_dlq(message)
```

Jitter prevents "retry storm" (all consumers retrying simultaneously after an outage).

### Circuit Breaker at Consumer
If downstream service (DB, API) is failing: stop processing, wait, retry.
Prevents overwhelming a recovering service with queued backlog.

## Popular Frameworks

### Apache Kafka
- **When:** High-throughput event streaming, audit log, data pipelines, replay capability required
- **Key concepts:** Topics → Partitions → Offsets. Consumer groups. Replication factor.
- **Throughput:** Millions of messages/sec per cluster. Sequential disk I/O + zero-copy.
- **Retention:** Default 7 days. Unlimited with compaction. Tiered storage to S3 for long-term.
- **Ecosystem:** Kafka Connect (source/sink connectors), Kafka Streams (stream processing), Kafka REST Proxy
- **Client libraries:** confluent-kafka-python, kafka-go, kafka-clients (Java)
- **Managed options:** Confluent Cloud, AWS MSK, Azure Event Hubs (Kafka-compatible)

### AWS SQS (Simple Queue Service)
- **When:** AWS-native, simple queue, work distribution, don't need replay
- **Standard vs FIFO:** Standard = at-least-once, unordered, unlimited throughput. FIFO = exactly-once, ordered, 3K TPS.
- **Visibility timeout:** Message becomes invisible after consumer receives it (default 30s). If not deleted within timeout: requeued. Set to `max(processing_time) × 2`.
- **Long polling:** `ReceiveMessage` waits up to 20s for a message → reduces API calls + latency vs short polling.
- **DLQ:** Configure `maxReceiveCount` → after N receives without delete → move to DLQ.

### RabbitMQ
- **When:** Complex routing (fanout, topic, headers exchanges), lower volume (< 50K msg/sec), message TTL, priority queues
- **AMQP model:** Producers → Exchange → Binding → Queue → Consumer. Exchange types: direct, fanout, topic (wildcard routing key), headers.
- **Acknowledgment:** Consumer sends ACK after processing. Unacknowledged messages requeued on consumer disconnect.
- **Quorum queues:** Raft-based replication for durability. Prefer over classic mirrored queues.
- **Management UI:** Built-in admin console for queue inspection.
- **Cons:** Message replay is not a feature. Not designed for high-throughput streaming.

### Google Cloud Pub/Sub
- **When:** GCP-native, global fan-out, exactly-once semantics built-in (vs manual idempotency)
- **Push vs Pull:** Push: GCP delivers to your HTTP endpoint. Pull: your service polls.
- **Exactly-once delivery:** Available on push subscriptions. Uses `ack_id` dedup.
- **Message ordering:** Optional per-message ordering key.

### Redis Streams
- **When:** Kafka-like but you already have Redis, lower volume, simpler ops
- **Features:** Append-only log, consumer groups, persistent (not in-memory only with AOF), XADD/XREAD/XACK
- **Limitations:** Not designed for multi-TB streams; better for < 1B events

### AWS Kinesis
- **When:** AWS-native Kafka alternative, streaming analytics, Lambda triggers
- **Shards:** Ordered within shard. Scale by adding shards (manual or auto via scaling API).
- **Enhanced fan-out:** Each consumer gets dedicated 2 MB/s throughput per shard (vs shared 2MB/s in standard).
- **Kinesis Firehose:** Managed delivery to S3, Redshift, Elasticsearch. No consumer code needed.

### Celery (Python task queue)
- **When:** Distributed task queue in Python ecosystem, periodic tasks, simple workflows
- **Brokers:** RabbitMQ (recommended), Redis (common), SQS
- **Result backend:** Redis, DB, S3
- **Beat scheduler:** For cron-like periodic tasks

## Read/Write Patterns

**High write throughput:** Kafka partitioned by write key. Producers batch messages. Large batches + compression = efficient.

**High read throughput:** Multiple consumer groups each reading independently. Scale consumers to match partition count.

**Backpressure:** Consumer is slower than producer. Queue depth grows. Solutions: (1) scale up consumers, (2) rate limit producer, (3) shard more (more partitions = more parallel consumers).

**Message ordering with multiple producers:** All producers must write to same partition for ordering to be guaranteed. Solution: partition key = entity_id.

## Interview Pitfalls

- **"I'll use Kafka for everything"** without knowing when a simple queue is better (SQS for task distribution is simpler and doesn't require offset management).
- **Not handling at-least-once correctly.** If you use Kafka/SQS, you WILL get duplicate messages on retries. Show you understand idempotent consumers.
- **Missing DLQ.** Always configure DLQ + alerting. "What happens when processing fails?" must have an answer.
- **Forgetting backpressure.** Producers can generate messages faster than consumers can process. Queue grows unboundedly → OOM or message loss. Must mention consumer scaling or rate limiting.
- **Kafka vs Queue confusion.** Kafka is not a queue — messages are not deleted on consumption. Multiple consumer groups can read the same data. This is the fundamental difference.
- **Not knowing partitioning impact on ordering.** "Kafka guarantees ordered delivery" is wrong — it's ordered within a partition.
