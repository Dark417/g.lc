## Core Concepts

**The first question: what queries do you need to run?** Design schema to serve your access patterns, not to model the domain perfectly. Normalization is a tool, not a goal.

**OLTP vs OLAP:**
- OLTP (Online Transaction Processing): many concurrent short transactions, row-level operations, low latency. PostgreSQL, MySQL, DynamoDB.
- OLAP (Online Analytical Processing): few complex queries, column-level scans, aggregations over large datasets. ClickHouse, BigQuery, Redshift, Snowflake.
- Hybrid (HTAP): TiDB, CockroachDB — can do both, tradeoffs on each.

**ACID:**
- **Atomicity:** Transaction fully completes or fully rolls back
- **Consistency:** DB goes from one valid state to another (constraints always satisfied)
- **Isolation:** Concurrent transactions don't see each other's intermediate state
- **Durability:** Committed transactions survive crashes (WAL)

**BASE (NoSQL trade):**
- **Basically Available:** System is always available (even with stale data)
- **Soft state:** State may change over time due to eventual consistency
- **Eventually consistent:** System converges to consistent state given time

## Relational Databases (RDBMS)

**When to use:**
- Complex relationships and joins
- ACID transactions required (financial, inventory)
- Schema is known and relatively stable
- Rich query flexibility needed (ad-hoc analytics)
- Referential integrity matters

**Key strengths:**
- Joins, foreign keys, constraints: enforce data integrity at DB level
- Rich indexing: B-tree, hash, partial, functional, full-text
- ACID transactions: multi-row, multi-table atomicity
- SQL: expressive query language; complex aggregations, window functions

**Scaling challenges:**
- Vertical scale is limited
- Horizontal sharding requires application-level logic (no transparent distribution)
- Joins across shards don't work
- Write throughput ceiling on single leader

**When not to use:**
- Unbounded horizontal write scale needed
- Schema is highly dynamic or nested documents
- Simple key-value access pattern (overkill)

### PostgreSQL

Most powerful open-source RDBMS. When in doubt, use Postgres.

- **Indexing:** B-tree (default), GiST (geometric), GIN (full-text, JSONB, arrays), BRIN (time-series), SP-GiST. Partial indexes: `CREATE INDEX ON orders(created_at) WHERE status = 'pending'` — only indexes pending orders.
- **JSONB:** Store semi-structured data with indexing. Use when schema varies but you still need relational joins.
- **PostGIS:** Geospatial extension. R-tree indexing for geo queries. `ST_DWithin(location, point, distance)`.
- **Window functions:** Crucial for analytics queries (rank, lag, lead, cumulative sum).
- **CTE (Common Table Expressions):** Readable complex queries; `WITH RECURSIVE` for hierarchical data.
- **pg_partman:** Partition tables by time range. Essential for time-series or large append-only tables.
- **Read replicas:** Stream WAL to replicas. Replicas can serve reads with slight lag (< 100ms typical).
- **Connection pooling:** PostgreSQL struggles with > 500 concurrent connections. Use PgBouncer (transaction-mode pooling) in front of Postgres.

Key settings: `work_mem`, `shared_buffers` (25% of RAM), `effective_cache_size` (75% of RAM), `max_connections`.

### MySQL / Aurora

- Aurora: AWS's MySQL-compatible rewrite. Separates storage from compute. 6-way replication across 3 AZs. Faster failover than vanilla MySQL. Aurora Serverless: scales to zero.
- MySQL binlog: used for replication and CDC (change data capture). Debezium reads binlog to stream changes to Kafka.
- MyRocks: Facebook's storage engine for MySQL. LSM tree instead of InnoDB's B-tree. Better for write-heavy workloads.

## NoSQL Databases

### Document Store — MongoDB, DynamoDB (document mode), Firestore

**When to use:**
- Document-oriented data (products with variable attributes, user profiles)
- Schema flexibility: different documents can have different fields
- Single-entity CRUD access pattern: no JOINs needed
- Hierarchical data that's always read together

**MongoDB:**
- Documents in BSON (binary JSON)
- Rich query operators, aggregation pipeline
- Atlas: managed service with global clusters
- Transactions: multi-document ACID since v4.0
- Indexing: compound, text, geo, TTL (documents auto-expire after N seconds)
- Best for: content management, catalogs, user profiles

**Tradeoff:** Joins are expensive (run in application, or use `$lookup`). Schema flexibility can become schema chaos without discipline.

---

### Wide-Column — Cassandra, HBase, ScyllaDB

**When to use:**
- Massive write throughput (100K+ writes/sec)
- Time-series data
- High cardinality keys with time clustering (event logs, chat messages, metrics)
- Linear horizontal scale is required

**Cassandra data model:**
```
Table = collection of rows
Each row has: partition key + clustering columns + regular columns
Partition key: determines which node stores the row (consistent hashing)
Clustering columns: ordering within a partition; enables range scans

Example: messages by conversation:
  PRIMARY KEY (conversation_id, message_id)
  → All messages for a conversation on the same node
  → Ordered by message_id within partition
  → FAST: single partition reads for "get all messages in conversation X"
```

**Key design rule:** Model your tables around your queries. One table per query pattern. Denormalize aggressively. Writes are cheap; reads that scan many partitions are expensive.

**Cassandra tuning:**
- Replication factor: typically 3 across 3 data centers
- Consistency: `LOCAL_QUORUM` = read/write from majority in local DC (strong local, eventual global)
- TTL: automatic data expiry (use for event logs, sessions, caches)
- Compaction: STCS (write-heavy) vs LCS (read-heavy) vs TWCS (time-series, best for append-only)

**ScyllaDB:** Drop-in Cassandra replacement written in C++. 10x throughput of Java-based Cassandra. Used by Discord, Comcast.

---

### Key-Value Store — Redis, DynamoDB (KV mode), etcd

**Redis:**
- In-memory, extremely fast (100K-1M ops/sec per node)
- Rich data types: String, Hash, List, Set, Sorted Set, Stream, HyperLogLog, Bitmap, Geo
- Persistence: RDB snapshots + AOF (append-only file). Choose `appendfsync everysec` for good balance.
- Cluster mode: hash-slot-based sharding (16384 slots). Automatic rebalancing.
- Sentinel: high-availability for single-shard Redis (leader election, automatic failover).
- Common patterns:
  - Caching (cache-aside)
  - Session storage (TTL-based expiry)
  - Pub/sub (real-time events)
  - Sorted sets for leaderboards/rate limiting
  - Distributed locks (SET NX + expire)
  - Counters (INCR is atomic)

**DynamoDB:**
- AWS-managed, fully serverless, infinite scale
- Key-value or document; single-table design preferred
- GSI (Global Secondary Index): enables different access patterns without table scan
- DynamoDB Streams: change data capture for triggering Lambda
- Provisioned vs On-Demand capacity: On-Demand for unpredictable traffic; Provisioned + auto-scaling for steady traffic
- Hot partition problem: if all traffic goes to one partition key → capacity exceeded. Mitigate with write sharding (add random suffix to partition key).

---

### Graph Database — Neo4j, Amazon Neptune, TigerGraph

**When to use:**
- Data is inherently graph-structured: social networks, knowledge graphs, recommendation graphs, fraud detection (entity relationship)
- Traversal queries: "find all friends-of-friends within 3 hops"
- Relationship-heavy queries that would require many JOINs in SQL

**Neo4j:**
- Property graph model: nodes + edges, both have properties
- Cypher query language: `MATCH (a)-[:FRIEND]->(b)-[:FRIEND]->(c) WHERE a.id = 123 RETURN c`
- Traversal: native graph storage, O(depth) not O(nodes) for graph traversal
- ACID transactions, clustering with Causal Cluster

**When SQL can emulate a graph:** Self-referential join with CTEs for hierarchy. PostgreSQL recursive CTEs work for moderate graph depth. At 10+ hops, graph DB wins.

---

### Time-Series Database — InfluxDB, TimescaleDB, Prometheus, VictoriaMetrics

**When to use:**
- Metrics, sensor readings, financial data: append-only writes, high cardinality, time-range queries
- "What was the CPU usage at 3:15 PM?" or "Average latency per minute in last 24 hours?"

**TimescaleDB:**
- PostgreSQL extension. Time-partitioned (chunks). Normal SQL queries work.
- Automatic compression of old chunks (90% size reduction)
- Continuous aggregates: materialized views updated automatically
- Best of both worlds: SQL flexibility + time-series performance

**InfluxDB:**
- Purpose-built time-series. Line protocol for ingestion. InfluxQL or Flux query language.
- Tags (indexed, low-cardinality): `hostname`, `region`. Fields (not indexed, high-cardinality): `value`.
- Retention policies: auto-delete data after N days.

**Prometheus:**
- Pull-based metrics collection (scrapes targets every 15s)
- PromQL for queries. Alert rules. Excellent for Kubernetes monitoring.
- 15-day local retention. For longer: remote write to Cortex/Thanos/VictoriaMetrics.
- VictoriaMetrics: drop-in Prometheus-compatible with better compression and performance.

## Access-Pattern-First Modeling

**The wrong approach:** Design a normalized schema first, then figure out queries.
**The right approach:** List all queries first, design schema to serve them.

**Steps:**
1. List all read queries with frequency and latency requirement
2. List all write operations with frequency
3. For each query: what's the access key? What data is needed?
4. Design tables / collections around the queries
5. Denormalize if it avoids joins that would be too slow

**Example — chat messages:**
```
Queries needed:
  - "Get last 50 messages in conversation X" (most frequent, < 20ms)
  - "Get all conversations for user Y" (moderately frequent, < 100ms)
  - "Search messages by content" (rare, can be slow)

Cassandra schema:
  messages_by_conversation:
    PRIMARY KEY (conversation_id, message_id DESC)
    → Serves query 1 perfectly

  conversations_by_user:
    PRIMARY KEY (user_id, last_updated DESC)
    → Serves query 2 perfectly

  (text search: Elasticsearch secondary index, async sync)
```

## Indexing Strategies

**B-tree index:** Default. Good for equality and range queries. Sorted order. Works for `=`, `<`, `>`, `BETWEEN`, `ORDER BY`.

**Hash index:** Only equality (`=`). Faster than B-tree for equality but no range. PostgreSQL's hash index is not WAL-logged (use for temporary data). Redis hashes are in-memory.

**Composite index:** Multiple columns. Rule: columns in order of selectivity (most selective first). Query must use leftmost prefix.
```sql
INDEX (status, created_at)
-- Works for: WHERE status = 'active'
-- Works for: WHERE status = 'active' AND created_at > '2024-01-01'
-- Doesn't work for: WHERE created_at > '2024-01-01' (skips status)
```

**Covering index:** Index includes all columns needed by a query. Query is satisfied entirely from the index (no table access). `CREATE INDEX ON orders (user_id, status) INCLUDE (total_amount)`.

**Partial index:** Index only a subset of rows. `CREATE INDEX ON orders(created_at) WHERE status = 'pending'`. Much smaller than full index; faster for queries that only touch pending orders.

**Functional/expression index:** Index a function of a column. `CREATE INDEX ON users (lower(email))` — enables case-insensitive email lookup.

**When indexes hurt:**
- Too many indexes → slow writes (every index must be updated on INSERT/UPDATE/DELETE)
- Low-cardinality columns (boolean, gender) — B-tree scan may be slower than full table scan
- Index bloat: dead index entries after mass deletes → VACUUM/REINDEX

## Interview Pitfalls

- **Jumping to "NoSQL for scale"** without explaining WHY. Cassandra has real tradeoffs (no joins, no ad-hoc queries, schema rigidity in practice).
- **Ignoring access patterns.** Designing a perfect normalized schema and then realizing the primary query requires a full table scan.
- **Not mentioning indexes.** Every query design should consider: is this indexed? What's the index selectivity?
- **"Just add a cache"** without mentioning invalidation strategy.
- **Not knowing CAP theorem.** CA systems (RDBMS) sacrifice partition tolerance. CP systems (ZooKeeper) sacrifice availability. AP systems (Cassandra, DynamoDB) sacrifice consistency.
- **Forgetting N+1 queries.** Fetching a list of 100 users and making 100 separate DB calls for each user's profile = disaster. Use `JOIN` or `IN` clause.
