## Core Concepts

**What interviewers are really evaluating:**
1. **Structured thinking:** Do you approach problems systematically or randomly?
2. **Communication:** Can you explain complex ideas clearly?
3. **Depth of knowledge:** Do you know details when asked to go deeper?
4. **Tradeoff awareness:** Do you understand that every choice has a cost?
5. **Collaboration:** Do you listen, incorporate feedback, and adjust?

**The core mistake:** Treating system design as a knowledge test. It's a conversation. Interviewers want to see how you think, not whether you can recite facts.

**What "Google L4" looks like:**
- Correctly scoped problem with FR/NFR
- Clean high-level architecture covering happy path
- At least 2 substantive deep dives with real detail
- Explicit tradeoff discussion (not just "here's my choice")
- Handles the follow-up questions without falling apart

**What "Google L5" adds:**
- Proactively identifies the hardest problems (not just answers them)
- Drives the conversation; doesn't wait to be asked
- Shows breadth (mentions adjacent concerns: ops, cost, security)
- Makes non-obvious design choices and can defend them rigorously

## The Interview Arc

A 45-minute system design interview should follow this arc:

```
0:00 - 0:05   Clarify scope and requirements
0:05 - 0:10   Propose high-level architecture; get alignment
0:10 - 0:20   API design + data model
0:20 - 0:40   Deep dives (2-3 hard problems)
0:40 - 0:45   Wrap-up: tradeoffs, what you'd do differently, questions
```

**Don't front-load:** Many candidates spend 20 minutes on requirements and run out of time before deep dives. Deep dives are where the differentiation happens.

**Don't be a silent designer:** Think out loud. Say "I'm considering X because Y, but the tradeoff is Z." Silence for more than 30 seconds reads as confusion.

## Problem Framing

**Step 1: Restate the problem in your own words.**
"So we're building a URL shortening service — users submit a long URL and get a short code, which redirects to the original. The key challenge is generating unique short codes at high volume and serving redirects with very low latency. Does that match what you're thinking?"

This serves two purposes:
- Confirms you understood the problem
- Gives the interviewer a chance to correct you early

**Step 2: Define functional requirements (FR) as user stories.**
```
Must have:
  - Shorten a URL → get a short code
  - Visit short URL → redirect to original
  - URLs don't expire by default

Out of scope:
  - User accounts
  - Custom aliases (can add in L5)
  - Analytics (can add in L5)
```

**Step 3: Define non-functional requirements (NFR) with numbers.**
```
- 100M URLs created/day, 10B reads/day (100:1 read:write)
- p99 redirect latency < 10ms (in-region)
- 5 years retention, ~99.9% availability
- URLs must not be guessable (security)
```

**Why numbers matter:** Every NFR number constrains a design decision. 10ms p99 redirect → must cache in Redis or CDN; can't afford a DB round trip.

## Communicating Tradeoffs

**The tradeoff structure:** State the choice → state what you're optimizing for → state what you're sacrificing.

```
"I'm going to use Cassandra for the tweet store rather than PostgreSQL.
 The reason is write throughput: we're at 100K writes/sec, which is beyond
 what a single Postgres primary can handle. Cassandra's LSM tree handles this
 well with linear write scalability.

 The tradeoff: Cassandra doesn't support joins or complex queries. For the
 read path, I'll denormalize the data (pre-compute timelines) to compensate.
 If we needed cross-tweet analytics, we'd need a separate Spark/ClickHouse
 pipeline — but that's a separate concern."
```

**The 3-option framing:** For hard decisions, briefly name 3 options and explain why you chose one.
```
"For ID generation, I see three options:
 1. UUID — trivial but 128 bits, not sortable, wastes index space
 2. DB auto-increment — simple but single point of failure, doesn't scale across shards
 3. Snowflake-style — 64-bit, sortable, distributed, no coordination needed

 I'll go with Snowflake because at this scale we need distributed generation
 with time-ordering. The operational cost of configuring node IDs is acceptable."
```

**When you don't know which is better:** Say so explicitly.
```
"This is a genuine tradeoff and the right answer depends on the actual access
 patterns. If most reads are recent (hot timeline), Redis sorted sets win on
 latency. If the access is random across history, Cassandra is better.
 In practice, I'd start with Redis and measure before adding Cassandra complexity."
```

## Handling Unknowns

**When you don't know a specific detail:**
Don't fake it. Interviewers can probe and you'll fail worse.

```
"I don't know the exact Cassandra compaction algorithm off the top of my head,
 but I know it's an LSM tree with periodic merges to reduce read amplification.
 The key tradeoff is: writes are fast (sequential append), reads may need to
 check multiple SSTables — which is why we add bloom filters to skip irrelevant files."
```

You demonstrate: (1) honesty, (2) you understand the principle even if not the detail.

**When asked about something outside your experience:**
```
"I haven't worked with Flink directly, but I understand stream processing
 conceptually — stateful operators, watermarks for handling late data, exactly-once
 with checkpointing. I'd apply those principles here."
```

**When you realize mid-design that your approach has a problem:**
Don't hide it — name it.
```
"Actually, I just realized that if I store the full fan-out in Redis, for
 a celebrity with 100M followers, that's a 100M-entry write per tweet.
 At 1K tweets/sec from celebrities, that's 100B Redis writes/sec — not feasible.
 I need to switch to a pull model for celebrities. Let me revise..."
```

Catching your own bugs is a positive signal, not a negative one.

## Managing Time

**The 20-minute rule:** If you've spent 20 minutes and haven't gotten to a deep dive, you're behind. Speed up.

**Fast-path through requirements:** Experienced candidates finish requirements in 5-7 minutes by using pre-built templates:
```
"Standard NFRs I'd start with: 99.9% availability (which means < 9h downtime/year),
 p99 latency < 200ms for API, p99 < 50ms for cache reads. Writes can be async where
 consistency allows. Does anything deviate from these for this problem?"
```

**Signal before going deep:** Before spending time on a deep dive, check if the interviewer wants it.
```
"I'd like to go deeper on the fan-out problem — how to handle celebrity accounts
 efficiently. Is that where you'd like me to focus, or is there another area?"
```

**If running out of time:**
```
"We're running short on time — let me quickly cover the two things I haven't touched:
 reliability (we'd add circuit breakers and retry with jitter between services)
 and the data model for the analytics pipeline (separate ClickHouse cluster, fed by Kafka).
 I'm happy to go deeper on either."
```

## Common Failure Modes

**1. The knowledge dump:** Listing technologies without explaining why. "I'd use Kafka, Redis, PostgreSQL, Elasticsearch, Kubernetes..." The interviewer hears noise.
- Fix: For every technology, explain the problem it solves here.

**2. The happy-path architect:** Designing only the success case; ignoring failures.
- Fix: For each component, ask yourself "what happens when this fails?"

**3. The one-size-fits-all answer:** Using the same architecture for every problem regardless of requirements.
- Fix: Let the NFRs drive choices. A URL shortener and a distributed database have different answers.

**4. Interviewer-driven design:** Waiting for the interviewer to ask questions rather than proactively covering ground.
- Fix: After each major section, say "Let me now move to [next topic]." Drive the conversation.

**5. Not listening to hints:** Interviewer asks "how would this work for a celebrity with 100M followers?" and candidate repeats the same answer.
- Fix: Hints are course corrections. Incorporate them.

**6. Ignoring scale:** "Let's just use PostgreSQL for everything" without addressing whether it can handle the load.
- Fix: Always connect your choices to the capacity numbers you established.

**7. Premature optimization:** Spending 10 minutes on Bloom filter false positive rates when the basic architecture isn't established yet.
- Fix: Get the 80% solution right first; then optimize.

## Phrases That Signal Seniority

**Instead of:** "I would use Kafka."
**Say:** "I'd use Kafka here because we need durable, ordered, replayable events. The consumer group model lets us add more workers independently. The alternative was SQS, but we lose ordering guarantees, which matters for the timeline."

**Instead of:** "We need a cache."
**Say:** "The DB read latency here is ~20ms, but our SLO requires p99 < 5ms for this path. We need to cache the hot user profiles in Redis — they're small (~1KB), frequently read, and update rarely. TTL of 5 minutes is acceptable given the staleness tolerance."

**Instead of:** "This might have consistency issues."
**Say:** "This creates a read-your-own-writes problem. After updating your profile, you might read it back from a replica that hasn't caught up yet. We handle this with sticky routing for the 30 seconds after a write — always read from primary after a mutation."

**Instead of:** "What if it fails?"
**Say:** "For the payment service, I'd apply the saga pattern with compensating transactions. If step 3 (capture payment) fails, step 2 (reserve inventory) has a compensating action (release inventory). Each step is idempotent, so retries are safe."

## Interview Pitfalls

- **Being too defensive.** If the interviewer pushes back on your choice, it's not an attack — it's an invitation to discuss tradeoffs. Say "That's a fair point, let me think about that..."
- **Asking too many clarifying questions.** Two or three is fine. Ten is stalling.
- **Starting to code.** System design interviews don't want code (except perhaps a short pseudocode snippet). Use boxes and arrows.
- **Not writing anything down.** Architecture diagrams help both you and the interviewer track the design. Always draw the high-level diagram early.
- **Treating the interviewer as an oracle.** You should drive; they react. Asking "what should I use?" instead of "I'm considering X for Y reason — does that match your expectation?" signals passivity.
- **Neglecting operational concerns.** How do you deploy this? How do you debug it? How do you know when it's unhealthy? Senior engineers think about ops, not just code.
- **Not summarizing.** End by recapping: "So in summary: we have a pull-based architecture for normal users, push-based fan-out for small accounts, a Redis cache for hot timelines, and Cassandra for historical storage. The main tradeoff we made is eventual consistency on the timeline in exchange for write throughput."
