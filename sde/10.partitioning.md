## Core Concepts

**Why partition:** Single node limits: disk capacity, memory, CPU throughput. When data exceeds single-node limits, partition across multiple nodes. A 10TB dataset doesn't fit on one 1TB disk; a 100K QPS workload exceeds one server's capacity.

**Horizontal partitioning (sharding):** Split rows across multiple nodes. Each node holds a subset of rows. Most common for large-scale systems.

**Vertical partitioning:** Split columns across tables or services. Separate rarely-accessed large columns (BLOBs) from frequently-accessed columns. Often used in database normalization or service decomposition.

**Functional partitioning (service decomposition):** Split data by domain. User data in user service, orders in order service. Each service owns its data. This is microservices architecture at the data level.

## Partitioning Strategies

### Range Partitioning

Assign rows to partitions based on a range of the partition key.
```
Partition 0: user_id 1 – 1,000,000
Partition 1: user_id 1,000,001 – 2,000,000

Or by time:
Partition 2024-01: events from January 2024
Partition 2024-02: events from February 2024
```

**Pros:** Efficient range scans. Locality for time-series data.
**Cons:** Hot spots on current-time partition. Uneven distribution.
**When to use:** Time-series data, archive queries needing range access.

### Hash Partitioning

```
partition = hash(key) % N
```

**Pros:** Uniform distribution. No hot spots for random access.
**Cons:** Range scans require scanning all partitions. Resharding is expensive (mitigated by consistent hashing).

### Composite Partitioning

```
Primary key = (tenant_id, user_id)
  → Partition by hash(tenant_id): distribute tenants across shards
  → Cluster by user_id within shard: efficient single-tenant queries
```

Used in: Cassandra (hash partition key + range clustering key), DynamoDB (hash PK + range SK).

### Directory-Based Partitioning

Maintain a lookup table: `key → node`. Flexible but lookup table is a SPOF and bottleneck.
Use for: per-key control (e.g., celebrity accounts needing isolated shards).

## Consistent Hashing

**Problem with naive hashing (`key % N`):** Adding/removing a node requires rehashing almost all keys → massive data migration.

**Consistent hashing:** Map both nodes and keys onto a circular ring. Key goes to the first node clockwise. Adding/removing a node remaps only K/N keys.

**Virtual nodes (vnodes):** Each physical node occupies V positions on the ring (150-200 typical). More uniform distribution. Weighted assignment possible.

**Implementations:**
- Cassandra: Murmur3 hash, 256 vnodes per node by default
- Redis Cluster: 16384 hash slots
- See `sd/2/05.consistent-hashing.md` for full implementation

## Hot Spot Mitigation

**Hot key problem:** One key/partition receives disproportionate traffic (celebrity user, flash sale item).

**Detection:** Monitor per-partition QPS. Alert when one partition is 3x above average.

**Mitigation for reads:**
- Read replicas for hot keys
- Local in-process cache (short TTL)
- Shard cache keys: `user:123:shard:0` through `user:123:shard:9`

**Mitigation for writes:**
```python
# Scatter-gather counter
shard = random.randint(0, 9)
key = f"counter:{entity_id}:shard:{shard}"
INCR key
# Read: SUM of all 10 shards
```

**Time-series hot partition:**
- Salting: `partition = hash(entity_id + hour_bucket)` → spreads writes
- Pre-allocate future partitions distributed across nodes

## Rebalancing

**When:** Node added, node failed, data skew detected.

**Strategies:**

**Fixed partitions (Kafka, Elasticsearch):**
Create many partitions upfront (e.g., 1000 partitions, 10 nodes). Adding a node: move partitions to new node. No rehashing.

**Dynamic partitioning (HBase, MongoDB):**
Partitions split when exceeding a size threshold. Automatic; no pre-planning.

**Consistent hashing (Cassandra):**
New node takes tokens from adjacent nodes. Data streamed in background. Rate-limited.

**Best practices:**
- Require human approval before large rebalancing operations
- Rate-limit migration to avoid network saturation
- Monitor migration progress and latency impact
- Ramp traffic to new node gradually

## Secondary Indexes on Sharded Data

**Local secondary index:** Each shard has its own index. Queries must scatter to all shards and gather results. O(shards) latency.

**Global secondary index:** Dedicated index shard per secondary key. O(1) shard lookup but requires cross-shard write coordination. DynamoDB GSI uses eventual consistency between primary and GSI.

## Popular Frameworks

**Vitess:** Horizontal MySQL sharding. Transparent to applications. Used by YouTube, Slack, GitHub.

**Citus (PostgreSQL):** Distributes PostgreSQL tables. Coordinator routes queries. Best for multi-tenant SaaS.

**CockroachDB:** Automatic range-based sharding. Raft per range. No application-level sharding needed.

**MongoDB Sharding:** Collection-level sharding with mongos router. Automatic chunk migration.

**TiDB:** MySQL-compatible distributed SQL. Automatic sharding via TiKV. HTAP (TiFlash for analytics).

**Cassandra:** Hash partitioning with vnodes. Best for massive write throughput.

## Interview Pitfalls

- **Bad shard key:** `user_id` is good (even distribution). `timestamp` creates hot current partition. `country` hotspots popular countries.
- **Not handling cross-shard queries:** "How do I join data from two shards?" — usually can't do it efficiently. Use materialized views or scatter-gather with awareness of cost.
- **Not mentioning rebalancing:** "What happens when you add a new node?" must have an answer.
- **Over-partitioning:** Too many partitions add coordination overhead. Start with 64-128 per table.
- **Ignoring hot spots at design time:** Identify celebrity user / flash sale scenarios before going to production.
