## Core Concepts

**Latency vs throughput:**
- **Latency:** Time for a single operation to complete (e.g., 50ms per request).
- **Throughput:** Operations per unit time (e.g., 10K requests/sec).
- They are often inversely related: higher throughput can increase latency under contention.
- At low load: latency is near baseline (network + processing). At high load: queueing adds latency.

**Little's Law:** `L = λW`
- L = average number of requests in the system
- λ = throughput (requests/sec)
- W = average time a request spends in the system (latency)

Example: 100 concurrent requests, 100ms avg latency → throughput = 100/0.1 = 1000 req/sec. Inversely: to achieve 10K req/sec at 50ms latency → need 500 concurrent workers.

**Amdahl's Law:** The speedup from parallelizing a program is limited by the fraction that must remain sequential.
```
Speedup = 1 / ((1 - P) + P/N)
P = fraction parallelizable, N = number of processors
If P=0.95, N=100: speedup = 1/(0.05 + 0.0095) ≈ 17x (not 100x)
```

**The Latency Numbers Everyone Should Know:**
```
L1 cache reference:          ~0.5 ns
L2 cache reference:          ~7 ns
Main memory (RAM) access:    ~100 ns
SSD random read (NVMe):      ~100 µs
Network within datacenter:   ~0.5 ms (500 µs)
SSD sequential read (1MB):   ~1 ms
Network SF → NY:             ~40 ms
Network SF → Europe:         ~150 ms
HDD seek + read:             ~10 ms
```

Implication: Redis (in-memory) is 1000x faster than SSD-backed DB for key lookup. Network within datacenter is 5000x faster than SSD random read.

## Latency Budget

**What is a latency budget:** The total allowed time for a request, broken down by step.

```
User expects response in 200ms total
  - Client network (request): 20ms
  - API Gateway: 5ms
  - Auth service: 10ms
  - Business logic: 50ms
  - Database query: 80ms
  - Cache lookup: 1ms
  - Response assembly: 5ms
  - Client network (response): 20ms
  - Margin: 9ms
  Total: 200ms
```

If any step exceeds its budget, the whole request fails the latency SLO. Track P99 per step.

## Throughput and Concurrency

**Horizontal scaling (scale out):** Add more instances. Stateless services scale horizontally trivially. Stateful services require distributed coordination.

**Vertical scaling (scale up):** Larger instance (more CPU, RAM). Cheaper to operate at first; hits ceiling eventually. Good for: databases (scale up before sharding).

**Concurrency models:**
- **Thread per request:** One OS thread per request. Simple but threads are expensive (~1MB stack). Max ~5000 threads per process.
- **Event loop (Node.js, Nginx):** Single thread handles many requests via non-blocking I/O. Excellent for I/O-bound workloads. Single-threaded = no CPU parallelism per process.
- **Async/await (Python asyncio, Go goroutines):** Lightweight coroutines. Go goroutines: ~2KB stack; millions of goroutines possible.
- **Thread pool:** Fixed pool of worker threads. Best for CPU-bound work (image processing, ML inference).

**Connection pooling:** Database connections are expensive (TCP handshake, SSL negotiation, DB auth). Pool maintains N persistent connections, reuses them. Without pool: N requests/sec = N connections/sec = slow. With pool: N requests use 1 connection (sequentially). PgBouncer (PostgreSQL), HikariCP (Java), SQLAlchemy pooling (Python).

## Scaling Patterns

**Stateless services:** Scale horizontally behind a load balancer. Session state in Redis (not local). Configuration from env vars or config service. Container orchestration (Kubernetes) handles auto-scaling.

**Database read replicas:** Add read replicas for read-heavy workloads. Writes go to primary (leader). Reads from replica. Replication lag: 1-100ms. Acceptable for: feeds, recommendations, product catalog. Not for: read-your-own-writes (use sticky routing or primary for those).

**CQRS (Command Query Responsibility Segregation):**
```
Write path (Command): → Command Handler → Write DB (PostgreSQL, normalized)
Read path (Query): → Query Handler → Read DB (Elasticsearch, Cassandra, denormalized)

Sync write→read: event-driven (Kafka CDC)
Tradeoff: more complex, eventual consistency between write and read models
When to use: write and read requirements are very different (scale, schema, latency)
```

**Fan-out vs fan-in:**
- Fan-out: one event triggers many parallel operations (send email + push + SMS simultaneously)
- Fan-in: wait for multiple async operations to complete (fetch from 5 DB shards, merge results)
- Fan-out parallelism: do I/O concurrently when operations are independent

## Database Performance

**N+1 query problem:**
```python
# BAD: N+1 queries
users = db.query("SELECT * FROM users LIMIT 100")
for user in users:
    orders = db.query(f"SELECT * FROM orders WHERE user_id = {user.id}")  # 100 queries!

# GOOD: 2 queries
users = db.query("SELECT * FROM users LIMIT 100")
user_ids = [u.id for u in users]
orders = db.query(f"SELECT * FROM orders WHERE user_id IN ({','.join(user_ids)})")
```

**Query optimization:**
- Use `EXPLAIN ANALYZE` to see actual query plan + costs
- Ensure index is used (not a full table scan)
- Avoid `SELECT *` — select only needed columns
- Avoid `ORDER BY` without index — requires full sort
- Limit result set early (use `WHERE` before `JOIN` when possible)

**Index selectivity:** Index is only useful if it filters a significant fraction of rows. An index on a boolean column is often useless. An index on a unique ID is very useful.

**Batching writes:** Don't write one row at a time.
```sql
-- BAD: 1000 separate INSERT statements
-- GOOD: single batch INSERT
INSERT INTO events (id, type, data) VALUES (1, 'click', '...'),(2, 'view', '...'), ... (1000 rows)
```

## Network Performance

**TCP keep-alive:** Don't establish new TCP connections for every request. Use HTTP/1.1 keep-alive (persistent connections) or HTTP/2 (multiplexed streams on one connection).

**Compression:** Enable gzip/Brotli for HTTP responses. 3-10x smaller payloads. Free performance improvement for text content (JSON, HTML, JS, CSS). Don't compress already-compressed binary (JPEGs, videos).

**CDN:** Serve static assets from CDN edge nodes near users. Eliminates round-trip to origin datacenter. Required for any global service. Cache-Control headers determine CDN caching behavior.

**DNS optimization:** DNS TTL affects failover speed and lookup overhead. Low TTL (60s) = faster failover. High TTL (24h) = fewer DNS lookups. Most clients cache DNS results.

## Load Testing and Benchmarking

**Tools:**
- **wrk:** Simple HTTP benchmarking. `wrk -t12 -c400 -d30s http://localhost:8080/api/users`
- **k6:** Scripted load tests in JavaScript. Supports thresholds, scenarios, output to Grafana.
- **Locust (Python):** User behavior simulation. Good for complex flows (login → browse → purchase).
- **JMeter:** Feature-rich, Java-based. GUI and CLI modes.
- **hey:** Simple HTTP load generator. Like Apache Bench but better.

**What to measure:** p50, p95, p99, p999 latency (not just average). Error rate. Max throughput (requests/sec before latency degrades).

**Benchmark configuration:**
```
Warm-up period: run at 10% load for 60s before measuring (cold JVM, empty cache)
Steady state: measure at target load for 5-10 minutes
Ramp test: gradually increase load to find breaking point
Spike test: sudden 10x load increase — how does system behave?
```

## Interview Pitfalls

- **"Just add more servers"** without knowing the bottleneck. If DB is the bottleneck, more app servers don't help.
- **Not knowing latency numbers.** "How fast is a Redis lookup?" You should know: sub-millisecond (< 0.1ms locally). "How fast is a DB query?" 1-50ms for indexed queries.
- **Confusing latency and throughput.** "It's slow" could mean high latency OR low throughput. They're different problems with different solutions.
- **Missing N+1 query antipattern.** This is one of the most common performance bugs. Know it by name.
- **Not considering connection pool exhaustion.** Services crash under load because they run out of DB connections. Connection pooling is non-optional.
- **Premature optimization.** Profile first, then optimize. "The DB query is the bottleneck" needs data, not assumption.
