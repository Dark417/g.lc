## Core Concepts

**Why consistency is hard in distributed systems:** In a single machine, transactions are guaranteed by the database. Across multiple services or databases, there is no coordinator that can guarantee atomicity — you must build it yourself.

**Consistency vs Availability vs Partition Tolerance (CAP):**
- **CA** (Consistent + Available): impossible under network partition (partitions will happen in real distributed systems)
- **CP** (Consistent + Partition tolerant): sacrifice availability when partitioned. ZooKeeper, etcd, HBase.
- **AP** (Available + Partition tolerant): sacrifice strong consistency during partition. Cassandra, DynamoDB, CouchDB.

**Practical interpretation:** You always have partitions in production (even transient ones). You choose: consistency (CP) or availability (AP). Most large-scale systems choose AP with careful eventual consistency handling.

**PACELC (extension of CAP):**
Even without partitions (P), there's a tradeoff between Latency (L) and Consistency (C):
- Low latency systems: write to local replica immediately → eventual consistency
- Strong consistency systems: wait for quorum write acknowledgment → higher latency

## Isolation Levels

**Why it matters:** Concurrent transactions can interfere in specific ways ("phenomena"). Isolation levels prevent specific phenomena at a cost to throughput.

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Use when |
|-------|-----------|-------------------|-------------|----------|
| Read Uncommitted | Possible | Possible | Possible | Almost never |
| Read Committed | Prevented | Possible | Possible | Default in most OLTP (PostgreSQL, Oracle) |
| Repeatable Read | Prevented | Prevented | Possible | MySQL default; financial reads |
| Serializable | Prevented | Prevented | Prevented | Financial transactions; correctness critical |

**Phenomena defined:**
- **Dirty read:** Read data written by uncommitted transaction that later rolls back
- **Non-repeatable read:** Read same row twice in one transaction, get different values (another committed transaction changed it between reads)
- **Phantom read:** Re-run a range query in one transaction, get different set of rows (rows inserted/deleted by another committed transaction)

**MVCC (Multi-Version Concurrency Control):**
PostgreSQL and most modern DBs use MVCC. Each row stores multiple versions (old committed values). Readers see a consistent snapshot without blocking writers. Writers create new versions without blocking readers. `BEGIN; SELECT... SELECT...` — both SELECTs see the same snapshot. No read-write blocking.

**Serializable Snapshot Isolation (SSI):**
PostgreSQL's implementation of Serializable isolation. Detects write-write conflicts via predicate locks. Aborts the transaction if a cycle is detected. Much better than traditional 2PL locking. Low overhead in read-heavy workloads.

## Distributed Consistency Models

### Strong Consistency (Linearizability)
Every read sees the most recent write. All operations appear to happen atomically at a single point in time. Requires quorum coordination → high latency. Used by: Google Spanner (using TrueTime), etcd, ZooKeeper.

### Sequential Consistency
All operations are in some total order consistent with the order they were performed at each node. No "real-time" guarantee. Slightly weaker than linearizability.

### Causal Consistency
Causally related operations seen in order. Concurrent operations may be seen in different orders by different nodes. Sufficient for most social applications. Used by: some CRDTs, MongoDB Causal Consistency sessions.

### Eventual Consistency
Given no new writes, all replicas will converge to the same value. Reads may see stale data. Highest availability. Cassandra, DynamoDB, DNS. Most internet-scale systems choose this with compensating mechanisms.

### Read-Your-Writes
After you write, you always read your own writes (even if from a different replica). Achieved by: sticky sessions (always read from the same replica), sync on write to designated replica, or client-side versioning. Common requirement that's easy to implement.

## Distributed Transactions

### Two-Phase Commit (2PC)
```
Phase 1 (Prepare):
  Coordinator sends PREPARE to all participants
  Each participant writes to WAL and responds VOTE_COMMIT or VOTE_ABORT

Phase 2 (Commit):
  If all voted COMMIT: coordinator sends COMMIT to all participants
  If any voted ABORT: coordinator sends ABORT to all participants

Participants execute commit/abort and release locks
```

**Problems:**
- Coordinator SPOF: if coordinator fails between phases → participants are blocked waiting (blocking protocol)
- High latency: 2 round trips minimum
- Availability: system is unavailable if any participant is down

**When to use:** Internal services where you control all participants and can tolerate the latency. Not suitable for cross-organization distributed transactions.

### Saga Pattern

Long-running business transactions without locking resources. Each step is a local transaction + compensating transaction if something fails.

**Choreography (event-driven):**
```
Step 1: Order Service creates order → publishes OrderCreated
Step 2: Inventory Service reserves stock → publishes StockReserved (or StockFailed)
Step 3: Payment Service charges → publishes PaymentCharged (or PaymentFailed)

On failure at step 3: Payment Service publishes PaymentFailed →
  Inventory Service listens → publishes StockReleased
  Order Service listens → publishes OrderCancelled
```

**Orchestration (coordinator-based):**
```
SagaOrchestrator manages state machine:
  State: ORDER_CREATED
  → Call: ReserveInventory
  → Success: State = INVENTORY_RESERVED
  → Failure: Call: CancelOrder → DONE

  State: INVENTORY_RESERVED
  → Call: ChargePayment
  → Success: State = COMPLETED
  → Failure: Call: ReleaseInventory, CancelOrder → DONE
```

**Saga tradeoffs:**
- No atomicity — intermediate states are visible
- Compensating transactions must be idempotent
- Choreography: more decoupled but harder to debug (distributed state machine)
- Orchestration: easier to understand and debug; orchestrator is complex but centralized
- Eventually consistent: brief window where state is inconsistent

### Outbox Pattern

**Problem:** "I need to write to DB and send a message to Kafka atomically."
Write to DB fails → Kafka message never sent. Kafka message sent → DB write fails → inconsistent.

**Solution:**
```sql
-- In same transaction:
INSERT INTO orders (id, ...) VALUES (...)
INSERT INTO outbox (id, topic, payload, status='PENDING') VALUES (...)

COMMIT;

-- Separate background process (polling or CDC):
outbox_worker:
  SELECT * FROM outbox WHERE status = 'PENDING' LIMIT 100
  FOR EACH row:
    kafka.publish(row.topic, row.payload)
    UPDATE outbox SET status = 'SENT' WHERE id = row.id
```

Outbox table is in the same DB as business data → atomic write guaranteed by ACID. Background worker publishes to Kafka with at-least-once semantics. Kafka consumer must be idempotent.

## Conflict Resolution

**Last Write Wins (LWW):** Each write has a timestamp. Higher timestamp wins. Simple but lossy — concurrent writes can be silently discarded. Vector clocks can determine if two writes are truly concurrent.

**Vector Clocks:**
```
Each event carries a vector clock: {node_A: 3, node_B: 2}
Event A happened before B if A's clock is element-wise ≤ B's clock
If neither A < B nor B < A: concurrent → conflict
```

**CRDTs (Conflict-free Replicated Data Types):**
Data structures designed to merge automatically without conflicts.
- G-Counter: grow-only counter. Each node has its own counter; total = sum of all.
- PN-Counter: positive + negative G-Counters → can decrement.
- OR-Set: add-wins set. Concurrent add + remove → add wins.
- LWW-Element-Set: last write wins per element.
- Used by: Riak, Redis CRDT mode, distributed databases for AP consistency

## Popular Approaches and Tools

**CockroachDB:**
- Distributed SQL with serializable isolation
- Powered by Raft consensus per range (contiguous key ranges)
- Globally consistent with no hot spots
- Compatible with PostgreSQL wire protocol
- Best for: multi-region strong consistency

**Google Spanner:**
- Globally distributed relational database with external consistency
- TrueTime API: uses GPS + atomic clocks to bound clock skew
- Writes commit at a timestamp guaranteed to be unique and monotonic
- Best for: global financial data, globally consistent applications
- Very expensive; use Cloud Spanner

**Apache ZooKeeper / etcd:**
- Coordination service: distributed lock, leader election, config management
- Raft consensus (etcd) / ZAB consensus (ZooKeeper)
- Strong consistency: linearizable reads and writes
- Not a general-purpose database — small data, high coordination overhead
- etcd is preferred for new systems; ZooKeeper is legacy (Kafka KRaft mode replaces ZK)

**Temporal (workflow engine):**
- Manages long-running workflows with automatic retries, timeouts, compensation
- Replays workflow from event history for deterministic execution
- Best for: complex business processes, saga orchestration
- Used by: Netflix, Uber, Datadog

## Interview Pitfalls

- **"I'll use a distributed transaction"** without knowing the cost (latency, coordinator SPOF, blocking). Evaluate if eventual consistency + saga is acceptable first.
- **Confusing consistency models.** "Eventual consistency" means "will converge eventually" — it does NOT mean "might lose data." Lost writes are a durability problem, not a consistency problem.
- **Not knowing the difference between ACID isolation and distributed consistency.** ACID is within a single DB. Distributed consistency is across services.
- **"Just add 2PC"** — 2PC is blocking and has a coordinator SPOF. Saga is usually preferable.
- **Ignoring compensating transactions.** "What happens if the payment succeeds but the inventory reservation fails?" must have an answer.
- **Not mentioning idempotency for sagas.** Compensating transactions will be retried. They MUST be idempotent.
