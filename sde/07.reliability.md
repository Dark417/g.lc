## Core Concepts and SRE Fundamentals

**SLA vs SLO vs SLI:**
- **SLI (Service Level Indicator):** Actual measurement. e.g., "error rate = 0.1% over last 24h"
- **SLO (Service Level Objective):** Target for an SLI. e.g., "error rate < 0.5%, p99 latency < 200ms, 99.9% of time"
- **SLA (Service Level Agreement):** Contract with customers. Breach → penalties/refunds. Usually less strict than internal SLO.

**Error budget:** SLO = 99.9% availability → 0.1% downtime budget per month = 43 min/month. When budget is exhausted: freeze feature work, focus on reliability.

**The four golden signals (Google SRE):**
1. **Latency** — time to serve a request (p50, p99, p999)
2. **Traffic** — requests/sec (QPS, messages/sec, etc.)
3. **Errors** — rate of failed requests (5xx, timeouts)
4. **Saturation** — how full the system is (CPU %, queue depth, disk %)

## Retry Patterns

**When to retry:** Transient errors only (network timeouts, temporary unavailability). Never retry on client errors (400, 422) or resource not found (404).

**Exponential backoff with jitter:**
```python
def retry_with_backoff(fn, max_retries=5, base_delay=0.1, max_delay=30):
    for attempt in range(max_retries):
        try:
            return fn()
        except TransientError:
            if attempt == max_retries - 1:
                raise
            # Full jitter: random(0, min(cap, base * 2^attempt))
            delay = random.uniform(0, min(max_delay, base_delay * (2 ** attempt)))
            time.sleep(delay)
```

**Why jitter:** Without jitter, all clients retry simultaneously after an outage → retry storm → makes the outage worse. Jitter spreads retries over time.

**Idempotency:** Retrying a non-idempotent operation (POST without idempotency key) can create duplicate resources. Always make retried operations idempotent.

**Retry budget:** Don't retry more than N times total across the request chain. Prevent retry amplification: A retries → B retries → C retries = 8x load.

## Circuit Breaker

**Problem:** Service A calls Service B which is failing. Each call waits for timeout (e.g., 30s). All A threads are blocked → A itself becomes unavailable.

**Circuit breaker states:**
```
CLOSED (normal)  → error_rate < threshold → stay CLOSED
                 → error_rate > threshold (e.g., 50%) → OPEN

OPEN (fast-fail) → immediately return error (don't call B)
                 → after timeout (e.g., 30s) → HALF-OPEN

HALF-OPEN (probe) → allow 1 request through
                  → success → CLOSED
                  → failure → OPEN again
```

**Configuration:** error_threshold (50%), measurement_window (100 requests), open_timeout (30s), half_open_requests (1-5).

**Libraries:** Netflix Hystrix (Java, legacy), resilience4j (Java, modern), Polly (.NET), py-circuitbreaker (Python), go-circuit-breaker.

## Timeouts and Deadlines

**Why timeouts matter:** Without timeout, one slow upstream can block all your threads. At 100 RPS and 100ms timeout: max 10 concurrent requests. At 100 RPS and 30s timeout: max 3000 concurrent requests blocked.

**Timeout types:**
- **Connection timeout:** Time to establish TCP connection. Should be short (1-5s).
- **Read timeout:** Time to wait for the first byte after connection. Set to expected max processing time.
- **Request timeout:** Total time from request start to response complete.
- **Deadline propagation:** Pass a deadline header through your entire call chain. Each hop should reduce the deadline by its own budget. If deadline already exceeded: return immediately without calling downstream.

```
Request enters with deadline: 500ms
  → API layer: 50ms budget → calls auth service
  → Auth service: 20ms budget → calls DB
  → DB: 15ms query
  → Returns to API (50ms elapsed)
  → API calls business logic: 300ms budget → ...
Total: 500ms

If auth takes 200ms: remaining budget is 300ms → API adjusts downstream calls accordingly
If deadline already passed: return 503 immediately, don't call any more downstream services
```

**gRPC deadlines:** Built-in. Context propagates deadline automatically through the call chain.

## Bulkhead Pattern

**Problem:** Slow requests in one use case consume all thread pool capacity, blocking other use cases.

**Solution:** Isolated thread pools (or semaphores) per use case.
```
Thread pool A (100 threads): user API requests
Thread pool B (20 threads): expensive report generation
Thread pool C (10 threads): background jobs

If report generation is slow → only pool B is exhausted → user API still works
```

**Semaphore-based bulkhead:** Limit concurrent calls to a downstream service.
```python
semaphore = Semaphore(10)  # max 10 concurrent calls to payment service

def charge_payment(amount):
    with semaphore:
        return payment_service.charge(amount)
```

**Microservice bulkhead:** Each service has its own resources (DB connection pool, thread pool). Slow service doesn't affect others. This is a key benefit of microservices over a monolith.

## Graceful Degradation

**Principle:** When a component fails, the system should degrade gracefully (serve reduced functionality) rather than failing completely.

**Patterns:**

**Fallback to cache:** Service is down → return last cached value (stale data is better than no data for many use cases).

**Default/empty response:** Recommendation service is down → return top-10 popular items (not personalized, but functional).

**Feature shedding:** Under extreme load, disable non-essential features. Shedding order: marketing > analytics > secondary features > core flow.

**Health check + load shedding:**
```
if (cpu_usage > 95% or memory > 90%):
    reject requests with 503 (with Retry-After header)
    → Better to reject quickly than to fail slowly
```

**Read vs write fallback:** Writes are more critical than reads. Under partial failure: accept writes (queue them), degrade reads (return stale data).

## Disaster Recovery

**RTO (Recovery Time Objective):** How long can the system be down before unacceptable impact? Minutes for payment systems; hours for reporting.

**RPO (Recovery Point Objective):** How much data loss is acceptable? Zero for financial systems; minutes for user activity logs.

**DR strategies:**

**Backup and restore (cheapest, highest RTO):**
- Periodic backups (hourly or daily) to off-site storage (S3, Glacier)
- On failure: restore from backup → RTO = hours; RPO = time since last backup
- Use for: development environments, non-critical data

**Pilot light:**
- Core infrastructure always running (DB, load balancers) but minimal capacity
- On failure: scale up quickly (minutes-hours)
- RTO: 1-4 hours; lower cost than full active-standby

**Warm standby:**
- Scaled-down but fully functional replica running
- On failure: scale up from warm standby
- RTO: minutes; higher cost than pilot light

**Active-active multi-region:**
- Traffic split across regions
- On region failure: redirect all traffic to remaining regions
- RTO: seconds (just DNS failover); RPO: near-zero
- Highest cost and complexity

**Chaos Engineering:**
- Deliberately inject failures in production to test recovery procedures
- Netflix Chaos Monkey: randomly terminates EC2 instances
- Netflix Chaos Kong: terminates entire AWS regions
- GameDay: simulate major failures with full engineering team response

## Popular Frameworks and Tools

**resilience4j (Java):** Circuit breaker, retry, rate limiter, bulkhead, timeout. Functional style. Spring Boot integration. Production-ready.

**Polly (.NET):** Retry, circuit breaker, timeout, bulkhead, fallback. Fluent API.

**go-resiliency / failsafe-go (Go):** Circuit breaker + retry for Go services.

**Envoy Proxy:** L7 proxy with built-in circuit breaking, retry policies, timeout, health checking. Used as sidecar in service mesh. Configured via xDS API (Kubernetes integration via Istio).

**AWS Auto Scaling:** Horizontal scale-out on CPU/memory/QPS metrics. Handles: capacity for peak traffic, auto-replace unhealthy instances.

**AWS Route53 health checks:** Failover routing: primary region health check fails → Route53 points to secondary region. DNS TTL = 60s (typical failover time).

**PagerDuty / Opsgenie:** Alert routing and on-call management. Escalation policies. Incident management.

## Interview Pitfalls

- **Not specifying timeout values.** "I'll add retries" without saying the timeout is incomplete. Set connection timeout (1-2s) and read timeout (2× expected p99).
- **Retrying non-idempotent operations.** Always check: can this be safely retried? If not, add idempotency key first.
- **Missing jitter on retries.** Retry storms are real. Always add jitter.
- **No fallback plan.** "What if Redis is down?" must have an answer. Don't say "it won't happen."
- **"Active-active is always better."** Active-active has higher cost and operational complexity. Match DR strategy to actual RTO/RPO requirements and budget.
- **Confusing RPO and RTO.** RPO = data loss tolerance. RTO = downtime tolerance. Both matter; they're different questions.
