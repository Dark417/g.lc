## What this topic does and why it matters
Requirements framing converts a vague prompt into an actionable engineering contract.

It solves three interview and real-world problems:
1. **Wrong-system risk**: teams build a technically strong system that does not match user/business needs.
2. **Over/under-engineering risk**: without explicit scale and SLO targets, architecture choices drift into guesswork.
3. **Misaligned tradeoffs**: consistency, availability, and cost priorities stay implicit and create hidden conflicts later.

If this step is weak, all downstream choices (API, DB, queue, caching, partitioning) are likely suboptimal.

## FR: what problem it solves and how to write them well
Functional requirements define expected user-visible behavior.

### FR quality bar
A strong FR is:
- **Observable** (can be tested from outside)
- **Atomic** (one behavior per statement)
- **Priority-tagged** (`must`, `should`, `nice-to-have`)
- **Boundary-aware** (explicit out-of-scope edges)

### FR writing patterns (implementation options)
#### Option A: User-story style
- “As a seller, I can create a listing with title, price, and inventory.”
- Best for product-facing systems and ambiguous prompts.

#### Option B: Use-case/action style
- “CreateOrder validates payment method, reserves inventory, and returns order ID.”
- Best for backend/API-heavy prompts.

#### Option C: Event-driven behavior style
- “On payment success, emit `order.paid` event within 500ms.”
- Best for async pipelines and microservice interviews.

### FR anti-patterns
- FR stated as implementation (“use Redis for session”) instead of behavior.
- Missing failure behavior (timeouts, retries, idempotency expectations).
- No admin/moderation/abuse ops for consumer-facing systems.

## NFR: constraints that drive architecture
NFRs encode performance and risk tolerances.

### Core NFR dimensions and what they decide
- **Scale (QPS, data growth)** → partitioning strategy, storage engine, queue throughput.
- **Latency (p95/p99)** → cache strategy, fanout model, region placement.
- **Availability** → redundancy model (active-passive vs active-active).
- **Consistency** → transaction model, conflict resolution, user-visible semantics.
- **Durability (RPO/RTO)** → backup/replication/DR architecture.
- **Security/compliance** → auth model, data segregation, key management.
- **Cost ceiling** → service tier, retention policy, storage class, compute headroom.

### Good NFR examples
- “p99 timeline read < 200ms at 150k peak QPS.”
- “Order placement must be exactly-once from API client perspective.”
- “RPO < 5 minutes, RTO < 30 minutes for regional outage.”

## Requirement decomposition methods (ways to implement the process)
### Method 1: User-journey decomposition
Break by end-to-end journey (onboard → create → consume → recover).
- **Pros**: intuitive, good for product systems.
- **Cons**: can miss backend operational constraints.

### Method 2: Domain-bounded decomposition
Break by bounded contexts (identity, catalog, order, payment, notification).
- **Pros**: aligns with microservice ownership.
- **Cons**: may hide cross-domain consistency needs.

### Method 3: SLO-first decomposition
Start from latency/availability targets, then derive required capabilities.
- **Pros**: strong for infra-heavy and high-scale problems.
- **Cons**: can under-spec user-facing behaviors if overused.

## Component-level impact map
| Requirement type | Typical component impact | Example decision |
| --- | --- | --- |
| High read QPS | CDN + cache + read replicas | Add Redis + cache-aside |
| Strict consistency on writes | Primary DB + tx pattern | Use outbox + idempotency key |
| Strong auditability | Immutable log/event store | Event-sourced order ledger |
| Low-latency global reads | Multi-region read topology | Geo-routing + regional caches |
| Tight compliance | Data governance pipeline | Field-level encryption + tokenization |

## Popular frameworks/templates and when to use them
- **RFC 2119 language** (`MUST`, `SHOULD`, `MAY`) for precise requirement docs.
- **C4 model + ADRs** for mapping requirements to architecture decisions.
- **SLO/SLI templates** (Google SRE style) for NFR precision.
- **PRD-lite + NFR appendix** for interview prep and faster stakeholder alignment.

## Which interview questions this is most critical for
- **Marketplace/order/payment systems**: consistency and failure semantics dominate.
- **Social feed/chat/notification**: latency, fanout, and eventual consistency semantics dominate.
- **Search/recommendation**: freshness vs ranking quality vs compute cost.
- **Storage/infra systems**: explicit durability and recovery targets are essential.

## Tradeoffs and failure modes
### Tradeoff 1: Broad scope vs time depth
- Broad scope improves completeness.
- Narrow scope improves depth and correctness.
- Interview strategy: define full scope quickly, then lock MVP for design depth.

### Tradeoff 2: strict NFR targets vs cost
- Strong p99/availability targets increase infra and operational cost.
- Say explicitly what target you relax under budget constraints.

### Common failure modes
- No numeric targets.
- No peak-vs-average distinction.
- No degraded-mode behavior.
- No explicit assumptions listed.

## Interview script (high-signal, low-time)
1. Restate product in one sentence.
2. Confirm 3–5 FRs with priority labels.
3. Confirm 5–7 NFRs with numbers.
4. Declare out-of-scope items.
5. State one key tradeoff (consistency vs latency/cost) before architecture.
