## What estimation does and what problem it solves
Back-of-envelope estimation turns qualitative scale claims into quantitative design constraints.

It solves:
- **Capacity ambiguity**: prevents architecture from being chosen without load numbers.
- **Bottleneck blindness**: reveals likely choke points (storage IOPS, network egress, queue lag).
- **Cost surprises**: highlights expensive components early (egress, managed DB replicas, hot storage).

## Estimation dimensions and formulas
### Throughput
- `avg_qps = daily_requests / 86,400`
- `peak_qps = avg_qps × burst_factor` (3–10 typical)
- `write_qps = total_qps / (read_write_ratio + 1)`

### Storage
- `raw_storage = writes_per_day × bytes_per_record × retention_days`
- `effective_storage = raw_storage × replication_factor × overhead_factor`
- Overhead factor usually `1.2–1.5` (indexes, compaction, metadata).

### Network
- `ingress_bw = request_qps × avg_request_size`
- `egress_bw = response_qps × avg_response_size`
- Add internal traffic: replication + cross-AZ + cache miss amplification.

### Concurrency
- Little’s Law: `L = λW`
- Useful for worker pool sizing and queue backlog reasonability.

## Ways to estimate (top-down vs bottom-up vs hybrid)
### Option A: Top-down estimation
Start from DAU/MAU and actions per user.
- **Best for**: product-facing interviews with user-scale prompts.
- **Pros**: fast and interviewer-friendly.
- **Cons**: weak precision for backend-heavy subcomponents.

### Option B: Bottom-up estimation
Start from operation costs (bytes/op, DB writes/op, events/op).
- **Best for**: infra, storage, and stream-processing questions.
- **Pros**: precise component sizing.
- **Cons**: slower and assumption-heavy.

### Option C: Hybrid estimation (recommended)
Top-down for global scale, bottom-up for hot paths.
- Best overall interview approach.

## Component-by-component impact
| Estimated metric | What it influences | Common implementation choice |
| --- | --- | --- |
| Peak read QPS | cache tier and read replicas | Redis + CDN + read replica fanout |
| Write QPS | partition key and DB engine | Cassandra/Dynamo for high write load |
| Retention TB/year | storage class and archival | Hot/warm/cold tier with lifecycle policy |
| Event throughput | broker partition count | Kafka partition planning |
| Egress bandwidth | region + CDN strategy | Edge cache, image/video optimization |

## Popular tools/frameworks and where they fit
- **Spreadsheets/Notion templates**: fastest for interview and early architecture.
- **AWS/GCP calculators**: cloud cost sanity checks.
- **Locust/k6/JMeter**: validation once prototype exists.
- **Prometheus/Grafana**: compare estimated vs observed after deployment.

## Common system-design question mappings
- **Rate limiter / API gateway**: estimate request cardinality and token bucket refill rates.
- **URL shortener / key-value service**: key count growth, redirect read skew.
- **Chat/feed systems**: fanout amplification and hot-key behavior.
- **Video/content platforms**: egress dominates; CDN hit-ratio assumptions critical.

## Tradeoffs and uncertainty handling
### Tradeoff 1: speed vs precision
- Fast estimates are coarse but sufficient for architecture screening.
- Precise estimates consume interview time.
- Strategy: provide ranges and sensitivity (low/base/high).

### Tradeoff 2: conservative sizing vs cost
- Conservative sizing improves reliability but raises baseline cost.
- Aggressive sizing saves cost but increases incident risk under burst.

### Handle uncertainty explicitly
State assumptions and sensitivity:
- “If average payload doubles, egress doubles linearly.”
- “If burst factor rises from 5x to 8x, shard count increases by ~60%.”

## Interview estimation playbook
1. Declare assumptions table (users, actions/day, payload sizes, retention).
2. Compute `avg + peak` QPS.
3. Compute hot storage + annual growth.
4. Compute major bandwidth edges.
5. Map each number to one architecture decision.
6. Add 20–30% headroom and call out confidence level.
