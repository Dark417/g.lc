## Core Concepts

**Why cache:** Reduce latency, reduce database load, increase throughput. A DB query that takes 20ms → cache hit takes 0.1ms (200x faster).

**Cache hit ratio:** (hits / total requests). Target > 90% for effective caching. If hit rate is < 50%, your cache key or TTL strategy is wrong.

**Cache types by location:**
- **Client-side cache:** Browser cache (HTTP cache headers). Zero server cost.
- **CDN cache:** Edge nodes cache content globally. Best for static/semi-static assets.
- **Reverse proxy cache:** Nginx, Varnish in front of app servers. Good for response caching.
- **Application-level cache:** Redis/Memcached called from application code. Most flexible.
- **Database cache:** Query result cache (MySQL query cache — deprecated; most DBs don't cache). Buffer pool (in-memory pages of B-tree).
- **CPU cache:** L1/L2/L3 — relevant for data-intensive code, not system design.

## Cache Placement Strategies

### Cache-Aside (Lazy Loading) — Most Common

```
Read:
  value = cache.get(key)
  if value is None:
    value = db.query(key)
    cache.set(key, value, ttl=300)
  return value

Write:
  db.write(key, new_value)
  cache.delete(key)  # or cache.set(key, new_value)
```

**Pros:** Only caches what's actually requested. Cache can go down without affecting correctness. Simple to implement.
**Cons:** Cache miss = 3 operations (read cache, read DB, write cache). Initial cold start has many misses. Risk of stale data if write path skips invalidation.

### Write-Through

```
Write:
  db.write(key, new_value)
  cache.set(key, new_value)  # update cache synchronously on every write

Read:
  value = cache.get(key)
  if value is None:
    value = db.query(key)  # should be rare; cache always populated
    cache.set(key, value)
  return value
```

**Pros:** Cache always fresh (writes go to both). No stale reads.
**Cons:** Every write touches cache (even for data never read again). Write latency includes cache write. Cache gets polluted with infrequently-read data.

### Write-Behind (Write-Back)

```
Write:
  cache.set(key, new_value)  # write only to cache (fast)
  queue.push({key: key, value: new_value})  # async write to DB

Background worker:
  while True:
    item = queue.pop()
    db.write(item.key, item.value)
```

**Pros:** Very fast writes (in-memory only).
**Cons:** Data loss if cache goes down before DB write. More complex. Risk of out-of-order writes.
**Use when:** Write volume is very high and slight data loss is acceptable (counters, analytics).

### Read-Through

Same as cache-aside but the cache library handles the DB call transparently. Application only calls cache. Cache framework handles miss → DB fetch → cache populate. Less common; requires smart cache client.

## Eviction Policies

**LRU (Least Recently Used):** Evict the item not accessed for the longest time. Best for temporal locality (recently used items likely to be used again). Default in most systems.

**LFU (Least Frequently Used):** Evict the item accessed fewest times. Better for workloads with hot/cold distinction that don't change. More complex to implement.

**ARC (Adaptive Replacement Cache):** Hybrid of LRU and LFU. Maintains two lists and adapts based on recent hit patterns. Used in ZFS, some Linux filesystems.

**FIFO:** Evict oldest inserted item. Simple but ignores access patterns. Generally inferior to LRU.

**Random:** Evict a random item. Surprisingly competitive with LRU in many real-world workloads (simpler implementation, similar hit rate).

**TTL-based expiry:** Separate from eviction. Items expire after a fixed time regardless of access pattern. Use for: data with known staleness tolerance, session data, rate limit counters.

## Cache Invalidation

"There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

**Strategies:**

**TTL-based (simplest):** Cache entries expire after N seconds. Simple, predictable. Allows stale reads up to TTL duration. Good for: tolerable staleness (user profiles, product catalog, recommendations).

**Event-driven invalidation:** On DB write, explicitly invalidate or update cache.
```python
def update_user(user_id, new_data):
    db.update(user_id, new_data)
    cache.delete(f"user:{user_id}")  # or cache.set(...)
```
Consistent but tightly coupled. Race condition: between DB write and cache delete, another reader might re-populate cache with old data.

**Cache stampede (thundering herd):** When a popular cached item expires, many concurrent requests all miss → all hit the DB simultaneously → DB overload → slow responses → more cache misses → death spiral.

**Mitigations for stampede:**
1. **Mutex/lock on cache miss:** Use a distributed lock (Redis SETNX). First request acquires lock, fetches from DB, populates cache. Others wait on lock or return stale value.
2. **Probabilistic early expiration (PER):** Before TTL expires, recompute with probability proportional to time remaining. Prevents sudden hard expiry.
3. **Background refresh:** Cache entry is served stale but triggers async background refresh when near expiry. Client always gets a response.

## Distributed Caching Patterns

### Redis Cluster

Sharding: 16384 hash slots divided across nodes. `CRC16(key) % 16384` → slot → node.
- Adding a node: slots migrate. Short period of `MOVED` redirects.
- Replication: each master has N replicas. On master failure: replica promoted.
- Hot key problem: single key with very high access rate → one shard is overwhelmed. Mitigation: read replicas for this key, or split key into N variants (read from all, write to all).

### Memcached vs Redis

| Feature | Memcached | Redis |
|---------|-----------|-------|
| Data types | String only | String, Hash, List, Set, Sorted Set, Stream, ... |
| Persistence | No | RDB + AOF |
| Clustering | Client-side hashing | Native cluster |
| Pub/Sub | No | Yes |
| Lua scripting | No | Yes |
| Memory efficiency | Slightly better for pure strings | More overhead per key |
| Multi-threading | Yes (multi-threaded) | Single-threaded event loop (Redis 6+ has I/O threads) |

**Choose Memcached when:** Pure caching, maximum memory efficiency, simple strings, don't need persistence or complex data structures.
**Choose Redis:** Almost always. The rich data types, persistence, and clustering are worth it.

### Local vs Distributed Cache

**Local (in-process) cache (e.g., Guava LoadingCache, Caffeine):**
- Zero network latency (nanoseconds vs milliseconds)
- Each server has its own copy → cache is not shared → inconsistency between servers
- Used for: configuration, lookup tables, data that changes rarely
- Combined approach: L1 = local in-process cache (TTL 5s) + L2 = Redis (TTL 5min). Very low latency for hot data, fallback to Redis for L1 miss.

## Failure Modes

**Cache availability:** If cache is down and you designed as "cache-aside with DB fallback": DB gets full traffic → DB may overload. Mitigate: circuit breaker on DB calls, fallback to simpler/cached responses.

**Cache poisoning:** Storing invalid data in cache (application bug, data corruption). Mitigation: cache checksums, short TTL, easy purge API.

**Cache coherence bugs:** DB is updated but cache still has old value. Most common cause of "why is the old data showing?" Systematic cache invalidation + logging all cache misses.

**Memory pressure:** Cache is evicting items too aggressively → hit rate drops. Monitor eviction rate and increase cache memory. Or: better TTL tuning, better key design (avoid caching data that won't be read again).

## Popular Frameworks

**Redis (remote dictionary server):**
- **When:** Shared cache across multiple servers; need complex data structures; need persistence
- **Key config:** `maxmemory-policy allkeys-lru` (evict LRU when full); `save 900 1` (RDB snapshot); `appendonly yes` (AOF persistence)
- **Clients:** redis-py (Python), ioredis (Node.js), Lettuce (Java async), Redisson (Java rich client)
- **Sentinel vs Cluster:** Sentinel for HA on single shard. Cluster for horizontal sharding.

**Memcached:**
- **When:** Pure cache, maximum throughput, simple use case
- **Key feature:** CAS (Check And Set) for atomic updates
- **Clients:** python-memcached, pylibmc, spymemcached (Java)

**Caffeine (Java in-process):**
- Best-in-class JVM cache. Window TinyLFU eviction (better than pure LRU).
- Async loading, refresh-after-write, maximum size policies.
- Used internally by many frameworks (Spring, Hibernate second-level cache).

**Guava Cache (Java):**
- Simpler than Caffeine, part of Google Guava library. Good for straightforward cases.

**Varnish:**
- HTTP reverse proxy cache. ESI (Edge Side Includes) for partial page caching.
- Complex VCL (Varnish Configuration Language) for cache logic.
- Used as CDN-origin cache layer.

**nginx proxy_cache:**
- Nginx can cache upstream responses. Good for: API gateway response caching, static file serving.
- Simple configuration: `proxy_cache_path`, `proxy_cache_valid`, `proxy_cache_bypass`.

## Interview Pitfalls

- **"I'll just add Redis"** without explaining what's cached, what the key is, what the TTL is, and how invalidation works.
- **Ignoring thundering herd.** Popular cache expiry → DB overload. Must mention mitigation.
- **Caching mutable data without invalidation.** Cache the user's balance: user makes a purchase → balance changes → cache is stale. Need event-driven invalidation.
- **Using cache for user authentication state** without thinking about logout/revocation. If you cache "user X is authenticated", you need a way to invalidate that on logout or permission change.
- **Wrong TTL:** Too short = high cache miss rate, DB overload. Too long = stale data for too long.
- **Not knowing LRU complexity:** LRU is O(1) with a HashMap + doubly-linked list. Know how to implement it.
- **Cache key collision:** Two different entities with the same cache key. Always namespace: `user:{id}`, `order:{id}`, not just `{id}`.
