## What architecture selection does and what it solves
High-level architecture chooses the system shape that best fits product constraints.

It solves:
- **Coordination complexity**: how teams and components interact.
- **Scaling boundaries**: where and how load can be isolated.
- **Failure blast radius**: how local failures affect global behavior.

## Selection dimensions (decision inputs)
Use these inputs before choosing a pattern:
- Team size and operational maturity.
- Domain complexity and boundary clarity.
- Read/write ratio and traffic burstiness.
- Consistency and auditability requirements.
- Change velocity and release independence needs.

## Pattern catalog: what each pattern does + when to use it
### 1) Layered monolith / modular monolith
**Does**: keeps deployment simple while enforcing logical module boundaries.
**Solves**: fast iteration with low operational burden.
**Use when**: early-stage products, uncertain requirements, small teams.
**Avoid when**: independent scaling/deploy cadence by domain is mandatory.

### 2) Microservices
**Does**: decomposes system by bounded contexts with independent deploy/scale.
**Solves**: organizational scaling and domain autonomy.
**Use when**: large teams + clear ownership domains.
**Avoid when**: platform maturity is low (CI/CD, observability, service governance).

### 3) Event-driven architecture
**Does**: decouples producers/consumers via asynchronous event contracts.
**Solves**: high fanout workflows and failure-tolerant processing.
**Use when**: notifications, ingestion pipelines, workflow orchestration.
**Avoid when**: strict synchronous consistency across many steps is mandatory.

### 4) CQRS + Event Sourcing
**Does**: separate write model from read model; store immutable event history.
**Solves**: auditability, replay, and heterogeneous read views.
**Use when**: finance/ledger/order-history/regulated systems.
**Avoid when**: simple CRUD product with limited replay/audit needs.

### 5) Request-driven + async hybrid (most common)
**Does**: synchronous API for user actions; async workers for side effects.
**Solves**: user-latency constraints with resilient background processing.
**Use when**: almost every modern internet system.

### 6) Edge-centric global architecture
**Does**: push caching/compute/routing to edge POPs.
**Solves**: global latency and origin offload.
**Use when**: media/content/global consumer apps.

## Component choices within each pattern
| Concern | Common choices | Tradeoff highlights |
| --- | --- | --- |
| API ingress | API gateway, service mesh ingress | governance vs latency overhead |
| Sync RPC | REST/gRPC | interoperability vs strict typing/perf |
| Async transport | Kafka/SQS/PubSub/RabbitMQ | ordering/replay vs simplicity |
| State store | SQL/NoSQL mix | consistency/query flexibility vs scale |
| Cache | Redis/CDN/local cache | freshness complexity vs latency gains |
| Workflow | Temporal/Cadence vs custom orchestration | reliability ergonomics vs platform overhead |

## Popular frameworks/platforms and best-fit jobs
- **Spring Boot / NestJS / FastAPI**: API-heavy service layers and modular monoliths.
- **Kubernetes + Istio/Linkerd**: microservice platform governance and mTLS.
- **Kafka + Kafka Streams/Flink**: event-driven pipelines, stream joins, stateful processing.
- **Temporal**: durable workflows with retries, compensation, and visibility.
- **CloudFront/Fastly/Cloudflare**: global content/API edge acceleration.

## Mapping to common interview question types
- **Design Twitter/News Feed**: hybrid sync+async + heavy cache + fanout logic.
- **Design Uber/Delivery**: event-driven + geospatial + real-time state propagation.
- **Design Payment/Order system**: modular service boundaries + strong consistency core + async side effects.
- **Design Search/Recommendation**: dual architecture (online serving + offline/stream pipelines).
- **Design YouTube/Netflix**: edge-centric architecture + object storage + transcoding pipeline.

## Tradeoffs and migration paths
### Typical migration path
1. Modular monolith.
2. Split high-change or high-scale domains.
3. Introduce event backbone for cross-domain side effects.
4. Add specialized serving/analytics planes as scale demands.

### Key tradeoffs
- **Simplicity vs autonomy** (monolith ↔ microservices).
- **Strong consistency vs availability/responsiveness**.
- **Real-time processing vs cost/operational complexity**.
- **Platform standardization vs team-level flexibility**.

## Interview framing template
- “Given team and scale assumptions, I’ll start with **X pattern**.
- The problem it solves here is **Y**.
- I reject **A/B** because of **tradeoff C**.
- I’ll include a migration path to **Z** once traffic/org complexity crosses threshold.”
