## Core Concepts

**Why capacity planning matters:** Systems that aren't sized correctly either fail under load (under-provisioned) or waste money (over-provisioned). In interviews, capacity estimation signals whether you understand the scale implications of your design.

**The goal in interviews:** Arrive at rough numbers that justify your architectural choices. You don't need precision — you need defensible order-of-magnitude estimates.

**Key principle:** Work top-down. Start from user-facing metrics (DAU, requests/sec), derive storage and compute needs, then size infrastructure.

---

## Estimation Methodology

**The 4-step approach:**

1. **Traffic:** DAU × requests/user/day → requests/sec (QPS)
2. **Storage:** writes/sec × payload size × retention period
3. **Bandwidth:** reads/sec × response size (+ CDN hit rate reduction)
4. **Compute:** QPS × latency target → concurrent requests → servers needed

**Useful approximations:**
```
1 million requests/day  ≈ 12 req/sec
1 billion requests/day  ≈ 12,000 req/sec (12K QPS)
Peak is usually 2-3x average

1 KB  = 10^3 bytes
1 MB  = 10^6 bytes
1 GB  = 10^9 bytes
1 TB  = 10^12 bytes
1 PB  = 10^15 bytes

30 days = ~2.5 million seconds
1 year  = ~31 million seconds

1 server handles ~1K-10K req/sec (depends on work type)
1 server handles ~100 concurrent DB connections
```

**When interviewers ask for estimates, they want:**
- Correct order of magnitude (not exact numbers)
- Structured approach (show your work)
- Conclusions that affect design decisions (not just numbers in a vacuum)

## Traffic Estimation

**Start with DAU (Daily Active Users):**
```
Twitter example:
  300M DAU
  Each user reads 20 tweets/day, writes 1 tweet/week on average

Reads:
  300M × 20 = 6B read events/day
  6B / 86,400 ≈ 70K reads/sec (average)
  Peak: 70K × 3 = 210K reads/sec

Writes:
  300M × (1/7) = 43M writes/day
  43M / 86,400 ≈ 500 writes/sec
  Peak: ~1,500 writes/sec

→ Read-heavy system (140:1 read:write ratio)
→ Justifies: read replicas, caching, CDN
```

**Read/write ratio matters for design decisions:**
- > 10:1 read-heavy: optimize for reads (caching, read replicas, denormalization)
- ~1:1 balanced: standard primary + replica setup
- > 2:1 write-heavy: Kafka ingestion, LSM-tree storage, async writes

## Storage Estimation

**Formula:** `storage = write_rate × object_size × retention × replication_factor`

```
YouTube example:
  500 hours of video uploaded per minute
  500 × 60 = 30,000 min of video/min uploaded
  Average compressed video: 300 MB/min (1080p)
  30,000 × 300 MB = 9,000 GB/min = 9 TB/min

  Per day: 9 TB/min × 60 × 24 = ~13 PB/day
  Replication factor: 3 → 39 PB/day raw storage
  Transcoding to multiple resolutions ≈ 2x → 78 PB/day total

  After 5 years: 78 × 365 × 5 ≈ 142,000 PB = 142 exabytes
  → Justify: distributed object storage (S3), erasure coding, tiered storage
```

**Hot/warm/cold tiering:**
```
Hot tier (last 30 days): SSD, fast retrieval, expensive
  → Accessed frequently, pay premium for IOPS

Warm tier (1-12 months): HDD or object storage
  → Accessed occasionally, medium cost

Cold tier (>1 year): Glacier/tape, high latency retrieval
  → Rarely accessed, minimal cost

Cost example (AWS S3):
  S3 Standard:        $0.023/GB/month
  S3 Standard-IA:     $0.0125/GB/month
  S3 Glacier:         $0.004/GB/month
  S3 Glacier Deep:    $0.00099/GB/month
```

## Compute Sizing

**Using Little's Law:** `concurrent_requests = throughput × latency`

```
Target: 10,000 req/sec, p99 latency = 50ms
Concurrent requests = 10,000 × 0.05 = 500

If one server handles 50 concurrent requests (thread pool):
  Servers needed = 500 / 50 = 10 servers

With 2x headroom for peaks: 20 servers
```

**Server capacity rules of thumb:**
```
Stateless API server (compute-light): 10K-50K req/sec per server
Stateless API server (compute-heavy, e.g., auth): 1K-5K req/sec
CPU-bound worker (transcoding, ML): 1-100 jobs/sec (depends on job)
PostgreSQL primary: 5K-50K simple queries/sec
Redis: 100K-1M ops/sec (single-threaded; pipelining helps)
Kafka broker: 100K-1M messages/sec (depends on message size)
```

**Memory sizing:**
```
Working set = hot data that fits in RAM
  If 20% of data accounts for 80% of reads → cache 20% for 80% hit rate
  1M users × 1 KB user profile = 1 GB → fits in one Redis node easily
  1B users × 1 KB user profile = 1 TB → Redis cluster (10 nodes × 128GB)
```

## Cost Modeling

**Unit economics:** Cost per unit of output (cost per request, cost per user, cost per transaction).

```
Example: Ride-sharing app
  1M rides/day
  Infrastructure cost: $50,000/month = ~$1,667/day
  Cost per ride: $1,667 / 1,000,000 = $0.0017 per ride

  Revenue per ride: $2 platform fee
  Gross margin: ($2 - $0.0017) / $2 = 99.9% before other costs
  → Infrastructure is not the margin concern; payment processing is
```

**Main cost categories (cloud):**
```
Compute (EC2, GCE, pods): 40-60% of cloud bill
  Right-size instances; use Reserved Instances for baseline (40-60% discount)
  Spot/Preemptible for batch jobs (70-80% discount; may be interrupted)

Storage: 20-30%
  Hot/warm/cold tiering
  Compression (saves 2-10x)
  Deduplication

Data transfer (egress): 10-20%
  Cloud providers charge for egress, not ingress
  CDN reduces origin egress significantly
  Multi-region transfers are expensive

Managed services (RDS, ElastiCache, Kafka): variable
  2-5x cost of self-managed but less ops burden
```

**Total Cost of Ownership (TCO):**
```
Cloud cost ≠ TCO
TCO = infra cost + eng hours (ops, on-call) + opportunity cost

Managed service at 3x price may be cheaper in TCO if it eliminates:
  - Database DBA work (40 hrs/month)
  - On-call incidents (10 hrs/month)
  - Upgrade overhead (20 hrs/quarter)
```

## Autoscaling

**Horizontal autoscaling (most common):**
- Scale out app servers based on CPU/memory/request rate
- AWS Auto Scaling Groups, Kubernetes HPA (Horizontal Pod Autoscaler)
- Target: keep CPU at 60-70% to absorb spikes without latency degradation

**Scale-up triggers:**
```
CPU > 70% for 3 minutes → add instances
Memory > 80% → add instances
Request queue depth > threshold → add workers
Custom metric (Kafka consumer lag > 1000) → add consumers
```

**Scale-down triggers:**
```
CPU < 30% for 10 minutes → remove instances
Be conservative on scale-down (hysteresis) to prevent flapping
Minimum instances: 2 per AZ for HA
```

**Pre-warming vs reactive scaling:**
- Reactive: scale out after metric crosses threshold. Lag = provisioning time (2-5 min for EC2, 30s for containers)
- Predictive: scale ahead of known traffic patterns (e.g., scale up 9 AM every weekday)
- Pre-warming: manually scale before scheduled events (flash sales, product launches)

**Stateful services don't scale as easily:**
- Databases: read replicas scale reads; writes require sharding (manual process)
- In-memory state: must be externalized to Redis before horizontal scaling
- Kafka: partitions are the unit of parallelism; set partition count at topic creation

## Cost Optimization Patterns

**Compute:**
- **Reserved Instances/Committed Use:** 1-3 year commitment → 40-60% discount vs on-demand. Use for predictable baseline load.
- **Spot/Preemptible Instances:** 70-80% cheaper; may be reclaimed with 2-min warning. Good for: batch jobs, ML training, CI/CD workers.
- **Right-sizing:** Many teams over-provision 2-4x. Use CloudWatch/Datadog to find underutilized instances.
- **Graviton/ARM instances:** AWS Graviton3 is 20-40% cheaper than x86 for same performance on many workloads.

**Storage:**
- **Compression:** Columnar formats (Parquet, ORC) compress 5-10x for analytics. Brotli/gzip for API responses.
- **Tiered storage:** Automatically move objects to cheaper tiers with lifecycle policies.
- **Deduplication:** Content-addressed storage (S3 dedup is implicit via ETags + chunking).
- **Delete what you don't need:** Logs are often the largest storage consumer. Set retention policies.

**Database:**
- **Connection pooling:** Reduces DB instance size needed (fewer idle connections consuming memory)
- **Read replicas vs caching:** Read replica adds latency (replication lag) and full cost. Redis cache is cheaper for hot data.
- **Query optimization:** One missing index can cost 100x more in compute than adding it.
- **Aurora Serverless:** Pay per ACU-hour; cost-effective for variable workloads with quiet periods.

**Networking:**
- **CDN for static assets:** Move 90% of bytes out of origin → 90% less egress cost.
- **VPC endpoints:** Keep traffic within AWS network (no NAT gateway cost, better security).
- **Batch API calls:** Fewer round trips = less latency + less egress data.
- **Compression in transit:** Reduces bandwidth and speeds up transfers.

## Interview Pitfalls

- **Skipping estimation entirely.** If you jump to architecture without sizing, the interviewer will ask "can this handle the load?" and you won't have an answer.
- **Over-precision.** "We need exactly 47.3 servers" looks like you don't understand approximations. Round to clean numbers.
- **Estimation without conclusion.** "We need 10 TB/day of storage" means nothing unless you follow with "so we need distributed object storage, not a RDBMS."
- **Forgetting replication factor.** Raw data × replication factor (3x) = actual storage cost. Always account for this.
- **Not mentioning peak vs average.** "We need 1000 req/sec capacity" should be "1000 avg, 3000 peak."
- **Ignoring data growth.** Estimating for today's load without considering 2-5 year growth leads to brittle architectures.
- **No cost tradeoffs.** "Use 3 AZs" has a cost implication (3x compute, cross-AZ data transfer). Know the tradeoffs.
