## What data modeling does and what it solves
Data modeling maps product behavior to durable state and query paths.

It solves:
- **Query latency mismatch**: schema/indexes do not match access patterns.
- **Write correctness risk**: constraints and transactions are underspecified.
- **Scale ceiling risk**: partitioning and key design are bolted on too late.

## Access-pattern-first method
1. List top read/write queries (with volume and latency target).
2. Classify each query as transactional, analytical, or background.
3. Design keys and indexes per query shape.
4. Validate hot-key and growth behavior.
5. Add lifecycle controls (TTL, archival, deletion compliance).

## Model families: relational/document/wide-column/KV/graph/time-series
### Relational (PostgreSQL/MySQL)
**Best for**: strong consistency, joins, constraints, transactional workflows.
**Common in**: order/payment/account systems.
**Tradeoff**: horizontal write scaling complexity.

### Document (MongoDB)
**Best for**: aggregate-centric reads, evolving schema, nested objects.
**Common in**: content metadata, profile/config workloads.
**Tradeoff**: relational integrity/joins become application-level complexity.

### Wide-column (Cassandra/Bigtable)
**Best for**: very high write throughput and predictable query patterns.
**Common in**: feed events, telemetry, large append-heavy datasets.
**Tradeoff**: denormalization burden, limited ad-hoc querying.

### Key-value (Redis/DynamoDB KV usage)
**Best for**: simple lookup by key, sessions, counters, ephemeral state.
**Tradeoff**: weak support for complex filtering/joins.

### Graph (Neo4j/JanusGraph)
**Best for**: relationship traversals and recommendation paths.
**Tradeoff**: operational complexity and specialized query semantics.

### Time-series (TimescaleDB/InfluxDB/ClickHouse)
**Best for**: metric/event series, downsampling, time-window analytics.
**Tradeoff**: not ideal for highly relational transactional domains.

## Implementation choices inside each family
### Key design
- Surrogate IDs (UUID/snowflake) vs natural keys.
- Composite keys for access locality and partition balance.

### Index strategy
- Cover primary query predicates and sort order.
- Use composite indexes following prefix rules.
- Limit index count on write-heavy tables.

### Denormalization strategy
- Precompute read views where read latency dominates.
- Maintain with event-driven pipelines/outbox updates.

### Data lifecycle
- Hot/warm/cold tiering by recency and access frequency.
- TTL for ephemeral entities.
- Hard-delete pipeline for compliance workflows.

## Popular databases/frameworks and best-fit jobs
- **PostgreSQL**: default OLTP + strong SQL + robust ecosystem.
- **MySQL**: mature transactional system in many high-scale stacks.
- **MongoDB**: flexible document workloads and rapid schema evolution.
- **Cassandra**: write-heavy globally distributed event storage.
- **DynamoDB**: managed low-ops KV/document at large scale.
- **ClickHouse**: high-performance analytical queries over event data.
- **Redis**: ultra-low latency cache/state adjunct (not primary source for most domains).

## Mapping to common interview question types
- **Design Instagram/Twitter feed**: write-optimized event store + denormalized read model.
- **Design e-commerce order/payment**: relational core for transactional correctness.
- **Design metrics/monitoring system**: time-series + rollup storage.
- **Design recommendation graph**: graph traversal + feature store hybrid.
- **Design URL shortener**: KV/relational hybrid with uniqueness + redirect hot reads.

## Tradeoffs and migration strategy
### Core tradeoffs
- Query flexibility vs scale specialization.
- Strong consistency vs global write availability.
- Normalization purity vs read-path performance.
- Operational simplicity vs tailored performance.

### Typical migration path
1. Start relational for correctness and development speed.
2. Add cache and read-optimized materialized views.
3. Split high-volume append-only domain to NoSQL/time-series store.
4. Keep authoritative source explicit to avoid dual-write corruption.

## Interview checklist
- Top 5 queries mapped to schema + index.
- Write path consistency semantics defined.
- Partition/hotspot strategy explained.
- Lifecycle (retention, archival, compliance delete) covered.
- Migration plan stated for 10x growth.
