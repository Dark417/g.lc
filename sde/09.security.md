## Core Concepts

**AuthN vs AuthZ:**
- **Authentication (AuthN):** Who are you? Verify identity. (Login, API key check, mTLS)
- **Authorization (AuthZ):** What are you allowed to do? Check permissions after identity is verified.

**Principle of least privilege:** Every service, user, and process should have only the minimum permissions needed to do its job. Not admin because it's easier.

**Defense in depth:** Multiple security layers. If one fails, the next catches it. Don't rely on a single control.

**Zero Trust:** "Never trust, always verify." Don't assume services inside the network perimeter are safe. Verify every request regardless of origin. mTLS between services. Short-lived credentials.

## Authentication (AuthN)

### API Keys
Simple fixed string included in request headers.
```
X-API-Key: sk_live_abc123xyz
```
- Easy to implement and use
- Must be rotated regularly
- Store hashed in DB, not plaintext
- Scope: usually single service, single permission level
- **Use for:** Service-to-service APIs, developer APIs

### JWT (JSON Web Tokens)
```
Header.Payload.Signature
{ alg: "RS256" }.{ user_id, roles, exp }.sign(private_key)
```
- Self-contained: no DB lookup for verification (verify signature with public key)
- Stateless: server doesn't need to store sessions
- Expiry: short-lived (15 min - 1 hour). Refresh tokens (long-lived, stored securely) provide seamless renewal.
- **Pitfall:** JWTs can't be revoked before expiry unless you maintain a blocklist (which reintroduces state). Short expiry mitigates this.
- **Use for:** User-facing APIs, microservice-to-microservice calls within an org

### OAuth2 / OpenID Connect (OIDC)
- **OAuth2:** Authorization framework. User grants permissions to third-party apps. Access token → API.
- **OIDC:** Authentication layer on top of OAuth2. Adds ID token (user identity).
- **Flows:**
  - Authorization Code + PKCE: web/mobile apps (most secure)
  - Client Credentials: service-to-service (no user involved)
  - Device Code: CLIs, smart TVs
- **Use for:** User login (Google/GitHub SSO), delegated authorization

### mTLS (Mutual TLS)
Both client and server present certificates. Bidirectional authentication.
- Server authenticates client (not just client authenticates server like regular TLS)
- Used in: service mesh (Istio), Kubernetes, financial institutions
- Certificate management: service mesh handles cert provisioning/rotation automatically
- **Use for:** Internal service-to-service authentication in zero-trust environments

### Session Tokens
Server issues session ID on login, stored in cookie (HttpOnly, Secure, SameSite=Strict).
- Requires server-side session store (Redis) → stateful
- Can be immediately revoked (delete from session store)
- CSRF protection required for cookie-based auth
- **Use for:** Web applications where server-side control of sessions is important

## Authorization (AuthZ)

### RBAC (Role-Based Access Control)
```
User → assigned Roles → Roles have Permissions
User: alice → roles: [admin, editor]
admin role → permissions: [create_user, delete_user, read_all]
editor role → permissions: [create_post, edit_post]
```
Simple to understand. Works for most applications. Limitations: coarse-grained (can't say "alice can only edit her own posts").

### ABAC (Attribute-Based Access Control)
Policies based on attributes of user, resource, environment.
```
Policy: user.role == 'editor' AND resource.owner == user.id AND action == 'edit'
→ alice can edit posts where alice is the owner
```
More expressive than RBAC. Complex policy management. Used in AWS IAM policies.

### ReBAC (Relationship-Based Access Control)
Permissions defined by relationships in a graph.
```
alice → OWNER → post:123
alice can EDIT post:123 because alice OWNS it
```
Used by: Google Zanzibar (Google Drive, Calendar, Gmail permissions). Auth0 FGA. Ory Keto.

### JWT Claims for AuthZ
```json
{
  "sub": "user123",
  "roles": ["editor"],
  "permissions": ["posts:write", "comments:write"],
  "org_id": "org456"
}
```
Permissions embedded in token → no AuthZ service round trip. Risk: stale permissions if token isn't refreshed. Short TTL + token refresh mitigates this.

## Encryption

**Encryption at rest:**
- Database: Transparent Data Encryption (TDE) — AWS RDS, PostgreSQL, MySQL have native TDE
- File/disk: AES-256 at the storage layer (AWS S3 SSE-S3 or SSE-KMS)
- Application-level: encrypt sensitive columns in app before writing to DB (for sensitive PII)
- **Envelope encryption:** Encrypt data with a per-record Data Encryption Key (DEK). Encrypt the DEK with a master key in KMS. Store encrypted DEK with the data. Decrypt: fetch KMS → decrypt DEK → decrypt data. Limits blast radius if master key is compromised.

**Encryption in transit:**
- TLS 1.3 everywhere. No 1.0 or 1.1.
- Certificate pinning for mobile apps (prevent MITM with forged certs)
- HSTS (HTTP Strict Transport Security): force HTTPS for 1 year, even for first request
- mTLS for internal service communication

**Hashing:**
- Passwords: bcrypt, scrypt, or Argon2. Never SHA-256/MD5 for passwords (too fast → brute-force).
- Data integrity: SHA-256. Files, checksums.
- API request signatures: HMAC-SHA256 with a secret key.

## Common Vulnerabilities

**Injection attacks:**
- SQL injection: NEVER string-concatenate user input into queries. Use parameterized queries.
- Command injection: Never pass user input to shell commands.
- NoSQL injection: MongoDB queries can be manipulated if using `$where` with user input.

**XSS (Cross-Site Scripting):**
- Never render user-supplied HTML without escaping.
- Content Security Policy (CSP) header limits script sources.
- HttpOnly cookies prevent JS access.

**CSRF (Cross-Site Request Forgery):**
- Malicious site tricks user's browser into making authenticated requests to your API.
- Prevention: CSRF token in forms, SameSite=Strict cookies, Origin header check.

**SSRF (Server-Side Request Forgery):**
- User-supplied URLs that your server fetches. Attacker sends requests to internal services.
- Prevention: allowlist of permitted outbound domains/IPs, block 169.254.x.x (AWS metadata).

**Broken Authentication:**
- Weak passwords, predictable session tokens, no rate limiting on login.
- Prevention: strong password policy, bcrypt, account lockout after N failures, 2FA.

**IDOR (Insecure Direct Object Reference):**
- `/api/orders/123` → accessing someone else's order by guessing ID.
- Prevention: always check that the authenticated user owns the resource.

## Secrets Management

**Never:**
- Hardcode secrets in source code
- Store secrets in environment variables committed to git
- Pass secrets as command-line arguments (visible in process list)

**Always:**
- Use a dedicated secrets manager
- Rotate secrets regularly (or automatically)
- Audit who accessed which secrets and when

**HashiCorp Vault:**
- Dynamic secrets: Vault generates temporary DB credentials that auto-expire
- PKI: issue short-lived TLS certificates
- Secret leasing: credentials have TTL, auto-revoked when expired
- Audit log: every secret access logged

**AWS Secrets Manager:**
- Managed, AWS-native. Supports automatic rotation for RDS, Redshift, DocumentDB.
- Secrets versioned; old version accessible during rotation window.
- Lambda rotation function: custom logic for rotating 3rd-party API keys.

**Kubernetes Secrets:**
- Base64-encoded (NOT encrypted by default). Enable encryption at rest via KMS.
- Use External Secrets Operator to sync from Vault or AWS Secrets Manager into K8s secrets.
- Avoid: storing secrets in ConfigMaps or environment variables in pod spec.

## Audit Logging

**Why:** Compliance (SOC2, PCI-DSS, HIPAA). Security incident investigation. "Who deleted that record?" needs an answer.

**What to log:**
- Authentication events (login success/failure)
- Authorization decisions (access granted/denied)
- All mutations to sensitive data (create, update, delete)
- Admin actions (user role changes, config changes)

**Requirements:**
- Tamper-evident: append-only, ideally write-only storage (S3 with object lock, WORM storage)
- Searchable: structured format, indexed by user_id, resource_id, timestamp
- Long retention: typically 1-7 years depending on compliance requirement
- Immutable: once written, cannot be modified or deleted

**Implementation:**
```json
{
  "event_id": "evt-uuid",
  "timestamp": "2024-07-01T15:30:00Z",
  "actor": { "user_id": "user-123", "ip": "1.2.3.4", "user_agent": "..." },
  "action": "DELETE",
  "resource": { "type": "payment", "id": "payment-456" },
  "outcome": "SUCCESS",
  "service": "payment-service",
  "request_id": "req-789"
}
```

## Popular Frameworks

**Keycloak:** Open-source IAM. SSO, OAuth2, OIDC, SAML, LDAP/AD integration. Self-hosted or managed.

**Auth0:** Commercial SaaS IAM. Easy integration, managed. Enterprise features: anomaly detection, breached password detection.

**Okta:** Enterprise SSO and identity platform. SAML, OIDC, SCIM provisioning.

**AWS IAM:** Identity and access management for AWS resources. Very fine-grained ABAC policies.

**HashiCorp Vault:** Secrets management, PKI, encryption as a service.

**OPA (Open Policy Agent):** Policy engine. Write authorization policies in Rego. Decouple policy from code. Used in Kubernetes admission controllers, API gateway AuthZ.

**OWASP:** Not a framework, but essential reference: OWASP Top 10 (most critical web vulnerabilities), OWASP ASVS (security verification standards), OWASP ZAP (free security scanner).

## Interview Pitfalls

- **"I'll store passwords hashed with MD5/SHA256"** — these are fast hashes, too easy to brute-force. Use bcrypt/argon2.
- **Not mentioning authorization.** Auth is both AuthN AND AuthZ. "Users log in with JWT" is incomplete — what controls what they can access?
- **Not considering token revocation.** JWT can't be revoked before expiry. For sensitive operations (payment, admin), use short-lived tokens + token blocklist.
- **Storing secrets in environment variables** in plain text in Docker Compose or K8s specs. Use Vault / Secrets Manager.
- **Not mentioning rate limiting on auth endpoints.** No rate limiting on login → brute-force. No limit on password reset → abuse.
- **No HTTPS between internal services.** "Internal traffic doesn't need TLS" is wrong in zero-trust model.
